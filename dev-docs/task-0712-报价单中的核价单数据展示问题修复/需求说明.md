## 11. 需求澄清备案（2026-07-12 技术总监 × 提出人 逐点澄清）

### 11.1 澄清结论

| # | 澄清点 | 结论 |
|---|--------|------|
| C1 | 具体错误现象 | **三面均异常**：① 编辑页核价单页签——刚建单**刷新前永久"加载中…"**；② 报价详情页——核价侧"**无组件数据**"；③ 核价管理进入——核价侧"**无组件数据**"。 |
| C2 | 时序 vs 真断裂 | 编辑页"加载中"**刷新/手刷后会好**（时序类）；但只读面（详情/核价管理）刷新也不好，因其不触发前端 warm。 |
| C3 | "一次性导入多比报价数据"语义 | **单报价单内多产品行**（一次导入一个报价单，含多个销售料号/产品行，每行都要渲染核价树）。 |
| C4 | 交互终态 | **后端建单时同步把明细行 + 卡片值（报价+核价含核价树）算好并落库，三面开箱即用**（否定了"前端强化 warm 回灌"方案——只读面无 warm，救不了）。 |
| C5 | N+1 | 后端核价树 `render()` 已是**多根一次批量查询**（实测 1 根 14 节点 / 2 根批量正确），**非 N+1**；本次只需保证建单走整单批量、不退化逐行。 |
| C6 | 边界 | 不改公式计算、不改报价侧展示逻辑、不改核价树递归 SQL 语义与 `bom_recursive_expand` 标记机制。 |
| C7 | 首版是否异步 | 首版走**同步物化**（实现简单、三面强一致、批量已够快）；大单性能问题另立 backlog，暂不做异步补算+轮询。 |

### 11.2 复现与根因归档（技术总监实测，2026-07-12）

**测试单**：QT-20260711-1366/67/68（客户罗克韦尔，核价模板含"配件(树)/元素/核价小计"，根料号 S-3120014539）。

- **后端完全正常**：生效递归 SQL（`costing_bom_tree_config` 的"核价树V1"，`is_active=true`）对种子产出完整 14 节点树；`ensure-card-values` 同步返回并**持久化**全部卡片值（含核价树各节点业务列 + 公式结果）；多根一次批量，无 N+1。
- **真正根因**：`V6QuotationCommitService.createQuotation`（全 103 行）**只建空报价单 + 写 hfPairs**，**不建明细行、不算任何卡片值**。明细行靠前端 `autoPopulate` 客户端建、卡片值靠前端 `warm` 异步补算。
  - 编辑页：warm 补算前"加载中"，warm 不稳定 → 刷新前永久加载中。
  - 详情页 / 核价管理（只读，**不触发 warm**）：直接读持久化的 `costing_card_values`（NULL）→ `ReadonlyProductCard.tsx:271 useSnap=false` → "无组件数据"。
- **修复方向**：把"明细行 + 4 份卡片值"的物化从"前端事后补算"上移到 `createQuotation` **服务端同步完成并落库**，复用现成 `CustomerPartCandidateService`（候选）+ `CardSnapshotService`（算+落库，已确认持久化）+ `CostingTreeRenderService.render`（批量）。详见同目录 `backtask.md` / `fronttask.md` / `api.md`。

> 复现脚本：`cpq-frontend/e2e/repro-costing-tree-import.spec.ts`（驱动真实导入 UI）。
