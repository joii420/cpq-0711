## 一、背景与问题（森萨塔 991 根因）

### 现象
导入 `森萨塔.xlsx`，第一步「物料BOM/组成件BOM 合并」在第 2 行（首个数据行）报错：

```
2  投入料号  报价料号跨客户串号
```

### 根因链（已代码 + DB 双向坐实）
1. `物料BOM` sheet 首数据行：销售料号=`1111122`，**材质料号=`991`**（Cu/Ag/Cu带）。文件无「投入料号」列，`MaterialBomMergeHandler.java:80-81` 回退读「材质料号」。
2. `MaterialNoResolver.resolve()`（`:40-43`）：**只要料号列非空，就无条件** `allocator.ensureRegistered(customerNo, "991")`。
3. `QuoteMaterialNoAllocator.ensureRegistered()`（`:64-75`）把 `991` 作为 QUOTE 行 upsert 进 `material_customer_map`；该表有部分唯一索引 `uq_mcm_quote_no = UNIQUE(material_no) WHERE system_type='QUOTE'`——**一个料号号码全局只能归属一个客户**。
4. DB 现状：`991` 已被 **CUST-1269** 占（历史测试导入登记）。本次以别的客户导 → owner≠customerNo → 抛 `CrossCustomerQuoteNoException` → 记为 `(rowNo=2, "投入料号", "报价料号跨客户串号")`。

### 用词误导（两层）
- 列标签硬编码成「投入料号」，实际数据来自「材质料号」列（handler fallback）。
- 错误话术统一叫「报价料号」，但 `991` 是**材质号**，只是被塞进了和销售料号共用的同一张全局唯一表。

### 本质判断（业务定调）
> `material_customer_map` 是**客户占号表**，应只登记**产品级料号**；把 BOM 组件级料号（材质/投入/组成件号）也塞进去做客户占号，是**语义错配**——通用原材料（如 991）被第一个导入的客户永久锁定，别的客户再也不能在 BOM 里引用，且触发本不该有的「跨客户串号」误伤。

## 二、精确不变量（本方案的核心目标）

> **`material_customer_map`（system_type='QUOTE'）只存产品级料号：**
> **① 客户既有产品销售料号（带 customer_product_no）；② 选配产品报价号（customer_product_no=NULL 但属选配产品）。**
> **不存 BOM 组件级料号（材质料号 / 投入料号 / 组成件料号）——它们只落 `material_master`（全局共享，客户无关）。**

关键澄清：不能用「customer_product_no 是否为空」当判据（选配产品报价号也是 NULL）。要按料号在业务里的**角色**（产品级 vs 组件级）区分。

补充语义确认：**材质料号 = 投入料号**，是同一字段在不同 sheet 的两种叫法，代码现状（`投入料号`→`材质料号` fallback）已如此处理，本方案沿用。

## 四、改造方案

### 4.1 写入侧（核心）

> **🔑 MUST-FIX（architect）：`mintAndRegister` 是共享方法，必须按调用方分流,不能整体改。**
> - `MaterialNoResolver.resolve:60`（**组件**铸号）→ 改走 **mint-only**（只铸号，不写 mcm 占号行）。
> - `ConfigureProductService:351 / :1139`（**选配产品**铸号）→ **保留 mint+register**（继续写 mcm 产品级②行）。现网 15 条选配报价号唯一来源就是这里，`ExistingProductService` 靠这条 mcm 行列出选配产品，删不得。

1. **`MaterialNoResolver.resolve()`**：解析**组件号**后**不再登记占号表**。
   - 料号有值：`trim` 后直接返回（去掉 `allocator.ensureRegistered`）。
   - 料号空+名称有值：按名匹配 `material_master`（现状 `findFirstByMaterialName`，客户无关，保留）；匹配不到时改调 **mint-only** 取稳定 component_no——见决策点 A。
   - 组件号写 `material_master` 的动作在各 handler 里已由 `accMaterialMaster` 完成，不动。
2. **`QuoteMaterialNoAllocator`**：
   - `ensureRegistered` 不再被组件链调用 → 变死方法，删除或保留仅供 Q02 复用（Q02 现走 `upsertQuote`，不依赖它，倾向删除）。
   - `mintAndRegister` **拆成两个入口**：`mintOnly(customerNo, yyMm)`（只从 `quote_material_no_seq` 铸号、不 INSERT mcm，供 `resolve` 用）+ 保留 `mintAndRegister`（铸号 + INSERT mcm 产品级行，供 `ConfigureProductService` 用）。二者共用 `getOrAllocateCustomerCode` 与铸号序列。
   - `getOrAllocateCustomerCode` **保留**（两条铸号路径都要客户四位码）。
3. **8 个 handler 的 `catch (CrossCustomerQuoteNoException)` 分支变死代码**：清理，避免误导后人（`docs/反模式.md` 记一条）。
4. **纵深防御**：给运行时 `component_sql_view` 表 4 条 `cp_view` 模板补回 `AND mcm.customer_product_no IS NOT NULL`（见 §3.2 证据订正）。

### 4.2 数据迁移（决策点 B）
一次性 Flyway 迁移，按 3.3 甄别规则清历史组件行：

```sql
-- 删除 QUOTE 侧组件级占号行（产品级：有 customer_product_no 或属选配/CFG- 一律保留）
DELETE FROM material_customer_map mcm
WHERE mcm.system_type = 'QUOTE'
  AND mcm.customer_product_no IS NULL
  AND mcm.material_no NOT LIKE 'CFG-%'
  AND NOT EXISTS (
    SELECT 1 FROM sel_part_signature sps
    WHERE sps.quote_part_no = mcm.material_no AND sps.customer_no = mcm.customer_no);
```

**安全网**（防生产环境误伤）：迁移前 `RAISE NOTICE` 打印待删清单；因已确认无读取方依赖组件行（3.2），删除对渲染/提交/候选零影响。architect 已给出现网 31 条待删逐行定性（19 条在 material_master 的真组件号 + 12 条无 BOM 引用的孤儿铸号），无漏删无误删。**保守替代**（决策点 B 另一选项）：只「向前修」不动存量——新导入不再污染，老脏行留着反正没人读，代价是森萨塔这类文件仍需手工清或换号才能重导。

> **⚠️ 部署时序（architect MUST-FIX）**：§4.1 写入侧「停登记」代码改动**必须与本 DELETE 清理同一次部署落地**。否则共享 cpq_db 上并发导入会在「清理完成 ~ 代码生效」之间重新写入组件占号行，清了白清。放同一 Flyway 版本 + 代码同批合并即可。

### 4.3 唯一索引语义（无需改）
`uq_mcm_quote_no` 保留。改造后表内 QUOTE 行只剩产品级料号，索引继续保证「一个销售料号/选配报价号 → 一个客户」，正是业务要的（与上一轮确认一致）。
