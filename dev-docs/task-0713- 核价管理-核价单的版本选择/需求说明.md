# 需求澄清纪要与设计定稿（2026-07-14 技术总监 × 业务方逐条澄清）

> 本节是本任务澄清后的**权威备案**，前后端任务文档（`fronttask.md`/`backtask.md`/`api.md`）以本节为准。上文原始立项内容保留作追溯。

## A. 一句话结论

在"核价管理→打开核价单"的详情页里：**报价侧读冻结副本保持不变；核价侧改成"可按版本重算"的视图**——核价树里每个"输出了约定版本列"的销售料号支持**版本下拉切换**，切换后按最小范围重查、成本 rollup 到根、更新单据总价，并把结果落库持久化。顺带修复该页面当前打开空白的渲染 bug。

## B. 现状调研结论（决定"为什么这么设计"）

1. **报价详情页是共享组件**：报价管理 `QuotationDetail`（live）与核价管理 `CostingReviewPage`（frozen）共用 `ProductDetailViews`（报价单/核价单 × 卡片/Excel 两级视图）。所以"核价管理打开的详情要和报价管理一致"架构上已具备。
2. **核价管理打开空白**：`CostingReviewPage` 走 `costingOrderService.getById(coid).frozenDto` → `ProductDetailViews frozen 模式`。空白根因方向 = 核价侧从 frozen 渲染出不来（frozen 时核价卡片值空 / 核价树 frozen 渲染缺链）。本任务把核价侧改为"从 V6 按 override 正确渲染 + 缓存"即修复。
3. **报价侧必须冻结、不能直读报价单**（已论证）：`saveDraft` 硬闸"仅 DRAFT 可改"；提交后唯一再编辑路径 = 财务驳回→`beginEdit` 转草稿→改→重提，而重提是**累积建新核价单**、老单变历史单。故历史核价单关联的报价单事后可能被改；若报价侧直读报价单会串味。现系统已有 `frozen_dto`（提交时整份 DTO 快照），报价侧**继续读它**即可，无需新增。
4. **遗留版本切换基本要推倒重做**：旧 `PartVersionDrawer`+`updateLineItemPartVersion` 是"整行/整卡切一个版本、仅 DRAFT、读 V44 废弃 `mat_part_version_log`、切一下 `regenerateAllSnapshots` 整单重算"，与本需求（核价树节点级、提交态可用、V6 取数、局部重查）完全不符。
5. **`:spineKeys` 宏已是死代码**：`CardSnapshotService` 注释确认 `SpineKeysContext.get()` 在 `SqlViewExecutor` 侧"从未被读取、纯 no-op、死代码"；新核价管线用 `CostingTreeVarsContext`（`:total_material_no`）取代之。**不在其上建设，可顺手清理。**
6. **活的核价树机制**：`costing_bom_tree_config` 里一条递归 SQL 产出 spine（`root_no/material_no/bom_version/parent_no/node_path`）→ `CostingTreeRenderService` 逐组件跑 $view、在 Java 按 `(parent_no,material_no)` 边/`material_no` 贴到 spine 节点；`CostingTreeVarsContext` 注入 `:total_material_no` 料号集合给非树页签。**"主树产出的料号数组" = `total_material_no`（整单）/`cardMaterialNo`（单卡）。**

## C. 版本机制（每 sheet 独立 · 约定列 + 一个宏 · 平台通用）

**背景事实**：版本是**每个核价 sheet 各自管理**的，版本列名各不相同——`material_bom_item.bom_version` / `unit_price.version_no` / `element_bom_item.characteristic`（元素的版本就是 characteristic，当前值 '2000' 与 bom_version 基线一致，**零新增列**）/ `capacity.version_no` / `plating_scheme.scheme_version` 等；全局费率表（汇率/电费/人工/fee_config）虽有 version_no 但不挂料号、按全局取，**不参与按料号版本切换**。主树对其他页签**只提供料号数组**，各页签走各自的版本管理。

**统一机制（组件作者只维护一条渲染 SQL）**：

- **约定列 `view_version`**：每个可版本化 $view 把本表版本列 `AS view_version` 输出。**有此列=该页签开启版本下拉+切换；无则不参与。**（各表版本列名差异在输出层被统一成一个名字。）
- **`:versionFilter(is_current列, 版本列, 料号键列)` 宏**：取代硬编码的 `is_current` 谓词。
  - **渲染模式**：该料号在本组件有 override → `版本列 = 选定版本`；否则 → `is_current = true`。override 来自扩展后的 `CostingTreeVarsContext`，按 `(component_id, part_no)` 取。
  - **列出模式（供下拉）**：展开为 `TRUE`（放开版本过滤），返回全版本行 → 平台收集 distinct `view_version`。
- **平台通用承担**（组件零参与）：版本下拉（同一 $view 列出模式 + 料号集合限定为被点料号，取 distinct view_version，**倒序**）、切换（写 override + 重跑 $view）、持久化、rollup+单据总价重算。
- **收益**：加一个新的可版本化页签 = 只要它的 $view「输出 `view_version` + 用 `:versionFilter`」，平台代码零改动。元素页签只是第一个落地示例（`ys_view`：`element_bom_item.characteristic AS view_version`）。
- **主树页签**同理（`view_version=bom_version`），特殊点在于"切结构→料号数组变→级联本卡所有页签重查"，级联逻辑由平台单独处理。

## D. 数据模型（轻）

- **新增 override 表** `costing_order_version_override(costing_order_id, component_id, part_no, view_version, ...)`：记"这张核价单里，哪个页签(component)的哪个销售料号被切到哪个版本"。
- **核价单缓存**：核价侧重算结果缓存回核价单（供重开秒开、不必每次现算）+ 更新单据总价。
- 报价侧 `frozen_dto` **不动**。

## E. 重查 / 重算范围规则

| 切换动作 | 重查（发新 SQL 取数） | 重算 | 完全不动 |
|---|---|---|---|
| **主树·任意节点** 版本 | 该产品卡片**所有页签**重查 | 全卡 rollup→根→单据总价 | 其他产品卡片 |
| **非主树页签·某料号** 版本 | 仅**本页签下该销售料号的整组数据**（一个料号在该页签可能有多行，整组一起重查，非单行） | 该料号成本变→rollup 到根→单据总价 | 其他卡片、其他页签的原始数据行、本页签其他料号 |

**原则**：① 触发重查的页签就触发重算；②"父节点不重查/不重建结构，但其成本因子节点变化而 rollup 更新"；③ 成本永远 rollup 到根、更新单据总价（3a）；④ **其他产品卡片永不受影响**；⑤ 业务响应 <3s、**禁 N+1**。

## F. 状态 / 权限 / 生命周期

- **仅 PENDING（待核价）可切版本**；角色 = 财务 / 管理员（核价管理菜单权限）。APPROVED 即锁定不可再改。
- **B1 生命周期**：每张核价单 override 独立；报价单驳回→改→重提生成的**新核价单从 is_current 起、不继承旧 override**；历史单保留自身 override + 缓存可回看。

# 架构评审落定（cpq-architect 2026-07-14 · D1/D2/D3 全部 GO）

> 以下为评审后的**绑定结论**，覆盖并细化上文 §I 的开放点；`api.md`/`backtask.md`/`fronttask.md` 已按此调整。

## 新发现·打开空白的真根因（务必据此修）
- 核价侧值本就存在报价行 `quotation_line_item.costing_card_values`(JSONB)，且 `QuotationDTO.LineItemDTO` 已把它映射进 `frozen_dto`。**空白 = 提交冻结时这些值还是 lazy NULL 没物化**（快路径首存留 NULL，见 `cpq-lazy-card-values-branch`），NULL 被冻进 frozen_dto → 打开就空。**不是 frozen 机制缺链**。
- **修复正手**：`CostingFreezeService.createForSubmission` 在 `buildFrozenDto` 之前先 `ensureCardValues(quotationId)` 物化（同 task-0712「materialize 须在写 v6 后调 + flush」纪律），再把已算好的核价值组装进 `costing_render`。此步是读已算好的行值再组装，几乎零增量成本。

## 硬约束·历史核价单无法 live 重算 → 缓存是唯一解
- 报价单驳回→beginEdit→改→重提时 `deleteLineItems` **物理删除并重建 line items（新 ID）**。故历史核价单关联报价行会消失、V6 也漂移 → 历史单核价侧**物理上无法 live 重算**。
- 结论（自洽）：**PENDING（唯一 active，line item 还在）→ 打开读缓存、切版本时 live 重算并回写缓存**；**APPROVED/REJECTED/WITHDRAWN → 只读 `costing_render` 冻结缓存、永不 live**。**打开永远读缓存、绝不 on-open 重算**（守 BL-0010 首开阻塞）。

## D1 落定
- 缓存落点：`costing_order` **新增两列** `costing_render jsonb`（`{lineItemId: {costingCardValues, costingExcelValues}}`）+ `costing_total_amount numeric(18,4)`。**绝不复用 `total_amount`**（那是含 Step3 折扣的报价总额，语义不同）。
- `frozen_dto` 保持"写一次永不改"（报价侧）；`costing_render` 是核价单级、APPROVED 前可变的独立快照。两条轴物理隔离。
- 前端守 AP-50：getById 返回 `costingRender` 后按 lineItemId 并回 `costingCardValues` 字段，`ReadonlyProductCard` COSTING 分支一行不改。

## D2 落定
- `VersionFilterMacro`（新类，逐条镜像 `SpineKeysMacro`）：`expandForExecution`(占位符 `:__vfPart/:__vfVer` 双 text[]) / `expandForListing`(→`TRUE`) / `expandForValidation`(→`is_current=true`)。渲染展开为单表达式 `EXISTS(unnest(:__vfPart,:__vfVer) 命中选定版本) OR (料号无override AND is_current)`；数组绑定复用 `SqlViewExecutor` 既有 null 保真通路（守 BL-0028）。
- 注入点：`SqlViewExecutor.injectCostingTreeVars`（既有，`executeAllRows`+`executeJdbc` 两处都调）加兄弟注入 `:__vfPart/:__vfVer`；**宏展开需新接进 `executeAllRows`/`queryRecursive`**（注意 spineKeys 的 expandForExecution 从未被接，别照抄"已接"假设）。RENDER/LIST 模式经 `CostingTreeVarsContext.Vars` 传递，在 SQL 构建期读。
- **主树递归 SQL 版本感知 = R1**：config SQL 把 `material_bom_item.is_current` 换成 `:versionFilter(...)`，`queryRecursive` 按占位符出现次数绑定 `:__vfPart/:__vfVer`（照抄现有 `:production_part_nos` 出现次数绑定范式）。**⚠️ 最高风险 = 父节点选版→子件成员集变（拓扑变）；必须先用多版本数据做最小 spike 验证再展开 B3**。
- 下拉列出模式 = 独立轻查（LIST 模式 + partNo 限定取 distinct view_version），**不走带 30s 缓存的 expandUncached**（守 AP-37 串号）。
- `SqlViewValidator` 镜像 spineKeys 的 3.5/3.6：保留 `:__vf` 前缀 + `expandForValidation` 后 EXPLAIN dry-run。
- spineKeys 清理走 B8 独立收尾，与 D2 新宏**解耦**（降爆炸半径）。

## D3 落定
- 新服务 `CostingVersionService.switchVersion(...)`：**单 `@Transactional` + `SELECT ... FOR UPDATE` 锁 costing_order** + upsert override + **flush** + 按范围重算 + 写 `costing_render`(仅受影响 line) + `costing_total_amount` + 返回受影响卡片。
- **重查 scope（最小远程 SQL）vs 重装 scope（整卡内存重装）分离**：主树切→该 line `render(templateId, [thisLine])` 各组件跑 $view；非树切→**仅该组件 $view 跑一次**(partNo 组限定)。两种情形远程查询次数**都与料号数无关**。**重装一律整卡**（含未重查页签用缓存值），保跨页签公式正确。
- rollup **落后端**（`buildCostingCardValues`/`assembleTabsWithFormulaResults`）；`costing_total_amount` = **Σ 核价成本 subtotal，不含 Step3 折扣**（核价=成本，折扣属报价轴）。前端只展示返回值。
- **不回写** `quotation_line_item.costing_card_values`（切版本只动 costing_order.costing_render，报价行实体保持干净）。
- 守 AP-51（受影响 line/页签整体 REPLACE、禁 Math.max 累加）、`cpq-expand-layer-not-threadsafe`（重查串行禁并行）、AP-31/37（前端只用返回值增量刷当前卡、不 getById 整单）。
- **开工前置**：后端须确认核价 FORMULA 有无跨页签依赖（决定非树重装粒度；默认取整卡重装=正确优先）。

## 前后端 5 条最关键叮嘱
1. `costing_render`/`costing_total_amount` 必须新列、绝不复用 `total_amount`；核价侧走 costing_render、报价侧 frozen_dto 写一次永不改，两轴物理隔离。
2. `createForSubmission` 冻结前先 `ensureCardValues` 物化——修空白正手，非可选。
3. 递归 SQL 版本感知最高风险：先多版本数据 spike 验证"父切版→子件集合变"，通过再展开 B3；一份 `versionFilter` 宏同喂页签 $view 与递归 SQL，别写两套。
4. 切版本 = 单 @Transactional + SELECT FOR UPDATE + upsert 后 flush 再重算；重查最小化、重装整卡；远程查询次数与料号数无关。
5. AP-51 整体 REPLACE 不 Math.max、AP-31 前端只增量刷当前卡不 getById 整单、expand 层串行——E2E 必须覆盖切版本前后行数/值稳定。
