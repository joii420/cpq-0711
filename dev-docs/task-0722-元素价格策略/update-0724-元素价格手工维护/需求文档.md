## 2. 需求背景与目标 ⭐

- **要解决什么问题 / 痛点**：
  - task-0722 交付后，元素价格**只能通过「价格导入」进入系统**。价格表（明细 / 矩阵两个 Tab）是纯只读的。
  - 痛点一（**补录**）：只需要改一两条价时，也必须走「下载模板 → 填 Excel → 上传」三步，成本与收益严重不匹配。
  - 痛点二（**纠错**）：导入完成后发现某条价填错了，当前唯一的纠正手段是**重导一次整批文件**去覆盖，没有就地改的能力。
- **本次要达成的目标**：
  - 在元素价格表内直接完成单条价格的新建 / 修改 / 删除，且每次改动强制留痕可追溯。
  - 手工录入的价格必须能被现有取价引擎 `f_customer_element_price` 正常取到，与导入进来的价格完全等价。
- **不做什么（明确边界）**：
  - **不替代「价格导入」**。导入仍是批量录价的主通道，手工维护是受控旁路。
  - 不做自动抓价（沿用 task-0722 §11.13 的裁决，本期依然不做）。
  - 不改取价引擎 `f_customer_element_price`，不改价格策略配置，不改报价单 / 核价单渲染链路。
  - 详见 §11 U11「明确不做清单」。

## 4. 功能详述 ⭐

### 4.1 用户场景 / 操作流程

**场景 A · 补录**
主数据维护 → 元素 → 「📊 元素价格表」→ 明细 Tab → 顶部工具栏「＋ 新建」→ 填 元素 / 源 / 日期 / 单价 / 货币 / 计价单位 → 保存 → 列表出现新行。

**场景 B · 纠错**
元素价格表 → 明细 Tab → 勾选填错的那一行 → 顶部工具栏「✎ 编辑」→ 单价改正（元素 / 源 / 日期 三个字段置灰不可改）→ 保存 → 该行 `fetch_status` 变为 `MANUAL`。

**场景 C · 清理**
键（元素 / 源 / 日期）录错的行无法通过编辑纠正 → 勾选 →「🗑 删除」→ Modal 列出所选项二次确认 → 删除 → 重新走场景 A 建正确的行。

**场景 D · 追溯**
元素价格表 → 「变更历史」Tab（本次新增第三个 Tab）→ 按 源 / 日期区间 / 元素 筛选 → 查看谁在什么时候把哪条价从多少改成了多少。

### 4.3 业务规则与计算逻辑（后端相关）

1. **源与日期必填**。这不仅是业务约束也是技术铁律：取价引擎按 `dp.source_id = w.source_id` 匹配，`source_id` 为空的行**永远取不到**（U0 实证 ①）。
2. **唯一键冲突拒绝而非覆盖**。新建时若 `(元素, 源, 日期)` 已存在 → 返回 `409`，提示改用编辑。不静默覆盖。
3. **键锁定**。`PUT` 只接受 `price` / `currency` / `priceUnit` 三个字段；请求体不含键字段，前端传了也忽略——不依赖前端置灰来保证。
4. **修改即改变身份**。任何一行（含 `fetch_status='IMPORT'` 的导入行）被手工修改后，`fetch_status` 一律写 `MANUAL`，使「被人工动过」这一事实在库里可见。重导仍按唯一键覆盖，不受影响。
5. **强制留痕**。新建 / 修改 / 删除三种动作各写一条变更历史，**与价格写入同一事务**（比照 task-0722 §11.22 对策略侧的强制要求，避免只落其一导致留痕断链）。
6. **单价必须 > 0**。
7. **源下拉只列 `ACTIVE` 源**（与导入一致）；编辑既有行时源已锁定，源被停用不影响该行编辑。
8. **元素下拉复用既有端点 `GET /api/cpq/elements`**（task-0722 §4.2 已改造的元素列表端点），取 `ACTIVE` 元素。
   > ⚠️ **不要**用 v1 的 `GET /element-prices/available-elements` —— 该端点本期随 v1 一并下线（U8）。task-0722 的策略例外抽屉选元素走的也是 `GET /api/cpq/elements`，本次与之保持一致。
9. **手工价与导入价完全等价**。取价引擎不过滤 `fetch_status`（U0 实证 ①），手工录入的价格与导入价格在取价时一视同仁，无优先级差异。

### 4.4 数据字段

**复用表 `element_daily_price` —— 结构不改**，现有字段已足够承载：

| 字段 | 含义 | 类型 | 必填 | 示例 / 取值 |
|------|------|------|:----:|------------|
| `element_name` | 元素符号（键之一） | varchar(64) | 是 | `Ag` |
| `source_id` | 价格源（键之一） | uuid | 是 | 指向 `element_price_source` |
| `price_date` | 价格日期（键之一） | date | 是 | `2026-07-24` |
| `raw_price` | 单价 | numeric(18,4) | 是 | `5860.0000`，须 > 0 |
| `currency` | 货币 | varchar(8) | 是 | `CNY` |
| `price_unit` | 价格计价单位 | varchar(16) | 是 | `kg` |
| `fetch_status` | 数据来源身份 | varchar(16) | 是 | 手工新建 / 手工修改后一律 `MANUAL` |
| `manually_filled_by` / `updated_by` | 操作人 | uuid | — | 后端由会话注入 |

**新建表 `element_daily_price_log`（变更历史）**，形状比照现役 `element_price_strategy_log`：

| 字段 | 含义 | 类型 | 必填 | 示例 / 取值 |
|------|------|------|:----:|------------|
| `id` | 主键 | uuid | 是 | `gen_random_uuid()` |
| `price_id` | 指向 `element_daily_price.id` | uuid | 否 | **不设 FK**：DELETE 后原行已不存在 |
| `element_name` | 键三元组冗余存 | varchar(64) | 是 | 删除后仍可追溯 |
| `source_id` | 键三元组冗余存 | uuid | 否 | 同上 |
| `price_date` | 键三元组冗余存 | date | 是 | 同上 |
| `action` | 动作 | varchar(16) | 是 | `CREATE` / `UPDATE` / `DELETE`，CHECK 约束 |
| `snapshot` | 完整值快照 | jsonb | 是 | 变更后的值；`DELETE` 存**删除前**的值 |
| `changed_at` | 变更时间 | timestamptz | 是 | `now()` |
| `changed_by` / `changed_by_name` | 变更人 | uuid / varchar(100) | 否 | 比照策略侧双写 |

> **`changes` 不入库**。已核实策略侧做法（`StrategyService.java:530`）：表里只存 `snapshot`，「只列变化项」的 `changes` 是查询时用相邻两条 snapshot 比对算出来的。价格侧照此办理，保持两边同构。

索引：`(element_name, COALESCE(source_id::text,''), price_date, changed_at DESC)` + `(changed_at DESC)`。

## 8. 验收标准 ⭐

**功能**

1. 新建一条价（选元素 / 源 / 日期 / 单价 / 货币 / 计价单位）→ 列表出现该行，`fetch_status='MANUAL'`。
2. 同一 `(元素, 源, 日期)` 再次新建 → `409`，提示改用编辑，**不覆盖原值**。
3. 修改一条 `fetch_status='IMPORT'` 的导入行的单价 → 保存成功，SQL 直查该行 `fetch_status` 已变为 `MANUAL`，单价为新值。
4. 编辑抽屉内 元素 / 源 / 日期 三个字段置灰不可改；构造请求强行传这三个字段 → 后端忽略，键不变。
5. 删除一条价 → Modal 列出所选项二次确认 → 删除成功，列表不再出现。
6. 单价填 0 或负数 → 校验拒绝。

**取价链路（核心）**

7. 为某客户配好策略后手工新建一条该源该元素的价 → `SELECT * FROM f_customer_element_price('<客户>', CURRENT_DATE)` 能取到该价（含系数加价换算后的值）。这是「手工价与导入价等价」的硬验证。
8. 删除策略窗口内的一条价 → 再次调用取价函数，结果按剩余数据重算（`AVG` 口径下均值变化 / `LATEST` 口径下回退到前一条）。

**留痕**

9. 三种动作各产生一条 `element_daily_price_log`，`action` 分别为 `CREATE` / `UPDATE` / `DELETE`；`DELETE` 记录的 `snapshot` 为**删除前**的完整值。
10. 「变更历史」Tab 能按 源 / 日期区间 / 元素 筛出对应记录，`UPDATE` 记录只列出真正变化的字段。
11. **事务原子性**：构造历史写入失败的场景，价格写入必须一并回滚，不得出现「价改了但没留痕」或「留痕了但价没改」。

**v1 下线**

12. 路由 `/element-price-center` 不可达；全工程 grep 无 `ElementPriceCenterPage` / `ManualPriceEntryDrawer` / `ElementPriceHint` 残留引用。
13. 下线的四个 v1 端点返回 404。

**强制自检（CLAUDE.md）**

14. 前端：`npx tsc --noEmit` 0 错误；每个改动的 `.tsx` 经 Vite transform 返回 200。
15. 后端：Quarkus 重启无异常；新 Flyway 迁移 `SELECT success FROM flyway_schema_history WHERE version='<NN>'` = `t`。
16. **E2E 强制项**：因改动 `QuotationStep2.tsx`（协议级文件），必须跑 `quotation-flow.spec.ts`，与干净 master 做 **A/B 同型对比**，**新增失败数 = 0**。
    > 该 spec 在干净 master 上已知恒 3 失败（夹具单缺产品分类，见记忆 `task0712-update071501-category-axis`），判定回归时**必须同型对比**，不得把基线失败误归因为本次改动。

## 9. 依赖与风险

| 风险 | 说明 | 缓解 |
|---|---|---|
| **改价静默影响在途报价单** | 改一条价会改变所有引用该源该日期窗口的客户报价成本。已提交 / 冻结的报价单有快照不受影响；DRAFT 重开会取到新价 | 该行为与「导入覆盖一条价」**完全同语义**，非本次引入，不额外拦截（见 U10）。验收 7/8 覆盖 |
| **留痕断链** | 价格写入与历史写入分属两条语句，任一失败即留痕不全 | 强制同事务，验收 11 专项验证。task-0722 §11.22 已在策略侧踩过同类风险 |
| **删除不可恢复** | 删掉一条价 = 策略取价窗口少一条数据，价格静默变化 | Modal 二次确认 + 列出所选项；`DELETE` 记录存删除前完整 snapshot，可据此人工重建 |
| **动了协议级文件** | `QuotationStep2.tsx` 在 E2E 强制清单内 | 本次仅删除一段恒不成立的死分支（U0 实证 ⑤），改动面极小；仍按规范强制跑 E2E |
| **`ElementPriceHint` 若不清理会休眠复活** | 该组件跨源混取最新 `MANUAL` 行；新功能上线后 `MANUAL` 行将大量增加，一旦将来有人给组件加了名为 `element_name` 的字段，它会突然生效并显示错乱的跨源价 | 本期直接删除，不留休眠代码（U8 裁决） |

**依赖**：无外部依赖。取价引擎 `f_customer_element_price`、价格源管理、元素主表均已随 task-0722 交付并在 master。

#### U1 · 功能定位 —— 导入的受控旁路，不是替代 ✅ 已确认

需求方确认场景为**补录**（只改一两条不值得做 Excel）+ **纠错**（导入完发现填错要就地改）两项。

**明确排除**的两个场景：不是「无源可导（电话询价 / 人工抄价）」，也**不是**「取代导入成为主要录价方式」。

→ 导入仍是批量录价主通道。task-0722 §11.14A 立的三条不变量全部保留：

| 原不变量 | 本次如何保住 |
|---|---|
| 每条价有**源** | 新建时源必填（U2） |
| 每条价有**日期** | 新建时日期必填（U2） |
| 每条价有**批次记录** | 换载体：逐条变更历史 `element_daily_price_log`（U5） |

---

#### U2 · 新建时源与日期必填 ✅ 技术铁律，无裁量空间

由 U0 ① 直接推出：`source_id` 为空的行取价引擎永远匹配不到，录了等于白录。日期同理（`price_date` 是 `NOT NULL` 且是取价窗口的唯一时间轴）。

此条**不作为可讨论项**，实现时后端强制校验。

#### U3 · 可修改导入行，改完 `fetch_status` 翻为 `MANUAL` ✅ 已确认

纠错场景要求必须能改 `fetch_status='IMPORT'` 的行。需求方裁决：**改完把身份翻成 `MANUAL`**。

- 收益：价格表能一眼看出哪些行被人工动过，保住了 §11.14A 关心的可追溯性；成本仅一个字段值。
- 不影响重导：重导仍按唯一键 `(element, source, date)` 覆盖，与 `fetch_status` 无关。
- 被否方案：① 保持原值不变（库里看不出被改过，导入回显「新增 9 条」与实际对不上账）；② 只能改 `MANUAL` 行（纠错场景实际未被满足）。

#### U4 · 键锁定 + 另提供删除 ✅ 已确认

唯一键是 `(元素, 源, 日期)`，改这三个字段等于换了一行的身份，与「改单价」不是同一回事。

裁决：**编辑只能改值**（单价 / 货币 / 计价单位），键字段置灰；**键录错了走删除重录**。删除作为危险动作，走 Modal 二次确认并列出所选项，且入变更历史。

- 后端硬保证：`PUT` 请求体不含键字段，传了也忽略——不依赖前端置灰。
- 被否方案：① 键可改不给删（改后撞行要处理 409，且垃圾行永远清不掉）；② 键可改且可删（编辑抽屉要处理「改键撞行」分支，变更历史要表达「从 A 行变成 B 行」，复杂度明显上升）。

#### U5 · 完整变更历史，与策略侧对齐 ✅ 已确认

§11.14A 禁手工录价的**唯一理由**是「保证每条价都有源、有日期、有批次记录」。本次开了旁路，源与日期靠必填保住，「批次记录」只能用变更历史补上，否则等于把当初的理由整个丢掉。

裁决：**新建独立表 `element_daily_price_log`**，形状比照 `element_price_strategy_log`（`action` + `snapshot` + `changed_by/changed_by_name`），`changes` 查询时算（U0 ⑦）。

**两条替代路已排除，不必再议**：

- *复用 `element_price_strategy_log` 加 `target_type`* —— 不可行。该表 `customer_no` 为 `NOT NULL`，而价格行是全局的、无客户维度，只能塞假值或改约束，两者都是污染。
- *不建表，靠多版本行追加* —— 不可行。唯一键 `(element_name, source_id, price_date)` 直接挡住，同一条价存不了第二行。

#### U13 · 技术总监拆任务时发现的两个契约缺口 ⚠️ 2026-07-23 补录

拆分前后端任务文档前，技术总监逐条复核 U0 的 7 项实证（全部成立 ✅），并在核对接口契约时发现**需求文档未覆盖的两个缺口**。两者都会直接导致功能做不出来或验收无法进行，已写入 `backtask.md` B3 与 `api.md` §4。

**缺口一：`ElementPriceRowDTO` 缺 `id`，`PUT`/`DELETE` 无法定位行**

- 实证：`pricetable/ElementPriceRowDTO.java` 共 11 个字段，**无 `id`**；`PriceTableService.listDetail:61-62` 的 SELECT 也未取 `edp.id`。
- 后果：§5 设计的 `PUT /prices/{id}` 与 `DELETE /prices/{id}` 靠行主键定位，但明细列表**根本不返回主键**，前端勾选后无从调用。
- 处置：DTO 加 `id`，`listDetail` SELECT 补 `edp.id`。同时前端 `rowKey` 必须从组合键 `${elementCode}__${sourceId}__${priceDate}` 改为 `id` —— `SelectableTable` 默认开启跨页保留选中，组合键跨页可能重复会导致选中串行。

**缺口二：`ElementPriceRowDTO` 缺 `fetchStatus`，验收 1/3 在 UI 上无法验证**

- 实证：同上，DTO 与 SELECT 均无 `fetch_status`。
- 后果：验收标准 1「列表出现该行，`fetch_status='MANUAL'`」与验收 3「SQL 直查该行 `fetch_status` 已变为 `MANUAL`」中，前者是**界面可见性要求**，当前列表无此列亦无此字段，无法达成。
- 处置：DTO 加 `fetchStatus`，明细 Tab 增加「数据来源」列（`MANUAL`→「手工」/`IMPORT`→「导入」）。
