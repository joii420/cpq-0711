## 2. 需求背景与目标 ⭐

- **要解决什么问题 / 痛点**：
  - 现状: 元素单价在报价单中配置的都是输入框列, 需要用户手动输入,并且没有一个可以系统性的元素价格管理页面
- 解决问题：客户与价格策略的关系, 针对客户的元素单价中,配置客户在报价时引入哪个源下的当前报价日期的价格单价，可以在报价单中关联查询展示
  - 分析: 由于最终单价需要在报价单和核价单进行关联引用, 根据每个客户的价格策略不同,引用其元素单价,是否需要给每个客户维护一个元素价格表,与系统的元素价格表分开,方便在SQL视图中配置组件渲染时,可以精确的关联出客户的元素单价.
    > **澄清结论（§11.18）**：不单建客户价格表，改为建 `v_customer_element_price` 视图实时计算，SQL 视图直接 JOIN 该视图即可精确取到客户元素单价。

- **不做什么（明确边界）**：
  - 非新增的元素单价维护模块不需要改动, 本次功能为额外的独立功能

# 11. 需求澄清结论（2026-07-22 澄清会 · 权威口径）

## 11.0 澄清前的系统现状盘点（开发必读）

澄清会前对现网做了盘点，四条事实决定了本方案的形态：

1. **元素管理页面已存在**：`主数据维护 → 元素` Tab（`cpq-frontend/src/pages/config/ElementManagement.tsx`，挂在 `MasterDataHubPage`）。采用 `SelectableTable` + Drawer 规范。本次三个新按钮加在此页。
2. **元素价格四张表在 V44 时期已建好骨架，基本未启用**：
   - `element_price_source`（价格源：名称/URL/类型/状态）
   - `element_price_fetch_rule`（抓取规则，JSONB）—— 本期不用
   - `element_daily_price`（每日价，含 raw_price / raw_open / raw_close / raw_high / raw_low）
   - `element_price`（客户级价格，含 `customer_id` / `premium_price`）
   - 现有 `ElementPriceService` 只实现了 v1「手工填单值」（只读写 `element_daily_price` 的 `fetch_status='MANUAL'` 行）；其 `listAvailableElements()` 仍读**已废弃**的 `mat_bom`（BL-0069 第 5 条实证失效路径），本期须一并纠正为读 `element` 主表。
3. **报价侧元素单价确为手填**：报价元素组件（`COMP-0029`，驱动 `$ys_view`，`system_type='QUOTE'`）的「单价」「货币」两列**无任何取数源**，其余列都有 `$ys_view.xxx`。
4. **核价侧元素单价已有取数，但口径不同**：核价元素组件（`COMP-0040`，`system_type='PRICING'`）的单价来自 `global_variable_value`（`var_code='COST_ELEMENT'`，按元素符号取）——一个**不分客户、不分日期的固定成本价**。本期按 §11.11 替换。
5. **JOIN 键天然对齐**：`element_bom_item.component_no` 存的就是元素符号（Ag / Cu / 301 …），与 `element.element_code` 完全一致（实测 9 个取值全部命中）。因此价格视图按 `element_code` 关联即可，无需任何编码映射。

## 11.1 策略配置粒度 —— 客户级默认 + 元素级例外（两层）

- 每个客户配 **1 条默认策略**（源 + 取值方式 + 窗口 + 系数/加价），该客户所有元素默认走它。
- 可对**个别元素**（如 Ag）追加**例外行**覆盖默认，例外行字段结构与默认策略相同。
- 取值优先级：**元素级例外 > 客户级默认 > 无（留空）**。

## 11.2 取价基准日 —— 报价单创建日期

- 窗口从**该报价单的创建日期**往前推，不是"今天"。
- 理由：同一张单过一个月重新打开，元素单价不会自己变，可追溯、可复现。
- **技术实现**：比照现有 `:customerCode` 的注入方式，在 `SqlViewExecutor` 中新增 `:priceBaseDate` 命名参数自动注入（由运行时上下文 `SqlViewRuntimeContext.quotationId` 反查报价单日期）。无 `quotationId` 上下文时（如配置期预览）回退为当天。

## 11.3 导入粒度 —— 一行 = 一个元素的当日价，日期与源在抽屉选

- 一次导入 = **一个源 × 一个日期 × N 个元素**。
- Excel 列：`元素符号` / `单价` / `货币` / `计价单位`（元素符号必填，其余可留空走默认）。
- 抽屉上选「价格源」+「价格日期」，Excel 里**不带**日期列。

### 11.3.1 行级校验规则

| 校验项 | 规则 | 不通过时 |
|--------|------|---------|
| 元素符号 | 必填，且**必须在元素管理中已存在、状态为启用** | 该行失败，说明"元素符号「X」在元素管理中不存在"或"已停用" |
| 单价 | 必填，须为大于 0 的数字 | 该行失败 |
| 货币 / 计价单位 | 可空，空则取系统默认（`CNY` / `元/kg`） | 不失败 |
| 价格源 | 抽屉上选定，只列出**启用**状态的源 | — |

### 11.3.2 🔒 事务边界：部分成功，失败行不阻断

- **逐行处理，失败行不影响其他行入库**。一批 10 行中 1 行失败，其余 9 行**照常写入**。
- **禁止**做成"任一行失败即整批回滚"。理由：价格是逐条独立的事实数据，行与行之间无关联；整批回滚会让维护人为一个错字反复重导整个文件。
- 修正失败行后可重新导入，重导走覆盖逻辑（§11.16），不会产生重复。

### 11.3.3 导入结果回显

- 汇总一行：`新增 N 条，覆盖 M 条，失败 K 条` + 源 / 日期 / 操作人。
- 明细表列：`行号 / 元素符号 / 单价 / 货币 / 计价单位 / 结果(新增·覆盖·失败) / 说明`。
  - 「覆盖」行的说明列显示 `原值 X → 新值 Y`；
  - 「失败」行的说明列显示具体失败原因（见 §11.3.1）；
  - 行号 = Excel 中的物理行号，便于维护人直接定位到文件里改。

## 11.4 价格维度 —— 每元素每天一个单价

- 只用 `element_daily_price.raw_price` 单值；`raw_open/close/high/low` 四列本期不用（保留不删）。
- 因此"最近 30 天最高值" = 这 30 个**日价**里的最大值，语义无歧义。

## 11.5 窗口内无数据的兜底 —— 留空，销售手填

- 策略窗口内一条价格都没有 → **不往前兜底**，该单元格取空值，由销售手工填写（与今天的行为一致）。
- 不做"超窗口取最近一条"，避免用一个过期价静默顶上。

## 11.6 自动带出的单价 —— 可手工覆盖

- 自动取数作为**默认值**带出，销售仍可在报价单里改（与系统其他取数列行为一致）。
- 实现上即"字段仍是可输入类型，只是配了 `default_source`"，不改字段类型。

## 11.7 取值方式 —— 4 种

| 枚举 | 名称 | 含义 |
|------|------|------|
| `LATEST` | 最新一条价 | 基准日**当天或之前**最近的一条价（不看窗口长度） |
| `AVG` | 窗口内平均值 | 窗口内所有日价的算术平均 |
| `MAX` | 窗口内最高值 | 窗口内日价最大值 |
| `MIN` | 窗口内最低值 | 窗口内日价最小值 |

- `LATEST` 不需要填窗口；其余三种必须填窗口。

## 11.8 加价规则 —— 源价 × 系数 + 加价

- 最终单价 = `取值结果 × factor + premium`
- `factor` 默认 1，`premium` 默认 0，均可为空（空等同默认值）。
- 计算顺序固定为**先乘后加**，不可调换。

## 11.9 窗口口径 —— 滚动区间

- 「最近 3 个月」= 基准日往前推 3 个月到基准日（如基准日 7/22 → 4/22 ~ 7/22，闭区间）。
- 窗口单位保留 **天 / 周 / 月 / 年** 四种，SQL 侧统一按 `interval` 处理。
- **不**采用自然日历口径（不是"上 3 个完整自然月"）。

## 11.10 货币与计价单位 —— 一并带出，不做换算

- 价格库存的 `货币` / `计价单位` 随单价一起带到报价单对应列。
- 与 BOM 侧计价单位不一致时**给提示，不自动换算**（换算规则易错，宁可人工确认）。
- 本期不建单位换算表。

## 11.11 核价侧 —— 走 `_GLOBAL_` 全局策略（2026-07-22 第 5 轮评审**改写**）

> ⚠️ **本节已推翻初版裁决**。初版（澄清会第 7 题）定为"核价也按客户策略取、与报价一致"，
> 但该判断建立在错误前提上——当时未查清核价侧数据的客户维度。第 5 轮评审经业务方指出并实测查证后改写如下。

### 11.11.1 实测事实（改写依据）

```
element_bom_item                    material_bom_item
PRICING  _GLOBAL_     26 行          PRICING  _GLOBAL_    59 行
QUOTE    CUST-1290   129 行          QUOTE    CUST-1269   20 行
QUOTE    CUST-1292   118 行          QUOTE    CUST-1290   15 行
QUOTE    CUST-1269    21 行          ...
```

**核价侧（`system_type='PRICING'`）全部数据挂在 `customer_no='_GLOBAL_'` 下**，报价侧才是真实客户编码。
即核价基础数据（BOM / 工序 / 费用）本就是**全局口径、不分客户**。

**由此纠正两个误判**：
1. 现有核价元素视图 `WHERE` 里没有客户过滤，**不是遗漏、是设计**。初版曾要求"接价格视图时补 `customer_no = :customerCode`"，照做会因 `_GLOBAL_ ≠ CUST-xxxx` 直接查不到任何数据。该要求**作废**。
2. 现有 `COST_ELEMENT` 全局变量取数**方向本就是对的**（全局成本价），它缺的是"源 / 日期 / 取值方式"三个维度，**不是**缺客户维度。

### 11.11.2 裁决（B1）

- **核价侧只认 `_GLOBAL_` 策略**，不看报价单归属的客户。
- 未配置 `_GLOBAL_` 策略时，核价元素单价**全部为空**（不回退到该报价单客户的策略）。
- 业务含义：**成本归成本、售价归售价**，两者差额即毛利。同一料号不因客户不同而核出不同成本。
- 因此**报价侧与核价侧的元素单价可以不同**，这是**预期特性，不是 bug**（初版"两侧同值"的验收要点相应作废，见 §11.21#6 改写）。

### 11.11.3 落地要点

- 视图侧两边各传各的，核价侧不需要客户上下文，**天然无跨客户串价风险**：
  ```sql
  -- 报价侧
  LEFT JOIN f_customer_element_price(:customerCode, :priceBaseDate) cep
  -- 核价侧
  LEFT JOIN f_customer_element_price('_GLOBAL_',    :priceBaseDate) cep
  ```
- 核价元素组件（`COMP-0040`）单价取数从 `global_variable_value(COST_ELEMENT)` 改为 `cep.unit_price`；旧全局变量数据**保留不删**（回滚退路）。
- 核价侧基准日仍取所属报价单的创建日期（与报价侧同一基准日，只是策略不同）。

### 11.11.4 🔒 强制技术约束（建表时定死）

**策略表的客户维度必须用 `customer_no VARCHAR` 存，禁止用 `customer_id UUID` 外键指向 `customer` 表。**
原因：`_GLOBAL_` 不是一张真实客户记录，UUID 外键无处安放。此决定影响建表 DDL、所有查询与 UI 取值，**后期更改代价极大**。

## 11.11E `_GLOBAL_` 策略的配置入口（B1 的必然配套）

`_GLOBAL_` 不是真实客户，不会出现在定价策略页的客户列表里，因此**必须给它一个配置入口**，否则该策略无处可配、核价侧永远无价。

- **形态**：定价策略页左侧客户列表**顶部固定一个特殊项**「🌐 全局（核价成本口径）」，与客户项视觉区分（图标 + 底色 + 分隔线），不参与客户搜索与分页。
- 选中它后，右侧「元素价格策略」Tab 的内容与客户完全同构：**全局默认策略 + 元素级例外 + 策略试算**，字段一模一样。
- 顶部给一条说明：「此处配置的是**核价成本口径**，仅核价单取用；报价单取各客户自己的策略。」
- 「折扣策略」Tab 在选中全局项时**不显示**（折扣是客户维度概念，全局无意义）。

## 11.12 价格源 —— 必填单选一个

- 客户策略（默认行与例外行）里「价格源」**必填且只能选一个**。
- 取价无歧义、可追溯到具体行情网站。

## 11.13 自动抓价 —— 本期不做

- 价格源只记录 `名称 / 网址 / 类型 / 备注 / 状态`，网址仅供人工参考。
- 价格一律靠 Excel 导入。`element_price_fetch_rule` 表保留不用，为将来爬虫预留。

### 11.13.1 价格源的操作与停用语义

- **操作集**：新建 / 编辑 / 停用（沿用列表操作规范：动作在顶部工具栏，行内只保留主入口链接）。**不提供物理删除**——源被历史价格引用，删除会造成价格记录悬空。
- **唯一性**：`源名称 + 网址` 组合不可重复（复用既有唯一索引 `uq_eps_name_url`）。
- **停用后的行为**（三条都要满足）：
  1. **不可**再被新的价格策略选用（策略的价格源下拉只列启用态）；
  2. **不可**再被新的价格导入选用（导入抽屉的源下拉只列启用态）；
  3. 该源下的**历史价格照常保留、照常显示**（价格表、元素抽屉各源最新价里仍可见，标注「源已停用」）；已经引用该源的存量策略**继续按原样取价**，不自动失效——避免停用一个源导致一批客户报价突然无价。
- 停用可在编辑抽屉里改回启用。

## 11.14 元素价格表 —— 抽屉，两个 Tab

- **Tab 1 明细**：一行一条 —— 元素符号 / 中文名 / 价格日期 / 源 / 单价 / 货币 / 计价单位 / 录入人 / 录入时间。顶部筛选：源、日期区间、元素（符号或中文名模糊）。
- **Tab 2 矩阵**：行 = 元素，列 = 日期，格 = 当日价。用于看趋势。
  - 日期跨度须限制（默认最近 30 天，最长 90 天，超出提示收窄）。
  - **价格源必填、且只能选一个**（与明细 Tab 允许"全部源"不同）：同一天多个源各有各的价，一个格子放不下，不先定源则无法成矩阵。
  - 某元素某天在该源下无记录时显示「—」（周末 / 未录入），不补零。
- 两个 Tab 都支持**导出 Excel**，导出内容 = 当前筛选结果（此项属需求原文「价格表支持下载」，非加法项）。

## 11.14A 元素抽屉 —— 各源最新价格（2026-07-22 原型评审修正）

- **主列表不加价格列**。原型 v1 曾在元素管理列表加一列「最新价」，评审否决：**一个元素在多个源下各有各的价，单列无法表达是哪个源的**。
- 改为在**元素编辑抽屉**下半部新增「各源最新价格」区块，**一行一个源**：`价格源 / 最新价 / 货币 / 计价单位 / 价格日期`。
- 规则：
  - 仅**编辑态**显示（新建元素时不出现）；
  - **只读**，不在此处录价——录价统一走「价格导入」，保证每条价都有源、有日期、有批次记录；
  - 已停用的源仍显示其历史价，整行置灰并标「源已停用」；
  - 该元素一个源都没价时显示空态文案：「该元素暂无任何价格记录，请通过『价格导入』录入」。
- 抽屉宽度由现有 `520` 调整为 `640`。

## 11.14B 元素列表「最后修改时间」—— 价格导入也算修改（2026-07-22 原型评审修正）

- 元素管理列表原有「创建时间」+「修改时间」两列，**合并为一列「最后修改时间」**，列表默认按该时间**倒序**（最近动过的排最前）。
- **口径**：该时间 = `MAX(元素主档 updated_at, 该元素所有价格记录的 updated_at)` —— 即**导入该元素的价格也算一次修改**。
- **实现口径（技术决策，不写库）**：查询时取两者较大值，**不**在价格导入时反写 `element.updated_at`。理由：
  1. 不污染主档语义（`element.updated_at` 仍严格表示"元素资料本身"何时被改）；
  2. 导入是批量操作，若逐元素反写主档会产生额外写入与行锁竞争；
  3. 存量历史数据天然正确，无需数据迁移回填。
- 排序规则相应调整为：**启用优先 → 最后修改时间倒序**（原规则为 启用优先 → 改时倒序 → 建时倒序）。

## 11.14C 策略试算（2026-07-22 原型评审确认保留）

- **入口**：定价管理 → 定价策略 → 选中客户 → 「元素价格策略」Tab → **内容区右上角「🧮 策略试算」按钮**（与"取值优先级"说明同一行）。
- **不**挂在「客户级默认策略」卡片头上：试算算的是**默认 + 例外合并后的最终结果**，挂进默认卡片会让人误以为只试算那一条。
- **行为**：
  - 弹抽屉（720px），可选**基准日**（默认今天；实际报价时基准日 = 报价单创建日期）；
  - 结果表逐元素列出：`元素 / 命中规则(元素例外 or 客户默认) / 源 / 取值方式 / 取值结果 / × 系数 / + 加价 / 最终单价 / 参与天数`；
  - 窗口内无价的元素整行标黄、最终单价显示「无价」，与报价单留空行为对应；
  - 按**当前页面上的配置**试算（含尚未保存的改动），支持"先试算、满意再保存"；
  - **只读不落库**，纯计算预览（不提供导出）。
- **价值**：策略是"窗口 + 取值方式 + 系数加价"的组合，配置本身看不出最终金额；试算是维护人自检与验收核对（§11.21#5）的最直接工具。

### 第 11 项（无价黄底）被砍的经过与依据

第 4 轮曾保留此项为"唯一视觉增补"，第 6 轮业务方追问其识别机制后砍掉：

- **识别问题**：渲染引擎无法区分"这个空格是应有价却没取到"与"普通空输入框"。实测全库 **220 个 INPUT 字段中 198 个（90%）都配了 `default_source`**，若按"配了取数源且解析为空即标黄"的通用规则，全系统 198 个字段的空值会同时变黄，信号被噪音淹没。
- **可行方案均需付费**：① 字段配置加新 config JSON 键 → 触发 AP-44 十七点联动，代价过高；② 视图输出约定列（如 `__price_miss__单价`）→ 0.5 人日。
- **砍掉的实质理由（非省成本）**：
  1. **零退步**——元素单价现状 100% 手填，改造后有价的自动带出、无价的仍手填，销售体验只升不降；
  2. **需求方错位**——"哪些元素缺价"的真正需求方是价格维护员而非销售；
  3. **已在正确的地方满足**——策略试算（§11.14C）结果表中窗口内无价的元素整行标黄并显示「无价」，同一信息无需在报价单再做一遍。

**连带结论（重要）**：
- 因角标与黄底全砍，**价格视图无需输出任何元数据列**（`__price_meta__*` / `__price_miss__*`），只返回最终单价 + 货币 + 计价单位，SQL 复杂度下降一档。
- **报价单/核价单前端零改动**成为事实而非口号：本期报价渲染侧只改组件字段的取数配置，不动任何 `.tsx`。
- 因此**不触发** CLAUDE.md「协议级改动必须跑 Playwright E2E」中关于 `QuotationStep2.tsx` 等文件的强制条款（仍建议跑一轮回归确认取数正确）。

## 11.14F 价格策略变更历史 + 最后变更信息（2026-07-22 业务方新增需求）

> 本节为**业务方主动提出的新增功能**，非技术总监加法项，不适用 §11.14D 的砍除清单。

### 11.14F.1 立项理由

价格策略直接决定报价金额——改一个系数，该客户全线报价跟着变。必须留痕可追溯：改了什么、谁改的、什么时候改的。

### 11.14F.2 记录范围

| 变更对象 | 记录的操作 |
|---------|-----------|
| 客户级默认策略 | 新建 / 修改 |
| 元素级例外 | 新建 / 修改 / 删除 |
| `_GLOBAL_` 全局策略（默认 + 例外） | 同上 |

### 11.14F.3 存储与展示口径

- **存快照、展示差异**：每次变更在历史表存一份**完整策略快照**（源 / 取值方式 / 窗口 / 系数 / 加价）；界面上自动比对相邻两版，**只显示变化项**（`旧值 → 新值`），未变项不展示，不需要人工对照。
  - 选择快照而非仅存差异的理由：快照可无损重建任意时点的策略全貌，差异记录一旦中间某条丢失即无法复原。
- **删除操作**记录删除前的完整内容（供追溯"当时删掉的是什么"）。
- 历史记录**只读**，不提供在历史界面上编辑。

### 11.14F.4 界面

- **入口**：「元素价格策略」Tab 右上角「🕘 变更历史」按钮，与「🧮 策略试算」并排。
- **形态**：抽屉 960px。筛选项：变更对象（全部 / 客户级默认 / 某个元素例外）、变更时间区间、变更用户。
- **列**：`变更时间 / 变更用户 / 变更对象 / 操作(新建·修改·删除) / 变更内容`。
- **最后变更时间 + 变更用户**同时显示在两处（业务方明确要求）：
  1. 「客户级默认策略」卡片头部：`最后变更 2026-07-22 10:15 · 张三`；
  2. 「元素级例外」表格新增两列：`最后变更时间`、`变更用户`（原「修改时间」列并入）。

### 11.14F.5 待确认项

## 11.18 客户元素价格的存储形态 —— 实时算，不物化

> **承载形态更正（第 5 轮）**：本节原写"建 SQL 视图 `v_customer_element_price`"，实际落为 **PG 表函数 `f_customer_element_price(客户编码, 基准日)`**（理由见 §11.20）。**"实时计算、不物化"的决策本身不变。**

- **不**建"每客户一张元素价格表"。改为由表函数按客户策略 + 基准日**当场算出**单价。
- 理由：
  - 物化表必须解决"何时重算"（新导价 / 改策略 / 日期推移都会让结果变），漏算即静默错价；
  - 视图逻辑单点，改策略立刻生效，永不失步；
  - 历史报价单靠现有卡片快照冻结机制固化，不会被后来的价格带跑。
- 数据量评估：`element_daily_price` 按每天 ~30 元素 × 数个源计，一年数万行，索引足够。

## 11.19 组件改造范围 —— 先改两个标杆组件

- 本期**接通并验证** `COMP-0029`（报价侧）+ `COMP-0040`（核价侧）两个标杆组件的「单价」列。
- 其余元素类组件副本（`COMP-0029__imp1` / `COMP-0020__imp1` / `COMP-0020__imp1__imp1`，分属不同客户模板）**不动**，随任务文档给出配置说明，由业务在组件管理里自行绑定（符合现有配置驱动架构）。

## 11.20 数据模型（澄清结论汇总）

| 表 | 用途 | 处理方式 |
|----|------|---------|
| `element_price_source` | 价格源 | **复用** V44 既有表；本期启用 CRUD |
| `element_daily_price` | 每日元素价（源 × 日期 × 元素） | **复用** V44 既有表；只用 `raw_price` 单值；导入写 `fetch_status='IMPORT'`（与既有 `MANUAL` 行区分） |
| `element_price_fetch_rule` | 抓取规则 | 本期**不用**，保留 |
| `element_price` | 旧的客户级元素价 | 本期**不用**（结构与新策略模型不符）；新建策略表承载 |
| `element_price_strategy`（新） | 客户元素价格策略 | 客户级默认 + 元素级例外同表，用 `element_code IS NULL` 区分默认行；🔒 客户维度必须为 **`customer_no VARCHAR`**（容纳 `_GLOBAL_`），禁止 `customer_id UUID` 外键（§11.11.4）；含 `updated_at` / `updated_by` 供「最后变更」显示 |
| `element_price_strategy_log`（新） | 策略变更历史（§11.14F） | 每次变更存一份**完整策略快照** + 操作类型（CREATE/UPDATE/DELETE）+ 操作人 + 操作时间；展示时相邻两版比对出差异；与策略写入**同事务** |
| `f_customer_element_price`（新**表函数**） | `(客户编码, 基准日) → 该客户各元素最终单价` | ⚠️ **是 PG 表函数，不是视图**（下方说明） |

### 为什么是表函数而不是视图（2026-07-22 第 5 轮确认）

初版 §11.18 写的是"建视图 `v_customer_element_price`"，实现形态已改为**表函数** `f_customer_element_price(p_customer_no TEXT, p_base_date DATE)`，原因有二：

1. **PG 普通视图不能带参数**，而取价必须按「客户 + 基准日」两个运行时入参计算；
2. 实测本项目 `component_sql_view.sql_template` **不支持内嵌引用另一个 `$view`**（全库 63 条视图均为 `scope=COMPONENT`，无先例），因此价格逻辑无法做成"共享 `$view`"给其他视图引用。

表函数一次调用返回该客户在该基准日下的全部有价元素（约几十行），驱动视图 `LEFT JOIN` 它即可，不存在 N+1。
§11.18 的"实时计算、不物化"这一**决策本身不变**，变的只是承载形态。

## 11.21 验收要点（可测口径）

1. **源管理**：新建 / 编辑 / 停用价格源，停用后不可再被新策略或新导入选用，历史数据照常显示。
2. **导入**：下载模板 → 填 3 个元素 → 选源与日期 → 导入成功，回显"新增 3 / 覆盖 0 / 失败 0"；同参数再导一次回显"新增 0 / 覆盖 3"，值被更新。
3. **价格表**：明细 Tab 按源 + 日期区间 + 元素筛选正确；矩阵 Tab 行列正确、超期提示生效；两 Tab 导出的 Excel 内容与筛选结果一致。
4. **策略**：某客户配「源 A + 近 30 天平均 × 1.05 + 2」，Ag 配例外「源 A + 最新价 × 1」，保存后回读一致；例外优先于默认。
5. **报价单取价（核心）**：该客户的报价单元素页签，Ag 行单价 = 源 A 在该单创建日期当天或之前最近一条价 × 1；其他元素单价 = 近 30 天平均 × 1.05 + 2；数值可人工用价格表数据复核。
6. **核价单取价**（§11.11 改写后）：核价元素页签单价按 `_GLOBAL_` 策略取，**与报价侧可以不同**（这是预期，不是 bug）；未配 `_GLOBAL_` 策略时核价元素单价全空；核价侧不受报价单归属客户影响（换个客户的单，核价元素单价不变）。
7. **可覆盖**：销售改掉自动带出的单价后，保存 → 重开仍是改后的值，不被自动值回冲。
8. **留空兜底**：把某元素的策略窗口内价格清空 → 该行单价显示空（不是 0、不是"加载中"），销售可手填。
9. **无回归**：未配策略的客户、其他页签、其他客户模板的元素组件行为与改动前逐位一致。
10. **`_GLOBAL_` 配置入口**（§11.11E）：定价策略页客户列表顶部可见「🌐 全局（核价成本口径）」固定项；选中后可配默认策略与元素例外；该项不参与客户搜索与分页；选中时「折扣策略」Tab 不显示。
11. **变更历史**（§11.14F）：
    - 改一次默认策略的系数 → 历史新增一条「修改」，**只列出变化项**（系数 旧→新），未变项不出现；
    - 新增 / 删除一条元素例外 → 分别产生「新建」/「删除」记录，删除记录含删除前完整内容；
    - `_GLOBAL_` 策略的变更同样入历史；
    - 按变更对象 / 时间区间 / 变更用户筛选正确；
    - 历史为只读，界面上无编辑入口。
12. **最后变更信息显示**（§11.14F.4）：默认策略卡片头部显示「最后变更 时间 · 用户」；元素例外表格的「最后变更时间」「变更用户」两列与历史记录最新一条一致。
13. **前端零改动核验**（§11.14D 连带结论）：本期报价单 / 核价单渲染相关 `.tsx` 文件 `git diff` 为空；取价效果完全由组件字段取数配置产生。
14. **最后修改时间口径**（§11.14B）：给某元素导入一条价格 → 该元素在列表中的「最后修改时间」更新为导入时刻并排到最前；但 `element` 主表的 `updated_at` **保持不变**（验证未反写主档）。
15. **元素抽屉各源最新价**（§11.14A）：某元素在 3 个源下有价 → 抽屉列出 3 行且日期/价格正确；停用其中一源 → 该行仍在并置灰标「源已停用」；对无任何价格的元素显示空态文案。
16. **导入部分成功**（§11.3.2）：一批 10 行含 1 行错元素符号 → 结果为"成功 9 / 失败 1"，且**9 行确已入库**（非整批回滚）；修正后重导 → 走覆盖、不产生重复行。

| ~~核价视图缺客户维度~~ **（已作废，勿照做）** | 初版误判 `COMP-0040` 无 `customer_no` 过滤是漏洞 | ❌ 初版对策"显式带 `:customerCode`"**已作废**——核价数据 `customer_no` 恒为 `_GLOBAL_`，加客户过滤会一行查不到。正确做法见 §11.11：核价侧传字面量 `'_GLOBAL_'`，视图 WHERE **不加**客户条件 |

| 废弃表引用 | 现有 `listAvailableElements()` 读已停写的 `mat_bom`（BL-0069#5） | 本期改读 `element` 主表，顺带闭合该条 P2 |
