> **状态**：第一版 · 仅覆盖「当前代码里**已实现版本化**的 sheet」，全部规则源自现有 handler 源码（非设想）。
> **用途**：供需求方审校现有轴/值/升版口径；调整定稿后再补充「需要新增版本化的 sheet」。
> **日期**：2026-07-09 起草（技术总监）。
> **代码依据**：`com.cpq.basicdata.v6.versioning.VersionedV6Writer` + `quote/Q*Handler` + `pricing/P*Handler`。

## 术语约定（贯穿全文）

| 术语 | 代码对应 | 含义 |
|------|---------|------|
| **轴** | `groupKeyColumns` / `masterGroupKey` / `childGroupKey` | 唯一确定"哪一组"的列集合。定位当前 `is_current=true` 的生效组；本组内容变化只在本组内升版，不影响其他组。 |
| **值** | `contentColumns` / `childContentColumns` | 参与"内容是否变化"比对的列。逐行、顺序无关地做 multiset 比较。 |
| **触发列** | `versionTriggerColumns` | 值列的子集，**只有它变化才升版**；未指定 = 全部值列都触发。用于"金额类原地更新、结构类才升版"的场景。 |
| **版本列** | `versionColumn` / `characteristic` / `bom_version` / `version_no` / `scheme_version` / `calc_version` | 存版本号的列，VARCHAR 存数字字符串。 |
| **生效标记** | `is_current` | 每组当前生效的行为 `true`，历史行为 `false`。 |

## 一、核心升版机制（所有 sheet 共用，`VersionedV6Writer`）

所有已版本化 sheet 都走同一套算法，理解一次即可套用到每个 sheet。

### 1.1 单表行集版本化（`writeVersionedGroup(s)`）

对某个 **轴** 定位到的一组行，与库里当前生效组做比对，走三分支：

```
按 轴(groupKey) 取当前 is_current=true 的整组行 existing
newRows = 本次导入的该组整套行

比较（multiset，顺序无关；数字 stripTrailingZeros 归一；NULL 安全 IS NOT DISTINCT FROM）：
  triggerSame = existing 与 newRows 在【触发列】上是否全等
  contentSame = existing 与 newRows 在【全部值列】上是否全等

┌── (a) triggerSame && contentSame ── 完全相同 → 复用旧版本号，不写库
├── (b) triggerSame && !contentSame ── 仅非触发值(金额等)变了 → 原地更新当前组
│                                      （删当前 is_current 行 + 用【旧版本号】重插；版本号不变、不升版）
└── (c) !triggerSame（或首次导入）──── 触发列变了 → 升版
                                       新版本号 = 该轴历史 MAX(数字版本)+1；旧组整体 is_current=false；
                                       新组写入 is_current=true
```

> **要点**：只有当 sheet 显式配置了「触发列」时，(b) 分支才有意义（如产能：金额改了不升版、工序集合改了才升版）。**大多数 sheet 没配触发列 → 触发列=全部值列 → 只有 (a)/(c) 两态：任意值变化即升版。**

### 1.2 版本号规则

- VARCHAR 存数字字符串，**首版 = `"2000"`**；升版 = 该轴下 `MAX(CASE WHEN 版本 ~ '^[0-9]+$' THEN 版本::int END) + 1`。
- 非数字历史版本（如导入的 `V_DEFAULT`/`V1`）在取 max 时被正则忽略，不影响新号生成。

### 1.3 主从 BOM 版本化（`writeVersionedMasterDetail(s)`）

用于 `element_bom`(主)+`element_bom_item`(子)、`material_bom`(主)+`material_bom_item`(子)：

- **子表行集**决定是否升版（比对子表 `childContentColumns`）；子表全等则整组不写。
- 升版号 = **主表**版本列 `MAX+1`；主、子 `is_current` 同步翻转。
- 现役全部 BOM 走 `childVersionColumn != null`（子表写版本列）→ **子表多版本保留**（历史明细行留存，`is_current=false`）。
- `masterFixedColumns`：主表 NOT NULL 固定列（如 `bom_type`），整批恒定，不进版本比较。
- `masterContent`：主表**逐组可变的描述列**（如 `production_no`），写入主行但**不进版本比较**。

### 1.4 通用护栏（写入器强制校验，违反直接抛错）

1. `newRows` 为空 → 拒绝（整组下线要走专门 API，不能用空写入静默清组）。
2. 值列为空 → 拒绝。
3. 值列与轴列**列名重叠** → 拒绝。
4. `system_type` 维度表（`capacity / plating_scheme / element_bom(_item) / material_bom(_item)`）的轴**必须含 `system_type`**，否则抛错（防 QUOTE/PRICING 跨域污染版本号与 is_current）。
5. 并发：同轴前缀走 `pg_advisory_xact_lock` 串行化（随事务提交/回滚释放），避免双 current / 重复版本。
6. 批量入口要求轴里至少有一个**跨组恒定列**（如 `system_type`）作锁/加载前缀。

## 四、现状里需要需求方**拍板/统一**的口径差异（审校重点）

以下都是"现有代码就这么写"的既成事实，但两侧/各 sheet 口径不完全一致，需你确认是"有意为之保留"还是"应统一"：

- **差异① · element_bom_item 的"值"两侧不一致**：报价 Q04 子值含 `composition_qty/issue_unit/base_qty`（用量变化会升版），核价 P07 子值只有 `seq_no/component_no/content/scrap_rate`（用量变化**不**升版）。同一张表，升版敏感度不同。
- **差异② · 单价类"金额即升版"**：报价 10 张 unit_price sheet 都**无触发列**，`pricing_price`/`cost_ratio` 等金额一变就升版；而 capacity/P08 用了触发列做到"金额原地更新不升版"。是否要给单价类也引入"金额原地更新、仅结构变化升版"的触发列口径？
- **差异③ · labor_rate 无独立版本**：P08 的人工单价借 capacity 版本号 upsert，若 capacity 未升版但人工单价变了 → 原地覆盖、无历史。第二步把 labor_rate 纳入版本化时需定它自己的轴/触发列。
- **差异④ · pricing_price 表未版本化**：多张报价 sheet 除写 unit_price 外还"另写 pricing_price 表"，该表本身没有版本化。是否纳入？
- **差异⑤ · 客户维度**：报价侧轴含 `customer_no`（按客户各自版本线），核价侧 `customer_no` 为常量（全局共享一条版本线）。确认这是期望语义。

## 五、核价侧版本升级规则（定稿 · 2026-07-10 澄清 C5–C12）

> `核价落库方案.md` 旧 §四/§五 的 `sales_part_no` 设计是**已被 V315 反做删除的过时 WIP，作废**。

### 5.0 核价升版总原则

| 维度 | 口径 |
|---|---|
| 版本来源 | **甲·系统自增**：忽略 Excel 自带版本列（元素价格版本/计算版本/汇率版本/版本编号…），由 `VersionedV6Writer` 内容指纹比对生成（首版 `2000`、内容变 `max+1`、旧组 `is_current=false` 翻转） |
| 取数 | **A·永远取 `is_current=true` 最新版**；不按版本号复现历史；「核价版本」P04 版本包退化，出本期范围 |
| 版本总轴 | **销售料号（`material_no`）**；`production_no` 仅描述、**不进轴、不参与升版比对** |
| 分类 | **每个 sheet 独立管版本**；版本组 = `(sheet/price_type, 销售料号)` |
| 粒度 | **粗·多行一组**：一个 (sheet, 销售料号) 的整批行为一个版本组；子维度（来料料号/要素/工序/项次/模具编号…）当作**值**，不作轴 |
| 比对 | **顺序无关** multiset（数字 stripTrailingZeros 归一、NULL 安全）；**不因导入行先后顺序不同而升版** |
| 触发 | 甲下**无触发列**：任一值变化即整组升版（原报价侧 capacity 的"金额原地更新不升版"特例，核价侧不采用，见 §5.4） |

### 5.2 `production_energy` 表结构重构（根因修复，C11）

- **根因**：建表未加类型区分，折旧、能耗塞 `depreciation_unit_price` / `energy_unit_price` 两列、**按 (material_no,process_no) 合并进同一行** → 两 sheet 共用行，无法独立版本。
- **改法**：
  1. 增列 `price_type`（枚举 `ENERGY`=能源 / `DEPRECIATION`=折旧）；
  2. 把 `depreciation_unit_price` + `energy_unit_price` **合并为单列 `unit_price`**；
  3. P09 折旧写 `price_type=DEPRECIATION`、P10 能耗写 `price_type=ENERGY`，**各写各行、不再共用行**。
- **收益**：来源 sheet 明确 + 两 sheet **可各自独立版本管理**（轴 = `system_type, material_no, price_type`，工序整批一组），与 unit_price 费用类模型同构。
- **迁移**：加列 + 合并列；存量行按非空单价拆成对应 `price_type` 行（数据量小，或清空重导）。
- **命名**：`production_energy.unit_price` 列与同名 `unit_price` 表仅名字相近、不同命名空间，语义="单价（按 price_type 区分类型）"。

### 5.3 `tooling_cost` 加版本列

- 现状：有 `is_current` 但**无版本列** → 无法系统升版。
- 本次：新增版本列（`calc_version` VARCHAR，与能耗对齐），接入 `VersionedV6Writer`；轴 = `system_type, material_no`，模具明细整批为一组。

### 5.4 `capacity` + `labor_rate` 去触发列（C12）

- 甲下任一值变即升版。核价 `capacity` **去掉现有触发列**（`process_no`），金额/币种/单位变化也升版；`labor_rate` 同口径（其整列即金额，本就必须金额升版）。
- 报价侧 `Q14 组装加工费` 同款触发列**不在本期范围**，留档暂不动（短期内产能"金额是否升版"报价/核价不一致，待报价侧统一时拉平）。

### 5.5 与现有写入器对接（实现约束，详见 backtask）
