### 4.1 用户场景 / 操作流程

### 4.2 界面与交互（前端相关）

### 4.3 业务规则与计算逻辑（后端相关）
- 移除了 组成件BOM 的sheet
- 物料BOM sheet 的投入材质-> 投入料号 , 投入材质名称-> 投入料号名称, 并增加了组成用量
- 物料BOM 现作为 主件的组成BOM的组成架构,  投入料号列可能是材质,可能是零件,可能是外购件, 与当前的 material_bom_item.characteristic 有三种类型： 'RECIPE' 材质， 'ASSEMBLY' 组成件 ， 'OUTSOURCED' 外购件 类型吻合
- 现约定  自制加工费 sheet 的投入料号为 零件类型的料号, 组成件其他费用 的组成件料号 为外购件类型的料号,  物料与元素BOM中的材质料号为材质类型的材质料号,成品其他费用和组装加工费,客户料号与宏丰料号的关系 内的销售料号均为主件料号(存储类型为零件类型,因为作为客户本次下单的料号,故作为主件料号区分). 所以物料BOM和来料固定加工费的投入料号中没有字段声明料号类型,需要根据这三个sheet内的料号进行匹配,来确认物料BOM中的料号类型,匹配不到并且数据库中不存在的料号均视为零件料号
- 电镀方案,来料年降,组装加工费年降,年降系数 四个sheet本次忽略暂不修改
- 其中红色的单元格为必填项, 蓝色的单元格为必填其一, 如果填写的是名称列,根据名称列进行查库查找对应的料号(先匹配类型 材质,零件,外购件. 再根据名称查找对应的表, 如果是料号,不存在则进行发号,如果是材质,提示未找到材质,报错)
- 本次导入如果存在错误行,全部回滚不导入,无法创建报价单,只有全部通过了才可以进行正确导入

### 2026-07-23 · update-0723 第一轮澄清（新导入模板「报价系统模板0723.xlsx」）

> 技术总监用 openpyxl 逐 sheet 比对 `报价系统功能基础数据功能结构所需字段V3.xlsx`（旧）→ `报价系统模板0723.xlsx`（新），并贴现役导入代码复核后，与需求方逐条确认。以下为交付前口径。

**U0 · 模板结构差异全清单（含需求文档未提、比对发现的 3 处）**

| # | 差异 | 需求文档是否提及 | 处置 |
|---|------|:---:|------|
| 1 | `元素单价` sheet 删除 | ❌ 未提 | Q01ElementPriceHandler 下线；元素价格改由另立项的**元素价格表功能**维护（已在开发） |

**U1 · 料号类型推断（物料BOM / 来料固定加工费 / 来料其他费用 / 来料回收折扣 的投入料号）**
- 物料BOM 现作为主件的组成 BOM 组成架构，投入料号可能是材质 / 零件 / 外购件，对应 `material_bom_item.characteristic` 三态 `RECIPE / ASSEMBLY / OUTSOURCED`。
- **类型权威来源（三个 sheet 的料号 + 名称，两侧都算）**：
  - 材质 = `物料与元素BOM`（材质料号 / 材质料号名称）
  - 零件 = `自制加工费`（投入料号 / 投入料号名称）
  - 外购件 = `组成件其他费用`（组成件料号 / 组成件名称）
  - 主件 = `成品其他费用` / `组装加工费` / `客户料号与宏丰料号的关系` 的销售料号（存储类型 = **零件**，作本次客户下单主件区分）
- **匹配口径**：投入料号的**料号列或名称列命中权威 sheet 其一即算**（U4）。
- **类型冲突**（同一料号 / 名称同时命中 ≥2 个类型集）→ **报错，整单回滚**（U6·洞①）。
- **库内兜底顺序**（三 sheet 都没命中时）：查 `material_recipe`（code 或 name 命中 = 材质）→ 查 `material_master`（material_no 或 material_name 命中 = 取其 `material_type` 映射类型）→ 都不中 = **零件**（U6·洞②）。

**U2 · 定型后的落库动作**

| 类型 | 行内填了料号 | 只填名称（蓝色必填其一） |
|------|-------------|------------------------|
| 材质 | 查 `material_recipe`，无 → **报错「未找到材质」** | 按名查 `material_recipe`，无 → **报错「未找到材质」** |
| 零件 / 外购件 | 库内无 → **直接用该料号落库**（新建登记 material_master + 占号），**不另发号** | 按名查 `material_master`，无 → **发号** `XXXX-YYMMNNNNNN` + 登记 |

**U3 · `material_master.material_type` 值域**
- 值域 = **`{零件, 外购件}`**，与 `characteristic` 的 `ASSEMBLY / OUTSOURCED` 一一对应。
- 材质料号**不进** `material_master`（沿用 repair-2 决策：材质走 `material_recipe`，不铸号不登记，防跨客户串号）。
- 存量旧数据里的 `material_type='组成件'` **不迁移**（§6 存量不处理）；新导入按新值域写入 / 覆盖。

**U4 · 只填名称行的定型**：名称也参与权威 sheet 匹配（料号列或名称列命中其一即算），再走 U1 库内兜底。

**U5 · 工艺路线断供修复**
- `组成件BOM` 删除后 `material_bom_item.operation_no` 断供（下游 `QuotationService` seed `quotation_line_process` 依赖它）。
- 修法：**从「自制加工费」的工序编号反填** —— 同一 `(销售料号, 投入料号)` 的工序编号写回 `material_bom_item` 对应零件子行的 `operation_no`；下游 seed SQL 不动。
- 说明：`process_master` 是全系统工序主数据（只做存在性校验 JOIN），「某料号走哪几道工序」的工艺路线来自 `material_bom_item.operation_no`，故必须补源。

**U6 · 全量回滚 + 事务方案（需求方与技术总监共同敲定）**
- **两阶段**：Phase 1 全量校验（解析所有 sheet + 类型推断 + 必填/冲突/材质缺库/引用校验，**零写库**，一次性返回全部错误行）→ 全通过才进 Phase 2 写入。
- **Phase 2 = 外层开一个事务，内层 13 个 handler join 共享同一物理事务**：handler 方法结构保留，事务传播从 `@Transactional(REQUIRES_NEW)` 改为 **join 外层**（`MANDATORY` / 由外层 `REQUIRES_NEW` 统一开启），全成功一起 `commit`、任一异常一起 `rollback`。
- 技术边界备案：数据库原子性单位是单个物理事务，「13 个独立 commit 的事务一起回滚」在物理上不成立（已 commit 无法真回滚，Saga 补偿删除脆弱不采用）；「13 个 handler 共享一个外层事务」即等价实现，方法结构不必合并。
- **副作用备案**：发号序列 `quote_material_no_seq` 非事务性，整体回滚后已 `nextval` 的序号不回退，会留下**料号号段空洞**（不影响唯一性 / 正确性，可接受）。

**U7 · 「有错全部回滚，全部通过才导入」**：本次导入若存在任何错误行，整单不导入、不创建报价单；仅全部校验通过才写库。

**U8 · 性能验收口径**：按本次模板量级（百行内）**端到端 < 2s**（点击导入 → `import_record` 变 `SUCCESS`）；禁止 N+1（批量 SQL / 一次查询内存计算）；大文件（千行级客户文件）不卡此指标。

**U9 · 4 个忽略 sheet**（`电镀方案` / `来料年降` / `组装加工费年降` / `年降系数`）：规则不改、照旧导入，但**纳入全量校验与整单回滚**（写入阶段同处一个物理事务，无法单独豁免）。

**U10 · 来料三表口径统一**：`来料固定加工费` / `来料其他费用` / `来料回收折扣` 的投入料号走同一套「类型推断 + 名称反查 + 发号」（描述同一批投入料号的不同费用维度，口径必须一致）。

**U11 · 模板配色语义**：红色 = 必填、蓝色 = 必填其一（料号或名称）、绿色 = 等同选填（无特殊语义）。

**U12 · 必填校验口径（以本条为准，覆盖 §4.3「红色即必填」的字面表述）**：本次**只强制校验关键键列**（销售料号 / 项次 / 料号类），**业务值列不强制**。典型例外：`物料与元素BOM.组成含量（%）` 模板标红但样例 3 行全空（仅填净用量）→ 该列不强制，组成含量与净用量二者能算出元素用量即可。

**U13 · 与升版任务衔接**：本次导入走 pending（`pending_quotation_id`）机制、**不触发真实升版**；回填升版由 `task-0721-报价升版逻辑` 负责（机制已在主干就位）。报价侧本身**零升版逻辑改动**。

**U14 · 范围边界**：只改「从基础数据导入 → 创建报价单」链路；前端**无代码改动**（现有异步进度条 + `sheetResults` 错误清单渲染已够用，仅配合回归）；其余模块功能不变。
