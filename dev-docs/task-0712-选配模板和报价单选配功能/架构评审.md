## 决策 1 — 单材质料号 quantity=2 的产品建模

### 现状事实
- D11：明细表一行 = 一个材质料号 + 数量；D12：数量合计 ≥ 2 触发组合工艺 = COMPOSITE（"≥2 件即需组装"）。
- 现役 `validateRequest` 用**行数**判定：SIMPLE=1 行、COMPOSITE=2~8 行。**单行 quantity=2 目前既过不了 COMPOSITE（parts.size<2 被拒），走 SIMPLE 又丢掉"组装"语义**。
- `computeComposite` 的 `COMBO=子件料号:数量` **已经是"去重子件 : composition_qty"口径**（不是把 qty 展开成多条），`writeCombomaterialBomV6` 也已按 `composition_qty=qty` 写 ASSEMBLY 行。
- `quotation_line_item` **没有 quantity 列**，数量无处安放（除非改报价单主表 = 触碰"禁改"红线）。

### 选项对比

| 选项 | 建模 | COMBO 指纹 | ASSEMBLY BOM | 组装加工费(capacity) | 需要改报价单主表? |
|---|---|---|---|---|---|
| **A（推荐）父COMPOSITE + 去重子件，composition_qty=行数量** | 单行 qty=2 → 父COMPOSITE + **1 个子件** `composition_qty=2` | `COMBO=P:2` | 1 行 `component_no=P, composition_qty=2` | 有（组装语义成立） | 否 |
| B｜SIMPLE + 数量字段 | 单行 qty=2 仍 SIMPLE，数量挂 line item | 走 `computeSimple`，**不含 qty** | 无 ASSEMBLY | 无 | **是（line_item 加 quantity 列）** |
| C｜父COMPOSITE + 把 qty 展开成 N 条相同子件 | qty=2 → `childHfPartNos=[P,P]` | `COMBO=P:1,P:1` | **2 行同 component_no** | 有 | 否 |

### 推荐：**选项 A**（父 COMPOSITE + 去重子件 + `composition_qty = 行数量`；判定按数量合计 Σqty）
- SIMPLE = `Σqty == 1`（唯一一行且 qty=1）；COMPOSITE = `Σqty ≥ 2`（**允许 1 行但该行 qty≥2**）。
- 单行 qty=2 → `childHfPartNos=[P]`（该行 resolvePart 得的子件料号）+ `childQtys=[2]` → `COMBO=P:2` → 父发 COMPOSITE 料号 + capacity 组装 + ASSEMBLY BOM `composition_qty=2`。

### 理由 / 取舍
1. **与现役指纹口径自洽（决定性）**：`computeComposite` 本来就是 `子件:composition_qty`。选 A 无需改指纹算法，只需放开判定闸门；选 C 会产出 `P:1,P:1` 这条**与 A 不同的串** → 同一个产品在"1 行 qty2"与"2 行"两种录入下指纹不一致 → 复用错配/错价（AP-52 类契约错配）。
2. **与导入等价（D16 底线）**：导入 §3 物料BOM 的 ASSEMBLY 对"2 个相同件"写的是**1 行 composition_qty=2**，不是 2 行。选 A = 导入同形；选 C 写 2 行同 component_no，既不等价导入，还可能撞 `material_bom_item` 唯一键。
3. **不碰报价单主表（硬约束）**：`quotation_line_item` 无 quantity 列。选 B 必须给主表加列 = 直接违反"报价单其他功能禁止变动"。选 A 把数量落在 `material_bom_item.composition_qty`（BOM 语义本位），line item 树只出"父 + 1 子 PART 行"，与导入组合品渲染一致。
4. **符合 D12 业务语义**：2 件即需组装，A 产出真正的 capacity 组装加工费；B 无组装费 → 与"≥2 件需组装"矛盾。

### 对指纹 / 落库 / 文档的影响
- **指纹**：口径不变（`COMBO=子件:数量`）；仅需保证 `childQtys` 取自 `pr.quantity`（现已如此）。**存量含"1行qty2"的选配此前根本发不了号 → 无历史指纹需兼容**。
- **落库**：单行 qty≥2 现在会进 COMPOSITE 落库分支（父 material_master=COMPOSITE + ASSEMBLY 头/子 BOM + capacity），需保证 B2 完整落库改造对"1 个子件的 COMBO"也成立（子件数=1 不是特例，循环天然支持）。
- **文档**：backtask B2.3 / api §3.3 的"数量合计 ≥ 2 → COMPOSITE"要落成**代码级判定口径**，并明确"单行 qty≥2 = 父COMPOSITE + 1 子件 composition_qty=qty，非展开多子件"。

### 落地要点 / 风险
1. **放开 `validateRequest`**：COMPOSITE 下限从 `parts.size()>=2` 改为 `Σqty>=2`（`parts.size()` 上限仍 ≤8 保留，指"去重子件行数"）；SIMPLE 改为 `Σqty==1`。**判定应后端按 Σqty 兜底裁决，不盲信前端 productType**（前端 Σqty 计算与后端必须同口径）。
2. **放开组合工艺参与配件校验**：`validateRequest` 现在硬求 `participatingPartIndexes.size()>=2`；单行 qty≥2 只有 1 个去重子件 → 需允许"单去重子件、该件 qty≥2"的组合工艺（participating 用 `[0]` 或该件被装配 qty 次）。这是本决策**唯一新增的边界放开点**，务必在 B2 一并改，否则单行 qty2 选组合工艺会 400。
3. 风险：前端若仍用"行数"判 SIMPLE/COMPOSITE（老逻辑），会与后端 Σqty 口径分叉 → 单行 qty2 前端发 SIMPLE、后端却应判 COMPOSITE。**前后端判定口径必须在本任务对齐为 Σqty**（写进 api §3.3）。

## 决策 2 — 组合工艺双轨收敛（R4）

### 现状事实
- 现存**两套**：① 独立 `composite_process_def`（6 条种子，**带 param_schema** 参数化：铆接压力/焊接电流…）；② 新口径 D13 = 工序库 `process_master` 过滤。
- **`process_master.process_category` 里根本没有 '组合工艺' 这个值**（0 行）。真实存在的是 `ASSEMBLY`（4 行：总装配/部件装配/螺栓连接/焊接装配，英文枚举，与 MACHINING/INSPECTION 同族）。
- 现役 `insertCompositeProcessCapacityV6` 写 `capacity.process_no = def code`、`process_name` 读 `composite_process_def`；**存量 QUOTE_ASSEMBLY 行 process_no='焊接'/'铆接'（中文名）、process_name 空** → 标识空间当前已经是乱的（正是 R4 要治的"双引擎交叉"）。
- **无任何视图引用 `composite_process_def`**（渲染不依赖它）；`buildSnapshot` 读 capacity 的 `process_no` 直接当 defCode 回显。→ 切换承载源是**受控、低渲染风险**的。
- §14 权威口径：`capacity.process_no = 组装工序 = 工序编号`（"取工序编号对应值"）。**这本就要求 process_no 是工序库的工序编号，而非 def code** —— 现役代码写 def code 其实一直偏离 §14。

### 选项对比

| 选项 | 候选源 | capacity.process_no | composite_process_def | 参数化(param_schema) |
|---|---|---|---|---|
| **2A（推荐）单一工序库口径** | `process_master WHERE process_category='ASSEMBLY'` | = `process_master.process_no`（工序编号，如 MRO-AS-0001） | 选配停止读写，仅留给 v0.4 configurator | 放弃（process_master 无此列，超范围） |
| 2B｜维持双轨 | 保留 def catalog + 另映工序编号 | def code 或工序编号（两套并存） | 继续读写 | 保留 | ← R4 要消灭的正是这个 |
| 2C｜把 def 迁进工序库 | 迁 6 条 def → process_master(ASSEMBLY) + 新工序编号 | = 迁移后的工序编号 | 迁移后废弃 | 需给 process_master 加列（超范围） |

### 推荐：**选项 2A**（单一工序库口径，标识锚点 = `process_master.process_no`）
并**订正文档口径**：候选过滤条件不是 `process_category='组合工艺'`，而是 **`process_category='ASSEMBLY'`**（DB 实值；无需新增枚举/种子，现成 4 行即候选）。

### 理由 / 取舍
1. **§14 就要求 process_no=工序编号**：2A 让候选/指纹/落库全用 `process_master.process_no`，一举把现役"写 def code"的偏离纠正到 §14 权威口径，选配 COMPOSITE 落 capacity 从此与导入组装加工费同源。
2. **单一事实来源，根除 AP-44 类双引擎**：组合工艺身份在**候选端点 → 前端选择值 → 指纹 CPROC → capacity.process_no → quotation_line_composite_process.def_code** 五处必须同一标识；2A 把它锚定为 `process_no`，任何一处漏改即静默错配（R4 核心诉求）。
3. **切换成本低**：无视图依赖 `composite_process_def`，改动只集中在 3 处读它的代码（候选端点 / `insertCompositeProcessCapacityV6` / `insertCompositeProcessesPerQuote` 校验）+ 指纹入参。
4. **取舍（必须让 PM 拍板的点）**：2A **放弃 param_schema**（铆接压力/焊接电流等参数录入）。现役 6 条参数化 def 会退出候选，取而代之是 4 条通用 ASSEMBLY 工序。需求 §4.1 组合工艺流程**只说"选择组合工艺"、未提参数录入**，且 D13 明确"组合工艺来自工序库" → 判定放弃参数化属**符合需求的减法**。若 PM 坚持要参数化，则须给 `process_master` 加 param_schema 列（本次范围外，另起需求）。

### `composite_process_def` 去留结论
- **不删表**（v0.4 configurator 迁移 V240-252 可能 FK 依赖，删 = 迁移影响面外扩）；标注为"v0.4 专用、选配不再引用"。
- 选配侧**三处解绑**：
  1. 候选端点 `GET /composite-processes` → 改读 `process_master WHERE process_category='ASSEMBLY'`，DTO `{code=process_no, name=process_name, currency=standard_currency, unit=standard_unit, defectRate=default_defect_rate}`（`icon/paramSchema` 置空或移除）。
  2. `insertCompositeProcessCapacityV6` → `process_no=选中的 process_master.process_no`、`process_name` 读 `process_master`（不再读 `composite_process_def`）。**同时补 §14 缺列**：`fixed_cost`(留 NULL，单价由 INPUT 层维护) / `default_defect_rate` / 计量单位写 **`capacity_unit`**（不是 `unit`，§14/backtask 订正）。
  3. `insertCompositeProcessesPerQuote` 的 `CompositeProcessDef.findByCodeOrThrow` → 改校验 `process_no` 存在于 `process_master`(ASSEMBLY)；`quotation_line_composite_process.def_code` 语义变为"工序编号"。

### 对指纹 / 落库 / 文档影响
- **指纹**：`computeComposite` 的 `compositeProcessCodes` 入参由"def code"改为"`process_no`"。算法不变，但**口径值域变了** → 新旧指纹不互通。因存量全测试数据（需求 §6 不处理），可接受；backtask 需注明"CPROC 值 = 工序编号"。
- **落库**：capacity.process_no 从 def code 切工序编号，并补齐 §14 全列（fixed_cost/currency/capacity_unit/default_defect_rate/seq_no）。存量脏 QUOTE_ASSEMBLY 行（中文 process_no）成孤儿，不清理（测试数据）。
- **文档**：D13 / backtask B6 / api §3.4 的 `process_category='组合工艺'` 全部订正为 `='ASSEMBLY'`；补"标识锚点=process_no 五处一致"清单。

### 落地要点 / 风险
1. **五处标识对齐清单**（PR 自检硬项，AP-44 精神）：候选端点、前端选择值、指纹 CPROC、capacity.process_no、quotation_line_composite_process.def_code —— 全部 = `process_master.process_no`。漏一处 = 组合工艺静默错配。
2. `process_master` ASSEMBLY 4 行 currency/unit/defectRate 均空 → capacity 落库要有兜底（currency→CNY、capacity_unit→留空或 KG，与自制加工费口径一致）。
3. 需 **cpq-architect 已评审**（本文即评审结论）；实现落 cpq-backend。

## 决策 3 — "规格"字段来源

### 现状事实
- `material_customer_map` **无独立规格列**。
- `material_master` **有 `specification`（规格）+ `dimension`（尺寸）**两列；`material_no` 唯一索引。
- 落值现状：`specification` 0/39（空）、`dimension` 15/39（客户产品行 dimension 形如 `3.5×3.5×0.6`）。
- 已有产品列表本就要 LEFT JOIN `model_config` 带 3D（api §2.1）。

### 选项对比

| 选项 | 取值 | 数据可用性 | 语义正确性 | N+1 |
|---|---|---|---|---|
| **3A（推荐）关联 material_master** | `spec ← material_master.specification`（可选 COALESCE dimension 兜显示） | specification 现空、dimension 有值 | specification 语义最正 | +1 LEFT JOIN（material_no 唯一索引，无 N+1） |
| 3B｜解析 customer_material_name | 从品名串里拆规格 | 品名多为空、无结构 | 靠猜、脆弱 | 无 JOIN 但逻辑脏 |
| 3C｜暂返 null 占位 | 恒 null | — | 不解决 | 无 |

### 推荐：**选项 3A**（LEFT JOIN `material_master`，`spec = specification`）
- 主取 `material_master.specification`；鉴于测试数据规格实际落在 `dimension`，**建议返回值用 `COALESCE(NULLIF(specification,''), dimension)`**，使原型/演示有值，同时 specification 优先（一旦规范落库即以其为准）。
- 过滤（api §2.1 的 `spec` query）对同一表达式做模糊匹配。

### 理由 / 取舍
1. **规格是 material_master 的一等列**，语义正确、有唯一键可 JOIN，远优于从品名解析（3B 脆弱）或恒空（3C 不解决）。
2. **N+1 可控**：已有产品列表 = `material_customer_map` LEFT JOIN `model_config`(3D) LEFT JOIN `material_master`(spec)，两个 JOIN 都命中唯一/部分唯一索引，**单条 SQL、无逐行查**，满足 R2/验收硬指标。再加一个 LEFT JOIN 对分页列表完全可接受。
3. **取舍**：specification 现全空，若只取它则 spec 全 null。用 COALESCE 兜 dimension 是"让演示有值"的务实选择，但**需 PM 确认"规格"语义到底映 specification 还是 dimension**——二者都在 material_master、都是单 JOIN，切换零成本。

### 对指纹 / 落库 / 文档影响
- 与指纹/落库无关（纯读展示字段）。
- **文档**：api §2.3 从"二选一待定"定稿为"`spec = material_master.specification`（LEFT JOIN material_no；测试期 COALESCE dimension 兜显示）"；backtask B3 规格映射同步定稿。

### 落地要点 / 风险
- JOIN 键 `material_customer_map.material_no = material_master.material_no`（唯一索引）。
- 风险低。唯一待确认 = specification vs dimension 语义（留给 PM 一句话定）。

## 附：§4.6 完整落库改造（B2）评审中发现的架构风险（不扩范围，仅列出）

1. **§14 列名订正（必修）**：`capacity` 实际计量单位列是 **`capacity_unit`**，无 `unit`。§14 表格与 backtask B2⑤ 写的 `unit` 会导致落库写错列/漏列。改造时以 DB 真实列为准。
2. **AP-53 纪律**：B2 所有 `insertXxxV6` 落的是 V6 表（material_master/material_bom(_item)/element_bom(_item)/unit_price/capacity），组件 `data_driver_path`/字段 `basic_data_path` 必须走 `$<sql_view_name>` 引用 V6，禁直接表名——B2 只落数据不碰渲染 driver，符合基线，保持即可。
3. **AP-51 行数纪律**：`writeCombomaterialBomV6` / capacity 走 `VersionedV6Writer` 版本翻转，勿引入 `Math.max(rowCount, baseRows)` 累加；幂等复用路径（指纹命中 return）已在落库前拦截，守住"不重复累加"。
4. **元素 characteristic 递增口径**：backtask B2③ 说"同主件不同组成则 +1"，但现役 `insertElementBomV6` 固定 `'2000'`。B2 改造若要支持"一个成品多材质料号"（D11 多行），element_bom 的 characteristic 需真正按组成区分，否则多材质料号元素会挤进同一 characteristic 桶。这是 B2 落库改造里最容易被漏、且直接影响元素 Tab 正确性的点，建议 architect 在 B2 详设时单列。
