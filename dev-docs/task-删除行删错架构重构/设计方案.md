## 0. 结论先行（摘要）

**根因一句话**：报价侧 driver 行有**两份平行存储**——`comp.rows`（row_data，承载 INPUT 显示值与写回目标）与「快照展开」`activeDriverExpansion`（由 `quoteCardValues.baseRows` 墓碑过滤而来，承载 driverRow / basicDataValues / effKey）——二者**只按数组下标 `i` 硬配**，而删除只给「快照展开」加墓碑过滤（少一行），`comp.rows` 不同步过滤（仍是全量），导致下标 `i` 之后两集合语义错位；叠加**行身份在 `rowKeyFields` 全空时退化为位置键**（`effKey="0"`），受控 `<input>` 本地态按错位下标写回，形成逐帧劣化的**写回反馈环**。

**推荐架构**：**方案 A ——以「内容指纹身份」（`uniqFp`）作为 driver 行的唯一身份键**，统一 display 取值 / 删除墓碑 / React key / 写回目标四条路径，彻底废除「位置 effKey + 双存储按下标对齐」。分两浪交付：

- **Phase 1（止血，低风险，用现有契约）**：删除端点改为**返回重算后的整单投影**，前端删除后**原子重灌 `comp.rows` + `quoteCardValues` + `deletedRowKeys`**（方案 D），把「双存储 desync 窗口」从「直到整页刷新」压缩到「一次往返」，立即消除持续性搅坏。
- **Phase 2（根治，中风险，改前后端行键契约）**：落地内容指纹身份 `uniqFp`，把 `editRows`/`formulaResults`/墓碑/React key/写回全部改由 `uniqFp` 定址，退役位置 `effKey`；顺带为「退役 `comp.rows`（RECORD Task6）」铺路。

**最高风险点**：Phase 2 的**前后端行键契约切换**（`editRows`/`formulaResults`/墓碑三处 key 语义变更），必须前后端同批、带**向后兼容读**、并用现网存量单（QT-1982 干净 + QT-1960 desync）先做 spike 验证 editRows 编辑值不丢。

## 1. 根因确认（逐层，真实代码行号）

四层根因均已在 master `456a1c5` 基线代码中核实。前端文件位于 `cpq-frontend/src/pages/quotation/`，后端位于 `cpq-backend/src/main/java/com/cpq/quotation/`。

### 1.1 Layer 1 —— 行身份在 `rowKeyFields` 全空时退化为「位置键」

**前端** `useCardSnapshots.ts`：
- `computeRowKey` L117：`if (!rowKeyFields || rowKeyFields.length === 0) return String(rowIndex);`
- `computeRowKey` L133：`if (!any) return String(rowIndex);` —— rowKeyFields 有配但**每段取值全空**时也退化为行号。

**后端逐字节对端**（三处口径一致，均以行号兜底）：
- `FormulaCalculator.computeRowKey`（L875 双参 / L904 四参）：全空 → `return null`（调用方按行号兜底）。
- `CardEffectiveRows.computeRowKey` L217 / L258：`return String.valueOf(idx)`。
- `RowDataMaterializer` L181：`rawKeys.add((rk != null && !rk.isEmpty()) ? rk : String.valueOf(i));`

**成立性**：来料页签 `rowKeyFields` 值全空 → `effKey` = `uniquify("0","1","2")` = `"0"/"1"/"2"`（纯位置）。实测拦截到 delete-driver-row 请求 `effKey:"0"`、fp 的 rkf 段为 `∅∅`，与此吻合。

> 注意：墓碑「双命中」= `effKey`（位置）**且** `fp`（内容指纹）。`fp` 由 `rowFingerprint` 对 driverRow **全键排序**取 canon（`deletedRows.ts` L31-47 / `DeletedRowKeys.java` L79-95），因此三行 AgNi11#-Ⅰ / H65带 / 组成件1 的 **fp 互不相同**。这一点是 Phase 2 用 `fp` 当身份的关键前提：**内容指纹本来就能唯一区分这三行，只有 `effKey` 不能。**

### 1.2 Layer 2 —— 双存储只过滤一半（核心结构缺陷）

**存储 A（快照展开，被墓碑过滤）** `QuotationStep2.tsx` `buildSnapshotExpansions` L1395-1474：
- L1435-1449：`kept = baseRows.filter(keepRow(uniqFull[i], fp, tombs))` —— 按墓碑剔行。
- L1454：`rowCount: kept.length` —— **过滤后行数**。
- L1461：`__effKey: uniqFull ? uniqFull[i] : undefined` —— 每保留行盖**完整集原始下标 i** 对应的 effKey（AP-54 C3，不随过滤重算）。

**存储 B（`comp.rows` / row_data，不被墓碑过滤）**：渲染期 `comp.rows` 始终是上次落库的全量行。

**按下标硬配二者** `manualRows.ts`：
- `splitRows` L24-28：`driverCount = exp.rowCount`（**过滤后** N-1）；`driverEditRows = rows.filter(!isManualRow)`（**未过滤** `comp.rows` 的非手动行，N）。
- `rowAt` L44-45：`if (i < driverCount) return { row: driverEditRows[i], expIndex: i }` —— 屏幕第 `i` 行的**显示行对象**取 `comp.rows[i]`（存储 B，未过滤），**同一个 `i`** 又用作 `expIndex` 去取 `activeDriverExpansion.rows[i]`（存储 A，已过滤）。

**渲染层组装** `QuotationStep2.tsx` `effectiveRows` L2381-2474：
- `ra.row` = `driverEditRows[i]` → `baseRow`/`row` = 显示与 INPUT 值来源（存储 B）。
- `basicDataValues`/`driverRow`/`__effKey`（L2448-2450, L2463-2464）= `activeDriverExpansion.rows[expIndex=i]`（存储 A）。

**删除按钮 callsite** L2696-2701：`handleDeleteDriverRow(componentId, rowKey=driverEffKey, driverRow=exp.rows[expIndex].driverRow)` —— **删除键与指纹都来自存储 A（过滤后）**，而用户眼睛看到、点的那一行是存储 B（未过滤）。

**结论**：删掉任一行后，存储 A 少一行、存储 B 不变，`rowAt` 用 `i` 同时索引两个**长度已不同**的集合 → 屏幕第 `i` 行显示 `comp.rows[i]`，却携带 `exp.rows[i]` 的 `__effKey`/`basicDataValues`（另一物理行）→ **删除键错位**（点第 1 行删掉视觉第 2/3 行）。四层里这是「删错行」的直接结构成因。

### 1.3 Layer 3 —— 受控 input 本地态粘连

`components/ComponentCell.tsx` `EditableCellInput` L202-239：本地 `useState(local)`（L212），仅在**未聚焦**时同步外部值（L214-218），失焦才 `onCommit`（L231-236）。React 按 `key` 复用 DOM 节点。

`QuotationStep2.tsx` `<tr key={rowKey}>` L2552，`rowKey = driverEffKey ?? activeUniqRowKeys[i] ?? String(i)`（L2451-2453）。因 Layer 1，`driverEffKey` = 位置键 `"0"/"1"/"2"`。删中间行后剩余行位置键重排（若删除同时 re-index），React 按 key 复用旧 DOM → `EditableCellInput` 本地 `local` 残留旧值 → 字段值「跨行串行」。

> L1 已合入 master 的修复（`<tr key={rowIndex}>` → `key={rowKey}`）**只部分缓解**：`rowKey` 本体仍是位置键（`__effKey` 用完整集下标，删除后 `"1"/"2"` 相对稳定，但仍是位置语义），一旦 baseRows 本身重排/重展开即失稳。根治仍需 Layer 1 的稳定身份。

### 1.4 Layer 4 —— 写回反馈环（致命，逐帧劣化）

**driver 行的写回下标未做身份映射** `QuotationStep2.tsx` L2454-2457：
```
const realRowIndex = ra.isManual
  ? activeComponent.rows.indexOf(ra.row)   // 手动行：对象引用映射（正确）
  : i;                                     // driver 行：直接用 i（注释自承认假设"非手动行顺序一致"）
```
该假设在存储 A/B desync 后**不成立**。

**多个 `comp.rows` 写入者都按此错位下标写**：
- `onCellChange` L2623 → `handleRowChange(activeComponentDataIndex, realRowIndex=i, k, val)`（`handleRowChange` L1599-1620 按下标 patch，越界补空行 L1611-1614）。
- `onCellBlur` L2624-2641 → `handleInputBlur(idx, realRowIndex=i, k)`（L1846）+ `handleSnapshotCellEdit(componentId, rowKey, k, committedVal)`（L1934，按 rowKey=位置 effKey 写 editRows）。
- **快照回填 bake effect** L1712-1770：`for ri in exp.rowCount` 写 `comp.rows[ri]`（L1765 `rows[w.ri][w.key]=w.value`）——desync 后 `exp.rowCount=N-1` 却按 `ri` 写全量 `comp.rows`，把默认值/空值烘进错位行（这解释了「未显式编辑也逐帧被搅坏」）。
- **prune effect** L1547-1571：按 `driverRows.slice(0, exp.rowCount)` 裁 `comp.rows`，desync 时会把「多出来」的正确行裁掉。

**受控 input 残留旧值（Layer 3）× 错位下标（Layer 4）** → `EditableCellInput.onBlur` 用残留 `committed`（旧/空）经 `handleRowChange` 写 `comp.rows[i]`（错行），实测把 `组成件1.料件` 写成 `null`、`realRowIndex` 从 2 退成 1，每次重渲染再劣化一层。

### 1.5 顶层结构判定：为什么点修不住

- **删除端点不回灌**：`QuotationService.deleteDriverRow` L2849-2881 只在 DB 追加墓碑并 `refreshQuoteCardValues`，**端点返回 `ApiResponse<Void>`**（`QuotationResource` L589），`quotationService.deleteDriverRow`（`services/quotationService.ts` L192）**丢弃响应**。
- **前端只做乐观墓碑**：`handleDeleteDriverRow` L1642-1650 只 append 墓碑到 `comp.deletedRowKeys`，**不 splice `comp.rows`、不重灌快照**。
- 于是 `comp.rows`（N）与快照展开（N-1）的 desync **持续存在到下一次整页 reload**，任何 autosave / onBlur / bake 在此窗口内按下标写都错位。

> 这解释了用户「盖 `__origIndex` + `realRowIndex` 按对象引用 indexOf」的收敛修复为何**首帧对、随后又坏**：它只重排了「读」的对齐，没有消除「存储 B 永不过滤」这个 desync 源，写回环继续按下标污染 `comp.rows`。**问题不在单点对齐，在双存储 + 位置身份这个设计本身。**

## 2. 架构决策（候选对比 + 推荐）

### 2.1 目标不变量（任何方案必须满足）

1. **单一行身份**：display 取值、删除定址、React key、写回目标四条路径用**同一个稳定身份**，禁止按数组下标跨存储对齐。
2. **两存储永远同过滤**：只要有一处按墓碑剔行，所有读/写/持久化都只见同一份过滤后集合（或干脆退役其中一份）。
3. **身份对删除/重排稳定**：删第 1 行不改变第 2/3 行的身份；baseRows 重展开且内容不变时身份不变。
4. **COSTING 侧隔离不动**（spec §3.7）：COSTING 侧 `__effKey=undefined`、无删除，本次改动 QUOTE-only。
5. **AP-51 行数权威**：`rowCount` 取过滤后有效行数（`effRows.length`），不 `Math.max` 累加历史脏行。
6. **前后端逐字节对齐**：任何身份/墓碑口径改动，`deletedRows.ts ↔ DeletedRowKeys.java`、`useCardSnapshots.ts ↔ FormulaCalculator/CardEffectiveRows/RowDataMaterializer` 必须同改同测。

### 2.2 候选方案对比

| 方案 | 核心做法 | 优点 | 缺点 / 风险 | 判定 |
|---|---|---|---|---|
| **A. 单一身份存储** | 用稳定行身份（内容指纹 `uniqFp`）统一 display/delete/key/writeback；driver 行不再按下标跨 `comp.rows` 与快照对齐 | 根治四层；顺带修撞键、editRows 重排丢值、未来退役 `comp.rows`；身份内容派生、无需持久 UUID/迁移新数据 | 改前后端行键契约（editRows/formulaResults/墓碑 key 语义）；需向后兼容读 + 存量单验证 | **推荐（目标态）** |
| **B. 删除时原子同步双存储** | `handleDeleteDriverRow` 同时 splice `comp.rows` + 加墓碑，写回按身份 | 直觉简单 | **splice 哪一行？** 位置 effKey 撞键/全空时无法可靠定位要删的 `comp.rows` 行 → **仍需稳定身份**，B 不能独立成立；且未解决 autosave/bake 其他下标写入者 | 并入 A（身份到位后 splice 才可靠） |
| **C. `comp.rows` 也走墓碑过滤** | 渲染/autosave/写回全部只认过滤后 `comp.rows` | 消除 desync | 过滤后子集下标 ≠ 原集合下标，写回仍要映射回原下标（AP-54 老坑重演）；且 `comp.rows` 与 baseRows 双源仍在，治标 | 不单独采纳；被 A 覆盖 |
| **D. 删除服务端权威 + 原子重灌** | 端点返回重算后的 `quoteCardValues` + 物化 `row_data`；前端删除后**同时替换两存储** | 用**现有契约**即可落地、低风险、可立即测；恢复索引对齐（两存储都 = baseRows 减墓碑、同序） | 失去乐观即时性（一次往返）；未改位置身份 → baseRows 重排/撞键仍脆；治标 | **推荐（Phase 1 止血）** |

**关键论证（D 为何能恢复对齐、A 为何是终态）**：
- 前端快照展开（`buildSnapshotExpansions`）按 baseRows **原序**保留非墓碑行；后端 `RowDataMaterializer` L204-221 同样按 baseRows **原序 skip 墓碑行并重编号**。**两者产出同序的 N-1 集合**。故只要删除后把 `comp.rows` 从服务端物化结果重灌（D），`driverEditRows[i]` 与 `exp.rows[i]` 立即恢复同序对齐 → Layer 2 desync 消失、Layer 4 按下标写回重新落到对的行。这是 D 能止血的原理。
- 但 D 未改「位置 effKey」：`editRows`/`formulaResults` 仍按位置 key 存，一旦 driver 重展开或撞键，编辑值仍会串。故 **D 是止血、A 是根治**。

### 2.3 推荐：A（内容指纹身份）为目标态，D 为 Phase 1 过渡

**为何用「内容指纹 `fp`」而非新铸 UUID 作身份**：
- driver 行是**查询派生**的，没有天然持久 PK；新铸 UUID 要么随重展开丢失、要么需额外持久化映射（还是得靠内容重匹配）。
- 系统里**早已存在**内容派生身份 = `rowFingerprint`（`fp`）。墓碑「双命中」用 `fp` 恰恰是因为 `effKey` 位置不稳、要靠 `fp` 内容消歧。**`fp` 就是现成的稳定身份**，前后端都已在算。
- 来料三行 `fp` 天然互异（Layer 1 注）；真正字节全同的行用 `uniquifyRowKeys` 同款 `#序号` 兜底（与既有撞键消歧一致，记忆 `cpq-rowkey-uniqueness-disambiguation`）。
- 无需为新数据做迁移（`fp` 从 baseRows 现算）；存量墓碑（`effKey`+`fp`）保留**双命中兼容读**即可。

> 术语：下文 **`rowIdentity` / `uniqFp`** = `uniquify(baseRows.map(fp))`，`fp = rowFingerprint(rowKeyFields, driverRow)`。它替代现在的 `effKey` 成为 driver 行的**唯一定址键**。

## 3. 稳定行身份设计（`uniqFp`）与契约影响

### 3.1 身份定义

- **driver 行**：`rowIdentity[i] = uniqFp[i]`，其中 `uniqFp = uniquifyRowKeys(baseRows.map(br => rowFingerprint(rowKeyFields, br.driverRow)))`。
  - `fp` 已对 driverRow 全键排序取 canon，天然内容派生、位置无关。
  - `uniquify` 处理字节全同行（`#0/#1`，按 baseRows 稳定原序）。
- **手动行**（`_origin='manual'`）：driver 无 fp，铸 `rowIdentity = "manual:" + 稳定序号`。建议在 `handleAddRow`（L1667-1676）落库时写一个持久 `_rowId`（如 `manual-<uuidv4>`），避免手动行也退化为位置键。手动行写回已用对象引用 `indexOf`（L2455-2456），身份化后更稳。

### 3.2 契约影响面（必须前后端同批）

| 载体 | 现状 | 目标 | 影响文件 |
|---|---|---|---|
| `editRows[].rowKey` | 位置 effKey | `uniqFp` | 后端 `CardSnapshotService.refreshQuoteCardValues`（写 editRows key）、`FormulaCalculator`（uniquify/编辑查表）、`RowDataMaterializer` L183/186-190；前端 `useCardSnapshots` L232-234/256-261、`handleSnapshotCellEdit` L1934-1954 |
| `formulaResults[].rowKey` | 位置 effKey | `uniqFp` | 后端 `FormulaCalculator.calculate`/`RowDataMaterializer` L160-171；前端 `snapByComp` L1920-1932、渲染读 L2487 |
| `deleted_row_keys[]` 墓碑 | `{effKey, fp}` 双命中 | `{rowId=uniqFp}` 单键（**保留旧双命中兼容读**） | `DeletedRowKeys.java`（新增 `keepMaskByRowId`，保留 `keepMask`）、`deletedRows.ts`（同步）、`QuotationService.deleteDriverRow` L2849-2881、`buildSnapshotExpansions` L1436-1449 |
| React `key` / 删除传参 | `driverEffKey` | `rowIdentity` | `QuotationStep2` L2448-2453/2552/2699 |
| 写回目标 | `realRowIndex`（下标） | 按 `rowIdentity` 定址（driver 行经 editRows；或按 fp 在 `comp.rows` `findIndex`） | `handleRowChange`/`patchRowField`/bake effect/`handleInputBlur` |
| `snapshot_rows` / baseRows | `[{driverRow, basicDataValues}]` | **不加物理列**（身份现算）；可选加冗余 `__rowId` 便于调试 | 无强制迁移 |

> **不需要 Flyway 迁移**（身份从既有 baseRows 现算）。若为可读性给 baseRows 冗余落 `__rowId`，属可选优化，不改主契约。

### 3.3 向后兼容读（存量墓碑）

现网存量单的 `deleted_row_keys` 是 `{effKey, fp}`。切 `uniqFp` 身份后：
- 新删除写 `{rowId: uniqFp, effKey, fp}`（三字段冗余写，便于回滚与兼容）。
- 过滤时优先按 `rowId` 命中；`rowId` 缺失（老墓碑）回退 `{effKey, fp}` 双命中。
- 因来料等场景 `fp` 本就唯一，`uniqFp`（单行时）≈ `fp`，绝大多数存量墓碑天然可映射。此兼容层在 Phase 2 落地、保留 2 个版本周期后可清理。

## 4. 写回环根治（Layer 4）

四条 `comp.rows` 写入路径必须去「按下标写 driver 行」：

1. **INPUT 编辑（首要）**：driver 行的用户输入**只写快照 `editRows`（按 `rowIdentity`）**，经 `editQuoteCardValue` 服务端重算回灌，不再靠 `handleRowChange` 按下标改 `comp.rows`。`ComponentCell` 的 `<input value>` 改读「editRows[rowIdentity][field] ?? driverRow ?? default」而非 `comp.rows[i][key]`。这是 RECORD Task6「row_data 退役」的正式落点，driver 行先行。
2. **受控 input 同步纪律**：`EditableCellInput`（L202-239）保持「聚焦不回灌」，但外部值来源改为「按 `rowIdentity` 取 editRows 值」；React `key=rowIdentity` 稳定后，删除重排不再复用错节点 → 本地态不再粘连。
3. **bake effect（L1712-1770）**：改为「按 `rowIdentity` 找目标行」写入，或在 driver 行退役 `comp.rows` 后直接烘进 `editRows`；迭代口径与 `exp.rowCount`（过滤后）一致，禁止按 `ri` 直写全量 `comp.rows`。
4. **prune effect（L1547-1571）**：driver 行退役 `comp.rows` 后本 effect 对 driver 行不再需要（只保留手动行裁剪）；过渡期若保留，裁剪必须按 `rowIdentity` 集合差，不按 `slice(0, rowCount)` 位置裁。

**过渡期（Phase 1，仍用 `comp.rows`）的最小纪律**：删除走服务端权威重灌（D）后两存储恢复同序，`realRowIndex=i` 暂时安全；但仍要在「墓碑乐观追加 → 服务端响应到达」窗口内**抑制写回**（pending-delete 标志：该组件处于删除在途时，onBlur/bake 不写 `comp.rows`），关掉唯一的窗口内污染源。

## 5. 必须覆盖的边界

| # | 边界 | 设计要求 |
|---|---|---|
| ① | 手动行 `_origin='manual'` 与 driver 行混排 | 手动行身份 `manual:<持久序号>`；`splitRows` 仍「driver 段在前、手动段在后」；手动行写回按对象引用/`_rowId`，不受 driver 过滤影响（现 L2455-2456 已正确，身份化后更稳） |
| ② | 连续多次删除 | 每次删除按 `rowIdentity` 单键墓碑，互不影响其余行身份；Phase 1 每删一次服务端重灌一次两存储；断言「删 N 次后剩余 M 行身份不变、值不串」 |
| ③ | 删除后刷新 / 重进编辑 | 墓碑持久化在 `deleted_row_keys`（不复活）；`enrichComponentData` 两处必须透传 `deletedRowKeys`（FixC2 教训）；`saveDraft` 重建 component_data 按 componentId 回填墓碑（FixC1 教训）；刷新后 `buildSnapshotExpansions` 用 `rowIdentity` 过滤 → 不复活、不错位 |
| ④ | `row_data` 与 baseRows 历史 desync 存量单（如 QT-20260712-1960 row_data=2 vs snapshot=4） | 身份化后 display 不再依赖 `comp.rows` 下标 → 天然免疫历史 desync；driver 行以 baseRows(=snapshot_rows) 为准，`comp.rows` 仅作 INPUT 值来源且按 `rowIdentity` 匹配（匹配不上 → 走 driverRow/default，不塞错行）。Phase 1 阶段：删除服务端重灌会把 `comp.rows` 拉回与 baseRows 同序，自动修 desync |
| ⑤ | 报价侧 vs 核价侧隔离 | COSTING 侧 `uniqFull=null → __effKey=undefined`（L1429-1432）、无删除按钮（L2702-2703 显示 🔗）；本方案所有身份/墓碑逻辑 `side==='QUOTE'` 门控，COSTING 行为零变化（spec §3.7） |
| ⑥ | 不破坏已合入 master 的展示修复 | 不动 footer 对齐、`.qt-cost-table-wrap` 横向自适应；L1 `<tr key={rowKey}>` 顺势升级为 `key={rowIdentity}`（同方向增强，非回退）；`ReadonlyProductCard.tsx`（L571/591/595/603/639 用 `splitRows`/`rowAt`/`buildUniqueRowKeys`）只读无删除，但 display 对齐同理，必须与编辑页同口径改（否则详情页 desync 复现） |

## 7. 验收标准（可测断言）

用户后续以 Playwright 拦截 `delete-driver-row` + 渲染层 console 仪表 + DB 反查复测。断言集：

**A. 删对行**
- A1：来料 3 行，点 DOM index 0 的 ✕ → 拦截请求 `rowId`（Phase 2）/ `effKey+fp`（Phase 1）对应 **AgNi11#-Ⅰ 行**；渲染后**恰好该行消失**，剩 H65带 / 组成件1 两行且**字段值逐字段不变**。
- A2：`activeDriverExpansion.rowCount === 2` **且** `comp.rows` 参与渲染的 driver 行数 === 2（两存储同数）。

**B. 多次重渲染不劣化**
- B1：删除后强制触发 ≥3 次重渲染（切 Tab 往返 / autosave tick），`组成件1.料件` **始终非 null**、`realRowIndex`（或身份定址）稳定不漂移；console 仪表 `'加载中' final count = 0`。

**C. 多删稳定**
- C1：连删 2 行 → 剩 1 行，身份/值正确；DB `deleted_row_keys` 恰 2 条墓碑；`row_data` 行数 = 1。

**D. 刷新不复活 / 不错位**
- D1：删 1 行后整页 reload / 重进编辑 → 仍 2 行、被删行不复活、剩余行值与顺序不变（验墓碑持久化 + FixC1/C2 透传）。
- D2：`restore-driver-rows` 后恢复 3 行，值与初始一致。

**E. 隔离与不回归**
- E1：COSTING 侧同单同页签 driver 行**无 ✕、不可删**，渲染与改前逐字节一致（`quoteCardValues`/`costingCardValues` 背靠背对比）。
