## 2. 需求背景与目标 ⭐

- **要解决什么问题 / 痛点**：

  V44 系表 `mat_*`（`mat_part`/`mat_bom`/`mat_process`/`mat_fee`/`mat_customer_part_mapping`/`mat_plating_fee`/`mat_plating_plan` + 各 `*_staging`）自 **2026-06-02 起停止真实业务写入**，新数据全部走 V6（`material_*`/`element_*`/`unit_price` + `basicdata.v6` 下 Q01~Q17/P01~P23 handler + `VersionedV6Writer`）。

  但**仍有活跃功能代码在读/写这些冻结的废弃表**——按业务判据「废弃表 + 活功能引用 = bug」，这类是最需要清理的隐患。同时，一批旧核价模板/视图/引擎已被 `核价通用模板`（V6）取代但仍残留，可一并清理。

  **冻结实证**（✅自验）：
  - `mat_part` 264 行，最后真实写入 2026-06-02；现役 ~85 个 V6 料号中仅 **2 个** 在 mat_part；`mat_part.unit_weight` 对全部 264 行均为 NULL。
  - `mat_customer_part_mapping` 167 行，冻结于 2026-05-22；现役料号仅 2 个在其中。
  - `mat_fee`/`mat_process` 在 2026-07-22 有零星写入，经查是**测试夹具**（合成料号 `CLR-A-PART`/`VER-T11-001` + 合成客户 `5x000000` UUID），非业务。
  - `import_record` 7 月全部来自 V6；import-session/staging 链路 5-15/5-22 后零新行。

- **本次要达成的目标**：

  1. 摘除唯一的活跃写入引信——「V5 增强导入」按钮（防止继续污染 mat_*）。
  2. 修复报价热路径上「活功能读冻结表」的 5 处 bug（= BL-0069），改读 V6。
  3. 清理已被取代的旧核价模板（7 个）→ 视图（7 张）→（后续）底表。
  4. 分阶段退役 V5 / import-session 死链路 + 旧核价引擎，最终让 `mat_*` 逐张可安全 DROP。

- **不做什么（明确边界）**：

  - **本期不直接 DROP 任何 `mat_*` 表**——它们仍被活代码读（见 §4 审计），退役是下游结果，非本期动作。
  - **本期不删共享底表**（`mat_fee`/`plating_fee`/`exchange_rate`/`mat_part`/`mat_customer_part_mapping`/`material_recipe`）——它们有活跃 V6 消费方。
  - 旧核价底表（16 张 `costing_part_*`/`costing_summary*`/`costing_*price`）的 DROP 需先删对应 Java 模块，属独立后续批次。
  - 历史报价单/核价单的冻结快照不受影响（自包含 JSONB，删定义/视图不动它）。

## 9.5 需求澄清与方案定稿（技术总监 2026-07-23）

### Q1 · 本期范围 = 全部 7 阶段 ✅ 已确认

不是分批止血，而是一次性做完 §9 施工顺序的 7 个阶段（含原本标注「多周·architect 主导」的 mat_* 退役）。全部在同一 worktree 分支，一次性交付统一验收。

### Q2 · DROP 全部改为「改名 `_drop` + 观察验证」✅ 需求方裁定

**核心手法变更**：所有表/视图**不执行 `DROP`**，一律 `ALTER TABLE/VIEW ... RENAME TO xxx_drop`。

- **可逆**：改回即回滚，数据零损失（比 DROP 安全得多）
- **大声报错**：遗漏的活代码触发 `relation "xxx" does not exist`，不是静默取空——这正是要的验证信号
