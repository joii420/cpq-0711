## 2. 需求背景与目标 ⭐

### 要解决什么问题 / 痛点

- 报价单的组成件 BOM 中，最终销售料号可能呈**树状结构**，当前报价卡片按平铺渲染，树结构不可见
- `material_bom_item.characteristic` 已有三态定义：`RECIPE` 材质 / `ASSEMBLY` 组成件 / `OUTSOURCED` 外购件（2026-07-20 已交付合 master）
- 用户需要在报价单 BOM 树上**新增行**，但新增行的业务类型无从确定
- 页签缺少类型语义，系统无法知道某个页签的料号列代表什么

### 本次要达成的目标

1. 报价卡片按 BOM 树渲染（复用核价侧 spine 树引擎），并支持树上编辑
2. 页签增加**类型属性**：`BOM` / `材质元素` / `零件` / `组成件` / `外购件` / `主件`，其中 **`BOM` 类型为树状页签**
3. 用户可在树的任意节点上新增叶子料号，其类型由系统自动判定（不需人工声明）
4. 报价数据可以正确展示各个页签的数据

### 不做什么（明确边界）

- **报价 Excel 视图 / 导出不做树状渲染** —— Excel 侧只展示主料号数据，维持平铺
- **新增行不回写 `material_bom_item`** —— 只进报价单快照
- **不支持新增非叶子节点** —— 树骨架的人工扩展仅限叶子层
- **不支持新增不存在于任何页签的料号** —— 按业务规则这是错误输入
- 核价侧渲染行为不得改变（共用引擎，须零回归）

## 4. 功能详述 ⭐

### 4.1 用户场景 / 操作流程

**场景一：查看树状报价单**

1. 用户导入 / 创建报价单
2. 打开报价单编辑页，配了 `BOM` 类型的页签按 BOM 树结构渲染
3. 树按 spine 全节点呈现：即使某节点在该页签没有业务数据，也补空行保留骨架
4. 支持折叠 / 展开

**场景二：树上新增叶子料号**

1. 用户在树的任意节点行点击 `+`
2. 系统弹出候选料号列表（来源：当前报价单各页签已渲染行的料号）
3. 用户选中一个料号
4. 系统按 §4.3 规则自动判定该料号的类型，无需用户声明
5. 新叶子挂在被点击节点之下，业务列留空待用户填写

**场景三：树上删除**

- 行级删除（`×`）：删除该节点内的一条明细行；删到只剩一行时退化为空行，保留骨架
- 节点级删除（剪枝）：移除整个节点及其子树，**跨所有树页签联动**

### 4.2 界面与交互（前端相关）

| 交互点 | 说明 |
|---|---|
| 树渲染 | 按 `__lvl` 缩进；节点主行显示折叠箭头；固定列显示 料号 / 父料号 / 版本（只读） |
| 行内 `+` | 每行前显示，点击新增子叶子；与默认的 `×` 删除行对应 |
| 行内 `×` | 行级删除 |
| 剪枝入口 | 区别于 `×`，仅在节点首行显示 |
| 删除确认弹窗 | **剪枝与行删除均必须弹窗**，展示完整删减明细：将被移除的树节点及子孙料号 + **各页签将被级联删除的行**；用户确认后才执行，不允许静默级联 |
| 行编辑 | **所有行**的业务列凡可填即可编辑；不因该节点是否拥有子节点而变化 |
| 系统列 | `__nodeId`/`__parentId`/`__lvl` 完全不可见；料号/父料号/版本显示为**只读**固定列 |
| `+` 置灰 | 节点类型为材质 / 外购件时禁用，hover 显示原因（不隐藏按钮，遵循列表操作规范） |
| 候选料号选择 | 本地下拉（数据已在前端 `componentData` 中），无需远程搜索；料号取自各页签 `part_no_field` 指定列（非启发式猜测） |

### 4.3 业务规则与计算逻辑（后端相关）

**规则一：页签类型属性 + 料号列标识**

页签（组件）声明三个配置，共同定义"该页签的料号列 / 料号名称列，及其所属的业务语义"：

| 配置 | 含义 | 取值 |
|---|---|---|
| `tab_type` | 该页签料号的业务语义 | 5 类枚举（下表） |
| `part_no_field` | 该页签**哪个字段是料号列** | 从该组件已有字段中选（字段 name） |
| `part_name_field` | 该页签**哪个字段是料号名称列** | 从该组件已有字段中选（字段 name），可空 |

> **2026-07-21 补充**：`part_no_field` / `part_name_field` 是立项时「(料号列/料号名称)」表述的落地，
> 初版文档遗漏。**类型判定（规则二）与加叶子候选料号采集必须依据 `part_no_field` 显式配置，
> 禁止靠字段名包含"料号"等启发式猜测**。
> - 树页签（`tab_type=BOM`）的料号取系统列 `__hfPartNo`，`part_no_field` 可不配
> - 非树页签（材质元素 / 零件 / 外购件 / 主件）**必须配 `part_no_field`**，否则该页签不参与类型判定匹配

值域共 **5 类**（2026-07-21 确认）：

| 类型 | 语义 | → `characteristic` |
|---|---|---|
| `BOM` | **树状页签**（结构角色，非物料语义） | — |
| `材质元素` | 该页签料号为材质 | `RECIPE` |
| `零件` | 该页签料号为零件 | `ASSEMBLY` |
| `外购件` | 该页签料号为外购件 | `OUTSOURCED` |
| `主件` | 成品 = 树根 | — |

> **已确认**：原提出的「组成件」与「零件」是同一个东西，值域**合并为「零件」**，不保留同义项。
> 理由：同义选项会导致配置者无从选择、判定链重复实现、后续误以为两者不同而分叉。

**树渲染的判据（2026-07-21 确认，Q1）**：

`tab_type = 'BOM'` 是「该页签是否按树渲染」的**唯一判据**。保存组件时后端**自动同步**既有字段
`bom_recursive_expand = true`，用户只需配置一个「页签类型」，不需要理解两个字段的关系。

> **背景**：`bom_recursive_expand` 是既有的**组件级全局开关**。现网 3 个开启该开关的组件
> （`COMP-0021__imp1__imp1` / `COMP-0039` / `COMP-0042`）共 34 处模板引用，**全部在 COSTING 模板，
> 报价侧 0 处**（2026-07-21 实查共享库）。
>
> ⚠️ **配置约束**：因该开关是组件级而非 Tab 级，同一组件被多个模板共用时开关一开全生效。
> 故报价侧树页签**必须新建专用组件，不得复用核价侧已有的树组件**，否则会把核价模板一并改成树。
> 开关下沉到 Tab 级（`template_component` override）的架构改造转 BACKLOG。

**规则二：新增叶子的类型判定链**

类型**不由用户声明**，由「料号出现在哪个类型的页签」推导：

| 序 | 条件 | 判定 |
|---|---|---|
| 1 | 料号出现在「材质元素」类型页签 | 材质 |
| 2 | 料号出现在「零件」类型页签 | 零件 |
| 3 | 料号未出现在「零件」页签，但其**直接子节点**挂有材质 | 零件（结构推导） |
| 4 | 料号出现在「外购件」类型页签 | 外购件 |
| 5 | 料号出现在「主件」类型页签 | **错误** —— 成品是树根，不能作为他人叶子挂入 |
| 6 | 以上皆不满足（零命中） | **错误** —— 该料号不是有效的报价产品，拒绝新增 |

- 规则 3 为**通用类型判定规则**，适用于树上所有节点（含 SQL 递归产生的中间层节点），不限于新增时刻
- 匹配范围为**当前报价单该页签已渲染的行**，不是基础数据主表全量
- **规则 3 只看直接子节点**（2026-07-21 确认，Q4）：语义是「由材质直接构成的零件」。若放宽到任意深度子孙，
  几乎所有中间节点都会命中，规则失去区分力
- **冲突粒度**（2026-07-21 确认，Q3）：只有命中**不同类型**的页签才算冲突（409 由用户裁决，不静默取第一个）；
  同一料号命中多个**同类型**页签（如两个「材质元素」Tab）**不算冲突**，类型无歧义，正常判定

**规则三：加子约束（双向，2026-07-21 确认）**

能否在某节点下加叶子，取决于**该节点的类型**，与它是不是叶子无关：

| 节点类型 | 能否加子 | 原因 |
|---|---|---|
| 零件 | ✅ 可以 | 零件可继续往下拆 |
| 材质 | ❌ 禁止 | 终端物料 |
| 外购件 | ❌ 禁止 | 终端物料 |

约束是**双向**的：

- **正向**：材质 / 外购件类型的节点，`+` 置灰不可加子
- **反向**：**已拥有子节点**的料号，禁止被添加到「材质元素」或「外购件」类型页签中（跨页签一致性校验）

**加子不影响父节点自身的数据** —— 父节点原本可填的单元格，加完子之后照样可改。
**不存在「变成非叶子就转只读」的规则**（该规则曾被提出，已于 2026-07-21 推翻）。

**规则四：级联删除（2026-07-21 确认，本需求的关键规则）**

剪枝 / 删行不只影响树页签，需**级联清理该料号在其余所有页签中的数据**。

**判定链**：

1. 执行删除后，重算被删料号（剪枝时为该节点及其**全部子孙**料号）在当前报价单树上是否还有**剩余 occurrence**
2. **仍有剩余** → 只删当前操作的行 / 枝，其余页签数据**不动**
3. **已无剩余** → 级联删除该料号在**所有其他页签**中的数据行

> DAG 重复子件（同一料号挂在多个父件下）是本规则的核心场景。例：料号 X 同时挂在零件A、零件B 下，
> 剪掉零件A 下的 X 后，零件B 下的 X 仍在使用材质页签中 X 的数据，此时**不得**删除。
> 现网已有实例（`3110520789` 同挂两个父件），必须覆盖测试。

**行级删除（`×`）同样适用**：删除的若是该料号在树上的**最后一行**，触发与剪枝相同的级联；否则只删当前行。

**级联的触发范围**（2026-07-21 确认，Q5）：只有 **`tab_type=BOM` 树页签内**的删除才触发级联判定。
其余页签（材质元素 / 零件 / 外购件）的行删除是普通删除，**不反向影响树**。
若某料号在其来源页签被删空，树上该节点退化为空行保留骨架（符合「spine 全节点呈现」原则）。

**实现方式**：一律走**墓碑机制**（标记删除），不做物理删除。
- 树节点 → `quotation_line_item.deleted_tree_nodes`
- 其余页签行 → 各组件的 `deleted_row_keys`

**交互要求**：执行前必须弹窗展示**完整删减明细**（将被移除的树节点及子孙料号 + 每个页签将被删除的行），
用户确认后才执行。不允许静默级联。

**规则五：树的产生与冻结**

- 树在**物化时**产生（建单 / 刷新快照阶段），写入 `snapshot_rows` 后**冻结**
- 已提交的报价单再次打开，树结构不随基础数据 BOM 变动而漂移
- 报价侧使用**独立的递归 SQL 配置**（与核价侧口径解耦）

**规则六：小计口径**

- spine 全节点展开后父节点通常为空行，`Σ 所有行` 不会父子重复累加
- 父子聚合（累乘用量、子树汇总）属**业务公式层**，不在渲染层实现
- 被剪枝节点必须同时从前端展示与后端小计中排除（两者必须同源）

### 4.4 数据字段

| 字段 | 含义 | 类型 | 必填 | 示例 / 取值 |
|------|------|------|------|------------|
| `component.tab_type` | 页签类型属性 | varchar | 否 | `BOM` / `材质元素` / `零件` / `外购件` / `主件`（5 类） |
| `component.part_no_field` | 料号列字段名 | varchar | 非树页签必填 | 该组件某字段的 name |
| `component.part_name_field` | 料号名称列字段名 | varchar | 否 | 该组件某字段的 name |
| `costing_bom_tree_config.usage` | 递归 SQL 配置的用途维度 | varchar(16) | 是 | `QUOTE` / `COSTING`，默认 `COSTING` |
| `quotation_line_item.deleted_tree_nodes` | 节点级墓碑（剪枝） | jsonb | 否 | `["<nodeId>", ...]` |
| `__nodeId` | 节点物化路径（per-occurrence 唯一） | string | 系统列 | 手工叶子为 `<宿主nodeId>/__manual_<uuid>` |
| `__parentId` | 父节点 id | string | 系统列 | |
| `__lvl` | 树层级（根=0） | int | 系统列 | |
| `__hfPartNo` / `__parentNo` / `__bomVersion` | 料号 / 父料号 / BOM 版本 | string | 系统列 | 显示为只读固定列 |
| `__manual` | 是否手工新增行 | boolean | 系统列 | `true` |
| `__sourceComponentId` | 手工叶子的料号来源页签 | uuid | 系统列 | 类型判定的权威来源 |

## 8. 验收标准 ⭐

| # | 验收项 | 判定标准 |
|---|---|---|
| AC-1 | 树渲染 | 配 `BOM` 类型的页签按 BOM 树结构渲染，层级缩进正确，折叠/展开可用 |
| AC-2 | spine 全节点 | 某节点在该页签无业务数据时补空行，骨架完整，无孤儿节点 |
| AC-3 | 加叶子 | 树任意节点可加叶子；候选料号来自当前报价单各页签；新叶子挂在正确位置 |
| AC-4 | 类型自动判定 | 按 §4.3 规则二五条链路各验一例，含结构推导与零命中拒绝 |
| AC-5 | 加子类型约束 | 材质/外购件节点 `+` 置灰且 hover 有原因；零件节点可加子；已有子节点的料号不能被加入材质/外购件页签 |
| AC-5b | 行可编辑性 | 所有行业务列可填即可编辑；加子后父节点数据仍可修改；系统列恒只读 |
| AC-6 | 行级删除 | 删除正确的行（不删错行）；删到只剩一行退化为空行保留骨架 |
| AC-7 | 剪枝 | 删节点连带子树消失，且**跨所有树页签联动**；小计同步排除 |
| AC-7b | 级联删除 | 料号在树上无剩余 occurrence 时，其在**其余所有页签**的数据一并墓碑删除 |
| AC-7c | DAG 不误删 | 料号仍有其他 occurrence 时（如 `3110520789` 同挂两父），其余页签数据**不得**被删 |
| AC-7d | 删除确认弹窗 | 剪枝与行删除均弹窗展示完整删减明细（树节点 + 各页签受影响行），确认后才执行 |
| AC-8 | 冻结 | 已提交报价单再次打开，树结构与提交时逐字节一致 |
| AC-9 | 刷新稳定 | 连续刷新 3 次行数稳定，不累加（AP-51） |
| AC-10 | 核价侧零回归 | 核价树渲染逐位不变 |
| AC-11 | 无「加载中…」残留 | E2E 全 Tab `'加载中' count = 0` |
| AC-12 | 三视图一致 | Step2 编辑页与详情页只读卡片渲染一致（AP-50） |
