## 2. 需求背景与目标 ⭐

### 要解决什么问题 / 痛点

- 现状：报价基础数据**导入即全局生效**（`QuoteImportService` 的 17 个 `Q*Handler` 在导入那一刻就通过 `VersionedV6Writer` 完成「比对→升版→翻 `is_current`」），数据立刻成为全局当前版本，**未经任何审核就能被其他报价单引用**。
- 痛点：一张未审核的报价单，其导入/编辑的料号数据会污染全局主数据；下游报价单在「从已有产品添加」里会看到、并引用这些未审核数据。

### 本次要达成的目标

1. **报价数据导入后不触发全局升版**，仅停留在当前报价单可编辑、可渲染，**其他报价单看不到、引用不到**。
2. **财务在核价单将报价单审核通过后**，把该报价单**此刻的有效数据**回填基础数据、触发升版落地。
3. **审核通过后，下一次引用这个产品料号，带出的数据与本次报价单完全相同**（这是本需求的核心验收语义 —— A-2「报价单即真相」的彻底形态）。
4. 「从已有产品添加」列表只出现**已审核通过**的产品料号；选配发号的产品同样跟随报价单审核通过后才能被引用。

### 不做什么（明确边界）

- **核价侧数据不做本套 pending/回填处理**（`system_type='PRICING'` 的导入与渲染维持现状）。
- 本需求聚焦「报价侧（`system_type='QUOTE'`）导入数据的延迟生效 + 核价通过回填升版」，其余业务逻辑不动。

## 4. 功能详述 ⭐

### 4.1 用户场景 / 操作流程

**场景一：导入报价数据（延迟生效）**
1. 用户：报价单管理 → 从基础数据导入报价单 → 导入报价数据 → 创建报价单。
2. 导入的 7 张版本化 V6 表数据落库为 **pending**（`is_current=false` + `pending_quotation_id=本单`），主档变更暂存、占号行写 pending。
3. 本报价单在**物化时**能渲染出自己的 pending 数据；**其他报价单看不到**。

**场景二：核价通过（回填升版）**
1. 财务在核价单/财务工作台点「核价通过」前，系统弹出**回填影响预览**：将升版 N 个组、新增 X 行 / 删除 Y 行 / 改值 Z 行，逐行可展开。
2. 财务只读确认后提交，系统在**同一事务**内：把报价单有效行集回填 7 张表（走 `VersionedV6Writer` 升版）→ upsert 暂存的主档变更 → 占号行 pending→approved。
3. 此后新建报价单在「从已有产品添加」里能看到该料号，且带出的数据 = 本报价单最终态。

**场景三：从已有产品添加（闸门）**
- 「从已有产品添加」列表只列 `pending_quotation_id IS NULL`（已审核/正式）的料号；未审核报价单的料号与选配发号料号一律不出现。

### 4.2 界面与交互（前端）

> 立项初稿写「前端本次无需修改」，澄清后**需增加两处**（2026-07-21 确认）：

| 交互点 | 说明 |

### 4.3 业务规则与计算逻辑（后端）

#### 规则一：受管表清单（7 版本化 + 2 特殊）

| 类别 | 表 | 版本列 | 写入 Handler | 处理方式 |
|---|---|---|---|---|
| 版本化 | `unit_price` | version_no | Q01/Q06~Q11/Q13/Q15/Q17 | pending + 回填升版（**有历史**） |
| 版本化 | `material_bom`（主）| bom_version | MaterialBomMerge | 同上（主子同步） |
| 版本化 | `material_bom_item`（子）| bom_version | MaterialBomMerge | 同上 |
| 版本化 | `element_bom`（主）| characteristic | Q04 | 同上（主子同步） |
| 版本化 | `element_bom_item`（子）| characteristic | Q04 / Q05 | 同上 |
| 版本化 | `capacity` | calc_version | Q14 | 同上 |
| 版本化 | `plating_scheme` | scheme_version | Q16 | 同上（**全局共享表，风险最高**，见 §9） |
| 占号 | `material_customer_map` | — | Q02 / 选配发号 | **闸门**：pending→approved（**无历史**） |
| 主档 | `material_master` | — | Q18 单重 / Q02 / Q04 / Q13 / MaterialBomMerge | **暂存**，通过时覆盖 upsert（**无历史**，方案甲） |

> ⚠️ 澄清纠错：`pricing_price` **不是表**，是 `unit_price` 的一个列，不单独成表。
> ⚠️ 立项初稿把受管范围写成 5 张表，澄清中经写入侧完整排查修正为 **7 张版本化表 + 2 张特殊表**（漏了两张 BOM 主表 `material_bom`/`element_bom` 与全局共享的 `plating_scheme`）。**只 pend 子表不 pend 主表 = 主子失步**（项目 V333/V339 已踩过 `uq_material_bom_item` 撞键坑），必须主子同步。

#### 规则二：延迟生效（导入侧，方案 A）

- 7 张版本化表各加列 `pending_quotation_id uuid`（NULL=正式行/历史行；非 NULL=某报价单私有草稿）。
- 导入时 `VersionedV6Writer` **照常算版本号**（比对基准仍是旧的 `is_current=true` 组），但新行写 `is_current=false` + `pending_quotation_id=本单`，**不翻旧组 `is_current`**。升版算法本身零改动。
- `material_master`：导入时把变更**暂存**（挂 pending 记录，不 upsert 主档）。
- `material_customer_map`：加 `pending_quotation_id`，Q02/选配发号写 pending。

#### 规则三：本单可见（渲染侧，仅物化期）

- **关键前提**：报价单渲染读 `snapshot_rows`（冻结、不二次 expand）——只有「物化时刻」（建单/刷新快照）才跑 expand 打 V6。故 pending 感知改写**只需发生在物化期**的 `SqlViewExecutor`。
- 改写内容（挂 `SqlViewExecutor.applyVersionFilter` 接缝，用 `SqlViewRuntimeContext.get().quotationId` 作 pending owner）：
  1. **表替换**：把用户视图模板 FROM/JOIN 里的白名单表 token 换成等价子查询，子查询内**重定义 `is_current` 列的值**为 `(is_current OR pending_quotation_id=:pq)`。**用户写的 `is_current=true` / `asy.is_current` 谓词一字不改。**
  2. **遮蔽**：本单有 pending 行的组，屏蔽旧 `current` 行（pending 优先，防行数翻倍）。
  3. **锚点**：注入 `<别名>.id AS __v6_id`，作为回填的行身份。
- **列映射**：不靠列名约定，用 pgjdbc `PGResultSetMetaData.getBaseTableName/getBaseColumnName` 让数据库直接返回「该输出列来自哪张表的哪个物理列」。中文别名、JOIN、子查询、CTE 全部准确追踪；计算列 / UNION 返回空 → **自动判不可回写（安全降级，绝不错写）**。
- **零配置**：以上规则只读 SQL 结构 + DB 元数据，不读任何组件名 / 页签名 / 列别名。用户随意命名不影响正确性。
- **启动期硬校验**：对每个报价侧视图跑「改写 → `LIMIT 0` 执行 → 校验 `__v6_id` 存在且基表∈白名单」，任一失败**启动报错并指名具体视图**，绝不带病上线。

#### 规则四：回填升版（核价通过时，方案 丙「全量对齐」）

目标语义：**新版本基础数据 ≡ 报价单此刻的有效状态**，所以增删改**全进**。

对每个受影响的（页签 → V6 组）：
1. **重建该组「有效行集」** = pending 行 ⊕ 用户改值 ⊕ 手工叶子（新增行）⊖ 墓碑行（剪枝/删行）⊖ **spine 补空行（骨架，无数据无意图，排除）**。
2. **手工叶子**（`__manual=true`、无 `__v6_id`）：合成新 V6 行——轴列由「挂在哪个父、哪个页签类型（→characteristic）」确定（NOT NULL 分析已验证轴列全部可从组上下文推得），业务列取用户填的值，空值 NULL 或从同组代表行继承。
3. **墓碑行**：该组少这行 → 升版后新版本不含它；**老版本降 `is_current=false` 留存**（物理不删、可审计）。
4. **DAG 去重**：同一料号多次出现按 **V6 身份（父+子）** 去重，不按树 occurrence（不同父下本就是不同 `material_bom_item` 行）。
5. 有效行集当 `newRows` 交 `VersionedV6Writer` → 升版、`is_current=true`，清理残留 pending 行。
6. `material_master`：upsert 暂存的主档变更（覆盖式，无历史）。
7. `material_customer_map`：`pending_quotation_id` 置 NULL（pending→approved）。
8. 以上**全部同一事务**。

> **为什么 丙 而非「只改值」**：手工叶子是成品 BOM 真实多出来的料，墓碑是真实删掉的料，都是「树根料号的真实记录」，必须传导才能满足「下次引用一致」。且 丙 是 `VersionedV6Writer` 整组模型的**原生用法**（喂整组有效行集），比「patch pending 行」更顺、代码更少。

#### 规则五：闸门（从已有产品添加）

- `ExistingProductService.list` 的 where 增谓词：`mcm.pending_quotation_id IS NULL`（只列已审核通过的料号）。单表谓词、**无 JOIN、零 N+1**。
- 与规则四第 7 步同事务翻转，不会出现「数据生效了但产品还看不见」或反向的中间态。

#### 规则六：回填影响预览（财务通过前）

- 核价通过前 dry-run 出影响清单（升版 N 组 / 新增 X 行 / 删除 Y 行 / 改值 Z 行，逐行可展开），财务**只读确认**后才真正提交。
- **不可挑选**（方案甲：只读知情确认，不允许勾选部分回填）。理由：`VersionedV6Writer` 按整组行集比对，部分勾选会写出「从未在任何报价单里存在过的行集」，导致料号基础数据与产生它的报价单**永久对不上账**。真要部分调整走「驳回 → 销售改 → 重交」。
- `previewToken` = 影响清单内容 hash；提交时重算比对，不一致返回 **409**（防 dry-run 与提交之间数据漂移的 TOCTOU）。

#### 规则七：状态机

| 场景 | 处理 |
|---|---|
| 核价驳回 | pending 行**保留**（销售改完再交，pending 跟着走） |
| 撤回已通过 | **不回滚**（避免抽走下游已引用的新版本数据）；状态机允许撤回，但已回填数据不动 |
| 重新提交 | **覆盖**（同报价单同时只有一批 pending；重交先清旧 pending 再写新） |
| 报价单删除 | 级联删该单 pending（未生效过，无保留价值） |

## 8. 验收标准 ⭐

| # | 验收项 | 判定标准 |
|---|---|---|
| AC-1 | 延迟生效 | 报价单导入创建成功后，7 张表新行为 `is_current=false` + `pending_quotation_id=本单`；旧组 `is_current` 不变 |
| AC-2 | 本单可见 | 该报价单物化后能渲染出自己的 pending 数据（各页签非空、无「加载中…」残留） |
| AC-3 | 他单隔离 | 另建一张报价单，其物化/渲染**看不到** AC-1 的 pending 数据；行数不因 pending 翻倍 |
| AC-4 | 闸门 | 未审核报价单的料号**不出现**在「从已有产品添加」列表；选配发号料号同样不出现 |
| AC-5 | 回填-改值 | 核价通过后，改值行的新值进入基础数据新版本；下次引用带出该新值 |
| AC-6 | 回填-新增 | 手工叶子（新增行）在通过后成为基础数据新行；下次引用带出该行 |
| AC-7 | 回填-删除 | 墓碑/剪枝行在通过后从新版本消失；老版本 `is_current=false` 留存可查（未物理删） |
| AC-8 | 一致性总纲 | **通过后新建报价单引用同一产品料号，带出数据与本报价单最终态逐字段相同** |
| AC-9 | 有历史 | 7 张表通过后有新旧两版本（新 `is_current=true` / 旧 `false`）；`material_master` 覆盖无历史（预期） |
| AC-10 | 已建单不受影响 | 回填前已创建的报价单再次打开，其渲染数据与回填前逐字节一致（读 snapshot_rows 冻结） |
| AC-11 | 预览 | 通过前弹回填影响预览，逐行明细正确；`previewToken` 漂移时提交返回 409 |
| AC-12 | 预览不可挑选 | 预览为只读；不提供部分勾选回填 |
| AC-13 | 状态机 | 驳回后 pending 保留；重交覆盖旧 pending；撤回已通过不回滚；删单级联删 pending |
| AC-14 | 主子同步 | `material_bom`/`element_bom` 主表与子表同步升版，无失步、无撞 `uq_material_bom_item` |
| AC-15 | 启动校验 | 某视图改写后追不到 `__v6_id` 或基表越白名单时，启动报错并指名该视图 |
| AC-16 | 无 N+1 | 闸门/回填/预览均无逐行查询（代码审查 + SQL 计数） |
| AC-17 | 核价侧零回归 | `PRICING` 侧导入/渲染逐位不变 |

## 9. 依赖与风险

### 依赖
- `VersionedV6Writer`（升版写入器，复用）、`SqlViewExecutor` / `SqlViewRuntimeContext`（改写接缝已存在，含 `quotationId`）。
- 树任务（`task-0721-报价侧树状结构与页签类型属性`）产生的 `snapshot_rows` 树元数据结构（手工叶子 / 墓碑）。

### 风险与兜底

| # | 风险 | 应对 |
|---|---|---|
| R-1 | 表 token 文本替换健壮性（避开字符串字面量/注释/同名 CTE）| 启动期全量校验：改写→`LIMIT 0`→校验锚点+基表，失败指名视图；不带病上线 |
| R-2 | 计算列 / UNION 无法回写 | pgjdbc 元数据返回空 → 自动判不可回写（安全降级，不会错写） |
| R-3 | 替换体把 `unit_price` 展开成 ~49 列，且 expand 是首存热路径 | 实测首存耗时 A/B 对比；超标加索引 / 物化 |

## 11. 与树任务的接触面（强关联，必读）

树任务把 `snapshot_rows` 扩展为 BOM 树结构（含手工叶子、spine 空行、跨页签级联墓碑）。本任务回填读的正是 `snapshot_rows`。协同约定：

| 接触点 | 结论 |
|---|---|
| 回填读取范围 | 只读 `__v6_id` + 改值 + 手工叶子标记 + 墓碑；树结构字段（`__nodeId`/`__parentId`/`__lvl` 等）**无视** |
| 手工叶子回写 | 树任务范围内**不回写 V6**（只落快照）；**由本任务在核价通过时回写**——生命周期分工，非矛盾 |
| spine 空行 | 回填时**排除**（骨架非数据行） |
| 树页签 vs 平铺页签 | 回填统一按 `__v6_id`，**不分叉**（递归 SQL 那条路径只需保证把 `__v6_id` 注进 driverRow） |
| DAG 重复子件 | 按 V6 身份去重 |

> **⚠️ 跨文档一致性**：树任务 `需求说明.md` §4 边界写「新增行不回写 `material_bom_item` —— 只进报价单快照」。该句在**树任务范围内**成立，但须加注解防止工程师误解为「手工叶子永不进基础数据」。已在树任务文档同步加注：**「由 task-0721-报价升版逻辑 在核价通过时统一回填」**。
