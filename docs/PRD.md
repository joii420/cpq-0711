## 一、项目背景与目标

### 1.1 项目背景

在制造业和工业零部件行业，产品报价流程复杂，涉及多工序选型、多级定价策略、客户分层管理等环节。传统的人工报价方式存在以下痛点：

- **效率低下**: 销售人员需要手动查询产品工艺参数、材料成本、加工费用等信息，制作一份报价单耗时较长
- **定价混乱**: 不同客户的折扣策略、批量折扣等规则依赖人工记忆，容易出错
- **版本失控**: 报价模板缺乏版本管理，修改历史难以追溯
- **协作困难**: 销售、工艺、财务之间的信息传递依赖线下沟通，导致报价数据不一致
- **客户管理分散**: 客户等级、信用额度、历史交易等数据散落在不同系统中
- **审批缺失**: 报价单缺乏规范化审批流程，定价合规性难以保证

### 1.2 项目目标

构建一套面向制造业的 **CPQ 报价系统**，实现：

| 目标 | 描述 | 衡量指标 |
|------|------|----------|
| **提升报价效率** | 通过模板化、自动化报价流程减少人工操作 | 报价制作时间缩短 70% |
| **统一定价管理** | 客户分层定价策略在线配置，系统自动应用 | 定价准确率 > 99% |
| **规范模板管理** | 产品卡片模板版本化、可视化配置 | 模板复用率 > 80% |
| **强化客户管理** | 客户全生命周期管理，分层运营 | 客户数据完整率 > 95% |

### 1.3 目标用户

| 角色 | 职责 | 核心场景 |
|------|------|----------|
| **销售代表** | 创建报价单、管理客户关系 | 快速生成报价、跟进客户 |
| **销售经理** | 管理报价审批、配置模板与工序、监督销售团队报价流程 | 模板配置、报价审批、团队管理 |

#### 2.1.2 功能需求

**客户列表页**
- 客户分类标签筛选：全部 / 活跃 / VIP / 潜在 / 不活跃
- 关键词搜索（客户名称、编码、联系人模糊匹配）
- 分页展示（每页 10/20/50 条可选）
- 批量勾选与批量删除操作
- 客户卡片展示：等级徽章（钻石/VIP/金牌/银牌/标准）

**客户详情/编辑（抽屉弹出）**
- 基本信息：客户名称（必填）、客户编码（自动生成）、客户等级、所属行业、所属区域
- 联系人管理：
  - 联系人列表展示，支持添加/编辑/删除联系人
  - 每个联系人包含：姓名（必填）、职责（采购/技术/财务等）、手机号（必填，格式校验）、邮箱、微信
  - 主要联系人标记（每个客户至少一个，用于报价单默认填充）
- 商务信息：信用额度（万元）、付款方式、地址、备注
- 统计面板（只读）：历史订单数、累计金额（自动计算，只读）、平均折扣率、信用额度

> **累计金额（自动计算，只读）**：系统自动累加该客户名下所有状态为 ACCEPTED 的报价单的 total_amount（折后总价）。每当一张报价单状态变为 ACCEPTED 时，系统通过原子 SQL 累加更新客户的 accumulated_amount 字段，与 ACCEPTED 标记操作在同一事务中执行，并发场景下数据正确性由数据库原子性保障。此字段不可手动修改。

**客户删除保护**
- 客户执行软删除（状态变为 INACTIVE）前，系统检查该客户是否存在状态为 DRAFT / SUBMITTED / APPROVED 的报价单
- 若存在，阻止删除并提示："该客户有进行中的报价单（X张），无法删除。请先处理相关报价单后再操作。"
- 已快照的 SENT / ACCEPTED / REJECTED / EXPIRED 报价单不受客户删除影响，可正常查看历史数据

**数据模型**

```
Customer {
  id: UUID (PK)
  name: String (NOT NULL, 客户名称)
  code: String (UNIQUE, 自动生成)
  level: Enum [DIAMOND, VIP, GOLD, SILVER, STANDARD]
  industry: String (所属行业)
  region: String (所属区域)
  address: String (地址)
  accumulated_amount: Decimal (累计成交金额，系统自动计算，等于所有ACCEPTED报价单折后总价之和)
  credit_limit: Decimal (信用额度)
  payment_method: String (付款方式)
  remarks: Text (备注)
  status: Enum [ACTIVE, INACTIVE]
  created_at: Timestamp
  updated_at: Timestamp
}

CustomerContact {
  id: UUID (PK)
  customer_id: UUID (FK → Customer)
  name: String (NOT NULL, 联系人姓名)
  role: String (职责，如：采购/技术/财务)
  phone: String (NOT NULL, 手机号，格式校验)
  email: String
  wechat: String
  is_primary: Boolean (是否主要联系人，每个客户至少一个)
  created_at: Timestamp
}
```

#### 2.1.3 非功能需求

- 客户列表加载时间 < 500ms（1000 条数据内）
- 搜索响应时间 < 300ms
- 支持并发操作，乐观锁防止数据覆盖
- 客户删除需软删除，保留历史引用完整性
- 手机号格式校验（中国大陆11位手机号）
- 每个客户至少保留一个主要联系人，不可全部删除
- 客户删除前检查关联进行中报价单（DRAFT/SUBMITTED/APPROVED），有则阻止

- 右侧：选中客户的策略配置区域

**策略配置**
- 策略名称
- 策略类型（批量折扣）
- 生效日期 / 到期日期（可选"长期有效"）
- 基础折扣率（百分比输入）
- 折扣起始金额门槛

- 同一客户的策略列表按优先级排序展示
- 当客户有多条 ACTIVE 策略时，列表顶部提示"当前有X条生效策略，按优先级依次匹配"

**定价规则**
- 批量折扣规则：订单金额阈值 → 折扣百分比
- 规则支持编辑和删除

**多策略匹配规则**（当客户存在多个 ACTIVE 状态的定价策略时）

> 1. 按 priority 字段升序排列所有 ACTIVE 策略
> 2. 依次检查每个策略的 min_order_amount 门槛，取第一个满足条件的策略进入折扣规则匹配
> 3. 若所有策略均不满足 min_order_amount 门槛，折扣率为 100%（不打折，等同无策略兜底规则）
> 4. 当前版本建议每个客户仅配置一条策略（priority=1），多策略为预留能力

**折扣规则匹配逻辑**（对上述匹配到的单条策略执行以下规则）

> 1. **批量折扣规则**（优先）：系统检查所有 BULK_DISCOUNT 规则，找出所有满足 threshold_amount ≤ 整单原始总价 的规则，从中选取 discount_rate 最低的（即折扣最优惠的）一条生效
> 2. **基础折扣率**（兜底）：若无任何批量折扣规则触发（原始总价低于所有阈值，或客户无批量折扣规则），则使用 base_discount 作为折扣率
> 3. **min_order_amount 门槛**：若整单原始总价低于 PricingStrategy.min_order_amount，则不进行任何折扣计算，折扣率为 100%（不打折）
> 4. 两者取其一，不叠加

**导入/导出**
- 支持导出当前客户策略为 Excel
- 支持从 Excel 模板导入策略数据

**数据模型**

```
PricingStrategy {
  id: UUID (PK)
  customer_id: UUID (FK → Customer)
  name: String (策略名称)
  type: Enum [DISCOUNT]
  base_discount: Decimal (基础折扣率, 0-100)
  min_order_amount: Decimal (折扣起始金额)
  effective_date: Date
  expiration_date: Date (NULL = 长期有效)
  priority: Integer (策略优先级，数值越小优先级越高，默认为1；同一客户多个ACTIVE策略时按priority升序匹配)
  status: Enum [ACTIVE, EXPIRED, DISABLED]
  created_at: Timestamp
  updated_at: Timestamp
}

PricingRule {
  id: UUID (PK)
  strategy_id: UUID (FK → PricingStrategy)
  rule_type: Enum [BULK_DISCOUNT]
  threshold_amount: Decimal (触发阈值)
  discount_rate: Decimal (应付比例, 0-100；如85表示8.5折)
  sort_order: Integer (仅控制列表展示顺序，不影响匹配逻辑；匹配逻辑始终取满足条件的最低discount_rate)
  created_at: Timestamp
}
```

> **折扣率字段统一规范**：全系统所有折扣率字段（base_discount、discount_rate、system_discount_rate、final_discount_rate）均以 0-100 的数值存储，语义为"应付比例"（如9折存储为90，表示支付原价的90%；不打折存储为100，表示支付原价的100%）。**统一计算公式：折后总价 = 原始总价 × 折扣率 / 100**。前端展示时直接显示数值并附"%"符号（如"折扣率：90%"即9折）。特别说明：无策略兜底时折扣率 = 100（不打折），而非 0。

#### 2.3.3 非功能需求

- 策略规则计算在报价生成时实时完成，延迟 < 200ms
- 策略过期由系统定时任务每日检查并自动标记
- Excel 导入支持最大 1000 条规则/次
- 策略变更需记录操作日志（审计追溯）
- 折扣规则由系统自动判定，不可人工干预
- 多条批量折扣规则并发匹配时，取最优惠规则（discount_rate 最小值），计算结果须与前端展示一致，误差为 0
- 同一客户多策略并存时，系统按优先级顺序计算，总耗时仍满足 < 200ms

> **折扣计算引擎实现说明（Drools）**：
> - 折扣计算由 Drools 规则引擎执行，规则数据来源于 PricingStrategy 和 PricingRule 表
> - 每次进入步骤三时，系统从数据库加载该客户所有 ACTIVE 状态的定价策略及其规则，动态构建 KieSession，输入整单原始总价，执行规则匹配，输出最终折扣率
> - 规则匹配优先级通过 Drools 的 salience 属性实现，映射规则为 salience = -priority（Drools salience 数值越大优先级越高，priority 数值越小优先级越高，两者方向相反，因此取负值映射：priority=1 → salience=-1 最高，priority=10 → salience=-10 最低）
> - 策略过期、新增、修改后，下次报价时自动加载最新规则，无需重新部署服务
> - 兜底规则（无策略时折扣率为 100%，即不打折）作为默认 Drools 规则内置，不依赖数据库

- "创建新草稿"按钮：基于当前最新 PUBLISHED 版本复制一份新 DRAFT 记录，用于下一次迭代配置

**模板发布**
- 模板右侧面板新增"发布"按钮（仅 DRAFT 状态可用）
- 发布前校验：subtotal_formula 不可为空（必须配置产品小计公式），未配置时阻止发布并提示"请先配置产品小计公式"；至少关联一个页签组件
- 点击发布 → 系统自动执行：①读取所有 TemplateComponent 关联的组件最新定义 ②写入 components_snapshot ③status 变为 PUBLISHED ④published_at 记录发布时间 ⑤版本号自动递增
- 已发布模板不可直接编辑；如需修改，需先"创建新草稿"（基于当前版本复制一份新 DRAFT）
- PUBLISHED 模板可被归档（status → ARCHIVED），归档后不可被新报价单选用，但历史报价单不受影响

**模板归档检查**
- 点击"归档"按钮时，系统检查是否有产品+工序组合的 ProductTemplateBinding 中**仅**绑定了该模板版本（即该产品的该工序组合下无其他 PUBLISHED 模板可用）
- 若存在此类绑定，弹出警告："归档后，以下 X 个产品的部分工序组合将无可用报价模板，销售代表将无法对这些组合发起报价。建议先为其绑定新版本模板。"，并列出产品名称及对应工序组合
- 系统管理员/销售经理可选择"强制归档"或"取消归档"；选择强制归档时记录操作日志
- 已被正在进行中报价单（DRAFT/SUBMITTED/APPROVED）使用的模板版本，不可归档（阻止操作并提示"该模板版本正在被X张进行中报价单使用，无法归档"）

**版本号自动递增规则**
- 版本号格式：v{大版本}.{小版本}（如 v1.0、v1.1、v2.0）
- 每次发布（DRAFT → PUBLISHED）自动递增小版本号（v1.0 → v1.1 → v1.2）
- 大版本号由销售经理在发布时手动指定（选填，不填则仅递增小版本）；通常在重大改版时手动升级大版本（如 v1.x → v2.0）
- 同一模板名下按版本号排序，模板版本比对（模块六）的下拉列表展示格式：模板名 - v{版本号}（{发布日期}）
- 第一次发布默认版本号为 v1.0

**模板版本存储方式**
- 每次发布（DRAFT → PUBLISHED）在 Template 表中新建一条独立记录，而非更新原有记录
- 同一"模板系列"（相同名称的模板）会有多条 Template 记录，每条代表一个独立版本
- 草稿（DRAFT）也是一条独立记录，发布后该草稿记录状态变为 PUBLISHED，不再新建
- QuotationLineItem.template_id 直接指向某个具体版本的 Template 记录，无需额外存储版本号字段
- 模板系列通过 template_series_id 关联，版本比对模块通过筛选同一 template_series_id 列出所有历史版本

**产品小计公式格式**

Template.subtotal_formula 采用与组件公式相同的表达式数组格式，支持两种引用类型：

- `component_subtotal`：引用页签组件的小计值
- `product_attribute`：引用 NUMBER 类型的产品属性值（如数量）

```json
[
  {"type": "bracket_open"},
  {"type": "component_subtotal", "component_code": "touLiao", "tab_name": "投料金额"},
  {"type": "operator", "value": "+"},
  {"type": "component_subtotal", "component_code": "huiLiao", "tab_name": "回料金额"},
  {"type": "bracket_close"},
  {"type": "operator", "value": "×"},
  {"type": "product_attribute", "attribute_name": "数量"}
]
```

上例含义：产品小计 = (投料金额小计 + 回料金额小计) × 数量。

报价生成器实时解析该表达式，将各页签组件的小计值和产品属性值代入计算产品总价。

> **产品属性引用约束**：仅 `field_type=NUMBER` 的产品属性可被拖入小计公式。模板配置 UI 的小计公式拖拽区左侧面板新增"产品属性"分组，仅展示当前模板中 NUMBER 类型的属性字段。TEXT 类型属性不出现在可拖拽列表中。

**数据模型**

```
Template {
  id: UUID (PK)
  template_series_id: UUID (NOT NULL, 模板系列标识；同一系列所有版本共享相同ID；创建新系列时自动生成UUID，"创建新草稿"时从源版本继承)
  name: String (NOT NULL, 模板名称，可修改，不影响版本关联)
  version: String (该记录的版本号，同一template_series_id下唯一，格式 v{大版本}.{小版本}，首次发布为v1.0)
  category: Enum [STANDARD_PARTS, CUSTOM_PARTS, RAW_MATERIALS]
  description: Text
  usage_note: Text (使用说明)
  product_attributes: JSONB (产品属性结构化定义列表, 每项含:
    name: String 属性名称
    field_type: Enum [TEXT, NUMBER] TEXT为文本，NUMBER为数字（可参与公式）
    required: Boolean 是否必填
    default_value: String 默认值（可选）
    sort_order: Integer
    示例: [
      {"name":"产品名称","field_type":"TEXT","required":true},
      {"name":"规格型号","field_type":"TEXT","required":false},
      {"name":"数量","field_type":"NUMBER","required":true,"default_value":"1"},
      {"name":"单位","field_type":"TEXT","required":false},
      {"name":"交货天数","field_type":"NUMBER","required":false}
    ]
  )
  subtotal_formula: JSONB (产品小计公式, 表达式数组格式, 可引用各页签组件小计和NUMBER类型产品属性)
  components_snapshot: JSONB (发布时快照的组件结构，含所有关联组件的字段和公式定义)
  status: Enum [DRAFT, PUBLISHED, ARCHIVED]
  created_by: UUID (FK → User)
  published_at: Timestamp (发布时间，DRAFT状态为NULL)
  created_at: Timestamp
  updated_at: Timestamp
}

TemplateComponent {
  id: UUID (PK)
  template_id: UUID (FK → Template)
  component_id: UUID (FK → Component, 引用组件管理中的组件)
  tab_name: String (页签显示名称, 默认使用组件名称)
  sort_order: Integer (页签排列顺序)
  created_at: Timestamp
}
```

> **说明**: 产品卡片模板不再独立定义标签页的列和公式（原 TemplateTab/TemplateColumn/TemplateFormula），而是通过 `TemplateComponent` 关联表引用组件管理中的组件定义。页签的列结构和公式由组件统一管理，模板只负责组装和排序。

> **components_snapshot 结构示例**：发布时将所有关联组件的完整定义快照为 JSON 数组，每个元素对应一个页签组件：
> ```json
> [
>   {
>     "component_id": "uuid-comp-1",
>     "component_code": "touLiao",
>     "tab_name": "投料金额",
>     "sort_order": 1,
>     "fields": [
>       {"name":"材料编码","field_type":"FIXED_VALUE","content":"M001","is_amount":false,"is_subtotal":false,"sort_order":1},
>       {"name":"材料单价","field_type":"DATA_SOURCE","datasource_binding":{"datasource_id":"uuid-ds-1","datasource_code":"material_price","datasource_name":"材料单价查询","param_bindings":[{"param_code":"material_code","param_name":"材料编码","bound_field_name":"材料编码"}]},"is_amount":true,"is_subtotal":false,"sort_order":2},
>       {"name":"使用数量","field_type":"INPUT","is_amount":false,"is_subtotal":false,"sort_order":3},
>       {"name":"小计","field_type":"FORMULA","is_amount":true,"is_subtotal":true,"sort_order":4}
>     ],
>     "formulas": [
>       {"name":"小计","expression":[{"type":"field","value":"材料单价"},{"type":"operator","value":"×"},{"type":"field","value":"使用数量"}],"result_type":"AMOUNT"}
>     ]
>   }
> ]
> ```
> 快照不包含 directory_id、status、created_at 等组件管理元数据，仅保留报价时所需的结构和计算定义。

> **说明**：Template 表中同一 template_series_id 可存在多条记录（DRAFT/PUBLISHED/ARCHIVED 各版本），通过 template_series_id + version 唯一确定一个模板版本，template_id 是数据库主键，代表该特定版本。模板名称（name）可自由修改，不影响版本关联。

#### 2.5.3 非功能需求

- 拖拽操作流畅，FPS > 30
- 模板保存支持自动保存（每 30 秒）
- 拖入组件后实时渲染对应列结构和公式预览，延迟 < 200ms
- 模板发布时自动快照所有关联组件的字段和公式定义到 components_snapshot，快照完成后状态变为 PUBLISHED
- 已发布模板的 components_snapshot 不可修改，报价单引用已发布模板时使用快照数据
- 组件变更后，引用该组件的**草稿状态**模板在打开时提示"该组件已更新，发布时将使用最新版本"；已发布模板不受影响

#### 2.7.2 功能需求

**产品库侧边栏（左侧）**
- 产品搜索
- 品类筛选：标准件 / 定制件
- 产品列表：状态徽章 + 工艺标签（表面处理、热处理、CNC加工等）
- 示例产品：精密轴承座、联轴器、法兰盘、特种齿轮

**配置区域（右侧，三标签页）**
- 工序选择标签页：分类网格展示工序卡片，复选框选择（配置该产品可选的工序范围）
- 工序组合与模板绑定标签页：
  - 顶部操作区："新增绑定"按钮
  - 点击后弹窗流程：
    1. **选择工序组合**：从该产品已关联的工序中多选，确定一个工序组合（可不选任何工序，表示"无工序"组合）
    2. **选择模板**：为该工序组合选择一个报价模板（从 PUBLISHED 状态的模板中选择）
    3. 确认后创建绑定记录
  - 绑定列表展示：工序组合（展示工序名称列表，空则显示"无工序"）、模板名称及版本号、是否默认、操作按钮
  - 同一产品下可以有多个工序组合，每个组合可绑定多个模板
- 模板版本查看标签页：展示该产品所有绑定的模板版本概览

**工序组合匹配规则**
- 报价时第3步模板列表通过**精确匹配**过滤：将销售代表在第2步选择的工序ID集合排序后与 ProductTemplateBinding.process_ids 精确比对，仅展示完全一致的绑定记录对应的模板
- 第2步未选择任何工序（跳过或该产品无工序）时，匹配 process_ids 为空数组 `[]` 的绑定记录
- 若精确匹配无结果，第3步显示"当前工序组合暂未配置报价模板，请联系销售经理配置"并禁用确认按钮

**模板版本可见性规则**
- 产品绑定模板时，列表中仅显示 status=PUBLISHED 的模板版本（归档版本不显示）
- 绑定的是具体版本（template_id），销售代表报价时看到的模板名称格式为"模板名 - v{版本号}"
- 若产品已绑定的某个模板版本被归档（ARCHIVED），该绑定关系保留但在报价弹窗中标记为不可用

**默认模板规则**
- 每个产品+工序组合最多设置一个默认模板（is_default=true），数据库层面使用 **PostgreSQL 部分唯一索引**（`CREATE UNIQUE INDEX ON product_template_binding (product_id, process_ids_hash) WHERE is_default = true`），确保同一产品+工序组合下最多一个默认模板
- 报价时三步弹窗第3步，匹配到的绑定记录中默认模板自动预选高亮，销售代表可切换选择其他已绑定模板
- 若该工序组合未设置默认模板，则第3步不预选，销售代表必须手动选择一个
- 在工序组合与模板绑定列表中，每个已绑定模板行有"设为默认"按钮，点击后切换默认模板

**数据模型**

```
ProductTemplateBinding {
  id: UUID (PK)
  product_id: UUID (FK → Product)
  process_ids: JSONB (工序ID有序列表，排序后存储，用于精确匹配；空数组[]表示"无工序"组合；示例：["uuid-1","uuid-2"] 或 [])
  process_ids_hash: String (process_ids排序后的SHA-256哈希，用于唯一约束和快速匹配；空数组哈希为固定值)
  template_id: UUID (FK → Template)
  is_default: Boolean
  created_at: Timestamp
}
```

> **唯一约束**：(product_id, process_ids_hash, template_id) 确保同一产品+同一工序组合下不会重复绑定同一模板。process_ids_hash 由后端在创建/更新时自动计算，前端无需感知。

> **匹配性能**：报价时第3步模板过滤通过 `WHERE product_id = ? AND process_ids_hash = ?` 查询，哈希字段建立索引，匹配效率 O(1)。

#### 2.7.3 非功能需求

- 搜索支持拼音首字母匹配
- 绑定关系变更实时生效
- process_ids_hash 字段建立索引，支持高效精确匹配

#### 2.8.1 用户与场景

|----------|----------|----------|----------|----------|
| G-01 | 销售经理 | 创建组件目录：建立组件分类树（如"生产管理"、"质检管理"） | 已登录系统 | 目录创建成功 |

| G-04 | 销售经理 | 配置公式：拖拽"材料单价"和"使用数量"字段构建 `材料单价 × 使用数量 + 回料·小计` 公式 | 字段已配置 | 公式保存成功，可引用其他组件的小计 |

- 画布标题与面包屑导航（如：生产管理 › 投料组件）

- 表头预览表格：实时展示组件列头效果

**DATA_SOURCE 字段配置弹窗（两步流程）**

当字段类型选择"数据源"时，"内容/来源"列变为"配置数据源"链接，点击后弹出两步配置弹窗：

*步骤一：选择数据源*
- 展示状态为 ACTIVE 的数据源列表（编号、名称、类型、描述），支持搜索
- 选中后展示该数据源的完整参数列表：参数顺序、参数名称、来源类型、参数说明
- 来源类型为"系统参数"的参数显示系统参数编码，标注"系统自动注入"，无需操作
- 来源类型为"用户字段引用"的参数在步骤二绑定

*步骤二：绑定用户字段参数*
- 仅展示来源类型为"用户字段引用"的参数行
- 每个参数提供下拉选择框，选项为当前组件内所有非 FORMULA 类型的字段（FIXED_VALUE / INPUT / 其他 DATA_SOURCE 字段）
- 所有参数必须完成绑定，有未绑定项时禁用"确认"按钮，提示"请为所有用户字段参数选择绑定字段"
- 确认后"内容/来源"列显示数据源名称，悬浮 tooltip 展示绑定详情

**组件配置 - 公式管理区域（中间下部）**
- 运算符工具栏：+、-、×、÷、(、)、%（可拖拽）
- 公式配置表格：
  - 公式名称（如：成本合计）
  - 公式表达式区域（拖拽式构建）：
    - 从右侧拖入本组件字段芯片（蓝色）
    - 从工具栏拖入运算符芯片（绿色）
    - 可引用其他组件的小计值（橙色，如"回料·小计"）
  - 结果类型选择：金额 / 数量 / 文本
  - 芯片可单个移除
- 新增公式 / 删除公式

**字段与小计面板（右侧）**
- 字段列表标签页：展示当前组件所有字段，可拖拽到公式区域
- 其他数据源标签页：展示其他组件的小计字段，可跨组件引用

```

  name: String (NOT NULL, 组件名称)

        "datasource_id": "UUID",
        "datasource_code": "String",    冗余，便于快照后追溯
        "datasource_name": "String",    冗余，便于展示

        ]

    is_subtotal: Boolean 是否小计字段
    note: String 说明
    sort_order: Integer
  )

    name: String 公式名称（必须与某个 FORMULA 类型字段的 name 完全一致）
    expression: Array 公式表达式(字段引用+运算符有序列表)
    result_type: Enum [AMOUNT, QUANTITY, TEXT]
  )
  status: Enum [ACTIVE, DISABLED]
  created_at: Timestamp
  updated_at: Timestamp

  sort_order: Integer
  created_at: Timestamp

#### 2.8.3 非功能需求

- 拖拽公式构建交互流畅，FPS > 30
- 组件变更后，引用该组件的**草稿状态**模板在打开时提示"该组件已更新，发布时将使用最新版本"；已发布模板不受影响，提示仅针对 DRAFT 状态模板
- 公式语法校验，保存前检测循环引用和语法错误
- 组件删除前检查是否被产品卡片引用，若有引用则阻止删除或提示确认
- 每个组件仅允许一个小计字段，用于被其他组件公式引用

> **公式计算引擎实现说明（JEXL）**：
> - JEXL 引擎负责两个层级的公式计算：
>   1. **组件行级公式**：组件内的公式字段（FORMULA 类型）。执行时从 Template.components_snapshot 中读取对应组件的 expression 数组，将其转换为 JEXL 表达式字符串，注入当前行的 INPUT 字段值和跨组件 subtotal 值，执行求值
>   2. **产品小计公式**：Template.subtotal_formula。执行时解析表达式数组，将各页签组件的小计值（component_subtotal 类型）和 NUMBER 类型产品属性值（product_attribute 类型，如数量）代入计算产品总价
> - **跨组件引用默认值**：当公式引用其他组件的小计值，但目标组件尚未填写数据（小计为 null 或无行数据）时，引用值默认为 **0**，公式正常计算不报错。前端在目标组件小计值变更后自动触发引用方公式重算
> - 循环引用检测：保存组件公式前，系统对 expression 中的跨组件引用执行有向图遍历，检测是否存在环路，有则阻止保存并提示"公式存在循环引用，请修改后重试"
> - JEXL 引擎不用于折扣计算和审批路由（那两个场景由 Drools 负责）

**步骤二：产品配置**
- 添加产品按钮 → 三步弹窗向导：
  1. **产品选择**（按品类：标准件/定制件/原材料）
  2. **工序选择**（多选复选框，从产品已关联的工序中选择）
  3. **产品卡片模板选择**（根据第1步选择的产品和第2步选择的工序组合，精确匹配 ProductTemplateBinding 记录，展示该工序组合下已绑定的模板列表；模板决定报价明细结构）
- 产品卡片展示（每个产品一个标签页，结构由选择的产品卡片模板决定）：
  - 基本信息区：由模板的 product_attributes 结构化定义动态渲染（如产品名称/规格型号/数量/单位/交货天数等），不同模板的属性字段可能不同；NUMBER 类型属性值变更后触发产品小计公式重算
  - **页签组件区**：根据模板中拖入的页签组件动态渲染
    - 每个页签组件展示为一个标签页（如：投料金额、回料金额、加工费用、税费等）
    - 页签内的列结构、字段类型、公式均继承自组件管理中的组件定义
    - 输入框类型字段可编辑录入数据
    - 数据源类型字段在报价时的交互行为：
      - 初始状态：显示占位符"—"，等待触发查询
      - 触发条件：当同一行中任意被绑定为参数来源的 INPUT 字段失焦且值发生变化后，经 300ms 防抖延迟，系统检查该行所有 is_required=true 的参数是否均有值：全部有值 → 调用后端执行数据源查询，单元格显示 loading，返回结果后填入，变为只读展示；存在必填参数为空 → 不执行查询，单元格保持"—"
      - 查询失败（超时/SQL报错/API异常）：单元格显示"查询失败"（红色字体），tooltip 展示错误信息；不阻断其他字段填写
      - 参数值被修改后（用户修改了被绑定的 INPUT 字段并失焦）：重新触发查询，覆盖上次结果
      - **同参数查询缓存**：同一报价单编辑会话中，相同数据源 + 相同参数值组合的查询结果缓存 5 分钟（前端内存缓存，key = datasource_id + 参数值 JSON hash），避免多行相同参数重复请求后端；参数变化时缓存自动失效
    - 数据源字段配置不完整警示：数据源字段处于占位符"—"状态且 is_required=true 时，该单元格显示红色边框，tooltip 提示"数据源未查询，请先完成关联字段的录入"；步骤五提交前校验新增一项：所有数据源字段已完成查询（值非占位符状态），否则高亮提示并阻止提交
    - 公式类型字段自动计算并展示结果
    - 每个页签组件有各自的小计
  - **产品小计**：自动汇总所有页签组件的小计（¥ 格式）

> **页签行操作规则**：
> - 模板只固定列结构（字段名、字段类型、公式），不固定行数
> - 报价时每个页签组件默认显示 1 行空白输入行
> - 销售代表可自由添加行（点击"+ 添加行"按钮）和删除行（每行右侧删除图标），最少保留 1 行
> - 每行数据独立存储，公式字段在每行内自动计算（如每行的小计 = 单价 × 数量）
> - 页签组件的小计 = 该页签所有行的小计列之和
> - 行数无上限限制，但超过 20 行时 UI 提示"行数较多，可能影响 PDF 导出排版"

- 标签页可展开/折叠
- 模板切换功能：

> **模板切换规则**：
> - 销售代表在产品卡片上点击"切换模板"时，系统弹出确认弹窗："切换模板将清空当前产品已填写的所有页签数据，此操作不可撤销。是否继续？"
> - 点击"取消"：关闭弹窗，保留当前模板和数据不变
> - 点击"确认切换"：清空当前产品所有 QuotationLineComponentData 行数据，加载新模板的页签结构，重置为 1 行空白输入行

- 产品添加/删除

> **异常处理**：
> - 若选中产品+选择的工序组合在 ProductTemplateBinding 中无精确匹配记录，第3步显示"当前工序组合暂未配置报价模板，请联系销售经理配置"提示，并禁用"确认"按钮
> - 若选中产品未绑定任何报价模板（所有工序组合均无绑定），第3步显示"该产品暂未配置报价模板"提示，并禁用"确认"按钮，引导用户重新选择其他产品或联系销售经理配置

> **报价时模板可用性规则**：
> - 三步弹窗第3步先按产品+工序组合精确匹配 ProductTemplateBinding 记录，再按模板状态过滤展示

> - ARCHIVED 状态的模板仍在列表中显示，但标记为"已归档（不可用）"，禁止选择
> - DRAFT 状态的模板不显示在列表中（草稿未发布，不可用于报价）
> - 若该工序组合下所有绑定的模板均已归档，系统提示"当前工序组合的所有报价模板已归档，请联系销售经理配置新版本"

**步骤三：价格策略应用**

> **步骤三原始总价来源（方案A：前端状态读取）**：
> - 原始总价直接读取前端状态管理中的实时产品小计之和，即步骤二页面上各产品卡片当前展示的小计值，无需额外 API 调用
> - 该值由前端 JS 公式引擎实时计算维护，始终反映用户当前录入的最新状态，不依赖草稿保存的时机
> - 进入步骤三时，系统以前端状态中的原始总价调用 Drools 折扣引擎计算折扣率
> - 若前端状态中任何产品小计为 0 或未完成计算，系统阻止进入步骤三，提示"存在产品小计未填写，请返回步骤二完成录入"
> - 手动调整的折扣率在以下情况自动清空并重新计算：用户从步骤三返回步骤二修改数据后再次进入步骤三，检测到原始总价发生变化时，原有手动折扣率清空，提示"产品数据已变更，折扣已重新计算，如需手动调整请重新操作"

> **无策略兜底规则**：若客户未关联任何定价策略，或所有关联策略均已过期/停用，步骤三显示"该客户暂无生效的定价策略，折扣率默认为 100%（不打折，即按原价）"。销售代表可在此步骤手动输入折扣率，手动输入同样需要填写调整原因（必填），并写入操作日志。

- 系统自动读取客户关联的定价策略，按优先级（批量折扣 > 基础折扣率）计算折扣
- 展示折扣计算明细：
  - 原始总价
  - 适用的折扣规则名称及折扣率
  - 折后总价
- 销售代表可手动调整最终折扣：
  - 输入自定义折扣率（百分比），覆盖系统自动计算结果
  - 手动调整需填写调整原因（必填）
  - 调整后显示"手动调整"标记，原系统计算结果保留作对比展示
- 折扣调整记录写入操作日志（审计追溯）

> **折扣作用层级说明**：折扣作用于整张报价单，而非单个产品行。系统按以下方式计算：
> 1. 汇总所有产品行小计，得到报价单原始总价
> 2. 根据客户定价策略按优先级匹配适用折扣规则，计算整单折扣率
> 3. 折后总价 = 原始总价 × 最终折扣率 / 100
> 4. 各产品行的 final_discount_rate 字段记录本次整单折扣率，用于历史追溯

**步骤四：商务条款配置**
- 付款方式（文本输入或下拉选择，如：预付30%/货到付款/月结30天）
- 交货周期（数字输入，单位：天）
- 说明：其他商务条款（质保期、违约条款等）为后续版本迭代内容，当前版本不包含

**步骤五：提交审批**
- 报价单完整预览（只读）：
  - 客户信息摘要
  - 产品明细汇总（产品属性值摘要/小计）
  - 折扣信息
  - 商务条款
  - 报价总金额（大字醒目展示）
- 提交前校验：
  - 所有必填字段已填写（含模板 product_attributes 中 required=true 的动态产品属性、各页签 INPUT 类型必填字段）
  - 至少包含一个产品
  - 产品小计均大于0
  - 校验失败时高亮提示具体问题项，阻止提交
- "提交审批"按钮：点击后触发快照机制（见下方说明），状态变为SUBMITTED，系统通知审批人（销售经理）
- 提交成功后跳转至报价单详情页，展示当前审批状态

> **提交时快照机制**：
> - 点击"提交审批"后，系统在状态变为 SUBMITTED 之前，自动将以下数据写入快照：
>   1. 当前客户的名称、等级、区域、行业、地址 → 写入 Quotation 快照字段
>   2. 每个产品行的 Product 表静态字段（SKU、品类、规格型号）→ 写入 QuotationLineItemSnapshot 表（注意：用户填写的产品属性值如"产品名称"已存储在 QuotationLineItem.product_attribute_values JSONB 中，天然具备快照性质，无需额外写入快照表）
> - 快照完成后状态变为 SUBMITTED，通知审批人
> - 快照后，即使客户被删除/改名、产品被停用/修改，该报价单的展示内容保持提交时的状态不变
> - DRAFT 阶段不做快照，实时显示最新客户和产品数据
>
> **提交操作原子事务保障**：
> - 上述步骤①②③④及状态变更须在同一个数据库事务中执行，执行顺序严格为：
>   1. 写入客户快照字段（snapshot_customer_* 系列字段）
>   2. 写入产品快照（QuotationLineItemSnapshot 表）
>   3. 执行 Drools 审批路由，将路由结果写入 Quotation.assigned_approver_id
>   4. 状态变更为 SUBMITTED
> - 任一步骤执行失败，整体事务回滚，报价单保持 DRAFT 状态不变，前端提示"提交失败，请稍后重试"
> - 审批通知（邮件 / 站内消息）在事务提交成功后异步发送，不纳入事务范围；通知发送失败不影响提交结果，系统记录失败日志供运维处理

> - FORMULA 类型字段（公式字段）不存储计算结果，展示时根据 Template.components_snapshot 中的公式定义实时计算
> - DATA_SOURCE 类型字段：存储查询结果的值快照（数值或字符串，与数据源返回值类型一致）。报价单提交时，将所有 DATA_SOURCE 字段的当前查询结果值随 row_data 一并固化为快照，后续数据源 SQL/API 结果变化不影响历史报价单展示。未完成查询的 DATA_SOURCE 字段在 row_data 中为 null，提交前校验会阻止此类情况通过
> - 格式示例：
> ```json
> [
>   {"row_index": 0, "材料编码": "M001", "使用数量": 5, "材料单价": 45.00},
>   {"row_index": 1, "材料编码": "M002", "使用数量": 3, "材料单价": 38.50}
> ]
> ```
> （INPUT/FIXED_VALUE 字段直接存值，DATA_SOURCE 字段存查询结果快照数值，FORMULA 字段不存储）

**报价单状态机**

```
DRAFT → SUBMITTED（销售代表提交）
SUBMITTED → APPROVED（审批人通过）
SUBMITTED → DRAFT（审批人退回）
SUBMITTED → DRAFT（销售代表撤回）
APPROVED → SENT（销售代表发送给客户）
SENT → ACCEPTED（客户确认接受，手动标记）
SENT → REJECTED（客户拒绝，手动标记）
SENT / APPROVED → EXPIRED（Quarkus Scheduler 定时任务每日 00:30 UTC+8 执行，条件：expiry_date < 当天日期 AND status IN (SENT, APPROVED)，自动标记为 EXPIRED 并记录操作日志）
```

- 草稿每10秒自动保存到后端，同时缓存到localStorage作为降级备份；后端保存失败时降级为仅本地缓存并提示用户。换设备登录后可从后端恢复未完成的草稿。
- 产品配置弹窗加载时间 < 500ms
- 金额计算精度：Decimal(18,4)，显示精度 2 位
- 单个报价单支持最多 50 个产品行
- 报价编号生成保证唯一性（PostgreSQL SEQUENCE 全局自增，格式 QT-YYYYMMDD-XXXX）
- 公式计算容差约定：前后端计算差异容许范围 ±0.01 元；超出容差时草稿保存报错提示，提交时阻止并要求用户确认

### 模块十：报价审批流程

#### 2.10.1 用户与场景

| 场景编号 | 用户角色 | 场景描述 | 前置条件 | 预期结果 |
|----------|----------|----------|----------|----------|
| AP-01 | 销售代表 | 提交报价单：完成报价填写后提交审批 | 报价单状态为DRAFT | 状态变为SUBMITTED，通知审批人 |
| AP-02 | 审批人 | 通过报价单：审核报价单内容后批准 | 报价单状态为SUBMITTED | 状态变为APPROVED，通知销售代表 |
| AP-03 | 审批人 | 退回报价单：审核不通过，填写退回原因 | 报价单状态为SUBMITTED | 状态变为DRAFT，退回原因通知销售代表 |
| AP-04 | 销售代表 | 重新提交：修改退回的报价单后再次提交 | 报价单已被退回 | 状态变为SUBMITTED，重新进入审批流程 |
| AP-05 | 系统管理员 | 配置审批规则：设置固定审批人或动态规则 | 已登录系统管理 | 审批规则配置生效 |

#### 2.10.2 功能需求

**审批规则**
- 所有报价单提交后必须经过审批，无金额或折扣豁免条件
- 审批人角色由**销售经理**担任。销售经理在系统中同时具备"模板/工序配置管理"和"报价单审批"两项职责
- 审批人支持两种配置方式，固定审批人和规则审批人均从"销售经理"角色的用户中选取：
  - **固定审批人**：在系统管理中指定固定的销售经理账号作为审批人
  - **规则审批人**：按报价单归属的销售代表所在部门/区域，动态匹配对应的销售经理作为审批人
- 系统管理员负责配置审批规则

**兜底审批人机制**
- 当报价单提交时，若未匹配到任何审批规则（包括固定审批人和动态规则），系统自动路由给**系统管理员**作为兜底审批人
- 多系统管理员场景：兜底时取 User 表中 role=SYSTEM_ADMIN 且 status=ACTIVE 的用户，按 created_at 升序取第一个（即最早创建的系统管理员）
- 系统管理员始终具备审批任意报价单的权限，无论审批规则如何配置
- 在系统管理模块的审批规则配置页，顶部显示"当前兜底审批人：{最早系统管理员姓名}"的提示

**审批流程**
- 销售代表提交报价单 → 状态变为 SUBMITTED
- 系统通知审批人（站内消息/邮件）
- 审批人操作：
  - **通过** → 状态变为 APPROVED，通知销售代表
  - **退回** → 状态变为 DRAFT，退回给原销售代表修改，需填写退回原因
- 销售代表修改后可再次提交，重新进入审批流程
- 报价单不支持"作废"操作，退回即回到草稿状态供修改

**审批超时策略**
- 当前版本无自动超时机制，审批人不处理则报价单持续等待
- 销售代表可在报价单详情页查看当前审批状态及提交时间
- 系统在提交后第 48 小时自动发送催办提醒邮件给审批人（销售经理）
- 后续版本迭代计划：支持超时自动升级（报价单超过N天未审批，自动通知上级销售经理）

**数据模型**

```
QuotationApproval {
  id: UUID (PK)
  quotation_id: UUID (FK → Quotation)
  approver_id: UUID (FK → User)
  action: Enum [APPROVED, REJECTED]
  comment: Text (审批意见/退回原因)
  acted_at: Timestamp
  created_at: Timestamp
}

ApprovalRule {
  id: UUID (PK)
  rule_type: Enum [FIXED, DYNAMIC]
  approver_id: UUID (FK → User, 固定审批人时使用)
  match_field: Enum [REGION, DEPARTMENT] (动态规则匹配维度，FIXED类型时为NULL)
  match_value_id: UUID (匹配值ID；当match_field=REGION时引用Region.id，当match_field=DEPARTMENT时引用Department.id；FIXED类型时为NULL)
  priority: Integer (优先级序号，数值越小优先级越高；多条规则并存时按 priority 升序匹配，取第一条匹配成功的规则作为审批人；FIXED 类型规则建议设置比 DYNAMIC 类型更小的 priority 值)
  created_at: Timestamp
}
```

> 兜底规则为系统内置，不可删除。当所有已配置规则均未匹配时，自动使用系统管理员账号作为审批人。

> **审批路由引擎实现说明（Drools）**：
> - 审批人路由由 Drools 规则引擎执行，规则数据来源于 ApprovalRule 表
> - 报价单提交时，系统从数据库加载所有审批规则，构建 KieSession，输入报价单归属销售代表的 region_id 和 department_id，执行规则匹配，输出审批人 user_id
> - 规则匹配优先级通过 Drools 的 salience 属性实现，映射规则为 salience = -priority（priority 数值越小优先级越高，salience 取 priority 的负值）
> - FIXED 类型规则（固定审批人）在 DRL 中统一赋予较高 salience（如 -priority + 1000 偏移），确保同 priority 数值下 FIXED 类型始终优先于 DYNAMIC 类型执行
> - 兜底规则（无匹配时路由系统管理员）作为最低 salience 的默认规则内置于 DRL 文件中
> - 审批规则变更后即时生效，下一次报价单提交时自动加载最新规则
> - 路由结果持久化：Drools 执行完毕后，将输出的审批人 user_id 写入 Quotation.assigned_approver_id 字段，提交即写入，重新提交时覆盖
> - 该字段支撑三个业务场景：
>   ① 销售经理"待我审批"列表：WHERE assigned_approver_id = 当前用户ID AND status = SUBMITTED
>   ② 详情页审批按钮权限判断：当前用户ID = assigned_approver_id 时才显示通过/退回按钮
>   ③ 审批规则变更不影响已提交报价单：规则变更后 assigned_approver_id 已固化，不会因规则变更而重新计算路由结果

#### 2.10.3 非功能需求

- 审批通知应在提交后 5 秒内送达审批人
- 审批操作日志完整记录，支持审计追溯
- 审批规则变更即时生效，不影响已提交的审批流程

#### 2.11.2 功能需求

**报价单详情页定义**

> 报价单详情页是报价单的统一只读视图，与报价单输出预览共用同一页面。所有进入报价单的操作（从报价单管理列表点击"查看详情"、提交成功后跳转、审批操作后跳转）均进入此页面。

**页面结构**：

- **顶部操作栏**：根据报价单当前状态和当前用户角色，动态展示可用操作按钮：
  - 销售代表视图：
    - DRAFT：继续编辑、删除、复制
    - SUBMITTED：复制（等待审批中，无其他操作）
    - APPROVED：发送给客户、复制、延期
    - SENT：标记客户已接受、标记客户已拒绝、延期、复制
    - ACCEPTED / REJECTED：复制
    - EXPIRED：复制
  - 销售经理视图（审批人）：
    - SUBMITTED（且该报价单的审批规则匹配到当前销售经理）：**通过**（绿色主按钮）、**退回**（红色按钮）、复制
    - SUBMITTED（但该报价单不在当前销售经理审批范围内）：查看、复制；通过/退回按钮置灰，悬浮提示"该报价单不在您的审批范围内"
    - 其他状态：复制、查看（无审批操作）
  - 审批操作区：
    - 点击"通过"→ 弹出确认弹窗 → 确认后状态变为 APPROVED，通知销售代表
    - 点击"退回"→ 弹出退回原因填写弹窗（必填）→ 确认后状态变为 DRAFT，通知销售代表

> **审批权限说明**：销售经理只能审批"审批规则路由到自己"的报价单。系统管理员不受此限制，可审批任意报价单（含兜底审批权）。

- **审批历史区**（折叠展开）：展示所有审批记录（操作人、操作时间、操作类型、审批意见），数据来源 QuotationApproval 表

- **正文区域**：即下方"报价单头部 + 产品明细区域 + 报价汇总"的完整内容，展示快照数据

- **操作日志入口**（折叠展开）：销售经理和系统管理员可见，展示该报价单的所有操作记录

> - **发送给客户按钮**：仅在状态为 **APPROVED** 时显示，其他状态不显示此按钮
> - DRAFT / SUBMITTED 状态：无发送按钮（未审批不可发送）
> - SENT 及之后状态：无发送按钮（已发送不可重复发送）

**报价单头部**
- 报价单号（格式 QT-YYYYMMDD-XXXX，XXXX 为全局4位补零流水号）
- 报价日期
- 客户名称与联系人
- 项目名称
- 到期日期（默认创建日期+30天）
- 付款条件（如：预付 30%）
- 交货周期
- **延期操作按钮**（仅在状态为 SENT 或 APPROVED 时可用）
- **客户反馈标记按钮**（仅在状态为 SENT 时显示，见下方说明）

> **数据展示规则**：报价单处于 SUBMITTED 及之后状态时，客户名称、联系人、产品信息均展示快照数据；仅 DRAFT 状态实时引用最新数据。

**客户反馈标记**（仅在状态为 SENT 时显示）
- "客户已接受"按钮：点击后二次确认，确认后状态变为 ACCEPTED，触发客户累计金额自动更新，记录操作日志
- "客户已拒绝"按钮：点击后二次确认，确认后状态变为 REJECTED，记录操作日志（附可选备注填写原因）
- 操作人：仅限该报价单的创建销售代表本人（权限校验）
- ACCEPTED / REJECTED 状态不可逆，操作后按钮消失，状态标签固定显示

**延期操作**
- 点击延期按钮 → 弹窗选择新的到期日期
- 保存后更新 expiry_date 字段，记录操作日志
- EXPIRED 状态的报价单不可延期，仅可复制创建新报价单

> **延期与审批状态说明**：
> - APPROVED 状态延期：仅修改到期日期（expiry_date），无需重新提交审批，状态保持 APPROVED 不变，操作记录到日志
> - SENT 状态延期：同上，状态保持 SENT 不变
> - 延期不属于"修改报价单内容"，不触发快照更新，不影响审批结论

**产品明细区域**
- 多产品卡片（透视表布局），每个产品的结构由其选用的产品卡片模板决定
- 每个产品：产品属性值（由模板 product_attributes 定义，如名称/规格/数量等）+ 产品卡片模板名称
- 页签组件明细表格（根据模板中的页签组件动态渲染）：
  - 每个页签组件渲染为一个数据表格（列结构继承自组件定义）
  - 如：投料金额表（材料编码/单价/数量/小计）、回料金额表、加工费用表、税费表等
  - 公式字段展示计算结果
- 每个页签组件的小计
- 产品小计（所有页签组件小计之和）

**报价汇总**
- 所有产品总计金额
- 商务条款区域
- 签章/审批区域

**导出操作**

*打印*
- 浏览器 print，打印样式优化

*PDF 导出*
- 点击"导出 PDF"弹出导出选项面板，包含：
  - **显示折扣信息**（勾选框，默认不勾选）：勾选后 PDF 中显示"原始总价 / 折扣率 / 折后总价"；不勾选则仅显示折后总价
  - **显示工序明细**（勾选框，默认不勾选）：勾选后 PDF 中显示各产品选用的工序列表
  - **显示页签详情**（勾选框，默认勾选）：勾选后 PDF 显示各页签组件的明细表格；不勾选则仅显示产品小计汇总
- 确认后后端生成 PDF 并下载

*Excel 导出*
- 点击"导出 Excel"弹出导出选项面板，包含：
  - **显示折扣信息**（勾选框，默认不勾选）：同 PDF
  - **包含原始数据 Sheet**（勾选框，默认不勾选）：勾选后额外生成一个 Sheet 包含所有行级原始数据，便于客户二次处理
- 确认后后端生成 Excel 并下载

> **折扣信息展示说明**：
> - 系统内部视图（步骤五预览、报价单详情页）：始终显示完整折扣信息，含原始总价、折扣率、适用规则名称、手动调整标记
> - PDF/Excel 导出：默认不显示折扣率（避免向客户暴露定价策略），由销售代表按需勾选
> - 邮件发送正文：不包含折扣信息，仅显示折后总价

*邮件发送*
- 点击"发送给客户"弹出邮件编辑面板：
  - 收件人：预填报价单联系人邮箱（contact_email 快照），可修改
  - 抄送：可选填（默认为空）
  - 邮件主题：预填"【报价单】{客户名称} - {报价单号}"（可修改）
  - 邮件正文：系统模板（含报价单号、产品数量、折后总价、到期日期，可编辑）
  - 附件：默认附 PDF（自动生成，不显示折扣信息），勾选后可同时附 Excel
- 发送后状态变为 SENT，记录发送时间和收件人邮箱到操作日志

**复制报价单**
- 入口：报价单详情页顶部操作栏，提供"复制"按钮，所有状态的报价单均可复制
- 复制后生成新报价单，以下内容**全部保留**：
  - 客户信息（客户ID、选中联系人）
  - 报价元数据（项目名称、商机编号、报价类型、优先级、阶段、预计成交日期）
  - 所有产品行（产品、产品属性值、选用的产品卡片模板）
  - 每个产品行的工序选择
  - 每个产品行的页签组件数据（QuotationLineComponentData 行数据复制，但 **DATA_SOURCE 字段的旧查询结果值清空为 null**，新草稿中需重新触发查询获取最新值；INPUT 和 FIXED_VALUE 字段值完整保留）
  - 商务条款（付款方式、交货周期）
  - 折扣信息（系统折扣率、手动调整折扣率及原因）（仅作为参考保留；进入步骤三时系统重新计算折扣率，若客户定价策略已变更则覆盖旧值，并提示"折扣已根据当前策略重新计算"）
- 复制后以下内容**自动重置**：
  - 报价单号（重新生成，格式 QT-YYYYMMDD-XXXX）
  - 报价日期（重置为今天）
  - 状态（重置为 DRAFT）
  - 到期日期（重置为今天+30天）
  - 审批记录（不复制历史审批记录）
  - 销售代表（重置为当前操作人）
- 复制完成后自动跳转至新报价单的草稿编辑页（报价生成器步骤一）
- 新报价单顶部显示"来源：复制自 QT-XXXXXXXX-XXXX"的提示标签

> **归档模板处理**：复制时检查每个产品行的模板状态（template_id 对应的 Template.status）：
> - 若模板为 PUBLISHED：正常复制，新草稿可直接使用该模板
> - 若模板为 ARCHIVED：复制成功但在新草稿步骤二中该产品行标记"⚠ 模板已归档，请重新选择可用模板"，产品行数据保留但禁止提交，直到重新选择有效模板
> - 若同名模板系列有更新的 PUBLISHED 版本，系统在标记旁提示"该模板已有新版本 v{X.X}，可点击升级"

#### 2.11.3 非功能需求

- PDF 生成时间 < 3s（10 个产品以内）
- Excel 导出时间 < 2s
- PDF 支持中文字体，排版与预览一致
- 导出文件命名规范：报价单号_客户名_日期
- 导出选项配置不持久化，每次导出独立选择

#### 2.12.2 功能需求

**用户管理**
- 用户列表：展示所有用户，支持按角色/状态筛选、关键词搜索
- 新增用户：用户名（必填）、姓名（必填）、邮箱（必填）、初始密码（系统生成）、角色分配（单选：销售代表/销售经理/定价经理/系统管理员）、所属区域（下拉选择，来源Region字典表）、所属部门（下拉选择，来源Department字典表）
- 编辑用户：支持修改姓名、邮箱、角色、所属区域/部门（下拉选择）
- 停用/启用用户：软停用，历史数据保留，停用账号无法登录
- 不支持物理删除用户，保留历史引用完整性

**审批规则配置**
- 展示当前所有审批规则列表
- 新增规则：
  - 规则类型：固定审批人 / 动态规则
  - 固定审批人：从销售经理账号列表中选择
  - 动态规则：选择匹配维度（区域/部门）+ 下拉选择匹配值（从Region/Department字典中选择）+ 选择对应销售经理
- 编辑/删除规则
- 规则优先级排序（数字越小优先级越高，多条规则时按优先级匹配）
  > 优先级数字存储于 ApprovalRule.priority 字段，由 Drools 的 salience 属性（= -priority）在规则执行时体现
- 规则变更即时生效

**账号安全**

*登录*
- 登录页：用户名（或邮箱）+ 密码登录
- 登录失败超过 5 次，账号锁定 30 分钟，锁定期间提示"账号已锁定，请30分钟后重试或联系管理员"
- 登录成功后跳转至报价单管理页（各角色默认首页相同）
- Session 有效期：8小时，超时自动登出并跳回登录页

> **认证机制**：采用服务端 Session + HttpOnly Cookie 方案（Quarkus Security 内置支持）。登录成功后后端创建 Session 并通过 Set-Cookie 返回 Session ID（HttpOnly、Secure、SameSite=Strict），前端 SPA 每次请求自动携带 Cookie，无需手动管理 Token。Session 存储于服务端内存（单实例部署，与 KieBase 缓存约束一致）。登出时后端销毁 Session 并清除 Cookie。不使用 JWT，避免单实例场景下引入不必要的复杂度。

*首次登录*
- 系统管理员创建账号后，系统发送初始密码邮件给用户
- 用户首次登录时，强制跳转至修改密码页面，不可跳过
  > 强制跳转逻辑依赖 User.is_first_login 字段：登录后端接口检查该字段，为 true 时在响应中携带 force_change_password=true 标志，前端收到后强制导航至改密页，改密成功后后端将 is_first_login 置为 false
- 初始密码有效期 24 小时，超期后需系统管理员重置
  > 有效期通过 User.initial_password_expires_at 字段追踪：创建用户时写入 created_at + 24h；登录时 is_first_login=true 则校验该字段，过期则拒绝登录并提示用户联系管理员；系统管理员重置密码时同步将 initial_password_expires_at 更新为 NOW() + 24h，重新计算过期时间

*修改密码（已登录用户）*
- 入口：右上角用户菜单 → 修改密码
- 需输入：当前密码（验证）、新密码、确认新密码
- 密码复杂度要求：最低8位，必须包含字母和数字

*忘记密码/密码重置*
- 登录页提供"忘记密码"链接
- 用户输入注册邮箱，系统发送重置链接（有效期 1 小时）
- 点击链接后跳转至重置密码页，输入新密码完成重置
- 重置密码链接一次性有效，使用后立即失效
- 系统管理员也可在用户管理页直接为指定用户"重置密码"，系统重新发送初始密码邮件

*用户、角色、权限控制*
- 每个用户有且仅有一个角色（单角色模型）
- 角色变更即时生效：用户当前 Session 在下次 API 请求时重新校验权限，如角色已变更则强制重新加载权限
- 系统管理员不可删除自身账号，不可将自身角色从系统管理员降级（防止系统无管理员）
- 系统中必须至少保留一个有效的系统管理员账号，停用最后一个系统管理员时系统阻止并提示

**区域/部门字典管理**
- 区域列表：编码、名称、状态（启用/停用）、排序；支持新增/编辑/停用
- 部门列表：编码、名称、状态（启用/停用）、排序；支持新增/编辑/停用
- 停用保护：停用前检查是否有 ACTIVE 状态用户关联该区域/部门，有则警告并列出用户列表
- 字典值在用户管理、审批规则配置中以下拉选择方式使用，禁止自由文本输入

**操作日志**
- 日志列表：操作时间、操作人、操作类型、操作对象、变更内容摘要
- 筛选条件：时间范围、操作人、操作类型（折扣调整/审批操作/策略变更/用户变更等）
- 日志只读，不可删除
- 支持导出为 Excel

**数据模型**

```
User {
  id: UUID (PK)
  username: String (UNIQUE, NOT NULL)
  full_name: String (NOT NULL)
  email: String (UNIQUE, NOT NULL)
  password_hash: String (NOT NULL)
  role: Enum [SALES_REP, SALES_MANAGER, PRICING_MANAGER, SYSTEM_ADMIN]
  region_id: UUID (FK → Region, 所属区域，用于动态审批规则匹配，下拉选择)
  department_id: UUID (FK → Department, 所属部门，用于动态审批规则匹配，下拉选择)
  status: Enum [ACTIVE, INACTIVE]
  is_first_login: Boolean (首次登录标识；创建用户时默认为 true；用户首次登录完成强制改密后由系统置为 false；后端每次登录时检查此字段，为 true 则强制跳转改密页面，不可跳过)
  initial_password_expires_at: Timestamp (初始密码过期时间；创建用户时写入 created_at + 24小时；仅在 is_first_login=true 时有效；用户首次登录时后端检查：若 NOW() > initial_password_expires_at 则拒绝登录并返回错误码，提示"初始密码已过期，请联系管理员重置"；首次改密成功后 is_first_login 置 false，此字段不再参与校验)
  created_at: Timestamp
  updated_at: Timestamp
}

PasswordResetToken {
  id: UUID (PK)
  user_id: UUID (FK → User)
  token_hash: String (NOT NULL, UNIQUE, 令牌的哈希值，原始令牌通过邮件发送，服务端仅存储哈希，防止数据库泄露后令牌被直接使用)
  expires_at: Timestamp (NOT NULL, 令牌过期时间，创建时设为当前时间+1小时)
  used_at: Timestamp (NULL表示未使用；令牌使用后立即写入使用时间，后续请求检查此字段非NULL则拒绝，实现一次性有效)
  created_at: Timestamp
}

OperationLog {
  id: UUID (PK)
  operator_id: UUID (FK → User)
  operation_type: String (操作类型，如 DISCOUNT_ADJUST / APPROVAL / PRICING_CHANGE)
  target_type: String (操作对象类型，如 Quotation / Customer / PricingStrategy)
  target_id: UUID (操作对象ID)
  summary: Text (变更内容摘要)
  created_at: Timestamp
}

Region {
  id: UUID (PK)
  code: String (UNIQUE, NOT NULL, 区域编码)
  name: String (NOT NULL, 区域名称，如"华南"、"华东"、"华北")
  sort_order: Integer
  status: Enum [ACTIVE, DISABLED]
  created_at: Timestamp
}

Department {
  id: UUID (PK)
  code: String (UNIQUE, NOT NULL, 部门编码)
  name: String (NOT NULL, 部门名称，如"销售一部"、"销售二部")
  sort_order: Integer
  status: Enum [ACTIVE, DISABLED]
  created_at: Timestamp
}
```

> **密码重置令牌安全规范**：
> - 令牌原始值为随机生成的安全字符串（建议 UUID v4 或 SecureRandom 32字节）
> - 服务端存储令牌的哈希值（SHA-256），邮件中发送原始值
> - 使用令牌时：查找匹配哈希 → 检查 expires_at 未过期 → 检查 used_at 为 NULL → 三项全通过才允许重置密码 → 重置成功后写入 used_at
> - 同一用户申请新令牌时，系统在同一数据库事务内执行以下两步：①将该用户所有未使用的旧令牌（`used_at IS NULL`）写入 `used_at = NOW()`，标记为已失效 ②插入新令牌记录。两步在同一事务内原子执行，确保同一时刻该用户最多存在一个有效令牌，消除并发申请导致多个有效令牌并存的竞态问题
> - 系统管理员重置密码走单独的管理后台接口，不使用此令牌机制

#### 2.12.3 非功能需求

- 密码使用 bcrypt 加密存储，salt rounds ≥ 12，初始密码系统邮件发送
- 重置密码邮件发送延迟 ≤ 30 秒
- 登录接口限流：同一 IP 每分钟最多 20 次请求，超出返回 429
- 操作日志保留至少 2 年
- 审批规则变更记录到操作日志
- 用户角色变更即时生效，当前已登录的 Session 在下次请求时重新校验权限
- PasswordResetToken 表须建立 (user_id, expires_at) 复合索引，支持高效查询未过期令牌
- 过期令牌由定时任务每日清理（DELETE WHERE expires_at < NOW()），避免表膨胀

### 模块十四：数据源管理

> **定位说明**：数据源管理是组件管理的基础设施模块。系统管理员在此配置可供组件字段引用的数据查询服务（SQL查询或第三方API请求），组件管理中的 DATA_SOURCE 类型字段通过绑定数据源实现报价时的动态取值。

#### 2.14.1 用户与场景

| 场景编号 | 用户角色 | 场景描述 | 前置条件 | 预期结果 |
|----------|----------|----------|----------|----------|
| DS-01 | 系统管理员 | 新增SQL数据源：创建查询材料单价的数据源，定义查询SQL、返回栏位、参数列表 | 已登录系统管理 | 数据源创建成功，可在组件管理中选用 |
| DS-02 | 系统管理员 | 新增API数据源：配置调用外部汇率服务的数据源，定义请求URL、返回栏位、参数列表 | 已登录系统管理 | 数据源创建成功，可在组件管理中选用 |
| DS-03 | 系统管理员 | 编辑数据源：修改查询SQL或API地址 | 数据源已存在 | 修改成功；已引用该数据源的组件在模板下次发布时使用新配置 |
| DS-04 | 系统管理员 | 删除数据源：下线不再使用的数据源 | 数据源未被任何组件引用 | 删除成功；若有引用则阻止并提示引用组件列表 |
| DS-05 | 系统管理员 | 测试数据源：填入测试参数值，验证SQL或API返回结果正确 | 数据源已配置 | 返回查询结果预览，确认取值正确 |

#### 2.14.2 功能需求

**数据源列表页**
- 展示所有数据源，列显示：编号、名称、类型（SQL/API）、参数数量、状态、创建时间
- 支持按类型筛选、关键词搜索（编号/名称）
- 每行操作：编辑、测试、删除

**数据源新增/编辑**

表单分三个区域：

*基本信息区*
- 数据源编号（UNIQUE，必填，字母数字下划线，创建后不可修改）
- 数据源名称（必填）
- 数据源类型：SQL查询 / API请求（必选，创建后不可修改）
- 状态：启用 / 停用
- 描述（选填）

*查询配置区*（根据类型展示不同内容）

SQL类型：
- 查询SQL（必填，多行文本框）；参数占位符使用 `?`，按 DataSourceParam.param_order 顺序与参数表对应
- 返回栏位名（必填）：SQL查询结果中需要取值的列名（如 unit_price）；报价时取第一行该列的值填入数据源单元格
- SQL安全约束（系统强制，后端校验）：仅允许 SELECT 语句（大小写不敏感）；禁止包含分号（防多语句注入）；执行时使用 PreparedStatement 参数化查询

API类型：
- 请求URL（必填）；用户字段参数占位符使用 `{param_code}`，系统参数占位符同样使用 `{param_code}`（param_code 即 DataSourceParam.param_code）
- 请求方法：GET / POST（必选）
- 请求头（选填）：键值对列表，支持添加/删除；敏感值在界面展示时脱敏（显示为 ****），执行时明文使用
- 请求体模板（POST时必填）：JSON模板文本框，占位符格式同URL
- 返回值路径（必填）：JSONPath表达式，指向返回JSON中目标字段（如 $.data.price）；取第一个匹配值
- 超时秒数（必填，默认5秒，最大30秒）

*参数配置区*
- 参数列表表格，每行一个参数，包含：
  - 参数顺序（自动编号，可拖拽调整；SQL类型对应 `?` 的位置序号）
  - 参数编码（同一数据源内唯一，字母数字下划线；API类型用于URL/请求体中 `{param_code}` 占位符替换）
  - 参数名称（显示用）
  - 参数来源类型：用户字段引用 / 系统参数
    - 用户字段引用：报价时由组件配置中绑定的同组件字段提供值（需在组件管理中完成绑定）
    - 系统参数：系统自动注入，无需在组件中绑定；在此直接选择系统参数编码
  - 系统参数编码（来源类型为系统参数时显示此列，从固定枚举中选择）
  - 是否必填（必填参数为空时不执行查询）
  - 参数说明（选填，报价时展示辅助信息）

*系统参数枚举*（固定内置，不可增删）：
- SYS_CUSTOMER_ID：当前报价单客户ID（UUID）
- SYS_CUSTOMER_LEVEL：客户等级（String，如 VIP）
- SYS_CUSTOMER_REGION：客户所属区域（String）
- SYS_QUOTE_DATE：报价单创建日期（Date，格式 yyyy-MM-dd）
- SYS_SALES_REP_ID：销售代表ID（UUID）

**数据源测试功能**（编辑页内嵌）
- 列出所有参数并提供测试值输入框（系统参数也可手动覆盖，仅测试用）
- 点击"执行测试"：后端执行查询/请求
- 展示：原始返回数据（折叠）、提取到的目标值（高亮）、执行耗时

**数据模型**

```
DataSource {
  id: UUID (PK)
  code: String (UNIQUE, NOT NULL, 数据源编号，创建后不可修改)
  name: String (NOT NULL)
  type: Enum [SQL, API] (创建后不可修改)
  status: Enum [ACTIVE, DISABLED]
  description: Text
  -- SQL专用（type=API时为NULL）
  sql_query: Text
  sql_result_column: String (取查询结果第一行该列名的值)
  -- API专用（type=SQL时为NULL）
  api_url: String
  api_method: Enum [GET, POST]
  api_headers: JSONB ([{"key":"...","value":"..."}])
  api_body_template: Text (POST时使用，GET时为NULL)
  api_result_path: String (JSONPath，如 $.data.price)
  api_timeout_seconds: Integer (默认5，最大30)
  created_by: UUID (FK → User)
  created_at: Timestamp
  updated_at: Timestamp
}

DataSourceParam {
  id: UUID (PK)
  datasource_id: UUID (FK → DataSource)
  param_order: Integer (NOT NULL, 从1起；SQL类型对应?位置序号)
  param_code: String (NOT NULL, 同一数据源内唯一；API类型用于占位符替换)
  param_name: String (NOT NULL)
  source_type: Enum [USER_FIELD, SYSTEM_PARAM]
  system_param_code: String (source_type=SYSTEM_PARAM时填写，如SYS_CUSTOMER_LEVEL；USER_FIELD时为NULL)
  is_required: Boolean (NOT NULL, 默认true)
  description: Text
  created_at: Timestamp
}
```

#### 2.14.3 非功能需求

- SQL查询执行超时统一限制10秒，强制中断
- **SQL 权限隔离**：数据源 SQL 查询使用独立的只读数据库连接（Quarkus 多数据源配置，命名为 `datasource-readonly`），配置专用的只读数据库用户；该用户仅授权 SELECT 权限，且仅授权访问业务数据表（如材料主数据表、价格表等），禁止访问系统表（User、OperationLog、Quotation、PasswordResetToken 等）；Quarkus 中通过 `quarkus.datasource.datasource-readonly.*` 配置独立连接池
- API请求超时由 api_timeout_seconds 控制，仅允许 HTTPS 地址
- **API 认证信息加密存储**：api_headers JSONB 中的敏感值（如 Authorization、API-Key 等请求头的 value）使用应用层 AES-256 加密后存储，运行时解密使用；加密密钥通过 Quarkus 配置属性注入，不硬编码在代码中；界面展示时显示为 `****`，编辑时留空表示保持原值不变
- 数据源删除保护：删除前检查 Component.fields 中是否存在引用该数据源的字段（通过 datasource_binding.datasource_id 判断）；有引用则阻止，提示引用组件名称列表
- 数据源的新增、编辑、删除操作记录到 OperationLog，target_type 为 DataSource
- 数据源执行不阻塞报价主流程：查询失败不影响其他字段录入和草稿保存
- **数据源配置版本说明**：数据源配置（SQL/API）变更后，新的报价单在查询时使用最新配置；已有报价单的历史数据不受影响，因为 DATA_SOURCE 字段的查询结果值已在 row_data 中作为快照存储，后续数据源配置变更不改变历史快照值

- **站内消息**：写入 Notification 表，前端每 30 秒轮询未读数量接口（GET /api/cpq/notifications/unread-count），数量变化时刷新通知列表

- 两种方式并行触发，互不依赖

### 模块十七：并发锁机制

> **定位说明**：并发锁机制为产品导入与 DDL 扩列操作提供互斥保障，防止并发冲突导致数据损坏。两种锁相互感知、双向互斥，通过数据库行锁实现，无需引入额外中间件。V1 阶段提供管理员 API，监控 UI 留至 Phase 5。

#### 2.17.1 业务背景

- **产品导入**操作会锁定特定客户或料号范围，期间禁止同范围的并发导入
- **DDL 扩列**操作（未来 Phase 1 第 25 项）需对全表加结构锁，期间禁止任何产品导入
- 两种锁需双向互斥：存在活跃 DDL 锁时拒绝导入申请，存在活跃产品导入锁时拒绝 DDL 锁申请
- 拒绝时统一返回 HTTP 423（Locked）

#### 2.17.2 产品级悲观锁

**自适应粒度**
- 料号数 ≤ `import.product_lock_downgrade_threshold`（默认 100）：申请**料号级锁**（精确锁定涉及的每个 part_no）
- 料号数 > 阈值：降级为**客户级锁**（锁定整个 customer_id，粒度粗但申请代价低）
- 阈值可通过系统配置中心动态调整（仅 SYSTEM_ADMIN 可修改）

**锁生命周期**

```
申请（ACTIVE）→ 心跳续期（每 30s）→ 业务完成（释放：RELEASED）
                                   → 超时未续期（自动过期：EXPIRED）
                                   → 管理员强制释放（RELEASED）
```

- 超时扫描：定时任务每 60 秒扫描 `expires_at < NOW()` 的 ACTIVE 锁，置为 EXPIRED
- 心跳续期：前端导入进行期间每 30 秒调用心跳接口延长 `expires_at`

**数据模型**

```
product_import_lock {
  id: UUID (PK)
  customer_id: UUID (NOT NULL, FK → Customer)
  lock_scope: Enum [PART_NO, CUSTOMER]（锁粒度）
  locked_part_nos: JSONB（PART_NO 级时存料号列表；CUSTOMER 级时为 NULL）
  status: Enum [ACTIVE, RELEASED, EXPIRED]
  locked_by: UUID (FK → User, 持锁用户)
  expires_at: Timestamp (NOT NULL)
  created_at: Timestamp
  updated_at: Timestamp
  -- 唯一约束：同一 customer_id + part_no 同时只能有 1 条 ACTIVE（部分唯一索引）
}
```

**API 端点（产品导入锁）**

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| POST | /api/cpq/import/locks | 申请产品导入锁 | 已登录 |
| DELETE | /api/cpq/import/locks/{id} | 业务完成后释放锁 | 锁持有人 |
| POST | /api/cpq/import/locks/{id}/heartbeat | 心跳续期 | 锁持有人 |
| GET | /api/system/locks/product-imports | 查询所有活跃产品导入锁（监控） | SYSTEM_ADMIN |
| POST | /api/system/locks/product-imports/{id}/release | 管理员强制释放 | SYSTEM_ADMIN |

#### 2.17.3 DDL 全局锁

**设计原则**
- 单行锁（`lock_key = 'global'`），通过 UPSERT 实现原子申请
- 全局唯一，存在 ACTIVE 状态时拒绝所有并发申请（HTTP 423）
- 与产品导入锁双向互斥：申请前互查对方活跃状态

**数据模型**

```
ddl_operation_lock {
  id: UUID (PK)
  lock_key: String (NOT NULL, UNIQUE，固定值 'global')
  status: Enum [ACTIVE, RELEASED, EXPIRED]
  locked_by: UUID (FK → User)
  operation_desc: String（操作说明，如"扩列：新增价格字段"）
  expires_at: Timestamp (NOT NULL)
  created_at: Timestamp
  updated_at: Timestamp
}
```

**API 端点（DDL 锁）**

| 方法 | 路径 | 说明 | 权限 |
|------|------|------|------|
| GET | /api/system/locks/ddl | 查询 DDL 锁当前状态 | SYSTEM_ADMIN |
| POST | /api/system/locks/ddl/release | 管理员强制释放 DDL 锁 | SYSTEM_ADMIN |

#### 2.17.4 锁互斥协议

```
申请产品导入锁前：
  SELECT 1 FROM ddl_operation_lock WHERE status = 'ACTIVE'
  → 有结果 → 返回 HTTP 423，提示"系统正在执行结构变更，请稍后重试"

申请 DDL 锁前：
  SELECT 1 FROM product_import_lock WHERE status = 'ACTIVE'
  → 有结果 → 返回 HTTP 423，提示"存在进行中的产品导入，请等待完成后重试"

产品导入锁申请（同范围）：
  使用 PostgreSQL SELECT ... FOR UPDATE SKIP LOCKED 检测冲突
  → 冲突 → 返回 HTTP 423，提示持锁用户信息和预计释放时间
```

#### 2.17.5 非功能需求

- 锁申请、释放、心跳接口响应时间均 < 200ms
- 超时扫描任务：Quarkus Scheduler，`@Scheduled(every = "60s")`，失败时记录日志不重试
- V1 不提供 UI 监控页面（REST API 可用），UI 留至 Phase 5（编号 26 号任务）
- DDL 锁的完整扩列业务流程（TECH-4 完整实现）留至 Phase 1 第 25 号任务

### 3.2 核心业务流程

**配置阶段（管理员预先配置，一次配置多次复用）：**

```
[组件管理: 创建页签组件]
  │  定义列结构、字段类型、计算公式
  │  如：投料组件(材料编码/单价/数量/小计)
  ▼
[产品卡片模板配置: 拖拽组装产品卡片]
  │  将多个页签组件拖入画布，配置产品属性和小计公式
  │  如：标准件卡片 = 投料金额 + 回料金额 + 加工费用 + 税费
  ▼
[产品选型与模板关联: 绑定产品+工序组合↔产品卡片]
  │  为每个产品的不同工序组合指定可用的产品卡片模板
  ▼
[配置完成，等待销售使用]
```

### 3.3 交互设计原则

- **渐进式表单**: 报价生成采用步骤向导，减少认知负担
- **上下文感知**: 客户选择后自动加载关联数据（联系人列表、策略、历史）
- **实时反馈**: 金额计算、公式验证即时呈现结果
- **防错设计**: 必填校验、草稿自动保存、必选工序不可取消
- **可视化配置**: 组件管理与产品卡片配置均采用拖拽式设计，降低配置门槛
- **分层复用**: 组件 → 产品卡片模板（发布时快照）→ 报价单三级复用架构。组件变更仅影响草稿状态的模板（打开时提示更新），已发布模板保持快照隔离，确保历史报价单数据稳定。

### 4.4 计算引擎架构

系统内存在三个独立的计算引擎，职责边界明确，互不耦合：

| 引擎 | 技术实现 | 触发时机 | 规则来源 |
|------|----------|----------|----------|
| **折扣计算引擎** | Drools KieSession | 进入报价步骤三时 | PricingStrategy + PricingRule 表（动态加载） |
| **审批路由引擎** | Drools KieSession | 报价单提交审批时 | ApprovalRule 表（动态加载） |
| **公式计算引擎** | Apache Commons JEXL | 报价时每行数据录入后及产品属性变更后实时触发 | Template.components_snapshot（快照读取）+ Template.subtotal_formula |

**设计原则**：
- Drools 负责"条件-动作"型规则（if 金额满足阈值 then 应用折扣率），规则可由业务人员通过系统界面配置，无需修改代码
- JEXL 负责"表达式求值"型计算，包括两个层级：① 组件内行级公式（材料单价 × 使用数量），由组件管理中拖拽配置；② 产品小计公式（各页签组件小计运算，可引用 NUMBER 类型产品属性如数量），由模板配置中拖拽配置。两级公式均存储为 JSONB 表达式数组
- 两者严格分工，不混用
- Drools salience 与业务 priority 的映射统一使用 `salience = -priority` 规则（FIXED 类型审批规则额外叠加 +1000 偏移量以确保类型优先级），开发团队须严格遵守此约定，禁止直接将 priority 值赋给 salience
- **公式计算引擎前后端分工**：公式计算采用前后端双执行策略。前端用 JavaScript 实现与 JEXL 等价的表达式解析逻辑，用于报价时每行数据录入后的实时计算展示（无网络请求，零延迟）；后端 JEXL 在草稿每次保存和提交审批时执行最终校验计算。**前后端计算一致性约定**：引入精度容差参数，允许误差范围 ±0.01 元（对应分位精度）。草稿保存时，后端 JEXL 校验结果与前端展示值差异在容差内，静默以 JEXL 结果覆盖存储；差异超出容差时，返回错误响应，前端高亮标红差异行并提示"第X行计算结果与本地不一致，请检查输入数据后重新保存"。提交审批时，后端 JEXL 重新校验所有行，任何行差异超出容差则阻止提交，提示"产品明细计算结果存在偏差，请返回步骤二确认数据后重新提交"。不允许静默覆盖：金额差异必须明确告知用户，不得在用户不知情的情况下修改其录入的计算结果。
- **Drools KieBase 缓存策略**：KieBase（规则编译结果）为重操作，须采用缓存机制避免每次重复构建。以客户所有 ACTIVE 策略的 updated_at 最大值作为缓存版本标识：策略未变更时复用缓存 KieBase，仅创建轻量级 Stateless KieSession 执行规则匹配；策略发生变更（新增 / 修改 / 状态变化）时清除该客户对应的 KieBase 缓存，下次报价时重新构建。审批路由 KieBase 同理，以 ApprovalRule 表 updated_at 最大值为版本标识，规则变更时清除缓存。
- **Drools 版本选择说明**：本项目选用 Drools 7.74.x 而非 8.x/Kogito，原因是 Kogito（8.x）已移除运行时动态 DRL 加载能力，规则须在编译期打包进 jar，不支持从数据库动态加载规则的业务需求。Drools 7.74.x 的 KieServices → KieFileSystem → KieBuilder 动态编译链路完整可用，是当前需求下的正确选择。在 Java 17 下需在 Quarkus 启动参数中添加：`--add-opens java.base/java.lang=ALL-UNNAMED` `--add-opens java.base/java.util=ALL-UNNAMED` 以解除 JDK 内部 API 访问限制。项目须以 JVM 模式运行，不使用 GraalVM native image（native 模式不包含运行时 Java 编译器，无法支持动态 DRL 编译）。
- **KieBase 缓存架构约束（重要）**：当前 KieBase 缓存存储于 JVM 堆内存中，为单实例本地缓存。本系统架构约束为**单实例部署**，不支持水平横向扩展（scale-out）。原因：多实例部署时各实例 KieBase 缓存独立，规则版本可能不同步，导致同一报价单的两次提交被路由到不同审批人，产生数据不一致问题。如未来需要水平扩展，须在扩展前将 KieBase 缓存迁移至分布式缓存（如 Redis），并实现集群级别的缓存失效同步，此为重大架构升级，需独立立项评估。运维团队须知晓此约束，严禁在未完成缓存分布式改造前部署多实例。
- **Quarkus 线程模型约束（@Blocking）**：Drools KieSession 执行（折扣计算、审批路由）和 JEXL 表达式求值（公式计算后端校验）均为 CPU 密集型阻塞操作，须在 Quarkus Worker 线程池上运行，严禁在 Vert.x I/O 事件循环线程上直接调用。实现要求：所有调用 KieSession 或 JEXL 的 Service 方法须标注 `@io.smallrye.common.annotation.Blocking` 注解；未标注时 Quarkus 检测到 I/O 线程执行阻塞操作将抛出运行时异常，在 RESTEasy Reactive 下尤为明显。此约束须在 M4a 技术 Sprint 中作为独立验收项验证，验证方式见 Sprint 10-11 验收标准第5条。

### 4.6 定时任务清单（Quarkus Scheduler）

所有定时任务统一使用 Quarkus Scheduler（`@Scheduled` 注解 + cron 表达式），在应用启动时自动注册。

| 任务名称 | cron 表达式 | 执行时间 | 说明 | 失败处理 |
|----------|-------------|----------|------|----------|
| 报价单过期标记 | `0 30 0 * * ?` | 每日 00:30 | 扫描 status IN (SENT, APPROVED) AND expiry_date < 当天日期，批量标记 EXPIRED，逐条记录操作日志 | 单条失败不阻断，记录错误日志继续下一条 |
| 定价策略过期标记 | `0 0 1 * * ?` | 每日 01:00 | 扫描 status=ACTIVE AND expiration_date < 当天日期（expiration_date 非 NULL），标记 EXPIRED | 同上 |
| 审批催办提醒 | `0 0 9 * * ?` | 每日 09:00 | 扫描 status=SUBMITTED AND created_at < NOW()-48h AND 尚未发过催办通知，发送站内消息+邮件给 assigned_approver_id | 邮件失败仅记录日志，站内消息独立写入 |
| 过期令牌清理 | `0 0 3 * * ?` | 每日 03:00 | DELETE FROM PasswordResetToken WHERE expires_at < NOW() | 失败记录日志，下次重试 |
| 过期通知清理 | `0 0 4 * * MON` | 每周一 04:00 | DELETE FROM Notification WHERE created_at < NOW()-6个月 | 同上 |

**通用规范**：
- 所有定时任务须标注 `@Blocking`（同 Drools/JEXL 约束），避免阻塞 I/O 线程
- 每次执行开始和结束均输出结构化日志（任务名称、处理条数、耗时、异常数）
- 任务执行期间不加全局锁（单实例部署无并发风险），但须做幂等处理（重复执行不产生副作用）
- 时区统一为 UTC+8（通过 `quarkus.scheduler.cron-type=unix` + JVM 时区参数 `-Duser.timezone=Asia/Shanghai`）

### 5.3 风险与对策

| 风险 | 影响 | 概率 | 对策 |
|------|------|------|------|
| 模板配置复杂度超预期 | 配置中心延期 | 中 | 优先实现核心配置，高级功能分期迭代 |
| 公式引擎性能不足 | 报价计算缓慢 | 低 | 预计算 + 缓存策略 |
| PDF 中文排版问题 | 导出效果差 | 中 | 提前进行 POC 验证 |
| 需求变更频繁 | 整体延期 | 高 | 需求冻结机制 + 变更评审流程 |
| 第三方邮件集成受限 | 发送功能延期 | 低 | 预留接口，邮件服务可插拔 |
| 审批流程配置复杂度 | 审批模块延期 | 中 | 提前与业务确认审批规则，优先实现固定审批人模式 |
| 外部系统产品数据同步接口对接 | V2 产品同步功能延期 | V1 不涉及 | V1 仅实现手动 CRUD 和 Excel 导入，同步功能推迟到 V2；V2 启动前需提前确认外部系统 API 规范，数据模型已预留 external_id/last_synced_at 字段 |
| Drools 7.74.x 与 Quarkus 3.23.3 + Java 17 兼容性问题 | 定价与报价模块延期 | 低 | Sprint 1-2 阶段完成POC验证（动态DRL加载链路 + --add-opens参数配置 + KieBase缓存机制）；POC不通过时降级为纯Java Service实现折扣计算和审批路由，接口定义不变，业务逻辑等价，对上层功能零影响 |
| 规则引擎并发性能不达标 | 步骤三响应超时，违反 200ms 约束 | 低 | KieBase 缓存机制 + 压测验证（模拟 100 并发用户同时进入步骤三）；压测不达标时降级为纯 Java Service 实现，保留接口不动 |

### 6.1 术语表

| 术语 | 说明 |
|------|------|
| **CPQ** | Configure, Price, Quote（配置、定价、报价） |
| **工序** | 产品制造过程中的工艺流程环节 |
| **产品卡片模板** | 定义产品在报价单中展示结构的可配置模板 |
| **定价策略** | 面向特定客户的折扣规则集合 |
| **组件** | 模板中可复用的字段/公式配置单元 |
| **报价行项目** | 报价单中的单个产品报价记录 |
| **审批人** | 负责对报价单进行审核通过或退回操作的用户 |
| **主要联系人** | 客户下标记为主联系人的联系人，用于报价单默认填充 |
| **折扣率（应付比例）** | 以 0-100 数值表示客户应付原价的百分比。100=不打折，90=9折（付原价90%），计算公式：折后价=原价×折扣率/100 |
| **模板系列** | 同一 template_series_id 下的所有模板版本集合，代表同一产品卡片模板的迭代历史 |
| **站内通知** | 系统内部消息机制，审批事件等触发后写入通知表，前端铃铛角标实时展示 |
