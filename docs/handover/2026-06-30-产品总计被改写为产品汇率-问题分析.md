# 问题分析报告：`[产品(总计)]` 保存后被改写为 `[产品.汇率]`

> 日期：2026-06-30 ｜ 模块：组件管理 · 页签连表公式 ｜ 类型：序列化往返(round-trip)信息丢失
> 结论：**已定位根因（确认级），属页签连表公式 token 序列化的"非对称/有损往返"缺陷。本报告仅做问题分析，未改代码。**

---

## 1. 现象

在「组件管理 → 报价小计(COMP-0034_imp1) → 配置页签连表公式」抽屉里：

1. 用户在公式表达式区点了「页签总计」分组下的绿色虚线按钮 **`产品(总计)`**，表达式区出现 `[产品(总计)]`（Image #1）。
2. 点「保存」后回到公式管理列表，「公式2」的表达式被显示成 **`[产品.汇率]`**（Image #2）。

即：用户选的"整页签总计"被静默改写成了该页签的"汇率小计列"。

---

## 2. 复现链路（数据流）

| 步骤 | 代码位置 | 行为 |
|---|---|---|
| ① 插入按钮 | `tabjoin/TabFieldMatrix.tsx:238` | 「页签总计」按钮 `onInsert(`[${ref}(总计)]`)` → 插入 **`[产品(总计)]`** |
| ②（对照）小计列按钮 | `tabjoin/TabFieldMatrix.tsx:207` | 「汇率(小计)」按钮 `onInsert(`[${ref}.${f}]`)` → 插入 **`[产品.汇率]`** |
| ③ 保存：串→token | `TabJoinFormulaDrawer.tsx:427/468` → `expressionToTokens()` | 把表达式串解析成 `FormulaToken[]` 落库 |
| ④ 回显：token→串 | `ComponentManagement.tsx:384` → `tokensToDrawerExpression()` | 列表「表达式」列用 token 反序列化成串显示 |

保存触发 ③→④ 一次完整往返，问题就出在这次往返。

---

## 3. 根因（确认级）

**`component_subtotal` token 无法区分"整页签总计 `(总计)`"与"具体某个小计列 `.列名`"，而解析(parse)侧在处理 `[产品(总计)]` 时提前把"第一个小计列名"塞进了 `value`，丢掉了"这是整页签总计"的意图；序列化(serialize)侧又把任何非空 `value` 一律当成列引用回显 `[页签.列]`。**

拆成两半看：

### 3.1 解析侧把 `(总计)` 急切地落成了第一个小计列

`formulaSerialize.ts:818-847`（处理无点、以 `(总计)` 结尾的 `[产品(总计)]`）：

```ts
const primarySubtotal = (tabDef.subtotalCols ?? [])[0] ?? '';   // 产品.subtotalCols[0] = '汇率'
result.push({
  type: 'component_subtotal',
  value: primarySubtotal,          // ← '汇率'，意图"整页签总计"已丢失
  tab_name: primarySubtotal,       // ← '汇率'
  component_code: tabDef.alias,
  label: '产品·汇率',
});
```

产品页签(COMP-0027_imp1)的小计列依次是 `汇率 / 利润比例 / 单率`（Image #1 小计列分组顺序），`[0]` = **汇率**，与最终错误输出 `产品.汇率` 完全吻合。

> 关键副作用：`[产品(总计)]` 解析出的 token 与小计列按钮 `[产品.汇率]` 解析出的 token（见 `formulaSerialize.ts:802-810`）**逐字段完全相同**。
> 也就是说两种来源在存储模型里已经无法分辨 —— 信息在 parse 阶段就丢了，不只是显示问题。

### 3.2 序列化侧"非空 value 即列引用"

`formulaSerialize.ts:913-925`：

```ts
case 'component_subtotal': {
  const col = token.value ?? '';        // '汇率'
  if (col) {
    parts.push(`[${label}.${col}]`);    // ← 命中，回显 [产品.汇率]
  } else {
    parts.push(`[${label}(总计)]`);     // 只有 value 为空才回显 (总计)
  }
}
```

只有当页签 **没有任何小计列**（value=''）时，`[页签(总计)]` 才能原样往返；只要页签有小计列，`(总计)` 必被改写成 `[页签.第一个小计列]`。

---

## 4. 影响面与严重度（2026-06-30 修订：算值确实改变）

> ⚠️ 早先初稿里"算值基本不变"的说法**错误，已作废**。经核对三处求值器，算值会变。

### 4.1 求值器实证：`value` 非空 → 命中"列小计"，根本走不到"整页签总计"

`componentSubtotals` 这张表同时登记两类键（`QuotationStep2.tsx:1219-1231`）：

```ts
const byCol = computeTabSubtotalsByColumn(comp, ...);  // {汇率:X, 利润比例:Y, 单率:Z}
let subtotal = 0;
for (const [colName, colVal] of Object.entries(byCol)) {
  subtotal += colVal;                                  // 累加所有小计列
  componentSubtotals[`${id}#${colName}`] = colVal;     // 列键：alias#汇率 = X
}
componentSubtotals[comp.componentId] = subtotal;       // 裸键：alias = X+Y+Z（整页签总计）
```

- **裸键** `componentSubtotals[component_code]` = **X+Y+Z**（该页签所有小计列之和）= `[产品(总计)]` 应得值
- **列键** `componentSubtotals[component_code#汇率]` = **X**（仅汇率列）= 被改写后的 `[产品.汇率]` 所得值

三处求值器对 `component_subtotal` 的解析（均"列键优先、value 为空才回退裸键"）：

| 求值器 | 位置 | 行为 |
|---|---|---|
| 前端 | `formulaEngine.ts:263-274` | `value` 非空→ `code#value` 列键(X)；空→ `code` 裸键(X+Y+Z) |
| 后端·报价 | `FormulaCalculator.java:140-152` | 同上，注释明写"与前端 1:1 对齐" |
| 后端·配置 | `FormulaCalculationService.java:187-191` | **只读 `component_code` 裸键**，恒取整页签总计 |

因为损坏的 token 里 `value='汇率'`（非空），前端与后端·报价都命中 `alias#汇率` = **X**，拿不到整页签总计 **X+Y+Z**。

### 4.2 结论

- **算值实质改变**：`[产品(总计)]` 期望 X+Y+Z，实际算成单列 X（X≠X+Y+Z，除非其余小计列恰为 0）。这是**线上错算**，不只是显示问题。
- **从首次保存即损坏**：`formulaSerialize.ts:838` 在 parse 阶段就把 `value` 塞成 `汇率`，所以**第一次保存起，求值就已经只取汇率列**；列表回显改字只是把已损坏的 token 暴露出来。
- **范围**：所有"页签连表公式 / 卡片公式 / SUBTOTAL 组件"里用 `[页签(总计)]` 且该页签有 ≥1 小计列的场景，保存后均被降级成"首个小计列"。非个例。

---

## 5. 确认可执行的方案（不改变公式计算逻辑）

**目标**：让 `[产品(总计)]` 解析出的 token 落成"整页签总计"形态（`value` 空、仅 `component_code`），由**现有未改动**的求值器经裸键路径算出 X+Y+Z；并能原样回显 `[产品(总计)]`。

**唯一改动点**：`formulaSerialize.ts` 第 838 行 —— bare `[alias(总计)]` 分支**不要**再把首个小计列名塞进 token，`value`/`tab_name` 置空：

```ts
// 现状（错）：
const primarySubtotal = (tabDef.subtotalCols ?? [])[0] ?? '';   // 急切取首个小计列 → 污染 value
// 改为：
const primarySubtotal = '';                                     // 整页签总计：value 留空，求值器走裸 component_code
```

改后该分支既有的 `value:''/tab_name:''/label:componentName` 逻辑（840-846 行的 else 支）会自然生效。

**为何满足"不改变公式计算逻辑"**：
- 三处求值器（`formulaEngine.ts` / `FormulaCalculator.java` / `FormulaCalculationService.java`）**一行都不动**；
- 序列化器 `tokensToDrawerExpression`（913-925 行）**一行都不动** —— value 空时它本就回显 `[label(总计)]`；
- 仅停止 parse 阶段对 token 的污染，让"既有求值数学"按其本来的裸键回退算出整页签总计。
- `[产品.汇率]` 小计列引用（parse 走 802-810 行、value='汇率'）路径完全不受影响，照旧算 X、回显 `[产品.汇率]`。

**对称往返单测**（`formulaSerialize.test.ts` 补 2 条）：
- `[产品(总计)]` → tokens（断言 `value` 为空、`component_code` 正确）→ 串，必须仍是 `[产品(总计)]`；
- `[产品.汇率]` → tokens（断言 `value='汇率'`）→ 串，必须仍是 `[产品.汇率]`。

### 5.1 ⚠️ 存量数据的不可自动修复性（必须告知）

修复前，「点 `产品(总计)`」与「点 `汇率(小计)`」保存出的 token **字节级相同**（都 value='汇率'）。因此**已存库的 `value='汇率'` token 无法被脚本安全区分**到底原意是"总计"还是"用户确实选了汇率列"。结论：

- 代码修复后，**新的**保存才会正确（`产品(总计)` → value 空）。
- **存量**误存为"总计但 value=列"的公式，只能由用户**重新打开该公式、删掉旧块、重新点一次 `产品(总计)`** 再保存来纠正；不做盲目数据迁移（否则会把"用户本就想要的列引用"误改成总计）。
- 受影响存量清单可另行排查（grep 快照/组件 JSONB 里 `component_subtotal` 且 value∈subtotalCols 的 token），但修复动作仍须人工确认逐条意图。

### 5.2 不采纳的替代方案

| 方案 | 不采纳原因 |
|---|---|
| serialize 端按"value===subtotalCols[0]"猜回 `(总计)` | 无法区分"用户本就想引用首列小计"与"整页签总计"，会把合法 `[产品.汇率]` 反向改成 `(总计)`，制造对称反向 bug |
| 给 token 加 `wholeTab` 判别位、保留 value=列 | 改 token schema + 求值器需识别新位，触及"公式计算逻辑"，且与"value 空即整页签总计"的现有约定重复，徒增维度。直接置空更简洁、零求值改动 |

---

## 6. 一句话总结

`[产品(总计)]` 在"串→token"(parse `formulaSerialize.ts:838`)时被急切替换成首个小计列名 `汇率` 存入 `value`，导致三处求值器都命中"汇率列小计 X"而非"整页签总计 X+Y+Z"——**自首次保存即错算**，列表回显成 `[产品.汇率]` 只是症状暴露。根因是 parse 阶段污染了 `component_subtotal` 的 `value`。修法：bare `(总计)` 分支不再塞首列名、`value` 置空，由**完全不改动**的求值器走裸 `component_code` 算出整页签总计——满足"不改变公式计算逻辑"。
