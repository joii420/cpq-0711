# 配置方法论：组件 · 模板 · 公式

> **用途**：CPQ 系统中"组件配置 / Excel 模板配置 / 公式配置"的快速决策手册。沉淀自 V96~V115 实战，避坑清单覆盖 14 个真实事故。
>
> **读者**：业务管理员、SYSTEM_ADMIN、AI 协作者。
>
> **使用方式**：拿到一个配置需求，按本文「决策树」定位配置层 → 套对应「模式模板」→ 跑「自检清单」。30 分钟内完成一个新指标的端到端配置。
>
> **前置**：先读 `Excel模板配置指南.md` 了解列字段语义，再读 `数据一致性方法论.md` 了解数据流约束。

## 一、四层架构（看图定位）

```
L0  物理层  ────  costing_*_price / mat_bom / mat_fee / costing_part_* 等表 + v_costing_*_price 视图
                ↑ 维护入口: 「核价基础数据」/ 数据导入 (V96 前) → 「全局变量配置」(V109 后)

L1  注册层  ────  global_variable_definition (V104) — 给 L0 数据起名字、暴露给公式
                ↑ 维护入口: 「全局变量配置」菜单 (CRUD + 变更日志, V106)

L2  组件层  ────  component (BASIC_DATA / FORMULA 字段 + token 数组)
                ↑ 维护入口: 「组件管理」菜单, FieldConfigTable + FormulaBuilder

L3  视图层  ────  costing_template (Excel 列定义: VARIABLE / FORMULA + comparison_tag)
                ↑ 维护入口: 「Excel 模板配置」菜单, CostingTemplateConfig 页

—— 运行时 ——
组件 BASIC_DATA / 模板 VARIABLE 列 → FormulaEngine.evaluate("{path}") → ImplicitJoinRewriter
注入 partNo/customerId/driver-row 同名列 → DataLoader → PostgreSQL → 单值/单行/多行
```

每层的"语义边界"清晰：**改 L0 数据 → L1~L3 实时反映；改 L2 组件 → 已发布报价不动 (snapshot 冻结)；改 L3 列定义 → 同样**。

## 二、决策树：先选层

### Q1: 我要配什么？

| 需求 | 选哪层 | 配置入口 |
|---|---|---|
| 给字段加"取数能力"（取一个值出来） | **L2 组件** BASIC_DATA | 组件管理 → 字段类型 BASIC_DATA + path |
| 给字段加"计算公式"（算出一个值） | **L2 组件** FORMULA | 组件管理 → 字段类型 FORMULA + token 表达式 |
| 给报价/核价单加"一列数据展示" | **L3 Excel 模板** VARIABLE | Excel 模板配置 → 加列 → source_type=VARIABLE |
| 给报价/核价单加"一列计算结果" | **L3 Excel 模板** FORMULA | Excel 模板配置 → 加列 → source_type=FORMULA |
| 给比对视图加"一个维度差异指标" | **L0 ComparisonTag + L3 列 tag** | V114 SQL 注册 + Excel 列设 comparison_tag |
| 让公式取"参考价格/汇率/系数" | **L1 全局变量** | 全局变量配置 → 注册 + 维护数据 |
| 让公式跨组件汇总 | **L2 component_subtotal token** | 公式编辑器 → "其他数据源" tab → 选组件小计 |
| 让公式按"当前行的元素"动态查表 | **L1 全局变量动态 key 模式** | 全局变量 picker → "⚡ 动态查表" |

### Q2: 字段值是单值还是多行？

| 场景 | 答案 | 实操 |
|---|---|---|
| 整张报价单一个值（如汇率、税率） | **单值** | 全局变量静态 key 或 BNF 带谓词路径 |
| 每料号一行（如 BOM、工序成本） | **多行** | 组件 `data_driver_path` 设主表，字段 path 不带谓词靠隐式 JOIN |
| 每电镀方案行（嵌套于料号下） | **多行** | driver 视图加 hf_part_no + 业务键，隐式 JOIN partNo 自动收窄 |

### Q3: 跨组件取值？

| 场景 | 答案 |
|---|---|
| 单值跨组件（如总成本卡片引用 PLATING-SCHEME 的 ∑行电镀材料费） | `component_subtotal` token，对方组件需有 `is_subtotal=true` 字段 |
| 跨表按当前行元素查表（如电镀方案行的元素 → 元素单价） | 全局变量动态 key 模式（path 不带谓词，靠 ImplicitJoinRewriter 注入 driver row 的 element_name） |

## 三、组件配置三种模式

### 模式 A：多行展示型组件

**典型**：`COMP-V4-RAW-BOM`、`COMP-V4-PLATING-SCHEME`、`COMP-V4-ELEMENT-BOM`

```
data_driver_path = 'costing_part_material_bom'    ← 主表
                 = 'mat_bom[bom_type=''ELEMENT'']'  ← 带枚举谓词
                 = 'v_part_plating_scheme'           ← 复合视图

字段:
  - 项次/料号/数量等              BASIC_DATA → costing_part_material_bom.<col>
  - 单价(动态查全局变量)           BASIC_DATA → v_costing_element_price.costing_price
                                    + global_variable_code='ELEM_PRICE'
  - 行小计                         FORMULA, is_subtotal=true (单组件最多 1 个)
```

**关键约束**：
- 字段 path 留空谓词（不写 `[hf_part_no='X']`），靠 ImplicitJoinRewriter 隐式注入
- driver 行需要包含被字段 path 所引用视图的同名列（V103 给 v_part_plating_scheme 加 element_name 别名就是为了这个）
- 跨表查找用全局变量动态 key 模式（V109 + V110 picker UI）

### 模式 B：单行汇总型组件

**典型**：`COMP-V4-PLATING-COST`、`COMP-V4-EXCHANGE-RATE`、SUBTOTAL 类组件

```
data_driver_path = NULL  或  'plating_fee'   ← 单行视图

字段:
  - 输入项                BASIC_DATA → 单行视图的 col
  - 跨组件汇总            FORMULA → component_subtotal token
  - 全局变量              BASIC_DATA + global_variable_code   或   FORMULA + global_variable token
  - 最终值                FORMULA, is_subtotal=true
```

### 模式 C：跨组件 SUBTOTAL 组件

**典型**：`COMP-V4-TOTAL-CNY`（总公式聚合 8 个组件小计 + finished_other 比例）

```
component_type = 'SUBTOTAL'

公式 expression:
  [{type:'component_subtotal', value:'<字段名>', component_code:'<引用组件 code>'}, ...]

注: 引用方组件必须有 is_subtotal=true 字段, 前端 ReadonlyProductCard.tsx
   reduce(comp.rows, computeFormula(comp, stField.name, row)) 跨行 SUM 后传入此 token.
```

## 四、Excel 模板配置模式

### 列结构 5 字段

| 字段 | 含义 | 示例 |
|---|---|---|
| `col_key` | 公式中 `[X]` 引用名（A-Z + 后缀） | `B` / `B_PURE` / `S` |
| `title` | 表头显示 | `材料成本 (行82)` |
| `source_type` | `VARIABLE` 取数 / `FORMULA` 计算 | `VARIABLE` |
| `variable_path` 或 `formula` | 路径 或 公式串 | `v_costing_summary_full.material_cost` 或 `=[B_PURE]+[B_PROC]` |
| `comparison_tag` | 比对维度标签（可空） | `MATERIAL_COST` |
| `hidden` | 是否隐藏列（仍参与公式） | `true` / `false` |

### 隐藏列模式（推荐 V96 起标准做法）

```
B_PURE   VARIABLE  hidden=true   v_costing_summary_full.pure_material_cost
B_PROC   VARIABLE  hidden=true   v_costing_summary_full.incoming_process_fee
B_OTHER  VARIABLE  hidden=true   v_costing_summary_full.incoming_other_fee
B_RECYCLE VARIABLE hidden=true   v_costing_summary_full.recycle_cost
B        FORMULA   hidden=false  =[B_PURE]+[B_PROC]+[B_OTHER]-[B_RECYCLE]   ← 用户看到的"材料成本"
```

**好处**：原始数据列保留可审计 + 用户视图只看公式结果，符合"金额列由公式合成"原则。

### 变量路径三种写法

| 写法 | 示例 | 何时用 |
|---|---|---|
| **legacy 占位符** | `{HF_PART_NO}` `{customer_drawing_no}` | 报价单元数据，前端从 lineItem 直接映射 |
| **BNF 路径** | `v_costing_summary_full.material_cost` `mat_part.unit_weight` | 90% 场景 — 后端 FormulaEngine 求值 |
| **全局变量** | `v_costing_exchange_rate[from_currency='CNY' AND to_currency='USD'].costing_rate` | 通过 「🌐 全局变量」按钮选，自动编译 |

### Excel 公式语法

```
=[col_key1] + [col_key2] - [col_key3]
=([B] + [C] + [D] + [E] + [F])
=[P] / 1000 / [Q]
=[P] * [S]
```

- `[X]` 引用同行其他列的已计算 cellValue
- 不支持 token 类型；想用全局变量 → 把它写成一个 VARIABLE 列，再在 FORMULA 列里 `[那个 col_key]` 引用

## 五、公式 token 完整清单（组件公式专用）

| token type | 用途 | 必填字段 | 示例 |
|---|---|---|---|
| `field` | 引用同行其他字段 | `value` | `{type:'field', value:'电镀面积(cm²)'}` |
| `operator` | 运算符 | `value` | `{type:'operator', value:'*', label:'×'}` |
| `bracket_open` / `bracket_close` | 括号 | - | `{type:'bracket_open', value:'('}` |
| `number` | 数字常量 | `value` | `{type:'number', value:'100000'}` |
| `component_subtotal` | 跨组件小计 | `component_code` | `{type:'component_subtotal', component_code:'COMP-V4-PLATING-SCHEME'}` |
| `product_attribute` | 产品属性 | `attribute_name` | `{type:'product_attribute', attribute_name:'材料'}` |
| `quotation_field` | 报价单级字段（税率等） | `value` | `{type:'quotation_field', value:'tax_rate'}` |
| `path` | BNF 路径直接取 | `path` | `{type:'path', path:'mat_part.unit_weight'}` |
| `global_variable` | 全局变量 | `code`, `key_values` 或 `key_field_refs` | `{type:'global_variable', code:'EXCHANGE_RATE', key_values:{from_currency:'CNY', to_currency:'USD'}, path:'...'}` |

**重要**：`global_variable` token 创建时由前端编译产物 `path` 字段（BNF），运行时 evaluator 走 path resolver — 不要手写。

## 六、运行时数据流速查

```
[前端] LinkedExcelView / ProductCard
   ↓ formulaService.evaluate({expression:'{path}', partNo, customerId})
[后端] FormulaEvaluateResource → FormulaEngine.evaluate
   ↓ resolvePathValue
[ctx] EvaluationContext {dataLoader, partNo, customerId, driverRow, bindings}
   ↓ dataLoader.loadByPath(path, driverRow, partNo, customerId)
   ↓
[Rewriter] ImplicitJoinRewriter.rewriteWithContext
   - 合并 effective driver: {hf_part_no:partNo, customer_id:customerId} ∪ driverRow
   - 查目标视图 information_schema.columns (V112 起空集不缓存)
   - 注入 effective 中目标视图存在 + 原谓词没用过 + 非黑名单 的列
   ↓ rewritten path: v_costing_summary_full[hf_part_no='3100080003'].pure_material_cost
[Compiler] CachedSqlCompiler → SQL
[JDBC] PreparedStatement → PostgreSQL view/table
   ↓ 单行 → 标量值 / 单行多列 → Map / 多行 → List<Map>
[前端] formatPathValue 兜底
   - 单值: 直返
   - List > 1: 「首值（共N项）」  ← 看到此即 ImplicitJoin 没注入, 排查 Rewriter
   - null: 显示 「—」
```

**关键节点排错**：
- 看到「共N项」→ ImplicitJoinRewriter 没注入 → 视图 DDL 后 cache 残留 → 重启 Quarkus
- 看到「—」+ 全行空 → V111 noCostingData 路径正常工作（料号无核价数据）
- 看到「0」铺满 → 视图 COALESCE(0) 错误填充 → 让 NULL 传递（V111）

## 九、快速配置清单（实战用例）

### 用例：给 Excel 模板加一个「能耗成本」维度对比

需求：报价单比对视图增加"能耗成本"差异指标。

| 步骤 | 操作 | 输出 |
|---|---|---|
| 1. 数据源 | 查清 `costing_part_process_cost` 中 `cost_type='ENERGY_DEDICATED'` | 数据已存在 |
| 2. 视图列 | 在 `v_costing_summary_full` 增 `energy_cost = SUM(unit_price WHERE cost_type='ENERGY_DEDICATED')` | 写 V_NN migration（重建视图，CASCADE 重启 Quarkus）|
| 3. Excel 列 | 在「核价 Excel 视图模板（完整公式版）」DRAFT 派生新版，加 `col_key=ENERGY` `source_type=VARIABLE` `variable_path='v_costing_summary_full.energy_cost'` `comparison_tag='ENERGY_COST'` | 列定义齐全 |
| 4. 标签注册 | V_NN+1 INSERT comparison_tag (code='ENERGY_COST', label='能耗成本', group_name='成本明细', tag_sort_order=25) | 比对视图能识别 |
| 5. 触发 | touch java 文件 → Quarkus 热重载 + Flyway 跑新 migration | success=t |
| 6. 验证 | 浏览器强刷报价单 → Excel 视图列 ENERGY 显示数值 → 比对视图"成本明细"组多一行"能耗成本" | UI 正确 |

**预计耗时**：30 分钟（含编译/热重载）。

### 用例：给报价单 Excel 视图复用核价 Excel 视图

参见 V113 实操：核价 Excel 模板 v1.2（30 列）→ 复制到报价单 Excel 模板（QUOTATION 类型）→ status=PUBLISHED + is_default。

```sql
UPDATE costing_template
SET name='报价单Excel视图模板（完整公式版）',
    columns=(SELECT columns FROM costing_template WHERE id='<源模板>'),
    referenced_variables=(SELECT referenced_variables FROM costing_template WHERE id='<源模板>'),

```

## 十、决策点参考（历史决策沉淀）

| 决策 | 选项 | 推荐 | 理由 |
|---|---|---|---|
| 字段从全局变量取还是 BNF 直查 | global_variable / 直接 BNF | **global_variable + global_variable_code 元数据** | UI 显示 🌐 徽章，链路清晰 |
| 静态 key vs 动态 key | 绑定具体值 / 按行查表 | **多行 driver 用动态; 单值（如汇率）用静态** | 多行场景错绑会"每行同值" |
| 公式跨组件 | component_subtotal / SQL 视图聚合 | **component_subtotal 更显式** | 可视化在 token 列表，不依赖 SQL 改动 |
| 隐藏中间列 | 全部显示 / 隐藏中间合成 | **隐藏中间，只露公式列** | UI 简洁，用户看不到"原始数据 + 合成"的冗余 |
| 视图 NULL 还是 COALESCE 0 | NULL 传递 / 填 0 兜底 | **NULL 传递（V111）** | 让前端能识别"无数据 vs 真 0"，整行清空 |
| 比对维度从模板列还是 SQL 字段 | 列加 tag / SQL 字段加 tag | **列加 comparison_tag** | 与 Excel 视图列定义对齐，单一真相源 |

## 多行数据展示问题专题（高频 / 必读）

> 多行 driver 展开链路在"配置 → 求值 → 渲染"3 段任意一段断裂都会显示成 `X (共N项)` / `{字段:null} (共N项)`。这族问题踩过 5 次以上，本专题作为**配置者必读**。完整反模式见 [`反模式.md` AP-22](./反模式.md)。

### 一、链路全景（4 个收窄关卡）

```
SQL 物理表 / 视图           ← 关卡 1: WHERE 谓词收窄
        ↓
ImplicitJoinRewriter        ← 关卡 2: 隐式注入 hf_part_no / customer_id / driver row 同名列
        ↓
useDriverExpansions hook    ← 关卡 3: driver 展开后, 每行单独 evaluate, basicDataValues 单值
        ↓
ProductCard 渲染 cell       ← 关卡 4: BASIC_DATA cell "四级回退链", 优先 row[fieldKey] > globalPathCache
        ↓
formatPathValue             ← 兜底: list.length > 1 → "首值（共N项）"
```

任意一关卡断裂 → 多行 array 漏到末层 → "（共N项）"。

### 二、配置时强制核对清单（逐项打勾）

| # | 检查项 | 通过条件 |
|---|---|---|
| 1 | 视图 / 表是否暴露 `hf_part_no` 列? | `\d+ <view>` 看到该列 |
| 2 | 跨表隐式 JOIN 列名是否对齐? | driver 行有 `element_name`, 目标视图也有 `element_name` 列（V109 的 element_code AS element_name 别名模式） |
| 3 | driver 行是否含字段路径目标表的业务键? | mat_fee multi-row driver 含 seq_no/dim_element_name 等 → 字段 path 自动按这些列收窄 |
| 4 | 字段 path 是否需要带显式谓词? | 仅当 driver 行不含目标列业务键时才写 `[X='Y']` 静态谓词 |
| 5 | 视图 NULL vs 0 语义? | 无业务数据 → 视图返 NULL（不要 COALESCE 0），让前端识别"无数据行" |
| 6 | comparison_tag code 在 comparison_tag 表注册? | `SELECT * FROM comparison_tag WHERE code IN (...)` 全部命中 |
| 7 | 视图 DROP CASCADE / 重建后是否重启 Quarkus? | touch java 文件 + 401 响应验证 |
| 8 | BASIC_DATA cell 的渲染分支有"四级回退链"? | driver 展开 → row[fieldKey] → globalPathCache → 加载占位 |

### 三、症状 → 根因 5 秒决策树

```
看到 "(共N项)" 后缀
   ↓
是 Excel 视图列? ─── 是 ──→ 类别 A: SQL 隐式 JOIN 失效
                          (1) 用 SQL 验证 SELECT col FROM view WHERE hf_part_no='X' 是否单行
                          (2) 单行 → 重启 Quarkus 清进程缓存（V112 安全网）
                          (3) 多行 → 视图本身缺收窄键, 加 PIVOT 包宽表
   ↓
是产品卡片视图 cell? ── 是 ──→ 类别 B: 渲染层 fallback 漏读 row
                              F12: 看 comp.rows.length 是否 > 1 + row[fieldKey] 是否标量
                              是 → BASIC_DATA cell 缺优先级 2 (V118 模板) — 立刻补
                              否 → 进入类别 A 排查（数据真的多行）

显示数值都是 0 但语义是"无数据"? ──→ 类别 C: 视图 COALESCE(0) 遮蔽
```

## :spineKeys 复合键跨页签过滤（核价）

> **2026-06-06** 新增。仅适用于核价（COSTING）模板 / 组件的 SQL 视图 `sql_template`。

### 用途

核价卡片以"生产料号"为根，经 `BomClosureService` 算出整棵 BOM 闭包，产出三元组集合 `(子件料号, 父件料号, 子件自身当前 BOM 版本)`。消费方页签的 SQL 视图用 `:spineKeys(...)` 宏可精确按这组三元组过滤，解决单列 `:hfPartNos` 无法区分"同一料号在不同父件/版本下多 occurrence"的问题。

### 语法

```sql
WHERE :spineKeys(<子件料号列>, <父料号列>, <版本列>)
```

三个实参按位对应：第 1 → 子件料号、第 2 → 父件料号、第 3 → 子件自身当前版本。实参可以是列名或表达式（如 `nullif(trim(t.x),'')` / `t.col::text`）。

### 展开形式

框架自动展开为 NULL-safe `EXISTS + unnest + IS NOT DISTINCT FROM`：

```sql
EXISTS (
  SELECT 1
  FROM unnest(:__skP::text[], :__skPP::text[], :__skV::text[]) AS k(p, pp, v)
  WHERE (<子件料号列>) IS NOT DISTINCT FROM k.p
    AND (<父料号列>)   IS NOT DISTINCT FROM k.pp
    AND (<版本列>)     IS NOT DISTINCT FROM k.v
)
```

- **NULL-safe**：根节点父料号 NULL、叶子节点版本 NULL 均可命中对应 NULL 行，不会误过滤。
- **无超长 IN**：无论 BOM 树有多少节点，始终只占 3 个数组绑定参数，绕开 PG 65535 参数上限。
- **空 spine → 0 行**：不产生 `IN ()` 语法报错，也不误匹配全表。

### 数据来源与上下文限制

三元组由框架从卡片 BOM 闭包派生并环境注入（类比 `:hfPartNos`）：

- **核价卡片展开上下文**（`CardSnapshotService.expandTemplateDriverBaseRows`）：三数组有值，过滤生效。
- **其他上下文**（报价单渲染、standalone dry-run 等）：三数组为空 → `EXISTS` 恒 false → 匹配 0 行（安全降级）。

### 完整示例

```sql
-- 核价模板 sql_template：按 BOM 树三元组过滤，避免跨卡片串数据
SELECT
    t.material_no,
    t.parent_no,
    t.bom_version,
    t.unit_cost,
    t.quantity
FROM v_costing_bom_items t
WHERE :spineKeys(t.material_no, t.parent_no, t.bom_version)
  AND t.customer_no = :customerCode
```

### 保留前缀与限制

- `:__sk*` 前缀为框架保留，**禁止**在 sql_template 中自定义 `:__skP` / `:__skPP` / `:__skV` 等占位符。
- 实参不支持含括号外层逗号的 SQL 字符串字面量（`'a,b'` 这类会误切实参分隔）。
- 同一 sql_template 可多次使用 `:spineKeys(...)`，共享同一组三数组（不重复绑定）。
- 保存期 dry-run 展开为校验形（`ARRAY[]::text[]` 字面量），`spineKeys` / `__sk*` 不进 `required_variables`。

### 版本语义说明（2026-06-06 变更）

核价 BOM 树"版本"列口径已统一为**子件自身当前 is_current BOM 版本**（非边版本/父件 BOM 版本）：

- 每个节点取 `material_bom_item WHERE material_no = 本节点 AND is_current = true` 的 `bom_version`。
- 叶子节点（自身无 BOM 记录）→ 版本为 NULL（树上显示空白）。
- `:spineKeys` 的第 3 实参与此口径对齐，才能正确命中对应行。
