# 选配产品 V6 迁移 — 受影响业务逻辑诊断方案

> **结论速览（一句话）**：选配的 **"落库写入端" 仍停留在 V44 老表**（`mat_part` / `mat_bom` / `mat_process` / `mat_composite_process`），而 **"搜索 / 取数 / 渲染读取端" 已全面切到 V6 新表**（`material_master` / `material_bom_item` / `element_bom_item` / `process_master`）。两端数据互不可见 → 选配在多个环节断链。
>
> 之前为什么"正确"：旧链路 **写 V44、读 V44** 端到端自洽。AP-53 把渲染/搜索切 V6 后，落库没跟上 → **写 V44、读 V6** → 断裂。这是 AP-53 续 5 同款"迁移半拉子"反模式，但发生在**整条选配生命周期**而非单点。
>
> **状态**：诊断 → **Route A Phase 1 已实施**（2026-05-28，AP-53 续 6 / V266）：料号+元素+子件落库迁 V6（渲染基线零改），修复 B-2（提交不崩）+ B-4（指纹复用过渡期正常）+ B-1 的材质/元素/子件 Tab。**工序列表 Tab + 组合工艺 Tab 仍空 = Phase 2**（V6 无承载结构，需新建业务表 + mirror UNION，走 architect）。详见 RECORD.md [2026-05-28] + §6。
> **立项日期**：2026-05-28。**关联**：`docs/反模式.md` AP-53（续 2/续 5/续 6）、`docs/选配与基础数据料号材质关系.md`、`docs/三大核心模块基线.md`。

## 2. 逐环节诊断（独立产品 SIMPLE + 组合产品 COMPOSITE）

### Step1 搜料号 —— ⚠️ 搜索集骤减 + 选 V6-only 料号埋雷
- **现状**：`searchParts` 查 V6 `material_master`，仅 **5** 行（`mat_part` 有 255）。
- **断点**：用户能搜到的料号从 255 → 5。绝大多数历史料号搜不到。
- **埋雷**：`material_master` 的 5 行里，`TEST-SIMPLE-001/002` **不在** `mat_part`（仅 3100080003/3120012574/3120012575 重叠）。选中 V6-only 料号 → Step5 校验崩（见下）。
- **证据**：`SELECT COUNT(*) material_master=5 / mat_part=255`；overlap 查询 TEST-SIMPLE-001/002 → `in_mat_part=f`。

### Step2 材质取数 —— ✅ 已修（续 5）
- `getForExistingPart` 已迁 V6，绑定字典派/BOM 派均走 V6。此环节本身 OK（前提是料号在 material_master）。

### Step3 工序字典 —— ✅ 字典有数据
- 查 V6 `process`（41 行），Step3 列表正常。**注意**：这是"工序字典"，与"工序落库 `mat_process`"是两回事。

### Step4 组合工艺字典 —— ✅ 字典有数据
- 查 V6 `composite_process_def`（6 行），Step4 列表正常。

### Step5 提交 —— ❌❌ 主断点（校验 + 落库全在 V44）
- **5a 存在性校验崩**（existing 路径）：`resolvePart:184` `SELECT … FROM mat_part WHERE part_no=:p`。若 Step1 选的是 V6-only 料号（不在 mat_part）→ `rows.isEmpty()` → **抛 "料号不存在"，提交直接失败**。
- **5b 指纹复用失效**（custom 路径）：`lookupHfByFingerprint:111` 查 `mat_part.config_fingerprint`。`material_master` 无此列；且新建料号只落 V44 → 同配置反复新建，且与 V6 搜索/渲染脱节。
- **5c 落库全 V44**：新料号 + 元素 BOM + 工序 + 组合工艺全部 INSERT 到 `mat_part`/`mat_bom`/`mat_process`/`mat_composite_process`，**完全不写** V6 `material_master`/`material_bom_item`/`element_bom_item`。

### 渲染（提交后报价单 Tab）—— ❌❌❌ 核心断链：配置完成但 Tab 全空
- **现状**：报价单 Tab 通过组件 `data_driver_path = $<mirror_view>` 走 driver 展开，mirror 视图**全部读 V6**。
- **断点**：Step5 把数据写进 V44，V6 mirror 视图里没有刚写的料号 → **材质/工序/元素含量/子配件/组合工艺 Tab 全部空白**。
- **SIMPLE**：元素含量读 `element_bom_item`（V6）/ 配置写 `mat_bom`(ELEMENT, V44) → 空；工序读 `composite_child_processes_mirror`（V6 material_bom_item）→ 空。
- **COMPOSITE**：子件读 `zcj_bom`/`material_bom_item`(characteristic='ASSEMBLY', V6) / 配置写 `mat_bom`(ASSEMBLY, V44) → 空；组合工艺读 V6 mirror / 配置写 `mat_composite_process`(V44) → 空。
- **维度错配加成**：mirror 按 `customer_no`(customer.code) 过滤；配置写 `mat_process.customer_id`(UUID) —— 即便未来补镜像，也要解决 customer_no↔customer_id 维度对齐。
- **证据**：5 个 mirror 视图 `sql_template` 全部 `FROM element_bom_item / material_bom_item / material_master / process_master`（见 §7 附录）。

## 4. 为什么"独立产品和组合产品都不行"
- **共因 B-1**：两类产品的渲染都走 V6 mirror 视图，落库都在 V44 → 都渲染空。
- **SIMPLE 额外**：元素含量 + 工序 Tab 空。
- **COMPOSITE 额外**：子件清单 + 组合工艺 Tab 空（assembly 行 / 组合工艺只在 V44）。
- 唯一"看起来还行"的特例：existing 路径选**同时存在于 V44+V6 的重叠料号**（3120012574/575）→ 校验过 + V6 本就有导入数据 → 能渲染；但这是**复用已导入料号**，不是真正"配置新品"。

## 6. 修复方向（候选，需选型后单独立项）

> 核心目标：让"落库写入端"与"读取/渲染端"落到**同一套表**。三条路线：

### 路线 A（推荐）：选配落库改写 V6 表
- `ConfigureProductService` 全面改写：`mat_part→material_master`、`mat_bom(ELEMENT)→element_bom_item`、`mat_bom(ASSEMBLY)→material_bom_item(characteristic='ASSEMBLY')`、`mat_process→`（V6 工序业务表，**当前缺**，需新建或在 material_bom_item.operation_no 维度承载）、`mat_composite_process→`（V6 缺对应业务表，需新建）。
- 校验/指纹改查 material_master（需给 material_master 加 `config_fingerprint` 列，镜像 V44 机制）。
- 必须解决 **customer_no(code) 维度**：落库时写 mirror 视图过滤所需的 `customer_no`。
- **缺口**：V6 目前**没有**"工序业务数据表"与"组合工艺业务数据表"（只有字典 `process`/`composite_process_def`）。路线 A 需先补这两张 V6 业务表 + mirror 视图对接。
- 优点：彻底，端到端 V6。缺点：工作量最大，涉及新建 V6 业务表 + 数据模型设计（建议走 architect）。

### 路线 B（过渡）：渲染端临时回读 V44 / 双读
- mirror 视图改成 V6 UNION V44，或选配产生的料号单独走 V44 driver。
- 优点：快速恢复选配可用。缺点：违背 AP-53 "禁查 V44 老表"，技术债，且与已导入 V6 数据口径不一致。

### 路线 C（最小可用）：选配落库"双写" V44 + V6
- configure 在写 V44 的同时，把同一份数据镜像写入 V6（material_master / material_bom_item / element_bom_item + customer_no），渲染端不变。
- 优点：渲染端零改、风险隔离在落库。缺点：双写一致性维护成本；仍需补 V6 工序/组合工艺业务表（否则工序/组合工艺 Tab 仍空）。

> **共性前置依赖**（任何路线都要先解决）：
> 1. V6 **缺"工序业务表"+"组合工艺业务表"**（只有字典）。这是 COMPOSITE/SIMPLE 工序渲染的硬缺口。
> 2. V6 基础数据导入未补齐（material_master 仅 5 行）—— 影响 existing 路径可选范围。
> 3. customer_no(code) ↔ customer_id(uuid) 维度统一。
