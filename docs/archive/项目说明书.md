## 1. 项目简介

CPQ（**Configure 配置 / Price 定价 / Quote 报价**）是面向制造业 / 工业元器件行业的报价管理系统。它把过去散落在多份 Excel 中的料号 BOM、工艺费、电镀方案、汇率、元素价格等"算价逻辑"统一沉淀到一个配置中心，确保**每次出价有据可查、版本可追溯、数据来源可还原**。

### 1.1 系统能做什么

- **客户基础数据**：客户 / 联系人 / 客户料号 → 宏丰料号映射，统一管理。
- **报价模板与核价模板**：通过组件 + 公式可视化拖拽，无需改代码即可调整算价口径。
- **报价单 5 步向导**：选客户 → 添加产品 → 优惠策略 → 交易条款 → 提交审批，全流程留痕。
- **Excel 一键导入**：销售上传客户提供的 14 个 Sheet 的 .xlsx，系统自动差异检测 + 字段级冲突决策。
- **审计与漂移追溯**：提交时数据快照冻结；引用版本变化自动弹"黄色横幅"提醒。

### 1.2 三种典型用户与日常工作

| 角色 | 典型一天 |
|---|---|
| 销售代表 (SALES_REP) | 收到客户询价 → 上传 Excel → 处理差异/冲突 → 检查公式与漂移 → 提交报价单 |
| 销售经理 (SALES_MANAGER) | 审批待办 → 看数据来源 / 历史版本 → 调整毛利率门槛 |
| 系统管理员 (SYSTEM_ADMIN) | 维护配置中心（全局变量、组件、模板、Excel 模板）、维护元素价格中心、监控锁与变更日志 |

## 2. 总体架构与四大模块的关系

### 2.1 全景图

```
┌──────────────────────────────────────────────────────────────────────┐
│                            配置中心（唯一真理源）                       │
│                                                                        │
│   ① 全局变量           ② 组件管理            ③ 模板管理                │
│   (轻量配置 KV)   →    (字段 + 公式可重用)   (拖拽组装 + 版本)         │
│                                                                        │
└────────────┬─────────────────────┬──────────────────────┬─────────────┘
             ↓                     ↓                      ↓
       报价单/核价单渲染      Excel 视图渲染           Excel 导入/导出
                       (业务层零硬编码，全部按配置驱动)
                                      │
                                      ▼
┌──────────────────────────────────────────────────────────────────────┐
│                            ④ 报价单管理                                │
│                                                                        │
│  5 步向导：客户 → 产品（含 Excel 视图）→ 优惠 → 条款 → 提交审批         │
│  状态机：DRAFT → SUBMITTED → APPROVED → EXPIRED/CLOSED                 │
└──────────────────────────────────────────────────────────────────────┘
```

### 2.2 四大模块各自的定位

| 模块 | 解决的问题 | 谁来维护 | 频率 |
|---|---|---|---|
| **全局变量** | 默认值 / 阈值 / 系数 / 折扣率（业务无关、轻量） | 系统管理员 / PRICING_MANAGER | 偶尔（业务策略调整时） |
| **组件管理** | 可重用的字段表（投料 / 回料 / 加工 / 电镀…），每个字段定义来源与算法 | 销售经理 / 管理员 | 模板结构调整时 |
| **模板管理** | 把组件拼成产品卡片 + 公式定义算价口径 + 关联 Excel 视图列结构 | 销售经理 / 管理员 | 模板版本演进时 |
| **报价单管理** | 销售对外出价的载体，承载基础数据导入 + 公式自动计算 + 审批 + 输出 | 销售代表（建/填）+ 销售经理（审批） | 每天 |

> **关键认知**：前三者属于"上游配置"，最后一个"报价单"是消费这些配置的运行时。**配置一旦调整，所有引用它的报价单立即生效**（H1 自动同步 snapshot 机制）。

## 3. 配置中心 — 三层模型与"唯一真理源"原则

### 3.1 三层模型

```
┌────────────────────────────────────────────────────────────┐
│              配置中心（唯一真理源）                          │
├────────────────────────────────────────────────────────────┤
│  全局变量定义 + 值                                          │
│    (global_variable_definition + global_variable_value)    │
│  组件配置                                                   │
│    (component.fields JSON)                                 │
│  模板配置                                                   │
│    (template.components_snapshot JSON)                     │
│  公式定义                                                   │
│    (formula_tokens)                                        │
└────────────────────────────────────────────────────────────┘
                          ↓
       ┌──────────────────┼──────────────────┐
       ↓                  ↓                  ↓
   报价单渲染          核价单渲染         Excel 视图渲染
   (零业务逻辑)         (零业务逻辑)       (零业务逻辑)
```

**核心原则**：业务渲染层**不做任何字段语义推断、不硬编码字段映射、不二次解析配置**。配置层定义→运行时按定义渲染。

### 3.2 字段类型矩阵（贯穿组件 / 模板 / 报价单）

| field_type | 数据来源 | 渲染形态 | 关键配置 |
|---|---|---|---|
| `FIXED_VALUE` | 字面量 | 只读 | `content` |
| `INPUT_TEXT` | 用户输入 | 文本框 | (可选 `default_source`) |
| `INPUT_NUMBER` | 用户输入 | 数字框 | `content` 静态兜底 + `default_source` |
| `BASIC_DATA` | BNF 路径查物理表/视图 | 只读 | `basic_data_path` |
| `DATA_SOURCE` | 多类型数据源 (4 种) | 只读 / 下拉 | `datasource_binding.type` |
| `FORMULA` | 公式引擎计算 | 只读 | `formula_tokens` |

### 3.3 三级兜底链（INPUT_NUMBER / INPUT_TEXT 取值顺序）

```
用户输入 row[key]
   → default_source.GLOBAL_VARIABLE → basicDataValues['@gvar:CODE']
   → default_source.BNF_PATH        → basicDataValues[{path}]
   → field.content                  → 静态字面量兜底
   → 0
```

这个链路决定了"报价单上每一个空白单元格默认填什么"。

## 4. 全局变量配置

### 4.1 全局变量是什么

> 「全局变量」= **轻量、业务无关的配置项**（默认值 / 阈值 / 系数 / 折扣率 / 默认成材率 / 默认毛利率…）

它**不是**：
- ❌ 业务主数据（核价价格 → 那是核价模块的事）
- ❌ 用户操作记录（订单状态 → 业务流转）
- ❌ 计算逻辑（毛利计算公式 → 公式定义）

### 4.2 两种形态

| 形态 | 用途 | 示例 |
|---|---|---|
| `LOOKUP_TABLE` | 按 key 查表 | 工序默认成材率 — 按 `process_code` 查 |
| `SCALAR` | 单一值 | 系统默认毛利率 = 20% |

### 4.3 两种可见性

| visibility | 全局变量页 | Picker 选择器 | 维护数据 |
|---|---|---|---|
| `PUBLIC` | ✅ 默认显示 | ✅ 显示 | ✅ 可编辑 |
| `COSTING_INTERNAL` | ❌ 默认隐藏（toggle 可查看） | ✅ 显示 + 「核价」紫色 tag | ❌ 禁用，提示去核价模块 |

> 核价 3 张（ELEM_PRICE / MAT_PRICE / EXCHANGE_RATE）使用 `COSTING_INTERNAL` 隔离，**只能在核价模块维护，不能在"全局变量"页改**。

### 4.4 新建全局变量的 SOP（纯 UI，90% 场景）

1. 打开「**配置中心 → 全局变量**」→ 点「**新建变量**」
2. 填写表单：
   - `code`（大写英文+数字+下划线，如 `PROCESS_DEFAULT_YIELD`）
   - `name`（中文，如「工序默认成材率」）
   - `变量类型`（`LOOKUP_TABLE` 或 `SCALAR`）
   - `key 列名`（LOOKUP_TABLE 必填，如 `process_code`；SCALAR 留空）
   - `单位`（选填，如 `%`、`CNY/KG`）
3. 创建后行内点「**维护数据**」抽屉添加 (key, value) 行
4. 去「组件管理 → 字段配置」给目标字段配 `default_source` 引用此变量

零 Flyway、零 Java、立即生效。

### 4.5 维护数据

- 列表里找到变量 → 点「维护数据」
- 表格列出所有 `(key, value)` 行
- 「新增」/「编辑」/「删除」均走二次确认 Modal
- 保存后**立刻生效**（`GlobalVariableService.upsertEntry` 内置缓存失效闭环 `invalidateDependentCaches`）

### 4.6 组件字段如何引用全局变量

```jsonc
// 场景 A：作为 INPUT_NUMBER 的默认值（最常用）
{
  "name": "成材率",
  "field_type": "INPUT_NUMBER",
  "content": "100",                          // DB 查不到时的静态兜底
  "default_source": {
    "type": "GLOBAL_VARIABLE",
    "code": "PROCESS_DEFAULT_YIELD",
    "key_field_refs": {}                     // 留空 = 同名映射(key 列 process_code → driver row.process_code)
  }
}

// 场景 B：作为公式 token 直接参与计算
{
  "name": "本工序成本",
  "field_type": "FORMULA",
  "formula_tokens": [
    { "type": "global_variable",
      "code": "PROCESS_DEFAULT_PRICE",
      "key_field_refs": { "process_code": "工序代码" } },
    { "type": "operator", "value": "*" },
    { "type": "field", "value": "数量" }
  ]
}
```

报价单上 placeholder 会显示「**默认 15 · PROCESS_DEFAULT_YIELD**」，用户输入即覆盖（覆盖值仅活在当前草稿）。

### 4.7 配置自检 checklist

```
□ global_variable_definition 已注册 (visibility=PUBLIC, value_source_type=KV_TABLE)
□ global_variable_value 至少 1 行（按 distinct key 全 seed,避免漏）
□ component.fields JSON 含 default_source 或 formula_tokens
□ template.components_snapshot 已同步（组件保存会自动同步,H1 机制）
□ 强刷后报价单 placeholder 显示「默认 X · CODE」(有后缀 = DB 命中)
□ 「全局变量」页能看到新变量并可「维护数据」
```

## 5. 组件管理

### 5.1 组件的定位

组件是**字段的可重用集合**。一个组件 = 一组字段（投料 BOM、加工费、电镀方案、商务条款…）+ 它的数据驱动路径（driver path）。模板拖拽组件组装成产品卡片。

### 5.2 字段配置表（FieldConfigTable）逐列说明

| 列 | 说明 |
|---|---|
| 字段名 | `name`，也是公式 token 的引用 key |
| 字段类型 | 6 种 `field_type`（见 [3.2 字段类型矩阵](#32-字段类型矩阵贯穿组件--模板--报价单)） |
| 内容/配置 | 按 field_type 动态切换 UI（固定值 / 路径徽章 / 默认值来源…） |
| 金额 | `is_amount=true` 时按货币格式化 |
| 小计 | `is_subtotal` ，公式聚合用；**组件内最多 1 个** |
| 备注 | `notes`，配置者备忘，不参与渲染 |
| 排序 / 删除 | UI 操作 |

### 5.3 6 种字段类型详解

#### `FIXED_VALUE` — 字面量

直接输入 `content`。常用于：标签 / 单位字符串 / 报价说明文字。
**坑**：`is_amount=true` 时 `content` 必须是数字。

#### `INPUT_TEXT` / `INPUT_NUMBER` — 用户输入

报价单可编辑单元格，自动走[三级兜底链](#33-三级兜底链input_number--input_text-取值顺序)。

字段配置「内容/配置」列：
- 左侧 `content` 输入框：静态兜底值
- 右侧「**+ 默认值来源**」按钮 → 打开 DefaultSourceEditor 抽屉
  - 选 type（动态从后端拉取，不硬编码）
  - 填配置 → 用「测试 driverRow JSON」+「测试解析」按钮实时验证
  - 保存

#### `BASIC_DATA` — BNF 路径查表

只读字段，按 BNF 路径解析（如 `mat_part.unit_weight`），后端 `ImplicitJoinRewriter` 自动隐式 JOIN driver row。

「内容/配置」列两个入口：
- **配置物理表路径** → PathPickerDrawer
- **🌐 全局变量** → GlobalVariablePickerDrawer

**职责单一原则**：BASIC_DATA 只查不算。需要计算用 FORMULA。不要在 path 里写 `mat_part.weight * mat_bom.pct`，要拆成两个 path token 放到 FORMULA。

#### `DATA_SOURCE` — 4 种数据源

| type | 配置字段 | 解析行为 | 适用场景 |
|---|---|---|---|
| `DATABASE_QUERY` | `datasource_id` + `params` | `DataSourceService.execute` | 数据源管理页注册的查询 |
| `GLOBAL_VARIABLE` | `global_variable_code` + `key_field_refs` | 按 driver row 解 | 引用配置中心变量值 |
| `BNF_PATH` | `bnf_path` | `DataLoader.loadByPath` | 与 BASIC_DATA 同源 |
| `HTTP_API` | `api_config.{url_template, response_path, auth_token_env}` | 外部 REST 调用 | 集成外部系统（**默认关闭**） |

> HTTP_API 必须在 `application.properties` 配 `cpq.http-api.allowed-hosts=api.example.com`，并通过环境变量传 Token（详见 `docs/HTTP_API_安全配置.md`）。

#### `FORMULA` — 公式引擎计算

公式 token 在模板编辑器拖拽配置，当前支持 11 种：

| token | 引用对象 | 必填字段 |
|---|---|---|
| `field` | 同行其他字段 | `value` |
| `number` | 数字字面量 | `value` |
| `operator` | `+ - * /` | `value` |
| `bracket_open` / `bracket_close` | 括号 | — |
| `component_subtotal` | 跨组件小计 | `component_code` 或 `tab_name` |
| `product_attribute` | 产品属性 | `attribute_name` |
| `quotation_field` | 报价单字段 | `value` |
| `path` | BNF 路径标量 | `path` |
| `global_variable` | 全局变量 | `code` + `key_values` / `key_field_refs` |
| `previous_row_subtotal` | driver 上一行 subtotal（工序累加） | `fallback_component_code` |
| `datasource_field` | 同行 DATA_SOURCE 字段解析值 | `name` |

**示例：工序累加公式**

```json
{
  "name": "小计",
  "field_type": "FORMULA",
  "is_amount": true,
  "is_subtotal": true,
  "formula_tokens": [
    { "type": "previous_row_subtotal",
      "fallback_component_code": "COMP-CFG-ELEMENT-BOM" },
    { "type": "operator", "value": "/" },
    { "type": "field", "value": "成材率" },
    { "type": "operator", "value": "*" },
    { "type": "number", "value": "100" },
    { "type": "operator", "value": "+" },
    { "type": "field", "value": "单价" }
  ]
}
```

含义：`上行小计 / 成材率 × 100 + 单价`，第 0 行兜底取跨组件 fallback 小计。

### 5.4 组件保存的自动同步机制（H1）

`ComponentService.update` 成功后**自动**调 `TemplateService.refreshSnapshotsByComponent`，**所有引用该组件的 `template.components_snapshot` 立即同步刷新**。

**含义**：组件改字段后**不需要**手工 SQL 改 template snapshot — 报价单立即看到新配置。

例外：如果有人手工 `UPDATE component SET fields = ...` 绕过 service 层，需调 `POST /api/cpq/components/{id}/refresh-template-snapshots`（PRICING_MANAGER+）或批量 `POST /api/cpq/config-center/refresh-all-snapshots`（SYSTEM_ADMIN）。

### 5.5 字段类型切换的自动清理矩阵

| 切换前 → 切换后 | 自动清理 | 保留 |
|---|---|---|
| INPUT_NUMBER → BASIC_DATA | default_source / content | name / notes |
| INPUT_NUMBER → DATA_SOURCE | default_source / content | name / notes |
| BASIC_DATA → DATA_SOURCE | basic_data_path / global_variable_code | name / notes |
| DATA_SOURCE 内 type 切换 | 切走类型的配置 | 公用字段 |
| 任何 → FORMULA | 数据来源相关字段 | name / notes / is_amount / is_subtotal |

### 5.6 组件提交前自检 checklist

```
□ field_type 在 6 种枚举内
□ 组件内 is_subtotal=true 至多 1 个
□ DATA_SOURCE 字段按 type 含必填字段
□ INPUT_NUMBER/TEXT 关键字段配 content 静态兜底
□ FORMULA token 引用的 field name 在同组件 fields 列表里存在
□ FORMULA token 引用 DATA_SOURCE 字段必须用 datasource_field type
□ BASIC_DATA 字段不要写表达式
□ HTTP_API 类型必须在 application.properties 配白名单
□ 强刷浏览器验证报价单渲染正常
```

## 6. 模板管理

### 6.1 模板的三类

| 模板类型 | 表 | 用途 | 绑定到报价单的字段 |
|---|---|---|---|
| **报价模板** | `template` (template_kind='QUOTATION') | 报价单产品卡片视图 + Excel 视图源 | `quotation.customer_template_id` |
| **核价模板** | `template` (template_kind='COSTING') | 核价单产品卡片视图 + Excel 视图源 | `quotation.costing_card_template_id` |
| **Excel 模板** | `costing_template` | Excel 视图渲染的列结构 | 通过 `linked_template_id` 反查 |

### 6.2 模板状态机

```
DRAFT  ──发布──►  PUBLISHED  ──归档──►  ARCHIVED
  ▲                  │
  │                  │ 派生新草稿
  └──────────────────┘
```

- `PUBLISHED` / `ARCHIVED` **不可改** → 列表页点「派生新草稿」复制一份，version 自增（v1.0 → v1.1）继续编辑。
- 通过 `template_series_id` / `costing_template.series_id` 把"同一业务用途的多个版本"串成一条 series；UI 默认只显示 series 内最新一份。

### 6.3 模板编辑器的核心动作

1. **拖入组件**：从左侧组件库拖到画布，按业务习惯排列 Tab。
2. **公式配置**：FORMULA 字段点开公式编辑器，拖拽 token 拼算式。
3. **产品属性绑定**：模板上可声明产品级属性（如：板厚、是否电镀），向导第 2 步填值后驱动组件展开。
4. **保存草稿 → 发布**：发布时把当时的组件 fields/formulas 写入 `template.components_snapshot`，作为"模板版本快照"。

> 单组件后续修改会通过 H1 机制**自动同步** snapshot — 不需要重新发布模板，但模板版本号语义保持不变。

### 6.4 Excel 模板（核价单 / 报价单 Excel 视图）

Excel 模板独立于 QUOTATION/COSTING 模板，描述的是**Excel 视图的列结构**。每列定义如下：

```jsonc
{
  "col_key": "material_cost",
  "title": "材料成本",
  "source_type": "VARIABLE",          // VARIABLE | FORMULA
  "variable_path": "v_costing_summary_full.material_cost",
  // source_type=FORMULA 时:
  //   "formula": "[材料成本] + [加工费]"   ← [X] 引用同行其他列
  //                                          { code } 引用全局变量
  "comparison_tag": "MATERIAL_COST"   // 比对视图分组标签
}
```

`source_type` 两种：
- **VARIABLE**：取数列。`variable_path` 可填 ①BNF 路径（`v_costing_summary_full.material_cost`）②lineItem 简写（`{hf_part_no}`）③显式谓词路径
- **FORMULA**：公式列。`formula` 用 `[X]` 引用同行其他列、`{code}` 引用全局变量

**隐式 JOIN**：后端 `ImplicitJoinRewriter` 自动注入 `hf_part_no = '<lineItem.productPartNo>'` 与 `customer_id = '<报价单.customerId>'`（目标表/视图含相应列时）。

**绑定关系**：`costing_template.linked_template_id` 反查到对应 QUOTATION/COSTING 模板。同一 `linked_template_id` 内只能 1 份 PUBLISHED 默认 Excel 模板（`is_default=true`）。

### 6.5 模板提交前自检 checklist

```
□ template_kind 正确（QUOTATION / COSTING）
□ components_snapshot 含所有需要的组件
□ FORMULA 字段引用的字段名在同组件存在
□ Excel 模板 columns JSON 的 variable_path 在 basic_data_config 中可解
□ 默认 Excel 模板有且仅一份（同 linked_template_id 下 is_default=true）
□ 发布后用「数据来源 → 健康检查」验证 components_total / templates_published
```

## 7. 报价单管理与 5 步向导

### 7.1 报价单是什么

> 报价单（Quotation）是销售对外的"价格清单"，承载 客户 / 产品 / 数量 / 单价 / 折扣 / 税费 / 币种 / 有效期，最终输出 PDF / Excel 给客户。
> 它与核价单（Costing Summary）通过 `quotation_line_item.costing_summary_id` 软关联（删核价不影响报价）。

### 7.2 状态机

```
DRAFT  ──提交──►  SUBMITTED  ──销售经理审批──►  APPROVED
  ▲                  │                            │
  │ 驳回              │                            ▼
  └──────────────────┘                       EXPIRED / CLOSED
```

提交时把以下字段冻结进 `quotation_line_item.*_snapshot`：
- `product_part_no_snapshot` / `product_name_snapshot`
- `unit_price_snapshot` / `discount_rate_snapshot`
- `formula_used_snapshot` / `data_source_snapshot`

**目的**：基础数据（料号、价格、汇率）后续可能变；快照保证半年后看回这份报价依然能复现当时的算法。

### 7.3 5 步向导（QuotationWizard）

| 步 | 名称 | 主要动作 | 关键产出 |
|---|---|---|---|
| 1 | 选择客户 | 选客户、联系人、报价模板 / 核价模板 / 产品分类 | `quotation.customer_id / customer_template_id / costing_card_template_id` |
| 2 | **添加产品** | 添加产品行；填属性；自动加载组件 / 公式；可视化报价、核价、Excel 视图 | `quotation_line_item[]` |
| 3 | 优惠策略 | 应用客户级折扣 / 价格调整 / 阶梯折扣 | `system_discount_rate / pricing_strategy_id` |
| 4 | 交易条款 | 付款条件 / 交期 / 有效期 / 币种 / 备注 | `payment_terms / expiry_date / base_currency` |
| 5 | 提交审批 | 冻结快照 → DRAFT → SUBMITTED | `*_snapshot` 列落盘 |

### 7.4 第二步「添加产品」的双轴视图

第二步页面（`QuotationStep2.tsx`）顶部有 **mainTab × viewType** 双轴切换：

|  | 产品卡片视图 | Excel 视图 | 比对视图 |
|---|---|---|---|
| **报价单 Tab** | 按报价模板的组件渲染卡片 | 按 `customer_template_id` 反查 `costing_template` 渲染表格 | 同模板/同产品的多份报价历史并排比对 |
| **核价单 Tab** | 按核价模板的组件渲染卡片 | 按 `costing_card_template_id` 反查 `costing_template` 渲染 | — |

> **关键**：报价单与核价单**共用同一份 `lineItems` 内存对象**，但渲染时按各自模板的 componentId 集合做白名单过滤。编辑回写走"updater 跑在视图态 → union-merge by componentId 回底层完整数据"的 sandwich 模式（避免双视图共享时的索引错位）。

### 7.5 提交前 hard gate

提交按钮置灰条件（任一满足即不可提交）：

- 存在红色 ERROR 单元格（公式运行时错误）
- 页面顶部有黄色"漂移检测"横幅未处理
- 报价单状态非 DRAFT
- 当前用户无提交权限

## 8. Excel 基础数据导入实战

> 销售最常用的入口：**报价单管理列表页 → 「从基础数据导入」**。

### 8.1 Excel 文件要求

- 格式：`.xlsx`（不支持 .xls / .csv）
- 单次每个 Sheet 行数 ≤ 2000，否则系统拒绝
- 表头行不允许合并单元格（数据行允许）
- 中间空行自动跳过，尾部空行自动截断

### 8.2 14 个 Sheet 一览

| Sheet 名称（中文，区分大小写） | 主要内容 | 关键必填列 |
|---|---|---|
| 来料BOM | 原材料 BOM 结构 | 宏丰料号、BOM 类型、物料名称、净用量、毛用量、损耗率 |
| 元素BOM | 贵金属元素组成 | 宏丰料号、元素名称、组成含量(%) |
| 客户料号与宏丰料号的关系 | 客户料号 ↔ 宏丰料号映射 | 客户ID、客户料号、宏丰料号 |
| 来料固定加工费 | 原材料固定加工成本 | 宏丰料号、客户ID、费用类型、费用值、货币 |
| 来料其他费用 | 原材料其他成本项 | 宏丰料号、客户ID、费用类型、费用值 |
| 来料年降 | 来料年度成本降幅 | 宏丰料号、客户ID、年降比例 |
| 成品固定加工费 | 成品固定加工成本 | 宏丰料号、客户ID、费用类型、费用值 |
| 成品其他费用 | 成品其他成本项 | 宏丰料号、客户ID、费用类型、费用值 |
| 组成件BOM及单价 | 成品组成件明细 | 宏丰料号、客户ID、组成件料号、数量、单价、货币 |
| 组装加工费 | 组装工艺费用 | 宏丰料号、客户ID、加工费类型、费用值 |
| 组装加工费年降 | 组装加工年度降幅 | 宏丰料号、客户ID、年降比例 |
| 电镀费用 | 电镀工艺费用 | 宏丰料号、客户ID、电镀方案编码、加工费、材料费 |
| 电镀方案 | 电镀工艺方案参数 | 方案编码、版本、镀层厚度 |
| 单重 | 物料单重数据 | 宏丰料号、单重（g）、单位 |

> 还有几张辅助 Sheet：元素单价（仅参考价，v1 不入公式）、年降系数字典等，详见 `docs/操作说明.md §6.1`。

### 8.3 4 步导入流程

**第 1 步：上传文件**
- 选择 .xlsx，系统解析进度条。格式错误或超 2000 行立即报错，不写入。

**第 2 步：预览与差异检测**
- 系统自动对比导入内容与数据库现有数据：
  - 新增记录：**绿色**
  - 更新记录：**黄色**（旧值 → 新值）
  - 无变化：**灰色**
- 同时检测：
  - **基础资料差异**（表结构级，走 UI-2）
  - **客户资料冲突**（字段级，走 UI-1）
  - 业务校验（含量之和、单重为 0 等）

**第 3 步：决策（UI-2 → UI-1）**

| 抽屉 | 宽度 | 行为 | 操作 |
|---|---|---|---|
| **UI-2 基础资料差异** | 960px | 表结构级差异（料号信息/BOM 结构/电镀方案） | **整体确认 / 取消**；备注必填（唯一审计线索） |
| **UI-1 客户资料冲突** | 1200px | 字段级冲突（组成件单价/各类费用/电镀费用） | **逐字段决策**「采用导入值 / 保留现有值」；CRITICAL 字段⭐⭐排顶部；500ms 自动保存草稿 |

**第 4 步：确认写入**

点击「确认导入」后系统执行：
1. 单一事务写入所有 14 张物理表（**全成功或全回滚**）
2. 同步写入 `change_log`
3. 释放产品导入锁
4. 如果从报价单页发起，自动创建或关联报价单草稿

### 8.4 公式自动计算 + 漂移检测

写入完成后系统自动重算所有公式字段：

- 单元格出现红色 **ERROR** → 点 ⓘ 看具体错误（路径解析失败 / 类型不匹配 / 除数为零等）
- 页面顶部黄色横幅「**检测到数据漂移**」→ 引用版本被后续导入更新；点击横幅看变更详情，可「刷新版本引用」更新到最新版本

> 漂移不阻止保存，但**阻止提交**。元素单价（Ag/Au/Cu）需销售手动在元素 BOM 行填写（v1 不自动取值），输入框旁会弹"参考价"气泡（管理员录入，仅参考）。

### 8.5 错误码对照

| HTTP 状态码 | 场景 | 处理方法 |
|---|---|---|
| 400 | 文件格式错误 / 必填列缺失 / 超 2000 行 | 按提示修正 Excel 后重试 |
| 409 | 预览到确认期间被其他人改动 | 重新上传重新预览 |
| 423 | DDL 全局锁活跃 / 料号被其他导入锁定 | 等待释放；卡死联系管理员强制释放 |
| 422 | 导出操作超 10,000 行 | 缩小筛选范围 |

## 9. 端到端实战：从零完成一份报价

> 假设场景：销售收到客户 ABC 的新询价（料号 HF-001），需配置一个"工序默认成材率"全局变量，并通过 Excel 导入完成报价。

### Step 1：管理员配置全局变量（一次性，可复用）

1. 进入「配置中心 → 全局变量」→ 点「新建变量」
2. 填写：
   - code: `PROCESS_DEFAULT_YIELD`
   - name: 工序默认成材率
   - 类型: LOOKUP_TABLE
   - key 列名: `process_code`
   - 单位: `%`
3. 行内「维护数据」→ 添加：
   - `Z350` → `15`
   - `Z029` → `100`
4. 保存（缓存自动失效）

### Step 2：管理员/销售经理配置组件字段

1. 进入「配置中心 → 组件管理 → 工序组件 (COMP-CFG-PROCESS)」
2. 找到「成材率」字段（INPUT_NUMBER）
3. 「内容/配置」列点「**+ 默认值来源**」→ 选 `GLOBAL_VARIABLE` → 选 `PROCESS_DEFAULT_YIELD` → `key_field_refs` 留空（同名映射）
4. 用「测试 driverRow JSON」填 `{"process_code":"Z350"}` → 点「测试解析」应返 15
5. 保存组件 → 系统**自动**同步所有引用该组件的模板 snapshot

### Step 3：销售上传 Excel 创建报价单

1. 进入「报价中心 → 报价单管理」→ 点「**从基础数据导入**」
2. 选客户「ABC」→ 上传 `客户ABC-料号HF-001.xlsx`
3. 系统弹出 **UI-2** 抽屉（基础资料差异）→ 填备注「客户首版基础资料」→ 整体确认
4. 系统弹出 **UI-1** 抽屉（客户资料冲突）→ 逐字段决策（CRITICAL ⭐⭐ 优先）→ 确认所有字段
5. 点「确认导入」→ 事务写入 14 张表 → 自动跳到报价单草稿

### Step 4：销售填写报价内容

1. 第 2 步「添加产品」→ 系统已自动加载 HF-001 行
2. 工序卡片上「成材率」字段 placeholder 显示「默认 15 · PROCESS_DEFAULT_YIELD」→ 接受默认或输入覆盖
3. 元素 BOM 行 Ag/Au/Cu 单价手动填（参考价气泡显示管理员最新参考价）
4. 切换 viewType 看 Excel 视图核对最终列：所有公式列自动重算
5. 第 3 步选定价策略 → 第 4 步填条款 → 第 5 步「提交」

### Step 5：销售经理审批 → 输出

1. 销售经理进入「报价单管理」→ 筛选 SUBMITTED
2. 点报价单详情 → 看「数据来源」Tab（基础资料快照、引用版本、元素单价手填记录）
3. 顶部「审批」按钮 → 通过/驳回 + 审批意见
4. APPROVED 后可下载 PDF / Excel 发给客户

| **技术实现** | 使用 `cpq-frontend/src/components/SelectableTable.tsx` |

详细规范见 `docs/列表操作规范.md`。

### 10.2 开发规范

| 规范 | 要求 |
|---|---|
| **需求变更必同步 PRD** | 修改 `docs/PRD-v3.md` 对应章节，并在第 9 章演进史记录调整 |
| **不要修改 PRD.md** | 已废弃归档，仅作变更决策回溯 |

| **完成工作后追加 RECORD.md** | 格式：`[日期] 模块 - 简要描述 \| 涉及文件 \| 关键决策或注意事项` |

### 10.3 修改后强制自检（每次代码改动结束前必须跑）

**前端改动**：
1. `cd cpq-frontend && npx tsc --noEmit -p tsconfig.json` → 必须 0 错误
2. 对每个改动的 .tsx 文件 `curl http://localhost:5174/src/<相对路径>` → HTTP 200
3. 故障路径含具体页面时必须 curl 拿到 200 才报"修复"

**后端改动**：
1. `touch` 一个 java 文件强制 Quarkus 重启（等 5-7 秒）
2. `curl http://localhost:8081/q/health` 或具体 endpoint → 期望 200/401（不要 500）
3. Flyway 迁移：查 `flyway_schema_history` 必须 `success=t`
4. **不要手动 `psql -f V_xx.sql`**！让 Quarkus dev mode 自己跑 Flyway

**视图 DROP CASCADE / 重建后必须重启 Quarkus**，原因：`ImplicitJoinRewriter.tableColumnsCache` 等 ApplicationScoped 进程级缓存可能在视图被临时删除时缓存空集，导致 BNF 路径求值不再注入 `hf_part_no` 谓词，UI 出现"首值（共N项）"错乱。

**完成宣告必须含一行"已自检"声明**，例：
> TS 0 错误 ✅；CostingPartDataPage.tsx → Vite 200 ✅；后端 /api/cpq/.. → 401（auth 正常）✅；V77 success=t ✅

### 10.4 配置中心禁止事项

- ❌ 不要为新全局变量建独立物理表 — 一律走 `global_variable_value` 单表
- ❌ 不要硬编码字段映射 — 业务渲染层零硬编码
- ❌ 不要让数据源类型固化在代码 — 用 `datasource_binding.type` 显式区分
- ❌ 不要绕过缓存失效闭环 — 全局变量写入必走 `GlobalVariableService.upsertEntry`
- ❌ 不要让核价价格表混进"全局变量"UI 编辑 — 通过 `COSTING_INTERNAL` 隔离
- ❌ 不要再写 `DO $$` 循环刷 template snapshot — H1 已自动化

## 11. 常见问题与避坑速查

### 11.1 全局变量 / 组件配置

| 现象 | 原因 | 修法 |
|---|---|---|
| 改组件字段后报价单还是旧配置 | snapshot 没同步 | H1 已自动同步；或手工 `POST /api/cpq/components/{id}/refresh-template-snapshots` |
| placeholder 显示「默认 100」没有 ` · CODE` 后缀 | DB 查不到 key | 检查 `global_variable_value` 是否含该 key；走 `INSERT ... SELECT DISTINCT FROM mat_xxx ON CONFLICT DO NOTHING` 全 seed |
| DATA_SOURCE 字段保存报 400 缺 datasource_id | 改了 type 但后端校验未识别新 type | 已在 K hotfix 按 type 分发校验 |
| 字段切换 type 后老配置残留 | JSON 松散结构 | FieldConfigTable 的 type 切换 onChange 已自动清相关字段（H2） |
| 测试解析返 null | driverRow JSON 缺 key 列字段 | 测试 driverRow 至少含 `def.keyColumns` 内所有列 |
| HTTP_API 不工作 | 白名单未配 | application.properties 加 `cpq.http-api.allowed-hosts=...`；token 用环境变量 |

### 11.2 报价单 / Excel 导入

| 现象 | 原因 | 修法 |
|---|---|---|
| 上传 Excel 后报"未识别到任何 Sheet 数据" | Sheet 名拼写不匹配（全角/半角空格） | 对照 §8.2 Sheet 名核对 |
| 单元格显示红色 ERROR | 公式运行时错误 | 点 ⓘ 看具体错误：路径大小写 / `TO_NUMBER()` 转换 / `IF` 防除零 |
| 顶部黄色横幅"漂移检测" | 引用的基础数据版本被新导入覆盖 | 点横幅看变更详情；可"刷新版本引用" |
| 提交按钮置灰 | 存在 ERROR / 漂移横幅 / 状态非 DRAFT / 无权限 | 按上方 hard gate 逐条排查 |
| 数据来源 Tab 不显示内容 | DRAFT 状态下显示"当前跟随最新版本"实时状态 | 提交后才有完整快照 |
| ELEMENT_PRICE / PREMIUM_PRICE 公式返空 | v1 不可用 | 改用销售手填元素单价 + `{报价单.element_unit_price}` 引用 |
| 导入卡住进度条不动 | 产品锁未释放 | 联系管理员到"锁监控"强制释放 |

### 11.3 多行数据展示

| 现象 | 原因 | 修法 |
|---|---|---|
| Excel 模板列显示「— (共4项)」/「0.138 (共3项)」 | 视图被 CASCADE 重建瞬间缓存了空集 | V112 后自愈，但需要重启 Quarkus 清进程级缓存 |
| 「工序」「材料名」列显示「—」但 DB 行有 process_code | INSERT 时漏写 view 渲染依赖的 name/描述列 | 写表前 grep 该表所有 basic_data_path 引用，所有渲染依赖列一并 INSERT |

更全的多行数据展示问题在 `docs/反模式.md` AP-22；配置决策树在 `docs/配置方法论.md`。
