# CPQ 数据一致性 / WYSIWYG 持久化方法论

> 配合 `docs/反模式.md` 使用。
> 反模式记的是"出过的坑"；本文档记的是"防新坑的方法论"——给出系统排查思路与命中清单，每个新功能上线前都要走一遍。

---

## 1. 问题模型

CPQ 是典型的"主数据 + 报价配置 + 公式衍生"三层数据系统：

| 层 | 例子 | 特性 |
|---|---|---|
| 主数据 | mat_part / mat_bom / customer / product / template | 长寿命、可被运营改动、跨报价共享 |
| 报价配置 | quotation / quotation_line_item / quotation_line_component_data.row_data | 每张报价单一份，需要审计与历史回溯 |
| 公式衍生 | 公式引擎 / driver 展开 / BASIC_DATA 路径解析 | 运行时计算，依赖主数据 |

**致命 anti-pattern**：把"运行时计算结果"当成"渲染时再算就好"，**保存只 dump 用户输入字段**。结果：

- 当时看屏幕一切正常（因为再算一次也能算出来）
- 主数据后续被改 / 删 → 老报价的历史值变了甚至读不出
- 提交流的 submission_snapshot 也只快照了 row_data 现状，连同空缺一并冻结
- 几个月后审计时发现"屏幕显示 75.05、数据库写 null"，无法解释

详见 `反模式.md` AP-11。

## 2. 排查方法（用在 PR / Code Review / 上线前）

### 2.1 起点：列出所有"展示态"

对每个有"业务数据展示"的页面 / 组件，把屏幕上所有可见列 / 文本字段都列出来，按**数据来源**分四类：

| 类别 | 标记 | 数据从哪来 |
|---|---|---|
| INPUT | I | 用户敲在表单里、存 row state |
| FIXED | F | 模板里写死的常量 |
| BASIC_DATA / 主数据 lookup | B | 运行时查 mat_part / mat_bom / customer / product 等主表得到 |
| FORMULA / 衍生 | C | 前端 / 后端公式引擎实时算 |
| DATA_SOURCE / 数据源 | D | 走 datasource 接口实时查（可能命中外部库） |

### 2.2 关键问题（每一项都要回答）

对每一个**非 I 非 F**字段（即 B / C / D 中的任何一个），逐一回答：

1. **保存时这个字段进 payload 了吗？**
   - 在 `buildXxxPayload` / `toRequestDTO` 里 grep 字段名，要么 grep 到，要么走 snapshot 函数
   - 没进 → AP-11 命中
2. **该字段的数据源（lookup 表 / 公式依赖）有可能后续被改 / 被删吗？**
   - 是 → 必须落 snapshot
   - 否 → 可以延迟 lookup
3. **submission_snapshot / 类似冻结结构里有这个字段吗？**
   - 提交后冻结、确保审计可回溯
4. **导出（PDF / Excel）走 snapshot 还是 live 查？**
   - 必须走 snapshot；否则导出文件跟报价快照对不上
5. **多用户并发场景（用户 A 改 mat_bom，用户 B 看老报价）下，老报价能否正确显示当时的值？**
   - 走 snapshot → 能；live 查 → 不能

### 2.3 命中后的两种修复策略

**策略 A — Save-time snapshot**（修在前端 buildPayload）：
- 序列化 `row` 之前先注入 BASIC_DATA / FORMULA / DATA_SOURCE 计算结果
- 适用：值的"权威来源"在前端运行时（前端公式引擎 / driver 展开缓存）
- 例：QuotationWizard.snapshotRows（AP-11 修复）

**策略 B — Read-time snapshot column**（修在后端 entity + DTO）：
- 给被 join / 反查的字段加 `_snapshot` 列；保存 / 创建实体时一并写入快照
- DTO 回读时 `lookup → snapshot fallback` 优先级
- 适用：值的"权威来源"在后端（DB join / 服务端查询）
- 例：`quotation.snapshot_customer_*` / `quotation_line_item.product_part_no_snapshot`（AP-3 修复）

**两者可叠加**：前端 save-time 写 row.fieldName + 后端持久化时另外写 snapshot 列做防腐——金额型字段 / CRITICAL 字段建议双写。

在 `cpq-frontend/src` 下：
```bash

```

在 `cpq-backend/src/main/java/com/cpq` 下：
```bash
# 找 export / snapshot 服务里直接 join 取主数据的地方（应改读 snapshot 列）
grep -rn "Customer.findById\|Product.findById" src/main/java/com/cpq/quotation

grep -rn "import com.cpq.product.entity.Product" src/main/java/com/cpq/quotation
```

## 5. 接口 404 排查方法（配合 AP-12 反模式）

页面在正常导航中暴露的 404 几乎 100% 不是路由缺失，而是「主资源 OK 但懒资源未生成」。下面是排查手顺：

### 5.1 是不是真路由缺失？

```bash
# 用 URL 路径片段定位 Resource 类
grep -rn "@Path.*<片段>" cpq-backend/src/main/java/com/cpq/
```

- 找到 `@Path` + `@GET/@POST` 完整定义 → 路由存在；问题在 service 层
- 没找到 → 真路由缺失，前端 service 的 URL 写错或者后端漏实现

### 5.2 路由存在 — 谁抛的 404？

```bash
grep -rn "BusinessException(404" cpq-backend/src/main/java/com/cpq/<module>
```

抓到具体方法后，回答 3 个问题：

1. 该资源是**懒生成**的吗？（用户操作触发才存在；DRAFT 状态本来就没有这行）
2. 同 service 内其他方法对相同语义有没有**空 DTO 兜底**？（一致性比较——常见的那个对，错的那个就该改）
3. 前端 service 调用方有没有 catch 兜底？前端是否在 catch 里把整页降级了？

### 5.3 修法选择

| 场景 | 修法 |
|---|---|
| 主资源 ID 真不存在 | KEEP_404 |
| 主资源 OK + 懒资源未生成（DRAFT / 未启用 / 未配置） | GRACEFUL_NULL：返回空骨架 DTO（quotationId 透传、列表字段空数组、status null） |
| 后端历史包袱难改、改了风险大 | FRONTEND_HANDLE：前端 service 在 `catch (e.response?.status === 404)` 时返回等价空数据，注释里指明对应后端方法 |

GRACEFUL_NULL 优于 FRONTEND_HANDLE：避免前端散点都做 404 兜底；后端单点处理，前端代码干净。
