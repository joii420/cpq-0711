## 0. 一句话心智模型

> **核价树 = 全局一条递归 SQL（产结构）+ 每个页签自己的 `$view`（产数据），后端按 `material_no` 把数据行分配到各卡片、按 `node_path` 把树页签的行拼成树。**
>
> - **① 全局递归 SQL**（存 `costing_bom_tree_config` 表，组件管理 [核价树配置] 页签维护，全局只有一条"生效"）：输入本报价单**所有卡片的生产料号**，一次性递归出全部卡片的 BOM 结构，产出 `root_no / material_no / bom_version / parent_no / node_path` 五列。**只跑一次**，不管报价单有多少张卡片。
> - **② 每个页签组件的 `$view` 取数 SQL**：只按 `material_no` 取业务数据，**每个页签只跑一次**（不管报价单有多少张卡片、树里有多少节点）。
> - 后端拿到①的结构 + ②的数据后，**在内存里**按 `root_no` 分组出每张卡片的料号集合、按 `material_no` 把②的行分配/复制到对应卡片和节点，不再逐卡片逐组件跑 SQL。

**与旧机制的本质区别**：旧机制是"每卡片 × 每递归组件各跑一次闭包 SQL + 用 spineKeys 三元组精确挂数据"，SQL 执行次数随卡片数 × 组件数增长；新机制是"**1 次递归 + N 次页签查询**（N=模板页签数，与卡片数无关），其余全是后端内存分组"。旧的 `BomClosureService`、`SpineKeysMacro`/`SpineKeysContext`、`buildSpineBaseRows`、组件级 `bom_recursive_expand` 触发"闭包"的旧语义、4A/4B 视图写法二选一，**均已删除/废止**。

## 1. 原理：一棵核价树是怎么被组装出来的

```
本报价单所有卡片的生产料号（seed = distinct(li.productPartNo)，去重）
                          │
① CostingTreeRenderService 跑「生效」的全局递归 SQL（仅 1 次）
      绑 :production_part_nos = seed
      → treeRows[]: { root_no, material_no, bom_version, parent_no, node_path }
                          │
   后端内存分组（不再查库）：
      cardMaterialNo[root_no] = distinct(material_no where root_no=该卡)（含根本身）
      totalMaterialNo         = distinct(所有 treeRows.material_no)  ← 各卡并集
      treeRowsByRoot[root_no] = 该卡的树行列表（保序，树页签建树用）
                          │
② 对模板每个页签组件的 $view 各跑 1 次
      绑 :total_material_no = totalMaterialNo
      SQL 内部自己写 WHERE 料号列 = ANY(:total_material_no)
      → 输出必含 material_no 列
      → rowsByMaterial[组件]: Map<material_no, List<业务行>>
                          │
③ 对每张卡片 × 每个页签，后端按 material_no 分配/复制：
   - 普通页签：material_no ∈ cardMaterialNo[该卡] 的行平铺输出；不属任何卡的行丢弃
   - 树页签（组件勾了「核价树」）：
       遍历该卡 treeRowsByRoot 的每个节点（保留多 occurrence，不去重）
         → 按 **(node.parent_no, node.material_no) 边键** 取 rowsByEdge，有则输出、无则补空行
           （2026-07-03 起：树页签数据是「父→子」边特有的，按边键精确挂，同子件挂多父不再串入别父件的边）
         → 注入系统列 __hfPartNo / __bomVersion / __parentNo / __nodeId / __parentId / __lvl
                          │
④ 前端 QuotationStep2（渲染分支未改，协议沿用旧版）：
      activeComponentBomTree = 核价侧 && 行里有 __sys.nodeId
      为真 → 渲染成树 + 固定带出「料号 / 版本」两列，按 __parentId → __nodeId 建父子层级
```

**执行次数**：1（递归）+ N（模板页签数），**与报价单卡片数、树节点数无关**——这是本次重构相对旧机制的核心收益。

**匹配键分两类（2026-07-03 树页签升级为边键）**：
- **普通（平铺）页签**：按 `material_no` 匹配（料号维度，如元素成分——同料号元素与父件无关）。
- **树页签**：按 **`(parent_no, material_no)` 边键** 匹配 → **树页签 `$view` 必须同时输出 `parent_no`（父件料号）+ `material_no`（子件料号）两列**。因为 BOM 树每个节点的数据是「父→子」边特有的（如用量随父件不同）；仅按 `material_no` 会把同一子件在**别父件下**的边也挂进本节点 → 重复/挂错父。对 `material_bom_item`：`parent_no = material_no(父)`、`material_no = component_no(子)`。
- **多 occurrence 小计语义不变**：同一子件挂多父 → 多个树节点，各节点挂各自边的业务行，小计按位置各算一次（业务预期），不引入旧版 spineKeys 三元组。

## 2. 配置步骤（四步）

### Step 1 — 组件管理 → [核价树配置] 页签 → 新增/编辑全局递归 SQL

- 入口：组件管理 Hub → 「核价树配置」页签（`CostingBomTreeConfigTab`，对应后端 `/api/cpq/costing-bom-tree-config`）。
- **契约（逐字，写错会被保存期 dry-run 拦截）**：
  - 输入变量：`:production_part_nos`（`text[]`），必须在 SQL 中引用。
  - 输出列（必须**逐字**全部出现，共 5 列）：
    - `root_no` —— 根生产料号（隐藏，仅用于按卡片分组）
    - `material_no` —— 当前料号（显示）
    - `bom_version` —— 当前料号（节点自身）版本号（显示；叶子自身无 BOM 可为 NULL）
    - `parent_no` —— 上级料号（仅显示/信息用，**不再是结构键**）
    - `node_path` —— **节点在树中的物化路径（结构键）**：从根到本节点的祖先路径，用 `/` 连接，根为单段，**每个节点（含同料号多 occurrence）全局唯一**。后端只按这一列精确建树。
  - `node_path` 硬约束：
    - 根节点 = 单段（如 `P1`）；子节点 = 父路径 + `/` + 本节点标识（如 `P1/A`、`P1/B/A`）。
    - 段内标识**不得含 `/`**（后端按最后一个 `/` 切出父路径）；若料号本身可能含 `/`，改用边序号/代理键做段标识，只需保证同一父下各子唯一。
- 写法参考（`WITH RECURSIVE` 累加 `node_path`，分区条件自己写）：
  ```sql
  WITH RECURSIVE bom AS (
    SELECT p AS root_no, p AS material_no,
           (SELECT bom_version FROM material_bom_item x
             WHERE x.material_no = p AND x.customer_no='_GLOBAL_'
               AND x.system_type='PRICING' AND x.is_current LIMIT 1) AS bom_version,
           CAST(NULL AS varchar) AS parent_no,
           p::text                AS node_path          -- 根 = 单段
    FROM unnest(:production_part_nos) AS p
    UNION ALL
    SELECT b.root_no, ch.component_no,
           ch.bom_version, ch.material_no,
           b.node_path || '/' || ch.component_no          -- 累加路径
    FROM material_bom_item ch
    JOIN bom b ON ch.material_no = b.material_no
    WHERE ch.customer_no='_GLOBAL_' AND ch.system_type='PRICING'
      AND ch.is_current AND ch.component_no IS NOT NULL
  ) CYCLE material_no SET is_cyc USING pth
  SELECT root_no, material_no, bom_version, parent_no, node_path FROM bom;
  ```
- 保存后点「设为生效」——**全局同一时刻只能有一条生效**（唯一索引保证，设为生效时后端在单事务内先把旧的 active 置 false 再置新的 true）。

### Step 2 — 目标组件勾选「核价树」

- 组件管理 → 目标组件 → 勾上字段配置里的 **「核价树」**（对应字段 `bomRecursiveExpand`，实体列 `bom_recursive_expand`）。
- 默认关，必须手动勾。
- **一个核价模板最多一个树页签**：保存模板 / 渲染前会校验，超过一个直接报错。
- 语义已变：不再触发"该组件走 BOM 闭包递归"，而是"**该组件页签渲染为树 + 三系统列**"，结构来自 Step 1 的全局递归结果，不再由组件自己发起闭包查询。

### Step 3 — 配置各页签组件的 `$view` 取数 SQL

- 树页签、普通页签**同一套契约**，都只按料号取数：
  - 输入变量：`:total_material_no`（`text[]`）= 本报价单所有卡片料号的并集。
  - SQL 内自己写 `WHERE <料号列> = ANY(:total_material_no)`（或等价 `IN`）。
  - **输出必含 `material_no` 列（逐字）**；保存期 dry-run 会校验缺列则拦截。
  - 分区条件（`system_type='PRICING'` / `customer_no='_GLOBAL_'` / `is_current` 等）**由 SQL 自己写死**——执行器**不注入** `customer_id` / `quotation_id` / `lineItemId` 等报价侧上下文变量。
- 组件字段（`fields`）配置沿用现状：`field_type=BASIC_DATA`、`basic_data_path=$视图名.列名`，列名须与视图 SELECT 输出列名逐字一致。

示例（料号维度取数，最常见形态）：
```sql
SELECT
  ebi.material_no  AS material_no,   -- ★ 必须：用于分配到卡片/节点
  ebi.component_no AS 元素,
  ebi.content       AS 含量
FROM element_bom_item ebi
WHERE ebi.system_type='PRICING' AND ebi.is_current=true
  AND ebi.material_no = ANY(:total_material_no)
```

### Step 4 — 重算报价单

- 改了①全局递归 SQL / 切了生效行、②组件「核价树」勾选、③页签 `$view` 或字段配置之后，**必须重算报价单**（重开该单或走 refresh-snapshot），旧快照不会自动追溯。
- 切换 `costing_bom_tree_config` 生效行或改其 SQL 后，需要让相关核价草稿快照失效重算，否则打开的仍是旧配置算出的快照。

## 3. 硬规则（配置任何核价树相关 SQL 都要守）

1. **递归 SQL 必出 5 列**：`root_no / material_no / bom_version / parent_no / node_path`，且引用 `:production_part_nos`。缺列或未引用变量 → 保存 `costing_bom_tree_config` 时 dry-run 拦截。
2. **`node_path` 是唯一结构真理源**：`__nodeId = node_path`；`__parentId` = 去掉最后一段（无 `/` 的单段根 → `null`）；`__lvl` = 段数（`/` 数 + 1）。**不要**指望 `parent_no` 参与建树，它只是显示列。
3. **`node_path` 段内不得含 `/`**，且同一节点（含同料号多 occurrence）全局唯一——否则后端切分父路径会切错、挂错枝。
4. **页签 `$view` 必出 `material_no` 列**，且用 `:total_material_no` 过滤。缺列 → 保存期 dry-run 拦截。
5. **匹配键只有 `material_no`**：不带版本、不带父件维度。同一料号命中多张卡片 → 该行进多卡；一张卡内多 occurrence → 业务行按 `material_no` 复制到各节点，小计各算一次（业务预期，非 bug）。
6. **一个核价模板最多一个「核价树」组件**（`bom_recursive_expand=true`），超过一个保存/渲染前拦截。
7. **无生效递归 SQL 时，含树页签的模板渲染会明确报业务错**，不会静默出空树。
8. **分区条件（PRICING/`_GLOBAL_`/`is_current`）由每条 SQL 自己写**，执行器不会替你注入 customer/lineItem 等上下文——这点与报价侧（QUOTATION）视图不同，报价侧原有上下文注入保持不变、不受本次改动影响。
9. **改①递归 SQL/生效行、②组件核价树勾选、③页签视图或字段后必须重算报价单**——系统列是算快照那一刻烘进去的，不追溯旧单。
10. **硬切换、不迁移存量**：本次重构对旧引擎是彻底替换，不做任何存量数据迁移。**存量核价报价单 / 依赖旧 spineKeys 视图的模板不受支持**，打开可能报错或空，必要时按需清理（不影响报价侧 QUOTATION，只影响核价侧 COSTING 且模板含旧机制残留配置的场景）。

## 4. 三个"固定系统列"（料号 / 父料号 / 版本 / 树结构）

- 不是配置员在视图里写的，是后端为树页签**自动注入**：`__hfPartNo`（= node.material_no）、`__bomVersion`（= node.bom_version）、`__parentNo`（= node.parent_no，仅建层级参考、不单独显示）、`__nodeId`（= node_path）、`__parentId`（= node_path 去尾段）、`__lvl`（= node_path 段数）。
- 前端渲染条件：`核价侧 && 该组件行里有 __sys.nodeId`。满足才**固定带出「料号 / 版本」两列** + 建树。此渲染分支协议沿用旧版，**未随本次后端重构改动**。
- 版本列口径 = 节点自身当前 BOM 版本，叶子（自身无 BOM）显示空——设计如此，非 bug。
- 两条渲染面（产品卡片视图 + Excel 视图 `buildLineTreeRows`）现在**共用同一份快照 `__nodeId` 骨架**：Excel 视图已改为直接读快照，不再自己重算闭包/树结构。若详情页 `ReadonlyProductCard` 也读 `__sys.nodeId` 协议，同样只读快照、不重算。
- **看不到这三列 = 该单快照没有按新管线算过**，常见原因：①勾「核价树」当时没保存生效的递归 SQL、②勾了/改了配置没重算、③浏览器没刷新。**先确认有生效递归 SQL → 重算 → 强刷。**

## 5. 常见坑速查

| # | 症状 | 根因 | 修法 |
|---|---|---|---|
| 1 | 整单核价树渲染报错 | `costing_bom_tree_config` 没有生效行（is_active=true 的那条不存在或被删了） | 组件管理 [核价树配置] 页签新增/选一条「设为生效」 |
| 2 | 树塌成一坨 / 同料号多 occurrence 的子件挂错父 | `node_path` 没做到"同一节点全局唯一"，或段内标识含 `/` 被切错父 | 检查递归 SQL 的 `node_path` 累加逻辑，确保多 occurrence 各自独立路径、段内不含 `/` |
| 3 | 某列（或整表）`#ERROR` / 空 | 字段 `basic_data_path` 引用了页签视图没有的列名 | 视图列名 ↔ 字段路径对齐 |
| 4 | 整表 0 行 / 页签数据全空 | 页签视图没写 `material_no = ANY(:total_material_no)`，或输出漏了 `material_no` 列 | 按 §2 Step 3 契约补齐 |
| 5 | 模板保存时报"最多一个核价树组件" | 模板里勾了「核价树」的组件超过一个 | 只保留一个组件勾选，其余作为普通页签用 `material_no` 平铺 |
| 6 | 改了递归 SQL / 切了生效行没生效 | 相关核价草稿快照未失效重算 | 重算该单（重开或 refresh-snapshot） |
| 7 | 勾了核价树还是没有三列 | 旧快照没走新管线（勾选前/无生效递归 SQL 时算的）/ 没重算 / 浏览器缓存 | 确认生效递归 SQL 存在 → 重算该单 + 强刷 |
| 8 | 打开某张存量核价单报错或空 | 存量数据依赖已删除的旧引擎（`BomClosureService`/spineKeys），本次硬切换不迁移存量 | 属预期后果；必要时清理该存量核价数据/模板/视图后按新契约重配 |
| 9 | 页签视图报"缺 material_no 列"或递归 SQL 报"缺列/未引用变量" | 保存期 dry-run 校验拦截（§3 规则 1/4） | 按 §2 契约补齐输出列/变量引用 |

## 6. 验证方法（DB + UI）

**DB 级（不需登录）——确认快照对不对：**
```sql
-- 某产品某页签的树骨架 + 数据（新版：__nodeId 直接是 node_path）
WITH tab AS (
  SELECT c FROM quotation_line_item li, jsonb_array_elements(li.costing_card_values->'tabs') c
  WHERE li.quotation_id='<报价单id>' AND li.product_part_no_snapshot='<根料号>'
    AND c->>'tabName'='<页签名>' LIMIT 1)
SELECT (r->>'__lvl')::int lvl,
       repeat('  ',(r->>'__lvl')::int-1)||(r->>'__hfPartNo') 料号,
       r->>'__nodeId' 节点路径, r->>'__parentId' 父节点路径, r->>'__bomVersion' 版本,
       r->'driverRow' 业务数据
FROM tab, jsonb_array_elements(tab.c->'baseRows') r ORDER BY (r->>'__nodeId');
```
- 看到多层 `__lvl` + `__nodeId/__parentId/__hfPartNo/__bomVersion` = 已按新管线建好树；
- 业务数据格有值 = 页签视图取数正确。

**全局递归 SQL 自检（某生产料号能否展开出结构）：**
```sql
-- 把 costing_bom_tree_config 生效行的 sql_template 取出来，手动绑一个料号数组跑一次
-- 期望：出多行，且 node_path 每行唯一、根为单段
```

**UI 级：** 组件管理 [核价树配置] 确认有一条「生效」 → 组件勾「核价树」 → 页签视图配好 `material_no`/`:total_material_no` → 重算该单 → 强刷 → 核价单 → 产品卡片视图 / Excel 视图：应见「料号/版本」两列 + 多层树，两处树结构一致。

**协议级改动**（动 `CostingTreeRenderService` / `CostingTreeGrouping` / `SqlViewExecutor` / 递归开关语义）按 CLAUDE.md 属协议级，须跑 Playwright E2E（`docs/E2E测试方法.md`）。

## 7. 检查清单（配完对照）

- [ ] `costing_bom_tree_config` 有且仅有一条生效行，SQL 输出 5 列（`root_no/material_no/bom_version/parent_no/node_path`）且引用 `:production_part_nos`
- [ ] `node_path`：根单段、子路径累加、段内不含 `/`、同一节点（含多 occurrence）全局唯一
- [ ] 目标组件已勾「核价树」，且**同一核价模板内只有这一个**勾选的组件
- [ ] 各页签视图输出含 `material_no` 列，且用 `:total_material_no` 过滤
