# 核价树页签组件 配置指南

> 面向:配置"报价单核价侧、按 BOM 树递归展开子料号"的页签组件的人(开发 / 配置员 / 测试)。
> 版本:2026-07-02(基于当前代码 + 实机验证沉淀)。
> 关联:`docs/superpowers/specs/2026-06-04-核价BOM递归展开-design.md`、`2026-06-06-spinekeys-复合键跨页签过滤-design.md`、`2026-06-06-核价BOM递归展开-组件级开关-design.md`、`docs/三大核心模块基线.md`(核价单渲染基线)、`docs/反模式.md` AP-51/AP-53。

## 0. 一句话心智模型

> **核价树 = 两件事的组合,缺一不可:**
> - **① 组件递归开关(`bom_recursive_expand`)** → 后端算出**树的骨架**(哪些节点、父子层级、版本)+ 把整棵树的料号(`partSet`)喂给视图。**产结构,不产数据。**
> - **② 组件的 `$view` 取数 SQL** → 给每个节点**取业务数据**。**产数据,不产结构。**
>
> 后端最后按 **`hf_part_no`** 把"②的数据行"挂到"①的骨架节点"上,组装成树。

---

## 1. 原理:一棵核价树是怎么被组装出来的

```
产品卡片根料号(li.productPartNo)
  └─ ① BomClosureService.compute(根料号)          ← 组件开关=true 才对该组件建树
        ├─ partSet   = {根 + 全部子孙料号}          → 注入视图 :hfPartNos
        └─ spine     = 树骨架(每 occurrence 一节点,含 hfPartNo/parentNo/bomVersion/lvl)
                          │
  ② 组件 $view 取数(执行器包一层 SELECT * FROM (你的SQL) inner_q)
        └─ 返回按 hf_part_no 分组的业务行
                          │
  buildSpineBaseRows(spine, byPart):
        对每个 spine 节点 → byPart.get(节点.hfPartNo)
             有行 → 用该行 + 注入系统列  │  无行 → 补空行 + 注入系统列
                          │
        注入的系统列(`__` 前缀):
          __nodeId / __parentId / __lvl / __hfPartNo / __parentNo / __bomVersion / __isCycle
                          │
  前端 QuotationStep2:
        activeComponentBomTree = (核价侧 && 行里有 __sys.nodeId)
        为真 → 渲染成树 + 固定带出「料号 / 父料号 / 版本」三系统列
        并按 __parentId → __nodeId 建父子层级
```

**闭包(①)的固定口径(代码写死,`BomClosureService.CLOSURE_SQL`)**:
- 数据源:唯一 `material_bom_item`(父子边:`material_no`=父、`component_no`=子);
- 硬过滤:`customer_no='_GLOBAL_'` + `system_type='PRICING'` + `is_current=true` + `component_no IS NOT NULL`;
- 全层级自动展开(PG `WITH RECURSIVE ... CYCLE`,成环标记不展开);
- 版本:每节点取**子件自身当前 BOM 版本**(叶子无自身 BOM → NULL)。

## 2. 前置条件 / 数据准备(最容易被忽略,先查)

核价树只认 **核价侧分区** 的数据。配之前务必确认:

| 数据 | 必须满足 | 不满足的后果 |
|---|---|---|
| BOM 父子(`material_bom_item`) | `system_type='PRICING'` + `customer_no='_GLOBAL_'` + `is_current=true` | 闭包爬不出子料号 → **只出根一行、没有树** |
| 业务数据(如 `element_bom_item` / 单价等) | 同样在核价侧可查(PRICING;元素/材质等按各自表口径) | 树骨架在、**每格空** |

> ⚠️ **最常见的"没有数据"根因**:产品的 BOM/元素数据其实在 **`QUOTE`（报价侧）** 分区,而核价树只读 `PRICING`。表现为:大多数产品核价树是空的,只有被专门导入 PRICING 的少数料号有树。**解决 = 把要核价的产品 BOM/业务数据按 `PRICING/_GLOBAL_/is_current` 导入。**

自检 SQL:
```sql
-- 某根料号在核价侧有没有可递归的子件
SELECT count(*) FROM material_bom_item
WHERE material_no = '<根料号>' AND system_type='PRICING'
  AND customer_no='_GLOBAL_' AND is_current=true AND component_no IS NOT NULL;
-- >0 才有树
```

## 3. 配置步骤(六步)

### Step 1 — 备齐 PRICING BOM 父子数据
按 Step 2 前置(§2)导入/确认 `material_bom_item` 的 `PRICING/_GLOBAL_/is_current` 父子行。**这是树的结构来源,没有它一切白搭。**

### Step 2 — 组件勾选「BOM 递归展开」开关
- 组件管理 → 目标组件 → 勾上 **"BOM 递归展开"**(`bom_recursive_expand=true`)。
- 默认是**关**(V296 起存量全置关),**必须手动勾**。
- 作用:让后端对该组件走"闭包 + spine"路径,并自动带出三系统列。

### Step 3 — 配置组件的 `$view` 取数 SQL(核心,见 §4)
- 组件 `data_driver_path = $你的视图名`(如 `$ys_view`);
- 编辑该视图 SQL,**务必满足 §5 的硬规则**(尤其 `hf_part_no` = 节点自身料号)。

### Step 4 — 配置组件字段(fields),路径对齐视图列
- 每个展示字段 `field_type=BASIC_DATA`,`basic_data_path = $视图名.列名`;
- **`列名` 必须与视图 SELECT 的输出列名逐字一致**,否则该格 `#ERROR`(见 §6 坑 3)。

### Step 5 — 把组件加入核价模板
- 核价模板(`template_kind=COSTING`)里挂上该组件;
- 确认模板挂的是**你勾了递归的那个组件 id**(小心导入副本重名,见 §6 坑 6)。

### Step 6 — 重算 + 验证
- **勾开关 / 改视图 / 改字段后,必须重算报价单**(重开该单或走 refresh-snapshot),否则旧快照没有 spine;
- 按 §7 验证 DB + UI。

## 4. 两种取数视图写法(按数据表形态二选一)

**先判断你的业务数据表是"按料号存"还是"按边存":**

### 4A. 料号维度表(推荐首选)—— 如 `element_bom_item`(一个料号一组数据)
直接按料号出,`hf_part_no` 取该料号,**不用 spineKeys**,靠执行器自动 `hf_part_no=ANY(partSet)` 过滤:

```sql
SELECT
  ebi.material_no  AS hf_part_no,     -- ★ 节点自身料号(必须)
  ebi.material_no  AS 料号,
  ebi.component_no AS 元素,
  ebi.content      AS 含量,
  ...
FROM element_bom_item ebi
LEFT JOIN material_master mm ON mm.material_no = ebi.material_no
WHERE ebi.system_type='PRICING' AND ebi.is_current=true
-- 不写 hf_part_no 过滤:执行器自动包 WHERE inner_q.hf_part_no = ANY(:hfPartNos)
```
- 优点:简单、天然不重复(料号维度)、无 spineKeys 版本坑。
- 每个 spine 节点按自己的料号命中自己那组数据。

### 4B. 边维度表 —— 如 `material_bom_item`(一行=一条父子边,同子件可挂多父/多版本)
用 **`:spineKeys(子件, 父件, 子件自身版本)`** 精确对号(区分同料号的多 occurrence):

```sql
SELECT
  mbt.component_no AS hf_part_no,     -- ★ 节点自身 = 子件(不是父件!)
  mbt.material_no  AS parent_no,
  ...
FROM material_bom_item mbt
WHERE mbt.system_type='PRICING' AND mbt.customer_no='_GLOBAL_' AND mbt.is_current
  AND :spineKeys(
        mbt.component_no,                                          -- 子件料号
        mbt.material_no,                                           -- 父件料号
        (SELECT cm.bom_version FROM material_bom_item cm           -- ★ 子件"自身"当前版本(叶子→NULL)
          WHERE cm.material_no=mbt.component_no AND cm.customer_no='_GLOBAL_'
            AND cm.system_type='PRICING' AND cm.is_current LIMIT 1)
      )
```
- **含 `:spineKeys` 时执行器"自治",不再追加 `hf_part_no=ANY(:hfPartNos)`**;由 spineKeys 三元组自己圈范围。
- `:spineKeys` 会被框架展开成:
  ```sql
  EXISTS (SELECT 1 FROM unnest(:__skP::text[], :__skPP::text[], :__skV::text[]) AS k(p,pp,v)
          WHERE (子件列) IS NOT DISTINCT FROM k.p
            AND (父件列) IS NOT DISTINCT FROM k.pp
            AND (版本列) IS NOT DISTINCT FROM k.v)  -- NULL-safe
  ```
  三个数组由后端从闭包 spine 注入(`__skP`=子件、`__skPP`=父件、`__skV`=子件自身版本,**叶子=NULL**)。

> **只做"绑定",不做"聚合"**:spineKeys 只是把边行精确挂到节点,不会累乘用量、不会按维度汇总。要"装配件显示子树滚动汇总",见 §8。

## 5. 硬规则(配任何核价树视图都要守)

1. **必出 `hf_part_no` 列,且 = 该行数据所属"节点自身料号"。**
   - 边源表里 = **子件 `component_no`**(不是父件 `material_no`);
   - 后端 `buildSpineBaseRows` 用 `byPart.get(节点.hfPartNo)` 按它配对,写成父件会**塌树 / 节点空**。
2. **A / B 二选一,不要混:**
   - 料号维度表 → 不写 spineKeys,靠自动 `hf_part_no=ANY(partSet)`;
   - 边维度表 → 写 `:spineKeys(...)`(此时执行器不再自动加料号过滤)。
3. **`:spineKeys` 第 3 参 = 子件自身当前版本(语义 B),不是边版本。**
   - 写成 `mbt.bom_version`(= 边/父版本)会让**叶子(自身版本=NULL)匹配不上 → 叶子空**;
   - 正确用子件自身版本的子查询(叶子返 NULL,靠 NULL-safe 命中)。
4. **数据分区必须 `PRICING`(边源另加 `_GLOBAL_`)+ `is_current`**,与闭包口径对齐。
5. **字段 `basic_data_path` 的列名 = 视图 SELECT 输出列名,逐字一致**(改视图列名要同步改字段,见 §6 坑 3)。
6. **改开关 / 视图 / 字段后必须重算报价单**——系统列是算快照那一刻烘进去的,不追溯旧单。

## 6. 三个"固定系统列"(料号 / 父料号 / 版本)

- 它们**不是你在视图里写的**,是后端为递归组件**自动注入**的:
  `__hfPartNo`(料号)、`__parentNo`(父料号)、`__bomVersion`(版本)。
- 前端渲染条件:`核价侧 && 该组件行里有 __sys.nodeId`(即 spine 已建)。满足才**固定带出这三列** + 建树。
- 版本列口径 = **子件自身当前 BOM 版本**,叶子(自身无 BOM)显示**空**——这是设计(不是 bug)。
- **看不到这三列 = 该单快照没有 spine**,几乎都是:①组件开关当时没勾,或②勾了没重算,或③浏览器没刷新。**先重算 + 强刷。**

## 7. 常见坑速查(本项目实际踩过)

| # | 症状 | 根因 | 修法 |
|---|---|---|---|
| 1 | 树塌成一坨 / 根一层就堆很多行 | `hf_part_no` 取了**父件** `material_no` | 改成子件 `component_no` |
| 2 | 叶子节点空、只有中间件有值 | **两因叠加**:①`:spineKeys` 第 3 参用了**边版本** `mbt.bom_version`(叶子自身版本=NULL,对不上);②**后端 bug**(2026-07-02 修):`SqlViewExecutor` 绑数组时 `String.valueOf(null)` 把 NULL 变成字符串 `"null"`,即使第 3 参写对(子件自身版本=NULL)也 NULL-safe 匹配失败 → 叶子被误滤 | ①第 3 参改子件自身版本子查询;②已修后端(null 保留为 SQL NULL,不转 `"null"`) |
| 3 | 某列(或整表)`#ERROR` / 空 | 字段 `basic_data_path` 引用了视图**没有的列名**(改了视图列名没同步字段) | 视图列名 ↔ 字段路径对齐;删无源字段 |
| 4 | 整表 0 行 | 边源 + spineKeys(边版本)把有数据的叶子滤掉,`INNER JOIN` 又把过了 spineKeys 的装配件滤掉(两刀叠加) | 按 §4B 修 spineKeys + hf_part_no;或改 §4A |
| 5 | 大多数产品核价树空 | 数据在 `QUOTE` 域,核价只读 `PRICING` | 把 BOM/业务数据按 `PRICING/_GLOBAL_` 导入 |
| 6 | 改了视图/开关没生效 | 组件有**导入副本重名**(如 `COMP-0021__imp1` vs `__imp1__imp1`),改到了没被模板引用的那个;或视图有同名多行 | 确认模板引用的组件 id;去重同名视图 |
| 7 | 勾了核价树还是没三列 | 旧快照没 spine(勾选前算的)/ 没重算 / 浏览器缓存 | 重算该单 + Ctrl+Shift+R |
| 7b | 导入后首屏是平表、刷新才出树 | 前端首屏 `useSnapCosting=false` 走实时兜底(仅根层),快照后台异步、不自动重拉 | 手动强刷(临时);根治见 BL-0027 |
| 8 | 行数每次刷新变多 / 直接子件重复 | ①driver 行数累加(AP-51);②**边源子件同时挂多父**,无 spineKeys 时按料号过滤把别树的边也捞进来 | ①守 driver 行数权威;②边源必须用 `:spineKeys` 按(子,父,版本)精确对号 |

## 8.（进阶)装配件显示"子树滚动汇总"—— 递归聚合视图

需求:装配件/根节点不只显示自身,而是**把子树里所有原料按用量加权汇总**(如产品整体元素成分)。

- **组件递归(①)不做聚合、spineKeys 也不做聚合**——聚合要在**视图 SQL 里自己写 `WITH RECURSIVE`**:沿 BOM 往下累乘 `composition_qty`、按维度加权求和、归一化。
- 与 §4 的区别:§4 是"每节点显示自身";这里是"每节点显示子树聚合值"。两者都靠组件递归(①)提供骨架 + partSet,只是视图算法不同。
- 示例骨架(元素成分滚动,详见团队内已验证版本):
  ```sql
  WITH RECURSIVE walk AS (
    SELECT part AS node_no, part AS cur_no, 1.0::numeric AS cum_qty FROM <所有料号>
    UNION ALL
    SELECT w.node_no, mbt.component_no, w.cum_qty * COALESCE(mbt.composition_qty,1)
    FROM material_bom_item mbt JOIN walk w ON mbt.material_no=w.cur_no
    WHERE mbt.system_type='PRICING' AND mbt.customer_no='_GLOBAL_' AND mbt.is_current
  ) CYCLE cur_no SET is_cyc USING path,
  agg AS (
    SELECT w.node_no, ebi.component_no AS elem, SUM(w.cum_qty*ebi.content) AS wc
    FROM walk w JOIN element_bom_item ebi ON ebi.material_no=w.cur_no AND ebi.system_type='PRICING' AND ebi.is_current
    GROUP BY w.node_no, ebi.component_no
  )
  SELECT node_no AS hf_part_no, elem AS 元素,
         ROUND(100.0*wc/NULLIF(SUM(wc) OVER (PARTITION BY node_no),0),4) AS 含量, ...
  FROM agg ...
  ```
- 前置:要有加权原语——`composition_qty`(用量,当前多为归一化占比)或 `material_master.unit_weight`(单重)。
- 口径需先定:①按 `composition_qty` 累乘还是引单重真实质量加权;②要不要归一化到 100%。

## 9. 验证方法(DB + UI)

**DB 级(不需登录)——确认快照对不对:**
```sql
-- 某产品某页签的树骨架 + 数据
WITH tab AS (
  SELECT c FROM quotation_line_item li, jsonb_array_elements(li.costing_card_values->'tabs') c
  WHERE li.quotation_id='<报价单id>' AND li.product_part_no_snapshot='<根料号>'
    AND c->>'tabName'='<页签名>' LIMIT 1)
SELECT (r->>'__lvl')::int lvl,
       repeat('  ',(r->>'__lvl')::int-1)||(r->>'__hfPartNo') 料号,
       r->>'__parentNo' 父, r->>'__bomVersion' 版本,
       r->'driverRow' 业务数据
FROM tab, jsonb_array_elements(tab.c->'baseRows') r ORDER BY (r->>'__nodeId');
```
- 看到多层 `__lvl` + `__hfPartNo/__parentNo/__bomVersion` = spine 已建 → 三列会渲染;
- 业务数据格有值 = 视图取数正确。

**闭包引擎自检(某根料号能否展开):**
```sql
WITH RECURSIVE bt AS (
  SELECT '<根料号>'::varchar node UNION ALL
  SELECT ch.component_no FROM material_bom_item ch JOIN bt ON ch.material_no=bt.node
  WHERE ch.customer_no='_GLOBAL_' AND ch.system_type='PRICING' AND ch.is_current AND ch.component_no IS NOT NULL)
SELECT * FROM bt;  -- 出多行 = 有树
```

**UI 级:** 勾选核价树 → **重算该单** → 强刷 → 核价单 → 产品卡片视图 → 目标产品 → 该页签:应见「料号/父料号/版本」三列 + 多层树。

**协议级改动**(动 `BomClosureService` / SQL 视图执行 / 递归开关)按 CLAUDE.md 属协议级,须跑 Playwright E2E(`docs/E2E测试方法.md`)。
