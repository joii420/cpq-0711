# 方案 — 加产品整份快照(报价单展示报价单自己的数据)

> **立项日期**:2026-05-29
> **状态**:Phase 0 设计冻结;**Phase 1-4 已实施并 E2E 验证(2026-05-29)**。渲染已读快照、基础变化不影响已快照行、saveDraft 跨保存重快照、"从基础刷新"保编辑。详见本文 §10 + RECORD.md。
>
> **冻结语义说明(实现取舍)**:saveDraft 全量重建行=新 UUID 且前端不回传行 id,故采用"加产品冻结 + 每次保存/刷新从当前基础重冻、编辑层 row_data 始终保留"(非严格"仅加产品一次冻结")。配合 driver 缓存 evictAll 保证重冻读到当前基础。严格"仅加产品冻结"需让快照随行穿越保存(前端携带或 saveDraft 拷贝),列为后续可选优化。
> **关联**:`docs/三大核心模块基线.md`、`docs/组件级数据源SQL方案.md`、`docs/反模式.md`(AP-31/37/38/44/50/51)、`docs/选配V6迁移诊断方案.md`、`SnapshotCollectorService`(提交快照)。

## 1. 业务设计与原则(已与业务确认)
- **基础表是报价单数据的"初始来源",不是渲染的实时数据源**。
- **加产品(选配确认)那一刻,把该产品所有组件 Tab 的数据整份快照进报价单**;之后报价单的**展示与编辑都读这份 per-quote 副本**,基础表后续变化**不影响**已加的报价单。
- 用户按**组件字段类型**编辑,形成报价单数据,**不回写基础表**。
- **料号身份仍共享去重**(指纹复用,CFG-* 不变);快照的是"展示数据副本",与料号身份是两件事。

## 2. 关键设计决策(评审已确认)
| # | 决策 |
|---|---|
| 适用范围 | **所有组件 Tab**(材质/工序/元素/子件/费用/成本/重量 + 选配-*)|
| 冻结层级 | **driver 展开层冻结整行 basicDataValues**(三类值全冻:`{$view.列}` BASIC_DATA / `@gvar:CODE` DATA_SOURCE / `{基础表 BNF path}`);渲染直返冻结行、跳过实时解析 |
| DATA_SOURCE | **冻结**(工序单价等存快照,全局变量后改不影响本单)|
| FORMULA | **不存,渲染按"冻结输入 + 编辑值"重算** |
| 存储 | 新增 `snapshot_rows`(JSONB);**两层**:基础冻结层(`snapshot_rows`)+ 编辑层(现有 `row_data`)|
| 从基础刷新 | 提供;**只换基础冻结层,保留用户编辑**(故必须两层分开)|
| 生效范围 | 只对**新加/重配**的产品;存量老行无快照 → 回退实时(平滑兼容)|
| 重配某行 | 重新选材质/工序 = 换产品语义 → 重建 `snapshot_rows`**并清 `row_data`(编辑)**;"刷新"才保编辑 |
| 提交收口 | `submission_snapshot` 改为基于 add-time 快照定稿;`DriftDetectionService` 转义为"基础较快照已变,提示可刷新" |

## 3. 对"改组件视图 SQL 查 JSONB"疑问的结论(重要)
最初设想"把组件视图 SQL 改成查 JSONB 快照"。核实后**不走这条路**:
- 一行的值来自**三类解析器合并**进 `basicDataValues`:`{$view.列}`(组件视图 SQL)、`@gvar:CODE`(全局变量服务,**不经视图 SQL**)、`{mat_bom.x}` 等(其它 BNF path 直查基础表,**也不经该组件视图 SQL**)。只改视图 SQL 只能冻结第一类,DATA_SOURCE/其它 BNF 冻结不住。
- **正确做法**:加产品时把整行全部解析值冻进 `snapshot_rows`;渲染时 `ComponentDriverService.expand` **直接反序列化 JSONB 返回冻结行**,不跑视图 SQL、不做 JSONB 关系化 unwrap、不实时解析全局变量/BNF。
- **对 BNF path 渲染零影响**:前端仍 `basicDataValues[normalizedPath]` 逐格取值(key 不变),只是来源从"实时展开"变"冻结快照";JSONB 天然保留数字/字符串类型,无需 `::numeric` 转型。
- 组件视图 SQL **保留**,只在加产品/刷新时作"种子"跑一次。

## 4. 数据模型(两层)
`quotation_line_component_data`(grain = 报价行 × 组件 × Tab)新增:
- `snapshot_rows JSONB` —— 基础冻结层:整组行的全部解析值 `[{driverRow:{...}, basicDataValues:{...}}, ...]`(= `ExpandDriverResponse.rows` 形状)。加产品/刷新时写。
- `snapshot_at TIMESTAMPTZ` —— 快照时间。
- `row_data`(现有)—— 编辑层:用户 INPUT 值。
- 渲染有效值 = **编辑值优先,否则取冻结基础值**;FORMULA 由二者重算(沿用 `QuotationStep2` 现合并逻辑,仅把 basicDataValues 来源换成 `snapshot_rows`)。
- 刷新 = 重算覆盖 `snapshot_rows`,不动 `row_data`。

## 5. 关键流程
- **加产品 `configure`**:建行后,读 `quotation.customer_template_id → template.components_snapshot`,遍历各组件 → `ComponentDriverService.expand(componentId, customerId, partNo, …, lineItemId, compositeType)` → 把 `rows` 写入 `snapshot_rows`(per 行×组件×Tab)。
- **渲染(Phase 2)**:`expand` 先查 `snapshot_rows`(按 lineItemId×componentId);命中直返,未命中(老行)回退实时。
- **编辑**:写 `row_data`。**刷新**:重跑 expand 覆盖 `snapshot_rows`,保 `row_data`。
- **提交(Phase 3)**:`submission_snapshot` 基于已存快照定稿。

## 6. 分阶段
- **Phase 0 设计冻结** — 完成(本文档)。
- **Phase 1 写快照(加性、低风险)** — configure 加产品时展开全部组件并写 `snapshot_rows`;**渲染不变**;做"快照值 vs 实时值"一致性对账。快照写入失败**不阻断加产品**(try/catch 降级)。
- **Phase 2 渲染读快照(行为变更,强制 E2E)** — `expand` 命中快照直返;前端不再实时 batch-expand(未命中回退);缓存键加报价行维度。
- **Phase 3 编辑/刷新/收口** — 编辑写编辑层;"从基础刷新"(保编辑);`submission_snapshot` 收口 + `DriftDetectionService` 转义。
- **Phase 4 清理** — 渲染路径退役实时基础查询;`ComponentDriverService` 降级为种子展开。

## 7. 影响面
- **后端**:`ConfigureProductService.configure`(加快照,Phase 1)、`ComponentDriverService.expand`(命中快照直返,Phase 2)、`QuotationService` load/save(两层 merge)、`SnapshotCollectorService`(收口)、`DriftDetectionService`(转义)、`ExcelViewService`。
- **前端**:`useDriverExpansions`(读快照)、`QuotationStep2`(basicDataValues 来源换快照,合并逻辑不变)、autosave(写编辑层)、刷新按钮、`ReadonlyProductCard`/`QuotationDetail`/`ExcelView`。
- **DB**:`quotation_line_component_data` 加 `snapshot_rows`/`snapshot_at`。
- **性能**:加产品变重(全组件展开一次),渲染变轻(读 JSON 不实时查/不解析全局变量),净收益在渲染。
- **存储**:每行存整组冻结值,体量上升。

## 8. 风险/回滚/验证
- 渲染链路核心区(AP-31/37/38/44/50/51),协议点多 → 分阶段 + 开关/灰度 + 全程 E2E。
- Phase 1 纯加性(只写不读)→ 可随时停;快照写入用 try/catch 降级,绝不阻断加产品。
- Phase 2 行为切换设开关,出问题回退实时链路。
- FORMULA 重算口径需对账(避免金额漂移)。
- 存量单只新加生效,免批量回溯风险。
- 验证:`quotation-flow` + `composite-product-flow` 不回归;Phase 1 加"快照已写 + 快照值=实时值"对账。
