# 同模板双轨支持组合产品

> **2026-05-20 立项**。单一报价模板（v1.11 等）同时支持 **SIMPLE 独立产品 + COMPOSITE 组合产品** 两种产品形态。核心机制：**字段级双轨存储 + 运行时按 lineItem.compositeType 切换有效 path**。无 Flyway，向后兼容。
>
> **TL;DR — 三句话**：
> 1. JSONB 内单字段加 `_composite` 后缀（`basic_data_path_composite` / `dataDriverPathComposite`），SIMPLE 用原字段，COMPOSITE 用 `_composite` 后缀。
> 2. 前端 `useDriverExpansions.tasks` / `computeAllFormulas` / cell render 按 `lineItem.compositeType` 切换 effective path；后端**完全透明**（前端发送时 path 已确定）。
> 3. 数据修复走 admin endpoint `POST /api/cpq/templates/admin/{id}/patch-composite`，给 PUBLISHED 模板的 `template_component.fields_override` 注入 `_composite` 后缀字段。

---

## 一、设计背景

### 1.1 为什么需要双轨

CPQ 报价单产品分两种形态：

- **SIMPLE 独立产品** — 1 个 lineItem 对应 1 个料号 (`hf_part_no`)。字段值取自 `mat_xx` 单子件表（如 `mat_process`、`mat_bom`）。
- **COMPOSITE 组合产品** — 1 个父 lineItem + N 个子 PART lineItem。父级 `hf_part_no=CFG-COMBO-*`，**在 mat_xx 表自己无行**。子件数据需要通过聚合视图 `v_composite_child_*` 按父级反查 N 子件聚合。

**旧设计的两种失败**：

1. **模板 driver 配死成 mat_xx**（V194 之前）：组合产品下父级查不到数据 → 所有"选配-*" Tab 显示空或报错（→ 渲染层"加载中"或 `process_default_cost 不存在`）。
2. **V195 SQL 强制改 snapshot driver 为 `v_composite_child_*`**（V194~V195 引入聚合视图后）：组合产品下渲染对，但**独立产品下也走聚合视图 → 父级 hf_part_no 在视图无行 → 渲染空**。绕过组件管理 UI 直接改 snapshot 还导致用户在"组件管理"里看不到这 4 个 Tab 配置（AP-40 命中之一）。

**两难**：单一 driver 配置无法同时满足两种产品形态。

### 1.2 双轨方案对比（曾考虑的 3 个）

| 方案 | 优点 | 缺点 | 状态 |
|---|---|---|---|
| **A 字段双轨**（加 `_composite` 后缀）| 改动最小、向后兼容、schema 不变 | schema 表面变胖（每字段双份） | ✅ **采用** |
| B tc-variant 整体变体 | 表达力最强、可在两种产品下用差异化字段集 | 后端 H1 + 前端 enrich 都要大改、工作量翻倍 | ❌ 否决 |
| C 路径模板字符串运行时插值 | schema 不变、单字段表达两场景 | 路径语法变复杂、调试不直观、反模式检测难 | ❌ 否决 |

## 二、存储格式（字段双轨）

### 2.1 数据库层

**无 Flyway 改动** — `template_component.fields_override` 和 `template.components_snapshot` 都是 JSONB，新增键即可。

### 2.2 JSONB 字段结构

#### `template_component.fields_override` / `template.components_snapshot[i].fields` 内每个字段：

```jsonc
{
  "name": "工序代码",
  "field_type": "BASIC_DATA",
  "basic_data_path": "mat_process.process_code",                    // SIMPLE 用
  "basic_data_path_composite": "v_composite_child_processes.process_code",  // COMPOSITE 用
  "content": "",
  "is_amount": false,
  "is_subtotal": false
}
```

#### `template_component` / `template.components_snapshot[i]` 顶层：

```jsonc
{
  "componentId": "...",
  "tabName": "选配-工序列表",
  "data_driver_path": "mat_process",                                // SIMPLE 用
  "data_driver_path_composite": "v_composite_child_processes",      // COMPOSITE 用
  "fields": [ ... ],
  ...
}
```

### 2.3 字段缺省行为（向后兼容）

| 场景 | basic_data_path | basic_data_path_composite | 行为 |
|---|---|---|---|
| 老配置 | 非空 | **缺失** | SIMPLE/COMPOSITE 都用 `basic_data_path`（fallback，行为同旧） |
| 双轨升级 | 非空 | 非空 | SIMPLE 用 `basic_data_path`，COMPOSITE 用 `basic_data_path_composite` |
| 仅 composite | **空** | 非空 | SIMPLE 显示 "—"（路径不存在），COMPOSITE 正常 |
| 全空 | 空 | 空 | 渲染 "未配置路径" |

数据驱动路径 (`data_driver_path` / `data_driver_path_composite`) 同此规则。

### 2.4 "子件"字段约定（关键）

组合产品 Tab 需要"子件"列识别"这行属于哪个子件"。SIMPLE 独立产品则不需要（它只有一个料号）。设计：

```jsonc
{
  "name": "子件",
  "field_type": "BASIC_DATA",
  "basic_data_path": "mat_part.part_no",                            // SIMPLE: 反查自身 part_no
  "basic_data_path_composite": "v_composite_child_materials.child_part_name"  // COMPOSITE: 各子件名
}
```

SIMPLE 模式下 `mat_part.part_no` 按 `hf_part_no` 隐式 JOIN 注入 → 返自身料号；COMPOSITE 模式下走聚合视图返 N 子件名。

## 三、运行时切换逻辑（前端）

### 3.1 切换条件

```ts
const isComposite = lineItem.compositeType === 'COMPOSITE';
```

`lineItem.compositeType` 三种取值：
- `'SIMPLE'` — 独立产品（默认）
- `'COMPOSITE'` — 组合产品父级（**双轨切换**）
- `'PART'` — 组合产品子件 lineItem（前端 filter 不渲染卡片，AP-37 根因 6）

### 3.2 协议传播点（AP-44 矩阵扩展 — 15 处 → 17 处）

任何字段类型变动 / 双轨字段新增/修改必须同步以下 17 处（在 AP-44 矩阵 15 处基础上 +2）：

新增第 ⑯ 处：**`useDriverExpansions` tasks + fingerprint 按 compositeType 切换 effective driver/fields**

```ts
// useDriverExpansions.ts
const isComposite = (item as any).compositeType === 'COMPOSITE';
const effectiveDriver = isComposite
  ? (compAny.dataDriverPathComposite || comp.dataDriverPath || undefined)
  : (comp.dataDriverPath || undefined);
const effectiveFields = (isComposite && Array.isArray(comp.fields))
  ? (comp.fields as any[]).map((f: any) =>
      f.basic_data_path_composite ? { ...f, basic_data_path: f.basic_data_path_composite } : f)
  : comp.fields;
```

新增第 ⑰ 处：**`QuotationStep2` 渲染层路径切换 (cell + computeAllFormulas + computeTabSubtotal)**

```ts
// 渲染 BASIC_DATA cell
```

## 四、数据迁移 SOP（PUBLISHED 模板修复）

### 4.1 admin endpoints

#### `POST /api/cpq/templates/admin/{templateId}/delete-tcs`
**用途**：删 PUBLISHED 模板的某些 Tab（如 V195 自动写入的"标准 Tab"）。绕过 `TemplateService.update` 的 DRAFT-only 限制。

**Body**：
```json
{ "sortOrders": [0, 1, 2, 4] }
```

**响应**：`{ deletedTcs, snapshotBefore, snapshotAfter }`

**权限**：`SYSTEM_ADMIN` 专用。

#### `POST /api/cpq/templates/admin/{templateId}/patch-composite`
**用途**：给 PUBLISHED 模板某 tc 注入双轨字段 + composite driver。

**Body（两种模式）**：

模式 A — patch 模式（只 patch basic_data_path_composite 到匹配字段）:
```json
{
  "tcId": "...",
  "dataDriverPathComposite": "v_composite_child_processes",
  "fieldComposites": [
    {"name": "工序代码", "basicDataPathComposite": "v_composite_child_processes.process_code"}
  ]
}
```

模式 B — replace 模式（整体替换 fieldsOverride，**加新字段时用**）:
```json
{
  "tcId": "...",
  "dataDriverPathComposite": "v_composite_child_processes",
  "fieldsOverride": [
    {
      "name": "子件",
      "field_type": "BASIC_DATA",
      "basic_data_path": "mat_part.part_no",
      "basic_data_path_composite": "v_composite_child_processes.child_part_name"
    },
    ...
  ]
}
```

**响应**：`{ tcId, dataDriverPathComposite, fieldsCount, replaceMode }`

**权限**：`SYSTEM_ADMIN` 专用。

### 4.2 完整数据迁移示例（v1.11 模板）

## 五、聚合视图清单（V194 引入）

| 视图 | 用途 | 关键列 |
|---|---|---|
| `v_composite_child_materials` | 跨子件材质字典 | `hf_part_no`(父) / `child_part_name` / `material_code` / `chemical_symbol` / `material_name` / `spec_label` / `recipe_type` |
| `v_composite_child_processes` | 跨子件工序聚合 | `hf_part_no`(父) / `child_part_name` / `seq_no` / `process_code` / `assembly_process` |
| `v_composite_child_elements` | 跨子件元素 BOM | `hf_part_no`(父) / `child_part_name` / `seq_no` / `element_name` / `composition_pct` |
| `v_composite_child_weights` | 跨子件单重 | `hf_part_no`(父) / `child_part_name` / `unit_weight` |

视图按父级 `hf_part_no` 查所有子件聚合行。`ImplicitJoinRewriter` 自动注入 `WHERE hf_part_no=...` 谓词。

### 视图谓词注入要求

要触发自动 JOIN，视图必须含 `hf_part_no` 列（语义=父级）。聚合视图实现时 SQL 类似：

```sql
CREATE OR REPLACE VIEW v_composite_child_processes AS
SELECT
  mcp.hf_part_no AS hf_part_no,           -- 父级
  mcp.child_part_name,
  mp.seq_no,
  mp.process_code,
  mp.assembly_process
FROM mat_composite_part_part mcp
JOIN mat_process mp ON mp.hf_part_no = mcp.child_hf_part_no
WHERE mp.is_current = true;
```

## 八、历史命中事故

### 事故 1：V195 自动 SQL 写入"标准 Tab" → 用户在组件管理看不到（2026-05-18 ~ 2026-05-20）

**触发**: V195 `UPDATE template SET components_snapshot = ...` 用 SQL 把"材质/工序/元素含量/组合工艺" 4 个 Tab 的 fields_override 改成 `v_composite_child_*` 驱动 + 字段路径。绕过组件管理 UI。

**结果**: 用户配的 4 个"选配-*" Tab 在组合产品下渲染错（driver=mat_xx 父级查不到），但 V195 加的 4 个"标准 Tab"渲染对。**用户感知**："系统自己加了 4 个我没配的 Tab，组件管理看不到，但我配的 4 个又不对"。

**修复**: 双轨方案落地（2026-05-20）：
1. 删 4 个 V195 自动 Tab
2. 给 4 个用户配的"选配-*" tc 注入双轨字段 + composite driver
3. 单一 v1.11 模板同时支持 SIMPLE + COMPOSITE

### 事故 2：H1 `firstResult()` 反向污染（AP-40）

**触发**: 用户改 component fields 触发自动 H1 sync，但 H1 `TemplateComponent.find(...).firstResult()` 只拿第一个 tc，用它的 fieldsOverride 覆盖所有同 cid snapshot entry。

**结果**: 4 个 cid 跨标准/选配-* 双 Tab，反向污染让用户配置丢失。

**修复**: AP-40 — H1 按 sortOrder 精确匹配 tc。详见 `docs/反模式.md AP-40`。

### 事故 3：v1.10 多产品场景 — driver path snapshot-only 字段被刷掉（AP-48, 2026-05-20）

**触发**: 用户给 v1.10 选配-* tab 通过 admin patch-composite 设了 `data_driver_path_composite=v_composite_child_*`, 数据正确写入 snapshot entry; 但 `refreshSnapshotsByComponent` 触发后 entry 字段被抹掉, COMPOSITE driver 失效, 产品 2 渲染全 "—"。

**结果**: `data_driver_path_composite` 只存在 snapshot 内, 没有源列 → publish/refresh 重建 snapshot 时无法保留。

**修复**: 加 `template_component.data_driver_path_composite TEXT` 列 (Flyway V205), 4 处协议传播点同步写 (publish / createNewDraft / refresh / admin patch-composite)。详见 `docs/反模式.md AP-48` + 本文 §四「双轨二期 (V205)」。

## 十、双轨二期 (V205) — 2026-05-20

### 10.1 LIST_FORMULA `formula_composite` 双轨

**背景**: V200 双轨只覆盖 BASIC_DATA path, LIST_FORMULA 字段的 `branches[i].formula` 仍单轨。COMPOSITE 产品视角下 partNo=CFG-COMBO-XXX 在物理表里查不到, BNF 公式 (如 `{mat_bom[...].net_qty}*15`) 必然返 0 → 业务上必须给组合视角配不同公式 (如用 `v_composite_child_*` 视图字段)。

**Schema 扩展**:

```jsonc
{
  "list_formula_config": {
    "per_item_rules": {
      "MRO-AS-0001": {
        "branches": [{
          "condition": "[工序代码] = 'MRO-AS-0001'",
          "formula":           "{mat_bom[element_name='Sn'].net_qty}*15",  // SIMPLE
          "formula_composite": "100"                                       // COMPOSITE (新增)
        }],
        "default_formula":           "",
        "default_formula_composite": ""  // (新增, COMPOSITE 默认值兜底)
      }
    }
  }
}
```

**运行时选择** (`QuotationStep2.tsx:1470`):

```ts
chosenFormula = (isCompositeItem && b.formula_composite)
  ? b.formula_composite
  : b.formula;
```

缺失 `formula_composite` 时**自动 fallback** 到 `formula`，向后兼容旧 LIST_FORMULA。

**协议传播 4 处**:

| # | 文件 / 函数 | 改动 |
|---|------|------|
| 1 | `QuotationStep2.tsx` cell render LIST_FORMULA 分支 | 按 `isCompositeItem` 选 `formula_composite ?? formula` (branch + default 都要选) |
| 2 | `usePathFormulaCache.collectListFormulaBnfPaths` | 扫 `branches[*].formula` + `formula_composite` + `default_formula` + `default_formula_composite` |
| 3 | `ComponentDriverService.collectBnfPathsFromFormula` (后端) | 同样扫 4 处 |
| 4 | `TemplateService.mergeListFormulaConfigComposite` | createNewDraft 时按 rule code + branch index 保留 `formula_composite` (不被 component 改动冲洗) |

### 10.2 `template_component.data_driver_path_composite` 列

**背景**: V200 设计 `data_driver_path_composite` 仅存于 `components_snapshot.entry`, 没有源列 → publish/refresh 重建 snapshot 时无法恢复 (AP-48)。

**Flyway V205**:

```sql
ALTER TABLE template_component
    ADD COLUMN IF NOT EXISTS data_driver_path_composite TEXT;
```

**Entity 字段** (`TemplateComponent.java`):

```java
@Column(name = "data_driver_path_composite", columnDefinition = "text")
public String dataDriverPathComposite;
```

**4 处协议传播点**:

| # | 函数 | 行为 |
|---|------|------|
| 1 | `publish()` | 把 `tc.dataDriverPathComposite` 写入 `snapshot.entry.data_driver_path_composite` |
| 2 | `createNewDraft()` | 拷贝 `stc.dataDriverPathComposite → newTc.dataDriverPathComposite` |
| 3 | `refreshSnapshotsByComponent()` | 重建 snapshot 时**从 tc 列读, 重新写入 entry** (而不是仅依赖 entry 旧值) |
| 4 | `patchTemplateComponentCompositeOverrides()` (admin endpoint) | 同时落 tc 列 + 写 snapshot entry (双写, 让后续 publish/refresh 不丢) |

### 10.3 createNewDraft 合并 fields_override (AP-39 架构修法)

**用户场景**: 改组件公式 → 创建新版本模板 → 期望新版本含组件最新公式 + 保留模板独有的 _composite 双轨字段。

**旧行为** (AP-39): `createNewDraft` 把 `stc.fieldsOverride` 整段拷给新 DRAFT, 组件改动永远进不来。

**新行为** (2026-05-20 实现, `mergeFieldsOverrideForNewDraft`):

```
对 fields_override 数组每个字段, 按 name 索引匹配 component.fields:
  1. 以 component.fields 顺序为基础, 每字段拿组件最新所有属性
  2. 保留旧 override 的顶层 `_composite` / `Composite` 后缀字段 (V200 双轨)
  3. LIST_FORMULA 字段深入 list_formula_config 嵌套合并 `formula_composite` /
     `default_formula_composite` (按 rule code + branch index)
  4. 兜底: 旧 override 里有 + 组件里没的字段, 整字段保留 (V195 历史 SQL 兜底)
```

**用户工作流升级**:

```
1. 组件管理 改 LIST_FORMULA 公式 → 应用 → 保存组件
   (此时 H1 自动 refreshSnapshotsByComponent 触发, 但只影响 components_snapshot,
    不影响 tc.fields_override — 这是设计意图保留模板硬覆盖)

2. 模板管理 → 创建新版本 (POST /templates/{id}/new-draft)
   → 后端调用 mergeFieldsOverrideForNewDraft
   → 新 DRAFT 的 tc.fields_override = component.fields 最新值 + 双轨保留

3. 检查 DRAFT (无需手工改公式 — 已自动同步)

4. 发布 (POST /templates/{id}/publish)
   → snapshot 重建用 tc.fields_override (含新公式) → snapshot 含新公式

5. 新报价单选新版本 → 渲染走新公式
```

### 10.4 双轨二期 PR 自检清单

新增 / 改动 LIST_FORMULA 字段或 driver 双轨配置时:

- [ ] 公式双轨: 给 `branches[i].formula_composite` + `default_formula_composite` 也填值 (缺失会 fallback formula, 兼容; 但对 COMPOSITE 视角无意义)
- [ ] driver 双轨: 设 `tc.data_driver_path_composite` (列, 非 snapshot 内字段)
- [ ] admin patch-composite endpoint 调完后**用 component-refresh 验证** 不会丢失双轨字段
- [ ] `createNewDraft` 后检查新 DRAFT 的 `fields_override.list_formula_config.formula_composite` 是否保留
- [ ] E2E `multi-product-flow.spec.ts` 跑通 (产品1 独立 + 产品2 组合, 8 Tab × 2 = 16 cell 都无 "加载中" "(共 N 项)" "#ERROR")
- [ ] 自检后端 `expand-driver` 返的 `row.basicDataValues` 含 LIST_FORMULA 内 BNF path key (含花括号 `{table.col}`)
