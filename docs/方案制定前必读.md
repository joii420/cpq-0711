## 🚨 V6 基础资料表使用规则（2026-05-26 立项，禁止违反）

> **背景**：2026-05-26 起 V218~V222 落地 23 张 V6 基础资料表，V44 `mat_part / mat_bom / mat_process / mat_fee / plating_plan / mat_customer_part_mapping / element_price / element_price_source / element_price_fetch_rule / element_daily_price / customer_tax` 等老表**全部废弃**。

### 强制规则
1. **基础数据落库**：新数据全部走 V6 表（material_master / material_bom / material_bom_item / element_bom / element_bom_item / unit_price / capacity / plating_scheme / process_master / fee_config / exchange_rate_v6 / material_customer_map / material_version_mgmt / labor_rate / annual_discount / equipment / production_energy / auxiliary_energy / tooling_cost / production_consumable / packaging_consumable / electricity_price / resource_group），**严禁**写入 V44 mat_* 老表。
2. **组件级 `data_driver_path` 禁用直接 PG 视图名**：必须用 SQL 视图引用语法 `$<sql_view_name>` 或 `$$<componentCode>.<sql_view_name>`（参 `docs/archive/组件级数据源SQL方案.md`）。**禁用** `v_composite_child_processes` / `mat_bom[bom_type='ASSEMBLY']` 这种直接表名/视图名。
3. **字段级 BNF 路径同样禁用直接表名**：`basic_data_path` 必须用 `$<sql_view_name>.<col>`，**禁用** `mat_process.process_code` 这种直接表名引用。
4. **SQL 视图（component_sql_view.sql_template）内部禁止查询 V44 表**：所有 mirror / wrapper SQL 必须 FROM V6 表。已存在但 SQL 模板还在查 mat_* 的视图（如 4 个 *_mirror）必须按 V228+ 迁移重写。
5. **V6 表没有 `quotation_line_item_id` 维度**：V6 设计是 customer × material 共享数据，同客户同料号在多张报价单看到相同基础数据。如需 per-报价单差异化，须走 quotation_line_item 自身的字段（如 productAttributeValues），不在基础资料表上做隔离。
6. **导入服务**：报价基础数据 19 Sheet → V6 表（`com.cpq.basicdata.v6.quote`），核价 24 Sheet → V6 表（`com.cpq.basicdata.v6.pricing`），**禁止**新数据写入 `BasicDataImportServiceV5` 链路（已 @Deprecated）。
7. **🆕 任何运行时搜索 / 取数 endpoint 一律走 V6 表**（2026-05-26 选配 Step1+Step2 迁移后加）：
   - 包括但不限于 ConfigureSearchResource / MaterialRecipeService / CustomerPartCandidateService / 任何 native SQL 直查 `mat_*` 或 `material_recipe*` 的 Resource
   - 新增 endpoint 时如发现 `FROM mat_part / FROM material_recipe / FROM mat_bom` 必须立即迁移到 `material_master / element_bom_item / material_bom_item`
   - 不允许「先写 V44 后续再迁」的渐进策略 — 渐进会产生 V6 数据 + V44 endpoint 错配的死链路（QT-20260526-1629 / 选配 Step1 类）
   - 例外：`material_recipe` 字典管理页（`/material-recipes` admin route，配置中心）暂保留 V44 直至单独决策是否废弃
8. **🆕 element_bom_item.hf_part_no 列（V245 加）**：Q04 Sheet 第 1 列「宏丰料号」必须导入到此列；mirror SQL + getForExistingPart 都按 `hf_part_no` 维度查；`material_no` 字段保留「投入料号」语义。**禁止**再省略「宏丰料号」列。
9. **🆕 material_bom_item 维度方向铁律**（2026-05-27 选配 Step1 修正后立项）：写涉及该表的 NOT EXISTS / JOIN / IN 子查询前必须明确**业务诉求方向**：
   - `material_no` = 父件维度（Q12 写「宏丰料号」=主件，如 3120012574）
   - `component_no` = 子件维度（Q12 写「组成件料号」=子件，如 8881-8885）
   - 「独立产品搜索」(顶层报价产品) 业务诉求是「保留父件 + 独立料号、排除已被装配的子件」→ `WHERE NOT EXISTS asy.component_no = mm.material_no`
   - 反向 `WHERE NOT EXISTS asy.material_no = mm.material_no` 会错排父件 — 这是 2026-05-27 实际撞到的坑
   - 任何「类 SIMPLE 过滤」语义都不能机械照搬 V44 `product_type` 写法 — V6 无该列，必须通过本表维度方向推导
10. **🆕 选配/渲染 SQL 视图必须按 customer_no 过滤铁律**（2026-05-27 "页签数据两份" 后立项）：
   - V6 表是 customer × material 共享，**同一料号被多客户导入会各存一份**。任何按料号查 V6 表（material_bom_item / element_bom_item / unit_price 等）的 SQL 视图，WHERE **必须同时过滤** `customer_no = :customerCode`，否则跨客户数据叠加 → 页签数据出现 N 份（N=导入过该料号的客户数）
   - `SqlViewExecutor` 只自动注入 `hf_part_no = ANY(:hfPartNos)`，**customer 过滤必须视图 SQL 自己写** `customer_no = :customerCode`
   - 占位符已由后端注入：`:customerId`(quotation.customerId UUID) + `:customerCode`(SqlViewExecutor.enrichCustomerCode 从 UUID 解析 customer.code，如 CUST-1269)。V6 表 `customer_no` 列存 **code 不是 UUID**，所以视图过滤用 `:customerCode`
   - 已落地：zcj_bom + 4 个 composite_child_*_mirror（V263）。新建任何选配/渲染视图照此办理
   - dry-run 兼容：SqlViewValidator 把 :customerCode 替换为 NULL，customer_no=NULL 语法合法不报错

### 例外
- V44 老表本身保留（含历史种子数据），可读不可写，作历史追溯。
- 老 PG 视图 `v_composite_child_*` 暂保留作历史报价单兼容查询，新单不再使用。

### 🚨 严格 BNF 通道规则（2026-05-26 加，AP-53 续）
**BNF path 查询只能使用组件配置的 SQL 视图（`component_sql_view` 表），禁止用 PG view**。

1. **组件级 `data_driver_path`** 必须用 `$<view_name>` / `$$<componentCode>.<view_name>` 形式（SqlViewExecutor.executeAllRows 内部执行 mirror SQL 返整行集）。**禁用** `v_composite_child_processes` / `mat_bom[...]` 这种直接 PG 表名/视图名。
2. **字段级 `basic_data_path`** 必须用 `$<view_name>.<col>` / `$$<componentCode>.<view_name>.<col>` 形式。从 driverRow 短路取值是**合法的**——因为 driverRow 本身来自 mirror SQL（V235+V236 后），短路只是性能优化，不破坏 mirror-only 单一数据真相源。
3. **SqlViewExecutor 已加 `executeAllRows`**（V236）支持无列名形态的 driver path；DataLoader.loadByPath 检测 `$<view>`（无列名）→ executeAllRows，`$<view>.<col>` → execute。
4. **PG `::cast` 与命名占位符冲突**：SqlViewExecutor.NAMED_PARAM 用负 lookbehind `(?<!:):` 排除 `::uuid` 等 cast 误判（V236 修）。
5. **跨组件引用**：当组件 A 想用组件 B 的 mirror 视图时，B 的 mirror 必须 `scope='GLOBAL'`，A 的路径写 `$$<B-componentCode>.<view_name>`。例：CHILD-PARTS 复用 MATERIAL 的视图 → driver path = `$$COMP-CFG-MATERIAL-RECIPE.composite_child_materials_mirror`。

### 违反后果案例
- **QT-20260526-1629 工序列表加载为空**（2026-05-26）：模板组件 data_driver_path=v_composite_child_processes（V44）+ mirror SQL 模板查 mat_process（V44）+ V6 导入只写 V6 表 → 渲染数据源完全不通。详见反模式.md AP-53。
- **driver path 用 `$<view>` 时字段值变 List 而非标量**（2026-05-26）：早期实现 SqlViewExecutor.execute 只支持 `$<view>.<col>` 含列名形态，无列名形态报"非法 SQL 视图路径语法"。修复见 V236 + SqlViewExecutor.executeAllRows。
- **mirror SQL 模板含 `::uuid` cast 被命名占位符正则误识为 `:uuid` 占位符**：NAMED_PARAM 改 `(?<!:):` 负 lookbehind。修复见 V236（执行）+ 同日同源修复 SqlViewValidator（校验/dry-run 通路漏修，双链路必须同步）。
- **选配-元素含量永远 0 行**（2026-05-26）：Q04 Handler 按方案文档 §4「宏丰料号不导入」决定丢弃主件维度 → `element_bom_item.material_no` 只存投入料号 → mirror SQL 按 `lineItem.partNo=3120012574` 查永远查不到。修复见 V245 加 `hf_part_no` 列 + V246 `characteristic=MAX` 过滤 + Q04Handler 同步写 hfPartNo + 官方方案文档撤销「不导入」决定。
- **选配添加 → 独立产品搜索查不到 V6 数据**（2026-05-26）：ConfigureSearchResource.searchParts / MaterialRecipeService.getForExistingPart 两个选配 Drawer 数据入口仍直查 V44 mat_part + material_recipe + mat_bom。V228+ 系列只迁了组件 driver_path + mirror SQL，遗漏选配 Drawer 链路。修复后两 endpoint 改查 material_master + element_bom_item（V246 同口径）。新规则 §7「任何运行时搜索 / 取数 endpoint 一律走 V6」立项以防再有遗漏。

## 二、按改动类型的决策树

### 改动 1：**新增字段类型 / 改 field_type 枚举**
- ❗强制：先读 `docs/配置方法论-合并版.md §11 字段类型联动协议（AP-44 核心规范）`（17 处协议传播点）
- 改动清单 grep 全工程，按矩阵勾掉每格
- E2E 双 spec 强制（quotation-flow + composite-product-flow）
- PR 必含 17 处 grep 命中输出 + E2E loading=0 + 三视图（报价单/核价单/详情页）截图

### 改动 2：**改 fallback 链 / 渲染分支**
- ❗强制：先读 AP-50 + 三大核心模块基线.md §4 报价单渲染章节
- 修法必须**两端同步**（ComponentCell 抽出后 ReadonlyProductCard + ProductCard 都过它）
- 禁止：单端改完不同步另一端 — AP-44 矩阵 #14/#15 已合一，改 1 处即可
- 自检：必须用 readonly=true 和 readonly=false 两个场景手测

### 改动 3：**改 driver expansion / batchExpand 链路**
- ❗强制：先读 AP-37 + AP-51 + CLAUDE.md §5.5
- driver 行数权威纪律：rowCount 必须取 `expansion.rowCount`，禁止 `Math.max(expansion.rowCount, baseRows.length)`
- cache key 6 维不能省（lineItemId / partNo / componentId / customerId / dataDriverPath / fieldsHash）
- enrich 完成后必须 invalidateDriverExpansions(partNos)
- 写后端 mat_* 流程后必须 evictAll() ComponentDriverService 缓存

### 改动 4：**改全局变量 / GV 解析 / @gvar:CODE 链路**
- ❗强制：先读 AP-49 + AP-52 + 全局变量使用指南.md
- 前后端契约：`basicDataValues['@gvar:CODE']` 是唯一 key 命名空间（不要再发明 BNF path 格式）
- evaluateExpression 的 global_variable token 必须**先查 @gvar:CODE，再 fallback BNF path**
- KV_TABLE 类 GV 字段必须显式声明 `datasource_binding.key_field_refs`（不依赖 findAliasValue 探测兜底）
- 字段绑 global_variable_code 时必须确认 GV.value_column 与字段语义对齐（如"单位"字段不能绑 value_column=costing_price 的 GV）

### 改动 5：**改 snapshot / 模板字段元数据 / refreshSnapshotsByComponent**
- ❗强制：先读 AP-39 + AP-40 + 三大核心模块基线.md §3.3
- Flyway 迁移必须**同时改 3 个层**：component.fields + template_component.fields_override + template.components_snapshot
- 同 cid 多 tc 实例必须按 sortOrder 精确匹配（AP-40 firstResult 反向污染坑）
- 重启 Quarkus 验证 snapshot 各 Tab fields_override 独立保留

### 改动 6：**改 ImplicitJoinRewriter / BNF path 求值 / 视图 schema**
- ❗强制：先读 CLAUDE.md §视图 DROP CASCADE / 重建后必须重启 Quarkus
- tableColumnsCache 是进程级 ApplicationScoped 缓存 — schema DDL 后必须重启清空
- 优先用 @gvar:CODE 通道绕开 ImplicitJoinRewriter（gvar task 直查 GV SPI，不依赖 tableColumnsCache）
- 不要手工 psql -f Vxxx.sql — 让 Quarkus dev mode 自动跑 Flyway
- **模板 Excel 视图列路径**改动同样适用：调 `/formulas/(batch-)?evaluate` 时若 Excel 列 `variable_path` 引用了 `$view.col`（模板 SQL 视图路径），前端必须携带 `templateId`，后端 `SqlViewRuntimeContext` 注入 ownerType=TEMPLATE 才能路由到 `template_sql_view` 表。`FormulaEvaluateResource` 已支持 `EvaluateRequest.templateId` 字段；前端 `LinkedExcelView.batchEvaluate` 已透传；漏传 templateId 会导致 `SqlViewExecutor` 找不到视图而返回 `null`（显示"—"）

### 改动 7：**改 autoSave / snapshotRows / buildDraftPayload**
- ❗强制：先读 AP-51 + RECORD.md L5193 driver 行迭代规范
- 持久化的 row 数据 = 渲染**值**（公式结果 / 用户输入 / driver 解析的 BASIC_DATA），**禁止持久化行数维度**
- React state 的 comp.rows.length **永远 ≤ expansion.rowCount**（多余行 prune 截短）
- snapshotRows / computeTabSubtotal / 渲染层迭代器 **全部按 expansion.rowCount**

### 改动 9（HTML 原型 / 静态文档）：**模板字符串里的 HTML 解析陷阱（双坑）**

#### 坑 1: `inline onclick` 嵌套转义
- ❌ **禁止**在 ES6 模板字符串 (\`...${expr}...\`) 内嵌的 HTML `onclick="..."` 属性里用 `\\'` 嵌套转义
- 反例（会抛 SyntaxError 整段 script 失效）：
  ```js
  `<a onclick="${cond ? 'showToast(\\'msg\\')' : 'doX(\\''+id+'\\')'}">`
  ```
- 正例：把条件分支逻辑挪到函数内部，inline onclick 只传简单参数
  ```js
  `<a onclick="handleClick('${id}')">`  // 简单
  // 函数内: function handleClick(id) { if (locked) {...} else {...} }
  ```
- 教训来源：早期 v0.1 原型曾因此 bug 整页失操作 (2026-05-24，原型已清理)

#### 坑 2: 模板字符串内不能嵌 `<script>` 标签（**整页 JS 失效**）
- ❌ **禁止**在 render 函数返回的模板字符串里嵌 `<script>...</script>` 块
- 原因：浏览器 HTML 解析遇到 `</script>` 时**会提前关闭外层顶层 script 标签**（不管它在不在 JS 字符串字面量里 — 这是 HTML5 解析规则）
- 反例（顶层所有函数定义丢失，开页 openEditor is not defined）：
  ```js
  function renderDetail() {
    return `
      <div>...</div>
      <script>                   ← 浏览器解析时这里关闭了顶层 script
        function someFunc() {}
      </script>
    `;
  }
  ```
- 正例：所有函数定义放顶层 script 块，inline 事件用 onclick/onchange/onmouseenter 调用
  ```js
  // 顶层 script:
  function handleAttachModeChange(mode) { ... }

  // 模板字符串内仅用 inline 调用:
  return `<input type="radio" onchange="handleAttachModeChange('OVERLAY')" />`;
  ```
- 触发场景：动态渲染 detail 面板 / Tab 内容 / 弹层 mockup
- 教训来源：`docs/html/v0.4-3D选配模板管理端原型.html` 因此 bug 整页 JS 失效 (2026-05-26)

## 三、连环 bug 案例（QT-20260522-1590~1604）— 浓缩教训

本次 14 轮派遣修了 1 个主线 + 3 轮 regression，根本原因是**多个独立隐患在 ComponentCell 抽取后同时暴露**。

### 时间线

| 阶段 | bug 现象 | 根因 | 修法 | 反模式 |
|---|---|---|---|---|
| 主线 | 详情页/编辑页 6 类字段渲染不一致 | 双轨实现 | 抽 ComponentCell | AP-50 |
| 第 1 轮 | ELEM_PRICE 解析依赖 tableColumnsCache 缓存 | schema DDL 后缓存残留 | gvar task 注入 + findAliasValue 别名探测 | AP-44 + CLAUDE.md §视图重启 |
| 第 1 轮 | 详情页 BASIC_DATA "加载中..."永久占位 | ComponentCell 缺 readonly 守卫 | 3 处 `if (readonly) return —` 守卫 | AP-31 / AP-38 |
| 第 2 轮 | "单位"列显示价格 5800 | 字段元数据错绑 global_variable_code: ELEM_PRICE | V214 数据迁移 | AP-52 新案例 |
| 第 2 轮 | "单价"列显示 0 | key_field_refs 缺失，findAliasValue 三规则无法映射 "key" 字段 | V215 数据迁移 | AP-52 新案例 |
| 第 3 轮 | 工序列表 4 行 × 7 次 = 28 行 | snapshotRows Math.max 持久化死锁 | 去 Math.max，driver 权威 | **AP-51** |
| 第 4 轮 | 单价首次渲染 0（即使 V215 已修） | formulaEngine global_variable token 查 BNF path 而非 @gvar:CODE | 加 @gvar:CODE 优先查找 | **AP-49 续集 / AP-52** |

### 教训提炼

1. **多隐患连锁触发**：单个修复（抽 ComponentCell）暴露了 4 个独立隐患（缓存残留、字段错绑、key_field_refs 缺失、公式引擎契约不对齐）。**任何架构层重构前都要预演下游链路**。
2. **数据配置错误 vs 代码 bug 的区分**：V214 单位字段错绑、V215 key_field_refs 缺失都是数据层问题，不是代码 bug。**先怀疑数据再怀疑代码**。
3. **前后端契约的 key 命名空间**：`@gvar:CODE` 必须前后端用同一格式。**任何新建数据通道前必须确认前后端 key 一致**。
4. **driver 行数纪律是隐性死锁源**：Math.max(driver, base) 看似兼容，实际是死锁源头。**driver 是唯一权威**。
5. **新建报价单 vs 已存报价单暴露不同问题**：测试必须覆盖两种入口。**E2E 只测已存报价单 ≠ 全覆盖**。
