# 组件管理 — 字段配置指南

> 2026-05-18 Phase K 后版本。配套阅读：[配置中心架构](./配置中心架构.md)、[全局变量使用指南](./全局变量使用指南.md)、[数据源类型扩展指南](./数据源类型扩展指南.md)、[HTTP_API_安全配置](./HTTP_API_安全配置.md)。

## 一、字段配置全景

「组件管理」→ 编辑组件 → 字段配置表（FieldConfigTable）是配置中心最频繁的入口。每个字段由 `field_type` 决定行为，「内容/配置」列按类型动态切换 UI。

### 字段配置表字段

| 列名 | 说明 |
|---|---|
| 字段名 | `name` — 也作为公式 token 的引用 key (`{"type":"field","value":"<name>"}`) |
| 字段类型 | `field_type` — 6 种枚举见下表 |
| 内容/配置 | 按类型动态：固定值输入框 / 数据源按钮 / 默认值来源按钮 / BNF 路径 / 公式提示 |
| 金额 | `is_amount` — 渲染时按货币格式化 |
| 小计 | `is_subtotal` — 公式聚合用；**组件内最多 1 个** |
| 备注 | `notes` — 配置者留言，渲染不展示 |
| 排序 / 删除 | UI 操作 |

## 二、7 种字段类型

| field_type | 数据来源 | UI 渲染 | 「内容/配置」列 | 关键 JSON 字段 |
|---|---|---|---|---|
| `FIXED_VALUE` | 字面量 | 只读单元格 | 固定值输入框 | `content` |
| `INPUT_TEXT` | 用户输入 | 文本输入 | content 兜底 + 默认值来源按钮 | `content`、`default_source` |
| `INPUT_NUMBER` | 用户输入 | 数字输入 | 同上 | `content`、`default_source` |
| `BASIC_DATA` | BNF 路径查物理表/视图 | 只读单元格 | 路径徽章 / 全局变量按钮 | `basic_data_path`、`global_variable_code` |
| `DATA_SOURCE` | 多类型数据源 | 只读 / 下拉选 | type 切换 + 各自配置入口 | `datasource_binding.{type,...}` |
| `FORMULA` | 公式引擎计算 | 计算结果只读 | "在产品模板拖入公式"提示 | `formula_tokens`（在模板编辑器配） |
| `LIST_FORMULA` | 主数据列表 × 逐项公式 | 列驱动多行（每行求值不同公式） | 「源列表 + 逐项公式」抽屉 | `list_formula_config.{source_table, item_key_column, per_item_formulas}` |

### 2.1 FIXED_VALUE — 固定字面量

最简单。直接输入 `content`，渲染时按字面量显示。常用于：标签 / 单位字符串 / 报价说明文字。

**典型坑**：`is_amount=true` 时 content 必须是数字字面量，否则货币格式化报错。

### 2.2 INPUT_TEXT / INPUT_NUMBER — 用户输入（三级兜底链）

报价单可编辑单元格。**核心能力**：default_source 默认值机制（V190 引入）。

#### 三级兜底链（行值空时按顺序解析）

```
用户输入 row[key]
  → default_source.GLOBAL_VARIABLE → basicDataValues['@gvar:CODE']
  → default_source.BNF_PATH        → basicDataValues[{path}] / pathCache
  → field.content                  → 静态字面量兜底
  → 0
```

#### UI 操作

字段配置「内容/配置」列：
- 左侧 `content` 输入框：静态兜底值（如成材率 100）
- 右侧「**+ 默认值来源**」按钮 → DefaultSourceEditor 抽屉

DefaultSourceEditor 抽屉：
1. 选 type（从后端动态拉，K2 后不再硬编码）
2. 各 type 填配置
3. 「测试 driverRow JSON」输入测试值
4. 「测试解析」按钮 → 调 `POST /api/cpq/data-sources/resolve` 实时验证
5. 「保存」

#### 报价单 placeholder

- `default_source` 命中 → `默认 X · CODE`
- 仅 `content` 兜底 → `默认 X`（无后缀）
- 用户输入即覆盖（覆盖值只活在当前草稿）

### 2.3 BASIC_DATA — BNF 路径查表

只读字段，按 BNF 路径解析。`basic_data_path` 字符串通过 ImplicitJoinRewriter 隐式 JOIN driver row。

#### UI 操作

「内容/配置」列两个入口：
- **配置物理表路径** → PathPickerDrawer 选 BNF 路径
- **🌐 全局变量** → GlobalVariablePickerDrawer 选变量 → 编译为 BNF path + 设 `global_variable_code` 徽章

#### `$` / `$$` 引用语法（2026-05-25 扩展，组件 SQL 视图数据源）

BNF path 不仅可以引用物理表/视图，还能引用**用户在组件管理配置的 inline SQL 视图**（详见 `docs/archive/组件级数据源SQL方案.md`）。语法对照：

| 场景 | 写法 | 示例 |
|---|---|---|
| 物理表/视图（既有） | `<sheet>[谓词].<col>` | `mat_part[hf_part_no={lineItem.partNo}].unit_weight` |
| **本组件 SQL** | `$<sql_view_name>[谓词].<col>` | `$element_view[bom_type='ELEMENT'].composition_pct` |
| **跨组件 GLOBAL SQL** | `$$<componentCode>.<sql_view_name>[谓词].<col>` | `$$COMP-FEE.assembly_fee[hf_part_no={lineItem.partNo}].fee_value` |

**PathPickerDrawer** 新增"SQL 视图"Tab，列出本组件可用的 SQL 视图 + 跨组件 GLOBAL 视图。

**关键纪律**：
- ✅ 渲染层 / cache key / ComponentDriverService 三分支 / ImplicitJoinRewriter / ComponentCell 不动 —— `$` 仅作 BNF 解析层别名替换
- ❌ 用户 SQL 内**禁用** `:hfPartNo` 标量占位符（由外层 batch 注入）
- ❌ 用户 SQL **禁止 DDL/DML**（保存时 EXPLAIN 校验）
- ⚠️ 双层冻结：模板 PUBLISHED 冻结至 `template.sql_views_snapshot`，报价单 SUBMITTED 冻结至 `quotation_component_sql_snapshot`

#### 设计原则

- **职责单一**：BASIC_DATA 只查不算。需要计算用 FORMULA。
- **跨表用 driver row 隐式 JOIN**：driver path 决定行展开，字段路径按 driver row 字段自动注入谓词。
- **不要写表达式**：`mat_part.weight * mat_bom.pct` 这种**禁止**，拆成两个 path token 在 FORMULA 里组合。

### 2.4 DATA_SOURCE — 4 种数据源类型（H2/I2/J2 落地）

#### 4 个 type 对照

| type | 配置字段 | 解析行为 | 适用场景 |
|---|---|---|---|
| `DATABASE_QUERY` | `datasource_id` + `params` | 走 `DataSourceService.execute`（SQL/API） | 数据源管理页注册的查询 |
| `GLOBAL_VARIABLE` | `global_variable_code` + `key_field_refs` | 按 driver row 解 → `global_variable_value` 单表 | 引用配置中心变量值 |
| `BNF_PATH` | `bnf_path` | `DataLoader.loadByPath` | 与 BASIC_DATA 同源（视作"数据源"形态） |
| `HTTP_API` | `api_config.{url_template, response_path, auth_token_env}` | 外部 REST 调用（严格安全） | 集成外部系统 |

#### UI 操作

「内容/配置」列：
- type Select（**动态从后端拉，K2 后**）
- 选 type 后右侧出现对应配置入口：
  - DATABASE_QUERY → 「配置数据源」按钮跳转
  - GLOBAL_VARIABLE → 「🌐 选择变量」（复用 GlobalVariablePickerDrawer）
  - BNF_PATH → 「{path}」编辑（复用 PathPickerDrawer）
  - HTTP_API → 「🌐 配置 HTTP API」inline Modal

#### 后端校验（K hotfix 后）

`ComponentService.validateFields` 按 `datasource_binding.type` 分发：

| type | 必填字段 | 报错消息 |
|---|---|---|
| DATABASE_QUERY (缺省) | datasource_id | `DATA_SOURCE/DATABASE_QUERY 缺 datasource_id` |
| GLOBAL_VARIABLE | global_variable_code | `DATA_SOURCE/GLOBAL_VARIABLE 缺 global_variable_code` |
| BNF_PATH | bnf_path (非空) | `DATA_SOURCE/BNF_PATH 缺 bnf_path` |
| HTTP_API | api_config.url_template | `DATA_SOURCE/HTTP_API 缺 api_config.url_template` |

#### HTTP_API 默认关闭

`HttpApiResolver` 必须在 `application.properties` 配置：
```properties
cpq.http-api.allowed-hosts=api.example.com
```
不配则整个 resolver 返 null。Token 走环境变量取 (`auth_token_env: "MY_VAR"` → `System.getenv("MY_VAR")`)。详见 `HTTP_API_安全配置.md`。

### 2.5 FORMULA — 公式引擎计算

公式 token 在模板编辑器拖拽配置。当前支持 11 种 token：

| token type | 引用对象 | 必填字段 |
|---|---|---|
| `field` | 同行其他字段值 | `value` (字段名) |
| `number` | 数字字面量 | `value` |
| `operator` | + - * / | `value` |
| `bracket_open` / `bracket_close` | 括号 | - |
| `component_subtotal` | 跨组件小计 | `component_code` 或 `tab_name` |
| `product_attribute` | 产品属性 | `attribute_name` |
| `quotation_field` | 报价单字段 | `value` |
| `path` | BNF 路径标量 | `path` |
| `global_variable` | 全局变量 | `code` + (`key_values` 或 `key_field_refs`) |
| `previous_row_subtotal` | driver 上一行 subtotal | `fallback_component_code`（行 0 兜底） |
| `datasource_field` (K1) | 同行 DATA_SOURCE 字段解析值 | `name` (字段名) |

#### 跨行/跨组件示例（工序累加公式）

```json
{
  "name": "小计",
  "field_type": "FORMULA",
  "formula_tokens": [
    {"type": "previous_row_subtotal", "fallback_component_code": "COMP-CFG-ELEMENT-BOM"},
    {"type": "operator", "value": "/"},
    {"type": "field", "value": "成材率"},
    {"type": "operator", "value": "*"},
    {"type": "number", "value": "100"},
    {"type": "operator", "value": "+"},
    {"type": "field", "value": "单价"}
  ]
}
```

### 2.7 LIST_FORMULA — 配置模板 + 条件分支公式（2026-05-19 Phase B）

绑定**独立的配置模板**（一级菜单"配置模板"维护），按选定**大类**的明细项数 expand 行；每项可配**多条件分支**（IF-ELSE-IF 链），运行时第一个 condition=true 的 branch 取其 formula 求值。

**关键能力（vs Phase A）**:
- 数据源从 `basic_data_config` 物理表 → **独立配置模板** (`config_template`/`config_category`/`config_item` 三表)
- 单一公式 → **IF-ELSE-IF 条件分支链**（不同字段值对应不同算法）
- 三态机：DRAFT 不可引用 / PUBLISHED 可用 / ARCHIVED 历史保护

**配置示意**：

```jsonc
{
  "name": "工序成本",
  "field_type": "LIST_FORMULA",
  "is_amount": true,
  "list_formula_config": {
    "config_template_id": "...",
    "config_template_code": "STD_RULES",
    "config_template_name": "标准定价规则",
    "category_code": "PROCESS",
    "category_name": "工序",
    "per_item_rules": {
      "RIVET": {
        "branches": [
          { "condition": "[厚度] > 5 AND [材质] = 'AgCu85'", "formula": "[厚度] * 2.5" },
          { "condition": "[厚度] > 2", "formula": "[厚度] * 1.8" }
        ],
        "default_formula": "[厚度] * 1.5"
      },
      "LASER_WELD": {
        "branches": [],
        "default_formula": "100"
      }
      // 没显式配的 → 走 config_item.default_value
    }
  }
}
```

**条件表达式（`evaluateCondition`）**:

| 语法 | 例 |
|---|---|
| `[字段] op X` (op: `> < = != >= <=`) | `[厚度] > 5` |
| 字符串字面值 (单/双引号) | `[材质] = 'AgCu85'` |
| 字段间比较 | `[a] > [b]` |
| AND/OR 扁平左结合 (大小写不敏感) | `[厚度] > 5 AND [材质] = 'AgCu85'` |
| 空字符串 / null | 视为"总是 true"（默认分支） |

不支持：括号嵌套 / NOT / IN / 函数（Phase C 再加）

**公式语法**（同 Phase A 的 `evaluateListFormulaString`）:

| Token | 含义 |
|---|---|
| `[字段名]` | 本行其他字段值 |
| `{GV_CODE}` | 全局变量（Phase C 接入；当前求值为 0） |
| 数字 / `+ - * / ( )` | 标准算术 |

**求值时序**:
1. ProductCard 启动, `useConfigTemplates(lineItems)` 收集 LIST_FORMULA 字段的 config_template_id
2. 后端 `GET /api/cpq/config-templates/{id}` 拉详情（含 categories.items）
3. 含 LIST_FORMULA 字段的组件: effectiveCount = `category.items.length`，每行 = 一个明细项
4. LIST_FORMULA cell 求值:
   1. `per_item_rules[item.code].branches` 按顺序求 `condition`
   2. 第一个 true 的 branch → 取其 `formula`
   3. 全不命中 → `rule.default_formula`
   4. 仍无 → `item.default_value`（config_item 上）
   5. 都空 → 渲染 `—`

**同组件多 LIST_FORMULA 字段**: 必须绑同 `(config_template_id, category_code)`（行数语义对齐）。

**自检要点**:
- 配置模板必须 PUBLISHED 才能被字段引用（DRAFT 在 Drawer 里过滤）
- 配置模板归档前请确认无字段引用（系统不强制校验，PR 自检注意）
- 条件表达式语法严格按 `[字段] op X` + AND/OR；解析失败 → false（保守不命中）
- 模板里加新明细项 → 引用此模板的所有组件自动新增行（无需改字段配置）

**关联文档**:
- `docs/反模式.md` AP-31 关于"加载中"渲染需谨慎
- `cpq-frontend/src/utils/conditionEngine.ts` 条件解析器单元测试覆盖 30 个 case

## 三、组件保存自动同步引用模板 (H1)

`ComponentService.update` 成功后自动调 `TemplateService.refreshSnapshotsByComponent`，所有引用该组件的 template.components_snapshot 立即同步刷新。

**含义**：组件改字段后**不需要**手工 SQL 改 template snapshot — 报价单立即看到新配置。

**例外**：手工 SQL 直接 `UPDATE component SET fields = ...` 绕过 service 层不会触发同步，必须调用 `POST /api/cpq/components/{id}/refresh-template-snapshots`（PRICING_MANAGER+）或批量 `POST /api/cpq/config-center/refresh-all-snapshots`（SYSTEM_ADMIN）。

## 四、自检 checklist（PR 评审用）

提交字段配置变更前对照：

```
□ field_type 在 6 种枚举内 (FIXED_VALUE/INPUT_TEXT/INPUT_NUMBER/BASIC_DATA/DATA_SOURCE/FORMULA)
□ 组件内 is_subtotal=true 至多 1 个 (validateFields 强制)
□ DATA_SOURCE 字段 datasource_binding 按 type 含必填字段:
    DATABASE_QUERY  → datasource_id
    GLOBAL_VARIABLE → global_variable_code
    BNF_PATH        → bnf_path 非空
    HTTP_API        → api_config.url_template
□ INPUT_NUMBER/TEXT 关键字段配 content 静态兜底 (避免 default 解不出退到 0)
□ default_source.GLOBAL_VARIABLE 的 key_field_refs 留空走同名映射 (LOOKUP key 列名 = driver row 物理列名)
□ FORMULA token 中引用的 field name 在同组件 fields 列表里存在
□ FORMULA token 引用 DATA_SOURCE 字段必须用 datasource_field type (K1), 不要用 field type
□ BASIC_DATA 字段不要写表达式 (如 a*b), 用 FORMULA 拆成多 path token
□ 字段 name 简洁 ASCII / 中文, 避免特殊符号 / (虽然 React/JSON 兼容, 公式 token 引用更稳)
□ 组件保存后无需手工同步 template snapshot (H1 自动)
□ 修改前在 DefaultSourceEditor 用「测试解析」验证 default_source 配置可解
□ HTTP_API 类型必须在 application.properties 配 cpq.http-api.allowed-hosts (默认关闭)
□ 强刷浏览器验证报价单渲染正常
```

## 五、避坑速查表

| 坑 | 现象 | 修法 | 反模式 |
|---|---|---|---|
| 改组件字段后报价单还是旧配置 | snapshot 没同步 | `ComponentService.update` 已自动同步 (H1); 或手工 `POST /components/{id}/refresh-template-snapshots` | AP-27 |
| 加新 default_source 字段元数据但前端拿不到 | 三个 builder 没同步 | 同步改 `enrichComponentData` + `buildComponentDataFromTemplate` + `buildField` | AP-26 |
| placeholder 显示「默认 100」无 ` · CODE` 后缀 | DB 查不到 key | 检查 `global_variable_value` 是否含该 key; 走 `INSERT ... SELECT DISTINCT FROM mat_xxx ON CONFLICT DO NOTHING` 全 seed | — |
| DATA_SOURCE 字段保存报 400 缺 datasource_id | 改了 type 但后端校验未识别新 type | K hotfix 已修, 按 type 分发校验 | AP-26 反向版本 |
| 「工序」字段三行重复显示同值 | driver path 按 mat_process 1:1 展开, 但字段绑 `assembly_process` 同 process_code 多行相同值 | 接受现状 / 改回 `component_name` / 加聚合 driver path | — |
| 「成材率」改成 DATA_SOURCE 后报「查询失败」 | 用户 field_type 切换但 driver expand 不识别 `datasource_binding.GLOBAL_VARIABLE` | K hotfix 已修, `parseGvarDefaultTasks` 同时识别两条路径 | — |
| 自动同步失败但保存成功 | snapshot 刷新失败被 try-catch 吞 | LOG.warn 看后端日志, 或手工 `POST /components/{id}/refresh-template-snapshots` 重试 | — |
| 测试解析返 null | driverRow JSON 缺 key 列字段 | 测试 driverRow 至少含 `def.keyColumns` 内所有列 | — |
| HTTP_API 不工作 | 白名单未配 | application.properties 加 `cpq.http-api.allowed-hosts=...`; token 用环境变量 | — |
| 字段切换 type 后老配置残留 | 切换前的 basic_data_path / datasource_id 留在 JSON | FieldConfigTable 的 type 切换 onChange 已自动清相关字段 (H2); 老组件配置可手工清 | — |
| 产品级组件（无 dataDriverPath）BASIC_DATA 全部「加载中」 | 后端 expand 看 dataDriverPath 空就跳过解析, 前端 globalPathCache 路径不稳 | 后端 hotfix: dataDriverPath 空时按「虚拟单行」展开, driverRow={hf_part_no, customer_id}, 解所有 BASIC_DATA paths 塞 basicDataValues | AP-29 |
| 「工序」「材料名」等列显示「—」但数据库行有 process_code 等 key | INSERT 时只写 key 列, 漏写 name/描述等 view 渲染依赖列 | 写表前 grep 该表所有 basic_data_path 引用, 把所有渲染依赖列一并 INSERT | AP-30 |
| 选配工序后报价单显示历史 Z350/Z029 不是新选工序 | fingerprint 只算 recipe+elements 不算 processIds → existing 路径直接复用老 hfPartNo | FingerprintCalculator 加 3-arg 重载传 processIds; existing+processIds 非空时生成新 hfPartNo 复制 mat_part/mat_bom | — |
| DATA_SOURCE.type=GLOBAL_VARIABLE 字段保存 400 「缺 datasource_id」 | 后端 validateFields 只识别 DATABASE_QUERY type | H2/K hotfix: validateFields 按 datasource_binding.type 分发校验 4 种形态 | — |
| DATA_SOURCE.GLOBAL_VARIABLE 字段在报价单「查询失败」 | 旧 noParamDsFields filter 不区分 type, 误调老 /datasources/{undefined}/execute 报 404; DATA_SOURCE 渲染分支只走老 dsLoading 状态机 | filter 加 type 判断, 仅 DATABASE_QUERY 走老 execute; DATA_SOURCE 渲染按 type 分发: GLOBAL_VARIABLE 走 basicDataValues['@gvar:CODE'] / BNF_PATH 走 bnfDriverLookupKey / HTTP_API 走 row[key] | — |

## 六、字段类型切换矩阵

UI 上切 field_type 时不同字段的清理规则：

| 切换前 → 切换后 | 自动清理 | 保留 |
|---|---|---|
| INPUT_NUMBER → BASIC_DATA | default_source / content | name / notes |
| INPUT_NUMBER → DATA_SOURCE | default_source / content | name / notes |
| BASIC_DATA → DATA_SOURCE | basic_data_path / global_variable_code | name / notes |
| DATA_SOURCE 内 type 切换（如 DATABASE_QUERY → GLOBAL_VARIABLE） | 切走类型的配置（datasource_id / bnf_path 等） | type 公用字段 |
| 任何 → FORMULA | 数据来源相关字段 | name / notes / is_amount / is_subtotal |

**注意**：UI 切换是「就地修改字段」，老组件配置文件里的残留字段可能存在（JSON 是松散结构）。校验通过即可，残留字段不影响业务但占 JSON 体积。

## 七、关联反模式

| 编号 | 反模式 | 字段配置场景中的体现 |

## 十一、字段类型联动性矩阵（**改字段类型 / 加新类型 强制 SOP**）

> **2026-05-19 沉淀，2026-05-22 AP-50 更新，2026-05-25 §11.x 新增 BNF 解析层第 18 处**：B1~B4 + AP-40~43 一连串事故证明：**"字段类型"不是单点属性，而是跨 15+ 文件的协议变换**。改一处必动 15 处，漏一处必有静默失败（不报编译错也不报运行时错，只是 UI 渲染不对）。

### 关键教训

> "字段类型" **不是一个属性**，是**跨前后端 + 跨多视图 + 跨持久化层** 三个维度的协议变换：
>
> 1. **跨前后端**: 后端定义校验 + 路径采集 + 公式 token, 前端定义 enrich + 渲染 + 公式求值
> 2. **跨多视图**: 报价单 / 核价单 / 详情页 / 比对页 / 批量导入 — 任何独立渲染 lineItem.componentData 的视图都要同步
> 3. **跨持久化层**: component.fields (源) + template.components_snapshot (publish 定格) + template_component.fields_override (Tab 级覆盖) — schema 迁移必须三处同步
>
> **没有 E2E 复测的字段类型变更 PR 一律拒收**。不是建议是规范 — 因为没 E2E 就不可能验证 15 处都改对了, 而漏一处必有静默失败.

```jsonc
            "formula": "{mat_bom[element_name='Sn'].net_qty}*5"  // ← BNF 路径
          }
        ]
      }
    }
  }
}
```

**Token 解析规则** (`evaluateListFormulaString` line 302-313):
- `{...}` 含 `.` → 视为 BNF 数据库路径, 从 `basicDataValues` (driver expand 行级) 取值, 缺值 fallback 到 `pathCacheState` (partNo 级 driver-free)
- `{...}` 不含 `.` → 视为全局变量 code, 从 `globalVarValues` 取
- `[字段名]` → 当前行其他字段值 (来自 rowFieldValues)
- `[表名.列名]` → 列表项原生列 (sourceTable 前缀匹配)

**协议传播 4 处** (必须同步改):
- 前端 `formulaEngine.ts:evaluateListFormulaString` — 入参增加 `basicDataValues` / `partNo` / `pathCacheState`
- 前端 `usePathFormulaCache.collectListFormulaBnfPaths` — 扫 `branches[*].formula` + `default_formula` 内 `{table.col}` token
- 后端 `ComponentDriverService.parseBasicDataPaths` + `collectBnfPathsFromFormula` — 同步采集让 batch-expand 行级 basicDataValues 含这些 path
- 后端无需改 BNF 解析器本身

**关键坑** (AP-46):
- 公式形如 `{mat_bom.net_qty}*5` (无谓词过滤) 返多行 array → `evaluateListFormulaString` 当 null → 求值 0
- 必须用 `{mat_bom[element_name='Sn'].net_qty}` 单值形态
- driver 行注入 ImplicitJoinRewriter 会把 driverRow 列也作 AND 谓词. driver=mat_process 时 seq_no 会注到 mat_bom 路径, 与 Sn 行 seq_no 不匹配 → 空集 → null. 前端 fallback 链回退到 partNo 级 cache 解决

#### 2. `formula_composite` 双轨 (SIMPLE/COMPOSITE 视角分别配公式)

同模板支持 SIMPLE + COMPOSITE 时, LIST_FORMULA 公式同样需要双轨:

```jsonc
{
  "list_formula_config": {
    "per_item_rules": {
      "MRO-AS-0001": {
        "branches": [{
          "condition": "[工序代码] = 'MRO-AS-0001'",
          "formula":           "{mat_bom[element_name='Sn'].net_qty}*15",  // SIMPLE 视角
          "formula_composite": "100"                                       // COMPOSITE 视角占位 (业务按需改)
        }],
        "default_formula":           "",
        "default_formula_composite": ""  // COMPOSITE 视角的默认值兜底
      }
    }
  }
}
```

**运行时选择**: `chosenFormula = (isCompositeItem && b.formula_composite) ? b.formula_composite : b.formula`. 缺失 `formula_composite` 时**自动 fallback 到 formula** (向后兼容: 旧 LIST_FORMULA 字段不需要立即升级双轨).

**为什么需要双轨**:
- 组合产品 partNo = 合成主键 (如 CFG-COMBO-000021), **不在任何子件物理表中**
- SIMPLE 公式 `{mat_bom[...].net_qty}` 对组合 partNo 必然查空 → 0
- COMPOSITE 需要用 `v_composite_child_*` 视图字段 (按 parent_hf_part_no=组合料号 JOIN 子件)

**协议传播扩展 (从 17 处 → 约 20 处)**:
- 前端 cell render (`QuotationStep2.tsx:1470` 附近) `branches.formula_composite` 优先选取
- 前端 `collectListFormulaBnfPaths` — 扫两条公式
- 后端 `collectBnfPathsFromFormula` — 同步扫
- 后端 `TemplateService.mergeListFormulaConfigComposite` — `createNewDraft` 合并时按 rule code + branch index 保留 `formula_composite` (组件改公式时不冲洗模板独有的 _composite 配置)

详见反模式 [AP-47 LIST_FORMULA 单轨公式失效](./反模式.md#ap-47-list_formula-formula-单轨--组合产品视角下公式失效)。

### 「其他数据源」选项限制同目录 — 2026-05-20

组件管理 UI 右侧「其他数据源」Tab 默认只列**同目录下其他组件**的 `is_subtotal=true` 字段。

**为什么**:
- 跨目录引用容易导致模板组装漏配 (不同业务目录的组件常常不在同一模板出现)
- 同目录组件通常对应同一业务流程, 跨组件小计语义清晰

**实现**: `ComponentManagement.tsx:505` `otherCompSubtotals` 加 `c.directoryId === selectedComponent?.directoryId` 过滤. `selectedComponent` 未选择或无目录归属时返空数组.

**例外场景**: 如果业务确实需要跨目录引用 (罕见), 通过 SQL 直接改 component.formulas 内 token (`type: 'component_subtotal'`) 写组件 code, 后端不强校验目录归属.
