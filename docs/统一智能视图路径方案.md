# 同模板独立/组合产品统一渲染 — 智能视图路径方案

> **2026-05-21 立项**。基于用户硬约束："所有产品卡片内容都根据组件管理和模板管理的设置进行渲染加载，禁止使用代码影响模板逻辑"。
>
> **TL;DR**：现有"双轨 `basic_data_path_composite`"方案违反了上述约束 — 前端通过 `lineItem.compositeType==='COMPOSITE'` 切换 path 就是"代码影响模板逻辑"。本方案改为：**组件管理统一配 `v_composite_child_*` 智能视图路径，视图自适应 SIMPLE/COMPOSITE，前端零特殊逻辑**。

---

## 一、问题深度分析

### 1.1 用户反馈的场景

**场景 1（组件管理配置）**：
"选配-元素含量" Tab 字段"元素"配置：
- 字段类型：BASIC_DATA
- 内容（path）：`mat_bom[bom_type='ELEMENT'].element_name`
- 实际查询：`SELECT element_name FROM mat_bom WHERE bom_type='ELEMENT' AND hf_part_no=<卡片料号>`

**场景 2（独立产品 vs 组合产品冲突）**：
- 独立产品（如 3120012574 自身）：卡片料号 = 一个物理 `hf_part_no` → 查 mat_bom 正确返回 N 行元素
- 组合产品（CFG-COMBO-000XXX 父级）：卡片料号 = 父级 `hf_part_no` → mat_bom 父级**无** ELEMENT 行（元素挂在子件下）→ 返 0 行 / "—"

**用户期望**：同一模板被两种产品共用，不需要任何前端特殊代码就能正确渲染。

### 1.2 现有"双轨方案"的根本错误

我（前一轮）的修法是给每个字段配两份 path：

| 字段属性 | SIMPLE 用 | COMPOSITE 用 |
|---|---|---|
| `basic_data_path` | `mat_bom[bom_type='ELEMENT'].element_name` | — |
| `basic_data_path_composite` | — | `v_composite_child_elements.element_name` |
| `data_driver_path` | `mat_bom[bom_type='ELEMENT']` | — |
| `data_driver_path_composite` | — | `v_composite_child_elements` |

前端 `useDriverExpansions` 通过 `lineItem.compositeType === 'COMPOSITE'` 切换 effective path：

```ts
// useDriverExpansions.ts (现状, 反模式)
const isComposite = item.compositeType === 'COMPOSITE';
const effectiveDriver = isComposite
  ? (comp.dataDriverPathComposite || comp.dataDriverPath)
  : comp.dataDriverPath;
const effectiveFields = isComposite
  ? fields.map(f => ({ ...f, basic_data_path: f.basic_data_path_composite ?? f.basic_data_path }))
  : fields;
```

**这正是用户禁止的"代码影响模板逻辑"**。

### 1.3 V202 视图已经实现了正确设计（但被绕过了）

`v_composite_child_*` 视图（V194 → V196 → V202 演进）的设计目标就是 SIMPLE/COMPOSITE 自适应：

```sql
-- v_composite_child_elements (V202)
-- 分支 A: COMPOSITE 父级 (mat_bom 含 ASSEMBLY 行)
SELECT asy.hf_part_no, child_part_name, elem.seq_no, elem.element_name, elem.composition_pct
FROM mat_bom asy
JOIN mat_bom elem ON elem.hf_part_no = COALESCE(asy.child_part_no, asy.input_material_no)
                  AND elem.bom_type = 'ELEMENT'
WHERE asy.bom_type = 'ASSEMBLY'

UNION ALL

-- 分支 B: SIMPLE 自身 (mat_bom 无 ASSEMBLY 行)
SELECT elem.hf_part_no, elem.hf_part_no AS child_part_name, elem.seq_no, elem.element_name, elem.composition_pct
FROM mat_bom elem
WHERE elem.bom_type = 'ELEMENT'
  AND NOT EXISTS (SELECT 1 FROM mat_bom WHERE hf_part_no=elem.hf_part_no AND bom_type='ASSEMBLY');
```

**对任何产品（SIMPLE 或 COMPOSITE），同一查询 `SELECT * FROM v_composite_child_elements WHERE hf_part_no=:p` 都能正确返回数据**：
- SIMPLE 产品 → 走分支 B → 返自身 ELEMENT 行
- COMPOSITE 产品 → 走分支 A → 返子件聚合 ELEMENT 行

**双轨方案的本质错误**：明明 V202 视图已经做对了，前端却又加了 `isComposite` 分支切回物理表 mat_bom — 把 DB 层已经解决的问题倒回到应用层，并散布到 17 处协议传播点。

### 1.4 实际验证

通过 admin 接口拉所有模板的"选配-元素含量"Tab 的"元素"字段配置：

```
v1.10~v1.22 几乎全部:
  basic_data_path           = mat_bom[bom_type='ELEMENT'].element_name   ← SIMPLE 物理表路径 (覆盖不到 COMPOSITE)
  basic_data_path_composite = v_composite_child_elements.element_name    ← 智能视图路径 (理论上覆盖 SIMPLE+COMPOSITE)
```

**两个 path 的差异**：智能视图路径包含了物理表路径的语义（SIMPLE 分支 B），且额外支持 COMPOSITE（分支 A）。所以**智能视图路径是物理表路径的超集**，物理表路径可以被完全替换。

## 二、完美方案设计

### 2.1 核心思路

**一份配置，DB 视图自适应，前端零分支**。

| 层 | 现状（错误） | 完美方案（正确） |
|---|---|---|
| 组件管理 path | `mat_bom[bom_type='ELEMENT'].element_name` (SIMPLE only) | `v_composite_child_elements.element_name` (统一智能视图) |
| 模板 fields_override | 双份 `basic_data_path` + `basic_data_path_composite` | 单份 `basic_data_path` |
| 模板 tc driver | 双份 `data_driver_path` + `data_driver_path_composite` | 单份 `data_driver_path` |
| 前端 useDriverExpansions | `isComposite` 切换 effective driver/fields | 无分支，统一查同一 path |
| 前端 QuotationStep2 | `computeAllFormulas(isCompositeItem)` 参数 | 无参数 |
| 前端 enrich mapper | 5 处透传 `_composite` 字段 | 无 |
| 后端 createNewDraft | `mergeListFormulaConfigComposite` 合并双轨 | 无 |
| AP-44 协议传播 | 17 处 | 回到 15 处 |
| AP-45 反模式 | 存在 | 删除 |

### 2.2 视图层（已有，不动）

V202 的 4 个 `v_composite_child_*` 视图已经实现 SIMPLE/COMPOSITE 自适应：

| 视图 | 用途 | SIMPLE 行为 | COMPOSITE 行为 |
|---|---|---|---|
| `v_composite_child_materials` | 材质明细 | 自身 mat_part → recipe | 子件聚合 |
| `v_composite_child_elements` | 元素含量 | 自身 mat_bom ELEMENT | 子件聚合 |
| `v_composite_child_processes` | 工序 | 自身 mat_process | 子件聚合 |
| `v_composite_child_weights` | 单重 | 自身 mat_part.unit_weight | 子件聚合 |

**子件标识列**：所有视图都有 `child_part_name` 列。组件管理"子件"字段配 `v_composite_child_*.child_part_name`：
- 独立产品 → child_part_name = 自身料号（用户看到自己的料号）
- 组合产品 → child_part_name = 真实子件料号（用户看到子件名）

### 2.3 组件管理规范（强制约定）

**任何打算被"组合产品报价模板"复用的 Tab 组件**，BASIC_DATA path 必须用 `v_composite_child_*` 视图，**禁止**直接用 mat_bom / mat_process / mat_part 物理表路径。

| Tab 语义 | 推荐 path 前缀 | 子件列 path |
|---|---|---|
| 材质明细 | `v_composite_child_materials.*` | `v_composite_child_materials.child_part_name` |
| 元素含量 | `v_composite_child_elements.*` | `v_composite_child_elements.child_part_name` |
| 工序列表 | `v_composite_child_processes.*` | `v_composite_child_processes.child_part_name` |
| 单重 | `v_composite_child_weights.*` | `v_composite_child_weights.child_part_name` |

**组件管理 UI 增强**（可选 P1）：
- 在字段编辑器里把 `v_composite_child_*` 视图标为"产品明细（智能）"，物理表 `mat_*` 标为"主数据（独立产品专属）"
- 提示用户："这是组合产品兼容字段，请使用'产品明细（智能）'类型"
- 模板新建/发布时检测"选配-*" Tab 字段是否用了非智能视图路径，给警告

### 2.4 模板层（清理）

**fields_override 字段简化**：
```jsonc
// 现状 (双轨, 反模式)
{
  "name": "元素",
  "field_type": "BASIC_DATA",
  "basic_data_path": "mat_bom[bom_type='ELEMENT'].element_name",
  "basic_data_path_composite": "v_composite_child_elements.element_name"
}

// 完美方案 (单轨, 智能视图)
{
  "name": "元素",
  "field_type": "BASIC_DATA",
  "basic_data_path": "v_composite_child_elements.element_name"
}
```

**tc 顶层简化**：
```jsonc
// 现状
{ "data_driver_path": "mat_bom[bom_type='ELEMENT']", "data_driver_path_composite": "v_composite_child_elements" }

// 完美方案
{ "data_driver_path": "v_composite_child_elements" }
```

### 2.7 模板数据迁移

所有 v1.0~v1.22 PUBLISHED 模板的"选配-*" Tab 字段：

1. 把 `basic_data_path_composite` 的值**覆盖**到 `basic_data_path`
2. 删除 `basic_data_path_composite` 键
3. 把 `data_driver_path_composite` 的值**覆盖**到 `data_driver_path`
4. 删除 `data_driver_path_composite` 键
5. 对 LIST_FORMULA 同样处理 `formula_composite`

### 2.8 子件列的统一语义

| 场景 | child_part_name 值 |
|---|---|
| SIMPLE 独立产品 | = hf_part_no（卡片料号）| 显示为产品自身料号 |
| COMPOSITE 父级 | = 真实子件 hf_part_no | 显示为子件名 |

**UI 影响**：
- SIMPLE 时"子件"列每行都是同一个料号 — 视觉冗余，但语义正确
- 可选 UX 优化：组件管理在"子件"字段加属性 `hideWhenSingleRow: true`，单行时不显示该列 — 但这是 P2 优化，不影响数据正确性

## 六、风险评估

| 风险 | 等级 | 缓解 |
|---|---|---|
| 视图性能：`v_composite_child_*` UNION ALL 比物理表慢 | 中 | 实测 mat_bom < 10K 行，视图 UNION 不到 100ms。如成瓶颈，加 mat_bom(bom_type) / (hf_part_no, bom_type) 索引 |
| 视图列不全：智能视图缺某个物理表列 → 现有模板字段无法迁移 | 高 | 阶段 1 体检时枚举所有模板用过的 mat_bom/mat_process/mat_part 列，确认视图覆盖；缺失列写入视图 SELECT 补充 |
| LIST_FORMULA `per_item_rules` 兼容性：现行 formula 引用了 `{mat_bom.X}` BNF token | 高 | 一并在阶段 2 迁移 — formula token 内的 `mat_bom.X` 也改为 `v_composite_child_*.X` |

## 十三、终极配置化方案：上下文变量字典 + 显式条件 path（2026-05-21 第三轮重设计）

> **触发**：用户挑战"现有 BNF path 自动注入 hf_part_no=卡片料号 是隐式硬规则"，要求**完全开放条件参数**，让用户能声明任意字段（如 customer_id、产品分类、行级字段）参与查询条件判断。

### 13.1 当前 BNF 的隐式硬规则审计

| 隐式规则 | 出处 | 用户能否在配置层覆盖？ |
|---|---|---|
| `WHERE hf_part_no = <卡片料号>` 自动注入 | `ImplicitJoinRewriter` 硬编码 | ❌ 不能 |
| `WHERE customer_id = <报价单客户>` 在 mat_process 部分自动注入 | `ComponentDriverService` 局部判断 | ❌ 不能 |
| `WHERE is_current = true` 在部分视图硬编码 | V202 视图 SQL | ❌ 不能（除非删视图重建）|
| `WHERE bom_type = 'ELEMENT'` | path 里字面常量 `[bom_type='ELEMENT']` | ✅ 能（用户在 path 里写）|
| `WHERE element_name = <行级元素名>` (DATA_SOURCE.key_field_refs) | datasource_binding 配置 | ✅ 能 |
| `WHERE quotation_line_item_id = <Bug B lineItemId>` | V207 ImplicitJoinRewriter 视图列谓词 | ❌ 不能 |

**结论**：BNF path 的字面常量谓词（`[col='value']`）是用户可控的，但**所有上下文变量谓词（料号 / 客户 / lineItemId）都是后端硬规则注入**，用户在 UI 上看不到、改不了。这正是用户挑战的根源。

### 13.2 新设计核心：上下文变量字典 + 显式谓词表达

#### 13.2.1 上下文变量字典（系统级标准变量）

定义一组"运行时上下文变量"，用户在组件管理 / 模板 / 公式编辑器里能引用：

```jsonc
// runtime context (后端组装, 前端透传给 BNF resolver)
{
  "lineItem": {
    "partNo": "3120012574",
    "compositeType": "SIMPLE" | "COMPOSITE" | "PART",
    "id": "<uuid>",                 // Bug B 隔离用
    "tempId": "<uuid>"
  },
  "quotation": {
    "id": "<uuid>",
    "customerId": "<uuid>",         // 罗克韦尔
    "customerCategory": "VIP",
    "productCategoryId": "<uuid>",
    "salesRep": "<uuid>"
  },
  "user": {
    "id": "<uuid>",
    "role": "SALES_REP"
  },
  "row": {
    // 行级字段, 用于父子表 JOIN (如元素含量 Tab 行的 element_name 用于查 ELEM_PRICE)
    "元素": "Ag",
    "工序代码": "MRO-AS-0001"
    // ... 动态, 由当前 row 决定
  },
  "global": {
    "CURRENT_FISCAL_YEAR": 2026     // 全局变量, 复用现有 global_variable 表
  }
}
```

后续可扩展（如 `quotation.deliveryDate`、`row.<任意字段名>`），保持开放。

#### 13.2.2 BNF path 语法升级

**当前语法**（隐式注入硬规则）：
```
mat_bom[bom_type='ELEMENT'].element_name
            ↑ 字面常量谓词
↑ 后端自动注入 hf_part_no = <卡片料号>
```

**新语法**（完全显式）：
```
mat_bom[
  bom_type = 'ELEMENT'
  AND hf_part_no = {lineItem.partNo}
  AND customer_id = {quotation.customerId}    ← 用户可加
].element_name
```

谓词类型：
- **字面常量**：`bom_type = 'ELEMENT'` / `is_current = true`
- **上下文变量插值**：`hf_part_no = {lineItem.partNo}` / `customer_id = {quotation.customerId}`
- **行级字段引用**：`element_name = {row.元素}` （父子表 JOIN）
- **全局变量**：`fiscal_year = {global.CURRENT_FISCAL_YEAR}`

**向后兼容**：
- 老 path `mat_bom[bom_type='ELEMENT'].element_name` 自动 desugar 为
  `mat_bom[bom_type='ELEMENT' AND hf_part_no={lineItem.partNo}].element_name`
- ImplicitJoinRewriter 变为"默认值兜底"：如果 path 显式声明了 hf_part_no 条件 → 不注入；如果没声明 → 兜底注入 `hf_part_no={lineItem.partNo}`
- 用户可以**显式声明 `[no_implicit_filter=true]`** 关闭兜底注入（特殊场景）

#### 13.2.3 组件管理 UI 谓词编辑器

字段编辑器从"输入 path 字符串"升级为结构化编辑：

```
[字段] 单价
[类型] 基础数据 (BASIC_DATA)
[数据表] mat_process
[取值列] unit_price
[过滤条件]
  + 自动关联 (默认勾选, 由系统注入 hf_part_no = 卡片料号)
  ☑ hf_part_no = {lineItem.partNo}     [×]
  ☑ customer_id = {quotation.customerId} [×]   ← 用户新加的条件
  ☑ is_current = true                  [×]
  [+ 添加条件...]
      条件列: [下拉: 表的所有列]
      操作符: [= / != / > / < / IN / IS NULL]
      值类型: ○ 常量  ○ 上下文变量  ○ 行级字段  ○ 全局变量
      值:    [文本框 / 下拉]
```

这样用户能在 UI 上看到完整的查询语义，每个条件都可加可删。

#### 13.2.4 Tab 显隐统一表达式

替代 §12.3 的 `applicableProductTypes` 多选属性，升级为通用表达式：

```jsonc
// 组件 schema
{
  "componentCode": "COMP-CFG-COMPOSITE-PROC",
  "componentName": "选配-组合工艺",
  "visibleWhen": "{lineItem.compositeType} = 'COMPOSITE'"
}

// 另一例: 仅 VIP 客户看的 Tab
{
  "visibleWhen": "{quotation.customerCategory} = 'VIP' AND {lineItem.compositeType} IN ('SIMPLE','COMPOSITE')"
}

// 另一例: 仅特定销售负责人看的 Tab
{
  "visibleWhen": "{user.id} = '<某 UUID>'"
}
```

`visibleWhen` 留空 = 默认显示。UI 也可降级为多选属性形态，但底层存的还是表达式（保持表达力）。

#### 13.2.5 公式编辑器扩展

LIST_FORMULA / FORMULA 的 token 字典扩展，允许引用：
- `{field.<字段名>}` — 同行其他字段（现有能力）
- `{lineItem.partNo}` — 卡片料号（新增）
- `{quotation.customerId}` — 报价单上下文（新增）
- `{global.<变量名>}` — 全局变量（现有能力）
- `mat_part.length` BNF token — 走 BNF resolver 解析（现有能力）

公式语义：表达式求值时，token 按上下文变量字典展开。

### 13.3 全局架构图

```
                  ┌────────────────────────────────────────┐
                  │     RuntimeContext (运行时上下文)       │
                  │  lineItem / quotation / user / row /    │
                  │  global                                 │
                  └─────────────┬──────────────────────────┘
                                │ 透传
        ┌───────────────────────┼───────────────────────┐
        ↓                       ↓                       ↓
   ┌──────────┐         ┌──────────────┐        ┌─────────────┐
   │ BNF path │         │ visibleWhen  │        │ Formula     │
   │ 谓词解析  │         │ 表达式求值    │        │ token 展开  │
   └─────┬────┘         └──────┬───────┘        └──────┬──────┘
         │                     │                       │
         ↓                     ↓                       ↓
   ┌──────────┐         ┌──────────────┐        ┌─────────────┐
   │  SQL     │         │ Tab 显隐决策  │        │ 计算结果    │
   │ WHERE 子句│         └──────────────┘        └─────────────┘
   └──────────┘
```

**统一原则**：上下文变量是**唯一**的运行时输入通道。前端代码不在"上下文"以外引入隐式分支；模板/组件配置完全决定行为。

### 13.4 数据契约扩展

#### 13.4.1 path 持久化格式

为了不破坏现有 `basic_data_path: string` 字段，path 仍是字符串，但支持新语法。后端 parser 升级：

```
旧: "mat_bom[bom_type='ELEMENT'].element_name"
新: "mat_bom[bom_type='ELEMENT' AND hf_part_no={lineItem.partNo} AND customer_id={quotation.customerId}].element_name"
```

或者更结构化（可选 P2）：
```jsonc
{
  "name": "元素",
  "field_type": "BASIC_DATA",
  "data_source": {
    "table": "mat_bom",
    "column": "element_name",
    "filters": [
      {"col": "bom_type", "op": "=", "value": {"type": "LITERAL", "value": "ELEMENT"}},
      {"col": "hf_part_no", "op": "=", "value": {"type": "CONTEXT", "ref": "lineItem.partNo"}},
      {"col": "customer_id", "op": "=", "value": {"type": "CONTEXT", "ref": "quotation.customerId"}}
    ]
  }
}
```

**P0 阶段用字符串语法（兼容现有 path）**，P1 阶段可选升级为结构化 JSON。

#### 13.4.2 ImplicitJoinRewriter 重构

原职责：自动注入 hf_part_no=卡片料号 谓词
新职责：
- 解析 path 已声明的谓词
- 检查关键维度（hf_part_no 等）是否已显式声明
- 未声明则注入默认值（向后兼容）
- 用户显式声明 `no_implicit_filter` 时不注入

代码层面：从"硬编码隐式规则"变为"读 path 配置 + 兜底默认"。

### 13.5 4 缺口在新方案下的最终落地

| 缺口 | 终极配置方案 |
|---|---|
| **G1** 元素单位/单价 | path 配 `v_costing_element_price[element_name={row.元素}].unit/costing_price`（行级 JOIN 显式化） |
| **G2** 工序单价 | 同 G1，path 配 `process_default_cost[process_code={row.工序代码}].unit_price` |
| **G3** 成材率公式 | 公式 token 用 `{row.工序代码}` 当 key，公式内引用 GLOBAL_VARIABLE PROCESS_DEFAULT_YIELD；或者用 `v_composite_child_materials.length` (视图加列归基础能力扩展，用户认可) |
| **G4** Tab 显隐 | 组件加 `visibleWhen: "{lineItem.compositeType} = 'COMPOSITE'"` |
| **双轨字段清理** | path 改成智能视图路径 `v_composite_child_*.X`，删 `_composite` 字段（视图自适应 + path 显式声明 hf_part_no = `{lineItem.partNo}`） |

### 13.6 实施分层（5 Layer）

| Layer | 内容 | 前提 | 工时估 |
|---|---|---|---|
| **L1 上下文变量基础设施** | 后端 RuntimeContext 类 + BNF parser 支持 `{...}` 插值 + 校验机制 | 无 | 2 天 |
| **L2 BNF path 谓词显式化 + 老 path desugar** | path parser 升级、ImplicitJoinRewriter 重构为"读配置 + 兜底" | L1 | 2 天 |
| **L3 组件管理 UI 谓词编辑器** | 字段编辑抽屉加结构化过滤条件编辑 | L1 + L2 | 3 天 |
| **L4 Tab 显隐 `visibleWhen` 表达式** | 组件 schema 加属性 + 渲染层求值 + UI 编辑 | L1 | 2 天 |
| **L5 公式编辑器扩展上下文变量** | LIST_FORMULA / FORMULA token 支持 `{lineItem.X}` `{quotation.X}` | L1 | 2 天 |

L1 + L2 是**必做基础**，其他 Layer 渐进上线。

### 13.7 兼容性 & 渐进迁移

| 现有资产 | 兼容方式 |
|---|---|
| 老 path `mat_bom[bom_type='ELEMENT'].element_name` | L2 desugar 自动补 `hf_part_no={lineItem.partNo}` 谓词，零改动生效 |
| `basic_data_path_composite` 双轨字段 | 删除（path 用智能视图后不需要）|

### 13.8 核心权衡

| 维度 | 当前架构 | 新架构 |
|---|---|---|
| 隐式硬规则 | 6 处（自动 hf_part_no / customer_id / is_current / lineItemId 注入等） | 0 处（全部 path 显式声明 + 兜底默认）|
| 用户可控范围 | 仅 path 内字面常量谓词 | 全部谓词 + 上下文变量 |
| 业务表达力 | 受限于隐式规则 | 任意条件组合 |
| 学习曲线 | 低（用户只配字段名）| 中（要理解上下文变量字典）|
| UI 复杂度 | 低（一个 path 字符串）| 中（结构化条件编辑器）|
| 实施工时 | — | L1+L2 必做 4 天，全 5 Layer 11 天 |

### 13.9 给用户的关键决策点

| 决策点 | 选项 |
|---|---|
| 路径语法 P0 实现形态 | A 字符串扩展（path 内写 `{var}`，老 path 自动 desugar）/ B 立即升级为结构化 JSON `filters[]` |
| 上下文变量字典完整度 | A 先开放 lineItem.partNo / lineItem.compositeType / quotation.customerId 3 个核心变量 / B 一次性开放全字典（user/row/global 全包）|
| Tab `visibleWhen` 表达式语言 | A 简单 DSL（`=`/`AND`/`OR`/`IN` 子集）/ B 引入完整表达式引擎（如 SpEL / mvel）|
| 实施顺序 | A 先 L1+L2（基础设施 + path 显式化），再渐进 L3-L5 / B 一次性 L1-L5 全做完才发版 |
| 历史模板迁移 | A admin endpoint 一次性 desugar 所有 path 为显式形态（用户能在 UI 上看到完整谓词）/ B 不迁移，老 path 走 desugar 兜底（用户看到的还是老 path）|

### 13.10 自查：本方案是否仍有"硬代码影响模板逻辑"？

| 嫌疑点 | 是否硬代码 | 理由 |
|---|---|---|
| L2 ImplicitJoinRewriter 兜底注入 `hf_part_no={lineItem.partNo}` | ⚠️ 半硬代码 | 是"默认值"语义，用户可在 path 显式声明 `[no_implicit_filter]` 关闭。**默认行为本身需要在文档说清楚**，不引入"如果是 COMPOSITE 则 X 否则 Y"这种业务分支 |
| L4 Tab `visibleWhen` 求值是代码 | ✅ 配置驱动 | 代码"读表达式忠实执行"，不引入新业务规则 |
| L5 公式 token `{lineItem.partNo}` 展开 | ✅ 配置驱动 | 同上 |
| 上下文变量字典本身（`lineItem.partNo` 等命名）| ⚠️ 半硬代码 | 系统提供的"词汇表"是元数据扩展，类似 SQL 语言的 `CURRENT_TIMESTAMP` 关键字 — 不算硬代码但需要文档化 |

**结论**：新架构把"代码影响模板逻辑"的 6 处反模式降为 **0 处**，剩下 2 处"半硬代码"都是系统级"词汇表/默认值"性质，类似 SQL 关键字，**用户能感知到但不能改是合理的**（除非要做完全可编程的 DSL，那是另一个量级的工作）。

### 12.2 G3 拆解：公式 token 用智能视图 + 视图列扩展归"配置词汇表"

**核心拆分**：
- **第一层（纯配置）**：LIST_FORMULA 公式编辑器允许用户从"可选字段下拉"选基础数据字段，下拉来源是 `basic_data_config` 注册过的所有 sheet/视图的列元数据。**用户选 `v_composite_child_materials.length` 而不是手敲 `{mat_part.length}`**。这本就是组件管理现有能力（公式编辑器的"字段引用"按钮）。
- **第二层（配置词汇表扩展）**：要让用户能选到 `v_composite_child_materials.length`，先要让该列存在于视图 + 在 `basic_data_config` 注册过。视图加列**不是针对某个 bug 的硬代码**，而是**给系统的"可配置词汇表"添加新项**，类似你在系统里加一张主数据表/全局变量 — 用户能在 UI 上看到、选到。

**类比**：用户希望系统支持"按月份查汇率"，那就需要先有"月份"维度的全局变量，这不是 bug fix，是系统能力扩展。视图加列同理。

**实施约束**：
- 视图加列必须有明确业务语义（"length/width/height 是子件的尺寸"）
- 必须同步注册到 `basic_data_config` 让 UI 下拉能列出来
- 不准为某个具体公式定制视图（如果出现"只有 X 公式用得到 Y 列"的情况，回归 DATA_SOURCE 路径）

**G3 替代选项**（更纯粹）：成材率 LIST_FORMULA 整个升级为 `DATA_SOURCE.GLOBAL_VARIABLE PROCESS_DEFAULT_YIELD` + `key_field_refs={process_code:"工序代码"}` — 全局变量层维护"工序→成材率"映射，**根本不用 length/width/height**。这是更优雅的纯配置路径，**前提是业务上能接受成材率不依赖物理尺寸**（这个判断要找用户/PM 拍板）。

### 12.3 G4 解：新增组件元数据 `applicableProductTypes`

**当前**：模板里所有 Tab 强制渲染。"选配-组合工艺" Tab 在独立产品下渲染为空 Tab（视觉污染）。

**完美配置方案**：

#### Step 1 - 组件 schema 加属性

`component` 表（或 component JSONB schema）加字段：

```jsonc
{
  "componentCode": "COMP-CFG-COMPOSITE-PROC",
  "componentName": "选配-组合工艺",
  "applicableProductTypes": ["COMPOSITE"],     // ← 新增。可选值: ["SIMPLE"] / ["COMPOSITE"] / ["SIMPLE","COMPOSITE"]
  // ... 其他字段
}
```

默认值：`["SIMPLE","COMPOSITE"]`（两种都展示，向后兼容）。

#### Step 2 - 组件管理 UI 增加属性编辑器

组件编辑抽屉新增"适用产品形态"多选：
- ☐ 独立产品 (SIMPLE)
- ☑ 组合产品 (COMPOSITE)

用户给"选配-组合工艺"组件勾掉"独立产品"。

#### Step 3 - 渲染层读配置自动隐藏

`QuotationStep2.tsx` Tab 渲染处：

```ts
const visibleTabs = componentData.filter(comp => {
  const applicable = comp.applicableProductTypes ?? ['SIMPLE', 'COMPOSITE'];
  return applicable.includes(lineItem.compositeType ?? 'SIMPLE');
});
```

**这是"代码读配置忠实执行"，不是"代码改变模板语义"** — 模板/组件层声明了规则，渲染层只是按规则隐藏 Tab。本质区别：
- ❌ 反模式：前端写 `if (compositeType==='COMPOSITE') use _composite_path` （修改查询路径，破坏单一配置语义）
- ✅ 配置驱动：前端写 `if (component.applicableProductTypes.includes(compositeType)) renderTab` （忠实执行组件声明）

### 12.5 "硬代码 vs 配置驱动"判断标准

判断一个改动是不是"代码影响模板逻辑"，用这两条：

1. **判断对象**：用户如果想改这个行为，能在 UI / 配置文件改吗？还是必须改源码？
   - 能在 UI 改 → 配置驱动 ✅
   - 必须改源码 → 硬代码 ❌
2. **代码是否引入新语义维度**：代码是只是"读取配置 + 忠实执行"？还是"在配置之外又加了 if-else 分支"？
   - 忠实执行 → 配置驱动 ✅
   - 额外 if-else → 硬代码 ❌

**对照表**：

| 代码片段 | 判断 | 理由 |
|---|---|---|
| `if (lineItem.compositeType==='COMPOSITE') use basic_data_path_composite` | ❌ 硬代码 | 第二条不通过 — 代码引入了"模板有两份 path"的隐式语义，用户在组件管理 UI 看不到这个分支逻辑 |
| `if (component.applicableProductTypes.includes(lineItem.compositeType)) renderTab` | ✅ 配置 | 两条都通过 — 用户在组件管理里看得到、改得了，代码只是读取并执行 |
| `if (tab.tabName.startsWith('选配-组合')) hideForSimple` | ❌ 硬代码 | 两条都不通过 — 按 Tab 名匹配特殊规则 |
