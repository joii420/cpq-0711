# CPQ 数据库 — 未使用视图 / 表 审计报告

> 版本：v1.0 | 日期：2026-06-18 | 范围：`cpq_db`（10.177.152.12）public schema 全部 **10 视图 + 147 表**
>
> 目的：识别**在实际系统逻辑中已不再被使用**的视图与表（旧硬编码逻辑 / 旧模板配置 / 废弃导入链路仍挂着的对象），给出可核验证据与分级清理建议。

---

## 0. 方法论与「使用」定义

一个对象判为「**在用**」，当且仅当被以下**任一**真实引用：

1. **活视图定义**：被某个还在被消费的 SQL 视图 `FROM/JOIN`；
2. **活配置**：`component_sql_view.sql_template` / `template_sql_view` / `component.data_driver_path` / 字段 `basic_data_path` / `global_variable` / `datasource` / **现役（is_default / PUBLISHED）模板的 columns 配置**；
3. **活代码**：Java 里**真实的** native SQL（`FROM/JOIN/INSERT/UPDATE`）或 ORM 查询，且该代码挂在**可达的**端点 / 前端**已挂载路由+菜单**上。

**不算「在用」**（本报告的核心判据）：

- ❌ **有业务数据≠在用**：表里有几千行数据，但无任何代码/视图/活配置读取 → 仍判未使用；
- ❌ **注释 / javadoc 示例**里出现名字；
- ❌ 仅存在于**冻结的旧模板快照**（PUBLISHED template snapshot）里的引用；
- ❌ 仅有**孤儿实体类**（`@Entity` 映射但 Java 里零引用）；
- ❌ 端点/前端 service 文件存在，但**前端无挂载路由或菜单入口**（用户不可达）；
- ❌ 仅被**只写不读**的导入链路写入（write-only 死端）。

**局限声明**：本报告基于代码 + 配置 + DB 元数据的静态证据。「端点在但 UI 未挂载」「导入写入但无消费」这类判定指向**代码层面无消费方**；个别对象是否仍有**运维/手工/外部**用途，需业务方最终确认。标 ⚠️ 的为需人工复核项。

## 6. ⚠️ 风险：活视图仍引用废弃表

`v_costing_summary_full`（**核价主视图、3 个默认模板的数据源**）的定义仍 `FROM mat_fee` + `FROM plating_fee`（V44/V125 废弃表）。

- 后果：违反 AP-53「配置 SQL 应 FROM V6 表」精神；这两张废弃表因被主视图聚合而**无法直接退役**，形成「废弃表被主链路反向钉住」。
- 建议：将 v_costing_summary_full 对 mat_fee / plating_fee 的聚合迁移到 V6/V125 对应表（unit_price / mat_plating_fee / costing_part_plating_fee），方可释放 §3.D 的清理空间。
