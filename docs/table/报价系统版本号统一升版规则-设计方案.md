## 一、范围与目标

### 1.1 范围
- 仅报价系统（`system_type='QUOTE'`）**Excel 导入落库**时的版本号生成 + 生效标志维护。
- 覆盖 5 张带版本表：`element_bom(+item)`、`material_bom(+item)`、`unit_price`、`plating_scheme`、`capacity`。
- **不覆盖**：核价（PRICING）表；版本生效日期区间、报价取数选版本等运行时逻辑（后续独立立项）。

### 1.2 目标
把 `element_bom` 现有的“内容指纹比对 + 数字递增”机制，与选配 V6 §5 的“is_current 生效翻转”机制**合并为一套统一规则**，推广到全部 5 张报价表，消除命名/格式/触发口径三重不统一。

### 1.3 与选配 V6 的关系（关键）
选配 V6 doc §6.7 明确规划“抽取**通用版本化写入工具**（分组键 → max+1 → is_current 翻转），供工序/组合工艺/元素/BOM 复用，并为将来导入复用预留”。本文即“导入复用”的规则定义，**调用同一个工具**，不另造轮子。

## 三、统一升版 + is_current 翻转机制（核心）

### 3.1 定义
- **分组键 (groupKey)** = 业务精简键，标识“同一个被版本化的业务对象”。**不是**唯一键去版本，而是该业务对象的自然身份（排除组内排序/明细维度 `seq_no` / `discount_order` / `item_seq` / `effective_date`，这些归入“内容”）。逐表定义见 §四。
- **指纹 (fingerprint)** = 该 groupKey 下**当前生效版本（is_current=true）整组行集**的内容（所有业务值字段 + 行数），按固定口径归一化后拼接。**行集排序键 = 每行所有 contentColumns 拼接后的归一化字符串（整行元组排序，非仅首列）**，保证行序无关、不会因首列重复而误判（决策 ③）。

### 3.2 写入算法（所有表统一，对齐选配 §5.1）

```
对本次导入按 groupKey 分组，每组 G：
    newFp     = fingerprint(本次导入 G 的整组行)
    curRows   = SELECT * FROM 表 WHERE groupKey=G AND is_current=true
    curVer    = curRows 的版本号（同组同版本）
    curFp     = fingerprint(curRows)

    if curRows 为空:                       # 该业务对象首次落库
        version = '2000'
        INSERT 全组行 (版本=version, is_current=true)

    elif newFp == curFp:                   # 内容/行数完全一致 → 复用，【完全不写库】
        version = curVer                   # 直接返回现版本号，一个字节都不动（对齐选配 P1；不刷新 updated_at/import_record_id）

    else:                                  # 内容变化 → 升版
        version = str( MAX(CAST(版本字段 AS INT)) + 1 )   # 该 groupKey 下全历史最大 +1
        UPDATE 表 SET is_current=false WHERE groupKey=G AND is_current=true   # 旧组整体下线
        INSERT 全组行 (版本=version, is_current=true)
```

- 主从型（element_bom / material_bom）：主表升版/翻转时，子表行同步写主表的 `version` 与 `is_current`（§5.2/§5.3）。
- **本批去重**：同一次导入内多次命中同一 groupKey 时，引擎缓存本批已生成版本号，不重复升版（沿用 element_bom 现有 `groupRows` 思路）。
- `MAX(版本字段)` 取该 groupKey 下**全历史**（含 is_current=false）的最大数字版本，保证版本号单调递增、不复用已下线号。

### 3.3 落地架构：**选配与导入共用同一个版本化工具**
> 🔒 **核心原则（2026-06-01 确认）**：选配落库与 Excel 导入**都需要版本号管理，且共用同一套版本号比对方式 + 升版逻辑**。因此**只有一个**版本化写入工具——选配 P1 已建的 `com.cpq.basicdata.v6.versioning.VersionedV6Writer`（参数对象 `VersionedGroupSpec`）。导入侧**不另建**写入器，直接复用它。

```
VersionedV6Writer (ApplicationScoped，选配 P1 已建，选配/导入共用)
    String writeVersionedGroup(VersionedGroupSpec spec)   # §3.2 全流程：组内行集指纹比对 → max+1 → is_current 翻转

VersionedGroupSpec (选配 P1 已建)
    - tableName / versionColumn
    - groupKeyColumns: Map<列名,值>      业务精简分组键（§四）
    - contentColumns: List<列名>         参与指纹比对的组内行值字段
    - newRows: List<Map<列名,值>>        本组本次要写的整套行
```

- **单表行集型**（unit_price / capacity / plating_scheme）：直接调 `writeVersionedGroup`。
- **BOM 主从型**（element_bom / material_bom）：当前 `VersionedV6Writer` 只做单表行集，**主表版本 + 子表跟随**是共享工具的**待扩展能力**。因选配 P4（补 BOM 主表）与导入 Q03/Q04/Q12 **都需要**，故作为**共享工具的 MASTER_DETAIL 扩展**统一加在 `VersionedV6Writer` 内（子表用 `writeVersionedGroup` 版本化 → 主表按同一版本号镜像写 + is_current 同步翻转），不在导入侧另写一份。element_bom 现有 `fingerprintExisting/fingerprintRows/nv()` 收编进该扩展。
- 各 Q* handler 改为：算 groupKey + 组装 newRows → 调 `VersionedV6Writer.writeVersionedGroup(...)`。
- 移除 `UnitPriceWriter` 的 `versionNo == null ? "V_DEFAULT"` 兜底。

## 四、每表分组键 + 指纹声明

> 🔒 **作用域隔离规则（升版比对与 is_current 翻转的 WHERE 必须遵守）：**
> 1. **含 `customer_no` 的表（unit_price / element_bom / material_bom）：groupKey 必须含 `customer_no`** → 按客户独立升版，A 客户导入绝不读到/翻转 B 客户的行。
> 2. **含 `system_type` 的表（同上三表）：groupKey 必须含 `system_type`，其值 = 当前导入系统（`ctx.systemType`）** → 报价单/选配导入取 `'QUOTE'`、核价单导入取 `'PRICING'`。升版比对与翻转**只作用于本系统的行**：报价导入绝不触碰 `PRICING` 行，核价导入绝不触碰 `QUOTE` 行，两者因 `system_type` 不同而天然隔离、互不干扰。`'QUOTE'` 不是写死的常量——本期范围恰为报价侧（Q* handler）故取 QUOTE；同一套共享工具 + 本规则对核价导入（P* handler，取 PRICING）直接复用。
> 3. **`capacity` / `plating_scheme` 既无 `customer_no` 也无 `system_type` 列** → 无法按客户/系统隔离，是**全局共享表**（与选配 §11 R4 一致）。**已确认（决策 ②）：本期接受全局共享，不加 `customer_no`**——不同客户同 `material_no`/`scheme_no` 落同一版本序列、相互翻转 `is_current`。升版/翻转作用于该 groupKey 全局；若将来需客户隔离，须另行加列（超出本期）。
>
> 通用：`unit_price` 各 Sheet 的 groupKey = 固定写入的 `(system_type=QUOTE, customer_no, price_type, cost_type)` + 业务主体标识列；`seq_no / discount_order / item_seq` 是**组内明细维度**——不进 groupKey、进指纹内容。`effective_date` **导入写 NULL**（决策 ⑪：V220 遗留 uq 维度，已被版本机制取代）——既不进 groupKey 也不进指纹。

## 五、is_current 翻转细则 + 边界

### 5.1 翻转语义（V277 已加列）
- 每张表（含 `material_bom_item`）均有 `is_current BOOLEAN NOT NULL DEFAULT TRUE`。
- 升版：`UPDATE 表 SET is_current=false WHERE groupKey=G AND is_current=true` → 再 INSERT 新版本行 `is_current=true`。
- 复用（内容未变）：**完全不写库**，直接返回现版本号；`is_current`、内容、`updated_at`、`import_record_id` 均不动（对齐选配 P1，决策 ①）。
- `capacity` / `material_version_mgmt` / `tooling_cost` 既有 `is_effective` **保留不改名**；生效判定**统一以 `is_current` 为权威**。

### 5.2 element_bom 主从同步
主表 `element_bom` 升版（characteristic max+1）+ 翻转后，子表 `element_bom_item` 整组按新 characteristic 写入、`is_current=true`，旧 characteristic 组 `is_current=false`。子表 uq 含 characteristic → **可物理保留多版本子表明细**。

### 5.3 ⚠️ material_bom 主从 + 子表多版本限制（已知约束）
- 主表 `material_bom` 版本走 `bom_version`，升版保留历史版本行（is_current 翻转）。
- 子表 `material_bom_item` **uq 不含版本维度**（characteristic 是语义标签 NULL=材质自指/ASSEMBLY=子配件，**非版本号**）→ 不同 bom_version 的子表行会在 uq 上冲突，**无法物理保留多版本子表明细**。
- 处理（选配 §10#1 方案 a）：子表 `ON CONFLICT DO UPDATE` 覆盖为当前内容，`is_current` 跟随主表当前版本（始终对应最新生效组）。**主表保留版本历史，子表仅保留当前版本明细**。
- 这是接受的限制（与选配 V6 一致）；若将来需子表全版本历史，须另行迁移给 `material_bom_item` 唯一键加 `bom_version`（超出本期范围）。

### 5.4 元素回收折扣（Q05）为更新，不升版（决策 ⑤）
按 **2 键 `(material_no, component_no)`** 匹配 `element_bom_item` 中 **`is_current=true`** 版本组的行，`UPDATE recovery_discount`。**不用 seq_no**（该 Sheet 不导入项次，§5 明细口径）；命中多行则批量更新（同元素理论上不应多行）。不产生新版本、不翻转 is_current。
> ⚠️ 与导入方案"通用规则"第 9 行的 3 键写法不一致——本设计以 §5 明细表为准（决策 ⑤），实现期同步回写导入方案的通用规则行。

### 5.5 其它边界
| 场景 | 处理 |
|------|------|
| 指纹归一化 | 数字去尾零（`70.00`==`70`）、null 与空串等价；**组内行按"整行 contentColumns 拼接归一化串"排序**（决策 ③，非仅首列，避免首列重复误判）；统一下沉到引擎 `nv()` 口径 |
| ⚠️ 共享工具改造（决策 ③） | 选配 P1 的 `VersionedV6Writer.rowsEqual` 现按 `contentColumns.get(0)` 单列排序 → 须改为按整行内容元组排序。属共享工具修正，选配同样受益，实现期与选配窗口协调 |
| `MAX(版本)` 非数字 | 空库不触发；若遇历史非数字（V_DEFAULT 等）解析失败则跳过取其余数字最大，全非数字按首版 `'2000'`（配合空库重导，基本不触发） |
| 空库/清档 | 不写存量归一迁移；TRUNCATE 后按新规则重导，全部从 `'2000'` 起 |
| `effective_date`（决策 ⑪） | 导入写 NULL；不进 groupKey/指纹；`UnitPriceWriter.newRow` 现有的 `effectiveDate = LocalDate.now()` 须改为不赋值（NULL）。它是 V220 遗留 uq 维度，版本历史由 version_no+is_current 管 |

## 八、测试方案
### 8.1 单元（VersionedV6Writer，含 MASTER_DETAIL 扩展）
- 首版：组无生效记录 → 版本 `'2000'`、is_current=true。
- 复用：相同指纹 → 版本不变、不新增行、is_current 维持 true。
- 升版：指纹变化 → 版本 max+1、新行 is_current=true、旧组行全部 is_current=false。
- 归一化：`70.00` vs `70`、null vs ''、组内行序无关 → 指纹一致。

### 8.2 集成（每表）
- 导入两遍完全相同数据 → 版本不变、生效行数不翻倍、`COUNT(is_current=true)` 稳定。
- 改一个值再导入 → 版本 +1，旧版本 `is_current=false` 保留，新版本 `is_current=true`。
- 主从（element_bom / material_bom）：改子表一行 → 主版本 +1；element_bom 子表保留双版本，material_bom 子表仅当前版本（§5.3）。
- plating_scheme：同 scheme_no 改一行 → 整 scheme 升一版。
