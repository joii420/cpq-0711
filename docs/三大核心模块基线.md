## 二、组件管理 (Components)

### 2.1 数据结构 (`component` 表)

```jsonc
{
  "id": "UUID",
  "code": "COMP-CFG-MATERIAL-RECIPE",
  "name": "选配-材质",                    // 渲染时变成 Tab 名
  "componentType": "NORMAL" | "SUBTOTAL",
  "dataDriverPath": "v_composite_child_materials",  // 单条 / 多条数据来源
  "fields": [...],                        // 字段定义 (核心)
  "formulas": [...],                      // FORMULA 字段公式定义
  "preset_rows": []                       // 预置行(常量)
}
```

### 2.2 字段类型矩阵

每个字段必须明确其类型 + 数据获取方式:

| field_type | 用途 | 关键配置 |
|---|---|---|
| **BASIC_DATA** | 从 DB 视图/表取值 | `basic_data_path` (BNF path, 支持 `{lineItem.partNo}` 等上下文变量) |
| **DATA_SOURCE** | 从结构化数据源取值 (主流: 全局变量) | `datasource_binding`: `{type:GLOBAL_VARIABLE, global_variable_code, value_field, key_field_refs}` |
| **FORMULA** | 同行公式求值 | `formula_tokens`: `[{type:field,value:含量},{type:operator,value:"/"},...]` |
| **LIST_FORMULA** | 多分支按行级 key 选公式 | `list_formula_config.per_item_rules`: `{ "MRO-AS-0001": {branches:[{condition,formula}]} }` |
| **INPUT_NUMBER** | 用户手填 (默认值可选) | `content` 默认值 |

### 2.3 driver path 语义 (BNF)

**结构**: `表/视图名[谓词条件].取值列`

谓词支持：
- 字面常量: `bom_type = 'ELEMENT'`
- 上下文变量: `hf_part_no = {lineItem.partNo}` / `customer_id = {quotation.customerId}`
- 行级字段: `element_name = {row.元素}` (父子表 JOIN)
- 全局变量: `fiscal_year = {global.CURRENT_FISCAL_YEAR}`

**兜底注入**：path 未显式声明 `hf_part_no` 谓词时，ImplicitJoinRewriter 自动注入 `hf_part_no = {lineItem.partNo}`（向后兼容）。

**显式关闭**：`mat_X[no_implicit_filter=true].column` 完全跳过兜底。

#### 2.3.1 `$` 前缀引用语法（2026-05-25 扩展，组件 SQL 视图数据源）

BNF path 支持 `$` / `$$` 前缀作为命名空间标记，引用用户在组件管理配置的 **inline SQL 视图**（物理上不发 DDL，但语义上等价 PG view）：

| 数据源 | 写法 | 解析后 |
|---|---|---|
| 物理表/视图 | `mat_part[谓词].col` / `v_q_*[谓词].col` | 不变（直接当 FROM 表达式） |
| **本组件 SQL** | `$<sql_view_name>[谓词].col` | `(SELECT ...) <sql_view_name>` 作为 FROM 表达式 |
| **跨组件 GLOBAL SQL** | `$$<componentCode>.<sql_view_name>[谓词].col` | 跨组件查找 + inline subquery（要求 scope=GLOBAL） |

**关键不变量**：
- ✅ 渲染层 / cache key / ComponentDriverService 三分支 / ImplicitJoinRewriter / ComponentCell **完全不动**
- ✅ batch 优化与既有 `ANY(:hfPartNos)` 机制自动融合（inline subquery 包装后仍受外层 batch filter）
- ✅ 用户在 SQL 中自己写 UNION 处理 SIMPLE/COMPOSITE，与 §6.3 红线"SIMPLE/COMPOSITE 配置层统一"一致
- ❌ 用户 SQL **禁用 `:hfPartNo`** 标量占位符 —— 由外层 batch 注入
- ❌ 用户 SQL **禁止 DDL/DML**，仅允许 SELECT（保存时 EXPLAIN 校验）

详见 `docs/archive/组件级数据源SQL方案.md` §五（运行时执行流程）。

### 3.2 components_snapshot 机制

**为什么需要 snapshot**：组件管理改字段后，老模板的历史报价单不能反向受影响（否则历史报价改写）。模板发布时把组件的当时状态冻结进 `componentsSnapshot` JSONB。

**snapshot 生成**：
- 模板 DRAFT → PUBLISHED 时由 `TemplateService.refreshSnapshotsByComponent` 拷贝 `component.fields` + `component.dataDriverPath` 进 snapshot
- 组件管理改字段 + 调 H1 端点时同步刷新所有 PUBLISHED 模板 snapshot (但只在用户明确触发时)

**snapshot 是渲染时的权威来源** — `template.componentsSnapshot[i].fields` 决定渲染什么字段。

#### 3.2.1 sql_views_snapshot 双层冻结（2026-05-25 扩展，与组件 SQL 视图同步）

组件 SQL 视图 (`component_sql_view`) 走与 `component.fields` 一致的双层冻结机制：

| 冻结层 | 触发 | 落地 |
|---|---|---|
| 模板层 | 模板 DRAFT → PUBLISHED | `template.sql_views_snapshot` JSONB（与 components_snapshot 同事务，由 `refreshSnapshotsByComponent` 扩展冻结） |
| 报价单层 | 报价单 DRAFT → SUBMITTED | `quotation_component_sql_snapshot` 表（与 `*_snapshot` 列同事务） |

**lookupSqlView 优先级**（运行时执行）：
1. 报价单 APPROVED/PUBLISHED → 优先读 `quotation_component_sql_snapshot`
2. 模板 PUBLISHED → 读 `template.sql_views_snapshot`
3. 兜底实时读 `component_sql_view`（DRAFT 期使用）

**跨组件 GLOBAL 引用闭包**：模板 PUBLISHED snapshot 时递归收集所有 GLOBAL scope 的 SQL 视图（即使引用了其他模板未挂载的组件），冻结至 snapshot 闭包，保证回放稳定。

详见 `docs/archive/组件级数据源SQL方案.md` §六（冻结策略）。

### 3.3 字段覆盖 (template_component.fields_override) — 已废弃

历史上模板可以通过 `template_component.fields_override` 覆盖组件字段（如修 path / 加新字段）。本轮 (2026-05-21) **完全废弃**：
- 60 PUBLISHED 模板的 fields_override 全部清空 (NULL)
- "子件"等历史 override 字段通过 `promote-override-to-component` 端点上升回 `component.fields`
- 单一字段来源 = component.fields → snapshot.fields → 渲染

**保留 `patchComposite` admin endpoint** 作为 @Deprecated 紧急运维入口，不在常规流程使用。

### 3.4 版本管理 (createNewDraft)

`TemplateService.createNewDraft(templateId)`:
- 拷贝当前模板为新 DRAFT (递增 version)
- `mergeFieldsOverrideForNewDraft` (override-priority 基底) 已废弃 _composite 处理，直接拷贝
- 用户在 DRAFT 上编辑 → 发布为新 PUBLISHED → 老 PUBLISHED 自动 ARCHIVED（同 series 版本）

### 3.5 admin 运维端点清单

| endpoint | 用途 | 状态 |

| `POST /templates/admin/{id}/delete-tcs` | 删 PUBLISHED 模板某 Tab | 保留 |

### 4.5 ComponentDriverService.expand — 三分支策略 (核心)

```java
boolean isCompositeAggregateView = effectiveDriverPath.contains("v_composite_child_");
boolean isCompositeParent = "COMPOSITE".equals(compositeType);

if (lineItemId != null && !(isCompositeAggregateView && isCompositeParent)) {
    // 分支 1: SIMPLE 场景 (或 SIMPLE 走聚合视图路径)
    // 注入: WHERE quotation_line_item_id = :lid OR IS NULL
    // 效果: 精确限定到当前 lineItem 的专属行 + 主数据 fallback
}
else if (isCompositeAggregateView && isCompositeParent 
         && childLineItemIds != null && !childLineItemIds.isEmpty()) {
    // 分支 2: COMPOSITE 父级聚合子件
    // 注入: WHERE quotation_line_item_id IN (:childIds) OR IS NULL
    // 效果: 限定到本报价单的子件 lineItem.id, 不再累积历史
}
else {
    // 分支 3: 兜底 (无 lineItemId 或 COMPOSITE 无子件)
    // 不注入 lineItemId, 走 hf_part_no 默认逻辑
}
```

### 4.6 footer 列小计单一来源化（columnSumsByComp）【2026-06-16】

**纪律**：报价 / 核价 / 详情三视图的 footer（tfoot）与 `allComponentSubtotals`，所有数值列（含**非小计**列）的合计值，统一从 `buildCrossTabRows` 的 `resolvedRows` 求和得到的 `columnSumsByComp` 单一来源读取，**不再另起炉灶二次重算**。

- `buildCrossTabRows` 改返回 `{ store, columnSumsByComp }`：`columnSumsByComp` 是「每组件 × 每数值列」从该组件已渲染的 `resolvedRows` 求和的权威值（走新原语 `sumColumnsCanonical`，读 canonical 值）。
- **废弃**死函数 `computeNonSubtotalColumnSums`（以**空 `crossTabRows`** 二次重算的旁路）—— 它与数据行所用的完整 `store` 分叉，是 footer 与行值对不上的根因。
- `buildCrossTabRows.computeRows` 补串 `prevRowValues`，修复其与渲染层 `effectiveRows` 之间隐藏的求值分叉（同一行在数据区与 footer 求和区取值不一致）。
- **`¥` 严格按 `field.is_amount === true`** 渲染金额前缀；非金额列（即便是 `is_subtotal` 列）显示**纯数字**，不再"小计列默认 ¥"。

**后端（A-保守，零改动）**：`backfillSubtotalsFromResolved` 仍只把 `is_subtotal` 列从 `resolvedRows` 求和落库（其本就是"从 resolvedRows 求和"的正确同源架构，本次是前端向它看齐）；非小计列的合计为纯前端运行时值，**不落库**。

**不变量**：footer 上某数值列的显示值 === 该列所有已渲染 `resolvedRow` 之和（无论小计 / 非小计、cross_tab / SUMIF / 普通列）。

**未做（后续 Phase，A-彻底）**：非小计列合计落库（`subtotalByColumn`）+ Excel 引用语义扩展到非小计列 + Phase4 零计算直读快照。详见 [[反模式 AP-57]]。

## 五、关键架构机制

### 5.1 V202 智能视图自适应 (DB 层屏蔽 SIMPLE/COMPOSITE 差异)

`v_composite_child_materials` / `v_composite_child_elements` / `v_composite_child_processes` / `v_composite_child_weights` 各有两个 UNION ALL 分支：

```sql
CREATE OR REPLACE VIEW v_composite_child_elements AS
-- 分支 A: COMPOSITE 父级 → 聚合子件 mat_bom ELEMENT 行
SELECT asy.hf_part_no, child_part_name, elem.seq_no, ...
FROM mat_bom asy
JOIN mat_bom elem ON elem.hf_part_no = asy.input_material_no AND elem.bom_type = 'ELEMENT'
WHERE asy.bom_type = 'ASSEMBLY'

UNION ALL

-- 分支 B: SIMPLE 自身 → 直接拿 mat_bom ELEMENT
SELECT elem.hf_part_no, elem.hf_part_no AS child_part_name, elem.seq_no, ...
FROM mat_bom elem
WHERE elem.bom_type = 'ELEMENT'
  AND NOT EXISTS (SELECT 1 FROM mat_bom WHERE hf_part_no=elem.hf_part_no AND bom_type='ASSEMBLY');
```

**重要**：视图本身不知道运行时 lineItem 类型，但因为 SQL `WHERE hf_part_no=:p` 同时只能命中一个分支（hf 要么是 CFG-COMBO 父级要么是物理子件），**视图层 + 应用层注入 lineItemId 谓词 = 完整自适应**。

### 5.2 ImplicitJoinRewriter (兜底注入 + 上下文插值)

L2 重构后的逻辑：
1. ContextInterpolator 展开 path 内的 `{var}` 占位符
2. 检测 `[no_implicit_filter=true]` → 完全跳过
3. 检测 path 是否显式声明 `hf_part_no` 谓词
4. 未声明 → 兜底注入 `hf_part_no = {lineItem.partNo}` (向后兼容)
5. lineItemId 维度: 见上面 §4.5 三分支策略

### 5.3 mat_process UNIQUE index (V210)

```sql
CREATE UNIQUE INDEX uq_mat_process_current
  ON mat_process (
    customer_id, hf_part_no, part_version, seq_no,
    COALESCE(sub_seq_no, -1),
    COALESCE(quotation_line_item_id, '00000000-0000-0000-0000-000000000000'::UUID)
  )
  WHERE is_current = true;
```

**关键**：用 COALESCE 把 NULL 折叠为哨兵值，避免 PG `NULL != NULL` 让重复行共存。

### 5.4 上下文变量字典 (L1 基础设施)

```java
// RuntimeContext.java — 系统级标准变量字典
class RuntimeContext {
    LineItemContext lineItem;    // {id, partNo, compositeType, tempId}
    QuotationContext quotation;  // {id, customerId, customerCategory, productCategoryId}
    UserContext user;            // {id, role}
    Map<String, Object> row;     // 行级 (动态)
    Map<String, Object> global;  // 全局变量
}
```

**ContextInterpolator** 提供 `interpolate(template, ctx)` 把 `{namespace.field}` 替换为实际值。

### 5.5 公式引擎运行时重写 + gvDefs 透传链路 (2026-05-21 沉淀)

公式引擎 `evaluateExpression` 是**前端纯函数求值器**，但全局变量 token 需要三种数据源支撑：

```
                     ┌──────────────────────────┐
                     │  evaluateExpression       │
                     │  (formulaEngine.ts)       │
                     └────────┬─────────────────┘
                              │
        ┌─────────────────────┼─────────────────────┐
        │                     │                     │
        ▼                     ▼                     ▼
  globalVariableDefs       currentRow            pathCache / _globalPathCache
  (GV 元数据字典)          (本行字段值)          (后端预热的 path → 值)
        │                     │                     │
        ▼                     ▼                     ▼
  QuotationStep2/Wizard/    effectiveRows.row    usePathFormulaCache
  QuotationDetail            (driver-bound 时    (按 partNo + path 写入)
  顶部 useState +            是 comp.rows[i])
  globalVariableService
  .list() useEffect
```

#### 5.5.1 三种全局变量 token 编译策略

| 场景 | token 形态 | 编译时机 | path 形态 |
|---|---|---|---|
| 静态 key | `{type, code, key_values:{key:'Ag'}}` | 编辑期 `compileGlobalVariableToPath` 即写入 token.path | `v_costing_*[col='val'].col2` 或 `global_variable_value[var_code='X' AND key_id='Ag'].value_number` |
| 动态 key | `{type, code, key_field_refs:{key:'元素'}}`, `path=undefined` | **运行时**按 currentRow 重写 | 求值期 `compileGlobalVariableTokenForRow(token, def, currentRow)` 现拼 |
| COSTING_VIEW lookup | `v_costing_element_price` 等真实视图 | V109 path 形式直引 | 走 ImplicitJoinRewriter pipeline |
| KV_TABLE lookup | 数据存 `global_variable_value` 单表 | V190 后统一 | `global_variable_value[var_code='X' AND key_id='Y'].value_number` |

#### 5.5.2 透传链路（必须 5 处全覆盖）

`evaluateExpression` / `computeAllFormulas` / `computeFormula` 凡是接受 `globalVariableDefs` 的函数，**所有调用点必须传**：

| # | 调用点 | 文件 | 漏传后果 |
|---|---|---|---|
| 1 | 编辑页 ProductCard 渲染 | `QuotationStep2.tsx` `computeAllFormulas` / `computeTabSubtotal` | 实时单价显示 0 |
| 2 | autoSave 落库 | `QuotationWizard.tsx` `buildDraftPayload → computeAllFormulas` | snapshotRows 写 0 → 重载后单价全 0 |
| 3 | 只读详情页 | `ReadonlyProductCard.tsx` `computeFormula` | 详情页单价显示 0 |
| 4 | partNo 透传 | `ReadonlyProductCard.tsx` `computeFormula` 内 `evaluateExpression` 第 7 参数 | cache lookup key 变 `'::path'` 永远 miss |
| 5 | 详情页预热 | `QuotationDetail.tsx` 必须自调 `usePathFormulaCache` | 详情页 `_globalPathCache` 空 → cache miss |

#### 5.5.3 fingerprint 依赖必须含 gvDefs

`usePathFormulaCache.fingerprint` useMemo 依赖数组：

```typescript
const fingerprint = useMemo(() => {
  // 收集动态 key path 并入指纹
  if (tok.type === 'global_variable' && !tok.path && tok.key_field_refs && tok.code) {
    const def = globalVariableDefs?.[tok.code];   // ← 依赖 gvDefs
    ...
  }
}, [lineItems, globalVariableDefs]);   // ← 必须列入, 否则 gvDefs 异步到位后 tasks 不重建
```

#### 5.5.4 诊断 SOP（已内置）

`formulaEngine.ts` case `'global_variable'` 内置 3 段诊断 log（守护 `window.__GV_DEBUG__`，零开销）：

| log | 含义 |
|---|---|
| `[gv-debug] def miss` | gvDefs 不含 token.code（race condition / list 失败 / code 命名错）|
| `[gv-debug] pathStr empty` | 重写失败（currentRow 缺字段 / token 缺 code）|
| `[gv-debug] cache miss` | path 重写成功但 cache 未命中（partNo 缺前缀 / path 字符串微妙差异）|

排查协议：用户 F12 跑 `window.__GV_DEBUG__ = true` → 刷新页面 → 截图 console → 按 log 类型按 §5.5.2 表查漏点。

详见 [[反模式 AP-49]]。

## 六、配置驱动原则（5 条红线）

### 红线 1: 字段渲染必须配置表达
- ✅ 允许：用户在组件管理 UI 看见、改得了的字段定义
- ❌ 禁止：前端代码 `if (compositeType==='COMPOSITE') swap_path` 切换查询

### 红线 2: 数据过滤条件必须通过 RuntimeContext 声明
- ✅ 允许：path 内 `{lineItem.partNo}` / `{quotation.customerId}` 显式插值
- ❌ 禁止：后端代码硬编码 WHERE 谓词

### 红线 3: SIMPLE/COMPOSITE 在配置层统一
- ✅ 同一模板 + 同一份 fields → DB 视图层 (V202) + ComponentDriverService 三分支 自动适配
- ❌ 模板字段不再有 `_composite` 后缀

### 红线 4: 模板字段单一来源
- ✅ template.components_snapshot 渲染时权威 (从 component.fields 拷贝)
- ✅ template_component.fields_override 永久空 (NULL)

### 红线 5: 上下文变量字典只通过系统级扩展
- ✅ 添加 `lineItem.X` / `quotation.X` 新变量是基础设施任务 (改 RuntimeContext 类)
- ❌ 在模板/组件里发明私有变量
