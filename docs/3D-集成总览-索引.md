## 二、按业务场景的决策树（v0.4 单一主线）

```
❓ 我要做什么？

┌─ 客户/销售要从头配置一台产品（型号/材质/颜色/部件/工艺）
│  并最终生成报价单
│       → 读 3D产品选配方案.md
│       → 体验 v0.4-3D产品选配原型.html

├─ 我要导入 UG NX 设计好的 .prt + .stp 文件到系统
│  或者已有现成的 .glb 文件直接上传
│       → 读 3D产品选配方案.md §六（UG NX 工作流）
│       → 看 CAD转换POC-技术验证.md（Docker 设计）
│       → 体验 v0.4-3D源文件上传与转换原型.html（含 .glb 直传分支）

├─ 我是产品经理，要维护选配模板（选项树 / 约束 / 3D 规则 / 价格规则）
│       → 体验 v0.4-3D选配模板管理端原型.html

├─ 我要给客户发个公网链接让他自己选配置
│       → 用 §十七 分享协作（CUSTOMER_SELF 类型）
│       → 体验 v0.4-客户自助选配原型.html

├─ 我要看所有客户的选配实例 / 绑定到报价单
│       → 体验 v0.4-选配实例列表原型.html
│       → 弱关联：选配实例 ↔ 报价单 双向独立

├─ 我要维护"产品特征字段库"（如型号 / 厚度 / 表面处理的取值集合）
│  让多个选配模板复用，避免每个模板手撕一遍
│       → 读 3D产品选配方案.md §18A（方案 B 快照复制）
│       → 体验 v0.4-特征库管理原型.html
│       → 在 v0.4-3D选配模板管理端原型.html 编辑器里点「📥 从特征库选择」即可导入
│
└─ 我要维护客户专属配置 / 审批规则 等灵活配置
        → 读 3D产品选配方案.md §十三~§十七
        → 暂无独立原型（管理后台直接配置）
```

## 八、阶段 5 待补算法清单（管理端 3 Tab 完全隐藏依据）

> 2026-05-26 决定：管理端原型 v0.4 中以下 3 Tab **完全从 UI 移除**（不留入口），强制等算法文档化完成后再开放。
> 目的：避免 3 个未文档化引擎并行实现 → 交叉 bug 不可调试（AP-44 同源）。
> 注：本节为路线图记录，不是原型展示项 — 防止后期忘记这些功能仍需实现。

| Tab | 缺失算法 / DSL | 应补章节 | 解锁前置条件 |
|---|---|---|---|
| ⚠ 约束规则 | 5 类约束 (REQUIRES/EXCLUDES/IMPLIES/HIDES/NUMERIC_RANGE) **执行顺序** + **循环依赖检测** + **HIDES 是 UI 层还是 evaluate 拒绝层** | `3D产品选配方案.md §3.4.1` 求值算法（伪代码 + 顺序图） | evaluate API §9.3 跑通 ≥ 100 个真实选配组合无死循环/无遗漏拒绝 |
| 👥 客户覆盖 | 多覆盖**合并优先级**（VIP 同时有 PRICE_OVERRIDE + OPTIONS_OVERRIDE + CONSTRAINT_RELAX 谁先）+ **跨模板传递规则** | `3D产品选配方案.md §14.3` 合并优先级表 + §14.5 算法伪代码 | 至少 2 家 VIP 客户配过覆盖且无冲突 ≥ 4 周 |
| ✅ 审批规则 | 触发条件 **DSL 语法**（字符串表达式 vs JSONB 结构化 vs 复用现有公式引擎）+ `has_custom_value` 等语义定义 | `3D产品选配方案.md §16.4` DSL 规范 + DSL 解析器复用决策 | 至少 1 条审批规则在阶段 4 灰度模拟跑通 |

**禁止行为**：
- ❌ 在算法文档化之前**直接实现这 3 个 Tab 的后端逻辑**（凭感觉实现 = AP-44 同源 bug 必然出现）
- ❌ 在 v0.4 阶段 3-4 提前在管理端 UI 加回这 3 个 Tab 入口（必须连同算法 PR 一起开放）

**解锁流程**：
1. 算法章节 PR 合并 + cpq-architect agent review
2. 后端实现 + cpq-tester ≥ 100 case 测试
3. v0.4 管理端原型 HTML 加回对应 Tab 导航 + 内容区 + 渲染函数（参考 git 历史 commit affda73 之后的版本）
4. RECORD.md + 本文档变更记录登记
