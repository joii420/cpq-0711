### ⚠️ 重要说明：本文件未触发「铸号」与「跨客户」两类路径

- **罗克韦尔文件所有料件均为预编号（文件既有号）**，不存在「无号有名」的组成件 → 未走 `mintAndRegister`，故 `quote_customer_code` **未给 CUST-1269 分配四位码**（符合规格：只有需要铸新号时才分配）。
- 因此规格中 **测试§3.5（BOM 无号组件铸报价料号）、§1.1（四位码分配）、§3.4/§3.8（跨客户守卫）** 三类路径**无法由本单客户文件驱动**，改由后端 JUnit 集成用例覆盖（见 §4，全绿）。

## 6. 附带发现 & 环境处置（需知会）

**问题**：测试开始后不久，共享 dev server（8081）进入 `Error restarting Quarkus`；首轮全量 v6 套件也报 1 个 `P06MaterialBomHandlerTest » Failed to start quarkus`。

**根因**：sibling worktree 已把 **V311（pricing sales part no）/ V312（industry dictionary）** 迁移应用到共享 DB，但 master 本地 `db/migration/` 只到 V310 → Flyway 严格校验报「**Detected applied migration not resolved locally: 311 / 312**」→ **每次 Quarkus 启动/热重载都崩**（`out-of-order=true` 不覆盖此类「已应用但本地缺文件」）。与本次 Spec 1 改动**零关联**。

**处置**：把两份**已应用**的迁移文件从 sibling worktree 复制进 master 的 `db/migration/`（未 `git add`，为 untracked），使 Flyway 能解析 → dev server 恢复 401 健康 → P06 转绿。

> ⚠️ **这两份 untracked 文件建议保留**：共享 DB 已应用 V311/V312，删除后 master dev server 下次重启会再次崩溃。它们不进 git 提交，仅供本地 Flyway 校验对齐。

**共享 DB 数据留痕**：本次向共享 dev DB 写入了罗克韦尔（CUST-1269）真实导入数据（material_customer_map 137 行 + BOM/element/unit_price/capacity 等），系测试固有副作用，如需清理请知会。
