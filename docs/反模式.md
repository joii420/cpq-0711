# CPQ 反模式速查

> 用途：记录 CPQ 项目里反复发生过、容易产生「数据看起来保存了但其实没保存」/「保存了但刷新后联动功能失灵」/「保存返回成功但实际 4xx」的失败模式。
> 每条都给：现象 / 根因 / 重现路径 / 防护措施 / 历史命中记录。
> 维护人：CPQ 多 Agent 共享记忆。新 Bug 命中以下任一模式时直接补「历史命中记录」，不要新开 Anti-Pattern；命中新模式时再追加章节。

## AP-1：UUID 字段以空字符串 `""` 形态传到后端

### 现象
- 前端调 `PUT /quotations/{id}/draft`、`POST /...` 等含 UUID 字段的接口，HTTP 返回 `400 HTTP 400 Bad Request`。
- 但是 `axios` 的 catch 分支或 `autoSave` 兜底逻辑会把错误吞掉（写 localStorage、`message.warning('网络异常')`、或干脆 `try{}catch{}` 静默），用户看到「保存成功」的提示也好、看不到任何报错也罢，**DB 始终没动**。
- 用户刷新后行还在但子表数据为空 → 误以为是渲染/读取 Bug。

### 根因
- 后端 DTO（`SaveDraftRequest.LineItemDraft.productId` 等）声明为 `UUID` 类型，Jackson 在反序列化阶段无法把 JSON 里的 `""` 转成 `UUID`，直接抛 `JsonProcessingException` → RestEasy 包成 `WebApplicationException 400`。
- 前端在某些分支为了"占位"把 UUID 字段写成 `''`（典型见批量导入流程：`productId: ''  // V5 流程不依赖 v3 product 表`）。
- 整次 PUT 全包失败 → 这次 saveDraft 的所有改动都丢。

### 重现路径
1. 任何 lineItem 类型对象里 `xxxId` 为空字符串。
2. 前端 `service.create / .update / saveDraft` 把整个对象 JSON 化提交。
3. 后端任意一个字段反序列化失败 → 整次请求 400 → 前端 catch 不展开错误。

### 防护措施（编写者必须做）
1. **写入侧（强制）**：所有发往后端的 payload 在序列化前对 UUID 字段一律 `value || null`，**禁止**把 `''` 传出去。`buildXxxPayload(...)` 这类聚合函数是统一关卡：
   ```ts
   productId: li.productId || null,
   templateId: li.templateId || null,
   contactId: values.contactId || null,
   ```
2. **后端侧（防御）**：能改的话给 `SaveDraftRequest`-类 DTO 增加 `JsonDeserializer` 把空串视为 null；至少在 GlobalExceptionMapper 里把 `JsonProcessingException` 落成有原始字段名的 400 消息，便于诊断。
3. **前端 service 侧（兜底）**：catch 分支不要静默吞 4xx；至少 `console.warn(err.response?.data)`，以便控制台显形。

| 日期 | 文件 / 行号 | 现象 | 修复 |
|------|-------------|------|------|
| 2026-04-30 | `BulkImportPartsDrawer.tsx:165` 给 `productId: ''` | 批量导入零件 → 保存草稿 → 刷新数据丢失（DB 里 `product_id=NULL`、所有 `row_data` 都是初始空值） | `QuotationWizard.buildDraftPayload` 统一把 productId/templateId 空串归零为 `null` |

## AP-2：保存请求 DTO 丢字段 — 前端有 X，后端 SaveRequest 没 X

### 现象
- 用户填了某个值（料号 / 客户料号 / 显示名称…），UI 看起来都对。
- 保存成功，刷新后该值消失 / 显示为空。
- 上层依赖该值的功能（driver 展开、BASIC_DATA 路径解析、公式引擎、price-mapping、ExcelView 渲染）跟着全部失效。

### 根因
- 前端 lineItem-type 状态里的字段是「客户端派生 + 客户端输入」混合体。
- 后端 `SaveDraftRequest.LineItemDraft` 只对应了**一部分**字段（早期偷懒）。
- saveDraft 主体在持久化前从 `liDraft.xxx` 取值，DTO 没有的字段就直接消失。
- DTO 回读 (`LineItemDTO.from(li)`) 时这些字段也是空 → 前端 `applyQuotationData` 看到的是后端给的稀疏视图，原本内存里那份 partNo 已经被 `setLineItems(api 返回)` 覆盖。

### 重现路径
1. 前端某个 drawer / wizard 构造 LineItem 时塞入「派生字段」（例：`partNo`、`customerPartNo`、`displayName`）。
2. `buildXxxPayload` 不把这些字段放进 payload。
3. 后端 saveDraft 也没看到、没写。
4. GET 时回不来 → 刷新后丢。

### 防护措施
1. **建模时立约**：每加一个 `lineItem.xxx` 字段，先回答：「刷新后这个字段还要不要？」要 → 必须：
   - SaveRequest DTO 加字段
   - saveDraft service 把 `liDraft.xxx` 写入实体（必要时落到 `_snapshot` 列）
   - ResponseDTO `from(entity)` 回读
   - 前端 `applyQuotationData / applyXxxData` 重新装载到 state
2. **DTO 自检清单**：把 SaveDraftRequest / *Response*DTO 的字段列举出来，跟前端 `LineItem` interface 对照逐字段 round-trip 测试。任何一个不能 round-trip 的字段必须在前端类型上加注释 `// transient — not persisted`。
3. **测试夹**：每个核心实体至少要有「保存→读取→比对」的集成测试（Given a payload with all fields → save → re-fetch → 字段值与 payload 等价）。

### 历史命中记录
| 日期 | 文件 / 字段 | 现象 | 修复 |
|------|------------|------|------|
| 2026-05-03 | `SaveDraftRequest.LineItemDraft` 缺 `productPartNo` / `productName` / `customerPartNo` | V5 批量导入分支 productId 始终为空，partNo 只活在前端内存；保存→刷新后前端 `useDriverExpansions` 在 `if (!item.productPartNo) continue;` 处直接跳过，"投料成本"中的 物料 / 元素 / 含量 / 材料损耗 4 列永远「加载中…」最后空白 | 1) `SaveDraftRequest.LineItemDraft` 补 3 字段；2) `QuotationService.saveDraft` 把 `liDraft.productPartNo/productName/customerPartNo` 写入 snapshot 列；3) `QuotationWizard.buildDraftPayload` 把这些字段放进 payload；4) `QuotationDTO.LineItemDTO` 用 Product join → snapshot 列回退顺序回读；5) `applyQuotationData` 装载 `customerPartNo` |

## AP-3：派生数据靠运行时 lookup，但 lookup 源可能为空

### 现象
- 某个功能（BASIC_DATA 路径解析 / 公式引擎 / driver 展开 / 客户折扣 / ExcelView 等）依赖一个 ID 反查（`productId → product.partNo`、`customerId → customer.level`…）。
- 当反查源行被删或者根本没建立（典型：V5 流程产品行 `product` 表没记录），lookup 失败 → 派生值空 → 上层全坏。

### 根因
- 把 ID 当成"指针"，需要时才解引用，本身没问题——直到那张被指向的表里行不存在。

### 防护措施
1. **关键派生值落 snapshot 列**：`product_part_no_snapshot`、`product_name_snapshot`、`snapshot_customer_name`、`snapshot_customer_level` 等；持久化时 product / customer 仍存在的话直接拿真值写一份。
2. **DTO 回读时 join 优先 + snapshot 回退**：
   ```java
   if (li.productId != null) {
       Product p = Product.findById(li.productId);
       if (p != null) dto.productPartNo = p.partNo;
   }
   if (dto.productPartNo == null) dto.productPartNo = li.productPartNoSnapshot;
   ```
3. **导入流程显式写 snapshot**：任何不走 `productId` 的导入分支（V5 mat_part 直入），保存时直接把已知 partNo 写到 snapshot 列。
4. **下游消费方做空保护**：useDriverExpansions / formulaEngine 都已经在头上 `if (!partNo) skip`，至少不能崩；但要有日志提示「driver expansion skipped: partNo missing for lineItem=…」便于排查。

### 历史命中记录
| 日期 | 模块 | 现象 | 修复 |
|------|------|------|------|
| 2026-05-03 | quotation_line_item.product_id（V5 批量导入分支为 NULL） | 刷新后 `productPartNo` 拿不到 → driver 展开跳过 → 4 列空白 | LineItemDTO 走 Product join → snapshot 列回退；saveDraft 同步写 snapshot 列 |
| 2026-04-29 | quotation.snapshot_customer_*（已有的最佳实践） | — | `QuotationService.create` / `saveDraft` 创建/更新时一并把客户档主信息打快照，避免后续客户改名导致历史报价显示错乱 |

## AP-4：自动保存定时器闭包陷阱

### 现象
- `setInterval(() => autoSaveDraft(), 10000)` 每 10 秒触发一次自动保存。
- 用户改了 lineItems，但是 DB 里始终是上一版甚至空版。
- 手动点「保存草稿」可以保住，自动保存"看起来"在跑（无错误日志）。

### 根因
- `autoSaveDraft` 是 `useCallback([quotationId, form, lineItems])`，依赖变化时函数引用变。
- 但是注册定时器的 `useEffect([quotationId])` 只跑一次，捕获的是首次渲染那版 `autoSaveDraft`，其内部 `lineItems` 永远是初始 `[]`。
- 第一次 fire 把 payload JSON 化记进 `lastSaveRef`；之后每次都跟初始空 payload 一样 → 跳过 → 用户感知不到，但 DB 永远没收到真实数据。

### 防护措施
- 把"最新版回调"放进 `useRef`，effect 不依赖它：
  ```ts
  const fnRef = useRef(autoSaveDraft);
  useEffect(() => { fnRef.current = autoSaveDraft; }, [autoSaveDraft]);
  useEffect(() => { const t = setInterval(() => fnRef.current?.(), 10000); return () => clearInterval(t); }, [quotationId]);
  ```
- 任何 `setInterval / setTimeout` + `useCallback`-依赖闭包的组合都是嫌疑组合，必须用 ref 模式。

### 历史命中记录
| 日期 | 文件 | 修复 |
|------|------|------|
| 2026-04-30 | `QuotationWizard.tsx` autoSave 注册 | 改 ref 模式，setInterval 始终调最新 autoSaveDraft |

## AP-5：JSONB 数据回读时 `rowData` 是字符串而非数组，前端假定是数组直接 map → 全空

### 现象
- 报价单刷新后行结构在、但每一格输入框 / 单元格全空。
- 数据库里实际有完整 `row_data`，但前端渲染当成空。

### 根因
- 后端 DTO 把 JSONB 列当 `String` 透传（`ComponentDataDTO.rowData: String`）。
- 前端类型 `ComponentDataItem.rows: Record<string,any>[]`，从 API 来的 `componentData` 里其实只有 `rowData: string`，没有 `rows`。
- 中间层 `enrichComponentData` 在 templateSnapshot 匹配失败 / fetch 失败时返回 raw saved → `comp.rows = undefined` → `useDriverExpansions` 仍按 driver 展开数渲染 N 行，每行 `row[key]` 都 `undefined` → 全空。

### 防护措施
- `enrichComponentData` 进入处先把 `rowData` 字符串解析为 `rows`，作为最低保障；catch 分支也返回带 `rows` 的版本而不是 raw saved。
- 任何接口契约里"DB 是 JSONB、DTO 是 String、前端期望解析后的 obj"的链路，前端必须有一个集中的 normalizer 函数，禁止散落 `JSON.parse(saved.rowData)` 散点解析。

### 历史命中记录
| 日期 | 文件 | 修复 |
|------|------|------|
| 2026-04-30 | `QuotationWizard.tsx::enrichComponentData` | 进入前先 parseJson 写到 `withRows`；catch 也返回 withRows |

## AP-6：实例字段以路径作 cache key，但路径在多行间共享

### 现象
- BOM 元素行展开后 4 行，但每一行公式算出来都用同一份（第一行）的 BASIC_DATA 值。
- 单价用户可以改，但其它行的计算结果跟第一行一致。

### 根因
- `formulaEngine.evaluateExpression` 对 `path token` 用 `${partNo}::${path}` 做 cache key——同一料号同一 path 全局只缓存一份"聚合多行结果"。
- driver 展开后明明每行都拿到了不同的 `basicDataValues`（按行返回），但是公式引擎在算的时候不知道这件事，依然走 partNo 维度 cache → 拿到聚合结果 → 取首值。

### 防护措施
- `evaluateExpression` 增加 `basicDataValues?: Record<string,any>` 参数：path token 优先按 `{path}` 从行级 driver 展开值取，回退才走 partNo 维度 cache。
- `computeAllFormulas` 同步透传 basicDataValues。
- 所有渲染层调用点必须把"当前行的 basicDataValues"传下去；不要靠全局 cache 算行级数据。

### 历史命中记录
| 日期 | 文件 | 修复 |
|------|------|------|
| 2026-04-30 | `formulaEngine.ts::evaluateExpression`、`QuotationStep2.tsx::computeAllFormulas` | 加入 basicDataValues 参数 + 调用点透传 |

## AP-7：列字段冲突 — 隐式 JOIN 把审计列也注进来

### 现象
- 公式 / 路径解析出报错：`操作符不存在: timestamp with time zone = character varying`。
- 看 SQL 发现 WHERE 子句里被注入了 `created_at = '2026-04-29 14:17:36...'` 这种条件。

### 根因
- `ImplicitJoinRewriter` 把 driver row 里所有跟目标表同名的列都拼成 `AND col = literal`。
- 审计列 `id / created_at / updated_at / created_by / updated_by / version` 等跟业务语义无关、却跟目标表里同名，于是被错误注入。
- 副作用一：类型不匹配（timestamptz 列 vs 字符串字面量）→ 5xx；副作用二：把 `mat_bom[bom_type='ELEMENT']` 这类多行查询收窄成单行（按 id 匹配只剩驱动行那一行）。

### 防护措施
- `ImplicitJoinRewriter` 永久黑名单：`id / created_at / updated_at / created_by / updated_by / version / deleted_at / is_deleted` 不参与隐式 JOIN。
- 凡是对 driver row 做隐式列拓展的地方，都要白名单（业务键）而不是黑名单（系统键）；遇到非业务字段一律绕开。

### 历史命中记录
| 日期 | 文件 | 修复 |
|------|------|------|
| 2026-04-30 | `ImplicitJoinRewriter.java` | 加 `SYSTEM_COLUMN_DENYLIST` 跳过审计列 |

## AP-8：导入差异检测对"Excel 缺列"误报为差异

### 现象
- 同一份 Excel 反复导入「基础数据预览&差异分析」，每次都给出**完全相同**的差异列表要求确认；用户感受是"没法收敛"。

### 根因
- `BasicDataImportServiceV5.compareField` 用 `valuesEqual(db, excel)` 判等：
  - `db != null && excel == null` → 返回不相等 → 报告为差异
- 但实际写库逻辑 `writePhysicalTables` 用 `COALESCE(:excelVal, db_col)`：Excel = null 时根本不会更新 → 不存在可执行的变更 → 不应作为差异要用户决策。

### 防护措施
- `valuesEqual` 当 Excel 端归一为 null/blank 时直接视为相等（匹配 COALESCE 的写库语义）。
- "diff" 的定义必须跟 "write" 的语义一一对应，不能 diff 报得比 write 实际做的更激进。
- 任何「预览 / 对比 / dry-run」类逻辑写完后，跟实际 write 对一遍：所有"看到差异 → 实际不会写"的场景都得抹掉。

### 历史命中记录
| 日期 | 文件 | 修复 |
|------|------|------|
| 2026-04-30 | `BasicDataImportServiceV5.valuesEqual` | Excel 端为 null 时返回 true（匹配 COALESCE） |

## AP-9：异步 enrichment / 远端结果用对象式 setState 整体覆盖，吞掉并发用户输入

### 现象
- 编辑页有"先用基础数据快速渲染骨架，再异步从远端拉模板/字典/快照补齐元数据"的两段式加载。
- 用户在第一段渲染后立即开始输入，第二段补齐到达时，用户在某些卡片上的输入**整体消失**。
- 表现极不对称：先输入的卡片整张被擦，后输入的卡片完整。

### 根因
- 第二段补齐用 `setLineItems(enrichedItems)` 整张替换；`enrichedItems` 是基于第一段 `basicItems`（即 API 原始返回）算出来的，里面 `rows` 全是默认空。
- 用户在第二段返回前的输入已经写入了 React state，但被这次整张替换无视。
- React 不知道这是"刷新元数据"而不是"刷新整张数据"。

### 防护措施
- 任何"先快速渲染、再异步增强"的二阶段加载必须用 **函数式 setState + 合并语义**：
  ```ts
  setLineItems(prev => prev.map((cur, i) => mergeMetaPreservingUserEdits(cur, enrichedItems[i])));
  ```
- 合并策略明确区分"元数据字段"（fields / formulas / componentId / tabName / 快照配置）与"用户数据字段"（rows / values / inputs）。前者拿增强结果，后者保留 `prev` 里的当前态；只有"用户没动过的初始空单元"才允许被默认值替换。
- 同样适用于其他增强类异步：BASIC_DATA path 拉取后写回行级缓存、driver 展开补 `basicDataValues`、快照刷新 etc. **只要在用户可输入的窗口期内**返回结果，都必须用函数式合并。

### 历史命中记录
| 日期 | 文件 | 现象 | 修复 |
|------|------|------|------|
| 2026-05-03 | `QuotationWizard.tsx::loadQuotation` 的 enrichComponentData Promise.then | QT-20260503-1281：两条产品共享同一模板，用户先在产品 2 各 BOM 行输入单价 → enrichment 完成 → `setLineItems(enrichedItems)` 整张盖回去 → 产品 2 输入清空（payload 只剩 1 行），产品 1 后输入因此保住（payload 完整 4 行） | 改成 `setLineItems(prev => prev.map(...))` + 合并 `componentData` 时按行检查 `hasUserInput`，保留用户已输入行，只替换"未动过的默认空行"承载的元数据 |

## AP-10：mutator 函数用对象式 onUpdate 取闭包旧值，与异步事件竞争丢字段

### 现象
- `handleRowChange / handleAddRow / handleDeleteRow` 等 row mutator 用 `onUpdate({componentData: item.componentData.map(...)})` 形式。
- 异步事件（autoSave 触发、DATA_SOURCE auto-query 回填、driver 展开完成、enrichment 合并）会触发 setLineItems。
- 在异步事件提交 state 与用户下一次按键之间，handleRowChange 闭包里的 `item.componentData` 仍是旧版本 → 它把旧 baseline 加上自己这一笔写回去 → 上一次的更改被吞。

### 根因
- 对象式 setState (`setX(newX)`) 不再读最新 state，必然和并发的异步 setState 竞争。
- React 的 props 在每次 render 后才更新，`item` 闭包到 onChange 触发时不一定是最新。
- 某些异步事件（autoSave、debounced query）发生频率比 React render 还快时，stale closure 不可避免。

### 防护措施
- 任何会改 lineItem / row / cell 的 mutator 必须使用函数式 `onUpdate(prev => newPatch(prev))`，避免读闭包里的 `item`。
- 同模块里 patchRowField 已经是这种写法，**所有同位 mutator 都要保持一致**——不要 mix。
- code review 启发式：搜 `onUpdate(\{` / `setLineItems(\[` / `setX({` 这种"object-form setState"，凡是同一模块里有 patchXxx 用 functional 形式的，那同位 mutator 也必须是 functional。

### 历史命中记录
| 日期 | 文件 | 现象 | 修复 |
|------|------|------|------|
| 2026-05-03 | `QuotationStep2.tsx::handleRowChange/handleAddRow/handleDeleteRow/handleAttrChange` 全部用对象式 onUpdate | 与 AP-9 同时生效：用户多行输入被异步事件清算时丢部分；保险起见全部转成函数式 | 4 个 mutator 改为 `onUpdate(prev => ...)` 形式；与同模块已有 patchRowField 的写法对齐 |

## AP-11：屏幕展示用「运行时计算」+ 保存只 dump 输入字段 → WYSIWYG 不一致 + 历史快照丢失

### 现象
- 报价单卡片上看到的 BASIC_DATA（物料 / 元素 / 含量 / 材料损耗）和 FORMULA（金额）数值都正常，但保存到 DB 的 `row_data` JSON 里这些字段是 **null**（仅 INPUT 类字段被写入）。
- 当时刷新页面看不出问题——driver 展开 / 公式引擎都会重新跑一遍把数据贴回屏幕。
- 真正的炸弹：**报价单一旦提交、几个月后被审计 / 被复制**，此时 mat_bom 元素行可能改了 / 删了，原有报价显示数据就读不出来或读错。
- 用户看到「实际数据 vs JSON」对比时会以为这是 Bug。

### 根因
- 渲染层直接从 `basicDataValues`（driver 展开返回，非持久化）和 `computeAllFormulas`（前端运行时算）取值贴屏。
- `row` state 只承载用户敲进来的 INPUT 字段；BASIC_DATA / FORMULA 类字段从来没写进去。
- `buildDraftPayload` 把 `cd.rows` 直接 `JSON.stringify` 出去，自然只能落 INPUT。
- 设计上想着"反正用得到时再 lookup"，忽略了**数据源被改 / 被删后历史就读不出**这一致命副作用。

### 防护措施
- **保存前快照 BASIC_DATA / FORMULA 到 rowData**：`buildDraftPayload` 在序列化每个 row 之前，按 component.fields 走一遍：
  - BASIC_DATA：从 driverExpansions 行级 basicDataValues 中按 `bnfDriverLookupKey(path)` 取值，写到 `row[fieldKey]`
  - FORMULA：跑 `computeAllFormulas` 拿到结果 map，写回 `row[fieldKey]`
  - INPUT / FIXED_VALUE / DATA_SOURCE：保持原 row 值
- 所有"显示=屏幕看到的"的字段都必须能被序列化保存，**即便它们在前端是衍生的**。"显示靠 lookup, 存只存输入"是隐性反模式。
- 如果有"快照 vs 即时" 的需求二选一（DRAFT 阶段让 BASIC_DATA 变化能反映 vs 提交后冻结），用提交时快照 + DRAFT 也快照（DRAFT 快照随每次保存覆盖即可）的方案；不要做"DRAFT 不快照"，因为 DRAFT 也会被刷新、被另一个用户接手、被另一个浏览器加载。

## AP-12：懒资源（lazy / optional sub-resource）GET 时硬抛 404，前端整页崩

### 现象
- 切换某个视图 / Tab 时整页报错 / 弹红 toast / 控制台 `404 Not Found`。
- URL 形如 `/api/cpq/quotations/{id}/<sub-resource>`，问题报价单 ID 本身是合法的。
- 重要特征：**问题不在路由缺失（endpoint 存在）**，而在后端 `service.getXxx(parentId)` 在子资源行不存在时硬抛 `BusinessException(404, "...not found...")`。
- 同一份 service 里其他方法（例如 `buildComparison`）反而做了"未找到 → 返回空 DTO"的兜底；存在自我矛盾。

### 根因
- 子资源是"按需创建 / 懒生成"的（核价表、Excel 视图 snapshot、提交快照、衍生比对…）。
- DRAFT 报价单刚建好、还没走过任何"初始化"动作，自然就没有这条子表行。
- 后端在 GET 时简单粗暴地"找不到就 404"，没有区分"主资源不存在（应当 404）"和"主资源 OK + 子资源还没生成（应当返回空骨架）"两种语义。
- 前端 `axios.get → catch` 通常把 4xx 当错误处理，弹 toast / 整页降级，体验断崖。

### 防护措施
- 区分两类语义：
  - **路径里 ID 找不到** → 真 404（主资源不存在）
  - **主资源 OK，但派生 / 关联子表还未建立** → 200 + 空骨架 DTO；让前端进入空态/初始化分支，而不是错误分支
- 检验模板：`grep -rn "BusinessException(404" cpq-backend/src/main/java/com/cpq/`，逐条问：
  - 这个对象**会不会**对一个合法主资源懒生成？
  - 前端会在主资源页面正常切换流程中**正常路径**触达这个 endpoint 吗？
  - 同 service 里其他方法已经做了空 DTO 兜底吗？（一致性）
- 命中即改，且要顺手统一同 service 内其他兜底。
- 前端 service 也应该对 404 做"等价空数据"兜底（双保险），避免后端某天又新增一处硬 404。

## AP-13：客户级版本表的"业务键 vs schema 唯一约束 vs 路径解析过滤"三方不一致

### 现象
- 报价单某个 tab（典型：施耐德其他费用）只显示 1 行甚至空白，但 Excel 主档明明导入了 4 行不同 dim_element_name 的内容（财务管理费 / 回收费 / 材料管理费 / 包装费…）。
- 看 mat_fee 表里同 part 同 fee_type 下有大量历史版本（149 / 87…）但只有 1 条 is_current=true。
- 重导一遍 Excel 仍然只剩最后写入的那 1 个 seq_no 是 current。

### 根因（4 处错位）
1. **schema 唯一约束太窄**：`uq_mat_fee_current = (customer_id, hf_part_no, fee_type) WHERE is_current=true`。同 part 同 fee_type 下只允许 1 条 current —— 多 seq_no 的业务场景被压扁。
2. **VersionedWriter 业务键没跟 schema 对齐**：`TableMeta("mat_fee")` 只有 `[fee_type]`，`seq_no` 进了 dataColumns。导入循环处理 4 个 seq_no 时互相覆盖：每写一个新的 seq_no 都把上一个标 `is_current=false`，自己变成唯一 current。
3. **组件 driver_path 缺失**：使用 mat_fee 路径的组件（如 COMP-0011 其他费用）没设 `data_driver_path`，前端 `useDriverExpansions` 直接 skip，整个 tab 只渲染单行。
4. **PathToSqlGenerator 不过滤 is_current**：客户级版本表（mat_fee / mat_process / plating_fee）做路径解析时不会自动加 `is_current=true`，会把所有历史版本都拉回来——即便 driver_path 配上，结果也是 N 个版本而不是 N 个业务行。

### 防护措施（永久消除）
- **三方对齐律**：客户级版本表的"schema 唯一约束键 = VersionedWriter 业务键 = 业务上区分行的字段集"必须三方一致。
  - 加新 fee_type 或新 dim_* 列时，三处都要同步更新；schema 改了但 VersionedWriter 没改 → 数据被覆盖；VersionedWriter 改了但 schema 没改 → INSERT 触发 unique violation。
- **路径解析对版本表必须默认过滤 is_current**：在 PathToSqlGenerator 维护 `VERSIONED_TABLES` 集合，遇到这些表自动在 WHERE 末尾追加 `is_current = true`；这一层是即便上层配置出错的最后防线。
- **组件配置审计**：BASIC_DATA 字段路径若指向多行返回的表，组件必须配 `data_driver_path` 一起指向同表的"行集合"过滤；否则前端只能贴第一行的值。Code Review 时按"组件 fields 引用了哪些表 → 该表是否多行 → 是否需要 driver_path"清单过一遍。
- **JDBC 绑定 UUID 字符串前先转 java.util.UUID 类型**：ImplicitJoinRewriter 注入的 customer_id 等 UUID 列条件是字符串字面量；DataLoader 在 `ps.setObject` 前要识别 UUID 模式并显式转 `UUID.fromString(s)`，否则 PG 报 `uuid = character varying` 找不到运算符。

|------|------|------|
| 2026-05-04 | mat_fee 全表 + COMP-0011 + PathToSqlGenerator + DataLoader UUID 绑定 + ImplicitJoinRewriter | (1) Flyway V69：重建 `uq_mat_fee_current = (customer_id, hf_part_no, fee_type, seq_no) WHERE is_current=true`；按 `(cust, part, fee_type, seq_no)` 找最大 version 重置 is_current；设 COMP-0011 driver_path = `mat_fee[fee_type='FINISHED_OTHER']`。(2) `VersionedWriter::TableMeta("mat_fee")` 业务键加 `seq_no`、dataColumns 去掉 `seq_no`。(3) `BasicDataImportServiceV5` mat_fee 写库时 `bk.put("seq_no", r.seqNo)`。(4) `PathToSqlGenerator` 引入 `VERSIONED_TABLES = {mat_fee, mat_process, plating_fee}`，自动注入 `is_current=true`。(5) `DataLoader.executeQuery` 将 UUID 形态字符串识别后转 `UUID.fromString` 再 `setObject` |

## AP-14：组件 data_driver_path 与 fields.basic_data_path 谓词不一致 + ImplicitJoinRewriter 自表注入坍缩

### 现象
- 组件 fields 用 `IN` 谓词扩多类型（典型 `mat_fee[fee_type IN ('FINISHED_OTHER','INCOMING_OTHER')].xxx`），但页面里只看到单类型行；或者各行内容彼此污染（应该是 INCOMING 的行显示 FINISHED 的值）。
- 后端 SQL 日志里出现一长串审计/数据列被注入成 AND 条件：
  ```
  mat_fee[fee_type IN (...) AND fee_ratio = 0.0500 AND dim_element_name = 'X'
          AND seq_no = 2 AND import_record_id = '...' AND imported_by = '...'
          AND status = 'ACTIVE' AND price_unit = 'KG' AND currency = '?']
  ```

### 根因（双层）

**层一：组件配置不一致**
- `component.data_driver_path`（决定 driver 展开的行数）和 `fields[*].basic_data_path`（每个字段的查询路径）是两份独立配置。
- 用户改了字段路径用 `IN` 多类型，**忘了同步把 driver_path 也改成 `IN`**。
- 结果：driver 展开还按单类型只返 N 行，但字段路径以为有更多类型可选；UI 渲染少行 + 数据混乱。

**层二：ImplicitJoinRewriter 在自表场景把整行复制成 WHERE**
- ImplicitJoinRewriter 设计意图是"driver 行的列同时存在于目标表 → 当 join 键注入"。这对 `mat_part → mat_bom` 这种**跨表**场景是正确的。
- 但当 `data_driver_path` 与 `basic_data_path` 查的是**同一张表**（典型 self-join：`mat_fee[...] → mat_fee[...].field`），driver 行就是 mat_fee 的一行；它的所有列都在目标表里，全部被注入成 AND，结果 WHERE 退化成"自己等于自己"。
- 副作用：
  1. 数据列（fee_value / fee_ratio / currency / price_unit / dim_input_material_name）被当 join 键，浮点等值匹配脆弱。
  2. 状态列（status / is_current）和导入溯源列（import_record_id / imported_by）也被混进，黑名单原本没盖。
  3. `IN` 多类型的扩展被压扁——每个 driver 行都被收窄成自己那 1 行，多类型 IN 等价于单类型。

### 防护措施
1. **配置一致性**：
   - 修改 `fields[*].basic_data_path` 时，**同步检查/更新 `component.data_driver_path`**——两者主表和谓词应该对齐。
   - 长远建议：组件保存校验时如果 fields 里出现谓词，driver_path 缺失或不一致 → warning 或自动同步。
2. **黑名单覆盖系统列**（已合入）：`SYSTEM_COLUMN_DENYLIST` 加入 `import_record_id / imported_by / status / is_current`。
3. **自表短路**（已合入）：`ComponentDriverService.evaluatePath` 在 driverRow 已经包含目标 leaf 字段时，直接读 driverRow，不再下发 SQL。绕开整层 ImplicitJoinRewriter 的副作用，且性能更好。
   - 边界：跨表场景（driverRow 来自 mat_part、target 是 mat_bom 之类）不会命中此短路，原 ImplicitJoinRewriter 路径仍然生效。

| 2026-05-04 | QT-20260504-1295 / COMP-0011（施耐德其他费用），fields 改成 IN ('FINISHED_OTHER','INCOMING_OTHER') 但 data_driver_path 还是 = 'FINISHED_OTHER'；扩展只渲染 4 行，row[0] 的 dim_input_material_name 还显示了 INCOMING 那一行的值 | (1) `UPDATE component SET data_driver_path = 'mat_fee[fee_type IN (...)]'` 同步谓词；(2) `ImplicitJoinRewriter.SYSTEM_COLUMN_DENYLIST` +4 列；(3) `ComponentDriverService.evaluatePath` 增加 driverRow 自表短路 + 新增 `extractLeafField()` 解析末段字段名。验证后 expand-driver 返 6 行（4 FINISHED + 2 INCOMING），各行字段独立、不串扰 |

## AP-15：客户级版本表的"业务键 / schema unique / 冲突 rowKey 三方"还需扩到 dim_* 维度（AP-13 续）

### 现象
- 用户在「客户数据冲突确认 (UI-1)」上点击"全部采纳新值"并确认导入，**下一次导入同一份 Excel 仍然出现完全相同的冲突**。
- 多次重复 → 冲突永远不收敛，DB 里同一个 (part, fee_type, seq_no) 下的 `is_current=true` 行只剩一条（最后写入的）。

### 根因
- AP-13 把 mat_fee 的 schema unique / VersionedWriter bk 从 `[fee_type]` 扩到了 `[fee_type, seq_no]`；但 Excel 业务上同一 `(fee_type, seq_no)` 下还允许多行，靠 `dim_element_name / dim_input_material_no / dim_input_material_name / dim_assembly_process / dim_sub_seq_no` 区分。
  典型："来料其他费用" sheet 在 H85 这一段下，`seq_no=2` 同时挂着「包装费 / 材料管理费 / 回收费」三个不同 `dim_element_name` 的费用项。
- 后端日志直接暴露：
  ```
  versioned mat_fee bk={fee_type=INCOMING_OTHER, seq_no=2} v167→v168, 6 log entries  ← 回收费
  versioned mat_fee bk={fee_type=INCOMING_OTHER, seq_no=2} v168→v169, 3 log entries  ← 材料管理费
  versioned mat_fee bk={fee_type=INCOMING_OTHER, seq_no=2} v169→v170, 6 log entries  ← 包装费
  ```
  同一次导入里 3 个 Excel 行连续覆盖同一个 bk，最后只剩"包装费"是 current。
- 检测客户冲突时 dbMap 的 key 也只用 `(part, fee_type, seq_no)`，三行 dim_* 行被压成最后一行；每次再导入：
  1. 第 1 行 Excel（回收费 0.05）vs DB 当前包装费 → 报"原值=空 → 0.05"
  2. 用户接受新值，写入回收费成为 current
  3. 第 2 行 Excel（材料管理费 0.02）vs DB 当前回收费 → 又报新冲突
  4. 写入材料管理费，回收费再被冲掉
  5. 第 3 行（包装费 0.4）→ 同样回弹
  6. 三轮过后 DB 只剩包装费，下次再来又是同样三个冲突，**永远收敛不到稳态**。

### 防护措施
- "schema unique = VersionedWriter bk = 冲突 rowKey = writePhysicalTables 用的 rowKey" 必须**四方都对齐到 Excel 真正能区分行的最细粒度**。
- 客户级版本表（mat_fee / mat_process / plating_fee）的设计要素一旦增加新的 dim_* 列，4 处都要同步加进来；否则要么 INSERT unique violation，要么覆盖丢失。
- 任何"versioned table"的 PR 在 review 时必须把这 4 个对齐点写在 PR 描述里勾选确认。

### 现象
- 用户在 A 菜单（如「模板配置」）创建了某种"X 模板"，确认已保存、已发布

4. 实际是接口指向了表 B（结构相似但完全独立），即使表 A 全发布也匹配不到

- 增加新菜单 / 新字段时，先确认底层表是否已有同义体系；若有 → 合并到一张表 + discriminator 字段（如 V71 给 `template` 表加 `template_kind`）

- 定 Reference 文档：每个领域名词背后到底是哪张表，避免新人/新 Agent 重蹈

## AP-18：dev mode 的 hot-reload ≠ Flyway 重跑 — `touch` 不一定让新 V_xx.sql 落库

### 现象
- 写好新 migration `V_NN__xxx.sql`，按 CLAUDE.md 跑 `touch <some>.java` 触发 Quarkus dev 重启
- 等 5–10 秒后查 `flyway_schema_history`，新 V_NN 不在表里 — 但旧 V_(NN-1) 还在
- 后端接口 / 视图查询 200，仿佛"也没出错"，但新 migration 的 schema 变更（视图 / 列 / 数据）确实没有被应用
- 进一步查 Quarkus 日志，看不到 Flyway 的"Migrating to version NN"日志行

### 根因
- Quarkus dev mode 监听的是 **java 源文件 / classpath 资源的 mtime + 内容 hash**，决定要不要走"完整重启"
- 单纯 `touch <file>.java` 只更新 mtime，**没改内容**，hash 不变 → dev mode 走的是"轻量 hot-reload"（只重新加载已变化的 bean / endpoint），不会触发 Flyway 重新扫描 migration 目录
- Flyway 仅在**完整应用启动**时执行（`quarkus.flyway.migrate-at-start=true`）
- 表面现象具有迷惑性：`/api/...` 端点 200、表查询 200 → 让人误以为新 migration 已生效；只有具体查"新视图列 / 新行"时才会暴露

### 重现路径
1. 写 V_NN.sql，包含 CREATE VIEW + INSERT 数据
2. `touch cpq-backend/src/main/java/.../SomeEntity.java`（不改内容）
3. 等 8 秒
4. 查 `select version from flyway_schema_history where version='NN'` → 0 行
5. 查具体接口数据 → 仍是旧数据 / 旧视图

### 防护措施
- **修改 java 文件的实际内容**（哪怕只加一行注释）→ Quarkus 检测到代码变化 → 走完整重启 → Flyway 重新扫描
- 或在 SQL 子目录 `db/migration/` 下加 / 删一个文件让 Quarkus 检测到 resources 变化（不可靠，依赖 Quarkus 版本）
- migration 写完后**强制按以下顺序自检**：
  1. 修改 entity 注释一行内容（不只是 mtime）
  2. 等 10–12 秒
  3. `psql -c "SELECT version, success FROM flyway_schema_history WHERE version='NN'"` 必须 success=t
  4. 查实际数据 / 视图字段确认落地
- 不要靠看 health 端点判断 — 旧 migration 已 committed，health 永远是 OK，掩盖新 migration 失败/未跑

## AP-20：BNF 隐式 JOIN 失效场景 — 目标视图缺 `hf_part_no` 列 / 多 metric 行未 PIVOT

### 现象
- Excel 视图配了一列 `costing_summary_result.value`，期望按当前料号取该 metric
- 实际渲染：所有产品行该列都是同一个值 / 全为空 / 全部"加载中…"不结束
- 切换不同料号的报价单结果一样 — 看起来"按料号过滤"完全没生效

- ImplicitJoinRewriter 自动注入 `hf_part_no` / `customer_id` 的前提：**目标物理表/视图本身有这列**

- 同时 `costing_summary_result` 是 long 表（每 metric 一行），单值取首条等价于"取任意一行的 value" — 渲染表面看起来"有值但全错"
- 视图 `v_costing_summary_full` 通过 PIVOT + 显式带出 hf_part_no 解决两个问题

### 重现路径
1. 配 Excel 列 `variable_path = costing_summary_result.value`
2. 所有产品行该列要么显示同一 metric 的同一个料号的值，要么 NULL
3. `\d costing_summary_result` 看到没 `hf_part_no` 列
4. 切到 `v_costing_summary_full.material_cost` → 立即按料号正确显示

### 防护措施
- **设计 BNF 取数源时优先做"宽表 / PIVOT 视图"**：每料号 1 行，业务字段横向铺开
  - 视图 SELECT 列表里**显式带出** `hf_part_no` / `customer_id` 等隐式 JOIN 关键字段
- 当原始数据是 long 表（key-value / discriminator），就用 `MAX(CASE WHEN ... END)` PIVOT 包一层视图
- BNF 路径配置时如果目标表没有 `hf_part_no` 列，UI 应**警告**"此路径无法按料号过滤，所有产品行将看到同一份数据"
- 隐式 JOIN 的"自动注入"日志（`ImplicitJoinRewriter.LOG.debugf`）需打开 / 检查 — 确认到底注入了哪些谓词
- 注册 basic_data_config sheet 时如果 target_table 不带任何隐式 JOIN 关键字段，标注"GLOBAL"避免误用

## AP-21：FORMULA 列写字符串字面量 / Excel 函数 — 安全闸吃掉直接显示 "—"

### 现象
- Excel 模板里某 FORMULA 列填了 `="KG"` 想显示固定文本，运行时永远显示 `—`
- 或者填了 `=SUM([A]:[T])` / `=IF([A]>0, [A], 0)` / `=VLOOKUP(...)`，同样永远 `—`
- 没有报错日志，浏览器 DevTools 也看不到异常 — 让人以为是数据问题

### 根因
- 前端 `LinkedExcelView.evaluateFormula` 出于**安全考虑**只允许字符 `[\d+\-*/().,\s%<>=!&|?:]`
- 任何字母（A-Z / a-z）、双引号、单引号 → 全部被安全闸拦下 → 返回 `'—'`
- `Function('"use strict"; return ' + expr)` 不能解析"KG" 也是字符串字面量需要引号转义 — 但安全闸已先拦
- Excel 函数 SUM / IF / VLOOKUP 全是字母，全被吃掉

### 重现路径
1. Excel 模板配 FORMULA 列：`formula = '="KG"'`
2. 渲染该列 → 永远 `—`
3. 改成 `=1+1` → 显示 `2`，证明引擎其他都正常

### 防护措施
- **要显示固定文本** → 改用 VARIABLE 列指向视图里的硬编码字段（视图 SELECT 里写 `'KG'::varchar AS weight_unit`，列 BNF 路径 = `v_xxx.weight_unit`）
- **要做条件判断** → 用三元 `[A]>1000 ? [A]*0.9 : [A]`（仅运算符 + 数字 + 列引用，不带函数名）
- **要做 Excel 函数级聚合** → 把逻辑下沉到后端视图（PIVOT / WINDOW / 聚合）；前端只负责取数
- 公式编辑抽屉应在保存前做 dry-run：用 mock 数 0/1/100 跑一遍 evaluateFormula，命中 `—` 给出明确警告"包含非法字符 / 函数"
- 公式语法文档在编辑抽屉就近显示（用户不必跳到 docs 才能找到）

## AP-22：多行数据被压缩成 "X (共N项)" / "{字段:null} (共N项)" 显示

### 现象

| 类别 | 根因 | 触发场景 |

| **A. SQL 层多行** | 视图本身按 partNo 维度有 N 行（如 v_costing_summary_full 4 个 partNo × 1 行/partNo），ImplicitJoinRewriter **没注入 hf_part_no 谓词** → 全表返 N 行 → list 被 formatPathValue 兜底成 "首值（共N项）" | 视图 DROP CASCADE 期间 ImplicitJoinRewriter.tableColumnsCache 缓存了空集 → 永久残留（V112 修） |

| **D. 比对标签未注册** | Excel 模板列打了 comparison_tag code 但 comparison_tag 表未注册该 code → tagLabelMap.getOrDefault(tag, tag) 返 raw code, tagToGroup 全归"其他"组 | V96 用 13 个 code 但表里没注册（V114 补） |

### 重现路径（典型 case：B 类）
1. 报价单 QT-XXX line item 关联多行 driver 组件（如 RAW-BOM driver=`costing_part_material_bom`）
2. 用户保存草稿 → snapshotRows 把 driver 展开的 3 行各自标量值写入 `row_data` JSONB
3. 用户重新打开报价单
4. `useDriverExpansions` hook 还没回完 expand-driver API（或 lineItem 状态导致 hook 没启动）

## AP-23：导入向导「上传即写库」— 事务边界紧贴解析，用户无后悔药

### 现象
- 用户上传 Excel 后立即看到 mat_* 正式表新增数据 / 出现升版日志
- 用户在「创建报价单」中途取消（关闭 Drawer / Esc / 浏览器关闭 tab），mat_* 已经被污染

- 同一份 Excel 重复上传两次会导致 mat_bom 等表出现重复行（COALESCE 空字段匹配漏失）
- 「全部升版/全部不升版」二选一 Modal → 用户没法精细决策 → 不想升版的料号也被升 / 想升版的料号被覆盖

- 决策 Map 入参只能携带 `Map<String,String>` 粗粒度 BUMP/NO_BUMP，前端 Modal 没空间放每料号的 toggle

1. 用户上传 Excel，UI2 显示 3 个料号有差异
2. 用户在 Modal 里点「全部升版」→ 后端 `POST /confirm` 提交事务 → mat_bom/process/fee 加入新版本行 + mapping.current_version=2001 + log 写一条

4. mat_* 已有新版本数据，新版本日志已写
5. 用户重新上传同一份 Excel → 检测出"新版本"是旧版本 → 又升一次 → 累积污染

2. **session 生命周期管理**：
   - 上传得 `sessionId`，每个决策更新 staging 表
   - 取消 → `DELETE FROM import_session WHERE id=?` CASCADE 清干净
   - 24h TTL + `@Scheduled` 清理 PENDING 过期 session
3. **决策表与暂存表分离**：用户每次切换 toggle/conflict/orphan 只更新 `import_session_decision` 表，不重写 staging（staging 上传一次即定）
4. **「全部 X」批量按钮 ≠ 唯一选择**：UI 上必须支持每行独立 toggle，批量按钮只是快捷方式
5. **物理表写入入口收敛**：任何写 mat_*（或同族正式表）的代码路径必须从「commit」端点进入；preview/upload 端点禁止写正式表

## AP-24：差异检测器（DiffDetector）四类不一致同时误判 — fingerprint 与字段级对比标准不齐

### 现象
- 用户未改 Excel 重新导入，「版本确认」列表仍显示全部 88 个料号待确认（但卡片显示「无 sheet 差异」`BOM(0)/工艺(0)/费用(0)/镀层费用(0)`）
- 用户改了一处 fee_value，整个料号反而**没**触发版本差异确认
- 同一料号 ELEMENT seq=1 显示「net_qty 0.03 → 0.30855」+ 「net_qty 0.0300000000 → 0.03」两条不应共存的诡异 diff
- commit 后报价单产品列表为空（用户改决策后）

### 根因（4 个独立缺陷叠加在同一个判定流程里）
| # | 缺陷 | 后果 |
|---|---|---|
| A | `computeBomDiff` / `computeRowLevelDiff` HashMap key 只用 `bom_type:seq_no`，**漏 element_name + input_material_no** | 同 seq 多元素（AgNi10 + CuZn36 都在 seq=1）后插入的覆盖前者 → Excel 行错配 DB 行 → 假差异 |
| B | `addIfChanged` 用 `Objects.equals(dbVal, excelVal.toPlainString())` **文本比较** | V178 精度扩展后 DB 存 `0.0300000000`、Excel `0.03` → 文本不等 → 全字段假差异 |
| C | staging vs mat fingerprint 用 ALL 列拼 `concat_ws`，但 `mat_bom` 比 `mat_bom_staging` 多一个 `child_part_no` 列 | 列数不同 → md5 永不等 → **永远 BUMP** 误报 |
| D | fingerprint WHERE `customer_product_no = :cpn` | `mat_fee.customer_product_no` 全是 NULL（V154 新列未回填）→ 两侧 0 行 → 都 "EMPTY" → **永远 NO_BUMP** 误压 |

C 和 D 互为对偶：C 把所有不应升版的判成 BUMP，D 把所有应该升版的判成 NO_BUMP。两个 bug 同时存在，前端就看到「列表里全是 BUMP，但每个都无业务差异；改了值反而不升版」。

### 防护措施（编写者必须做）
1. **DB 唯一索引 = 字段级 diff key = fingerprint 行匹配 key** 三方一致。任意一处偏离即 AP-13/24 同源。
   - 检查 `\d+ <table>` 看真实 UNIQUE INDEX
   - 代码里的 HashMap key 必须完全覆盖该 UNIQUE 的所有列
2. **字段值比较一律 BigDecimal.compareTo()**，不要 `toPlainString() + Objects.equals`：
   - V178 类精度扩展后 DB 列默认 trailing zeros，Excel 写入随机精度
   - 文本比较等于把"语义一致"判成"不一致"
3. **fingerprint 跨表（staging vs mat）必须用列集合交集**，且：
   - 排除 METADATA_COLS (`id/created_at/imported_by/...`)
   - 主键/外键列在两表都存在但有 NULL 值时，filter 要 NULL-tolerant（`col = :v OR col IS NULL`）
4. **fingerprint 与字段级 diff 标准不一致时 — 用字段级判定作权威**：
   - fingerprint 仅用作日志参考 / 性能短路
   - 最终 action 由 sheetDiffs + rowLevelDiff 决定
   - 这样 fingerprint 误报/漏报都不影响业务结论
5. **commit 阶段的 "数据范围"（hfPairs/候选集）应该从 staging 表独立查 distinct (cpn, hf)**：
   - 不要从 "用户决策" 推 — 决策只控"动作"（升版/不升版），不应控"作用域"
   - 决策与数据范围解耦后，前端过滤 NO_BUMP 显示不影响 commit 完整性

### 排查决策树（5 秒定位）
```
看到「列表全是料号但每个 sheet 差异都 0」
   ↓
fingerprint 报 BUMP 但 sheetDiffs 全 0 → 类别 C/D: fingerprint 误报
   - 用 \d+ 查两表列数差 → 找单边列
   - 用 SELECT count(*) 直查 cpn 过滤后行数 → 0 行说明 NULL 值被排掉
   ↓
看到「同 (seq, bom_type) 多条不应共存的 diff」 → 类别 A: HashMap key 漏字段
   - 检查 DB 唯一索引列 vs 代码 compound key 是否吻合
   ↓
看到「0.0300000000 → 0.03 这种数值等但文本异的 diff」 → 类别 B: 文本比较
   - 改 BigDecimal.compareTo

看到「改了值反而不升版」→ 类别 D: filter 排掉 NULL 行
   - 把 `col = :v` 改为 `(col = :v OR col IS NULL)`
```

## AP-29：产品级组件 (无 dataDriverPath) 走 globalPathCache 兜底, 但路径有 cache miss 永远加载中

### 现象
- 选配产品/报价单的「材质」「单重」等**产品级**组件 (1 行展示, 无按行展开) 的 BASIC_DATA 字段 5 列**全部显示「加载中…」**
- 强刷浏览器 / Quarkus 重启 / 数据库 view 验证全部正常都无解
- console 日志显示 `[path-formula-cache]` batchEvaluate 收到响应但 `result: null` 或 cache key 与渲染层不匹配
- driver-bound 组件 (工序 / 元素含量 — 有 dataDriverPath) 显示正常

### 根因
- `ComponentDriverService.expand` 对没有 `dataDriverPath` 的组件**直接返 `rowCount=0` + 空 rows**, 完全跳过 BASIC_DATA 路径解析
- 这种"产品级"组件的字段值要靠前端 `usePathFormulaCache` → `batchEvaluate` → `globalPathCache` 路径取数
- 但 globalPathCache 路径在多 lineItem / 多视图叠加场景下存在 cache key 不一致问题 (报价单 + 核价单同时渲染时 task count 与 cache size 对不上)
- 前端 BASIC_DATA cell 渲染**先**检查 `basicDataValues` (driver 路径), **再**走 `row[key]` (snapshot 写入), **最后**走 globalPathCache, 任何一级失败就「加载中」

### 重现路径
1. 组件 `dataDriverPath = null/empty`, 有 N 个 BASIC_DATA 字段
2. 前端调 `POST /components/{id}/batch-expand` → 后端 `expand` 看 dataDriverPath 空 → 返 `rowCount=0, rows=[]`
3. 前端 `activeDriverExpansion = {rowCount:0, rows:[]}` (非 undefined 但 rowCount=0)
4. `useDriver = false` → `effectiveRows[i].basicDataValues = undefined`
5. BASIC_DATA cell 跳过 driver 分支, 走 row → 行值是 null → 走 globalPathCache
6. globalPathCache miss (路径数 ≠ cache key 数, 可能 fingerprint useMemo 没把这些 path 计入) → 永远显示加载中

### 防护措施
1. **后端兜底 — 虚拟单行 driver**: `ComponentDriverService.expand` 对无 dataDriverPath 组件**也按"产品级单行"展开**, 用 `driverRow = {hf_part_no: partNo, customer_id: customerId}` 虚拟构造, 所有 BASIC_DATA paths + GVAR tasks 逐个求值塞 `basicDataValues`. 前端走 driver 优先级直接拿到值, **绕开 globalPathCache 不稳定路径**.
2. **设计原则**: 任何 BASIC_DATA 字段的取值都应该走**统一的 driver expand 路径**, 不再依赖前端独立的 path cache 机制. globalPathCache 仅作 ExcelView 列等场景的非组件路径兜底.
3. **PR 评审强制项**: 新建无 dataDriverPath 的组件时, 必须验证 BASIC_DATA 字段在报价单实际能渲染出值 (不能信任前端 path cache 路径).

### 4 个独立的失败模式（都会触发"加载中"）

| # | 失败模式 | 修复 |
|---|---|---|
| 1 | **invalidate 漏调** — 后端 mat_process / mat_bom 写完, cache 还停留在"写之前"的 0 行 | 任何调 `ConfigureProductService` / `BasicDataImportService` 之类的写入后, 必须 `invalidateDriverExpansions(partNos)` |

## AP-37："加载中…" / 页签隐藏 续集 — 新字段类型协议传播 + 同 componentId 多实例 cache 冲突

> **AP-31 的延伸**。AP-31 讲"加载中"是 4 类共因(fingerprint 漏维度 / pre-enrich 缓存 EMPTY / invalidate 漏调 / DATA_SOURCE 渲染缺 fallback)。AP-37 补充 4 个**结构性根因**, 都是引入新字段类型 (LIST_FORMULA / Phase B) 或新模板结构 (同 componentId 出现多次) 时漏掉的"协议传播点"。

### 现象 (2026-05-19, QT-1418/1419/1420 一连串报错)

- LIST_FORMULA 字段配进组件 + 报价单卡片渲染 → 整个 tab "加载中…" 永远不更新
- 模板里同 componentId 出现两次 (不同 driver path) → 部分 tab 自动隐藏 / 全部"加载中"
- 跨 4 个不同的根因都表现成"加载中" — 没系统的诊断会反复 hotfix 单点

### 4 个独立根因 (修一个还有 3 个隐藏)

#### 根因 1: enrich mapper 显式 spread 漏字段

`QuotationWizard.enrichComponentData` 和 `BulkImportPartsDrawer.buildComponentDataFromTemplate` 各有一个 field mapper, **显式列出每个要复制的属性**:

```ts
const fields: ComponentField[] = (snapshotComp.fields || []).map((f: any) => ({
  name: ..., field_type: ..., content: ..., is_amount: ..., is_subtotal: ...,
  formula_name: ..., datasource_binding: ..., basic_data_path: ..., ...
  // ← 加新字段类型(如 LIST_FORMULA 的 list_formula_config) 必须在这里 spread,
  // 否则 enrich 后该字段被静默丢弃, useXxx hook 看不到 → 永远不加载.
}));
```

#### 根因 2: normalizeFieldType 4 处独立函数

字段类型枚举在 4 个文件**各有一份** normalizeFieldType:

```
cpq-frontend/src/pages/quotation/QuotationStep2.tsx        line 27
cpq-frontend/src/pages/quotation/QuotationWizard.tsx       line 47
cpq-frontend/src/pages/quotation/BulkImportPartsDrawer.tsx line 80
cpq-frontend/src/pages/quotation/ReadonlyProductCard.tsx   line 48
```

新加字段类型 enum 时 4 处都要同步加 `if (t === 'XXX') return 'XXX';` + return type union 加新枚举值. **漏任意一处 → 该字段类型被悄悄降级到 INPUT**, 渲染分支跳过, 但不报错.

#### 根因 3: cache key 维度漏 dataDriverPath

`useDriverExpansions.driverExpansionKey` 老定义只用 `(partNo, componentId, customerId)`. 当模板出现**两个相同 componentId 不同 driver path 的实例** (如 v1.8 模板有两个 COMP-CFG-PROCESS — 一个绑 v_composite_child_processes 一个绑 mat_process), 两个组件**共享同一 cache 条目**:
- 第一个组件 fetch 完写入 cache (含 v_composite_child_processes.* 键)
- 第二个组件的 tasks dedup 时 `seen.has(key)` 命中, 不发 fetch
- 第二个组件查 mat_process.* 键 → cache miss → 永久"加载中"
- 或: 第二个组件 fetch 完后写入同一 cache slot → 前一个组件的数据被覆盖丢失

**修法**: key 必须 = `(partNo, componentId, customerId, dataDriverPath)`. tasks dedup + cache lookup + autoSave + prune 4 处全部加 driverPath 参数 (约 11 个 callsite).

#### 根因 4: 前端 batchKey 反查依赖 backend r.key (后者不含 override)

即使 cache key 含 driverPath, 前端 batchExpandDriver 完成后用 `r.key` 反查 localKey:

```ts
// OLD: 用 backend 返的 r.key (= cacheKey(cid, cust, part, ver) **不含 override**) 反查
const batchKeyToLocalKey: Record<string, string> = {};
for (const t of missing) {
  const bk = buildBatchKey(t.componentId, customerId, t.partNo, t.partVersion);
  batchKeyToLocalKey[bk] = t.key;  // ← 同 r.key 多 task → 后写覆盖前写
}
for (const r of results) {
  const localKey = batchKeyToLocalKey[r.key];
  cache[localKey] = r.data;  // ← 两个 result 写到同一 cache slot, 后一个赢
}
```

`batchKey` (后端 r.key 格式) 设计上**只含 (cid, cust, part, ver)**, 不含 override tag. 同 (cid, cust, part, ver) 不同 override 的多 task → r.key 相同 → map 反查必然冲突.

**修法**: 不用 r.key map, 改用 **task index 直接对应 result index**. backend 保证 `results.length == tasks.length` 且按 task 顺序返 (含 ERROR 兜底):

```ts
for (let i = 0; i < missing.length; i++) {
  const t = missing[i];
  const r = results[i];   // backend 顺序保证
  if (r?.status === 'OK' && r.data) cache[t.key] = r.data;
  else                                cache[t.key] = EMPTY_EXPANSION;
}
```

#### 根因 5: enrich 反查 `savedById[componentId]` 塌缩 + `saved.tabName` 覆盖 `snapshot.tabName`

`QuotationWizard.enrichComponentData` 把 saved componentData 索引成 `Record<string, any>` 再用 cid 反查:

```ts
// OLD: 同 cid 多条 saved 全塌缩到最后一条
const savedById: Record<string, any> = {};
for (const s of withRows) {
  if (s.componentId) savedById[s.componentId] = s;  // ← 后写覆盖前写
}
return snapshot.map(sc => {
  const saved = savedById[sc.componentId] || {};
  return {
    tabName: saved.tabName || sc.tabName || '',  // ← saved 覆盖 snapshot
    ...
  };
});
```

当模板里同 cid 出现 N 次 (典型: 一个 cid 既挂 v_composite_child_* 也挂 mat_* 形成两个 Tab):
1. 所有 N 个 snapshot entry 反查到**同一条** saved (Record 后写覆盖前写)
2. `saved.tabName || snapshot.tabName` → 所有 N 个 Tab 显示同一个 tabName
3. autoSave 把这堆错误的 tabName 持久化, 此后每次 reload 都从脏数据重生 → **不可逆污染**

**典型现象 (QT-1421)**: snapshot 里 4 对 (cid, 标准/选配-* 名) Tab → 渲染出 4 对全部叫"选配-*"的重复 Tab, 行数据彼此污染。

**修法**:
1. 按 cid 分组成**队列**, 同 cid 多条按 `(cid, tabName)` 精确匹配优先, 命中后从队列剔除:
   ```ts
   const savedQueueByCid = new Map<string, any[]>();
   for (const s of withRows) {
     if (s.componentId) {
       if (!savedQueueByCid.has(s.componentId)) savedQueueByCid.set(s.componentId, []);
       savedQueueByCid.get(s.componentId)!.push(s);
     }
   }
   // snapshot.map 内
   const q = savedQueueByCid.get(snapId);
   let idx = q ? q.findIndex(s => s.tabName === snapTab) : -1;
   if (idx < 0 && q && q.length > 0) idx = 0;
   const saved = (q && idx >= 0) ? q.splice(idx, 1)[0] : {};
   ```
2. **结构性字段一律以 snapshot 为权威, saved 只贡献行/小计**:
   ```ts
   tabName: snapshotComp.tabName || saved.tabName || '',  // snapshot 优先
   componentId: snapshotComp.componentId || saved.componentId || '',
   componentType: snapshotComp.component_type || saved.componentType || 'NORMAL',
   dataDriverPath: snapshotComp.data_driver_path || saved.dataDriverPath,
   ```
   理由: 模板 publish 时 snapshot 已经定格了"什么 cid 在什么 Tab 走什么 driver", 历史脏数据不应该有覆盖权。

3. **load 路径强制走 enrich, 不要因 fields 已存在就跳过**:
   ```ts
   // OLD: 漏掉脏 tabName 的 heal 机会
   const needEnrich = compData.length === 0 || !compData.every(cd => cd.fields?.length > 0);
   // NEW: snapshot 是唯一权威, 始终跑 enrich (fetchTemplateOnce SWR 缓存, 同模板复用零成本)
   const [enriched, productAttrs] = await Promise.all([
     enrichComponentData(li.templateId, li.componentData),  // 始终跑
     needProductAttrs ? loadProductAttributes(...) : ...,
   ]);
   ```
4. 一次 load+autoSave 后, 脏数据自愈, 后续都正常。

#### 根因 6: 详情页 / 编辑页 enrich 协议不对齐 + componentHasData "自动隐藏空 Tab" 与模板 WYSIWYG 冲突

**双套 enrich**: `QuotationStep2`/`QuotationWizard` 走 `enrichComponentData` (snapshot-driven 队列匹配), 而 `ReadonlyProductCard` 有**独立 inline enrich** (saved-driven, `snapshot.find(s => s.cid === saved.cid)` 永远返第一个匹配). 同 cid 多实例时, saved 5..8 也对到 snapshot 0..3 → tabName + fields 错位 + saved.tabName 覆盖 snapshot. **改 enrich 必须同时改两处**, 否则详情/编辑页 Tab 列对不上 (用户首屏直观感受到"丢 Tab"或"重复 Tab").

**componentHasData 与 WYSIWYG 冲突**: 早期为治"SIMPLE 产品挂 COMPOSITE 模板时空 Tab 显示加载中"而引入"自动隐藏空 Tab" (QT-1409 patch), 但弱副作用是**模板配 N 个 Tab 用户只能看到 ≤ N 个**, 而且详情/编辑两边判定逻辑还不一致 (详情没 driver/LIST_FORMULA 分支). 表面像"丢 Tab"实则被运行时数据状态隐藏.

**修法 (2026-05-19 用户决议方案 A)**:
1. **两边 enrich 必须同源**: ReadonlyProductCard 重写为 snapshot-driven 队列匹配 (与 enrichComponentData 完全对齐):
   ```ts
   const savedQueueByCid = new Map<string, any[]>();
   for (const s of withRows) {
     if (s.componentId) savedQueueByCid.set(s.componentId, [...(savedQueueByCid.get(s.componentId) || []), s]);
   }
   return snapshot.map(sc => {
     const q = savedQueueByCid.get(sc.componentId);
     let idx = q ? q.findIndex(s => s.tabName === sc.tabName) : -1;
     if (idx < 0 && q?.length) idx = 0;
     const saved = q && idx >= 0 ? q.splice(idx, 1)[0] : {};
     return { tabName: sc.tabName, fields: sc.fields, rows: saved.rows || [], ... };
   });
   ```
2. **结构性字段一律 snapshot 优先**: `tabName / componentType / dataDriverPath / componentId / componentCode` 全部 `sc.xxx || saved.xxx || default`, saved 只贡献行数据/小计.
3. **撤销 componentHasData**: 详情/编辑两边都改为 `normalComponents = componentData.filter(c => c.componentType !== 'SUBTOTAL')`. 空数据 Tab 内部表格自带"暂无数据"占位行.
4. **后端 admin heal endpoint 洗历史脏数据**: `POST /api/cpq/quotations/admin/heal-componentdata-tabnames?apply=true` 扫所有 lineItem → 按 snapshot 队列匹配 → 重写 tab_name + sort_order. dry-run 默认.

**关键教训**: enrich 是 **跨页面协议**, 不是单页内函数. 任何报价单视图 (编辑/详情/比对/打印…) 复用 lineItem.componentData 都必须走同一份 enrich 实现; 多份就有协议漂移. componentHasData 这种"按数据状态隐藏 UI 结构"的策略与模板 WYSIWYG 不兼容, **空数据用占位提示, 不要用隐藏 Tab**.

### DATA_SOURCE 4 子类型解析协议 (computeAllFormulas + 渲染 双路径)

DATA_SOURCE 字段有 4 个 `datasource_binding.type`, 数据流各不同:

| sub-type | 数据源 | 后端 expand 写入 | 前端 fieldValues 读取 | 前端渲染读取 |
|---|---|---|---|---|
| `GLOBAL_VARIABLE` | global_variable_value 单表 | `basicDataValues['@gvar:'+code]` (gvar task 链) | 同左 | 同左 |
| `BNF_PATH` | 任意视图/表(走 BNF resolver) | `basicDataValues['{bnf_path}']` (parseBasicDataPaths 链) | 同左, 缺失再查 pathCache | 同左 |
| `DATABASE_QUERY` | data_source 表配置的 SQL | 不写入 expand (走前端 dsLoading 状态机异步查询写回 row[key]) | `row[key]` | `row[key]` |
| `HTTP_API` | 外部 API (HttpApiResolver) | 不写入 expand (同 DATABASE_QUERY 走 dsLoading) | `row[key]` | `row[key]` |

任何一处分发逻辑写错 → 前端公式 token 取不到值 → 公式结果异常 / 渲染落"加载中".

### "同 componentId 多实例" cache key 检查

driverExpansion cache key + tasks dedup 必须包含**所有影响请求体的字段**:

```

1. 同 partNo + componentId 出现 2 次 (template 里) 时, 应看到 **2 个独立的 /batch-expand task**, 不是 1 个 (dedup 错位)
2. 后端响应 results 数组长度 = 请求 tasks 数组长度
3. 每个 result 的 basicDataValues 含对应 driver 的键 (v_composite_child_processes.* vs mat_process.*)

## AP-38："0 行 driver 鬼魂行" 加载中 — autoSave 留空 row + globalPathCache miss

> **2026-05-19 E2E 真实渲染验证暴露**。AP-31/37 之后的渲染层兜底链补丁。

### 现象

报价单产品卡片切到某个 Tab (典型 "组合工艺" / "选配-组合工艺")，driver=mat_composite_process 等需要历史业务数据填充的 driver，但当前产品是独立产品没该类数据 → driver expand 返 `rowCount=0`。

预期：Tab 应显示空表 / "—" / "暂无数据"。
实际：表格里有 **1 行**（autoSave 落库的空 row），每个 BASIC_DATA cell 都显示 **"加载中…"** ← UI 谎言

### 根因

`QuotationStep2.tsx#effectiveCount`（≈L1288）:

```ts
const effectiveCount = useDriver
  ? driverCount       // = driverExpansion.rowCount (= 0)
  : useListFormula
  ? lfRowCount
  : activeComponent.rows.length;  // autoSave 留的 row 数 ≥ 1
```

当 `useDriver = false` (driverExpansion.rowCount === 0)，effectiveCount 退化到 `activeComponent.rows.length` — autoSave 留了 1 行空 row → 渲染 1 行"鬼魂行"。

BASIC_DATA 渲染分支三级兜底（≈L1454-1498）:
1. `basicDataValues[lookupKey]` — `undefined`（no driver row）
2. `row[fieldKey]` — 空（autoSave 空行）
3. `globalPathCache[partNo::path]` — **永久 cache miss**（因为 path 是 `mat_composite_process.X`，按 partNo 查全表本就该空 → 但 cache miss 触发 "加载中…"）

### 修法

在第三优先级 `globalPathCache` 兜底**之前**加 **driver 已加载 + rowCount=0** 短路兜底：

```ts
// AP-38 (2026-05-19 前端修): driver 已 fetch 但返 0 行 → 该 Tab 没数据,
// 不该再去 globalPathCache 兜底 (cache miss 必然永久"加载中").
if (
  activeComponent.dataDriverPath
  && activeDriverExpansion
  && activeDriverExpansion.rowCount === 0
) {
  return <span className="qt-ds-placeholder">—</span>;
}
```

判断条件：
- `activeComponent.dataDriverPath` 非空 = 该组件本就是 driver 模式（产品级 `''` 的组件不进这分支）
- `activeDriverExpansion` 已加载 = 不是"fetch 中"的真实 loading
- `rowCount === 0` = driver 表查到的真值是空

### 防护措施

- BASIC_DATA cell 渲染必须区分 **"未加载"（fetch 中）** vs **"已加载且空"** 两个状态
- "加载中…" 仅出现在前者；后者一律 "—" 或 "暂无数据"
- 任何 `globalPathCache` 兜底前都要先判断 driver 状态 — 不能盲目降级

### 历史命中记录

| 日期 | 表现 | 修复 |
|---|---|---|
| 2026-05-19 | 独立产品报价单 "组合工艺" / "选配-组合工艺" Tab 4 列加载中 | `QuotationStep2.tsx` BASIC_DATA 渲染分支加 driver+rowCount=0 兜底 |

### 关键教训

> "加载中…" 不仅是 fetch 中的合法状态，也常被错误地当成**任何 cache miss 的兜底文案**。每个 cache miss 的语义都不同：
>
> - driver 未加载 = 真 loading
> - driver 加载完返 0 行 = 已加载且空 → "—"
> - 路径不存在配置 = 配置错误 → "未配置路径"
> - DATA_SOURCE 查询失败 = 业务错误 → "查询失败"
>
> 不要让单一文案"加载中…"承担 4 种含义。

## AP-40：H1 `refreshSnapshotsByComponent` 同 cid 多 tc 实例 firstResult() 反向污染

> **2026-05-19 B3 调试发现，最高危的 H1 bug**。任何模板 PUBLISH 后用户在组件管理改字段类型 → 自动 H1 sync → snapshot 被错误污染 → 用户配置丢失。

### 现象

- 用户在组件管理把字段 X 从 type A 升级到 type B (典型: 成材率 DATA_SOURCE → LIST_FORMULA + per_item_rules)
- 保存组件成功 → ComponentService.update 自动调 H1 `refreshSnapshotsByComponent`
- 报价单 UI 看不到 type B 的渲染效果 → 排查发现 **template.components_snapshot 里 type 仍 A** 或者 **type 是 B 但 list_formula_config 丢失**
- 同时**同 cid 的另一个 Tab (有 fields_override 的) 也被错误覆盖** 或 **没有 override 的 Tab 沿用了别的 Tab 的 override**

### 根因

`TemplateService.refreshSnapshotsByComponent(componentId)` 老实现:

```java
// 老逻辑 (2026-05-19 修前): 只查 1 个 tc, 用它的 fields_override 覆盖所有同 cid snapshot entry
TemplateComponent tc = TemplateComponent.find(
        "templateId = ?1 AND componentId = ?2", tplId, componentId).firstResult();
List<?> effectiveFields = (tc != null && tc.fieldsOverride != null && !tc.fieldsOverride.isBlank())
        ? parseJsonArray(tc.fieldsOverride) : compFields;
// 然后用这个 effectiveFields 写到所有含 cid 的 snapshot entry
```

**问题**：模板里同 cid 出现 N 次 = N 个 template_component 行 (V204 设计意图 — 每个 Tab 可有独立 fields_override)。`firstResult()` 只拿排序后第一行的 fields_override, 然后给**所有**同 cid snapshot entry 都用这同一个 effectiveFields。

**实际效果** (模板 v1.10 + cid=0a436b6c 2 个 Tab):
- "工序" Tab (sortOrder=1, tc 有 fields_override = 7 字段 DATA_SOURCE 成材率)
- "选配-工序列表" Tab (sortOrder=6, tc.fieldsOverride = null, 应走 component.fields = 6 字段 LIST_FORMULA 成材率)
- H1 跑完: 两个 Tab snapshot 都被改成 7 字段 DATA_SOURCE 成材率 → "选配-工序列表" 的 LIST_FORMULA + per_item_rules 数据**永久丢失**

### 修法 (2026-05-19 Java 代码修复, 无 Flyway)

把 tc 查找下沉到 snapshot entry loop 内, 按 `(templateId, componentId, sortOrder)` 精确匹配:

```java
for (Object entryObj : snapList) {
    Map<String, Object> entry = (Map<String, Object>) entryObj;
    Object cidObj = entry.get("componentId");
    if (cidObj == null || !componentId.toString().equals(cidObj.toString())) continue;

    // 按 sortOrder 精确匹配 tc (每个 Tab 用自己的 fields_override)
    Object sortObj = entry.get("sortOrder");
    Integer sortOrder = (sortObj instanceof Number) ? ((Number) sortObj).intValue() : null;
    TemplateComponent tc = (sortOrder != null)
            ? TemplateComponent.find(
                    "templateId = ?1 AND componentId = ?2 AND sortOrder = ?3",
                    tplId, componentId, sortOrder).firstResult()
            : TemplateComponent.find(
                    "templateId = ?1 AND componentId = ?2", tplId, componentId).firstResult();
    List<?> effectiveFields = (tc != null && tc.fieldsOverride != null && !tc.fieldsOverride.isBlank())
            ? parseJsonArray(tc.fieldsOverride) : compFields;
    // 然后 entry.put("fields", effectiveFields) — 仅覆盖当前 entry
}
```

### 防护措施

- **同 cid 多 tc 场景必须按 sortOrder 精确匹配**，禁止用 `firstResult()` 共用一份 effectiveFields
- 任何 H1 类（"按 source 批量同步多个 target"）的逻辑，**每个 target 必须独立查 source 配置**，不能把 N 个 target 强绑同一份 source
- 新增模板字段时，先在 DRAFT 复测：组件管理保存触发自动 H1 后，所有 Tab 的 snapshot 配置都是预期形态

### 历史命中记录

| 日期 | 表现 | 修复 |
|---|---|---|
| 2026-05-19 | 模板 v1.10 cid=0a436b6c "选配-工序列表" Tab snapshot 成材率从 LIST_FORMULA 被改成 DATA_SOURCE，list_formula_config 丢失 | `TemplateService.refreshSnapshotsByComponent` 按 sortOrder 精确匹配 (Java 代码修复, 无 Flyway) |

### 关键教训

> 后端"按 cid 同步"循环里，任何**外部跨多 entry 共用的查询结果**都是隐患。只要 source 端按 (cid + 其他维度) 区分，target 端的同步循环就必须**每个 entry 独立查 source**，不能在 loop 外提前 firstResult() 缓存。
>
> 这条规则跟 AP-37 第 ⑨ 处 cache key 含 fieldsHash 是**同一类协议变换 — N→1 退化漏掉了"其他维度"**。

## AP-41：prop drilling 漏传 — 报价单视图 vs 核价单视图 ProductCard 不对齐

> **2026-05-19 B3 调试发现**。表现为某个视图功能正常另一个视图功能失效，本质是 React props 显式枚举遗漏。

### 现象

- 同一份数据 (`lineItem.componentData`) 在两个视图都渲染
  - 视图 A (核价单 `costingLineItems`) — 功能正常
  - 视图 B (报价单 `quoteLineItems`) — 同样组件、同样字段，但某个 feature 完全失效 (典型: LIST_FORMULA 字段空白)
- 切换"产品卡片视图" / "Excel 视图" 看到对比差异

### 根因

父组件渲染两个视图时分别 `<ProductCard ... />` 实例化，**props 列表手写漏一个**:

```tsx
// 视图 A: 完整
<ProductCard
  item={...} index={...} customerId={...}
  driverExpansions={driverExpansions}
  configTemplates={configTemplates}   // ← 有
/>

// 视图 B: 漏掉
<ProductCard
  item={...} index={...} customerId={...}
  driverExpansions={driverExpansions}
  // configTemplates 漏传!
/>
```

ProductCard 内部 `configTemplates?.[id]` 总是 undefined → 该视图 LIST_FORMULA / 任何依赖 configTemplates 的 feature 静默失效。

### 修法

- **即时**: 给漏传的视图补 prop
- **长期**: 把 configTemplates / driverExpansions 等"跨视图共享数据"提到 Context 或 ProductCardProvider，避免 prop drilling

### 防护措施

- TS 应当能编译报错 — 但若 ProductCard prop interface 标 `configTemplates?: ...` 可选, 漏传就静默通过. **核心共享数据必须标必填**或用 Context
- PR review 必须**对比同组件多 callsite**：grep `<ProductCard` 看每处 props 列表是否一致
- E2E 必须**两个视图都跑**：报价单视图 + 核价单视图，任一缺一都不算完成

### 历史命中记录

| 日期 | 表现 | 修复 |
|---|---|---|
| 2026-05-19 | 报价单视图 ProductCard 漏传 configTemplates → 选配-工序列表"成材率" LIST_FORMULA 永远空白 | QuotationStep2.tsx L2269 加 configTemplates prop |

### 关键教训

> "跨视图共享数据" 是 React 应用最常见的 prop drilling 陷阱。任何关键 prop 缺省值是 undefined 时, 漏传 = 静默失效, **不报编译错也不报运行时错**, 只能靠两个视图对比测试才暴露.
>
> 现代解法：要么把这类数据提到 Context，要么 prop 标必填让 TS 强制检查。手写 `<X a={} b={} c={} />` 多 callsite 必出错。

## AP-42：spread `...rawRow` 用 null 字段反向覆盖 lfItem 自动映射

> **2026-05-19 B3 调试发现**。effectiveRow 包装层对"用户输入覆盖自动映射"语义有 bug：rawRow 含 null/undefined/'' 时不该算"用户输入"。

### 现象

- LIST_FORMULA 字段 condition `[工序代码] = 'MRO-AS-0001'` 在 driver-bound 行 (mat_process 2 行) 求值永远 false
- 即使该行 basicDataValues 含 `{mat_process.process_code}: 'MRO-AS-0001'`
- 调试 log 显示 `row['工序代码'] = null` (lfItem.code 应该填进来)

### 根因

```ts
const baseRow = lfItem ? {
  '工序代码': lfItem.code,  // 自动映射: 'MRO-AS-0001'
  '工序': lfItem.name,
  ...
  ...rawRow,                 // ← rawRow 是 autoSave 落库的"空"行: { 工序代码: null, 工序: null, ... }
                             // null 字段也是 own enumerable property, 会覆盖 lfItem.code
} : rawRow;
```

**spread 语义**: `{...obj}` 把所有 own enumerable keys 复制过来，包括 value 是 null/undefined 的 keys。后写覆盖前写 → rawRow 的 null 把 lfItem.code 擦掉。

### 修法 (2026-05-19 前端代码修复, 无 Flyway)

```ts
// 过滤 rawRow 内非"真实输入"的字段 (null/undefined/'')
const rawRowNonEmpty = rawRow ? Object.fromEntries(
  Object.entries(rawRow).filter(([_, v]) => v !== null && v !== undefined && v !== '')
) : {};

const baseRow = lfItem ? {
  ...lfItem 自动映射,
  ...rawRowNonEmpty,         // 只有真值才覆盖 lfItem
} : rawRow;
```

### 防护措施

- **`{...auto, ...user}` 模式默认是 user 覆盖 auto** — 但前提是 user 的值有意义. user 含 null/undefined 字段时, 必须先 filter
- 任何"自动填值 vs 用户输入"语义的合并, 在 spread 前 **filter 出非空字段** 是基础动作
- ESLint 加规则: 在已知"用户输入可能空"的合并模式上, 警告未 filter 的 spread

### 历史命中记录

| 日期 | 表现 | 修复 |
|---|---|---|
| 2026-05-19 | LIST_FORMULA condition 求值因 row['工序代码']=null 全失败 → 成材率显示 "—" | QuotationStep2.tsx effectiveRows lfItem 分支用 rawRowNonEmpty |

### 关键教训

> JS 对象 spread 不区分"显式 null" vs "缺失字段" — 都是 own enumerable property. "用户输入覆盖自动填值" 语义在 user 是稀疏对象时**必须显式 filter**.
>
> autoSave 落库的"空"行典型场景: 数据库读出的 row 含所有列 (即使 null), spread 进 React state 后所有 null key 都是 own enumerable. 不 filter 直接 spread 会"主动擦掉"自动填值.

## AP-43：Vite ESM 项目残留 `require()` ReferenceError

> **2026-05-19 B3 调试发现**。从 CommonJS 项目移植/AI 生成代码常见 — Vite/ESM 环境里 require 不存在，运行时抛 ReferenceError。

### 现象

- 渲染层条件计算明明正确 (调试 log 显示 chosenFormula='50') 但 UI 仍显示 "—"
- 因为整个 render 函数被 try/catch 包裹, require() 抛错被 catch 吃掉 → fallback 显示 "—"
- 没有 console 报错 (catch 没 log), 看不到任何线索

### 根因

```tsx
try {
  const { evaluateListFormulaString } = require('../../utils/formulaEngine');  // ❌ Vite ESM 抛 ReferenceError
  const v = evaluateListFormulaString(chosenFormula, ...);
  return <span>{v}</span>;
} catch {
  return <span>—</span>;  // ← 永远命中这里
}
```

Vite (无论 dev/build) 的代码都是 ESM (`<script type="module">`), 浏览器原生 `require` 是 undefined → 整段 try 块第一行抛.

### 修法

改顶部 ESM static import:

```tsx
// 文件顶部
import { evaluateExpression, evaluateListFormulaString } from '../../utils/formulaEngine';

// 使用处
try {
  const v = evaluateListFormulaString(chosenFormula, ...);
  return <span>{v}</span>;
} catch {
  return <span>—</span>;
}
```

如果确实需要 lazy load，用 `await import(...)` (动态 import 是 ESM 标准):

```tsx
const { evaluateListFormulaString } = await import('../../utils/formulaEngine');
```

### 防护措施

- **项目已用 `typescript-eslint/recommended`**（详见 `cpq-frontend/eslint.config.js`），其内置 `@typescript-eslint/no-require-imports` 规则原本能抓 `const x = require(...)`，但**前提是定期跑 `npm run lint`**。本次 AP-43 没被抓到说明此 lint 任务**没在 CI 强制**。修法二选一：
  - (a) 在 CI 流水线加 `npm run lint --max-warnings=0` 强制门槛
  - (b) 在 `eslint.config.js` 额外加 flat-config 形式的 `no-restricted-syntax`:
    ```js
    rules: {
      'no-restricted-syntax': ['error', {
        selector: "CallExpression[callee.name='require']",
        message: 'Vite ESM 项目禁用 require()，改用 import 或 await import()',
      }],
    }
    ```
- catch 分支不能"吃掉"错误 — 至少 `console.warn('LIST_FORMULA eval failed:', err)` 让排查时有线索
- AI 生成代码 review 时全文 grep `require(` — CommonJS 思维容易混入

### 历史命中记录

| 日期 | 表现 | 修复 |
|---|---|---|
| 2026-05-19 | QuotationStep2.tsx LIST_FORMULA 渲染分支 `require('../../utils/formulaEngine')` 抛 ReferenceError, 所有 LIST_FORMULA 字段渲染 "—" | 改顶部 ESM import |

### 关键教训

> **任何 `catch {}` 静默吃错误都是反模式**, 至少 `console.warn`. AI 生成 / 从 CommonJS 移植的代码尤其要全文 grep `require(`. Vite 项目里 require 100% 是错的.

## AP-44：**字段类型变动 / 新增 = 组件管理 + 报价渲染 强联动协议**（核心规范）

> **2026-05-19 用户提出的工程规范，由 B1~B4 + AP-40~43 一连串事故沉淀**。组件管理改字段类型 ≠ 单点变更，**必然牵动报价单渲染 N 个独立路径**。任何"只改一头"的尝试都会埋雷。

### 现象

- 用户在「组件管理」改某字段类型（FIXED_VALUE → DATA_SOURCE / DATA_SOURCE → LIST_FORMULA / 新增 X_TYPE 等），保存成功
- 报价单显示老样子 / 显示"加载中..." / 显示空白 / 公式按 0 算 / 渲染列错位
- 排查发现：
  - snapshot 没刷新 (AP-39 V109 散字段残留)
  - snapshot 刷新了但同 cid 多 Tab 串号 (AP-40 H1 firstResult)
  - cache slot 串号 (AP-37 第 ⑨ 处 fieldsHash)
  - 渲染分支没识别新类型 (AP-37 协议传播 ①~⑥)
  - fieldValues 不含新类型字段 (AP-37 ⑦ computeAllFormulas 分支)
  - 路径采集漏 (AP-37 ⑧ parseBasicDataPaths)
  - prop drilling 漏传 (AP-41 configTemplates)
  - effectiveRow 包装层 spread 反向覆盖 (AP-42)
  - 老代码用 require() 抛错 (AP-43)

**单点修复永远不够** — 必须把"字段类型 = 跨多文件协议"作为基础认知。

### 强制联动检查矩阵

**任何字段类型变动 / 新增** 必须扫这 **15 个检查点**（按数据流顺序）：

> **AP-50 更新（2026-05-22）**：原 #14/#15 已合并为 `ComponentCell.tsx` 单点；新增 #14b gvar hash 维度项。总数维持 15 项但结构重整，任何漏改 `ComponentCell` = 编译错误可感知，不再有"静默双源漂移"风险。

### 关键教训

> **"字段类型" 不是一个属性, 是一个跨 15+ 文件的协议**。改一处必动 15 处, 漏一处必有静默失败 (不报编译错也没运行时崩溃, 只是渲染不对).
>
> 这个协议有 **3 个性质**:
> 1. **跨前后端**: 后端定义校验 + 路径采集 + 公式 token 分支, 前端定义 enrich + 渲染 + 公式求值
> 2. **跨多视图**: 报价单 / 核价单 / 详情页 / 比对 / 批量导入 — 任何视图独立渲染 lineItem.componentData 的都要同步
> 3. **跨持久化层**: component.fields (源) + template.components_snapshot (publish 定格) + template_component.fields_override (Tab 级覆盖) — schema 迁移必须三处同步
>
> **新增字段类型 + E2E** 必须是同一个 PR — 没 E2E 就不算完成. 详见 `docs/E2E测试方法.md`.

## AP-46: BNF 路径在 driver 行上下文被反向收窄 (driver 列谓词污染)

### 现象
- LIST_FORMULA 字段公式形如 `{mat_bom[element_name='Sn'].net_qty}*5`, 路径自带 element_name='Sn' 过滤
- 直接调 `/api/cpq/formulas/batch-evaluate` 测试 OK, 返单值 1.288
- 在报价单选配-工序列表 cell 里渲染 = **0** (前端 fallback)
- 后端 `expand-driver` 返的 `row.basicDataValues["{mat_bom[element_name='Sn'].net_qty}"] = null`

### 根因
- `ComponentDriverService.evaluatePath` 走 `FormulaEngine.evaluate(path, ctx)` 把 driverRow 作上下文
- `ImplicitJoinRewriter` 把 driverRow 中所有"目标表也有的列"作为 AND 谓词注入路径
- driver = mat_process 行, 含字段 hf_part_no/seq_no/part_version 等
- mat_bom 表也有 seq_no 列 → 注入 `AND seq_no=1` (mat_process.seq_no=1)
- 但 mat_bom 中 element_name='Sn' 的行 seq_no=2 → AND 收窄到 0 行 → null

### 防护
- 前端 `evaluateListFormulaString` 加 fallback: basicDataValues 缺值 → 退到 partNo 级 `globalPathCache` (无 driver 注入, 走纯 BNF batch-evaluate). 调用方必须传 `partNo` + `pathCacheState` (React state cache 引用)
- 用户配 BNF 路径时尽量加上**唯一识别谓词** (如 `[element_name='Sn']`) 让结果与 driver 无关
- 后端 `ImplicitJoinRewriter.SYSTEM_COLUMN_DENYLIST` 已包含 id/created_at/status 等; 不宜把业务键 (seq_no) 也加进去 — 否则破坏正常隐式 JOIN. 走前端 fallback 链更稳

### 历史命中
- 2026-05-20 QT-20260520-1448 选配-工序列表"成材率" 渲染 0 (用户配 `{mat_bom[element_name='Sn'].net_qty}*5`, 期望 6.44). 修法: `evaluateListFormulaString` 加 partNo+pathCacheState 入参, 缺值时回退 globalPathCache. 详见 RECORD.md 2026-05-20 双轨二期

## AP-51：**`snapshotRows` Math.max 持久化累加死锁**

> 2026-05-22 立项。AP-31 续集 — autoSave 持久化路径未遵循"driver 行数权威"原则。

## AP-54：**过滤后下标 (filtered index) 当作底层数组下标 → 编辑写错位 Tab，受控 input 假死**

> 2026-05-27 立项（QT-20260527-1656 复现）。背景：渲染层用"过滤子集"的下标遍历展示，写路径却用同一个下标去索引"未过滤的原始数组"，两者下标语义不一致 → 编辑落到错位的对象上。

### 症状
- 报价单编辑页某 Tab 的**文本输入 (INPUT_TEXT) / 数字输入 (INPUT_NUMBER)** 框无法输入字符：键入后 value 立即回退为空（受控 `<input value={row[key]}>` 永不更新）
- 只读类型字段（FIXED_VALUE / FORMULA / BASIC_DATA / DATA_SOURCE / LIST_FORMULA）显示正常（本就不可编辑，看不出错）
- **不报编译错、不报运行时错**，纯静默失败；只在"真正去某个 Tab 键入"时暴露
- 现象与"哪个 Tab 在编辑"强相关：被过滤组件（如 SUBTOTAL）排在前面的 Tab 越靠后，错位越明显

### 根因
`QuotationStep2.tsx` 的 `ProductCard` 用 `normalComponents = item.componentData.filter(c => c.componentType !== 'SUBTOTAL')` 生成 Tab 列表。
- **渲染**正确用 `normalComponents[activeTab]`（过滤子集下标）。
- 但**所有写路径** mutator（`handleRowChange` / `handleInputBlur` / `handleDeleteRow` / `handleAddRow`）与 `dsStateKey` / `loadingKey` 直接拿 `activeTab` 去索引 `item.componentData`（**未过滤**原数组）。
- 当 SUBTOTAL 组件（本例「选配-总成本」）排在 `components_snapshot` 第 0 位时，`normalComponents[i] === item.componentData[i+1]`（整体偏移 +1）。编辑「选配-子配件清单」(activeTab=3) → 写到 `componentData[3]`=「选配-元素含量」→ 当前组件 `row[key]` 永不变 → 受控 input 冻结。

### 修法（最小改动）
在 `activeComponent` 之后计算底层真实下标，所有写路径改用它：
```ts
const activeComponentDataIndex = activeComponent
  ? item.componentData.indexOf(activeComponent)   // filter 保留对象引用 → indexOf 可靠
  : activeTab;
```
把 6 处 `activeTab` → `activeComponentDataIndex`：`loadingKey` / `dsStateKey` / `handleRowChange` / `handleInputBlur` / `handleDeleteRow` / `handleAddRow`。
**保持用 `activeTab`** 的 3 处（语义本就是过滤子集下标，不可改）：Tab 头高亮 `activeTab === ci`、`activeComponent = normalComponents[activeTab]`、越界钳制 `activeTab >= normalComponents.length`。

### 通用规范（核心教训）
**凡是"渲染用过滤/排序后的子集、写回用原始集合"的列表/Tab/表格，写路径的下标必须显式映射回原始集合下标（或直接按对象引用 / 稳定 ID 定位），禁止把过滤子集下标当原数组下标用。** 三种正确做法（按优先级）：
1. 按稳定 ID（componentId / key）在原集合 `findIndex` 定位 —— 最稳，免疫后续插入/重排
2. `originalArray.indexOf(filteredItem)` —— filter/slice 保留引用时可靠（本次采用）
3. 渲染时即携带原始下标（`map` 出 `{item, originalIndex}`），不要事后反推

## AP-56：**多 source 链式 SUM + KSUM 嵌套预聚合的求值落点 / 前后端对称 / inner 白名单坑**（2026-06-13 立项）

> `cross_tab_ref` 求值能力扩展（多 source 链 + KSUM 降维投影），是 [[AP-55]] 的续集。`FormulaToken` 纯新增 `sources`（多 source 链）+ `projectToHostKey`（KSUM），无 DB 列、无 snapshot 迁移。改求值器（`formulaEngine.ts` / `FormulaCalculator.java`）或序列化（`formulaSerialize.ts`）/ validator 时**必读**。完整能力见 `docs/配置方法论-合并版.md §3.4.1`。

### 6 个反模式根因（实现期实际踩到）

**① 决策 K 空集分流的真实落点（最易错）**
- 空集分流**必须**落在统一"空集→0 提前返回"**之前**：前端 `formulaEngine.ts` `aggregateRows` 的 `hits.length===0` 分支；后端 `FormulaCalculator.evalCrossTab` 的 `hits.isEmpty()→ZERO` 之前。
- **绝不能**改 AVG 的 `arr.reduce/arr.length`（前端）/ `.average().orElse(0)`（后端）死分支 —— 那些只在 hits 非空时触达，空集永不到。
- 语义：`KSUM`/`KCOUNT` 空集→0（静默）；`KAVG`/`KMAX`/`KMIN` 空集→**null**→外层注入非法表达式(`(null.x)` / `(0/0)`)→try/catch→整外层塌 0 + ⚠（I-2 污染范围 = 整个外层表达式，不是只该项）。

**② 前后端 mergedRow 必须合并驱动行（否则非空 match KSUM 分叉）**
- 前端 `evalRowExpr` 递归求值时传 `mergedRow = {...hostRow, ...ar}`（驱动行 ar 高优先覆盖）；后端 `targetRowValue` 的 `sub.currentRowRaw` **必须同款合并** `arow`（`new HashMap<>(ctx.currentRowRaw); putAll(arow)`）。
- 漏合并时：KSUM 子 token 带**非空 match**（按驱动行键精确过滤）会从宿主行而非驱动行取键 → 命中不同 → 前后端结果分叉。**全用 `match=[]` 的测试会遮盖此 bug**，必须有非空 match 对拍用例（`cross-tab-cases.json` T7-12）。

**③ 全局变量 token 的实际 type 是 `path` 不是 `global_variable`**
- spec/白名单写"允许 global_variable"，但本仓全局变量 `{路径}` 实际产出 `{type:'path'}` token（`brace_expr → path`）。
- 后端 `TokenMappabilityValidator` 的 KSUM inner 白名单**必须放行 `path`**，否则前端发 `path`、后端按字面 `global_variable` 校验 → 误拒 publish。回显 `tokensToDrawerExpression` 也须有 `path` case，否则 round-trip 丢 `{路径}`。

**④ 单列 KSUM 不复用单列 shortcut（C2）**
- `KSUM([外购件.费用])` 即使内层单列，也**强制**折叠成 `targetExpr:[{field}]` + `target:''`，**不能**把列名塞进 `target` 走单列 shortcut（求值器 KSUM 分支依赖 targetExpr 存在）。

**⑤ 多 source 更粗 source 按行键广播（buildCrossTabRows 不预 join）**
- `buildCrossTabRows` 每个组件的行**独立存储、各含自己的列、未预 join**。多 source 公式里更粗 source 的字段（source≠驱动）**不会**自动出现在驱动行 → 求值器必须在 `evalRowExpr`（前端）/ `targetRowValue`（后端）里对 `sources[1..]` 更粗 source 按 `match` 命中 `crossTabRows[source]` 的行、把列并入（0命中→项0 / 1命中→并入,驱动覆盖同名 / >1命中→multiMatchErr→ERR→0）。`token.sources` 若只序列化不在求值用 = dead code（多 source 静默失效）。

**⑥ insideKsum 配色需 FN_NAMES 含 K***
- `parseFormulaSegments` 的 `FN_NAMES`（insideFn 深度跟踪）**必须含 KSUM/KAVG/KMAX/KMIN/KCOUNT**，否则 KSUM(...) 内引用因不被识别为"在函数内"→ 误判 needs-agg 红。另需独立 `ksumOpenDepths` 栈跟踪 insideKsum，对区间内宿主块/第二被聚合页签块标 `insideKsum-illegal` 红。

## AP-57：**footer 小计"另起炉灶空-crossTabRows 旁路"族**（2026-06-16 立项）

> 报价单渲染 footer（tfoot / `allComponentSubtotals`）的列合计与数据行的求值**分叉**，导致同一数值列"行里一个值、合计里另一个值"。本质是 single-source 反模式在**列合计求和**这一新落点的再现（[[AP-50]]/[[AP-51]] 的列方向续集）。修法 = footer 列小计从 `buildCrossTabRows` 的 `resolvedRows` 单一来源求和（`columnSumsByComp`）。

### 症状

- 同一数值列，**数据行的值与 footer 列小计对不上**（`cross_tab` / `SUMIF` 这类需要逐行预聚合的列尤其明显）。
- **非金额列被强行加 `¥`**（小计行/合计行把本是纯数字的列也渲染成货币）。

### 根因

1. **footer 与数据行分叉**：footer 用 `computeNonSubtotalColumnSums` / `computeTabSubtotalsByColumn`，以**空 `crossTabRows`** 旁路二次重算；而数据行用 `buildCrossTabRows` 的**完整 `store`** 求值 —— 两条路径对同一列得出不同值。
2. **`¥` 无条件加**：货币前缀走无条件 `formatCurrency(...)`，**不看 `field.is_amount`**（误把"小计列"当"金额列"）。
3. **`computeRows` 漏串 `prevRowValues`**：`buildCrossTabRows.computeRows` 未把 `prevRowValues` 串进求值上下文，与渲染层 `effectiveRows` 隐藏分叉（依赖前序行结果的列在两处取值不同）。

### 纪律（写 footer / 列小计 / 货币前缀时必守）

1. **footer 列小计必须从 `buildCrossTabRows` 的 `resolvedRows` 单一来源求和**（`columnSumsByComp`），**禁止**任何以空 `crossTabRows` 二次重算的旁路（`computeNonSubtotalColumnSums` 已删除，勿复活）。
2. **任何"小计/合计"显示值必须 === 已渲染行值之和**（不变量）：footer 某列值 === 该列全部已渲染 `resolvedRow` 之和。改动后用此不变量自验。
3. **`¥` 一律按 `field.is_amount`**：`is_amount===true` 才加货币前缀；非金额列（哪怕 `is_subtotal`）渲染纯数字。
4. **`buildCrossTabRows.computeRows` 必须串 `prevRowValues`**，与渲染层 `effectiveRows` 求值上下文对齐，杜绝"数据区一套值、求和区另一套值"。

### 协议传播点（同源对齐）

| 视图 | 文件 | 关键点 |

| 报价编辑 | `QuotationStep2.tsx` | `buildCrossTabRows` 返 `{store, columnSumsByComp}`；footer tfoot + `allComponentSubtotals` 非小计列读 `columnSumsByComp`；删 `computeNonSubtotalColumnSums`；`¥` 按 `is_amount` |
| 详情页 | `ReadonlyProductCard.tsx` | footer 同源对齐（读 `columnSumsByComp` + `¥` 按 `is_amount`） |

## AP-58：**同一展示值前后端双引擎必然分叉**（2026-06-21 立项）

> 报价 Excel 视图卡片显示 0.93、Excel 视图显示 0.8527——同一数据、两个引擎各算各，结果永久分叉。本质是对同一展示值同时维护两套计算路径，导致不可避免的数值偏差。

### 症状

- 同一报价行的「卡片产品小计」与「Excel 产品小计」不相等（典型偏差：0.93 vs 0.8527）
- 编辑单元格后，卡片实时更新，但 Excel 视图维持旧值；或反之
- 导出 Excel 的数值与页面卡片不符，客户与销售各执一词
- 看代码时发现**两处** `buildExcelValues` 或类似重算调用点，且两者都在写同一列（`quoteExcelValues`）

### 反模式根因

**双引擎 = 对同一输入集合（`componentData + editRows + formulaResults`）存在两条互相独立的计算路径，各自产出同名结果列**。

典型结构：
```
前端引擎（buildExcelSnapshot）→ quoteExcelValues（via saveDraft）
后端引擎（buildExcelValues）   → quoteExcelValues（via editCardValue / refreshQuoteCardValues）
```

两条路径对「公式精度」「字段映射」「driver 行过滤」「单位换算」的处理细节永远无法完全对齐，即便今天对齐，下次任一侧改动又分叉。

### 正确模式：单引擎单写源

```
唯一计算引擎（前端 buildExcelSnapshot）
    ↓
唯一写入端点（POST saveDraft → QuotationService.java:380）
    ↓
唯一读取源（quote_excel_values 列）
    ↓
导出 / 提交 / 展示 全部读同一列
```

**判断口径**：若两处代码对同一 DB 列做赋值（`quoteExcelValues = ...`），且各自有独立的计算逻辑，则必有分叉——消除方法是**保留一条计算路径，删除另一条**，而不是"努力让两条路径产出相同结果"。

### 强制规范

1. **一列一引擎**：任何持久化列（`quote_excel_values` / `costing_excel_values` 等）在全代码库只允许一条计算+写入路径；发现第二条视为 AP-58 违规。
2. **写入点唯一性校验**：每次新增对 `*ExcelValues = ` 的赋值前，先 `grep -rn "*ExcelValues =" src/`，确认只有 saveDraft + bootstrap（`== null`）两处，否则拒绝合入。
3. **bootstrap 不是第二引擎**：`== null` 守卫的 bootstrap 调用只允许在新行首次落库时运行一次，之后立即被前端 saveDraft 接管；bootstrap 的存在不构成双引擎，但 bootstrap 之外的任何后端重算均构成双引擎。
4. **核价侧同样适用**：`costingExcelValues` 目前由后端单一引擎管理（核价不走前端编辑），属于合规的单引擎单写源，但一旦核价侧也引入前端计算，需同步退役后端重算路径。

### 修复检查清单

- [ ] `grep -rn "quoteExcelValues =" src/main/java` → 只剩 saveDraft 写入 + `== null` bootstrap（各一处）
- [ ] 删除 `editCardValue` 里的 `buildExcelValues` 调用（含 `if (excel != null) li.quoteExcelValues = excel`）
- [ ] 删除 `refreshQuoteCardValues` 里的 `buildExcelValues` 调用（同上）
- [ ] 保留 `snapshotLineValues` 里的 `if (== null)` 守卫并注明「bootstrap 兜底，非第二引擎」
- [ ] 前端 `buildExcelSnapshot` 在每次 saveDraft 前调用，覆盖所有编辑场景（单元格失焦 / 行增删 / 显式刷新）

### 关联

- [[AP-50]]（single-source 渲染层）— 本条是其「写入层」方向：AP-50 是读路径双 source，AP-58 是写路径双引擎

## AP-59：**前端单引擎接管"显示值"时，误以为"列定义/语义"也在客户端**（2026-06-21 立项）

> AP-58 的续集。把 Excel 视图从后端引擎搬到前端 `buildExcelSnapshot` 后，连续踩两个坑——**都源于一个错误前提：「列定义和 token 语义在客户端就能拿全」**。实际上「列从哪来、token 怎么解析」很大一部分是**服务端配置/语义**；前端只接管「值计算」时必须把这些服务端语义**完整取回或补齐**，否则静默全 0（不报错、不抛异常，纯显示 0）。

### 坑 1：v2 引用配置的列定义在服务端，客户端 `getExcelViewConfig` 拿不到列

- **症状**：报价 Excel 视图全列为 0；「前端算值」绿标已显示（前端路径已生效），但 `parsedColumns` 为空 → `buildExcelSnapshot` 无列可算。
- **根因**：报价模板 `template.excel_view_config` 存的是 `{excel_component_id, column_overrides, version}`（**服务端引用**），列由后端 `getEffectiveColumns` 解析 `excel_component_id → component.excel_columns + column_overrides`。前端 `getExcelViewConfig` 端点返回**原始 v2 配置**，客户端 `parseExcelViewColumns` 对 v2 得**空数组**。
- **错误假设**：spec/plan 初稿写「列复用 `useLinkedExcelRows` 客户端解析」「退役 `useBackendExcelRows`」——对 v2 配置直接失效。
- **正解**：列定义**仍由后端解析**。显示侧取 `useBackendExcelRows.parsedColumns`（`useBackendExcelRows` 改作「列来源」未退役）；saveDraft 侧加后端端点 `GET .../excel-view-config/effective-columns`。值仍走前端 `buildExcelSnapshot`。**列结构由后端解析不破坏恒等性（分叉只在值不在列）**。

### 坑 2：`expressionToTokens` 需要**完整 TabDef**（componentId + subtotalCols），`col.tabs` 子集不够 → TAB_JOIN 列全 0

- **症状**：列定义已正确取到（`cols= A,B,C`），`componentSubtotals` 也算对了（`来料#材料成本=0.0433`、`报价小计=62.19`），但最终列值 `A=0 B=0 C=0`。**小计算对了，列求值得 0**。
- **根因**：TAB_JOIN_FORMULA 列表达式（如 `[来料.材料成本]` / `[报价小计(总计)]`）须经 `expressionToTokens(expr, tabDefs)` 转 token。`expressionToTokens` 据 `TabDef.subtotalCols` 判 `[别名.列]` 是 **component_subtotal**（读 `componentSubtotals[别名#列]`）还是 **cross_tab_ref 明细**（读空的 crossTabRows）；据 `TabDef.componentId` 解析 `[别名(总计)]` 的组件引用。而 `col.tabs` 只有 `{alias, tabKey, rowKeyFields}`——**缺 `componentId`（只有 `tabKey`）、缺 `subtotalCols`** → `[别名.小计列]` 误判明细读空、`[别名(总计)]` 引用缺失 → 全 0。
- **正解**：按 `tabKey` 从 `item.componentData` 补全完整 `TabDef`：`componentId = tabKey`，`subtotalCols = fields.filter(is_subtotal).map(name)`，`detailFields` / `componentName` 同理（与后端 `tabDefsByComponent` 同口径）。
- **测试陷阱（重要）**：初版 vitest「通过」是假绿——夹具 `tabs` 里**手动补了 `componentId`** 且用 `[来料(总计)]`（whole-tab）规避了 `subtotalCols` 依赖，**没忠实复刻线上 `col.tabs` 子集形态**，掩盖了真实失败。**教训：构造 TAB_JOIN / 配置驱动的测试夹具，必须忠实复刻线上数据结构（缺哪些字段就别补），否则夹具"作弊"会让红用例变绿、放过真 bug。**

### 强制规范（前端接管任何"原后端计算的展示值"时）

1. **先问"语义在哪"**：列定义、token 解析规则、字段元数据（is_subtotal / componentId / subtotalCols / 单位换算系数 / 全局变量）哪些是服务端配置？前端接管"值计算"前必须把这些**完整取回（新端点 / 已有响应）或从已有数据补齐**，不能假设客户端天然有全集。
2. **全 0 / 全空 = 优先查"输入是否齐"**：症状是"算出来全 0"而非报错时，先用一行 `console.warn` 打 `列数 / 小计键数 / 关键中间值 / 最终列值`，定位是「输入缺」「中间算错」还是「末端取值键不对位」三段中的哪段（本案 = 末端 token 解析键不对位）。
3. **测试夹具忠实复刻线上结构**：配置驱动（Excel 列 / TAB_JOIN / 字段类型）的用例，夹具字段集必须 = 线上真实结构；缺字段不许私自补全，否则等于把"线上会缺的字段"在测试里填上，红用例假绿。
