## 0. 既定前提（复核确认）

1. `QuotationLineItem.productPartNoSnapshot` = **生产料号**，是递归种子；递归 SQL `unnest(:production_part_nos)` 起，`JOIN material_bom_item ch ON ch.material_no=b.material_no ... system_type='PRICING'`（`material_no`=生产料号）。**已复核 `CostingTreeRenderService.render:89`。**
2. 活跃递归 SQL（`costing_bom_tree_config` 名 `BOMV2`，is_active）**无 sales 过滤**；每层 `bom_version` 子查询 `LIMIT 1` 在 N 个 sales 版本间非确定。**已从 DB 拉全文复核。**
3. `material_customer_map` 已含 `system_type`/`production_no`/`sales_part_no` 三列（报价 Spec1 + 本 spec V311 均已在 worktree 库落地，**已 `\d` 复核**）。三码：销售↔客户 1:1；生产↔销售 **1:N**。
4. 渲染两段：**递归段**走 `CostingTreeRenderService.queryRecursive`（裸 JDBC，不经 SqlViewExecutor）；**driver 段**每个 `$view` 组件 `componentDriverService.expandUncached(compId, null)` 靠线程级 `CostingTreeVarsContext` 注入 `:total_material_no` 收窄、按 `material_no` 分桶。`SqlViewExecutor.injectCostingTreeVars`（:437）是现有注入点；`ImplicitJoinRewriter` 只注 `hf_part_no`+`customer_id`，denylist 含 `is_current`。

## A. 核心难点：批量渲染 + 1:N 下如何按销售料号隔离

### A.1 问题精确化

`render` 把整单**所有** line item 的种子合成一个数组跑**一次**递归 + driver 段跑一次。1:N 触发的两个致命点：

- **递归段扇出**：`JOIN material_bom_item ch ON ch.material_no=b.material_no`——若生产料号 M 有 S1/S2 两个 sales 版本均 `is_current`，每个父节点的子行复制 N 份，且内层 `bom_version` 子查询 `LIMIT 1` 跨 sales 非确定 → 树结构本身错乱（不止翻倍，取错版本）。
- **分组键塌缩**：`CostingTreeGrouping.group` 按 `root_no`（=种子生产料号）建 `treeRowsByRoot` / `cardMaterialNo`。若 li1=(生产 M, 销售 S1)、li2=(生产 M, 销售 S2)，两者 `root_no` 同为 M → **两棵树塌进同一 `byRoot[M]` 桶**，卡片料号集合被并集污染。`rootToLineItemIds` 也按生产料号 M → [li1,li2] 混指。**已复核 `CostingTreeGrouping.java:24-26` + `render:87-94`。**
- **driver 段分桶塌缩**：$view 按 `material_no` 分桶。M 的子件在 S1/S2 两套 capacity/element/unit_price 值不同（本 spec 的 sales 维度就是让它们不同），但都落进 `byKey[M]` → 卡片装配拉到 N 套值 → 单值 BNF 路径返数组（"共 N 项"）。

### A.2 候选解法评估

| 解法 | 机制 | 评价 |
|---|---|---|
| **(1) 种子升为 (生产料号, 销售料号) 对，递归 CTE 携带 sales 并每层 `AND ch.sales_part_no = 根 sales` 收窄** | 因**整树同 sales**（spec §4.1），沿边传播根 sales 即可精确隔离 S1/S2 两棵树；分组键改用 sales | ✅ **选定**。改动集中在递归 SQL + render 种子/分组键 + 分桶键；单次递归/单次 driver 段不变（保住批量性能） |
| (2) 按销售料号分组多次递归（每 sales 跑一遍 CTE） | N 次递归 + N 次 driver 段 | ❌ 退化批量为逐树，perf 回退；且 driver 段跨树重复 expand |
| (3) 只在渲染层分桶键升为 (material, sales)、递归不改 | 递归段仍扇出 + `bom_version` 取错版本 → 树结构已错，分桶救不回 | ❌ 修不了递归段（fact #2）；方案 B 的致命缺口原样复发 |

### A.3 选定方案对各触点的影响（总纲）

**核心洞察：销售料号 `sales_part_no` 全局唯一、整树共享 → 让它同时充当「树标识（分组键）」与「每层 sales 收窄锚」，一列两用，改动最小。**

1. **递归种子**：`List<String> seed`（纯生产料号）→ **两条平行数组** `productionSeed[]` + `salesSeed[]`（同序、按 sales 去重）。
2. **递归 SQL**：`unnest(:production_part_nos)` → `unnest(:production_part_nos, :sales_part_nos)`；**`root_no` 由「生产料号」改为「销售料号」**（充当树标识 + 每层 sales 过滤锚 `ch.sales_part_no = b.root_no`）；`material_no` 仍是生产料号（保 JOIN 与边键语义不变）。
3. **`CostingTreeGrouping` 零改**：仍按 `root_no` 分组——因 `root_no` 现在是 sales（唯一），S1/S2 天然分桶。`node_path/nodeId` 仍基于生产料号（边键语义不变；跨卡 node_path 相同无害，各在独立 `byRoot` 桶）。
4. **`rootToLineItemIds`**：key 由生产料号改为**销售料号**（li 经 §B.2 取值链解析得 sales）。
5. **driver 段分桶键**：sales-aware 组件由 `material_no` 升为 **(material_no, sales_part_no)**（树页签再含 parent_no）；sales-agnostic 组件（$view 不输出 sales_part_no）**保持 material_no 单键**（QUOTE/非版本化 driver 零影响）。
6. **收窄绑定**：`CostingTreeVarsContext` 增 `totalSalesPartNo` 维；`SqlViewExecutor.injectCostingTreeVars` 注入 `:total_sales_part_no`；6 必改视图 WHERE 引用之。

- **driver 段**：`render:129` 由 `set(new Vars(null, g.totalMaterialNo))` 改为 `set(new Vars(null, null, g.totalMaterialNo, totalSalesPartNo))`。

### B.2 `CostingTreeRenderService.render`——销售料号取值链（关键）

**line item 无直接 salesPartNo 字段**（已核 `QuotationLineItem`：仅 `productPartNoSnapshot`=生产料号、`customerPartNo`=客户料号）。**生产↔销售=1:N，故不能用生产料号反查 sales**（同一生产料号在同一客户下可对多个客户产品/多个销售料号）。**唯一确定性键 = 客户料号（`customerPartNo`），它与销售料号 1:1。**

**确切取值链**（每个 line item）：
```
custCode = customer.code(quotation.customerId)   // 调用方 CardSnapshotService 已持 q.customerId
salesPartNo = SELECT material_no
              FROM material_customer_map
              WHERE system_type='QUOTE'
                AND customer_no = :custCode
                AND customer_product_no = :li.customerPartNo
              LIMIT 1                              // 1:1 由 uq_mcm_quote_cust_prod 硬保证
```
QUOTE 行的 `material_no` = 报价料号 = 销售料号（报价 Spec1 §1），与版本化表 `sales_part_no` 同口径（本 spec §0.1「销售料号=报价料号，贯穿报价/核价」）→ 用它收窄核价 PRICING 视图正确。

> 等价备选（PRICING 行）：`SELECT sales_part_no FROM material_customer_map WHERE system_type='PRICING' AND customer_product_no=:customerPartNo LIMIT 1`。同样需 `customerPartNo`。选 QUOTE 行接法（不依赖生产料号、1:1 更干净）。

**取值来源接线**：`render` 当前签名 `render(UUID templateId, List<QuotationLineItem> lineItems)` **无客户上下文**。两选：
- **B2-a（选定）**：签名加 `UUID customerId` → `render(templateId, customerId, lineItems)`。调用方 `CardSnapshotService:501/619` 已有 `q.customerId`，直接传。render 内解析一次 custCode（复用 SqlViewExecutor 同款 `customer.code` 查询/缓存思路），批量查 map 得 `li → salesPartNo`。
- B2-b：render 内经 `QuotationLineItem.quotationId → Quotation.customerId` 反查（多一跳 DB，signature 不变）。→ 不选（render 已持 lineItems，加参更直接）。

**种子/分组键改造**：
```java
// customerPartNo 为空 → 该行 sales 不可解析(见 §C 风险): 记 warn, 该 li 退化为"生产料号单键"(无 sales 收窄, 保留旧行为兜底)
LinkedHashMap<String,String> salesToProduction = new LinkedHashMap<>(); // sales -> production, 按 sales 去重
Map<String, List<UUID>> rootToLineItemIds = new LinkedHashMap<>();      // key 改为 sales
for (li : lineItems) {
    String production = li.productPartNoSnapshot;
    String sales = resolveSalesPartNo(li, custCode);   // 上面取值链; 可空
    if (production==null||production.isBlank()) continue;
    String rootKey = (sales!=null && !sales.isBlank()) ? sales : production; // 兜底
    salesToProduction.putIfAbsent(rootKey, production);
    rootToLineItemIds.computeIfAbsent(rootKey, k->new ArrayList<>()).add(li.id);
}
List<String> productionSeed = new ArrayList<>(salesToProduction.values());
List<String> salesSeed      = new ArrayList<>(salesToProduction.keySet()); // 平行, 同序
List<String> totalSalesPartNo = new ArrayList<>(rootToLineItemIds.keySet());
```
- `queryRecursive(cfg.sqlTemplate, productionSeed, salesSeed)`（§B.3）。
- driver 段 `set(new Vars(null, null, g.totalMaterialNo, totalSalesPartNo))`。
- 分桶（§A.3.5）+ 装配：`cardSales = root`（=`rootToLineItemIds` 的 key）。

### B.3 递归 SQL（`costing_bom_tree_config.sql_template`，活跃 `BOMV2`）

**改后完整 SQL**（`root_no` 载销售料号；每层 `ch.sales_part_no = b.root_no` 收窄；内层 `bom_version` 子查询相关到本树 sales 消 `LIMIT 1` 非确定）：

```sql
WITH RECURSIVE bom AS (
  SELECT
    sp::text                                       AS root_no,       -- ★ 销售料号=树标识+每层过滤锚
    p::text                                        AS material_no,   -- 生产料号(根节点)
    (SELECT bv.bom_version::text
       FROM material_bom_item bv
      WHERE bv.material_no = p
        AND bv.customer_no = '_GLOBAL_'
        AND bv.system_type = 'PRICING'
        AND bv.is_current  = true
        AND bv.sales_part_no = sp                  -- ★ 相关到本树 sales, 消跨 sales LIMIT 1
      LIMIT 1)                                      AS bom_version,
    NULL::text                                     AS parent_no,
    p::text                                        AS node_path
  FROM unnest(:production_part_nos, :sales_part_nos) AS s(p, sp)     -- ★ 平行数组解种子对

  UNION ALL

  SELECT
    b.root_no,
    ch.component_no::text                          AS material_no,
    (SELECT bv.bom_version::text
       FROM material_bom_item bv
      WHERE bv.material_no = ch.component_no
        AND bv.customer_no = '_GLOBAL_'
        AND bv.system_type = 'PRICING'
        AND bv.is_current  = true
        AND bv.sales_part_no = b.root_no           -- ★
      LIMIT 1)                                      AS bom_version,
    ch.material_no::text                           AS parent_no,
    (b.node_path || '/' || ch.component_no)::text  AS node_path
  FROM material_bom_item ch
  JOIN bom b ON ch.material_no = b.material_no
            AND ch.sales_part_no = b.root_no       -- ★ 每层按 sales 收窄, 隔离 S1/S2 两棵树
  WHERE ch.customer_no  = '_GLOBAL_'
    AND ch.system_type  = 'PRICING'
    AND ch.is_current   = true
    AND ch.component_no IS NOT NULL
) CYCLE material_no SET is_cyc USING cyc_path
SELECT root_no, material_no, bom_version, parent_no, node_path
FROM bom
```

**`queryRecursive` 配套改造**（`CostingTreeRenderService.java:259`）：现只绑 `:production_part_nos`。改为按出现顺序绑两个 text[]：
- 签名 `queryRecursive(String sqlTemplate, List<String> production, List<String> sales)`。
- 扫描 SQL 中 `:production_part_nos` / `:sales_part_nos` 的出现，按**文本先后**逐个 `?` 绑对应数组（现 SQL 中各出现 1 次于 base case unnest；用 SqlViewExecutor.rewriteNamedParams 同款「命名→位置」扫描，避免手数下标）。
- `CostingTreeNode` 结构（root_no/material_no/bom_version/parent_no/node_path 5 列）**不变**——`root_no` 语义变（生产→销售）但列不变，无需改 DTO/Grouping。

**落点定夺（是否需 V315）**：**需要**，递归 SQL 必改（fact #2）。`BOMV2` 行由运行时插入（V307 仅建表，migration 无 INSERT，已核）。为可复现 + 与本 spec 全删重导迁移链一致，**走新 Flyway `V315`**（就地 UPDATE 活跃行）：
```sql
-- V315__costing_bom_tree_sales_dim.sql
UPDATE costing_bom_tree_config
   SET sql_template = $BOMV3$ ...(上面完整 SQL)... $BOMV3$,
       name = 'BOMV3_sales', updated_at = now()
 WHERE is_active;                       -- 就地改活跃行, 不碰 ux_cbt_active 部分唯一索引
```

### B.4 6 个必改 PRICING 视图模板（`component_sql_view.sql_template`）

### B.5 收窄机制定夺

**选定：绑定变量 `:total_sales_part_no`（driver 段集合）+ `:sales_part_nos`（递归段平行数组）。不扩 `ImplicitJoinRewriter` 注入、不动 denylist。**

理由：
- driver 段的 sales 收窄发生在 **$view 模板自身 WHERE**（`sales_part_no = ANY(:total_sales_part_no)`），在 $view 子查询求值时即生效——**无论该 $view 是被 driver 采集还是被单值 BNF 路径内联引用（`$view.col`）**，收窄都已烘进子查询，与外层 `ImplicitJoinRewriter` 注入的 `hf_part_no`/`customer_id` 谓词正交。故单值 basic_data 路径（zh capacity 不良率 / ys 元素单价）经这些 $view 也收窄到单行——这正是方案 A 相对方案 B 的根本优势。
- `is_current` 在 denylist、`sales_part_no` 又是运行时上下文里**本不存在**的维度（清单 §2.2 实测零命中）→ 只能显式绑定注入，不能靠 ImplicitJoin 自动注。绑定路径不碰 denylist，零副作用。

**注入点（driver 段 ctx → SqlViewExecutor 绑定链）**：
1. `render` driver 段 `CostingTreeVarsContext.set(Vars(..., totalSalesPartNo))`。
2. `SqlViewExecutor.injectCostingTreeVars`（:437-442）加一行：`if (v.totalSalesPartNo != null) namedParams.put("total_sales_part_no", v.totalSalesPartNo);`。
3. `rewriteNamedParams` 已通用处理 `:xxx`——未引用该占位符的模板（QUOTE/非版本化 $view）无副作用（不出现即不绑）。

**分桶 sales-awareness 容错**（关键，防 QUOTE/非版本化 driver 被误升键）：分桶时读 `r.driverRow.get("sales_part_no")`；**该组件所有行均无此列 → sales-agnostic → 保持 material_no 单键 + 装配 material-only（今行为）**；有此列 → sales-aware → 键含 sales。按组件粒度判定（首个非空行探测或列存在性），存 `Set<String> salesAwareCompIds`，装配分支同判。→ 6 必改 PRICING $view 走 sales-aware，其余零影响。

## C. TDD + E2E + 影响面

### C.1 单元测试（后端，worktree 的 `cpq-backend/` 内 `./mvnw`）

构造：同生产料号 **M**，两个销售料号 **S1/S2**，各一套 `is_current` 数据——`material_bom(+item)`（M 及其子件，sales_part_no 分别打 S1/S2）、`capacity`、`element_bom_item`、`unit_price` 费用；`material_customer_map` 各一 QUOTE 行（custCode + 各自 customer_product_no → S1/S2）+ PRICING 行。两个 line item：li1(生产 M, 客户料号→S1)、li2(生产 M, 客户料号→S2)。

| 断言 | 预期 |
|---|---|

## 汇总报告（回主线的五个定论）

1. **1:N 解法选定**：**种子升为 (生产料号, 销售料号) 平行数组；递归 CTE 用 `root_no`=销售料号一列两用（树标识 + 每层 `ch.sales_part_no=b.root_no` 收窄）；分组键/分桶键改用销售料号**。整树共享 sales（spec §4.1）使沿边传播即精确隔离，保住单次递归 + 单次 driver 段的批量性能。候选 (2) 多次递归、(3) 只改分桶 均否决（perf 回退 / 修不了递归段扇出与取错版本）。

2. **收窄机制选定**：**绑定变量 `:total_sales_part_no`（driver 段）+ `:sales_part_nos`（递归段），经 `SqlViewExecutor.injectCostingTreeVars` 注入，不扩 `ImplicitJoinRewriter`、不动 denylist**。收窄烘进 $view 模板 WHERE → 对 driver 采集与单值 BNF 内联引用一体生效，这是方案 A 能修 basic_data 单值路径的关键。分桶按组件 sales-awareness 容错，QUOTE/非版本化 driver 零回归。

3. **销售料号取值链**：line item 无直接 sales 字段；**必经 `customerPartNo`（客户料号，与销售料号 1:1）反查 `material_customer_map` QUOTE 行的 `material_no`**——**不能用生产料号反查（1:N 歧义）**。render 签名加 `customerId`（调用方已持 `q.customerId`）。

4. **Task 拆分**：T0 前置核对 → T1 ctx/注入 → T2 render（基线核心·高危·E2E）→ T3 递归 SQL+queryRecursive（骨架+DB config·高危·E2E）→ T4 六视图（AP-53·E2E）→ T5 重启+验证。T3/T4 config 并入同一 V315。

5. **最大风险点**：
   - **(R1·数据前提) `customerPartNo` 可空** —— 存在 `customerPartNo=null` 的 SaveDraft 流（`SaveDraftRequest.java:58` 注释实证）。空则 sales 不可解析 → 该卡退化生产料号单键（不崩但无 sales 隔离，1:N 时该卡仍可能扇出）。设计已给兜底 + warn；**须在实现期确认核价场景 `customerPartNo` 必填/必达**，否则方案 A 对这些行失效。列为落地前须确认项。
   - **(R2) `productPartNoSnapshot`=生产料号口径** 已随本设计复核确认（递归 JOIN 打 `material_bom_item.material_no`=生产料号），非存疑。
   - **(R3) 递归 config 是运行时数据非 migration** —— V315 就地 UPDATE `WHERE is_active` 需活跃行存在；强删重导后须确保 BOMV2 活跃行已插再跑 V315，或 V315 内 upsert 兜底。

6. **递归 SQL 是否需 V315**：**需要**。递归 SQL 必改（root_no=销售料号 + 每层 sales 收窄 + 内层子查询相关 sales + 双数组 unnest），且 `costing_bom_tree_config` 活跃行原由运行时插入、未纳入 migration → 走 `V315` 就地 UPDATE 活跃行的 `sql_template`（号段实现前按实测 `MAX(version)` 核准），与本 spec 全删重导迁移链一致；建议与 6 视图 config 改动并入同一迁移集中重启。
