## 0. 分析口径与关键前提

- **本次改动实质**：8 张版本化表（`unit_price`、`material_bom(+item)`、`element_bom(+item)`、`capacity`、`labor_rate`、`production_energy`、`auxiliary_energy`、`tooling_cost`）在 **PRICING（核价）行**上新增 `sales_part_no` 并进 uq / 升版 groupKey。结果：**同一生产料号 `material_no` 现在会有 N 个 `sales_part_no` 各自 `is_current=TRUE` 的行并存**。
- **扇出判定的唯一触发条件**：视图读上述某张版本化表，`WHERE` 只按 `system_type='PRICING'` + `is_current`（可能再加 `customer_no='_GLOBAL_'`）过滤、按 `material_no` 取数，**不带 sales_part_no 维度** → 单个生产料号命中 N 行 → 扇出 / 聚合翻倍 / BNF 单值路径返数组（"共 N 项"）。
- **QUOTE 侧不在本 spec 范围**：本次只写 PRICING 行。凡视图模板 `system_type='QUOTE'`（读报价数据、按 `customer_no=:customerCode` 过滤），其数据不被本次改动触碰。QUOTE 侧是否也有 sales 扇出，属**另一份 spec**（`2026-07-06-报价料号统一`）的评估范围，**本清单不判其为必改**，仅标注。
- **B 类物理视图两张（`v_c_summary_agg` / `v_costing_summary_full`）读的是 `costing_part_*` / `mat_fee` / `plating_fee` / `costing_summary` 家族——不是这 8 张版本化表**，与本次改动无关。

## 2. 架构问题一：销售料号上下文能否注入这些视图？

### 2.1 谓词注入机制现状（证据）

- **`ImplicitJoinRewriter`（`cpq-backend/.../formula/dataloader/ImplicitJoinRewriter.java`）**：BNF 路径求值时把"有效上下文 driver"以 `AND col=val` 注入。有效上下文 = `{hf_part_no: partNo, customer_id: customerId}` 与显式 driverRow 合并；注入条件 = 目标物理表**存在该列** + 原谓词未声明该列 + 值非 null。**`SYSTEM_COLUMN_DENYLIST` 显式包含 `is_current` / `status`**（永不注入）。**注入维度只有 `hf_part_no` 与 `customer_id`，没有 `sales_part_no`。**
- **`SqlViewPathRewriter`**：把 `$view` inline 成 `(SELECT sql_template) sqlview_<name>`，尾部 `[谓词].col` 作用在该子查询**输出列**上。即注入/过滤只能命中视图 `SELECT` 暴露出来的列。
- **核价树渲染路径（`CostingTreeRenderService.render`）**：
  - 整单以 `li.productPartNoSnapshot` 为种子，经 `costing_bom_tree_config` 递归 SQL 展开，`CostingTreeVarsContext` 只携带 `seed(种子料号)` + `totalMaterialNo`（**全是生产料号 material_no 集合，无 sales_part_no**）。
  - 每个 driver 组件的 `$view` 用 `componentDriverService.expandUncached(compId, null)`（**customerId/partNo/partVersion/lineItemId 全传 null**），仅靠 `:total_material_no` 收窄，跑**一次**，结果按 **`material_no`**（树页签按 `(parent_no, material_no)` 边键）分桶。
- **实测**：`sales_part_no` / `salesPartNo` 在 `CostingTreeRenderService` / `datasource/sqlview/` / `ImplicitJoinRewriter` **零命中**。

### 2.2 结论

**现状机制无法把 sales_part_no 注入这些视图——因为运行时上下文里根本没有 sales_part_no 这个维度。** 核价树渲染整条链路（种子 → 递归 → `:total_material_no` → 分桶）**只认生产料号**，`is_current` 又被 denylist 排除，所以即便不改视图，扇出也会在渲染/分桶层直接发生（一个 material_no 桶里塞进 N 个 sales 版本的行）。

要修正，二选一（或组合）：

- **方案 A · 过滤（注入/绑定 sales_part_no）**：把"该核价树对应成品的销售料号"（spec §4.1：一棵核价树内所有版本化行的 `sales_part_no` = 成品销售料号，全树同值）灌进运行时上下文——扩 `CostingTreeVarsContext` 增 `salesPartNo` 维 + 视图模板加 `:sales_part_no` 绑定变量（或扩 `ImplicitJoinRewriter` 有效上下文 + 视图 `SELECT` 暴露 `sales_part_no`）。
  - **优点**：单值 basic_data 路径（capacity 不良率、元素单价/含量等）能收窄到单行，从根上消 "共 N 项"。
  - **代价**：跨 `CostingTreeRenderService` / `CostingTreeVarsContext` / 视图模板 / 可能 `ImplicitJoinRewriter` 的上下文管线改造（**中等偏大**）。**这是唯一能修 basic_data 单值路径的方案。**
- **方案 B · 暴露 + 分组**：视图只 `SELECT sales_part_no`，渲染/分桶层把桶键从 `material_no` 升为 `(material_no, sales_part_no)`，再靠 line item 的成品销售料号选中正确桶。
  - **优点**：改动小，够 driver/list 页签（多行明细）用。
  - **致命缺口**：**修不了 basic_data 单值 BNF 路径**——单值求值仍在视图层拿到 N 行 → "共 N 项"（AP-22 原样复发）。zh_view（capacity 不良率）、ys_view（元素单价）这类单值消费点必须走方案 A。

**推荐**：以**方案 A（过滤）为目标态**——因为 spec §4.1 已定"整棵核价树共享成品销售料号"，该值在渲染根节点即可得（前提：确认 `productPartNoSnapshot` 与版本化表 `sales_part_no` 同口径，见 §5 存疑）。视图 `SELECT` 顺带暴露 `sales_part_no`（便于分桶/排障），但**收窄靠注入/绑定**，不能只暴露。

## 4. P18 `finished_material_no` 连带点清单

- **背景**：本次 P18/19/20/22/23（自制加工费等成品维度费用）的 `unit_price.finished_material_no` 由旧的 `=code` 改为 `null`（PRICING 行）。凡按 `finished_material_no` join 这些 cost_type 的 **PRICING** `unit_price` 行，该 join 会断（匹配 null）。
- **实测扫描结果（全库）**：
  - `component_sql_view` 中"读 `unit_price` 且 `system_type='PRICING'`"的模板 → **0 个**（正则 `unit_price` + `system_type.*='PRICING'` 命中为空；此前 ILIKE `%PRICING%` 的命中系 `pricing_price` 列名假阳性，已逐字核验排除）。
  - `pg_views` 中"join PRICING `unit_price` by `finished_material_no`" → **0 个**。
  - 18 个在册视图里所有 `unit_price` 读点（dd/gx/jg/qt/wgj + v_composite_child_processes）**全部 `system_type='QUOTE'`**。
- **结论**：**本轮 18 个视图中，无一按 `finished_material_no` join PRICING `unit_price` 的这些 cost_type → P18 的 `finished_material_no=null` 改动不会打断任何在册核价视图。连带必改点 = 0。**
