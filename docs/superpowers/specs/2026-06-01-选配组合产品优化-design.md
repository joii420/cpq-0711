# 选配组合产品优化 设计方案

- **日期**: 2026-06-01
- **状态**: 已评审通过，待实现
- **范围**: 新建报价单 → 选配添加 → 组合产品（COMPOSITE）流程的 4 项交互/数据优化
- **核心原则**: 选配只负责**数据入库**，不做任何系统自动计算；价格由用户配置的组件公式引用 BOM 字段自行计算

---

## 1. 背景与现状

当前选配添加是一个 4 步抽屉（`ConfigureProductDrawer.tsx`）：

```
Step0 产品类型(SIMPLE / COMPOSITE)
   → Step1 逐个配件: 料号搜索 → 材质/元素 → 工序
   → Step4 组合工艺(选工艺 + 勾参与配件 + 填参数)
   → Step5 汇总
```

现状痛点：

1. **没有配件数量概念** —— `material_bom_item.composition_qty` 在 `insertMaterialBomAssemblyV6` 中被**硬编码为 1**（`ConfigureProductService.java:642`），用户无法配置每个配件的组成用量。
2. **组合工艺步骤过重** —— 用户被迫勾选「参与配件 1/配件 2」并填写 `paramSchema` 参数，绝大多数场景用不到。
3. **工序 / 组合工艺选择只显示名称** —— 不显示编码；工序搜索只过滤 name/category，组合工艺根本没有搜索框。
4. **配件配置是纯线性流程** —— 配完配件 1 才能配配件 2，无法直观看到选配进度，也无法快速跳回已配的配件修改。

---

## 2. 目标（4 项需求）

| # | 需求 | 验收要点 |
|---|------|----------|
| 1 | 进入组合工艺前，展示所有已选配件清单，允许修改**配件数量**（默认 1） | 数量正整数、默认 1、min 1、step 1；写入 `material_bom_item.composition_qty` |
| 2 | 组合工艺步骤里用户**只选用哪些组合工艺**，不选参与配件、不填参数 | UI 去掉参与配件勾选区 + 参数输入区；默认全配件参与、参数空 |
| 3 | 工序、组合工艺选择时在名称下显示**编码**，支持按编码或名称**模糊搜索** | 工序显示 `process.code`；组合工艺显示 `composite_process_def.code` + 新增搜索框 |
| 4 | 配件配置阶段顶部展示**配件框**，显示选配进度，可切换到指定配件修改 | 顶部 chip 标注 已完成/当前/未开始；**只能点击跳回已完成的配件**，未到达置灰 |

非目标（YAGNI）：

- 不做配件数量参与价格的系统自动乘法（由用户公式计算）。
- 不给 SIMPLE 单产品加配件数量（SIMPLE 用原有报价数量即可）。
- 不允许配件数量填小数（只接受正整数）。

---

## 3. 改造后流程总览

```
Step0 产品类型(选 COMPOSITE + 配件数 N)
   ↓
Step1 逐个配件配置: 料号 → 材质 → 工序        ← 顶部【配件框】导航 (需求#4)
   ↓
Step2 配件清单 + 数量编辑 (新增步骤)            ← 需求#1
   ↓
Step3 组合工艺 (只选用哪些, 带编码 + 搜索)       ← 需求#2 + #3
   ↓
Step4 汇总确认
```

SIMPLE 单产品流程不变（无配件清单步、无组合工艺步），仅工序选择沿用需求 #3 的编码显示 + 搜索。

## 4. 模块详细设计

### 模块 1 — 配件清单 + 数量编辑（需求 #1）

- **位置**：所有配件配置完成后、进入组合工艺之前，新增一个 wizard 步骤（仅 COMPOSITE 出现）。
- **UI**：列表展示已选配件，每行只读显示 `料号 / 名称 / 材质符号 / 规格`，行尾一个 `配件数量` `InputNumber`：
  - 默认值 `1`，`min={1}`，`step={1}`，`precision={0}`（正整数），`parser` 拦截小数/非法输入。
  - 校验：所有配件数量必须 ≥ 1，否则禁止「下一步」。
- **状态**：`PartState` 增加字段 `quantity: number`（初始化为 1）。
- **数据落库**：数量写入 `material_bom_item.composition_qty`，对应 `characteristic='ASSEMBLY'` 且 `component_no=子料号` 的行。
  - 前端：`ConfigureProductRequest.PartRequest` 增加 `quantity`（或父级传 `childQuantities: number[]`，按子件顺序与 `childHfPartNos` 对齐——实现时二选一，推荐放在 `PartRequest` 内以保证与料号天然对齐）。
  - 后端：`ConfigureProductService.insertMaterialBomAssemblyV6` 接收每配件数量，把第 642 行 `VALUES (..., 1)` 的硬编码 `1` 替换为传入数量。
- **明确边界**：系统不做任何 `× 数量` 计算，仅把 `composition_qty` 入库，后续由用户配置的组件公式引用该字段自行计算。

### 模块 2 — 组合工艺简化（需求 #2）

- **去掉** `Step4CompositeProcess.tsx` 中：
  1. 勾选「参与配件 1/配件 2…」的 checkbox 区；
  2. 按 `paramSchema` 渲染的参数（number/text）输入区。
- **保留**：仅让用户从组合工艺库多选「使用哪些组合工艺」。
- **默认值（前端提交时填充）**：
  - `participatingPartIndexes` = 当前全部配件索引；
  - `paramValues` = `{}`（空对象）。
- **后端不变**：`insertCompositeProcesses` / `quotation_line_composite_process` 写库逻辑保持，只是收到的恒为「全配件参与 + 空参数」。组合工艺成本由工艺定义本身决定，不依赖参数。

### 模块 3 — 工序 / 组合工艺带编码 + 模糊搜索（需求 #3）

- **工序（`Step3Process.tsx`）**：
  - 左侧列表项在工序名称下方增显工序编码 `process.code`。
  - 搜索框过滤逻辑由「name/category」扩展为 **`code` 或 `name` 模糊匹配**（不区分大小写，包含匹配）。
- **组合工艺（`Step4CompositeProcess.tsx`）**：
  - 当前无搜索框 → **新增搜索框**。
  - 组合工艺库卡片在名称下增显编码 `composite_process_def.code`。
  - 支持按 `code` 或 `name` 模糊搜索。

### 模块 4 — 顶部配件框导航（需求 #4）

- **位置**：仅在「逐个配件配置」阶段（Step1：料号/材质/工序）顶部显示一排配件 chip/标签。
- **状态**：每个配件标注三态——`已完成 / 当前 / 未开始`，用颜色 + 图标区分。
- **交互**：**只能点击跳回已配置完成的配件**进行修改；未到达（未开始）的配件置灰不可点。
- **实现**：新增轻量进度组件（如 `configure/AccessoryProgressBar.tsx`）。由 `ConfigureProductDrawer` 维护「已完成配件索引集合」并下发；点击已完成 chip → 跳转到该配件的 Step1 子流程。

## 6. 数据模型说明

`material_bom_item`（V6 §4 物料 BOM 子表）相关列：

- 组合产品父行：`material_no = 父料号(COMPOSITE)`，`characteristic='ASSEMBLY'`。
- 每个配件一行：`component_no = 配件子料号`，`seq_no = 序号`，`composition_qty = 配件组成用量`（本次新增可配置）。
- `composition_qty` 列定义 `precision 18, scale 6`（库支持小数），但**业务层只允许正整数输入**。

无新增表 / 无新增列（`composition_qty` 已存在），仅前端补输入 + 后端去硬编码。因此**无 Flyway 迁移**。
