# 跨页签引用公式 设计方案

- 日期：2026-06-05
- 状态：已评审通过（待落实施计划）
- 路线：**甲（新增 `cross_tab_ref` token 类型）** — 2026-06-05 读真实代码后由路线乙修正而来
- 修正说明：最初选路线乙（往 JEXL `FunctionRegistry` 加 `X*` 函数）。但读代码发现 **B 页签列公式（FORMULA 字段）走的是 token 数组引擎**（前端 `formulaEngine.ts#evaluateExpression` + 后端 `FormulaCalculator#appendToken`），**不经过 JEXL**；JEXL 函数只服务衍生属性/独立 eval 端点。故 `X*` 函数无法出现在列公式里，改用与 `component_subtotal`/`previous_row_subtotal` 同构的**新 token 类型**。

## 1. 背景与需求

在组件（页签）公式配置中，需要支持 B 页签的列公式**引用另一个页签 A 的列数据**，按匹配键关联。两个具体需求：

1. **跨页签取值（VLOOKUP 式）**：`当 A.a = B.a 时，取 A.b`，结果用于 B 页签列公式。
2. **跨页签聚合（SUMIF 式）**：`当 A.a = B.a 时，sum(A.b)`，结果用于 B 页签列公式。

### 现状（为什么是新能力）

- 公式引擎在**后端**（JEXL3，`FormulaEngine.java`），支持 `SUM/AVG/COUNT/IF/LOOKUP` 等，但**没有条件聚合（SUMIF）**。
- 公式 token 作用域基本是**同组件、同行内**。跨页签目前只有两条受限通道：`component_subtotal`（只能引用对方组件小计单值）和"全局变量动态 key"（按当前行字段查全局变量表）。
- **公式里直接引用"另一个页签的某一列"目前不支持**。本需求即新增此能力。

## 2. 需求边界（已与用户逐条确认）

1. **匹配条件复杂度**：多列 AND 等值（`A.x = B.x 且 A.y = B.y ...`）。暂不做非等值/区间。
2. **需求1 取值的匹配数量语义**：匹配 0 行 → 返回 0；匹配多行且未用聚合 → **报错**。
3. **公式中的角色**：既能单独成列，也能作为子表达式参与更复杂公式（与现有 token/函数体系混用）。
4. **可引用范围**：仅**同一产品卡片（模板）内**的其它页签。
5. **循环依赖**：**禁止成环**，检测到 A↔B 互引 → 报错拦截。链式引用允许（A.b 本身可为另一跨页签结果），靠拓扑排序解决。
6. **聚合函数集**：SUM / AVG / COUNT / MAX / MIN（+ 取值 XLOOKUP）。
7. **配置方式**：引导式配置面板（抽屉），不手敲文本语法。
8. **生效范围**：报价单视图、核价单视图、详情只读页**三视图均需正确一致**。
9. **实现路线**：甲——新增 `cross_tab_ref` token 类型（见路线修正说明）。

### 边角默认值（已确认）

- A 的参与行 = 该卡片内 A 页签当前所有**数据行**（driver 展开后），**剔除 SUBTOTAL 小计行**。
- 匹配键比较**计算后的值**；A 或 B 任一方匹配键为 null/空串 → 视为不匹配。
- 需求1 取值目标列可为文本；若结果再参与 `+-*/` 而非数字 → 报错。
- 需求2 聚合目标列必须为数字，遇非数字行 → 报错（COUNT 除外，只数行数）。
- 计算时序：后端统一算，A 先于 B（拓扑序）；成环 → 配置时报错拦截。

## 3. token 结构（对外语义）

新增一种公式 token `cross_tab_ref`，与现有 `component_subtotal`/`previous_row_subtotal` 同族，由配置面板生成、可嵌入任意列公式：

```jsonc
{
  "type": "cross_tab_ref",
  "source": "<A 组件稳定标识>",     // componentId（运行时按 id；id 缺失退 componentCode + sortOrder）
  "sourceLabel": "投料",            // 仅回显用
  "target": "<A 目标列字段名>",      // agg=COUNT 时可空
  "match": [                        // 多列 AND；a=A 列名, b=B 当前行列名
    { "a": "子件编号", "b": "子件编号" }
  ],
  "agg": "NONE"                     // NONE | SUM | AVG | COUNT | MAX | MIN
}
```

求值语义（agg 决定）：

| agg | 对应需求 | 语义 | 0 匹配 | 多匹配 |
|---|---|---|---|---|
| `NONE` | 需求1 | 取匹配行 target 值 | 返回 0 | **报错** |
| `SUM` | 需求2 | 匹配行 target 求和 | 0 | 求和 |
| `AVG` | 需求2 | 平均 | 0 | 平均 |
| `COUNT` | 需求2 | 匹配行数（不取 target） | 0 | 计数 |
| `MAX`/`MIN` | 需求2 | 极值 | 0 | 取极值 |

- 聚合类（除 COUNT）target 遇非数字行 → 报错。
- NONE 取值 target 可为文本；若结果再参与 `+-*/` 而非数字 → 报错。

## 4. 双引擎对等实现（关键架构约束）

公式计算是**前后端双引擎镜像**，`cross_tab_ref` 必须在两处对等实现，否则三视图不一致（AP-41/AP-44 红线）：

- **前端引擎** `cpq-frontend/src/utils/formulaEngine.ts#evaluateExpression`：报价单编辑 / 核价单 / 详情只读三视图的**实时显示**值由它算。新增 `case 'cross_tab_ref'`。
- **后端引擎** `cpq-backend/.../quotation/service/FormulaCalculator.java#appendToken`：**快照落库 / 导出**由它算。新增对应 `case`。
- 两端逻辑用**共享测试夹具**锁一致（参考既有 `formula-reconcile-cases.json` 模式）。

求值时 `cross_tab_ref` 需要"A 组件全部已算行"。两端各自维护一个**同卡片"已算行存储"**（组件标识 → 行表，行 = 字段名→已算值，剔除 SUBTOTAL 行），是把现有 `componentSubtotals`（组件→单值）升级为组件→整张行表。

> 既有先例：跨页签按条件匹配/聚合的能力在 **Excel 模板列**里已实现（`CardFormulaEvaluator` 的 `topoOrder`+环检测、`CardDataProvider.rowsOf`、`CardRef` ROW_WHERE、`CardAggregateSource` SUM_OVER）。本需求是把该能力搬进**组件列 token 引擎**，算法可直接借鉴。

## 5. 核心新基建：跨组件求值编排

本需求真正的新基建（路线甲乙都需要）：

1. **组件依赖图 + 拓扑排序**：扫描卡片内每个组件公式里的 `X*` 函数，得出"B 依赖 A"边。按拓扑序决定计算顺序；检测到环（A↔B）→ 报错拦截（配置时 + 运行时双保险）。
2. **同卡片"已算行存储"**：每个组件算完后，把它所有数据行（driver 展开后、**剔除 SUBTOTAL 行**）的"字段名→已算值"快照存进 RuntimeContext，键 = 组件稳定标识。
3. **匹配求值**：B 求值时，从存储取 A 的行，逐行比对每个匹配列对（比**计算后的值**），任一方 null/空 → 该行不匹配；命中行集合再按函数做取值/聚合。

> 链式引用（A.b 本身是跨页签结果）天然由拓扑序解决。

## 6. 配置 UI（引导面板）

FormulaBuilder 新增按钮"跨页签引用"，弹**抽屉**（遵守项目 Drawer 规范）：

1. 选**源页签 A**（下拉，列出同卡片其它组件，含实例区分）。
2. 配**匹配列对**（可加多行：A 列 = B 列）。
3. 选**目标列 A.b**。
4. 选**聚合方式**（无=XLOOKUP / SUM / AVG / COUNT / MAX / MIN）。
5. 确认 → 生成对应 `X*(...)` 函数文本插入公式，并以**可读文案**回显（如 `跨页签[投料].单重 当[投料.子件=本.子件]`）。

## 7. 校验（模板保存时拦截）

- 源组件存在于同卡片；匹配列在 A、B 两侧均存在；目标列在 A 存在。
- 聚合函数目标列声明类型应为数值（能查到类型则静态拦截，否则运行时拦截）。
- 跨组件依赖**无环**。
- 范围越界（引用别的卡片）→ 报错。

## 8. 计算编排与三视图接入

计算是双引擎，编排需在两端各做（落点已勘定）：

- **后端**：`CardSnapshotService#assembleTabsWithFormulaResults`（PASS1 建 componentSubtotals、PASS2 逐 tab `FormulaCalculator.calculate`）。新增一步：按组件级 `cross_tab_ref` 依赖**拓扑排序组件计算顺序**，并在算完每个组件后把其行表写入"已算行存储"，传入 `FormulaCalculator`。环检测复用 `CardFormulaEvaluator.topoOrder` 同款 Kahn 算法。
- **前端**：`QuotationStep2.tsx#computeAllFormulas` + 其 componentSubtotals 编排循环（约 684–694 行）；同步改 `ReadonlyProductCard.tsx`（详情/核价只读）与 `useCardSnapshots.ts`。组件级拓扑排序 + 已算行存储与后端对齐。
- 现有"组件出现顺序"改为"按 `cross_tab_ref` 依赖拓扑序"，与 `component_subtotal`/`previous_row_subtotal` 的既有顺序依赖协同（拓扑序是其超集，不破坏旧行为）。

## 9. 测试

- **后端单测**（`FormulaCalculator` + `CardSnapshotService`）：
  - agg=NONE：0 行→0 / 1 行→值 / 多行→报错。
  - agg=SUM/AVG/COUNT/MAX/MIN。
  - 多列 AND 匹配；null 键不匹配；非数字聚合报错。
  - 组件级拓扑排序顺序；环检测报错；链式引用（A.b 本身是 cross_tab_ref）。
- **前端单测**（`formulaEngine.test.ts`）：`cross_tab_ref` case 与后端**共享夹具**逐项对齐。
- **E2E（协议级改动强制，AP-44 矩阵）**：跑 `quotation-flow.spec.ts`，新增跨页签场景 spec，验证报价单 + 核价单 + 详情三视图结果一致、无"加载中"残留。

## 10. 已知风险 / 注意点

- 与 AP-37（同 componentId 多实例 cache 冲突）相关：组件标识优先 componentId，退化时用 componentCode + sortOrder，匹配/存储键不能只用 componentCode。
- 与 AP-44 字段类型联动矩阵相关：新增 token 类型须对照矩阵补齐协议传播点（enrich / 渲染 / cache key / 后端校验白名单 / FormulaZone 回显 / getFormulaDeps 依赖采集等）。
- 三视图一致性是历史高发坑（AP-41/AP-44/AP-50）：前后端双引擎必须对等，且前端三处编排（QuotationStep2 / ReadonlyProductCard / useCardSnapshots）都要接，不得只改一处。
- `cross_tab_ref` 的依赖是**组件级**（不同于现有 FORMULA 字段级 `getFormulaDeps`）；`getFormulaDeps` 需扩展出"采集本公式引用的源组件"以喂组件拓扑排序。
