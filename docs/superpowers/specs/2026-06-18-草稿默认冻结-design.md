# 报价单草稿默认冻结（freeze-on-bake）设计

> **状态**：Phase 0 设计（待用户评审）。
> **性质**：`docs/三大核心模块基线.md`「报价单渲染」基线的破坏性变更，按基线纪律执行（评审 + 分阶段 + 强制 E2E + 回写基线）。
> **立项来源**：`docs/RECORD.md` 2026-06-08「严格冻结结论」标记的"待立项"，2026-06-18 正式立项。

## 1. 背景与问题

现行设计（`docs/superpowers/specs/2026-06-01-报价单产品行整份快照-design.md` §5/§8）规定：**DRAFT 草稿报价单"打开即重刷"**——前端进 Step2 自动调 `POST /quotations/{id}/refresh-card-snapshot` → `CardSnapshotService.refreshDraftQuoteCards` → `rebuildStructureForDraft`（重建结构）+ 逐行 `refreshQuoteCardValues`（重查 `$view`/driver + 重算公式 + 按 rowKey 保留编辑）。只有提交（SUBMITTED+）后才冻结。

**暴露的问题**：报价单 `QT-20260617-1758`（DRAFT）打开后单元格显示 `#ERROR[QUERY_ERROR]: 路径解析失败：$ll_view._类型`。根因链：
1. V2 来料组件 `3cb220be` 字段 `类型`/`组成用量` 的 `default_source.path` 写成 `$ll_view._类型`/`$ll_view._组成用量`（带 `_` 前缀），但 `ll_view` 实际输出列是 `类型`/`组成用量`（无前缀）。元素 `ad99c10d` 同类。
2. 07:00 对 V2 组件/模板的编辑引入了该不一致。
3. 13:32 用户打开草稿 → on-open 自动重刷 → `evaluatePath` 叶列短路落空 → 回退 `SqlViewExecutor` 严格 ASCII 正则拒中文列 → `FormulaError` 串被当作单元格值**冻进 `resolvedRows`**。

**产品诉求（用户拍板）**：草稿首次定型后即冻结，打开不再自动重查；只在用户显式点击「刷新基础数据」时才重算；模板结构创建时冻结、永不变更（R1）。

## 2. 目标与非目标

**目标**
- 草稿首次 bake（加产品/新建）后，**打开不再自动重查/重建**——结构与值都冻结。
- 提供显式「刷新基础数据」按钮：用户主动触发才按最新基础数据**重算值**（不重建结构）。
- 编辑仍按已存快照重算（不变）。
- 存量脏快照（含 `#ERROR`）通过"先修数据 → 一次性干净重烤"迁移到冻结态。

**非目标**
- 不改快照存储模型、不改渲染读路径、不新增数据库列。
- 不改核价侧（永久冻结，不变）。
- 不改编辑回写 `editCardValue` 的快照重算逻辑。

## 3. 核心设计

### 3.1 冻结判据（复用现有列，零新增）

`QuotationLineItem` 已有 `cardSnapshotAt`（加产品首次 bake 时间，写一次）与 `quoteValuesAt`（最近重刷/编辑时间）。

| 语义 | 判据 |
|---|---|
| 是否已首次冻结（行级） | `cardSnapshotAt != null` |
| 整单是否仍可变 | `quotation.status == 'DRAFT'`（SUBMITTED+ 永久冻结，不变） |

**否决新增列**：新增 `frozen` 与 `cardSnapshotAt` 语义重复，易成"两个真相源"。结构冻结判据用"结构快照已存在"（`quoteCardStructure != null`）。

### 3.2 两个写入口（明确边界）

| 入口 | 触发 | 行为 | 幂等 |
|---|---|---|---|
| 首次 bake（隐式） | 加产品 / 新建行 / copy | 仅当 `cardSnapshotAt == null` 跑 expand + 算四份快照 + 写 `cardSnapshotAt` | 已 bake 行二次调用 = no-op |
| 显式刷新（按钮） | 用户在 Step2 点「刷新基础数据」 | `force=true`：重 expand 值 + 重算 + 按 rowKey 保编辑；**不重建结构（R1）** | 每次点都执行；仅 DRAFT |

**`force` 语义集中在 service 方法签名**（不分散到各 caller，防 2026-06-08 类漏判）：
- 首次 bake / copy / add：`cardSnapshotAt == null` → 跑；否则跳过。
- 显式刷新 / `deleteDriverRow` / `restoreAllDriverRows`：`force=true` → 无视 `cardSnapshotAt` 强制重算值。

### 3.3 结构冻结（R1，用户拍板）

- 结构在创建/首次 bake 时固定，**此后永不重建**。
- `rebuildStructureForDraft` 从草稿刷新路径**整个移除**（不再被 on-open，也不被刷新按钮调用）。
- 模板重发布/改字段**不传播**到已有草稿。要最新模板结构 → 删单重建。
- 显式「刷新基础数据」**只刷值、不刷结构**。

## 5. 存量迁移

**时序硬约束：必须先修 Bug1，再重烤。** 否则重烤把同样的 `#ERROR` 冻回去。

### 5.1 前置：Bug1 字段路径↔视图列名对齐
1. **全库审计端点**（如 `GET /admin/components/audit-basicdata-paths`）：扫所有组件 `fields[].default_source.path`，凡 `$<view>.<列>` 的 `<列>` 不在该 view 实际输出列集合中者列为可疑，输出"组件id/字段名/配置 path/视图实际列/建议修正"。**不自动改**（`_前缀` 未必都错，须按视图真实列校验）。
2. **修复走配置端点**（改 `component.fields` 的 path），触发 `refreshSnapshotsByComponent` 同步模板 snapshot（防 AP-39 残留旧 path）。
3. **加固**：组件保存时对 `default_source.path` 的列名做存在性软校验，配错即告警（防回归）。

### 5.2 存量草稿干净重烤 → 冻结
- **一次性 admin 端点** `POST /admin/quotations/migrate-freeze-drafts`（支持 `dryRun`，可重跑）：
  - 对所有 `status=DRAFT` 行：先校验其引用组件路径已修（未修则拒绝并列脏组件）；再对每行 `force=true` 干净重烤（重 expand 值 + 重算）清掉 `#ERROR`。
  - 重烤值来源：`loadComponentsSnapshot(customerTemplateId)`——须确认其字段 path 已是修正后的（依赖 §5.1 已触发模板 snapshot 同步）。
  - **R1 下不重建结构**；若个别存量草稿的冻结结构快照本身携带旧 path 致渲染/expand 取错，作为**一次性迁移例外**允许该端点对受影响草稿重建一次结构（仅迁移时，非常态）。← plan 阶段据数据核实是否需要。
  - 返回每单结果（成功/失败/仍含 ERROR 字段清单）。
- **不走 Flyway**：迁移依赖 Java expand + 公式引擎，且不能硬编码 UUID。
- **迁移顺序**：部署新代码（on-open 不再自动刷）→ §5.1 审计+修复 → §5.2 重烤 → 抽样验证无 `#ERROR`。

## 6. 影响面与风险

| 能力 | 冲突 | 缓解 |
|---|---|---|
| 草稿打开自愈（6/17 单位换算修复依赖 on-open 重刷带出修好的底层数据） | on-open 不再自动刷 → 老草稿打开看不到新值 | 用户主动点「刷新基础数据」。预期行为变更，发版说明 + 按钮 tooltip 告知 |
| 草稿刷出后台改的基础数据（6/2：on-open 前 `evictForLineItem` 重查） | 同上 | `evictForLineItem` 保留在显式刷新路径，按钮点击时仍正确重查 |
| 老草稿拿不到模板新增字段/页签（R1 结构永冻） | R1 取舍 | 删单重建（产品已接受） |

**反模式雷区**：AP-51（显式刷新仍重 expand → 守 `rowCount` 权威、禁 `Math.max`，且冻结降低反复刷暴露面）；AP-54（刷新按 rowKey 对齐保编辑，不变）；AP-44（本方案不新增/改 field_type，不触发 17 检查点）；AP-39（§5.1 走配置端点防 snapshot 残留）。

**性能/体验**：打开草稿不再全行重 expand + 二次 getById + loading → 打开显著变快。风险：用户忘点刷新看到旧数据 → 顶部提示条缓解。

## 8. 测试策略

**E2E（强制，基线纪律）**：`quotation-flow.spec.ts` + `composite-product-flow.spec.ts`。
- 现有用例更新：打开草稿**不再**有"正在重新计算"loading、**不再**发 `refresh-card-snapshot` 请求；不变量 `'加载中' final count = 0` + 8 Tab 渲染保持。
- 新增用例：
  1. 打开不重查：后台改底层数据 → 打开 → 断言仍显示旧快照值 + 未发 `/refresh-card-snapshot`。
  2. 显式刷新才重查：点按钮 → 确认 → 断言发请求 + 值更新 + rowKey 命中行编辑保留、被删行编辑丢弃 + **结构不变（R1）**。
  3. 首次 bake 不被挡：新建 → 加产品 → 卡片非空白。
  4. 冻结幂等：重复打开 N 次 → `cardSnapshotAt` 不变、值不变（AP-51 守护）。
  5. 存量迁移（集成测试）：构造含 `#ERROR` 脏快照 DRAFT → 修路径 → 跑重烤端点 → 断言无 `#ERROR`、`类型`/`组成用量` 解出真实值。
  6. 编辑仍按快照重算（回归，不重 expand）。

**PR 证据**：双 spec passed + 加载中=0；打开（旧值、无重算 loading）vs 点刷新（新值）两态截图；新建加产品非空白截图；迁移端点 dryRun 输出 + 实跑后 DB 查询无 `#ERROR` 截图。
