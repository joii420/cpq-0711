## 1. 背景与现状

### 1.1 现状（已实现的部分）
- **值快照已存在**：`方案-加产品整份快照.md`（Phase 1–4，2026-05-29）给 `quotation_line_component_data` 加了 `snapshot_rows`（JSONB），加产品时冻结每个 Tab 每行的解析值（`$view.col` BASIC_DATA / `@gvar` DATA_SOURCE / BNF 路径值）。渲染经 `ComponentDriverService.expandWithSnapshot` 命中快照直返，**基础数据变化不影响已加行的值**。
- **结构未冻结**：页签（Tab）、字段清单、字段属性（`field_type`、默认值、是否可编辑/`is_required`、`basic_data_path`、`datasource_binding`、排序、公式）仍来自 `template.components_snapshot`——**模板级**快照，同模板下所有报价单共用。模板重发布 / `refreshSnapshotsByComponent` 会改动已存在报价单的**卡片结构**。
- **Excel 视图未脱钩**：报价/核价 Excel 视图（`LinkedExcelView` / `ExcelViewService`）的列来自独立的 `costing_template.columns`（VARIABLE/FORMULA），取值仍走 BNF 路径**实时查**基础数据（`excel_view_snapshot` 是可选缓存，非渲染主路径）。

### 1.2 问题
卡片"值"冻了但"结构"没冻；Excel 视图整体没冻。导致：模板/基础数据变化仍可能改变已存在报价单的展示，渲染未与数据源彻底脱钩。

### 1.3 目标
**每个报价单产品行存一份自包含快照，结构 + 值全部下沉到报价行级；渲染完全只读快照，与基础数据、模板都脱钩。** 要更新只能删行重加（永久冻结）。

## 2. 核心需求（已与用户确认）

| # | 决策 | 取值 |
|---|---|---|
| 脱钩程度 | 结构 + 值都下沉到产品行级，渲染完全读快照 | 彻底脱钩 |
| 刷新时机 | **永久冻结（A）**：加产品即冻死；基础/模板变化一律不影响；**不提供重建按钮**，要换只能删行重加 | A |
| 存储形态 | **产品行级整份自包含快照（①）** | ① |
| 编辑层 | **①a 内部三层** `structure` / `baseRows` / `editRows`；取值编辑优先、否则基础值 | ①a |
| 渲染范围 | **范围三**：① 报价卡片 ② 报价 Excel ③ 核价卡片 ④ 核价 Excel 四份；比对视图不存（②④ 衍生比对） | 范围三 |
| 计算口径 | **Z 细化**：渲染零实时计算/零基础查询；核价 ③④ 永久只读冻死；报价 ①② 仅在用户编辑可编辑字段时触发 FORMULA + Excel 重算并回写报价侧快照 | Z |
| 物理存储粒度 | **P2 四列分存**（非对称读写：核价冻死、报价可回写，物理隔离） | P2 |
| 旧链路处理 | **Q1**：旧链路保留实现，降级为"加产品/编辑重算的种子计算器"，从渲染读路径移除；删测试存量；无兼容回退 | Q1 |

### 2.1 四份数据模型（用户原话语义）
报价单加入一个产品时，用**报价模板**和**核价模板**分别跑一遍"查询 + 计算 + 渲染"，产出四份数据：① 报价视图·产品卡片、② 报价视图·Excel、③ 核价视图·产品卡片、④ 核价视图·Excel。四份全部冻进报价单快照；渲染各视图只读各自快照。比对视图不存，是 ②④ 结果值的实时比对（天然读快照）。

### 2.2 数据模型关联事实（核实结论）
- 报价单 `quotation`：全局绑 `customer_template_id`（报价模板，`template_kind='QUOTATION'`）+ `costing_card_template_id`（核价模板，`template_kind='COSTING'`）。
- `quotation_line_item.template_id` 指报价模板；报价/核价**共用同一份 lineItems**，按各自模板组件白名单过滤渲染。
- Excel 视图列来自独立 `costing_template.columns`（与报价/核价模板经 `linked_template_id` 关联），**不是**组件字段。
- 核价侧 V72 后也有产品卡片形态（复用 `ProductCard`，按 `costing_card_template_id` 加载组件）。

## 3. 数据模型（P2 四列分存）

`quotation_line_item` 新增 4 个 JSONB 列 + 时间戳：

| 列 | 内容 | 可变性 |
|---|---|---|
| `quote_card_snapshot` | 报价卡片：结构 + 值 | 加产品写；**用户编辑时回写** |
| `quote_excel_snapshot` | 报价 Excel：列定义 + 结果行 | 加产品写；**用户编辑时回写** |
| `costing_card_snapshot` | 核价卡片：结构 + 值 | 加产品写；**永久只读** |
| `costing_excel_snapshot` | 核价 Excel：列定义 + 结果行 | 加产品写；**永久只读** |
| `snapshot_at` | 四份冻结时间 | 加产品写 |
| `quote_snapshot_at` | 报价侧最近编辑重算时间 | 编辑回写 |

### 3.1 卡片快照 JSON 形状（①a）
```jsonc
{
  "version": 1,
  "templateId": "<uuid>",
  "templateKind": "QUOTATION | COSTING",
  "tabs": [
    {
      "componentId": "<uuid>",
      "tabName": "选配-材质",
      "sortOrder": 1,
      "componentType": "NORMAL | SUBTOTAL | ...",
      "dataDriverPath": "$cz_view",
      "fields": [
        {
          "name": "unit_price",
          "fieldType": "BASIC_DATA | DATA_SOURCE | FORMULA | INPUT_NUMBER | LIST_FORMULA | ...",
          "label": "单价",
          "sortOrder": 3,
          "isAmount": true,
          "isRequired": false,
          "editable": false,
          "defaultValue": null,
          "basicDataPath": "$cz_view[...].price",
          "datasourceBinding": { "type": "GLOBAL_VARIABLE", "global_variable_code": "..." }
        }
      ],
      "formulas": [ { "name": "总价", "expression": [ /* token 数组 */ ] } ],
      "baseRows": [ { "driverRow": { /* ... */ }, "basicDataValues": { /* 三类冻结值 */ } } ],
      "editRows": [ { "<field>": "<编辑值>" } ],          // 核价恒空
      "formulaResults": [ { "<formulaField>": "<算好的结果>" } ]
    }
  ]
}
```

### 3.2 Excel 快照 JSON 形状
```jsonc
{
  "version": 1,
  "costingTemplateId": "<uuid>",
  "columns": [ { "colKey": "...", "title": "...", "sourceType": "VARIABLE | FORMULA | EXCEL_FORMULA", "comparisonTag": "...", "hidden": false } ],
  "rows": [ { "<colKey>": "<算好的最终值>" } ]
}
```

> **零计算约束（关键）**：为保证"渲染零实时计算"，**FORMULA 结果必须落在快照里**——卡片用 `formulaResults`、Excel 用 `rows` 中已算好的 FORMULA 列值。加产品时算一次；报价侧每次编辑重算并回写；核价侧只算一次永久冻死。

## 4. 写时机一：加产品（四份一次冻结）

扩展现有 `ConfigureSnapshotService` → 抽出 `CardSnapshotService.snapshotLine(lineItem)`，对每个报价行：

1. **报价卡片**：遍历报价模板（`customer_template_id`）组件 → `ComponentDriverService.expand` 种子展开 → 组装 `structure + baseRows + 初始 formulaResults` → 写 `quote_card_snapshot`（`editRows` 空）。
2. **报价 Excel**：以报价模板 linked 的 `costing_template` 跑 `ExcelViewService.buildRowData` → 写 `quote_excel_snapshot`（列定义 + 算好的结果行）。
3. **核价卡片**：遍历核价模板（`costing_card_template_id`）组件 → 同法种子展开 → 写 `costing_card_snapshot`。
4. **核价 Excel**：以核价模板 linked 的 `costing_template` 跑 `buildRowData` → 写 `costing_excel_snapshot`。

**容错**：四步各自 `try/catch` 降级 + `REQUIRES_NEW` 事务隔离，单步失败不连坐、不阻断加产品（沿用 `方案-加产品整份快照.md` Phase 1 既有纪律）。

**组合产品（COMPOSITE）**：种子展开口径对齐现有 `ConfigureSnapshotService.snapshotLines` 的子件逐展开 + 聚合逻辑（AP-45）；含 `composite_child_*` 的组件逐子件展开拼接，聚合为空写 NULL 让该 tab 标记"未命中"。

---

## 5. 写时机二：用户编辑报价卡片（只动报价两列）

新增/扩展编辑回写端点（替代现有 autosave 写 `row_data` 的路径）：

1. 把编辑值写入 `quote_card_snapshot.tabs[].editRows`。
2. 基于 `baseRows + editRows` 重算该行所有 FORMULA → 更新 `formulaResults`。
3. 以"快照基础值 + 编辑值"为输入重算报价 Excel（`buildRowData` 等价口径）→ 回写 `quote_excel_snapshot.rows`。
4. 更新 `quote_snapshot_at`。
5. **核价两列（`costing_card_snapshot` / `costing_excel_snapshot`）物理不参与本次 UPDATE**——结构性隔离保证核价永不被编辑动到。

> FORMULA 重算口径必须与加产品时一致，对账防金额漂移。

---

## 6. 渲染读路径（零计算）

渲染取值优先级（卡片）：`editRows[field] ?? baseRows.basicDataValues[path] ??（FORMULA → formulaResults[field]）`。Excel 渲染直读 `rows[colKey]`。

前端四个入口全部改为"读对应快照列"，渲染路径不再调 `enrichComponentData`（模板快照）/ `useDriverExpansions`（实时 batch-expand）/ Excel 实时求值：

| 入口 | 文件 | 读 |
|---|---|---|
| 报价/核价卡片 | `ProductCard.tsx` | `quote_card_snapshot` / `costing_card_snapshot` |
| 详情只读卡片 | `ReadonlyProductCard.tsx` | 对应卡片快照 |
| 报价/核价 Excel | `LinkedExcelView.tsx` / `useLinkedExcelRows.ts` | `*_excel_snapshot.rows` |
| 比对视图 | `ComparisonView.tsx` | `quote_excel_snapshot.rows` vs `costing_excel_snapshot.rows` 直接比对 |

## 7. 旧链路降级（Q1）

- `ComponentDriverService.expand`、`ExcelViewService.buildRowData`、模板 `components_snapshot` enrich：**保留实现**，仅作"加产品 / 编辑重算的种子计算器"，从渲染读路径全部移除。
- `DriftDetectionService`（基础漂移提示）：随脱钩**退役**（基础变化已不影响快照，漂移检测无意义）。
- 存量测试报价单：**删除**（均为测试数据，用户确认可删）。
- **无兼容回退分支**：新链路为唯一渲染源（不保留"无快照 → 回退实时"逻辑）。

## 8. 分阶段交付（每阶段强制 E2E + 三视图截图）

- **Phase 0**：本设计文档冻结（评审通过）。
- **Phase 1（加性低风险）**：建 4 列 + `CardSnapshotService` 写四份；渲染不变；做"快照值 vs 实时值"对账。快照写入失败 try/catch 降级，绝不阻断加产品。
- **Phase 2（行为变更，强制 E2E）**：报价卡片读 `quote_card_snapshot` + 编辑回写重算（含报价 Excel 回写）；E2E `quotation-flow.spec.ts`。
- **Phase 3**：报价 Excel + 核价卡片 + 核价 Excel 读快照；E2E `composite-product-flow.spec.ts` + Excel 视图核对。
- **Phase 4**：ComparisonView 读快照 + 旧渲染链路降级为种子 + 删测试存量 + 清理 + `DriftDetectionService` 退役。

每阶段交付须附：E2E `passed` + `'加载中' final count = 0`；报价卡片 / 核价卡片 / Excel 三视图截图。

## 9. 主要风险与防护

- **核心渲染链路破坏性改动**：命中 `三大核心模块基线.md` 锁定链路 + AP-31/37/38/44/50/51 协议雷区。
- **字段类型联动协议点不减反增（AP-44）**：原 17 检查点现分裂为两端——**种子计算端**（加产品/编辑重算时要算对）+ **快照渲染端**（要读对）。任一端漏改 = 静默失败。新增/变动 `field_type` 必须双端同步 + 双 spec E2E。
- **driver 行数纪律（AP-51）**：种子展开冻 `baseRows` 时 `rowCount` 权威，禁 `Math.max(rowCount, baseRows.length)` 累加。
- **多行 / 隐式 JOIN（AP-22）、V6 表纪律（AP-53）**：种子计算端仍适用（`$<sql_view_name>` 引用、FROM V6 表、customer × material 维度）。
- **组合产品子件聚合（AP-45）**：种子展开口径对齐现有 `ConfigureSnapshotService`。
- **FORMULA 重算口径对账**：加产品 vs 编辑重算两处口径必须一致，防金额漂移。
- **存储体量上升**：每行存四份整份快照；可接受（换渲染零计算）。
