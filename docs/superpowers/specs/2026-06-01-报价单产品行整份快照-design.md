## 1. 背景与现状

### 1.1 现状（已实现的部分）
- **值快照已存在**：`方案-加产品整份快照.md`（Phase 1–4，2026-05-29）给 `quotation_line_component_data` 加了 `snapshot_rows`（JSONB），加产品时冻结每个 Tab 每行的解析值。渲染经 `ComponentDriverService.expandWithSnapshot` 命中快照直返，基础数据变化不影响已加行的值。
- **结构未冻结**：页签、字段清单、字段属性（`field_type`、默认值、是否可编辑、`basic_data_path`、`datasource_binding`、排序、公式）仍来自模板级 `template.components_snapshot`，模板重发布 / `refreshSnapshotsByComponent` 会改动已存在报价单的卡片结构。
- **Excel 视图未脱钩**：报价/核价 Excel 视图（`LinkedExcelView` / `ExcelViewService`）的列来自独立的 `costing_template.columns`（VARIABLE/FORMULA），取值仍走 BNF 路径实时查基础数据。

### 1.2 问题
卡片"值"冻了但"结构"没冻；Excel 视图整体没冻。导致模板/基础数据变化仍可能改变已存在报价单的展示，渲染未与数据源彻底脱钩。

### 1.3 目标
**报价单的四套视图（报价卡片 / 报价Excel / 核价卡片 / 核价Excel）渲染完全只读报价单自己的快照，与基础数据、模板彻底脱钩。** 结构在报价单创建时固定不变；值在加产品时冻结，草稿态报价侧可按最新基础数据重刷、保留用户编辑，提交后全部冻死。

### 1.4 与现有快照机制的关系（收口，非并存）
**重要事实（2026-06-01 架构评审核实）**：卡片渲染**当前已经读 `snapshot_rows` 快照**——`batch-expand` 端点（前端 `useDriverExpansions` 调）后端已走 `ComponentDriverService.expandWithSnapshot`（`ComponentResource.java:182/195`），命中 `quotation_line_component_data.snapshot_rows` 直返。报价/核价 Excel 视图则**仍是实时 BNF 求值**（`useLinkedExcelRows.ts` 走 `batchEvaluate`，`excel_view_snapshot` 列虽在写但渲染未读）。

因此本方案是对现有快照机制的**演进 + 收口**，不是从零新建，也**不并存双写**：

| 现有 | 新方案对应 | 收口时机 |
|---|---|---|
| `quotation_line_component_data.snapshot_rows`（行×组件×Tab 粒度，卡片值，已是渲染权威） | `quotation_line_item.quote_card_values` / `costing_card_values`（行级整份） | Phase 1 **复用同一次种子展开结果**组装新列、不重复展开；`snapshot_rows` 渲染读路径 Phase 2 切走，**Phase 4 退役** |
| `quotation_line_item.excel_view_snapshot`（已写、渲染未读） | `quote_excel_values` / `costing_excel_values` | Phase 1 写新列；Phase 3 Excel 渲染切读新列；`excel_view_snapshot` **Phase 4 退役** |
| 结构来自 `template.components_snapshot`（模板级共享） | `quotation_view_structure`（报价单级 4 份，创建即冻） | Phase 2 渲染读新结构 |

**纪律**：Phase 1 的 `buildCardValues` 必须复用 `ConfigureSnapshotService.snapshotLines` 的展开结果（不另跑一遍 expand），避免加产品耗时翻倍与两套值漂移。

### 1.5 基线条款影响清单（须 Phase 4 回写《三大核心模块基线》）
本方案把渲染权威从「模板 snapshot enrich + 实时 expand + RuntimeContext 求值」整体替换为「读行级值快照」，命中以下基线锁定条款，**适用阶段从"渲染期"前移到"种子期"（加产品 / 草稿重刷）**：

| 基线条款 | 现状（渲染期适用） | 本方案后 |
|---|---|---|
| §4.2 渲染链路 6 步（enrich→useDriverExpansions→cell 分发） | 渲染期执行 | Phase 2 起步骤 4–5（实时展开）移除，渲染只读快照 |
| §4.4 driverExpansionKey 6 维 cache | 渲染期 | 仅种子期（加产品/重刷）适用 |
| §4.5 ComponentDriverService 三分支 | 渲染期 | 降级为种子计算器 |
| §5.2 ImplicitJoinRewriter / RuntimeContext 求值 | 渲染期 | 仅种子期；渲染不再经 RuntimeContext |
| §3.2.1 组件 SQL 视图报价单层冻结 | — | 种子重刷时 `lookupSqlView` 走哪个优先级（DRAFT 实时 vs 已冻结）待 §12 细化 |

> 红线 1/2（字段渲染走配置 / 过滤条件走 RuntimeContext）**不被破坏**：种子计算阶段仍走 RuntimeContext + 三分支 + V202 视图，只是渲染期不再走。但基线文档本身必须在 Phase 4 更新，否则后人按 §4.2 改 enrich 会与脱钩后的渲染冲突。

### 2.1 四份数据模型（用户原话语义）
报价单加入一个产品时，用**报价模板**和**核价模板**分别跑一遍"查询 + 计算 + 渲染"，产出四份数据：① 报价视图·产品卡片、② 报价视图·Excel、③ 核价视图·产品卡片、④ 核价视图·Excel。四份全部冻进报价单快照；渲染各视图只读各自快照。比对视图不存，是 ②④ 结果值的实时比对（天然读快照）。

### 2.2 数据模型关联事实（核实结论）
- 报价单 `quotation`：全局绑 `customer_template_id`（报价模板，`template_kind='QUOTATION'`）+ `costing_card_template_id`（核价模板，`template_kind='COSTING'`）；状态机 `status`：`DRAFT → SUBMITTED`（`Quotation.java:64`）。
- `quotation_line_item.template_id` 指报价模板；报价/核价**共用同一份 lineItems**，按各自模板组件白名单过滤渲染。
- Excel 视图列来自独立 `costing_template.columns`（与报价/核价模板经 `linked_template_id` 关联），**不是**组件字段。
- 核价侧 V72 后也有产品卡片形态（复用 `ProductCard`，按 `costing_card_template_id` 加载组件）。

### 2.3 模板结构固定前提（关键简化）
四套结构在报价单创建/确定模板那一刻固定、整张报价单共享、之后绝不改变。由此：
- 结构无需在每个产品行重复存，**按报价单级单独存一份**。
- 草稿重刷**不触碰结构**，只重查值。
- "字段被删 / 字段类型变 → 编辑值如何对齐"的问题**不存在**（结构恒定，字段层面恒定）；草稿重刷只需处理**行层面**对齐（数据行可能因基础数据变化而增删/换身份）。

## 3. 数据模型（两层存储）

### 3.1 报价单级 · 4 份结构快照
报价单创建/确定模板时固定，永不变，整单共享。建议存于 `quotation`（或独立 `quotation_template_snapshot` 表，plan 阶段定）：

| 快照 | 内容 | 来源 |
|---|---|---|
| `quote_card_structure` | 报价卡片：tabs + 每 tab 的 fields（字段类型/属性/设置）+ formulas + `dataDriverPath` + `rowKeyFields` | 报价模板 `components_snapshot` |
| `quote_excel_structure` | 报价 Excel：columns 列定义（colKey/title/sourceType/comparisonTag/hidden） | 报价模板 linked `costing_template.columns` |
| `costing_card_structure` | 核价卡片：同报价卡片结构形状 | 核价模板 `components_snapshot` |
| `costing_excel_structure` | 核价 Excel：同报价 Excel 列结构形状 | 核价模板 linked `costing_template.columns` |

**卡片结构 JSON 形状**：
```jsonc
{
  "version": 1,
  "templateId": "<uuid>", "templateKind": "QUOTATION | COSTING",
  "tabs": [
    {
      "componentId": "<uuid>", "tabName": "选配-材质", "sortOrder": 1,
      "componentType": "NORMAL | SUBTOTAL | ...", "dataDriverPath": "$cz_view",
      "rowKeyFields": ["material_no"],                 // 行键（见 §5.1）
      "fields": [
        { "name": "unit_price", "fieldType": "BASIC_DATA | DATA_SOURCE | FORMULA | INPUT_NUMBER | LIST_FORMULA | ...",
          "label": "单价", "sortOrder": 3, "isAmount": true, "isRequired": false,
          "editable": false, "defaultValue": null,
          "basicDataPath": "$cz_view[...].price",
          "datasourceBinding": { "type": "GLOBAL_VARIABLE", "global_variable_code": "..." } }
      ],
      "formulas": [ { "name": "总价", "expression": [ /* token 数组 */ ] } ]
    }
  ]
}
```

### 3.2 产品行级 · 4 份值快照
`quotation_line_item` 新增列（P2 物理分开）：

| 列 | 内容 | 可变性 |
|---|---|---|
| `quote_card_values` | 报价卡片值：`tabs[].{baseRows, editRows, formulaResults}` | 加产品写；**草稿态打开重刷**；**用户编辑回写** |
| `quote_excel_values` | 报价 Excel：`rows[]`（算好的最终列值） | 加产品写；草稿重刷/编辑回写 |
| `costing_card_values` | 核价卡片值：`tabs[].{baseRows, formulaResults}`（editRows 恒空） | 加产品写；**永久只读** |
| `costing_excel_values` | 核价 Excel：`rows[]` | 加产品写；**永久只读** |
| `snapshot_at` | 加产品冻结时间 | 加产品写 |
| `quote_values_at` | 报价侧最近重刷/编辑时间 | 草稿重刷/编辑回写 |

**卡片值 JSON 形状（按 tab）**：
```jsonc
{
  "tabs": [
    { "componentId": "<uuid>", "tabName": "选配-材质",
      "baseRows": [ { "driverRow": { "material_no": "8851" }, "basicDataValues": { /* 三类冻结值 */ } } ],
      "editRows": [ { "rowKey": "8851", "values": { "<field>": "<编辑值>" } } ],   // 核价恒空
      "formulaResults": [ { "rowKey": "8851", "values": { "<formulaField>": "<算好的结果>" } } ] }
  ]
}
```
> `editRows`/`formulaResults` 以 `rowKey`（按 `rowKeyFields` 拼出）索引，便于草稿重刷后按行键对齐（见 §5）。

**Excel 值 JSON 形状**：`{ "rows": [ { "<colKey>": "<算好的最终值>" } ] }`。

### 3.3 零计算约束（关键）
为保证"渲染零实时计算"，**FORMULA 结果必须落快照**——卡片用 `formulaResults`、Excel 用 `rows` 中已算好的 FORMULA 列值。加产品算一次；报价侧草稿重刷 / 编辑时重算并回写；核价侧只算一次永久冻死。渲染取值：`editRows ?? baseRows.basicDataValues ??（FORMULA → formulaResults）`，Excel 直读 `rows[colKey]`。

## 4. 写时机一：加产品（固定结构 + 四份初始值）

加产品（扩展现有 `ConfigureSnapshotService` → 抽出 `CardSnapshotService`）：

1. **固定四份结构**（若该报价单尚未固定）：从报价/核价模板 `components_snapshot` 与各自 linked `costing_template.columns` 组装四份结构快照，写报价单级（含每个组件的 `rowKeyFields`）。
2. **算四份初始值**（产品行级），对每个报价行：
   - 报价卡片：遍历报价模板组件 → `ComponentDriverService.expand` 种子展开 → `baseRows` + 初始 `formulaResults`（`editRows` 空）→ 写 `quote_card_values`。
   - 报价 Excel：`ExcelViewService.buildRowData`（报价模板 linked costing_template）→ 写 `quote_excel_values`。
   - 核价卡片：遍历核价模板组件 → 同法 → 写 `costing_card_values`。
   - 核价 Excel：核价模板 linked costing_template → 写 `costing_excel_values`。

**容错**：各步 `try/catch` 降级 + `REQUIRES_NEW` 事务隔离，单步失败不连坐、不阻断加产品。
**组合产品（COMPOSITE）**：种子展开口径对齐现有 `ConfigureSnapshotService` 的子件逐展开 + 聚合逻辑（AP-45）。

## 5. 写时机二：草稿态打开重刷（只刷报价侧值）

**触发**：报价单 `status="DRAFT"` 且打开编辑向导（进入 Step2 编辑态）时，对全部报价行执行。
**范围**：只重刷报价侧两份值（`quote_card_values` / `quote_excel_values`）；**结构不动、核价两份不动**。

流程（每报价行）：
1. **重查基础值**：按报价模板各组件 SQL 视图（`ComponentDriverService.expand` 种子）实时查最新产品数据 → 新 `baseRows`。
2. **对齐保留编辑**：用 `rowKeyFields` 为新 `baseRows` 每行算 `rowKey`，把旧 `editRows` 按 `rowKey` 叠加回对应行；新数据里不存在的 `rowKey`（行没了）→ 丢弃该行编辑。
3. **重算公式**：基于（新 baseRows + 保留的 editRows）重算 → 新 `formulaResults`。
4. **重算报价 Excel**：以"快照值 + 编辑值"为输入重算 → 回写 `quote_excel_values.rows`。
5. 更新 `quote_values_at`。
6. **降级纪律**：重刷任一步失败 → 保留该行上一次的报价值快照，不报错、不阻断打开（与加产品同等降级）。

### 5.1 行键机制（`rowKeyFields`）
- 组件管理新增组件级配置 `rowKeyFields`（一个/多个字段名，作为该组件 driver 行的业务标识）。
- `rowKey` = 按 `rowKeyFields` 从 `driverRow` 取值拼接（稳定标识，**不含会随重排变化的纯顺序列**如 `seq_no`/`序号`）。
- **判定口径（已定，不留 plan）**：组件 `dataDriverPath` 非空（driver 驱动可能多行）**且** `fields` 含至少一个可编辑字段（`field_type` ∈ 可录入集合）→ 需声明 `rowKeyFields`，其中每个名字须存在于 `fields[].name`。无 `dataDriverPath`（单行/固定）或无可编辑字段（纯只读）→ 豁免。
- **复合键**：组合产品同一元素/工序会跨子件重复出现（如子件1、子件2 都有 Ag），单字段不唯一 → 必须用复合键（如 `["子件","元素"]` / `["子件","工序代码"]`）。校验对复合键唯一性给**告警**（不硬拦），唯一性责任在配置者。
- **无稳定业务键的豁免**：纯按行号排的清单类组件（driverRow 仅 `seq_no`、无任何稳定业务列）允许显式豁免——置 `rowKeyFields = ["__seq_no__"]`（哨兵）表示"按行号对齐、自担重排错位风险"，绕过强制校验。
- **校验上线策略（避免回归）**：
  - **新建组件**：硬拦（不声明则保存失败）。
  - **存量组件**：软校验（保存时仅告警、不阻断），并由 Task 1 的 Flyway data migration **预填**已知核心组件行键：`选配-元素含量=["子件","元素"]`、`选配-工序列表=["子件","工序代码"]`、材质组件=`["子件","材质代码"]`（具体字段名以各组件 `fields` 实际 name 为准，迁移前核对）。
- 行键写入报价单级结构快照（§3.1），草稿重刷读它做对齐。

## 6. 写时机三：用户编辑报价卡片（回写报价侧值）

草稿态用户编辑可编辑字段（替代现有 autosave 写 row_data）：
1. 编辑值写入 `quote_card_values.tabs[].editRows`（按 `rowKey` 索引）。
2. 基于 baseRows + editRows 重算 FORMULA → 更新 `formulaResults`。
3. 以"快照值 + 编辑值"重算报价 Excel → 回写 `quote_excel_values.rows`。
4. 更新 `quote_values_at`。
5. **核价两列物理不参与本次 UPDATE**——结构性隔离保证核价永不被编辑动到。

> FORMULA 重算口径在加产品 / 草稿重刷 / 编辑三处必须一致，对账防金额漂移。

## 7. 渲染读路径（零计算）

## 11. 主要风险与防护

- **核心渲染链路破坏性改动**：命中 `三大核心模块基线.md` 锁定链路 + AP-31/37/38/44/50/51 协议雷区。
- **字段类型联动协议点（AP-44）**：种子计算端（加产品/重刷/编辑要算对）+ 快照渲染端（要读对）双端同步；新增/变动 `field_type` 必须双端改 + 双 spec E2E。结构固定降低了"运行期字段漂移"风险，但 `rowKeyFields` 是组件配置新增项，需进字段配置指南的联动检查。
- **行键对齐错位（AP-54 同类风险）**：草稿重刷按 `rowKey` 对齐，禁止用过滤后下标 / 行号兜底替代业务键；`rowKey` 取值列必须稳定（排除纯顺序号）。
- **driver 行数纪律（AP-51）**：种子展开冻 `baseRows` 时 `rowCount` 权威，禁 `Math.max(rowCount, baseRows.length)` 累加。
- **DATA_SOURCE 不退化（AP-39/AP-52）**：`buildCardStructure` 从 `components_snapshot` 搬运时，DATA_SOURCE 字段的 `datasource_binding` 必须完整带过来（否则种子重算单价退化为 0）；种子重算全局变量时 `key_field_refs` 动态 key 重写口径须与现有渲染一致，防报价/核价单价漂移。结构快照测试须断言 DATA_SOURCE 字段 binding 非空。
- **多行 / 隐式 JOIN（AP-22）、V6 表纪律（AP-53）**：种子计算端仍适用。
- **组合产品子件聚合（AP-45）**：种子展开口径对齐现有 `ConfigureSnapshotService`。
- **FORMULA 重算口径对账**：加产品 / 草稿重刷 / 编辑三处口径一致，防金额漂移。
- **草稿重刷性能**：每次打开草稿对全部报价行重跑报价侧种子查询，需评估批量展开 + 缓存 evict 时机。
- **存储体量**：结构报价单级共享（省）；产品行级四份值（增）。净可接受。
