## 0. 为什么分期（评审 + 事实核验结论）

v1/v2 的核心机制"财务切料号版本 → 重算当前料号的**子料号**与数据"在当前代码**不现成**，已穿透核验：

- **BOM 闭包不吃版本**：`BomClosureService.CLOSURE_SQL` 硬编码 `is_current = true`（`BomClosureService.java:71, 88`），`compute(root, versionOverrides)` 在 P1 **显式忽略** `versionOverrides`（`:120-124`，注释"P1 不支持版本覆盖…P2 走 Java 逐层迭代展开"）→ **子料号集合不随版本变**。
- **核价卡片 expand 传 `partVersion=null`**（`CardSnapshotService.java:1631-1632`），到 `ComponentDriverService.expand:402` `PartVersionContext.set(null)` 清空上下文 → 核价卡片值不按版本算。
- 仅核价 **Excel 树**部分单元格值随 `li.partVersionLocked` 变（版本化表谓词），但树结构（哪些子料号）仍来自闭包、不变。

**结论**：切版本重算 = 依赖"版本感知 BOM 闭包展开（P2）"这块独立底层工程，**推到第二期**。第一期只做不依赖它、且立即有价值的部分。

**分期一利好**：第一期不切版本 → 核价数据 == 报价单提交快照、**永不分叉** → 无需"核价单独立快照 / 覆盖注入 / 单据总价覆盖 / 逐行 revision"这套（v2 的 B-1/B-2/B-3 整块）。第一期大幅瘦身。

## 1. 第一期范围（本 spec 交付）

**核价管理 = 财务审批节点 + 列表 + 只读核价工作台 + 通过/驳回/撤回**。

- 报价单提交审批 → 进入财务**角色队列**（任一财务/管理员处理）；财务在「核价管理」看到列表。
- 财务点开 → **核价工作台（只读）** 复用详情页"报价/核价/比对 × 卡片/Excel"视图复核数据；顶部操作栏 **核价通过 / 驳回（填原因）**；列表页亦可通过/批量驳回。
- 核价通过 → 报价单 `已审核(APPROVED)`；驳回 → `核价驳回(COSTING_REJECTED)`，销售可改单重提；销售/管理员可**撤回**到草稿。
- **第一期财务不调价**（无切版本）；调价（切版本重算）= 第二期（§9）。
- 角色 `PRICING_MANAGER` 显示名改"财务"；旧手动 CostingSummary 下线。

## 4. 数据模型（第一期，精简）

```sql
-- 核价单（精简）：每报价单 1 行，提交审批时自动建；锚定核价流程元数据 + 第二期扩展点
CREATE TABLE costing_order (
    id            UUID PRIMARY KEY,
    quotation_id  UUID NOT NULL REFERENCES quotation(id),
    submitted_by  UUID,                              -- 提交审批的用户(列表"提交人"; submit 现有 userId 可取)
    entered_costing_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (quotation_id)
);
-- 第二期再加 costing_order_revision / costing_order_line_snapshot（切版本快照）。
```
- 核价状态 = `quotation.status` 派生（SUBMITTED→待核价 / COSTING_REJECTED→核价驳回 / APPROVED+(SENT/ACCEPTED)→核价通过）；不在 costing_order 另存，单一真相源。
- 迁移（B-5）：`ALTER` `quotation.status` CHECK 加 `COSTING_REJECTED`（顺补 `CANCELLED`）；`quotation_approval.action` CHECK 加 `COSTING_APPROVED/COSTING_REJECTED`。

## 5. 审批状态流

```
DRAFT ──submit()──> SUBMITTED(待核价)  [建 costing_order; assignedApproverId 照旧写,核价不依赖它]
   (角色队列: 任一 财务/管理员)
   ├─ costingApprove()       ─> APPROVED(核价通过/已审核)
   └─ costingReject(reason)  ─> COSTING_REJECTED(核价驳回) ──销售改单 resubmit()──> SUBMITTED
{SUBMITTED|COSTING_REJECTED|APPROVED} ──withdraw()(销售/管理员)──> DRAFT  [解冻; costing_order 保留]
```

- `costingApprove(id, comment?)`（新，role-based）：校验 status=SUBMITTED + role∈{PRICING_MANAGER,SYSTEM_ADMIN} → APPROVED + `QuotationApproval(action=COSTING_APPROVED)`。
- `costingReject(id, reason)`（新）：同上 + reason 必填 → COSTING_REJECTED + `QuotationApproval(action=COSTING_REJECTED, comment=reason)`。
- **不改** `approve()/reject()`（保留供未来"流程审批"）。**不改** `submit` 的 routeApprover/assignedApproverId（避 B-4）。

## 6. 后端设计（第一期）

1. **迁移**（§4）：两个 CHECK 约束 ALTER。
2. **`costing_order` 实体/Repo** + `submit()` 末尾建一行（`submitted_by=当前 userId`）。幂等：已存在则不重建（重提复用）。

## 7. 撤回（M-2 处置，第一期）

- **范围**：`{待核价/核价驳回/核价通过(APPROVED)}` → DRAFT。**第一期排除 SENT/ACCEPTED**（已对客户发出 / 已影响 `customer.accumulated_amount`，回退逻辑独立、风险高，放第二期或单列子项）。
- **解冻步骤**（统一封装，回 DRAFT 必走）：清 `submission_snapshot`、清 `quotation_component_sql_snapshot`(SQL 视图闭包)、`total_amount` 标记待重算（再编辑后重算），建 `QuotationApproval(action=WITHDRAWN)`；`costing_order` 保留。
- **机制统一（已定）**：**废弃两步撤回** `QuotationWithdrawService`（request→approve/reject，`:35/56/68/101`）及其前端 `WithdrawSection.tsx`、相关端点；**统一为销售一步直接撤回**（扩展 `QuotationService.withdraw:1304` 从"仅 SUBMITTED"放宽到 `{SUBMITTED/COSTING_REJECTED/APPROVED}` + 加解冻步骤）。退役 `QuotationWithdrawService` 时核验无其它依赖（两步审批表/状态若有需停用，参照 CostingSummary 休眠做法）。
- 权限：该报价单创建人销售 / SYSTEM_ADMIN。
