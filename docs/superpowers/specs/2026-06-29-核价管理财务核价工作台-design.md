## 1. 背景与目标

把现有「核价管理」菜单（原"手动料号级核价单" CostingSummary，**就此废弃**）改造为**财务核价工作台**：报价单审批流驱动、数据自动同步、财务在此核价。

**业务流**：
```
报价单  提交审批(待核价=SUBMITTED)
   └→ 核价审批(财务：切料号版本重算 → 核价通过 / 驳回)
        ├ 核价通过 → 已审核(APPROVED)        〔流程审批·后期接流程引擎，此处先直达〕
        └ 驳回(填原因) → 核价驳回(COSTING_REJECTED) → 销售改单重提 → 回 待核价
```

**核心语义**：报价单一经提交，其数据为冻结快照；财务**不改销售原始快照**，而是**另存一份「核价单」快照**（切料号版本重算所得），且**核价单最新快照在所有读核价数据处覆盖报价单自身核价快照**（详情显示 / 金额 / 对外 PDF·Excel·邮件输出）。保留销售原版用于版本对比 / 流程记录。

**财务唯一可改的操作 = 切换某料号的版本号**，系统据此重算该产品卡片的子料号与核价数据；不可手改折扣/单价/字段、不可增删产品。

## 4. 数据模型

### 4.1 新增三表（Flyway V{N}）

```sql
-- 核价单：每报价单 1 份，提交审批时自动建
CREATE TABLE costing_order (
    id              UUID PRIMARY KEY,
    quotation_id    UUID NOT NULL REFERENCES quotation(id),
    latest_revision_id UUID,                       -- 指向当前生效 revision
    submitted_by    UUID,                          -- 提交审批的销售(列表"提交人"字段; 见 §7.1)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (quotation_id)
);
-- 注:核价状态(待核价/核价驳回/核价通过)= 报价单状态的派生(SUBMITTED/COSTING_REJECTED/APPROVED),
--    单一真相源在 quotation.status，costing_order **不**另存 status，列表 §6.3 join quotation 取，避免漂移。

-- 核价单修改记录：财务每次保存(切版本)追加一条；revision#0 = 提交时种子
CREATE TABLE costing_order_revision (
    id              UUID PRIMARY KEY,
    costing_order_id UUID NOT NULL REFERENCES costing_order(id),
    seq             INT NOT NULL,                  -- 0,1,2... 单调递增
    change_summary  JSONB,                         -- {lineItemId, hfPartNo, versionFrom, versionTo} 列表; revision#0 为 null/初始标记
    created_by      UUID,                          -- 该 revision 由谁产生(财务); revision#0 = 系统
    created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
    UNIQUE (costing_order_id, seq)
);

-- 核价单行快照：每 revision × 行，存该行核价侧冻结值 + 所用版本
CREATE TABLE costing_order_line_snapshot (
    id              UUID PRIMARY KEY,
    revision_id     UUID NOT NULL REFERENCES costing_order_revision(id),
    line_item_id    UUID NOT NULL,                 -- → quotation_line_item.id
    part_version_locked INT,                        -- 该行此 revision 采用的料号版本
    costing_card_values  JSONB,                     -- 与 QuotationLineItem.costing_card_values 同形
    costing_excel_values JSONB,                     -- 与 QuotationLineItem.costing_excel_values 同形(BOM 树)
    UNIQUE (revision_id, line_item_id)
);
CREATE INDEX idx_cols_revision ON costing_order_line_snapshot(revision_id);
```

> 不直接删旧 `costing_summary*`（FK 约束 + 历史数据）；本表族与之无关。

### 4.2 覆盖注入（D2）
`loadLineItems(quotationId)`（`QuotationService.java:2345`）装配时：
1. 取该报价单 `costing_order.latest_revision_id`；
2. 一次 `IN` 批查该 revision 的 `costing_order_line_snapshot` → `Map<lineItemId, {costingCardValues, costingExcelValues}>`；
3. `LineItemDTO.from(li)` 后，若 map 含该行 → 覆盖 `dto.costingCardValues/costingExcelValues`（**逐行**；无则保留报价单自身核价快照）。

效果：详情页核价视图、金额、对外输出**自动以核价单最新 revision 为准**；前端、`ReadonlyProductCard` 读取逻辑零改。`costingCardStructure` 不覆盖（结构不变）。

## 5. 审批状态流

新增状态：`COSTING_REJECTED`（核价驳回）。补 `VALID_QUOTATION_STATUSES`（`:92-94`）+ 前端 `statusConfig`（`:34-42`，标签"核价驳回"/红）。

```
DRAFT ──submit()──> SUBMITTED(待核价)
                       │  (角色队列: 任一 财务/管理员 可操作)
                       ├─ costingApprove() ─> APPROVED(核价通过/已审核)
                       └─ costingReject(reason) ─> COSTING_REJECTED(核价驳回)
COSTING_REJECTED ──销售改单 resubmit()──> SUBMITTED(待核价)   // 见 §6.4 重提
```

- `submit()`：路由不再走 `routeApprover`（指派单人），改为**进入财务角色队列**（不写 assignedApproverId，或写 null）；并**自动建 `costing_order` + revision#0**（§6.1）。记录 `submitted_by`。
- `costingApprove()`（新）：校验 status=SUBMITTED + 角色∈{PRICING_MANAGER,SYSTEM_ADMIN} → APPROVED；建 `QuotationApproval(action=COSTING_APPROVED, approverId=当前财务, comment 可选)`；**锁定核价单**（Q5）。
- `costingReject(reason)`（新）：校验同上 + reason 必填 → COSTING_REJECTED；建 `QuotationApproval(action=COSTING_REJECTED, comment=reason)`。
- 审批历史 `QuotationApproval` 无需扩列（action 为 String，新增 `COSTING_APPROVED/COSTING_REJECTED` 枚举值即可）。

## 6. 后端设计

### 6.1 提交时自动建核价单
`submit()`（`:782-906`）末尾（快照冻结后）：建 `costing_order(quotation_id, submitted_by=当前用户)` + `costing_order_revision(seq=0)` + 为每个非 PART 行写 `costing_order_line_snapshot`（**值 = 该行当前 `costing_card_values/costing_excel_values` 拷贝**，`part_version_locked = li.partVersionLocked`）；`latest_revision_id` 指向 revision#0。幂等：同 quotation 已有 costing_order 则跳过建、按 §6.4 处理重提。

### 6.2 核价专用切版本端点（D3）
`PUT /api/cpq/quotations/{id}/costing-order/line-items/{lineItemId}/part-version`，body `{version}`，`@RoleAllowed({"PRICING_MANAGER","SYSTEM_ADMIN"})`：
1. 校验报价单 status=SUBMITTED（待核价）；校验 version 在 `mat_part_version_log` 存在。
2. **只重算核价侧该行**：`PartVersionContext.set(version)` → `CardSnapshotService.buildCostingCardValues(li,...)` + `buildExcelValues(...,costingTree=true)` → 得该行新核价 card/excel 值（复用 §3.2 既有方法，串行，**勿并行**：expand 层非线程安全）。
3. **追加 revision**：新建 `costing_order_revision(seq=last+1, change_summary={lineItemId,hf,versionFrom,versionTo}, created_by=财务)`；为本单**所有非 PART 行**写 `costing_order_line_snapshot`——改动行用新值/新 version，其余行**沿用上一个 revision 的快照**（保证"最新 revision"自洽含全行）。`latest_revision_id` 指向新 revision。
4. **不动** `quotation_line_item.part_version_locked`、不动报价单自身 `costing_*_values`、不调 `regenerateAllSnapshots`（那是 DRAFT 全单重算路径）。
5. 返回该行新核价快照（工作台就地重渲染）。

> 与现有 `updateLineItemPartVersion`（`:2563`，DRAFT 限定 + 全单重算 + 覆盖报价单快照）并存、互不影响。

### 6.3 列表数据端点
`GET /api/cpq/costing-orders?status=&sort=`，`@RoleAllowed({"PRICING_MANAGER","SYSTEM_ADMIN"})`：`costing_order` join `quotation` 取投影 `{quotationId, quotationNumber, customerName, baseCurrency, submittedByName, status, createdAt}`，其中 **status 由 `quotation.status` 派生映射**（SUBMITTED→待核价 / COSTING_REJECTED→核价驳回 / APPROVED→核价通过；其余报价单状态不在核价管理列出）；支持按 status 过滤 + 排序。无创建/删除端点。

### 6.4 重提（D6）
`COSTING_REJECTED` 的报价单可被销售改单并 `resubmit()`（复用 submit 路径，允许从 COSTING_REJECTED→SUBMITTED）：重算报价单自身快照后，**追加新 revision，逐行种子 = 上一 latest_revision 的行快照**（延续财务结果）；该 revision 的行集以**当前报价单行**为准——老行（line_item_id 命中上一 revision）延续其核价快照、新增行回退到报价单最新销售核价快照、被删行丢弃。`latest_revision_id` 指向新 revision。
> 残留（实现前一句话敲定）：销售若改了销售侧内容，命中老行是否需用新销售结构重算——当前定义 = 延续上一 revision 行快照（不重算），仅新增行取销售快照。

### 6.5 角色/权限
- 现有 `approve()/reject()`（`:1239/1271`）保留供"流程审批"未来用；本期审批走新的 `costingApprove/costingReject`（角色队列校验，**不**校验 assignedApproverId）。
- `submit()` 的 `routeApprover` 改为不指派单人（队列模型）。

## 9. 性能与风险

- **切版本重算**：仅核价侧 + 改动行（非 `regenerateAllSnapshots` 全单两侧），范围小；**串行**（expand 层非线程安全，勿并行；记忆 `cpq-expand-layer-not-threadsafe`）。大单（77 行）下单行核价重算可接受。
- **覆盖查询**：`loadLineItems` 多 1 次 `IN` 批查（latest revision 行快照），与既有 `cdByLine` 同款，影响小。
- **AP-50 协议级**：核价工作台复用 `ReadonlyProductCard` 渲染分支 → 改动触发**强制双 spec E2E**（`quotation-flow.spec.ts` SIMPLE + `composite-product-flow.spec.ts` COMPOSITE）。
- **AP-51 行数纪律**：核价 card 行数以 driver/快照为权威，勿 Math.max 累加。
- **审批节点重定向**：现有 approve/reject（指派模型）保留但本期不用；切到角色队列，注意前端按钮显隐由"角色 + 状态"判定（不再判 assignedApproverId）。
