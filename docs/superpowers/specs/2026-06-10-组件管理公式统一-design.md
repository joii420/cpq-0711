## 1. 背景与目标

Excel 模板的「页签连表公式」配置页与公式配置功能已完成。本次做一次收敛重构，把**所有公式配置能力集中到组件管理**，并统一三类组件的配置体验。

**核心目标**

1. Excel 视图列配置**归属迁移**到组件管理（EXCEL 组件），模板配置页 Excel 视图改为"引用 EXCEL 组件即用"。
2. 三类组件（页签 / EXCEL / 小计）公式配置**统一复用编辑器** `TabJoinFormulaDrawer`（页签连表公式抽屉）。
3. 组件管理列表风格与模板管理对齐，配置风格统一。
4. 清理：移除冗余的右侧 `FieldPanel`；路径来源收敛为 SQL 视图；隐藏/合并若干配置项。

> **重要约束**：本次重构**不改动**「三大核心模块基线」里已 2026-05-21 终态锁定的**组件公式 `FormulaToken[]` 求值器与报价/核价/详情渲染读取链**。统一是在**编辑器（UX）层**，不是存储模型层（见 §3.2 决策 B）。

## 3. 架构与数据模型改动

### 3.1 Excel 配置归属迁移（点 1）——核心工作包之一

**现状**（评审核实）：
- Excel 视图列配置 = `Template.excelViewConfig`(JSONB)（`template/entity/Template.java:70-71`），经 `ExcelViewService`（`ExcelViewService.java:55,64`）读写。
- 前端 `ExcelViewConfigTab.tsx` 在模板配置里编辑；列模型 = `{col_key, title, source_type, hidden, formula, ...}`（`ExcelViewConfigTab.tsx:25-44`）。
- 注意：`CustomerExcelTemplate`（导入用）与本次无关，不动。

> ⚠️ **关键语义分叉（评审 🔴-1）**：`excelViewConfig` 这**一个 JSONB 同时承载两类数据** —— ①Excel 列定义数组；②`import_settings`（导入解析配置）。导入链读的是后者（`ImportExecutionService.java:374-383`）。归属迁移到 EXCEL 组件时，**必须先定调**：导入链是跟着迁到读组件，还是显式声明"导入仍读 template"。不定清楚会连带搞坏导入功能。

**改造**：
1. **新增组件类型** `component_type = 'EXCEL'`（详见 §3.4 传播清单）。
2. **列定义新存储位置**：Excel 列模型字段集与 `component.fields`(`FieldItem`) 不一致，**不强塞 fields**；在 EXCEL 组件上新增 jsonb 列（如 `excel_columns`）承载 `{col_key,title,source_type,hidden,formula,sort}`。`component.code` 有 unique 约束（`Component.java:25`），迁移时按规则生成 code（缺失/非 ASCII 时兜底生成，呼应决策 A 别名规则）。
3. **`excel_columns` 协议传播**（评审 🟡-3）：`ComponentExportBundle.Item` 当前只导 `fields/formulas/sqlViews`（`ComponentExportBundle.java:33`），**会丢 excel_columns** → 导入导出丢列。必须把 `excel_columns` 纳入 export bundle 进出 + snapshot 协议。
4. **隐藏列语义**：`hidden=true` 列仍参与公式计算、但不出现在 Excel 视图/导出/回写快照。落点在**渲染、导出、单元格回写快照**三侧（`updateExcelViewCell` 也重建快照）。
5. **数据迁移**：历史 `excelViewConfig` 均为测试数据，**重建即可（不保数）**；但"读取源切换"的代码改造**不可省**，否则 Excel 视图渲染空/导出缺列/导入断。

**读取源全量改造清单**（评审 🔴-1，`grep -rn excelViewConfig` 实测，迁移时逐处标动作）：

> **导入链定调结论（阶段 0 Task 0.1，2026-06-10 核实定调）**
>
> 现状核实（当前行号，非陈旧引用）：`ImportExecutionService` 同一个 `template.excelViewConfig`(JSONB) 既承载导入解析配置又承载列定义——
> - `:379` / `:548`：`getMap(config, "import_settings")` 读**导入解析配置**（sheet / header 行 / 行偏移 / part_no 列等）。
> - `:123` / `:380` / `:544`：`getList(config, "columns")`（含 `:383` / `:546` 老格式裸数组 fallback）读**列定义**。
> - `:687`：`record.configSnapshot = template.excelViewConfig`（整块快照留痕）。
>
> **定调**：`import_settings` 是**模板-导入绑定**（描述"这张 Excel 文件怎么解析进系统"），**语义不属组件**，强行迁组件会牵动 `ImportExecutionService` / `ImportMappingTemplateService` 4+ 处且语义错位 → **`import_settings` 留在 `template.excelViewConfig`，不迁**。
>
> **故阶段 3 的改造范围**：仅把**列定义读取源**（`:123` / `:380` / `:544` 的 `columns`，及其它读取点）切到"引用的 EXCEL 组件 `excel_columns`"；**不动**导入链对 `import_settings` 的读取（`:379` / `:548`）与 `configSnapshot` 留痕（`:687`）。迁移后导入链对某模板执行解析时 = 读 template 的 `import_settings` + 读其引用 EXCEL 组件的 `excel_columns` 列定义（跨读，列定义不再从 template 自身的 `columns` 取）。
>
> 验收锚点：列定义迁移后**导入功能不回归**（预览/执行/configSnapshot 三条路径的 `import_settings` 读取行为不变）。

### 3.2 公式模型——决策 B（抽屉只做编辑器，产出分类型落库）

**两套现状模型**（评审核实，不兼容）：
- 组件公式：`FormulaItem { expression: FormulaToken[] }`（`component/types.ts:119`）→ 存 `component.formulas` → 后端 `FormulaCalculationService` 按 token 求值 → 驱动报价/核价/详情渲染（**已终态锁定**）。
  - 关键事实：该 token 模型**已支持跨页签引用 + 聚合**（`cross_tab_ref` token 含 `agg/match/source`，由原 `CrossTabRefDrawer` 产出）。
- Excel 列公式：`{source_type:'TAB_JOIN_FORMULA', expression: string, tabs:[]}`（`TabJoinFormulaDrawer.tsx:84`）→ 存 `excelViewConfig` → tab-join 求值器。

**决策 B**：`TabJoinFormulaDrawer` 作为**共享编辑器 UI**；保存时按组件类型序列化到各自存储：
- 编辑**页签/小计组件**公式 → 序列化为现有**组件公式表示**（落 `component.formulas`，复用已锁定的 token 求值器，渲染链一行不动）。
- 编辑 **EXCEL 列**公式 → 维持**字符串表达式**（落 `excel_columns`，复用 tab-join 求值器）。

**默认分支 = (甲) 按类型分叉**（评审三轮后定调）：
- **EXCEL 组件公式** → 维持 tab-join 字符串（`excel_columns`），保留 N 元对称全外连的完整能力（这本就是"整表"该干的）。
- **页签/小计组件公式** → 落 `component.formulas` token，表达力**限定在 token 可表达范围**（同行字段 + 单源跨页签 `cross_tab_ref`）；保存前做一次**「可映射校验」**，不可映射则提示"此类复杂多表连表请用 Excel 组件"。
- 共享的是**抽屉 UI（编辑体验统一）**，不是存储/求值器。前后端渲染链（`computeAllFormulas` / `FormulaCalculator` token 求值）**一行不动**。

**为什么 (甲) 而非"完整 N 元能力下放页签"（乙）**：
- **页签/小计公式天然锚定宿主行**：当公式挂在某页签组件上，该组件自己的字段是**同行普通 field token**（非跨页签），外部引用都是**单源** `cross_tab_ref`（含 agg/match/targetExpr，`FormulaCalculator.evalCrossTab` 实测可求值）。→ 绝大多数页签/小计公式**可映射**。
- **真正不可映射**的只有"2+ 个**外部**表当裸明细、彼此 row-key 对齐、却不锚定本组件"——这语义上就是 Excel 整表的活（`TabJoinPlanEvaluator.alignByRowKey` N 元全外连），不该塞进页签字段公式。
- **(乙) 要把 tab-join 求值移植进前后端两套渲染链**（实测 `TabJoinPlanEvaluator` 唯一调用方是 `ExcelViewService`，渲染链没有字符串求值器），正撞已锁定核心 + AP-40~43/AP-50/AP-51 高发区，为边际需求重写最脆弱核心，不取。

**第 0 阶段门禁（保留为验证而非分叉判据）**：实现首阶段先用一组真实页签/小计公式样本，验证「可映射校验 + token 序列化桥」覆盖到位（顶层 +/- 拆项、单源 agg、targetExpr 子表达式、行键）。若发现某类**常见**页签公式无法映射且确属刚需，再单独走 architect 评估是否局部上 (乙)——但不作为默认路径。

**明确排除（A 方案）**：不把所有 `component.formulas` 迁成字符串、不替换 token 求值器、不改渲染读取链——风险面过大且违反基线。

### 3.4 新增 `component_type = EXCEL` 传播清单（独立于 AP-44 的 field_type 矩阵）

现状渲染/求值层是 **二元 NORMAL / SUBTOTAL** 假设（`componentType !== 'SUBTOTAL'`），`componentType` 影响面 codegraph 报 ~80 符号。加 EXCEL 必须显式三态化，至少覆盖（评审 🟡-2 补全后端读点）：

**🔴 DB 层（plan 第一批任务，评审 🔴-A）**：现有 `V22__add_component_type.sql:3` 有 `CHECK (component_type IN ('NORMAL','SUBTOTAL'))`，到 V297 无替换。**只加应用层校验不够** —— `INSERT component_type='EXCEL'` 会直接 `check_violation`。必须新增 Flyway（如 V298）：`ALTER TABLE component DROP CONSTRAINT chk_component_type; ... ADD CONSTRAINT chk_component_type CHECK (component_type IN ('NORMAL','SUBTOTAL','EXCEL'))`；同迁移加 `excel_columns jsonb` 列。

**前端**：
1. 联合类型 `ComponentType`（`component/types.ts:12`）加 `'EXCEL'`。
2. 所有 `=== 'SUBTOTAL'` / `!== 'SUBTOTAL'` 二分改**显式三态**（`QuotationStep2.tsx:762/908/960/1482`、`enrichComponentData.ts:279` 默认值、**`ComponentPalette.tsx:96-97`**（评审 🟡-C，现 EXCEL 会被并进"页签组件"桶））——避免 EXCEL 被当 NORMAL 去 driver-expand/求公式/出小计。
3. **详情/只读视图** `ReadonlyProductCard` / `ComponentCell` 加 EXCEL 分支（**AP-50 single-source 反模式**：不加分支会复用 NORMAL 渲染出僵尸数据）。
4. 组件管理详情按三态渲染对应形态（已在原型体现）。

**后端**：
5. `ComponentService.validateRequest`（`:374`，当前**不校验** componentType）加枚举校验。
6. `ComponentDriverService`（明确 EXCEL 是否跳过 driver-expand）、`CardSnapshotService.assembleTabsWithFormulaResults`、`SnapshotCollectorService`、`refreshSnapshotsByComponent`、`ComponentExportBundle`（`:33`，含 §3.1 的 `excel_columns`）逐处识别 EXCEL。

## 4. UI 设计（详见原型）

### 4.1 组件管理页（Master-Detail）

- 左栏：搜索/新建 + 批量动作 + 手风琴目录 → 三段固定分类（页签/EXCEL/小计）→ 纵向渐变卡片（左侧勾选框）。
- 右栏：选中组件内嵌详情（非抽屉）；空态提示。
- **列表规范合规**（点 3 修订）：卡片多选 + 批量动作（发布/停用/删除）**必须**走 `SelectableTable` 的 `runBatch` 聚合 + 危险动作 Modal 列出所选项二次确认；动作禁用用 `enabledWhen` + tooltip 原因，**不得**用原型里的裸 `<input type=checkbox>` + disabled button 草表实现（原型仅示意）。
- 三类详情差异：页签（合并配置行 + 字段/公式/SQL视图 Tab）；EXCEL（Excel 列配置 + 显示开关）；小计（仅公式，与页签一致）。

### 4.2 公式编辑 = `TabJoinFormulaDrawer`（共享编辑器）

- 详情「公式」列表点编辑/添加 → 弹该抽屉（唯一保留的抽屉）。
- 结构与现有 1:1：试算条 → 表达式 TextArea → 运算符+函数 → 规则提示 → 页签字段矩阵（行键锁定置灰）。
- 本次差异：去「结果类型」列、去「全局变量插入」；矩阵"可引用页签集 = 同目录组件"（决策 A，别名用 code）；标题/占位建议参数化（小计组件无"页签连表"语义）。
- **数据源契约新增**（评审核实——抽屉现全 templateId-scoped，`tabJoinFormulaService.ts:24-40`）：
  1. `GET 组件/目录级 tab-defs`（按 directoryId 聚合兄弟组件 → tabDefs，新端点 + 新 service，`ExcelViewService.tabDefsOfTemplate:746` 仅模板版）。
