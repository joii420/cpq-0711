## 1. 背景与目标

Excel 模板的「页签连表公式」配置页与公式配置功能已完成。本次做一次收敛重构，把**所有公式配置能力集中到组件管理**，并统一三类组件的配置体验。

**核心目标**

1. Excel 视图列配置**归属迁移**到组件管理（EXCEL 组件），模板配置页 Excel 视图改为"引用 EXCEL 组件即用"。
2. 三类组件（页签 / EXCEL / 小计）公式配置**统一复用编辑器** `TabJoinFormulaDrawer`（页签连表公式抽屉）。
3. 组件管理列表风格与模板管理对齐，配置风格统一。
4. 清理：移除冗余的右侧 `FieldPanel`；路径来源收敛为 SQL 视图；隐藏/合并若干配置项。

> **重要约束**：本次重构**不改动**「三大核心模块基线」里已 2026-05-21 终态锁定的**组件公式 `FormulaToken[]` 求值器与报价/核价/详情渲染读取链**。统一是在**编辑器（UX）层**，不是存储模型层（见 §3.2 决策 B）。

## 2. 决策清单（已与用户逐项确认 + 评审修订）

| # | 议题 | 决策 |
|---|------|------|
| 1 | Excel 配置归属 | **归属迁移（非"删实体"）**：当前 Excel 视图列配置存在 `Template.excelViewConfig`(JSONB) 上，**不是独立实体**。本次把"列定义 + 公式 + 显示/隐藏"迁为 `component_type = EXCEL` 组件的归属；模板侧 Excel 视图改为"引用 EXCEL 组件 + 可选列覆写"。详见 §3.1。 |
| 1b | EXCEL 组件粒度 | 一个 EXCEL 组件 = **一整张 Excel 表**（含全部列），模板 Excel 视图通常拖 1 个。 |
| 2 | 公式抽屉复用方式 | **复用 `TabJoinFormulaDrawer` 作为统一编辑器**；聚合可选。**存储不统一**——产出按组件类型分别落库（决策 B，见 §3.2）。 |
| 3 | 组件管理布局 | **左列表 + 右内嵌详情（Master-Detail 双栏）**。命中 `列表操作规范 §12` 例外白名单，但因含批量动作，**批量必须走 `runBatch` + 危险动作 Modal 列项二次确认**，不得用裸 checkbox（见 §4.1）。卡片复用 `tm-component-item` 渐变样式。 |
| 4 | 右侧 `FieldPanel` 去留 | **移除 FieldPanel，砍掉"全局变量 / 其他数据源插入公式"能力**。字段级全局变量取值（FieldConfigTable 🌐）/ 数据源绑定（DataSourceModal）独立存在，**不受影响**。 |
| 5 | 详情区精简 | 隐藏「树表（纯展示）」；「数据驱动路径」与「核价 BOM 递归展开」**合并一行**；公式列表**去「结果类型」列**。 |
| 6 | 路径来源收敛 | **仅移除 `datasource_binding.type === 'BNF_PATH'` 的 UI 选项 + 前端创建入口**；**保留** `default_source.type === 'BNF_PATH'`、`BnfPathResolver` SPI 注册、所有渲染层 BNF path 求值（值统一 `$view.列`）。详见 §3.3。 |
| A | 公式抽屉"可引用页签集" | **同目录组件即页签集**：矩阵列出同目录其他页签/EXCEL/小计组件。**别名用 component `code`（ASCII），显示用 name**（呼应历史教训：中文标识符需 ASCII 别名）。试算样本 = 反查"引用该组件的报价/核价卡片"，无样本则禁用试算、仅保存表达式。 |

**字段配置不变项**：`字段名 / 字段类型 / 内容(取值来源) / 金额 / 小计 / 行键 / 备注 / 排序` 全部沿用现有 `FieldConfigTable`，保持可编辑。唯一改动是路径来源收敛（点 6）。

## 3. 架构与数据模型改动

### 3.1 Excel 配置归属迁移（点 1）——核心工作包之一

**现状**（评审核实）：
- Excel 视图列配置 = `Template.excelViewConfig`(JSONB)（`template/entity/Template.java:71`），经 `ExcelViewService.saveExcelViewConfig/getExcelViewConfig`（`ExcelViewService.java:55-96`）读写。
- 前端 `ExcelViewConfigTab.tsx` 在模板配置里编辑；列模型 = `{col_key, title, source_type, hidden, formula, ...}`（`ExcelViewConfigTab.tsx:25-44`）。
- 渲染/导出：`ExcelViewService.buildExcelView` / `exportExcelView`（`ExcelViewService.java:617`）；tab-defs 构建 `tabDefsOfTemplate`（`:746`）。
- 注意：`CustomerExcelTemplate`（导入用）与本次无关，不动。

**改造**：
1. **新增组件类型** `component_type = 'EXCEL'`（详见 §3.4 传播清单）。
2. **列定义新存储位置**：Excel 列模型字段集与 `component.fields`(`FieldItem`) 不一致，**不强塞 fields**；在 EXCEL 组件上新增 jsonb 列（如 `excel_columns`）承载 `{col_key,title,source_type,hidden,formula,sort}`。`component.code` 有 unique 约束（`Component.java:25`），迁移时按规则生成 code。
3. **模板侧改造**：`Template.excelViewConfig` 退化为"引用哪个 EXCEL 组件 + 可选列覆写"。模板 Excel 视图渲染/导出读取源从 template 列**切换**为 EXCEL 组件列（`buildExcelView`/`exportExcelView`/`tabDefsOfTemplate` 全部要改读取源——这是断链风险点，逐处列改造清单）。
4. **隐藏列语义**：`hidden=true` 列仍参与公式计算、但不出现在 Excel 视图/导出。落点需在**渲染与导出两侧**实现。
5. **数据迁移**：历史 `excelViewConfig` 均为测试数据，**重建即可（不保数）**；但"读取源切换"的代码改造**不可省**，否则 Excel 视图渲染空/导出缺列。

### 3.2 公式模型——决策 B（抽屉只做编辑器，产出分类型落库）

**两套现状模型**（评审核实，不兼容）：
- 组件公式：`FormulaItem { expression: FormulaToken[] }`（`component/types.ts:119`）→ 存 `component.formulas` → 后端 `FormulaCalculationService` 按 token 求值 → 驱动报价/核价/详情渲染（**已终态锁定**）。
  - 关键事实：该 token 模型**已支持跨页签引用 + 聚合**（`cross_tab_ref` token 含 `agg/match/source`，由原 `CrossTabRefDrawer` 产出）。
- Excel 列公式：`{source_type:'TAB_JOIN_FORMULA', expression: string, tabs:[]}`（`TabJoinFormulaDrawer.tsx:84`）→ 存 `excelViewConfig` → tab-join 求值器。

**决策 B**：`TabJoinFormulaDrawer` 作为**共享编辑器 UI**；保存时按组件类型序列化到各自存储：
- 编辑**页签/小计组件**公式 → 序列化为现有**组件公式表示**（落 `component.formulas`，复用已锁定的 token 求值器，渲染链一行不动）。
- 编辑 **EXCEL 列**公式 → 维持**字符串表达式**（落 `excel_columns`，复用 tab-join 求值器）。

**B 的前置验证任务（实现计划首阶段必做）**：验证抽屉表达式的完整表达力（顶层 +/- 拆项、纯标量/总计只算一次、行键锁定置灰）能否**无损映射**到组件公式 token 语义。
- **通过** → 加一个"抽屉表达式 ↔ 组件公式表示"序列化桥即可。
- **不通过** → 折中：组件公式也存 `TAB_JOIN` 字符串、复用 tab-join 求值器，但**不迁移现有 token 公式**（新老并存，老的保持 token）；仍不重写锁定的求值器。

**明确排除（A 方案）**：不把所有 `component.formulas` 迁成字符串、不替换 token 求值器、不改渲染读取链——风险面过大且违反基线。

### 3.3 路径来源收敛（点 6 / AP-53）——精确范围

- **移除**：`datasource_binding.type === 'BNF_PATH'` 的 **UI 选项 + 前端创建入口**（`FieldConfigTable.tsx` / `types.ts:70` 子类型选择）；`PathPickerDrawer` 的 manual 手输路径 Tab；老表直引面向用户的警告入口。
- **保留（不得误删）**：
  - `default_source.type === 'BNF_PATH'`（INPUT 默认值来源，`types.ts:106`）——**与 datasource_binding 是两处**，误删会断 INPUT 默认值。
  - `BnfPathResolver` SPI 注册（`datasource/resolver/BnfPathResolver.java:23`）、`RESOLVER_TYPE_LABEL.BNF_PATH`。
  - 所有渲染层 `$view.列` / BNF path 求值（`QuotationStep2 / ComponentCell / treeTable / usePathFormulaCache` 等，散落 ~19 文件）。
- **数据**：删除历史含裸表路径（`mat_*` / `costing_part_*`）的测试配置。

### 3.4 新增 `component_type = EXCEL` 传播清单（独立于 AP-44 的 field_type 矩阵）

现状渲染/求值层是 **二元 NORMAL / SUBTOTAL** 假设（`componentType !== 'SUBTOTAL'`），`componentType` 影响面 codegraph 报 ~80 符号。加 EXCEL 必须显式三态化，至少覆盖：
1. 前端联合类型 `ComponentType`（`component/types.ts:12`）加 `'EXCEL'`。
2. 所有 `=== 'SUBTOTAL'` / `!== 'SUBTOTAL'` 二分分支改**显式三态**（`QuotationStep2.tsx:762/908/960/1482`、`enrichComponentData.ts:279` 默认值等）——避免 EXCEL 被当 NORMAL 去 driver-expand/求公式/出小计。
3. 后端 `ComponentService.validateRequest`（`:374`，当前**不校验** componentType）加枚举校验。
4. `refreshSnapshotsByComponent`、`ComponentExportBundle`（`:36`）、snapshot collector 等识别 EXCEL。
5. 组件管理详情按三态渲染对应形态（已在原型体现）。

> 写代码前用 `codegraph_impact` 列全 `componentType` 命中清单逐处勾；E2E 双 spec 验证。

## 4. UI 设计（详见原型）

### 4.1 组件管理页（Master-Detail）

- 左栏：搜索/新建 + 批量动作 + 手风琴目录 → 三段固定分类（页签/EXCEL/小计）→ 纵向渐变卡片（左侧勾选框）。
- 右栏：选中组件内嵌详情（非抽屉）；空态提示。
- **列表规范合规**（点 3 修订）：卡片多选 + 批量动作（发布/停用/删除）**必须**走 `SelectableTable` 的 `runBatch` 聚合 + 危险动作 Modal 列出所选项二次确认；动作禁用用 `enabledWhen` + tooltip 原因，**不得**用原型里的裸 `<input type=checkbox>` + disabled button 草表实现（原型仅示意）。
- 三类详情差异：页签（合并配置行 + 字段/公式/SQL视图 Tab）；EXCEL（Excel 列配置 + 显示开关）；小计（仅公式，与页签一致）。

### 4.2 公式编辑 = `TabJoinFormulaDrawer`（共享编辑器）

- 详情「公式」列表点编辑/添加 → 弹该抽屉（唯一保留的抽屉）。
- 结构与现有 1:1：试算条 → 表达式 TextArea → 运算符+函数 → 规则提示 → 页签字段矩阵（行键锁定置灰）。
- 本次差异：去「结果类型」列、去「全局变量插入」；矩阵"可引用页签集 = 同目录组件"（决策 A，别名用 code）；标题/占位建议参数化（小计组件无"页签连表"语义）。
- **数据源契约新增**（评审核实——抽屉现全 templateId-scoped，`tabJoinFormulaService.ts:24-40`）：
  1. `GET 组件/目录级 tab-defs`（按 directoryId 聚合兄弟组件 → tabDefs，新端点 + 新 service，`ExcelViewService.tabDefsOfTemplate:746` 仅模板版）。
  2. `GET 组件级 sample-cards`（反查引用该组件的报价/核价行，新端点）。
  3. `POST 组件级 dry-run`（组件上下文试算，`TabJoinFormulaDrawer.tsx:110` 现强制 templateId+sampleLi）。
  4. 抽屉 props + 三个 service 从 templateId 改为 directoryId/componentId 路由。
  > 这是 **4+ 个契约改动的工作包**，实现计划单列阶段。

### 4.3 模板配置页（点 1 模板侧）

- 调色板三组（页签/EXCEL/小计，渐变卡片可拖）。
- Excel 视图 = 单组件容器，拖入 EXCEL 组件即用；列与公式只读引用自组件，模板侧不再配公式。

## 7. 验收要点

1. 组件管理：左列表+右详情，三段分类卡片，三类详情形态正确；批量动作走 runBatch+Modal。
2. 三类组件配公式都弹 `TabJoinFormulaDrawer`：无结果类型列、无全局变量插入、矩阵=同目录组件（别名 code）。
3. 页签/小计公式落 `component.formulas`（token 求值器渲染正确）；EXCEL 列公式落 `excel_columns`（tab-join 求值）。
4. 字段配置保留 金额/小计/行键/备注 可编辑；路径来源只剩 SQL 视图；INPUT 默认值 BNF_PATH 与 BnfPathResolver 仍正常。
5. EXCEL 组件列保留显示/隐藏；模板 Excel 视图拖入 EXCEL 组件可渲染、可导出（隐藏列不出现）。
6. 详情区：树表隐藏、数据驱动路径与核价BOM递归合并一行。
7. component_type=EXCEL 三态传播全勾；E2E 双 spec（`quotation-flow` + `composite-product-flow`）+ 报价/核价/详情三视图截图通过（AP-44 SOP）。
