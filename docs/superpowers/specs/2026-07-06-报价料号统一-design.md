# 报价料号统一 设计方案（Spec 1 · 数据基座 + 发号服务）

> 日期：2026-07-06 ｜ 状态：草案（待用户评审）
> 项目：报价侧料号体系统一为「报价料号」`XXXX-YYMMNNNNNN`
> 本文 = **Spec 1**（三个 Spec 的第一个）。Spec 2＝选配发号统一，Spec 3＝客户料号维护页面。

---

## 0. 背景与目标

报价侧目前**散装**着多套"料号发号"机制，互不统一：

- **9 字头**（`MaterialNoResolver`）：报价 BOM 导入时，组件料号空但有名称 → `MAX(9字头)+1` 生成 `9000000001` 之类当 `material_no`。
- **`CFG-{符号}-{6位流水}`**（`PartNoProvider`/`AutoAllocatePartNoProvider` + `ConfiguratorInstanceService`）：选配产品生成 `hf_part_no`，如 `CFG-AgCu-000001`；BOM 导入明确**拒绝** `CFG-` 料号回填。

**统一目标**：报价侧所有料号收敛为**一套系统唯一料号「报价料号」**，格式 `XXXX-YYMMNNNNNN`：

- `XXXX`：客户唯一四位码（系统按客户首见递增分配，`0001..9999`）
- `YYMM`：发号当时年月
- `NNNNNN`：6 位流水，在 `(XXXX, YYMM)` 前缀下从 `000001` 递增

**三个料号的最终语义**（承载于 `material_customer_map`）：

| 料号 | 字段 | 含义 | 全局唯一 |
|---|---|---|---|
| **报价料号** | `material_no`（QUOTE 行） | **本系统唯一料号**（报价侧料号口径） | 是（QUOTE 内） |
| 生产料号 | `production_no`（新增） | 对接 ERP 的料号 | 否（可空、后补） |
| 客户料号 | `customer_product_no` | 客户方产品料号 | 否 |

**Spec 1 范围**：客户表结构 + 发号服务 + 通道① Q02 + 通道② BOM 无号组件改发报价料号（**取消 9 字头生成路径**）+ P05 打 PRICING 标 + 存量与唯一性 + 单测。
**不在 Spec 1**：选配 `CFG-` 统一（Spec 2）、维护页面（Spec 3）。

### 关键决策留档（与用户逐条确认）

| # | 决策 | 结论 |
|---|---|---|
| 报价↔生产基数 | 一个生产料号可对多个报价料号；一个报价料号只对一个生产料号 | 多对一 |
| 报价料号全局唯一 | 同一报价料号不跨两个客户 | 是 |
| pricing/quote 分离 | 不共用一列，加 `system_type` 维度按行隔离（方案 S-A） | S-A |
| 通道① Q02 | 用文件既有报价料号（现"宏丰料号"列 relabel），**不 mint**（Align-A） | Align-A |
| 通道② BOM | 全是报价料号；查不到即用文件值登记；无号有名组件改发报价料号 | 见 §4 |
| 生产料号 | 报价侧可空，后补（维护页面/ERP） | 可空 |
| 存量报价料号 | **不回填/不改写**，现值即报价料号；旧 9 字头/CFG- 号保留为合法历史号 | 不回填 |
| 9 字头规则 | **取消**（报价 BOM 无号组件改发 `XXXX-YYMMNNNNNN`） | 取消 |
| 客户四位码 | 系统按客户首见递增分配、持久化 | 自动分配 |
| 流水范围 | 每 `(客户码, YYMM)` 独立、从 `000001` 递增 | 每客户每月 |

## 1. 数据模型

### 1.1 `material_customer_map` 改造（Flyway 迁移）

现状（V218）：列 `material_no / customer_no / customer_name / customer_material_name / customer_product_no / customer_drawing_no / seq_no / payment_method / base_currency / quote_currency / exchange_rate`；唯一键 `(material_no, customer_no, customer_product_no)`。P05（核价）与 Q02（报价）**共用同表同列**。

**改动**：

1. `ADD COLUMN system_type VARCHAR(20)` —— `QUOTE` / `PRICING`，按行隔离两侧语义。
   - 存量回填规则见 §5.1（含分类不确定时的保守默认）。回填后设 `NOT NULL`。
2. `ADD COLUMN production_no VARCHAR(20) NULL` —— 生产料号（ERP，QUOTE 行后补；PRICING 行不用）。
3. **唯一键**：`DROP` 旧 `uq_material_customer_map`；`CREATE UNIQUE INDEX ... (system_type, material_no, customer_no, customer_product_no) NULLS NOT DISTINCT`（PG16 支持；使 `customer_product_no IS NULL` 的重复行仍被约束）。
4. **报价料号全局唯一**：`CREATE UNIQUE INDEX uq_mcm_quote_no ON material_customer_map(material_no) WHERE system_type='QUOTE'`。
   - 迁移前先跑校验（§5.2）：若 QUOTE 分类行内 `material_no` 有重复，**迁移显式失败并打印冲突行**，不静默建索引。

**列语义对照**（改造后）：

| 列 | QUOTE 行 | PRICING 行 |
|---|---|---|
| `system_type` | `QUOTE` | `PRICING` |
| `material_no` | 报价料号（系统唯一料号） | 生产料号（现状不变） |
| `production_no` | 生产料号（可空·后补） | 不用 |
| `customer_product_no` | 客户料号 | 客户料号 |
| 其余 | 不变 | 不变 |

### 1.2 发号服务用表

**`quote_customer_code`** —— 客户四位码（首见即分配，永久固定）
```sql
CREATE TABLE quote_customer_code (
  customer_no  VARCHAR(20) PRIMARY KEY,
  code         CHAR(4)     NOT NULL UNIQUE,   -- 0001..9999
  created_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE SEQUENCE quote_customer_code_seq START 1;   -- 下一个码 = lpad(nextval::text,4,'0')
```

**`quote_material_no_seq`** —— 每 `(客户码, YYMM)` 流水计数
```sql
CREATE TABLE quote_material_no_seq (
  customer_code CHAR(4) NOT NULL,
  year_month    CHAR(4) NOT NULL,               -- YYMM
  last_serial   INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (customer_code, year_month)
);
```

> 报价料号本身的权威登记就是 `material_customer_map`（QUOTE 行，`material_no` 唯一）。**不额外建登记表**——因为通道①/② 都最终落 `material_customer_map`，它即权威源；选配（Spec 2）也落此表。

## 2. 发号服务 `QuoteMaterialNoAllocator`

单一发号算法归处，entry-agnostic（谁调都走它）。Spec 1 建好 + 单测；调用方：通道② BOM（§4）。选配（Spec 2）后续接入。

### 2.1 接口

```java
/** 取或分配客户四位码（首见分配、之后固定）。 */
String getOrAllocateCustomerCode(String customerNo);

/** 铸造一个新报价料号 XXXX-YYMMNNNNNN（customerNo 决定 XXXX，yyMm 决定 YYMM）。 */
String mint(String customerNo, String yyMm);
```

### 2.2 客户四位码分配（并发安全）

```
SELECT code FROM quote_customer_code WHERE customer_no=:c;   -- 命中直接返回
-- 未命中：
code := lpad(nextval('quote_customer_code_seq')::text, 4, '0');
INSERT INTO quote_customer_code(customer_no, code) VALUES(:c, :code)
  ON CONFLICT (customer_no) DO NOTHING;
-- 竞态败者（另一事务先插）→ 再 SELECT 取既有 code（浪费一个序列值＝可接受空洞）。
if code > '9999' → 抛业务异常「客户四位码枯竭（>9999）」。
```

### 2.3 流水发号（原子）

```sql
INSERT INTO quote_material_no_seq(customer_code, year_month, last_serial)
VALUES (:code, :ym, 1)
ON CONFLICT (customer_code, year_month)
DO UPDATE SET last_serial = quote_material_no_seq.last_serial + 1
RETURNING last_serial;   -- 原子，无需 advisory lock
```
`report := code + '-' + ym + lpad(serial::text, 6, '0')`；`serial > 999999` → 抛「单客户单月流水溢出」。总长 15。

> 与旧 9 字头的 `advisory xact lock + MAX+1`（`MaterialNoResolver.generateNextMaterialNo`）不同：新方案用 `INSERT..ON CONFLICT DO UPDATE RETURNING` 单语句原子自增，简洁且天然并发安全。

## 3. 通道①：报价侧客户料号登记（Q02）

**`Q02CustomerMapHandler`（Sheet「客户料号与宏丰料号的关系」）**：

- 现有「宏丰料号」列 = **报价料号**（relabel）。兼容读取：优先读「报价料号」表头，回退「宏丰料号」。
- 值直接落 `material_no`，`system_type='QUOTE'`，**不 mint**（Align-A）；`production_no` 留空。
- 「客户产品编号」→ `customer_product_no`（客户料号）。
- 幂等：QUOTE 行报价料号全局唯一（§1.1），故 **upsert 冲突目标 = `material_no`（`WHERE system_type='QUOTE'` 部分唯一索引）**，而非 PRICING 用的 4 列复合键。`MaterialCustomerMapRepository` 的 upsert 需按 `system_type` 分派冲突目标：QUOTE→`(material_no) WHERE system_type='QUOTE'`，PRICING→`(system_type, material_no, customer_no, customer_product_no)`。这样"同报价料号、客户料号后来变更"走 update 而非撞唯一索引报错。
- Q02 现有「按客户先删后插（replace-per-customer）」逻辑**保留**——因报价料号是文件值、重导复用同值，删后重插号不变。

`MaterialCustomerMapRepository.MapRow` + `upsert/upsertBatch` SQL 增 `system_type`、`production_no` 两字段；冲突键含 `system_type`；`production_no` 用 `COALESCE(EXCLUDED, 现值)` 防止置空（后补的生产料号不被重导清掉）。

## 4. 通道②：报价侧 BOM 导入（取消 9 字头，改发报价料号）

**`MaterialBomMergeHandler`（物料BOM＋组成件BOM 合并）**：`投入料号 / 组成件料号 / 宏丰料号` 全是报价料号。

现状用 `MaterialNoResolver.resolve(料号, 名称, batch)` 解析组件料号：料号有值→返回；空+有名→按名匹配料号表复用，匹配不到→生成 9 字头。

**改造后组件料号解析算法**（取代 9 字头生成）：

```
resolveQuoteMaterialNo(报价料号, 名称, customerNo, yyMm, batch):
  1. 报价料号有值 → 直接返回（文件值即报价料号）；
     并「确保登记」：material_customer_map 无 (QUOTE, 该报价料号, customerNo) 行则插一行
     （material_no=报价料号, system_type=QUOTE, customer_no=customerNo, production_no=NULL）。   ← 通道②「查不到即登记」(Q4=a/Q5=B)
  2. 报价料号空 + 名称有值 → 先按名称匹配既有报价料号（沿用 MaterialNoResolver 的 findFirstByMaterialName 复用语义）；
     命中复用；未命中 → allocator.mint(customerNo, yyMm) 铸新报价料号，并登记 material_customer_map 一行。   ← 取代 9 字头
  3. 都空 → 报错（同现状）。
```

- `batch` 内维持"名称→报价料号"缓存（同批同名复用），与现有 `BatchState.nameToNo` 语义一致。
- `customerNo` 取导入上下文 `ctx.customerNo`；`yyMm` 取导入当时年月。
- `MaterialNoResolver` 的 9 字头生成路径（`generateNextMaterialNo` / `maxNineLeadingMaterialNo` / `lockForMaterialNoGeneration`）在报价侧**停用**（本 Spec 1 不物理删类，仅切断报价侧调用；彻底删除待所有调用方迁移后单独清理，避免误伤核价侧潜在引用——实现时用 `codegraph_callers` 核实调用面）。
- `CFG-` 前缀拒绝逻辑（`isCfg`）**本 Spec 保留不动**（选配统一在 Spec 2 处理）。

> **协议级提示**：`MaterialBomMergeHandler` 属报价导入核心，改动后须跑报价 E2E（`docs/E2E测试方法.md`），并对照 AP-51 行数纪律 / 组件渲染。

## 5. 存量与唯一性

### 5.1 `system_type` 回填（存量分类）

存量 `material_customer_map` 行需分 QUOTE / PRICING。**风险点**：现有行无内建标记区分两侧写入来源，需按实际数据判定。

- 分类信号候选（实现时用 `codegraph` + 数据核对确定）：写入来源（P05 vs Q02）无持久标记 → 依赖可观测信号（如 `customer_no` 取值域、是否存在对应核价 BOM / 报价模板引用等）。
- **保守默认**：分类不确定的存量行默认 `PRICING`（PRICING 侧 `material_no` 不加全局唯一约束，默认 PRICING 不会因 §5.2 校验失败）。
- **开放项（实现前必做）**：跑一段核对 SQL 确认存量 QUOTE/PRICING 可否干净分类；不能则与用户确认默认策略后再迁移。**此项在实现计划里作为第一个前置任务。**

### 5.2 报价料号唯一性校验（建索引前）

```sql
-- 期望 0 行；有行则迁移失败并打印
SELECT material_no, count(*) FROM material_customer_map
WHERE system_type='QUOTE' GROUP BY material_no HAVING count(*) > 1;
```

### 5.3 存量号处理

- 存量 `material_no`（含旧 9 字头 / CFG- 值）**保留不动**，视为合法历史报价料号。**不回填、不改写、不重发**。
- 仅"今后发号规则"统一为 `XXXX-YYMMNNNNNN`。
- 存量 `production_no` 全空，后补。

## 6. 边界与上限

| 情形 | 处理 |
|---|---|
| 客户数 > 9999（四位码枯竭） | 发号显式报错，不静默 |
| 单客户单月流水 > 999999 | 发号显式报错 |
| `customer_product_no` 为 NULL | 唯一键用 `NULLS NOT DISTINCT` 约束 |
| PRICING 行 `material_no`（生产料号）跨客户重复 | 允许（不加全局唯一约束） |
| 通道② 报价料号跨客户冲突（文件把 A 客户的号写进 B 客户 BOM） | 全局唯一（QUOTE）→ 登记时命中既有行（属别客户）报错，不静默串号 |

## 7. 测试

**`QuoteMaterialNoAllocator` 单测**
- 四位码：首见分配、二次复用同码、并发下唯一（序列空洞可接受）、`>9999` 报错。
- mint：格式 `XXXX-YYMMNNNNNN`（15 位）正确；同 `(码,YYMM)` 流水 `000001→000002`；跨 YYMM 归 `000001`；`>999999` 报错。

**通道① Q02 集成**
- 写入行 `system_type='QUOTE'`；relabel 列读取（报价料号/宏丰料号任一表头）；先删后插后报价料号不变（幂等）。

**通道② BOM 集成**
- 组件有报价料号 → 解析并「确保登记」`material_customer_map`（QUOTE 行、production_no 空）。
- 组件无号有名 + 无既有同名 → mint `XXXX-YYMMNNNNNN`（非 9 字头）并登记。
- 组件无号有名 + 有既有同名 → 复用既有报价料号（不新发）。
- 同批同名两组件 → 同一报价料号。
- 都空 → 报错。

**P05 集成**：写入行 `system_type='PRICING'`。

**Repository**：upsert 冲突键含 `system_type`；`production_no` COALESCE 不被置空。

**迁移/校验**：`system_type` 回填后无 NULL；QUOTE 报价料号无重复（§5.2）。

> 协议级（改 `MaterialBomMergeHandler`）→ 跑 `quotation-flow.spec.ts` E2E。

## 9. 影响面清单（实现前用 codegraph 核实）

- `material_customer_map` 全部读写点（视图 / driver / 报价候选查询 `FROM material_master WHERE material_no IN ...` 等）——加 `system_type` 后是否需在查询侧补维度。
- `MaterialNoResolver` 全部 `callers`（报价侧切断、核价侧是否有引用）。
- `MaterialCustomerMapRepository` 全部 `upsert/upsertBatch/deleteByCustomerNo` 调用点。
- 报价料号全局唯一索引对现有报价导入的影响（重复号是否会致导入失败）。
