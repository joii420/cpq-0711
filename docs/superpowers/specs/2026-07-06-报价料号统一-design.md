> 日期：2026-07-06 ｜ 版本：v4（第三轮评审 6 条 Chain 缺口全补：§5 坐标更正+V6路径JOIN、§6.3 对称校验、ensure跨客户检测、upsert双失败路径、V255 视图过滤、强判据落 SQL）｜ 状态：草案（收口版，待进实现）
> 项目：报价侧料号体系统一为「报价料号」`XXXX-YYMMNNNNNN`
> 本文 = **Spec 1**（三个 Spec 的第一个）。Spec 2＝选配发号统一，Spec 3＝客户料号维护页面。

## 0. 背景与目标

报价侧目前**散装**着多套"料号发号"机制，互不统一：

- **9 字头**（`MaterialNoResolver.resolve` → `generateNextMaterialNo`）：报价 handler 导入时，料号空但有名称 → 按名匹配 `material_master`，匹配不到 → `MAX(9字头)+1` 生成 `9000000001` 之类。**调用点 = 8 个报价 handler**（见 §4）。
- **`CFG-{符号}-{6位流水}`**（`PartNoProvider`/`AutoAllocatePartNoProvider` + `ConfiguratorInstanceService`）：选配产品生成 `hf_part_no`；BOM 导入明确**拒绝** `CFG-` 料号回填。

**统一目标**：报价侧所有料号收敛为**一套系统唯一料号「报价料号」**，格式 `XXXX-YYMMNNNNNN`：
- `XXXX`：客户唯一四位码（系统按客户首见递增分配，`0001..9999`）
- `YYMM`：发号当时年月
- `NNNNNN`：6 位流水，在 `(XXXX, YYMM)` 前缀下从 `000001` 递增

**三个料号的最终语义**（承载于 `material_customer_map`）：

| 料号 | 字段 | 含义 | 基数/唯一 |
|---|---|---|---|
| **报价料号** | `material_no`（QUOTE 行） | 本系统唯一料号（报价侧料号口径） | 全局唯一（QUOTE 内）；**↔ 客户料号 1:1** |
| 生产料号 | `production_no`（新增） | 对接 ERP 的料号 | 可空、后补；**↔ 报价料号 1:N** |
| 客户料号 | `customer_product_no` | 客户方产品料号 | ↔ 报价料号 1:1 |

**Spec 1 范围**：客户表结构 + 发号服务 + 报价侧**全部 8 个发号 handler 改发报价料号（取消 9 字头）** + 通道① Q02 relabel + P05 打 PRICING 标 + **受影响读点/视图加 `system_type` 过滤** + 存量分类回填与唯一性 + 单测。
**不在 Spec 1**：选配 `CFG-` 统一（Spec 2）、维护页面（Spec 3）。

### 关键决策留档（与用户逐条确认）

| # | 决策 | 结论 |
|---|---|---|
| 报价料号 ↔ 客户料号 | **1:1**（一个报价料号只对一个客户料号） | 1:1 |
| 生产料号 ↔ 报价料号 | 一个生产料号可对多个报价料号 | 1:N |
| 报价料号全局唯一 | 同一报价料号不跨两个客户 | 是 |
| pricing/quote 分离 | 加 `system_type` 维度按行隔离（方案 S-A） | S-A |
| 通道① Q02 | 用文件既有报价料号（现"宏丰料号"列 relabel），不 mint（Align-A） | Align-A |
| **9 字头取消范围** | **报价侧全部 `resolve()` 调用点（8 handler），名匹配不到就发报价料号** | 全量取消 |
| 生产料号 | 报价侧可空，后补 | 可空 |
| 存量报价料号 | 不回填/不改写，现值即报价料号；旧 9 字头/CFG- 号保留为合法历史号 | 不回填 |
| 客户四位码 | 系统按客户首见递增分配、持久化 | 自动分配 |
| 流水范围 | 每 `(客户码, YYMM)` 独立、从 `000001` 递增 | 每客户每月 |

## 1. 数据模型

### 1.1 `material_customer_map` 改造（Flyway 迁移）

现状（V218）：唯一键 `(material_no, customer_no, customer_product_no)`，`customer_product_no NOT NULL`。P05（核价）与 Q02（报价）**共用同表同列**。

**DDL 改动**：

1. `ADD COLUMN system_type VARCHAR(20)` —— `QUOTE`/`PRICING`，按行隔离两侧语义。存量回填见 §5.1，回填后设 `NOT NULL`。
2. `ADD COLUMN production_no VARCHAR(20) NULL` —— 生产料号（ERP，QUOTE 行后补；PRICING 行不用）。
3. **`ALTER COLUMN customer_product_no DROP NOT NULL`**（Blocker-3）—— 通道②/费用表 mint 出的 BOM 组件登记行无客户料号，须允许 NULL。
4. **唯一键**：`DROP` 旧 `uq_material_customer_map`；建两个索引：
   - `uq_mcm_composite` = `UNIQUE (system_type, material_no, customer_no, customer_product_no) NULLS NOT DISTINCT`（PG16；覆盖 PRICING 生产料号跨客户重复 + QUOTE 行）。
   - `uq_mcm_quote_no` = `UNIQUE (material_no) WHERE system_type='QUOTE'`（报价料号全局唯一；因 ↔ 客户料号 1:1，单列即足，不与复合键矛盾）。
   - **强制**加固 1:1（从客户料号侧，New-5）：`uq_mcm_quote_cust_prod` = `UNIQUE (system_type, customer_no, customer_product_no) WHERE system_type='QUOTE' AND customer_product_no IS NOT NULL` —— 它是"同客户同客户料号 ↔ 唯一报价料号"的**唯一硬保证**（否则同一客户料号会静默映射两个报价料号）。**不是可选**。
   - 建 `uq_mcm_quote_no` 前先跑 §5.2 校验，重复则迁移显式失败并打印冲突行。

**列语义对照**（改造后）：

| 列 | QUOTE 行 | PRICING 行 |
|---|---|---|
| `system_type` | `QUOTE` | `PRICING` |
| `material_no` | 报价料号（系统唯一料号） | 生产料号（现状不变） |
| `production_no` | 生产料号（可空·后补） | 不用 |
| `customer_product_no` | 客户料号（通道②组件登记行为 NULL） | 客户料号 |

### 1.2 发号服务用表

```sql
CREATE TABLE quote_customer_code (
  customer_no VARCHAR(20) PRIMARY KEY,
  code        CHAR(4)     NOT NULL UNIQUE,     -- 0001..9999
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE SEQUENCE quote_customer_code_seq START 1;

CREATE TABLE quote_material_no_seq (
  customer_code CHAR(4) NOT NULL,
  year_month    CHAR(4) NOT NULL,              -- YYMM
  last_serial   INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (customer_code, year_month)
);
```

> **报价料号的权威登记 = `material_customer_map` 的 QUOTE 行**（`material_no` 全局唯一）。不额外建登记表。可行的前提是 §3 把 Q02 的先删后插收窄到"仅客户料号映射行"，使 mint 登记的组件行（`customer_product_no IS NULL`）不被重导删除（见 §3 对 Major-8 的处理）。

## 2. 发号服务 `QuoteMaterialNoAllocator`

单一发号算法归处，entry-agnostic。Spec 1 建好 + 单测；调用方：§4 全部 8 个 handler（经 `MaterialNoResolver` 统一入口）。选配（Spec 2）后续接入。

### 2.1 接口

```java
String getOrAllocateCustomerCode(String customerNo);          // 取或分配客户四位码（首见分配、之后固定）
String mintAndRegister(String customerNo, String yyMm);       // 铸新报价料号 + 登记 material_customer_map QUOTE 行（组件行 customer_product_no=NULL）
void   ensureRegistered(String customerNo, String quoteNo);   // 料号有值（文件值）时确保 QUOTE 行存在，幂等（ON CONFLICT DO NOTHING）
```

> **New-2（甲）**：`material_customer_map` 的 QUOTE 行 = **全部报价料号的统一登记表**——既含真客户产品（`customer_product_no` 非空，来自 Q02），也含 §4 各 handler mint 的原料/元素/费用/子件料号（`customer_product_no=NULL`）。每个报价料号都有一行、都有 `production_no` 槽位（对 ERP 待补），语义统一。候选查询靠 `customer_product_no IS NOT NULL` 隔离出"真客户产品"（§5 New-1）。

### 2.2 客户四位码分配（并发安全）

```
SELECT code FROM quote_customer_code WHERE customer_no=:c;               -- 命中直接返回
-- 未命中：
code := lpad(nextval('quote_customer_code_seq')::text, 4, '0');
INSERT INTO quote_customer_code(customer_no, code) VALUES(:c,:code) ON CONFLICT (customer_no) DO NOTHING;
-- 竞态败者再 SELECT 取既有 code（浪费一个序列值＝可接受空洞）。
if code > '9999' → 抛「客户四位码枯竭(>9999)」。
```

### 2.3 流水发号 + 登记（原子）

```sql
-- ① 原子自增流水
INSERT INTO quote_material_no_seq(customer_code, year_month, last_serial)
VALUES (:code, :ym, 1)
ON CONFLICT (customer_code, year_month)
DO UPDATE SET last_serial = quote_material_no_seq.last_serial + 1
RETURNING last_serial;                       -- serial > 999999 → 抛「单客户单月流水溢出」
-- report := code || '-' || ym || lpad(serial::text,6,'0')   （总长 15）

-- ② 登记 material_customer_map QUOTE 行（组件级，无客户料号）；幂等（New-6）
INSERT INTO material_customer_map(system_type, material_no, customer_no, customer_product_no, production_no, created_at, updated_at)
VALUES ('QUOTE', :report, :customerNo, NULL, NULL, NOW(), NOW())
ON CONFLICT (material_no) WHERE system_type='QUOTE' DO NOTHING;
```

- `ensureRegistered(customerNo, quoteNo)`（料号有文件值、通道②"查不到即登记"，全 8 handler 用）：
  ```
  1. rows := INSERT ... VALUES('QUOTE', quoteNo, customerNo, NULL, NULL,...)
             ON CONFLICT (material_no) WHERE system_type='QUOTE' DO NOTHING;   -- 幂等（New-6）
  2. if rows==0:  -- 已存在同报价料号行 → 必须核对客户，防跨客户静默串号（Chain-3）
       owner := SELECT customer_no FROM material_customer_map WHERE material_no=quoteNo AND system_type='QUOTE';
       if owner != customerNo → 抛/记 "报价料号跨客户串号"（handler recordError，该行不落库）。
  ```
  - **为何不能只 DO NOTHING**：报价料号 R 已登记给客户 A（其前缀 `XXXX` 本就是 A 的四位码），B 客户导入的 BOM/费用文件出现 R 时，DO NOTHING 会 0 错误静默接受、handler 仍把 R 当 B 的 `material_no` 写进 `element_bom`/`unit_price` → 跨客户泄漏。第 2 步的客户核对把 §7 的"跨客户报错"从 Q02 扩到全 8 handler。

> 用 `INSERT..ON CONFLICT DO UPDATE RETURNING` 单语句原子自增，取代旧 9 字头的 `advisory lock + MAX+1`。

## 3. 通道①：报价侧客户料号登记（Q02）+ Repository 改造

**`Q02CustomerMapHandler`（Sheet「客户料号与宏丰料号的关系」）**：
- 现「宏丰料号」列 = **报价料号**（relabel；兼容读取：优先「报价料号」表头，回退「宏丰料号」）。值落 `material_no`，`system_type='QUOTE'`，**不 mint**（Align-A）；`production_no` 留空。
- 「客户产品编号」→ `customer_product_no`（客户料号，Q02 要求非空——保持现状）。
- 幂等 + 客户守卫（New-4）：报价料号 ↔ 客户料号 1:1 且 `material_no` 全局唯一 → **upsert 冲突目标 = `(material_no) WHERE system_type='QUOTE'`**：
  ```sql
  INSERT INTO material_customer_map(system_type, material_no, customer_no, customer_product_no, ...)
  VALUES ('QUOTE', :quoteNo, :customerNo, :custProdNo, ...)
  ON CONFLICT (material_no) WHERE system_type='QUOTE'
  DO UPDATE SET
    customer_product_no = EXCLUDED.customer_product_no,     -- ★ 必须进 SET：把组件先登记的 NULL 回填成客户料号
    customer_name = COALESCE(EXCLUDED.customer_name, material_customer_map.customer_name),
    customer_material_name = ..., customer_drawing_no = ..., 货币/汇率/付款 = ...,
    updated_at = NOW()
  WHERE material_customer_map.customer_no = EXCLUDED.customer_no;   -- ★ 客户守卫：跨客户命中→0 行更新
  ```
  - `customer_product_no` **必须进 `DO UPDATE SET`**（否则组件先登记的 `(报价料号, NULL)` 被 Q02 `(报价料号, X)` 命中时，X 回填不进去、客户料号丢——现有 upsert 因它原是冲突键而不在 SET 里，本次要挪进来）。
  - **客户守卫 `WHERE customer_no=EXCLUDED.customer_no`**：文件把 A 客户的报价料号写进 B 客户导入时，命中的是 A 的行、守卫使更新影响 0 行 → handler 检出"冲突键命中但 0 行更新"→ 显式报"报价料号跨客户串号"，不静默改客户。这与 §7 一致。
  - **两类失败必须分开处理（Chain-4）**：
    - (a) **0 行更新** = 上面的客户守卫拦截（跨客户串号）→ 该行 `recordError`。
    - (b) **抛 `unique_violation` 异常** = 撞 **`uq_mcm_quote_cust_prod`**（同 `(customer_no, customer_product_no)`、不同 `material_no`——即客户料号被改绑到新报价料号 / 一个客户料号误配两个报价料号）。因 `ON CONFLICT` 的 arbiter 是 `(material_no)` ≠ 该冲突索引，PG 直接抛异常，不是 0 行。
    - handler 是 `@Transactional(REQUIRES_NEW)`（`Q02CustomerMapHandler.java:36` 等），(b) 若不 per-row 捕获会**回滚整个 sheet**。故 upsert 须 **per-row `try-catch`（或 savepoint）** 把 (b) 转 `recordError("客户料号 1:1 冲突")`，单行毒不死整批；或导入期先预校验客户料号唯一性再写。

**`deleteByCustomerNo` 收窄（一并修 Major-4 + Major-8）**：
- 现状 `DELETE ... WHERE customer_no=:c`（无维度）。改为参数化：Q02 先删后插只删 `WHERE customer_no=:c AND system_type='QUOTE' AND customer_product_no IS NOT NULL`。
- 效果：① 不误删 PRICING 行（M4）；② 保住 §2/§4 mint 登记的组件行（`customer_product_no IS NULL`，M8）——它们不属 Q02 sheet 的"客户料号映射全集"，不应被 replace 波及。
- 依据：Q02 sheet 行 `customer_product_no` 恒非空（`Q02CustomerMapHandler` 必填校验），故 `IS NOT NULL` 是可靠判别。

**`MaterialCustomerMapRepository` 联动改造（Minor-9，逐处点名）**：
- `MapRow`：增 `systemType`、`productionNo` 两字段。
- `coalesceOver` 折叠 dedup 键：`List.of(materialNo, customerNo, customerProductNo)` → **加 `systemType`**，否则跨 system_type 行被错误折叠。
- `upsertChunk`：11 列硬编码 SQL → 13 列；`b = i*11` 步长改 `i*13`；VALUES 占位、参数绑定同步。
- `production_no` 用 `COALESCE(EXCLUDED.production_no, material_customer_map.production_no)`，后补的生产料号不被重导清空。

**P05（核价）**：`P05CustomerMapHandler` 写入行打 `system_type='PRICING'`；不受报价料号影响。

## 4. 报价侧发号统一：8 个 handler 取消 9 字头（Blocker-1）

`MaterialNoResolver.resolve()` 的生成分支（9 字头）当前被 **8 个报价 handler** 调用：
`MaterialBomMergeHandler`(投入料号:76 / 组成件料号:105)、`Q04ElementBomHandler`:61、`Q06FixedProcessFeeHandler`:61、`Q07IncomingOtherFeeHandler`:58、`Q08IncomingAnnualDiscountHandler`:57、`Q09IncomingRecoveryHandler`:55、`Q10SelfProcessFeeHandler`:62、`Q13ComponentOtherFeeHandler`:59。（`Q05ElementRecoveryHandler`:58 用 `resolveMatchOnly`，只匹配不生成 → 不改。）

**统一改造：把发号收敛进 `MaterialNoResolver`，一处改、8 处受益**：

1. `BatchState` 增 `customerNo`、`yyMm` 两字段（每次导入构造时由 handler 从 `ctx.customerNo` + 导入年月注入）。
2. `resolve(料号, 名称, batch)` 算法改为：
   ```
   料号有值 → 返回（文件值即报价料号）；
   料号空+有名 → 先按名匹配既有报价料号（findFirstByMaterialName / batch.nameToNo 缓存复用）；
                 命中复用；未命中 → allocator.mintAndRegister(batch.customerNo, batch.yyMm)（取代 9 字头）；
   都空 → 抛 MaterialNoUnresolvableException（同现状）。
   ```
3. 删除/停用 `generateNextMaterialNo`（9 字头）；`maxNineLeadingMaterialNo`/`lockForMaterialNoGeneration` 仅在确认无其它调用后清理（实现时 `codegraph_callers` 核实非报价侧无引用）。
4. 8 个 handler 唯一改动 = 构造 `BatchState` 时传 `customerNo`+`yyMm`；`resolve` 调用签名不变。
5. `material_master` 名称→料号复用副作用保留（各 handler 原有 upsert 不动）；mint 出的报价料号既进 `material_master`（经 handler 既有 upsert）又进 `material_customer_map`（经 §2.3 登记），两者一致。

> **登记范围（New-2 甲，显式声明）**：`resolve()` 的"确保登记"对**全部 8 个 handler**生效——料号有值 → `ensureRegistered(customerNo, 报价料号)`（幂等 DO NOTHING）；料号空+有名+需铸新 → `mintAndRegister` 已含登记。于是元素/费用/子件料号也登记进 `material_customer_map`（`customer_product_no=NULL`）。这是有意为之：map = 全部报价料号的统一登记表（每号一行、含 `production_no` 槽位）。候选查询靠 `customer_product_no IS NOT NULL` 隔离（§5 New-1）——这是采纳甲的**配套硬约束**，不可省。
> **`isCfg` 拒绝逻辑本 Spec 保留不动**（选配统一在 Spec 2）。
> **协议级**：改 `MaterialBomMergeHandler` 等报价导入核心 → 跑报价 E2E（`quotation-flow.spec.ts`），对照 AP-51 行数纪律。

## 5. 受影响读点/视图（Major-5，本 Spec 交付项，非调研）

加 `system_type` 后，下列读点会串到错误一侧或行扇出（AP-22 家族），**须在本 Spec 迁移中补 `system_type='QUOTE'` 过滤**：

> **⚠️ Chain-1 坐标更正**：以下按 `CustomerPartCandidateService` 真实代码坐标重列（v3 曾把污染子查询错标到 V6 路径）。实现时仍以 `codegraph` 复核实际行号为准，行号仅供定位。

1. **`CustomerPartCandidateService` 回退路径（手工"+添加产品"，`importRecordId==null`）**：
   - 裸子查询 `~:76` `WHERE p.material_no IN (SELECT material_no FROM material_customer_map)` → **加 `WHERE system_type='QUOTE' AND customer_product_no IS NOT NULL`**（New-1 真正要改这里）。该查询原意（`~:53-57` 注释）＝"候选＝有客户映射的产品、排除 BOM 子件原料"；甲登记后 map 有大量 `customer_product_no=NULL` 组件行，**只按 `system_type` 挡不住**，必须再加 `customer_product_no IS NOT NULL`。
   - 回退路径的 `LEFT JOIN material_customer_map m` `~:71` → JOIN 条件加 `AND m.system_type='QUOTE'`。
2. **`CustomerPartCandidateService` V6 优先路径（`listCandidatesV6`）**：
   - `WHERE p.material_no IN :hfs` `~:170`（`hfs` 源自 `V6QuotationCommitService:56` 已带 `customer_product_no IS NOT NULL`）→ **无需改**（上游已过滤，组件行不入候选）。
   - 但其 `LEFT JOIN material_customer_map m` `~:165` → **须加 `AND m.system_type='QUOTE'`**（否则同 `material_no` 命中 QUOTE+PRICING 两行扇出/误挂 PRICING 列）。此处 v3 遗漏（Chain-1）。
3. `V6QuotationCommitService.java:53-56`：`SELECT DISTINCT customer_product_no, material_no FROM material_customer_map WHERE 客户+时间窗 AND customer_product_no IS NOT NULL` → 再加 `AND system_type='QUOTE'`。（已含 `customer_product_no IS NOT NULL`，天然不含组件行。）

## 6. 存量迁移

### 6.1 `system_type` 回填（Major-6：默认方向按信号判 QUOTE）

存量行按**可观测列信号**分类（比"无标记默认 PRICING"可靠——后者会把真报价行错标、致 §5 读点对存量全返空）：

- **判 QUOTE（强信号）**：6 个"报价专有"列 `customer_material_name` / `customer_drawing_no` / `payment_method` / `base_currency` / `quote_currency` / `exchange_rate` **任一非空** → QUOTE。依据：`P05CustomerMapHandler` 这些列全传 `null`（:55-61）。
- **⚠️ New-3 更正**：**不能**用 `seq_no IS NULL → QUOTE`。P05「项次」非必填（`P05CustomerMapHandler` `row.getInt("项次")` 可空），项次空的**真 PRICING 行** 6 信号列全 null 且 `seq_no IS NULL` → 会被误判 QUOTE；且它与"可选列全空的最小 Q02 行"**列内容完全一致、无信号可分**。
- **歧义行（6 信号列全空）→ 不默认任何一侧**，走强判据 / 人工：
  - 强判据（Chain-6 落成具体 SQL）：核价/报价侧 BOM/单价表都带显式 `system_type` 列且写值可靠（如 `Q04ElementBomHandler.java:99` 写 `system_type='QUOTE'`，非循环依赖）。判据 = 该 `material_no` 是否被**报价侧**数据引用：
    ```sql
    -- 报价侧引用集（被引用→倾向 QUOTE；仅核价侧引用/无引用→倾向 PRICING）
    SELECT DISTINCT material_no FROM element_bom  WHERE system_type='QUOTE'
    UNION SELECT DISTINCT material_no FROM material_bom WHERE system_type='QUOTE'
    UNION SELECT DISTINCT material_no FROM unit_price   WHERE system_type='QUOTE'
    -- （实现时以 codegraph 复核报价侧还有哪些表带 system_type='QUOTE' 的 material_no 引用，补全并集）
    ```
    歧义行 `material_no ∈ 上集 → QUOTE；否则 PRICING`。
  - 实现前先跑 §6.2 量化歧义行占比：为 0 → 用下方强信号 SQL 即安全；非 0 → 按强判据或人工核对清单处理，**不得盲目默认**。
- 回填 SQL（仅对强信号明确者）：
  ```sql
  UPDATE material_customer_map SET system_type = 'QUOTE'
  WHERE system_type IS NULL
    AND (customer_material_name IS NOT NULL OR customer_drawing_no IS NOT NULL
         OR payment_method IS NOT NULL OR base_currency IS NOT NULL
         OR quote_currency IS NOT NULL OR exchange_rate IS NOT NULL);
  -- 其余（强信号全空的歧义行）按强判据/人工单独处理后再置 PRICING/QUOTE
  ```

### 6.2 疑似双源合并行（Major-7）

P05 与 Q02 今天共用唯一键 `(material_no, customer_no, customer_product_no)`：同一三元组若在报价簿与核价簿都出现，历史会 `ON CONFLICT DO UPDATE` **合并成同一物理行**，无法从单行还原两侧。

- **实现前必做（两类都量化）**：
  ```sql
  -- (1) 疑似双源合并行：既有 QUOTE 强特征、又有 PRICING 项次
  SELECT count(*) FROM material_customer_map
  WHERE (customer_material_name IS NOT NULL OR base_currency IS NOT NULL /*QUOTE 特征*/)
    AND seq_no IS NOT NULL /*PRICING 项次*/;
  -- (2) 歧义行（New-3）：6 强信号列全空——无法用 §6.1 强信号分类，含"项次空的 P05 行"与"最小 Q02 行"
  SELECT count(*) FROM material_customer_map
  WHERE customer_material_name IS NULL AND customer_drawing_no IS NULL
    AND payment_method IS NULL AND base_currency IS NULL
    AND quote_currency IS NULL AND exchange_rate IS NULL;
  ```
- (1) 非零 → 拆分策略（复制成 QUOTE+PRICING 两行，或人工核对清单）；(2) 非零 → §6.1 强判据/人工。两者皆 0 → §6.1 强信号回填即安全。**此项是实现计划的第一个前置任务。**

### 6.3 唯一性校验（建两个 QUOTE 唯一索引前，两段对称——Chain-2）

存量在旧键 `(material_no, customer_no, customer_product_no)` 下写入，本次两个新 QUOTE 索引都可能撞存量：

```sql
-- ① 建 uq_mcm_quote_no 前：QUOTE 内 material_no 唯一
SELECT material_no, count(*) FROM material_customer_map
WHERE system_type='QUOTE' GROUP BY material_no HAVING count(*)>1;

-- ② 建 uq_mcm_quote_cust_prod 前：QUOTE 内 (customer_no, customer_product_no) 唯一（New-5 的必要配套）
--    旧键允许 (M1,C,X)+(M2,C,X) 共存（material_no 不同即合法）→ 同客户同客户料号历史上可映射两个宏丰料号；
--    1:1 是本次新规、存量不受约束 → 不先校验，建强制索引会直接抛通用 unique_violation 崩迁移。
SELECT customer_no, customer_product_no, count(*) FROM material_customer_map
WHERE system_type='QUOTE' AND customer_product_no IS NOT NULL
GROUP BY customer_no, customer_product_no HAVING count(*)>1;
```
两段**都为 0** 才建索引；任一非 0 → 迁移显式失败并打印冲突行，按业务定 1:1 收敛策略（保留哪个报价料号、其余归并/废弃）后再建。

### 6.4 存量号处理
存量 `material_no`（含旧 9 字头/CFG- 值）**保留不动**，视为合法历史报价料号；不回填、不改写、不重发。`production_no` 存量全空、后补。
