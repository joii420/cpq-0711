# 报价料号统一 设计方案（Spec 1 · 数据基座 + 发号服务）

> 日期：2026-07-06 ｜ 版本：v2（据独立评审 9 条 findings + 用户两项业务决策回炉重写）｜ 状态：草案（待用户二次评审）
> 项目：报价侧料号体系统一为「报价料号」`XXXX-YYMMNNNNNN`
> 本文 = **Spec 1**（三个 Spec 的第一个）。Spec 2＝选配发号统一，Spec 3＝客户料号维护页面。

---

## 0. 背景与目标

报价侧目前**散装**着多套"料号发号"机制，互不统一：

- **9 字头**（`MaterialNoResolver.resolve` → `generateNextMaterialNo`）：报价 handler 导入时，料号空但有名称 → 按名匹配 `material_master`，匹配不到 → `MAX(9字头)+1` 生成 `9000000001` 之类。**调用点 = 8 个报价 handler**（见 §4）。
- **`CFG-{符号}-{6位流水}`**（`PartNoProvider`/`AutoAllocatePartNoProvider` + `ConfiguratorInstanceService`）：选配产品生成 `hf_part_no`；BOM 导入明确**拒绝** `CFG-` 料号回填。

**统一目标**：报价侧所有料号收敛为**一套系统唯一料号「报价料号」**，格式 `XXXX-YYMMNNNNNN`：
- `XXXX`：客户唯一四位码（系统按客户首见递增分配，`0001..9999`）
- `YYMM`：发号当时年月
- `NNNNNN`：6 位流水，在 `(XXXX, YYMM)` 前缀下从 `000001` 递增

**三个料号的最终语义**（承载于 `material_customer_map`）：

| 料号 | 字段 | 含义 | 基数/唯一 |
|---|---|---|---|
| **报价料号** | `material_no`（QUOTE 行） | 本系统唯一料号（报价侧料号口径） | 全局唯一（QUOTE 内）；**↔ 客户料号 1:1** |
| 生产料号 | `production_no`（新增） | 对接 ERP 的料号 | 可空、后补；**↔ 报价料号 1:N** |
| 客户料号 | `customer_product_no` | 客户方产品料号 | ↔ 报价料号 1:1 |

**Spec 1 范围**：客户表结构 + 发号服务 + 报价侧**全部 8 个发号 handler 改发报价料号（取消 9 字头）** + 通道① Q02 relabel + P05 打 PRICING 标 + **受影响读点/视图加 `system_type` 过滤** + 存量分类回填与唯一性 + 单测。
**不在 Spec 1**：选配 `CFG-` 统一（Spec 2）、维护页面（Spec 3）。

### 关键决策留档（与用户逐条确认）

| # | 决策 | 结论 |
|---|---|---|
| 报价料号 ↔ 客户料号 | **1:1**（一个报价料号只对一个客户料号） | 1:1 |
| 生产料号 ↔ 报价料号 | 一个生产料号可对多个报价料号 | 1:N |
| 报价料号全局唯一 | 同一报价料号不跨两个客户 | 是 |
| pricing/quote 分离 | 加 `system_type` 维度按行隔离（方案 S-A） | S-A |
| 通道① Q02 | 用文件既有报价料号（现"宏丰料号"列 relabel），不 mint（Align-A） | Align-A |
| **9 字头取消范围** | **报价侧全部 `resolve()` 调用点（8 handler），名匹配不到就发报价料号** | 全量取消 |
| 生产料号 | 报价侧可空，后补 | 可空 |
| 存量报价料号 | 不回填/不改写，现值即报价料号；旧 9 字头/CFG- 号保留为合法历史号 | 不回填 |
| 客户四位码 | 系统按客户首见递增分配、持久化 | 自动分配 |
| 流水范围 | 每 `(客户码, YYMM)` 独立、从 `000001` 递增 | 每客户每月 |

## 1. 数据模型

### 1.1 `material_customer_map` 改造（Flyway 迁移）

现状（V218）：唯一键 `(material_no, customer_no, customer_product_no)`，`customer_product_no NOT NULL`。P05（核价）与 Q02（报价）**共用同表同列**。

**DDL 改动**：

1. `ADD COLUMN system_type VARCHAR(20)` —— `QUOTE`/`PRICING`，按行隔离两侧语义。存量回填见 §5.1，回填后设 `NOT NULL`。
2. `ADD COLUMN production_no VARCHAR(20) NULL` —— 生产料号（ERP，QUOTE 行后补；PRICING 行不用）。
3. **`ALTER COLUMN customer_product_no DROP NOT NULL`**（Blocker-3）—— 通道②/费用表 mint 出的 BOM 组件登记行无客户料号，须允许 NULL。
4. **唯一键**：`DROP` 旧 `uq_material_customer_map`；建两个索引：
   - `uq_mcm_composite` = `UNIQUE (system_type, material_no, customer_no, customer_product_no) NULLS NOT DISTINCT`（PG16；覆盖 PRICING 生产料号跨客户重复 + QUOTE 行）。
   - `uq_mcm_quote_no` = `UNIQUE (material_no) WHERE system_type='QUOTE'`（报价料号全局唯一；因 ↔ 客户料号 1:1，单列即足，不与复合键矛盾）。
   - 可选加固 1:1（从客户料号侧）：`UNIQUE (system_type, customer_no, customer_product_no) WHERE system_type='QUOTE' AND customer_product_no IS NOT NULL` —— 保证"同客户同客户料号 ↔ 唯一报价料号"。
   - 建 `uq_mcm_quote_no` 前先跑 §5.2 校验，重复则迁移显式失败并打印冲突行。

**列语义对照**（改造后）：

| 列 | QUOTE 行 | PRICING 行 |
|---|---|---|
| `system_type` | `QUOTE` | `PRICING` |
| `material_no` | 报价料号（系统唯一料号） | 生产料号（现状不变） |
| `production_no` | 生产料号（可空·后补） | 不用 |
| `customer_product_no` | 客户料号（通道②组件登记行为 NULL） | 客户料号 |

### 1.2 发号服务用表

```sql
CREATE TABLE quote_customer_code (
  customer_no VARCHAR(20) PRIMARY KEY,
  code        CHAR(4)     NOT NULL UNIQUE,     -- 0001..9999
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);
CREATE SEQUENCE quote_customer_code_seq START 1;

CREATE TABLE quote_material_no_seq (
  customer_code CHAR(4) NOT NULL,
  year_month    CHAR(4) NOT NULL,              -- YYMM
  last_serial   INTEGER NOT NULL DEFAULT 0,
  PRIMARY KEY (customer_code, year_month)
);
```

> **报价料号的权威登记 = `material_customer_map` 的 QUOTE 行**（`material_no` 全局唯一）。不额外建登记表。可行的前提是 §3 把 Q02 的先删后插收窄到"仅客户料号映射行"，使 mint 登记的组件行（`customer_product_no IS NULL`）不被重导删除（见 §3 对 Major-8 的处理）。

## 2. 发号服务 `QuoteMaterialNoAllocator`

单一发号算法归处，entry-agnostic。Spec 1 建好 + 单测；调用方：§4 全部 8 个 handler（经 `MaterialNoResolver` 统一入口）。选配（Spec 2）后续接入。

### 2.1 接口

```java
String getOrAllocateCustomerCode(String customerNo);          // 取或分配客户四位码（首见分配、之后固定）
String mintAndRegister(String customerNo, String yyMm);       // 铸新报价料号 + 登记 material_customer_map QUOTE 行
```

### 2.2 客户四位码分配（并发安全）

```
SELECT code FROM quote_customer_code WHERE customer_no=:c;               -- 命中直接返回
-- 未命中：
code := lpad(nextval('quote_customer_code_seq')::text, 4, '0');
INSERT INTO quote_customer_code(customer_no, code) VALUES(:c,:code) ON CONFLICT (customer_no) DO NOTHING;
-- 竞态败者再 SELECT 取既有 code（浪费一个序列值＝可接受空洞）。
if code > '9999' → 抛「客户四位码枯竭(>9999)」。
```

### 2.3 流水发号 + 登记（原子）

```sql
-- ① 原子自增流水
INSERT INTO quote_material_no_seq(customer_code, year_month, last_serial)
VALUES (:code, :ym, 1)
ON CONFLICT (customer_code, year_month)
DO UPDATE SET last_serial = quote_material_no_seq.last_serial + 1
RETURNING last_serial;                       -- serial > 999999 → 抛「单客户单月流水溢出」
-- report := code || '-' || ym || lpad(serial::text,6,'0')   （总长 15）

-- ② 登记 material_customer_map QUOTE 行（组件级，无客户料号）
INSERT INTO material_customer_map(system_type, material_no, customer_no, customer_product_no, production_no, created_at, updated_at)
VALUES ('QUOTE', :report, :customerNo, NULL, NULL, NOW(), NOW());
```

> 用 `INSERT..ON CONFLICT DO UPDATE RETURNING` 单语句原子自增，取代旧 9 字头的 `advisory lock + MAX+1`。

## 3. 通道①：报价侧客户料号登记（Q02）+ Repository 改造

**`Q02CustomerMapHandler`（Sheet「客户料号与宏丰料号的关系」）**：
- 现「宏丰料号」列 = **报价料号**（relabel；兼容读取：优先「报价料号」表头，回退「宏丰料号」）。值落 `material_no`，`system_type='QUOTE'`，**不 mint**（Align-A）；`production_no` 留空。
- 「客户产品编号」→ `customer_product_no`（客户料号，Q02 要求非空——保持现状）。
- 幂等：报价料号 ↔ 客户料号 1:1 且 `material_no` 全局唯一 → **upsert 冲突目标 = `(material_no) WHERE system_type='QUOTE'`**（QUOTE 专用），有则更新、无则插。

**`deleteByCustomerNo` 收窄（一并修 Major-4 + Major-8）**：
- 现状 `DELETE ... WHERE customer_no=:c`（无维度）。改为参数化：Q02 先删后插只删 `WHERE customer_no=:c AND system_type='QUOTE' AND customer_product_no IS NOT NULL`。
- 效果：① 不误删 PRICING 行（M4）；② 保住 §2/§4 mint 登记的组件行（`customer_product_no IS NULL`，M8）——它们不属 Q02 sheet 的"客户料号映射全集"，不应被 replace 波及。
- 依据：Q02 sheet 行 `customer_product_no` 恒非空（`Q02CustomerMapHandler` 必填校验），故 `IS NOT NULL` 是可靠判别。

**`MaterialCustomerMapRepository` 联动改造（Minor-9，逐处点名）**：
- `MapRow`：增 `systemType`、`productionNo` 两字段。
- `coalesceOver` 折叠 dedup 键：`List.of(materialNo, customerNo, customerProductNo)` → **加 `systemType`**，否则跨 system_type 行被错误折叠。
- `upsertChunk`：11 列硬编码 SQL → 13 列；`b = i*11` 步长改 `i*13`；VALUES 占位、参数绑定同步。
- `production_no` 用 `COALESCE(EXCLUDED.production_no, material_customer_map.production_no)`，后补的生产料号不被重导清空。

**P05（核价）**：`P05CustomerMapHandler` 写入行打 `system_type='PRICING'`；不受报价料号影响。

## 4. 报价侧发号统一：8 个 handler 取消 9 字头（Blocker-1）

`MaterialNoResolver.resolve()` 的生成分支（9 字头）当前被 **8 个报价 handler** 调用：
`MaterialBomMergeHandler`(投入料号:76 / 组成件料号:105)、`Q04ElementBomHandler`:61、`Q06FixedProcessFeeHandler`:61、`Q07IncomingOtherFeeHandler`:58、`Q08IncomingAnnualDiscountHandler`:57、`Q09IncomingRecoveryHandler`:55、`Q10SelfProcessFeeHandler`:62、`Q13ComponentOtherFeeHandler`:59。（`Q05ElementRecoveryHandler`:58 用 `resolveMatchOnly`，只匹配不生成 → 不改。）

**统一改造：把发号收敛进 `MaterialNoResolver`，一处改、8 处受益**：

1. `BatchState` 增 `customerNo`、`yyMm` 两字段（每次导入构造时由 handler 从 `ctx.customerNo` + 导入年月注入）。
2. `resolve(料号, 名称, batch)` 算法改为：
   ```
   料号有值 → 返回（文件值即报价料号）；
   料号空+有名 → 先按名匹配既有报价料号（findFirstByMaterialName / batch.nameToNo 缓存复用）；
                 命中复用；未命中 → allocator.mintAndRegister(batch.customerNo, batch.yyMm)（取代 9 字头）；
   都空 → 抛 MaterialNoUnresolvableException（同现状）。
   ```
3. 删除/停用 `generateNextMaterialNo`（9 字头）；`maxNineLeadingMaterialNo`/`lockForMaterialNoGeneration` 仅在确认无其它调用后清理（实现时 `codegraph_callers` 核实非报价侧无引用）。
4. 8 个 handler 唯一改动 = 构造 `BatchState` 时传 `customerNo`+`yyMm`；`resolve` 调用签名不变。
5. `material_master` 名称→料号复用副作用保留（各 handler 原有 upsert 不动）；mint 出的报价料号既进 `material_master`（经 handler 既有 upsert）又进 `material_customer_map`（经 §2.3 登记），两者一致。

> **通道②"查不到即登记"**（Q4=a）由上面第 2 步覆盖：料号有值时若 `material_customer_map` 无对应 QUOTE 行，`resolve` 额外"确保登记"一行（`material_no`=报价料号, `system_type='QUOTE'`, `customer_no=batch.customerNo`, `customer_product_no=NULL`, `production_no=NULL`）。
> **`isCfg` 拒绝逻辑本 Spec 保留不动**（选配统一在 Spec 2）。
> **协议级**：改 `MaterialBomMergeHandler` 等报价导入核心 → 跑报价 E2E（`quotation-flow.spec.ts`），对照 AP-51 行数纪律。

## 5. 受影响读点/视图（Major-5，本 Spec 交付项，非调研）

加 `system_type` 后，下列读点会串到错误一侧或行扇出（AP-22 家族），**须在本 Spec 迁移中补 `system_type='QUOTE'` 过滤**：

## 6. 存量迁移

### 6.1 `system_type` 回填（Major-6：默认方向按信号判 QUOTE）

存量行按**可观测列信号**分类（比"无标记默认 PRICING"可靠——后者会把真报价行错标、致 §5 读点对存量全返空）：

- **判 QUOTE**：`customer_material_name` / `customer_drawing_no` / `payment_method` / `base_currency` / `quote_currency` / `exchange_rate` 任一非空，**或** `seq_no IS NULL`（Q02「报价表无项次列」恒 null；P05 `seq_no` 来自「项次」）。
- 依据：`P05CustomerMapHandler` 这些列全传 `null`（:55-61）。
- 回填 SQL（示意）：
  ```sql
  UPDATE material_customer_map SET system_type =
    CASE WHEN customer_material_name IS NOT NULL OR customer_drawing_no IS NOT NULL
           OR payment_method IS NOT NULL OR base_currency IS NOT NULL
           OR quote_currency IS NOT NULL OR exchange_rate IS NOT NULL
           OR seq_no IS NULL
         THEN 'QUOTE' ELSE 'PRICING' END
  WHERE system_type IS NULL;
  ```

### 6.2 疑似双源合并行（Major-7）

P05 与 Q02 今天共用唯一键 `(material_no, customer_no, customer_product_no)`：同一三元组若在报价簿与核价簿都出现，历史会 `ON CONFLICT DO UPDATE` **合并成同一物理行**，无法从单行还原两侧。

- **实现前必做**：跑核对 SQL 量化这类行（同时具备 QUOTE 特征列**且** `seq_no` 来自核价项次等冲突信号）：
  ```sql
  -- 量化「既像 QUOTE 又像 PRICING」的可疑合并行
  SELECT count(*) FROM material_customer_map
  WHERE (customer_material_name IS NOT NULL OR base_currency IS NOT NULL /*QUOTE 特征*/)
    AND seq_no IS NOT NULL /*PRICING 特征*/;
  ```
- 若量非零：按业务确认拆分策略（复制成 QUOTE+PRICING 两行，或产出人工核对清单）。若为 0：§6.1 分类即安全。**此项是实现计划的第一个前置任务。**

### 6.3 报价料号唯一性校验（建 `uq_mcm_quote_no` 前）

```sql
SELECT material_no, count(*) FROM material_customer_map
WHERE system_type='QUOTE' GROUP BY material_no HAVING count(*)>1;   -- 期望 0 行，非 0 则迁移失败+打印
```

### 6.4 存量号处理
存量 `material_no`（含旧 9 字头/CFG- 值）**保留不动**，视为合法历史报价料号；不回填、不改写、不重发。`production_no` 存量全空、后补。

## 7. 边界与上限

| 情形 | 处理 |
|---|---|
| 客户数 > 9999（四位码枯竭） | 发号显式报错 |
| 单客户单月流水 > 999999 | 发号显式报错 |
| `customer_product_no` NULL（通道②组件行） | 唯一键 `NULLS NOT DISTINCT`；`customer_product_no` 已 DROP NOT NULL |
| PRICING 行 `material_no`（生产料号）跨客户重复 | 允许（不加全局唯一） |
| 报价料号跨客户串号（文件把 A 客户号写进 B 客户 BOM） | `uq_mcm_quote_no` 全局唯一 → 登记时命中别客户既有行报错，不静默串号 |

## 8. 测试

**`QuoteMaterialNoAllocator` 单测**：四位码首见分配/复用/`>9999` 报错；mint 格式 15 位正确；同 `(码,YYMM)` 流水 `000001→000002`、跨 YYMM 归 `000001`、`>999999` 报错；`mintAndRegister` 落 `material_customer_map` QUOTE 行（`customer_product_no`/`production_no` NULL）。

**8 handler 发号集成**（抽样 `MaterialBomMergeHandler` + 至少 1 个费用 handler 如 `Q07`）：料号有值→解析并确保登记；无号有名+无既有同名→mint 报价料号（**非 9 字头**）并登记；无号有名+有既有同名→复用；同批同名→同号；都空→报错。

**通道① Q02 集成**：写入行 `system_type='QUOTE'`；relabel 列读取；先删后插**只删 `customer_product_no IS NOT NULL` 的 QUOTE 行**，验证 PRICING 行与组件登记行（`customer_product_no IS NULL`）**存活**；报价料号重导不变。

**P05 集成**：写入行 `system_type='PRICING'`；Q02 replace 不误删其行。

**Repository**：upsert 冲突键含 `system_type`；`coalesceOver` dedup 键含 `system_type`；`production_no` COALESCE 不被置空。

**读点/视图**：`CustomerPartCandidateService`/`V6QuotationCommitService`/V255 加 `system_type='QUOTE'` 后，对混入 PRICING 行的场景不再扇出/串号。

**迁移**：`system_type` 回填后无 NULL；§6.1 信号分类正确；§6.2 可疑行计数；§6.3 QUOTE 报价料号无重复。

> 协议级（改 `MaterialBomMergeHandler` + V255 视图）→ 跑 `quotation-flow.spec.ts` E2E + 视图重建后重启 Quarkus 自检。
