# 组装加工费（capacity）料号级整组升版 — 设计方案

**日期**：2026-06-02
**范围**：报价基础数据导入 → 组装加工费（Q14 / `capacity` 表）版本化逻辑修复 + 解析器重名表头防呆
**状态**：设计已与用户逐节确认，待写实现计划

---

## 1. 背景与根因

用户从"基础数据导入"导入 Excel `组装加工费` Sheet 后，发现工序数据与库中不同，但**未触发数据升版**。逐层排查定位到两个问题：

### 1.1 根因 A：解析器重名表头静默覆盖丢列（直接症状根因）

`组装加工费` Sheet 有**两列同名表头"组装工序"**：第一列是工序编码（`Z350`/`Z029`），第二列是工序名称（`焊接`/`铆接`）。

- `ExcelParserService.parseSheet`：表头 `headerMap` 是 `Map<列号,列名>`，但数据行 `cells` 是 `Map<列名,值>`，按列号顺序写入。两列同名 → `cells.put("组装工序", Z350)` 先写、`cells.put("组装工序", 焊接)` 后写**覆盖**。
- 结果 `row.getStr("组装工序")` 永远返回**第二列名称（焊接）**，工序编码 `Z350` 从未进入系统。
- 历史那次导入与本次导入**都**把 `process_no` 读成了 `焊接/铆接`，名称未变 → 系统认为无变化 → 不升版。库里的 `焊接/铆接` 脏数据，本身就是同一个解析 bug 上一次导入的产物。

> 用户已确认：源 Excel 的两列表头**写错了名字**（本应是"组装工序编码 / 组装工序名称"两个不同表头），数据本身没错。但系统**不应静默吞掉一整列数据**，必须防呆。

### 1.2 根因 B：升版分组粒度与业务需求冲突（架构性根因）

`Q14AssemblyProcessFeeHandler` 的版本分组键 `groupKey = (material_no, process_no, resource_group_no)`，**工序编码在分组键里**。

- 改一道工序编码（`焊接`→`Z350`）= 另起一个全新分组，旧组永远不会被"升版下线"。
- 用户重导后的实测数据印证：旧 `焊接/铆接`(2000) 与新 `Z350/Z029`(2000) **4 行全 is_current=true 并存**，旧组未下线。

`VersionedV6Writer.writeVersionedGroup` 中 `flip()`（旧组下线）只对**同一 groupKey** 生效；`(料号,焊接) ≠ (料号,Z350)` → 跳过 flip。

## 2. 业务需求（用户确认）

### 2.1 组装加工费升版规则

- **升版粒度 = 料号级整组**：同一料号的所有组装工序作为一组。
- **升版触发条件**：仅当 (a) **工序数量**变化，或 (c) **工序编码/名称**变化 → 整组升版（版本号 +1）、旧版整组 `is_current=false`（**保留为历史，不删除**）。
- **不触发升版**：(b) 组装加工费金额、(d) 货币 / 计价单位 / 拒收率(不良率) 变化 → **原地更新当前组的值、版本号不变**（避免改对一个错价格永远不生效）。

### 2.2 一致性边界（用户选定"方案 1"）

"金额/价格变化不升版、仅结构变化才升版"的规则**只应用到组装加工费（Q14 / `capacity`）**。

其余所有版本化表**保持现状**：任何 `content` 变化（含价格）都升版——因为对价格表而言，价格变化正是要留版本追溯的内容（报价历史/审计）。用户已确认组装加工费是"工序结构定版、价格随版原地浮动不另立新版"这一特殊语义。

## 3. 设计

### 3.1 设计 ①：升版分组改为料号级

`Q14AssemblyProcessFeeHandler`：

- `groupKey` 从 `(material_no, process_no, resource_group_no)` 改为 **`(material_no, resource_group_no)`**。
- `process_no` 下沉进 `content`（写入列）。
- 聚合逻辑：从"每 (material, process) 一行、后写覆盖"改为"**同一料号的所有工序行汇总成一组 `newRows`** 整体写入"。

**唯一约束兼容性**（已核对 V220 / V277）：
- `uq_capacity = UNIQUE(material_no, process_no, resource_group_no, COALESCE(calc_version,''))` → 同 (料号,资源组) 下多 process_no、多版本互不冲突 ✓
- V277 局部唯一索引 `(material_no, process_no, resource_group_no) WHERE is_current=TRUE` → 每道工序仅一行 current；升版后旧行 `is_current=false` 退出索引 ✓
- **减工序场景**：`flip()` 按 `(material, resource_group)` 整组下线全部 current 行 → 新版只插存活的工序 → 被删工序的旧行自动 `is_current=false`，正确从 current 消失（这是 per-process 分组无法做到、而料号级分组天然解决的）。

### 3.2 设计 ②：升版触发列与写入列分离（`VersionedV6Writer` 增强）

给 `VersionedGroupSpec` 增加**可选**字段 `versionTriggerColumns`（升版触发列）：

- **向后兼容**：`versionTriggerColumns == null` 时，触发列 = `contentColumns`（即所有其它表保持现状，行为不变）。
- Q14 传入 `versionTriggerColumns = [process_no, seq_no]`（工序编码 + 项次/数量），`contentColumns = 全部字段`（含 `fixed_cost, currency, capacity_unit, default_defect_rate, is_effective, process_no, seq_no, production_type`）。
- 约束：`versionTriggerColumns ⊆ contentColumns`（触发列必须是已加载/已写入的列），且与 `groupKeyColumns` 不重叠（沿用 I2 校验）。

`writeVersionedGroup` 决策逻辑改为：

1. `loadCurrentGroup` 载入当前生效组的 `contentColumns`。
2. 比较 `existing` vs `newRows`：
   - **触发列多重集相等**（结构未变）：
     - 若 `contentColumns` 也全等 → 完全相同，**no-op**，返回当前版本。
     - 否则（仅 b/d 变化）→ **原地更新**：复用当前版本号，覆盖当前生效组的值（实现可选"删除当前组 current 行后以同版本号 + `is_current=true` 重插"，或逐行 UPDATE），**不 flip、不升版号**。
   - **触发列多重集不等**（结构变化）→ 升版：`nextVersionOf` 版本号 +1、`flip()` 旧组整组 `is_current=false`（保留为历史）、写新版组。

> 注：原 `versionTriggerColumns==null` 路径与现行 `multisetEqual(existing, newRows, contentColumns)` 完全等价，确保其余 13 个 Handler 行为零变化。

### 3.3 设计 ③：解析器重名表头防呆

`ExcelParserService.parseSheet`：构建 `headerMap` 时检测**同一 Sheet 内归一化后重复的表头名**：

- 一旦发现重复表头 → **抛异常 / 记 `_sheet_` 级错误**，该 Sheet 标记失败但不影响其它 Sheet（`QuoteImportService` 已是 per-Sheet `REQUIRES_NEW` 隔离）。
- 错误信息明确："表头列名重复：组装工序（列 3、列 4）"，让问题在导入时立即暴露，杜绝静默写错。

### 3.4 设计 ④：修正当前已存在的脏数据（一次性 SQL）

```sql
-- 1) 旧版 焊接/铆接 整组下线（保留为 2000 版历史）
UPDATE capacity SET is_current = false
WHERE id IN (
  '7accbaff-39fa-4833-8a98-a196b402f746',  -- 焊接
  '10983bb5-6fee-47f0-969d-663d0673bc90'   -- 铆接
);
-- 2) 新导入 Z350/Z029 升版到 2001
UPDATE capacity SET calc_version = '2001'
WHERE id IN (
  '51ecedb1-02de-4557-80ae-f331b43b3a56',  -- Z350
  '55d8b2e2-33cc-4fed-a50a-83182cab1475'   -- Z029
);
```

> 说明：焊接/铆接 历史只能靠保留现有行实现（新 Excel 已无这两道名称行，重导不会重建）。执行前先 `SELECT` 核对，确认 4 行均属料号 `3120012004`。SQL 走 Flyway 迁移（`V***__fix_assembly_capacity_dirty_version.sql`），不手工 psql。

### 3.5 设计 ⑤：测试

**`VersionedV6Writer` 单测（含 `versionTriggerColumns`）**：
- (a) 工序数量变（增/减一道）→ 升版号 +1、旧组整组 `is_current=false`、新版含正确行数。
- (c) 工序编码变 → 升版。
- (b/d) 仅金额/货币/计价单位/拒收率变 → 版本号不变、值已更新、无新历史行。
- 完全不变 → 复用版本、不写。
- **回归**：`versionTriggerColumns==null`（其它表）行为与改前一致。

**`Q14AssemblyProcessFeeHandlerTest`**：补料号级整组场景（多工序汇总成一组、升版/原地更新分支）。

**`ExcelParserService` 单测**：重名表头 → 报错（`_sheet_` 级错误）。

**E2E**：导入 → 报价单渲染该料号组装加工费 Tab，确认升版后取到 `Z350/Z029` 当前版数据、旧版不再出现在 current。

## 4. 影响面与不改动项

**改动文件**：
- `VersionedGroupSpec.java`（加可选字段 `versionTriggerColumns`）
- `VersionedV6Writer.java`（触发列/写入列分离 + 原地更新分支）
- `Q14AssemblyProcessFeeHandler.java`（groupKey 料号级 + 聚合 + 传 versionTriggerColumns）
- `ExcelParserService.java`（重名表头防呆）
- 新 Flyway 迁移（脏数据修正）
- 对应单测 + E2E

**明确不改动**：
- 其余 13 个 Handler（Q01/Q03/Q04/Q06–Q13/Q15–Q19）的升版规则——`versionTriggerColumns` 默认 null 保持现行"任何 content 变化即升版"。
- `unit_price` / `plating_scheme` / BOM 主从表的分组键与 content。

## 5. 验收标准

1. 导入含重名表头的 Excel → 该 Sheet 报错，错误信息含重复列名（不再静默吞列）。
2. 表头改对后导入 → 工序编码 `Z350/Z029` 正确入库。
3. 改工序编码 / 增减工序 → 整组升版（版本号 +1）、旧版整组 `is_current=false` 保留为历史。
4. 仅改金额/货币/计价单位/拒收率 → 版本号不变、当前组值已更新。
5. 当前 4 行脏数据修正为：`Z350/Z029`@2001 current、`焊接/铆接`@2000 `is_current=false`。
6. 其余基础资料表升版行为零变化（回归单测通过）。
7. 全套自检：`tsc` 0 错、后端 health 200、Flyway success=t、E2E `加载中=0`。
