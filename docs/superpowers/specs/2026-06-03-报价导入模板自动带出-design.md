# 报价导入向导 · 模板自动带出 设计

- 日期:2026-06-03
- 状态:已通过 brainstorm,待实现
- 入口流程:基础数据导入 → 选择客户和报价数据文件 → 导入 → **选择报价和核价模板**(本设计针对最后这一步)

## 1. 背景与目标

当前 `QuoteBasicDataImportV6Drawer` 导入成功后进入 Step 2(`QuotationCreateForm`),用户每次都要重复选择产品分类、报价模板、核价模板。痛点集中在:报价模板存在多候选(客户专属 + 通用,即 `MIXED`)时,现有逻辑**故意留空**让用户手选。

**目标**:进入选模板这一步时,自动带出
- 该客户**上次使用的报价模板线**的最新版本;
- 当前可用的**最新版本核价模板**;
- 以及上次使用的**产品分类**;

三项预填、可改,消灭重复手选,同时保留一次「可见的确认」避免无感生成错报价单。

## 2. 选择规则(已与用户逐条敲定)

### 2.1 报价模板

主规则唯一:**有历史就跟随上次使用的那条模板线**;「客户专属优先」仅作为**无历史时的兜底**,两条规则不交叉。

1. 查该客户 `customerTemplateId` 非空的报价单,按 `createdAt DESC` 取第一张,确认其模板仍存在 → `lastTemplate`。
2. **有历史**:
   - `categoryId = lastTemplate.categoryId`(报价单表无 `category_id`,从模板反推);
   - `seriesId = lastTemplate.templateSeriesId`;取 `templateSeriesId=seriesId AND status='PUBLISHED' ORDER BY publishedAt DESC` 的第一条 = 该线最新发布版本 → `source = LAST_USED`;
   - 若该线已无 PUBLISHED 版本(整条归档)→ 落入兜底(沿用上面的 `categoryId`)。
3. **无历史**:`categoryId =「默认分类」.id`,落入兜底。
4. **兜底**:对 `(customerId, categoryId)` 调用现有 `TemplateService.matchCustomerQuoteTemplate`:
   - 客户专属非空 → 取最新发布(`source = CUSTOMER_SPECIFIC_FALLBACK`);
   - 否则通用非空 → 取最新发布(`source = GENERAL_FALLBACK`);
   - 否则 `source = NONE`(留空,沿用现有「未找到模板」提示)。

> 设计要点:`lastTemplate` 上次用的就算是「通用」线,也照样跟随该通用线(尊重历史);客户专属只有在完全无历史时才作为默认。

### 2.2 核价模板(独立,不做记忆)

按分类取最新发布的可用核价模板,客户专属优先于通用:

```
categoryId = ? AND templateKind = 'COSTING' AND status = 'PUBLISHED'
AND (customerId = ? OR customerId IS NULL)
ORDER BY (CASE WHEN customerId = ? THEN 0 ELSE 1 END), publishedAt DESC
```
取第一条 → `source = CUSTOMER_SPECIFIC / GENERAL / NONE`。

### 2.3 产品分类

从 §2.1 的 `lastTemplate.categoryId` 推导;无历史退「默认分类」。分类是模板匹配的源头,一并自动带出才能让链路闭环(否则改分类会把模板重置回手选)。

## 3. 架构与数据流

采用「新增后端自动默认端点」(路线 1):所有「最近报价单 / 模板线 / 最新版本」推导集中在后端一处,前端进 Step 2 调一次、预填表单。

```
QuoteBasicDataImportV6Drawer (导入成功, step→2)
        │  GET /api/cpq/templates/auto-defaults?customerId
        ▼
TemplateResource.autoDefaults → TemplateService.computeAutoDefaults(customerId)
        │  (复用 matchCustomerQuoteTemplate 做兜底)
        ▼
QuoteImportAutoDefaults DTO  →  预填 createForm 三个 ID + hint 文案
        ▼
QuotationCreateForm 现有 effect 的 "preserve if present" 逻辑保留预填值
```

## 4. 后端设计

### 4.1 端点

`GET /api/cpq/templates/auto-defaults?customerId={uuid}` → `QuoteImportAutoDefaults`。

### 4.2 DTO

```java
class QuoteImportAutoDefaults {
  UUID   categoryId;
  String categoryName;

  UUID   customerTemplateId;
  UUID   customerTemplateSeriesId;
  String customerTemplateName;
  String customerTemplateVersion;
  String customerTemplateSource;   // LAST_USED | CUSTOMER_SPECIFIC_FALLBACK | GENERAL_FALLBACK | NONE

  UUID   costingTemplateId;
  String costingTemplateName;
  String costingTemplateVersion;
  String costingTemplateSource;     // CUSTOMER_SPECIFIC | GENERAL | NONE
}
```

### 4.3 `TemplateService.computeAutoDefaults(UUID customerId)`

按 §2 规则实现:
1. 最近报价单查询(`Quotation` 关联 `Template`,过滤 `customerTemplateId` 非空、模板存在,`createdAt DESC` 取一)。
2. 有历史 → 取分类 + 模板线最新发布版本;线归档 → 兜底。
3. 无历史 → 默认分类 + 兜底。
4. 兜底复用 `matchCustomerQuoteTemplate(customerId, categoryId)`。
5. 核价模板按 §2.2 查询。
6. 组装 DTO,各 `source` 据来源赋值。

数据模型支撑(已核实):
- `Template.templateSeriesId`(UUID,模板线标识,同线各版本共享);
- `Template.categoryId / customerId / status / publishedAt / version`;
- `Quotation.customerId / customerTemplateId / costingCardTemplateId / createdAt`(报价单**无** `category_id`,故分类从模板反推)。

## 5. 前端设计

### 5.1 `QuoteBasicDataImportV6Drawer.tsx`

导入成功、`step` 切到 2 的分支里调用端点:
```ts
const d = await api.get('/templates/auto-defaults', { params: { customerId } });
setCreateForm({
  name: `${customerName} 报价单`,
  categoryId: d.categoryId,
  customerTemplateId: d.customerTemplateId,
  costingTemplateId: d.costingTemplateId,
});
setAutoHints({
  customerTemplateHint: hintOf(d.customerTemplateSource, d.customerTemplateVersion),
  costingTemplateHint:  hintOf(d.costingTemplateSource,  d.costingTemplateVersion),
});
```
端点失败 → 静默降级为现状(不阻断导入,不预填)。

### 5.2 `QuotationCreateForm.tsx`

- 新增可选 props `customerTemplateHint?: string` / `costingTemplateHint?: string`,在对应 Select 标签旁渲染一行小灰字提示。
- `readOnly`(编辑态)不显示提示。
- 现有三个 `useEffect`(默认分类 / `match-customer-quote` / 核价列表)**保持不动**:其 "若已选 ID 在结果里就保留、不覆盖" 的 hotfix 逻辑天然保留预填值。**不改这三个 effect = 不触碰 AP-31 / AP-44 字段类型联动协议。**

source → 文案映射:

| source | 文案 |
| --- | --- |
| LAST_USED | 上次使用 · 最新版 v{version} |
| CUSTOMER_SPECIFIC_FALLBACK | 无历史 · 客户专属最新 |
| GENERAL_FALLBACK | 无历史 · 通用最新 |
| 核价 CUSTOMER_SPECIFIC | 客户专属 · 最新 |
| 核价 GENERAL | 通用 · 最新 |
| NONE | (不显示提示,沿用现有「未找到模板」Alert) |

### 5.3 交互

进入 Step 2 即看到分类/报价模板/核价模板三项已填 + 提示;可临时改任意项(改分类触发现有 match 重跑,符合预期);点「生成报价单」走原有 commit 流程。

## 6. 错误处理与降级

- 端点异常 / 超时 → 前端静默降级为当前手选行为,导入与建单不受影响。
- 各 `source = NONE` → 沿用现有「未找到适用模板」Alert。
- 上次模板线整条归档 → 后端已在 §2.1 步骤 2 内退兜底,前端无需特判。

## 7. 测试

### 7.1 后端单测(`TemplateService` 层)

1. 有历史 → 取该线最新发布版本(`LAST_USED`);
2. 该线整条归档 → 退兜底;
3. 无历史 + 多条客户专属 → 取最新发布客户专属(`CUSTOMER_SPECIFIC_FALLBACK`);
4. 无历史 + 仅通用 → 通用(`GENERAL_FALLBACK`);
5. 无任何模板 → `NONE`;
6. 核价:客户专属 / 通用 / 无 三态。

### 7.2 前端自检

- `cd cpq-frontend && npx tsc --noEmit -p tsconfig.json` → 0 错;
- 改动的 `QuoteBasicDataImportV6Drawer.tsx` / `QuotationCreateForm.tsx` 各走 Vite `curl` 200。

### 7.3 流程验证(手动)

真实导入一次 → 进 Step 2 看三项是否自动带出且与该客户历史一致 → 改分类看是否正确重匹配。本次**不补**向导 Playwright E2E(改动未触碰 driver/snapshot 协议,且不在 CLAUDE.md 强制 E2E 文件清单内)。

### 7.4 回归

确认 `QuotationCreateForm` 在编辑态(`readOnly`)及非导入入口的复用调用方行为不变(新 props 可选,缺省即旧行为)。

## 8. 范围之外(YAGNI)

- 不新增「上次使用模板线」的持久化字段(从历史报价单推导即可,无需冗余存储)。
- 核价模板不做客户记忆。
- 不改 `match-customer-quote` 现有端点语义(新增独立端点,避免回归面)。
