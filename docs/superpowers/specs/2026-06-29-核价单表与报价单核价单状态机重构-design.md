## 1. 背景与目标

第一期把「核价管理」改造成**财务核价工作台**，落地精简版 `costing_order`（只存 `quotation_id + submitted_by + 时间`，状态派生、无单号、无数据克隆）。本轮按用户「完整状态需求单」升级为**完整核价单实体**并理顺状态机：

1. **新增完整核价单**：提交审批时把当前报价单数据**冻结**进核价单表（整份 DTO，含 enrich 后结构 + gvDefs；卡片/Excel 值依赖既有零计算快照），发**核价单号**，作为「提交审批的记录」+「核价单的基础数据来源」，可追溯通过/驳回记录、驳回原因、操作时间，且**历史单经报价单重做后仍 1:1 回看当时明细**。
2. **两侧状态联动且可分叉**：提交→核价单`待核价`；通过→两侧`已审核`；驳回→两侧`已驳回`；销售点编辑进向导→报价单`草稿`、核价单仍`已驳回`；销售撤回→报价单`草稿`、核价单`已撤回`。
3. **核价管理列表增强**：全状态过滤 + 报价单号搜索框 + 新增「退回原因」「修改时间」列。

### 本轮范围

- **财务只读核价**：查看核价单冻结数据 → 整单通过/驳回。**不切料号版本、不改数据、不重算**（切版本留第二期，依赖版本感知 BOM 闭包，BL-0005~0007）。
- **真冻结·务实版**（用户两轮拍板）：核价单自带冻结 DTO（结构 + gvDefs），值依赖既有 costing 卡片/Excel 快照；不做独立 path-cache 冻结。
- **不做**「核价单数据覆盖报价单读取层下沉」——本轮两侧不分叉（财务不改数据）；下沉留第二期切版本分叉时再做。

## 3. 状态模型

### 3.1 状态枚举与标签

**报价单 `quotation.status`（沿用现有枚举，仅改前端标签）：**

| 枚举码 | 旧标签 | 新标签 | 说明 |
|---|---|---|---|
| `DRAFT` | 草稿 | 草稿 | 不变 |
| `SUBMITTED` | 审批中 | **待核价** | 提交审批后 |
| `APPROVED` | 已批准 | **已审核** | 财务核价通过 |
| `COSTING_REJECTED` | 核价驳回 | **已驳回** | 财务核价驳回 |
| `REJECTED` | 已退回 | **客户已拒绝** | 客户拒绝已发送报价（`rejectByCustomer`），消歧 |
| `SENT` / `ACCEPTED` / `EXPIRED` / `CANCELLED` | — | 不变 | 不在本轮调整 |

> 枚举码全部沿用，**不动后端 `VALID_QUOTATION_STATUSES` 与 V304 的 `chk_q_status`**。只改前端 `statusConfig` 单点标签。

**核价单 `costing_order.status`（新增显式状态列，稳定英文码）：**

| 枚举码 | 标签 | 触发 |
|---|---|---|
| `PENDING` | 待核价 | 报价单提交审批 |
| `APPROVED` | 已审核 | 财务核价通过 |
| `REJECTED` | 已驳回 | 财务核价驳回 |
| `WITHDRAWN` | 已撤回 | 销售撤回 |

> **稳定英文码存库 + 前端单点映射中文**（规避第一期"英文枚举 vs 中文串"按钮永久禁用坑）。第一期 `deriveCostingStatus`（派生中文）**退役**；列表/详情读 `status` 列；DTO 一律返英文码。

### 3.2 状态转移表

| 动作 | 触发端点 | 报价单 status | 核价单 | 取哪条 | 副作用 |
|---|---|---|---|---|---|
| 提交审批 | `POST /quotations/{id}/submit` | → `SUBMITTED` | **新建一条** `PENDING` | 新建 | 后端组装 **frozen_dto** + 发号 + submitted_by + total |
| 财务通过 | `POST /quotations/{id}/costing-approve` | → `APPROVED` | → `APPROVED` | `findActive` | reviewed_by/at；`quotation_approval(COSTING_APPROVED)` |
| 财务驳回 | `POST /quotations/{id}/costing-reject` | → `COSTING_REJECTED` | → `REJECTED` | `findActive` | reject_reason + reviewed_by/at；`quotation_approval(COSTING_REJECTED, comment=原因)` |
| 销售点「编辑」进向导 | `POST /quotations/{id}/begin-edit` | `COSTING_REJECTED` → `DRAFT` | **不变**（仍 `REJECTED`）| — | 清报价单自身冻结快照（复用 unfreeze）；**不动 costing_order** |
| 销售撤回 | `POST /quotations/{id}/withdraw` | {`SUBMITTED`,`COSTING_REJECTED`,`APPROVED`} → `DRAFT` | → `WITHDRAWN` | **`findLatest`** | 清报价单自身冻结快照；`quotation_approval(WITHDRAWN)` |
| 重新提交 | `POST /quotations/{id}/submit` | → `SUBMITTED` | **再新建一条** `PENDING` | 新建 | 旧 `REJECTED`/`WITHDRAWN` 永久留存为历史 |
| 客户拒绝已发送 | `POST /quotations/{id}/reject-by-customer` | → `REJECTED` | — | — | 不在本轮调整 |

> **active 核价单** = `status in ('PENDING','APPROVED')` 那条（DB 部分唯一索引保证至多一条，§4.1·B3）。通过/驳回取 active。
> **撤回**用 `findLatestByQuotation`（createdAt desc，**含终态 REJECTED**）——B2 修正：撤回已驳回单时该条是 REJECTED，active 查不到，必须用 latest。

### 3.3 编辑入口收紧

「编辑」只对 `DRAFT` + `COSTING_REJECTED` 启用（按 `docs/列表操作规范.md`，禁用不隐藏、tooltip 给原因）：

- `DRAFT`：直接进向导。
- `COSTING_REJECTED`：点编辑 → 先 `await quotationService.beginEdit(id)` 翻 `DRAFT` → 再进向导。
- `SUBMITTED` / `APPROVED`：禁用，tooltip「请先撤回再编辑」。
- 其余（`SENT`/`ACCEPTED`/`EXPIRED`/`CANCELLED`/`REJECTED`）：不可编辑。

## 4. 数据模型（迁移 V305）

### 4.1 `costing_order` 表演进

```sql
-- V305: 核价单升级为完整实体 —— 解除一报价单一核价单、加单号/状态/原因/冻结DTO/总额/审核人

-- 1) 解除唯一约束（改为按提交累积多条）
ALTER TABLE costing_order DROP CONSTRAINT uq_costing_order_quotation;

-- 2) 新增列（v3：只 frozen_dto，无 frozen_path_cache）
ALTER TABLE costing_order
  ADD COLUMN costing_order_number VARCHAR(64),
  ADD COLUMN status               VARCHAR(32)   NOT NULL DEFAULT 'PENDING',
  ADD COLUMN reject_reason        TEXT,
  ADD COLUMN frozen_dto           JSONB,                       -- 整份 getById DTO（enrich 后结构 + gvDefs + 卡片/Excel 值快照）
  ADD COLUMN total_amount         NUMERIC(18,4),
  ADD COLUMN reviewed_by          UUID,
  ADD COLUMN reviewed_at          TIMESTAMPTZ,
  ADD COLUMN updated_at           TIMESTAMPTZ   NOT NULL DEFAULT now();

-- 3) 状态 CHECK
ALTER TABLE costing_order
  ADD CONSTRAINT chk_co_status CHECK (status IN ('PENDING','APPROVED','REJECTED','WITHDRAWN'));

-- 4) 核价单号序列 + 回填（B4：SENT/ACCEPTED/REJECTED/EXPIRED 均隐含已通过核价 → APPROVED）
CREATE SEQUENCE IF NOT EXISTS costing_order_number_seq START 1;
UPDATE costing_order co SET
  costing_order_number = 'HJ-' || to_char(co.created_at,'YYYYMMDD') || '-' || lpad(nextval('costing_order_number_seq')::text,4,'0'),
  status = CASE q.status
             WHEN 'SUBMITTED'        THEN 'PENDING'
             WHEN 'APPROVED'         THEN 'APPROVED'
             WHEN 'SENT'             THEN 'APPROVED'   -- B4
             WHEN 'ACCEPTED'         THEN 'APPROVED'   -- B4
             WHEN 'REJECTED'         THEN 'APPROVED'   -- B4 v3：客户拒绝隐含先已通过核价
             WHEN 'EXPIRED'          THEN 'APPROVED'   -- B4 v3：过期隐含先已通过核价
             WHEN 'COSTING_REJECTED' THEN 'REJECTED'
             ELSE 'WITHDRAWN' END                       -- DRAFT/CANCELLED
  FROM quotation q WHERE q.id = co.quotation_id;

ALTER TABLE costing_order ALTER COLUMN costing_order_number SET NOT NULL;
ALTER TABLE costing_order ADD CONSTRAINT uq_co_number UNIQUE (costing_order_number);

-- 5) B3：部分唯一索引 —— DB 层强约束"至多一条 active"，挡并发/重复提交双 PENDING
CREATE UNIQUE INDEX uq_co_active ON costing_order(quotation_id) WHERE status IN ('PENDING','APPROVED');

-- 6) 索引
CREATE INDEX idx_costing_order_status ON costing_order(status);
-- 保留既有 idx_costing_order_quotation
```

> **存量处理**：第一期 `costing_order` 现有行多为联调数据；上面 `UPDATE` 补号 + 按报价单当前态映射（B4 修正）；`frozen_dto` 留 NULL → 历史单详情对 NULL 降级显示「此单为旧版本，无冻结明细」，不报错。

### 4.2 `CostingOrder` 实体新增字段

```java
@Column(name = "costing_order_number", nullable = false) public String costingOrderNumber;
@Column(name = "status", nullable = false)               public String status = "PENDING";
@Column(name = "reject_reason", columnDefinition = "TEXT") public String rejectReason;
@Column(name = "frozen_dto", columnDefinition = "jsonb")  public String frozenDto;  // 大字段，list 查询禁投影（N5）
@Column(name = "total_amount", precision = 18, scale = 4) public BigDecimal totalAmount;
@Column(name = "reviewed_by")  public UUID reviewedBy;
@Column(name = "reviewed_at")  public OffsetDateTime reviewedAt;
@Column(name = "updated_at", nullable = false) public OffsetDateTime updatedAt;

// 通过/驳回取 active（部分唯一索引保证至多一条）
public static CostingOrder findActiveByQuotation(UUID qid) {
    return find("quotationId = ?1 and status in ('PENDING','APPROVED')", qid).firstResult();
}
// B2：撤回取最近一条（含终态 REJECTED）
public static CostingOrder findLatestByQuotation(UUID qid) {
    return find("quotationId = ?1 order by createdAt desc", qid).firstResult();
}
public static java.util.List<CostingOrder> findAllByQuotation(UUID qid) {
    return list("quotationId = ?1 order by createdAt desc", qid);
}
```

`@PreUpdate` 维护 `updatedAt`。

## 5. 真冻结机制（B1·务实版）

### 5.1 现有机制复用（大部分已天然冻结）

- **核价卡片值**：`quotation_line_item.costing_card_values` 已零计算冻结——`ReadonlyProductCard` 在 `useSnap=true`（该行有 `costingCardValues`）时**不发 batch-expand**（`useDriverExpansions(EMPTY_LINEITEMS)`），driver 行 / 公式结果 / 逐行 BASIC_DATA 全从快照 `baseRows[].driverRow` + `formulaResults` 读（`ReadonlyProductCard.tsx:240-287`，Phase4「详情页读快照零计算」成果）。
- **Excel 视图**：`ReadonlyExcelView` → `useExcelSnapshotRows` 纯读 `quote/costingExcelValues` 快照、**零 fetch**，列来自 frozen_dto 的 `quote/costingExcelColumns`。**真冻结**（评审复核确认）。
- `ProductDetailViews` 本身靠 `quotation` prop 渲染。

### 5.2 需补冻的两处结构（N3）

`getById` 的 `componentData` **不含 fields**（后端不持久化）；卡片渲染靠 `enrichComponentData(templateId,…)` live 拉模板补 fields，`globalVariableService.list()` live 拉 gvDefs（动态 key GV path 编译依赖）。模板/GV republish 即漂。务实版要"冻结构"，故：

1. **后端组装 frozen_dto**：提交时 `CostingFreezeService` 组装出**含 enrich 后 componentData（fields 已按提交时模板补全）+ gvDefs 快照**的完整 DTO，存入 `costing_order.frozen_dto`。结构在提交那刻定格，后续模板/GV republish 不影响该核价单。
   - 实现：复用提交时已冻结的模板 snapshot（quotation 提交即冻 `QuotationComponentSqlSnapshot`/submissionSnapshot）补 fields；gvDefs 取提交时 `global_variable` 全集快照。**parity 检查点**：后端补 fields 的口径须与前端 `enrichComponentData` 对齐（结构级，比 path 采集稳定；E2E 兜）。
2. **前端冻结模式**：`ProductDetailViews` 增加可选 `frozen` 标志（或检测 `quotation.__frozen`）。冻结模式下：
   - `ReadonlyProductCard` **跳过 `enrichComponentData`**，直接用 frozen_dto 里已 enrich 的 componentData；
   - `ProductDetailViews` **跳过 `globalVariableService.list()`**，用 frozen_dto 里的 gvDefs。

### 5.3 残留 live（务实版接受，§0 已述）

- `usePathFormulaCache` 仍 live 求值未被卡片快照覆盖的少数 path 单元（多数被快照遮蔽）；
- `useConfigTemplates`（LIST_FORMULA 配置）、比对视图 `comparisonTags` 仍 live。
→ 仅 V6/主数据/配置 republish 时漂，不被报价单重做触发。**不做** path-cache / config-template / tag 冻结（消除 N1/N2）。

### 5.4 冻结 DTO 内容

`frozen_dto` = 提交那刻组装的完整 DTO JSON：`lineItems[]`（**enrich 后 componentData** / templateId / costingCardValues / quoteCardValues / compositeType / id / productPartNo）+ 头层（customerId / quoteCardStructure / costingCardStructure / quoteExcelColumns / costingExcelColumns / **gvDefs** / originalAmount / totalAmount / finalDiscountRate / status）。`partVersionLocked` 落 JSON 保持 `Integer` 数值型（Minor 修正）。

`total_amount` = 提交时 `dto.totalAmount`（复用现有汇总口径，含 Step3 折扣按 `lineDiscountService`，不另立口径）。

### 5.5 读取路径（单源，反 AP-50）

- **核价管理列表 → 点核价单 → 工作台/详情**：`GET /costing-orders/{coid}` 返回 `frozen_dto` → 喂 `ProductDetailViews({quotation: frozenDto, frozen: true})`。任何核价单（含终态历史）读自己的冻结副本。
- **报价单详情（核价视图）**：保持第一期现状，读报价单自身 line items。本轮两侧不分叉。

## 10. 验收标准

1. 提交审批 → 核价管理列表出现一条 `待核价`（有 `HJ-` 号、提交人、创建时间），`frozen_dto` 非空（含 enrich 后 fields + gvDefs）。
2. 财务在工作台/列表 `核价通过` → 报价单与核价单两侧均 `已审核`。
3. 财务 `驳回` 填原因 → 两侧均 `已驳回`，列表「退回原因」列显示该原因。
4. 财务驳回后，**销售在报价单列表能按「已驳回」tab/label 找到该单**，点「编辑」→ 报价单转 `草稿` 进向导可编辑；核价管理列表里该核价单**仍** `已驳回`（N4 + 决策4）。
5. **改个值后重提** → 列表出现**第二条** `待核价`（新号），第一条 `已驳回` 历史仍在；**打开第一条 `已驳回` 单，结构 + 卡片/Excel 值仍是改之前的冻结值（1:1 历史保真，冻结模式 `/templates`、`/global-variables` 请求 0 次）**。
6. 销售对 `待核价`/`已驳回`/`已审核` 单「撤回」→ 报价单 `草稿`、核价单 `已撤回`（驳回单撤回不报错，B2）。
7. 核价管理列表可按全部 4 状态过滤、按报价单号搜索、按状态/时间排序；含「退回原因」「修改时间」列；同报价单多条历史各占一行（rowKey=costingOrderId，无撞键）。
8. 报价单详情 `SUBMITTED` 块 + 报价单列表均不再有销售经理「审批通过/退回」入口与「待我审批」tab。
9. 后端单测全绿（独立 test DB，含更新后旧测试 + 409 + 投影）+ 前端 tsc 0 + `quotation-flow.spec.ts` 1 passed + `e2e-costing-freeze.spec.ts` 通过（工作台结构冻结 8 Tab/加载中=0//templates+global-variables=0）。
