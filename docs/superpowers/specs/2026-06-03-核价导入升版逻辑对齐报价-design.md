# 核价导入升版逻辑对齐报价 — 设计方案

- 日期：2026-06-03
- 状态：待 review
- 分支：feat/selopt-v6-ingestion

## 1. 背景与问题

「主数据维护 → 导入核价数据」导入新料号后，`material_bom.bom_version` 被写成 `'V1'`，
而按版本规则应从 `2000` 起递增。根因：**核价（PRICING）导入链路对版本化表完全没有走升版逻辑**，
与报价（QUOTE）导入链路不一致。

### 现状（已逐行核对）

- 报价链路 `Q03MaterialBomHandler`（`cpq-backend/.../basicdata/v6/quote/Q03MaterialBomHandler.java:87`）
  调用 `VersionedV6Writer.writeVersionedMasterDetail`：groupKey = `{system_type='QUOTE', customer_no, material_no, bom_type}`，
  版本号 = 该组 `bom_version` 历史 `max+1`（非数字旧值被正则忽略，首版 **2000**），并维护 `is_current` 翻转。
- 核价链路写「版本化表」的 4 个 handler **全部未接 `VersionedV6Writer`**：
  - `P06MaterialBomHandler.java:51`：写死 `bom_version='V1'` + `ON CONFLICT` 覆盖
  - `P07ElementBomHandler`：直插 `element_bom`/`element_bom_item`，无升版
  - `P08CapacityHandler`：`calc_version` + `is_effective` + `ON CONFLICT`，无升版（另写 `labor_rate`）
  - `P21PlatingSchemeHandler`：用 Excel「版本」列直写 `scheme_version` + `ON CONFLICT`，无升版
- 其余 20 个核价 handler 写的是非版本化表（汇率/各类费用），本不该升版，**不在范围内**。
- `VersionedV6Writer`（`cpq-backend/.../basicdata/v6/versioning/VersionedV6Writer.java`）：
  - `ALLOWED_TABLES` 已含 `unit_price / capacity / plating_scheme / element_bom(_item) / material_bom(_item)`。
  - 版本序列与 `is_current` 翻转**完全按调用方传入的 `groupKey` 隔离**，写入器本身无需改动。
  - `nextVersionOf`（:305）：`MAX(CASE WHEN ver ~ '^[0-9]+$' THEN ver::int END) + 1`，无数字版本则 `2000`。
- `is_current` 列：V277 已给全部 4 张表加好（`V277__add_is_current_to_versioned_v6_tables.sql`）。

### 表结构两类差异（关键约束）

| 表 | 唯一键维度 | 有 system_type? | 版本列 |
|---|---|---|---|
| `material_bom` / `element_bom` | system_type + customer_no + material_no(+characteristic) | ✅ 有 | `bom_version` / element_bom 主表无版本列（多版本子表模型） |
| `capacity` | material_no + process_no + resource_group_no + COALESCE(calc_version,'') | ❌ 无 | `calc_version` |
| `plating_scheme` | scheme_no + scheme_version + seq_no | ❌ 无 | `scheme_version` |

- `capacity` 报价侧 `Q14AssemblyProcessFeeHandler.java:77` **已在用 `writeVersionedGroup` 升版**（groupKey 含 `resource_group_no='QUOTE_ASSEMBLY'`）。
- `plating_scheme` 报价侧 `Q16PlatingSchemeHandler.java:71` **已在用 `writeVersionedGroup` 升版**（忽略 Excel「版本」列，系统生成）。
- 读侧：报价 V280/V281 已给下游视图/sql_template 加 `is_current` 过滤，但**仅 `system_type='QUOTE'`**；
  核价 `v12_*` 视图（V255/V257）读 `material_bom_item`/`element_bom_item` 写死 `QUOTE`，读 `plating_scheme`（`V255:629 FROM plating_scheme ps`）**无 is_current 过滤**。

## 2. 目标与非目标

**目标**：核价（PRICING）导入对 `material_bom` / `element_bom` / `capacity` / `plating_scheme` 4 张表
执行与报价（QUOTE）**完全一致的存储升版**：整组比对 → 内容变化则 `max+1` 升版（首版 2000，非数字旧值忽略）
→ `is_current` 翻转；PRICING 与 QUOTE **各自独立**的版本序列。

**非目标**：
- 不改其余 20 个核价 handler（写非版本化表）。
- 不改 `VersionedV6Writer` 的核心算法。
- 不动报价侧已正确工作的升版语义（仅为新增 system_type 维度补 groupKey）。

## 3. 锁定决策

1. 范围 = 全部 4 张版本化表（`material_bom` / `element_bom` / `capacity` / `plating_scheme`）。
2. 与报价**存储升版逻辑一致**：整组升版 + `is_current` 维护。
3. PRICING 与 QUOTE **各自独立**版本序列（各自 2000 起算）。
4. `capacity` / `plating_scheme` **加 `system_type` 列**做物理隔离（与其余表一致）；
   报价导入写 `'QUOTE'`，核价导入写 `'PRICING'`。
5. 存量脏数据**写迁移规整**（不清空）。

## 4. 设计

### 4.1 Schema 改造（新 Flyway 迁移，建议 `V290`）

```sql
-- capacity 加 system_type + 改唯一键
ALTER TABLE capacity ADD COLUMN IF NOT EXISTS system_type VARCHAR(10) NOT NULL DEFAULT 'QUOTE';
ALTER TABLE capacity ADD CONSTRAINT chk_capacity_system_type
    CHECK (system_type IN ('QUOTE','PRICING','BOTH'));
DROP INDEX IF EXISTS uq_capacity;
CREATE UNIQUE INDEX uq_capacity ON capacity
    (system_type, material_no, process_no, resource_group_no, COALESCE(calc_version,''));

-- plating_scheme 加 system_type + 改唯一键
ALTER TABLE plating_scheme ADD COLUMN IF NOT EXISTS system_type VARCHAR(10) NOT NULL DEFAULT 'QUOTE';
ALTER TABLE plating_scheme ADD CONSTRAINT chk_plating_scheme_system_type
    CHECK (system_type IN ('QUOTE','PRICING','BOTH'));
DROP INDEX IF EXISTS uq_plating_scheme;
CREATE UNIQUE INDEX uq_plating_scheme ON plating_scheme
    (system_type, scheme_no, scheme_version, seq_no);
```

- `material_bom` / `element_bom` 已有 `system_type` + `is_current`，结构不动。
- `is_current` 4 张表均由 V277 具备，本次不再加。

### 4.2 `VersionedV6Writer`

- **核心逻辑不改**。仅需确认 `ALLOWED_TABLES` / `CHILD_UQ` 已覆盖（已覆盖）。
- 因 `nextVersionOf` / `is_current` 翻转均按调用方 `groupKey` 过滤，
  只要核价 handler 的 groupKey 带 `system_type='PRICING'`，PRICING 序列即与 QUOTE 天然隔离。

### 4.3 写侧 handler 改造

| handler | 改法 |
|---|---|
| `P06`(material_bom) | 删写死 `'V1'`+ON CONFLICT，改调 `writeVersionedMasterDetail`，镜像 `Q03`，masterGk/childGk `system_type='PRICING'` |
| `P07`(element_bom) | 镜像 `Q04ElementBomHandler`，`system_type='PRICING'` |
| `P08`(capacity) | 改调 `writeVersionedGroup`，镜像 `Q14`，groupKey 加 `system_type='PRICING'`；`labor_rate` 副作用保留（评估其 is_current/版本一致性） |
| `P21`(plating_scheme) | 改调 `writeVersionedGroup`，镜像 `Q16`（同样忽略 Excel「版本」列，系统生成 `scheme_version`），groupKey 加 `system_type='PRICING'` |
| `Q14`(capacity) | groupKey **补 `system_type='QUOTE'`**（新增维度） |
| `Q16`(plating_scheme) | groupKey **补 `system_type='QUOTE'`**（新增维度） |

> 注：`Q03`/`Q04` 本就传 `system_type='QUOTE'`，不受影响。

### 4.4 存量数据迁移（同一 `V290` 内，幂等）

- `material_bom` / `element_bom` 的 `system_type='PRICING'` 行：
  - `bom_version`（及 element_bom 子表版本）'V1'/非数字 → 规整为 `2000`；
  - 按组确保仅一条 `is_current=TRUE`（取最新一条置 TRUE，余置 FALSE）。
- `capacity` / `plating_scheme` 加列后回填 `system_type`（历史行 QUOTE/PRICING 不可严格区分，**启发式**）：
  - `capacity`：`resource_group_no='QUOTE_ASSEMBLY'` **或** `calc_version ~ '^[0-9]+$'` → `'QUOTE'`；其余 → `'PRICING'`。
  - `plating_scheme`：`scheme_version ~ '^[0-9]+$'` → `'QUOTE'`；否则 → `'PRICING'`。
  - 回填为 PRICING 的行：版本列洗到 `2000` 体系，并保证每组仅一条 `is_current=TRUE`。

> **⚠️ 待 review 假设**：上述 4.4 启发式回填规则。若历史数据无法满足该启发式，
> 退路是清空 `capacity`/`plating_scheme` 后用新逻辑重导（测试数据）。

### 4.5 读侧改造（防 AP-22「(共N项)」重复行）

升版后旧版本行 `is_current=FALSE` 仍在库，读侧不过滤会返重复行。需：

1. grep 全工程这 4 张表的所有读取点（`component_sql_view` / `template_sql_view` 的 `sql_template`、
   PG view、Java 直查）。
2. 核价 `v12_*` 视图读 `material_bom_item` / `element_bom_item` / `capacity` / `plating_scheme` 时，
   补 `system_type='PRICING' AND is_current=TRUE` 过滤（参照 V280/V281 报价侧做法，新增 `system_type` 维度）。
3. `v12_plating_scheme`（`V255:629`）补 `system_type + is_current` 过滤；capacity 相关视图同补。
4. 视图 DROP/重建后**必须 touch java 重启 Quarkus**（CLAUDE.md DDL 纪律，清 ImplicitJoinRewriter 等进程级缓存）。

## 5. 风险

- **读侧遗漏 → 重复行**（AP-22）：必须 grep 全量读取点，逐一加 is_current。最高风险项。
- **报价回归**：`capacity`/`plating_scheme` 新增 `system_type` 维度后，`Q14`/`Q16` 漏补 groupKey 会导致
  报价侧也出现版本/隔离错乱 → 必须跑报价 E2E 回归。
- **唯一键变更**：`DROP INDEX` + 重建期间并发写风险（dev 环境可接受；上线需评估）。
- **`labor_rate`**（P08 副作用）：是否需同样的 system_type/版本一致性，实现时评估。

## 6. 验证

- 后端：touch java 重启 → `flyway_schema_history` 中 `V290 success=t` → endpoint 200/401（非 500）。
- 视图 DDL 后再 touch 重启一次（DDL 纪律）。
- 功能验证：
  1. 核价导入新料号 → `material_bom.bom_version=2000`（非 'V1'），`is_current=TRUE` 唯一。
  2. 同料号改内容重导 → 升 `2001`，旧行 `is_current=FALSE`，核价单/Excel 渲染无「(共N项)」。
  3. 内容未变重导 → 复用版本，不新增行。
  4. PRICING 与 QUOTE 同 (customer, material) 各自独立 2000 序列，互不影响。
- E2E：核价单 spec + `quotation-flow.spec.ts`（报价回归），`'加载中' final count = 0`。
