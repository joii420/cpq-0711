# 导入组件冲突时用系统顺序编号（去掉 `__impN` 后缀）— 设计

- 日期：2026-07-20
- 范围：后端 `ComponentImportService`（组件导入）
- 决策：方案 A（副本导入后独立）+ 仅改今后的导入（存量 `__impN` 不迁移）

## 背景 / 问题

组件导入时，若 bundle 里的组件 `code` 与库内已有 `code` 冲突且策略为 RENAME，
现在通过 `nextFreeCode()` 在原 code 后追加 `__imp1 / __imp2 …` 后缀（如
`COMP-0063__imp1`）。用户反馈这个编号难以辨认，希望改为「按系统数字顺序自动递增」，
即与「新建组件」同源、走 `component_code_seq` 生成干净的 `COMP-####`。

## 现状关键事实

- **新建组件**已用系统序列：`ComponentService.create` → `nextval('component_code_seq')`
  → `String.format("COMP-%04d", seq)`。当前 `last_value=94`，下一个 = `COMP-0095`。
- **导入**（`ComponentImportService`）：
  - 无冲突 → `finalCode = 原 code`（保持不变）。
  - 冲突 + RENAME → `finalCode = nextFreeCode(原 code, reserved)` = `原 code + "__impN"`。
    调用点：第 232 行（真正落库）、第 124 行（**预览**，只算不落库）。
- `__impN` 后缀是**承重**的：`IMP_SUFFIX` 正则反推 base code，被三处依赖：
  1. 批次内跨组件引用重映射 —— 实际走 `codeMap`（原 code→finalCode），**与编号格式无关**。
  2. 源→副本自动同步（`ComponentService`：EXCEL 源组件改列后刷到副本，按 base code + `__impN` 找副本）。
  3. G4 目录级引用自动修复（按 base code 找副本）。

## 目标 / 非目标

**目标**
- 冲突（RENAME）时，分配下一个系统顺序编号 `COMP-####`（走 `component_code_seq`），取代 `__impN`。
- 无冲突导入行为不变（保持原 code）。

**非目标**
- 不迁移库里已存在的 `__impN` 老副本（保留其编号与源→副本同步行为）。
- 不改预览显示真实预测号（见下"预览"取舍）。

## 设计

改动集中在 `ComponentImportService.java`，纯后端：

1. **新增 helper** `nextSequentialComponentCode()`：
   `SELECT nextval('component_code_seq')` → `String.format("COMP-%04d", seq)`。
   序列全局唯一且单调递增，**天生不撞库内既有 code**，无需 `reserved` 防撞循环。

2. **落库路径**（现第 232 行）：
   `finalCode = nextSequentialComponentCode(); reserved.add(finalCode);`
   取代 `nextFreeCode(...)`。其后的 `codeMap.put(it.code, finalCode)` **不变**，
   第二遍 `FormulaRefRemapper` 按 `codeMap` 重写批次内跨组件引用 → **仍正确指向新顺序号**。

3. **预览路径**（现第 124 行）：预览只读、可重复调用，**不得消耗序列**（否则每次预览烧号、
   且预览号 ≠ 实际号）。改为把 `p.newCode` 置为占位提示（如 `"（导入时自动分配）"`），
   `action` 仍为 `RENAME`。真实编号在导入时才由落库路径分配。

4. **删除已无引用的 `nextFreeCode`**（两个调用点均替换）。**保留** `IMP_SUFFIX` /
   base-code 解析 / 源→副本同步 / G4 —— 供已存在的 `__impN` 老副本继续工作。

## 取舍（方案 A 的代价，已确认接受）

- 今后导入的顺序号副本，**不再**被"源→副本自动同步"和"G4 目录引用修复"发现
  （二者靠 `__impN` 反推 base）。副本导入后即独立。
- 存量 `__impN` 副本不受影响。

## 预览取舍

预览显示"（导入时自动分配）"而非具体号。可靠预测序列下一个值（不消耗）在并发下脆弱，
故不做。若将来需要预览显示真实号，另立二期。

## 测试

- **新增**：导入一个 code 冲突的 bundle（RENAME）→ 断言：
  - 新 code 匹配 `^COMP-\d{4}$`（无 `__imp`）、唯一、大于现有最大；
  - 批次内 `cross_tab_ref.source` / `component_subtotal.component_code` 重映射到新顺序号。
- **回归**：
  - `ComponentImportRefRemapTest`：若有断言期望 `__imp` 格式重命名 code，改为期望顺序号格式
    （重映射逻辑与格式无关，应仍通过）。
  - `DirectoryRefRemapServiceTest` / `ComponentDataEffectiveRowsTest`：不动（走保留的 `__imp` 兼容路径）。
- **自检**：`./mvnw test -Dtest=ComponentImport*` 全绿；touch java → 后端 `/api/cpq/components`=401（非 500）。
