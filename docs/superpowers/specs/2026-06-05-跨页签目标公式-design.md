# cross_tab_ref 目标公式（第一期）设计方案

- 日期：2026-06-05
- 状态：已评审通过（待落实施计划）
- 关联：扩展 `docs/superpowers/specs/2026-06-05-跨页签引用公式-design.md`（cross_tab_ref token，已实现）

## 1. 背景与需求

cross_tab_ref 现支持"目标列 = 从 A 组件选一个字段"。本需求把目标扩展为**可写公式**：目标可引用 **A（匹配行）的列 + B（当前行）的列 + 四则运算 + 全局变量**，在每个匹配的 A 行上求值。

### 范围裁剪（路线丙，已与用户确认）

完整函数库（ROUND/IF/汇率/SUM…）只存在于后端 JEXL 引擎；卡片公式引擎（前端 `formulaEngine.ts` + 后端 `FormulaCalculator`）**无函数调用机制**。给卡片引擎做"函数化"是独立大改。故采用分期：

- **第一期（本 spec）**：目标公式支持 **A 列 + B 列 + 四则运算 + 全局变量**（复用现有双 token 引擎，成本低）。
- **第二期（另立项）**：卡片引擎"函数化"（前后端双引擎加函数调用 token + 移植函数库 + 共享夹具锁一致）。届时目标公式自动获得函数能力。**本 spec 不含函数。**

### 已确认的需求边界

1. 目标公式可引用：A 匹配行列、B 当前行列、四则运算、全局变量（函数→第二期）。
2. 配置方式：默认"选单列"下拉 + "改用公式"开关展开紧凑公式编辑区（向后兼容老的单列配置）。
3. 聚合语义：**逐行先算、再聚合**（SUMPRODUCT 式）。`SUM` 目标 `A.单价×A.数量` = `Σ(单价ᵢ×数量ᵢ)`，非 `(Σ单价)×(Σ数量)`。NONE = 单匹配行上算一次。

## 2. token 结构（向后兼容）

```jsonc
{
  "type": "cross_tab_ref",
  "source": "<A 组件 componentId>",
  "sourceLabel": "投料",
  "target": "<A 目标列字段名>",     // 单列模式（默认/老配置不变）
  "targetExpr": [ /* ExpressionToken[] */ ],  // 公式模式（新增,可选）；非空时优先于 target
  "match": [ { "a": "A列", "b": "B列" } ],
  "agg": "NONE|SUM|AVG|COUNT|MAX|MIN"
}
```

`targetExpr` 内支持的 token：

| token | 含义 | 求值来源 |
|---|---|---|
| `field` | A（匹配行）的列 | 当前匹配 A 行的已算值 |
| `b_field` （新增） | B（当前行）的列 | B 当前行原始/已算值 |
| `operator` / `number` / `bracket_open` / `bracket_close` | 四则运算 | 字面量 |
| `global_variable` | 全局变量 | 见 §5 |

> 规则：`targetExpr` 非空 → 用之；否则回退 `target`（现状）。两者皆空且 agg≠COUNT → 校验报错。

## 3. 求值（前后端 evalCrossTab 对等）

对每个匹配到的 A 行 `arow`：
- 若 `targetExpr` 非空：用现有 token 引擎（前端 `evaluateExpression` / 后端 `FormulaCalculator.evaluateExpression`）求值，其中 `field`→`arow` 列值、`b_field`→B `currentRow` 列值、`global_variable`→当前 B 行上下文（§5）。得一个数。
- 否则：取 `arow[target]`（现状）。

聚合（与已有 cross_tab_ref 一致，只是"每行的值"改为"每行公式结果"）：
- `NONE`：0匹配→0；1匹配→该行结果；多匹配→**报错**。
- `SUM/AVG/MAX/MIN`：各匹配行结果聚合；某行结果非数字→**报错**；空集→0。
- `COUNT`：匹配行数（忽略 target/targetExpr）。

实现要点：evalCrossTab 内对 targetExpr 复用引擎现有单公式求值逻辑（逐行构造 fieldValues=arow + currentRow=B 行）。`b_field` 是新 token 分支，需在两个引擎的 token switch 各加一处。错误（多匹配/非数字）沿用现有 error→整公式 0 口径（前端 `(null.x)` 注入、后端异常哨兵）。

## 4. 配置 UI（CrossTabRefDrawer 目标区）

- 默认：单列 `Select`（现状，显示 `label||name`）。
- 新增"改用公式"开关。开启后展开紧凑编辑区：
  - 当前 `targetExpr` 以 chip 序列展示（复用 FormulaZone 风格回显，`field`/`b_field` 显示中文名）。
  - 插入按钮：【A 字段】（源组件字段下拉）、【本组件字段】（B 字段下拉）、【运算符】(+ − × ÷ ( )）、【数字】（输入）、【全局变量】（全局变量选择）。
  - 删末 / 清空 / 可读预览。
- 确定：公式模式且 `targetExpr` 非空 → 写 `targetExpr`，`target` 置空；否则写 `target`。
- 显示文案沿用 `label||name`（已实现的可读性修复）。

## 5. 全局变量解析上下文（设计决定）

`targetExpr` 内 `global_variable` 按**当前 B 行求值上下文**解析（复用引擎现有 global_variable 分支：`@gvar:CODE` / path cache；动态 key 用 B 当前行字段）。静态全局变量直接生效。"按 A 行作用域的全局变量"不在第一期范围（A 行的 basicDataValues 不在 crossTabRows 存储内）。

## 6. 校验

- 组件级（`ComponentService#validateFormulas`）：
  - 若 `targetExpr` 非空：校验其 `field` 引用存在于 source 组件、`b_field` 引用存在于本组件；token 类型在白名单内。
  - 非 COUNT 且 `target` 与 `targetExpr` 皆空 → 报错。
- 模板级（`TemplateService#validateCrossTabRefs`）：不变（targetExpr 只引用 source 组件列，不新增跨组件依赖；source 存在性 + 环检测已覆盖）。

## 7. 测试

- 后端单测（`FormulaCalculatorCrossTabTest`）+ 前端单测（`formulaEngine.test.ts`）：targetExpr 覆盖 A 列 / B 列 / 四则 / 全局变量；NONE 与 SUM 的"逐行先算再聚合"；非数字结果报错；target 与 targetExpr 互斥回退。
- 共享夹具 `cross-tab-cases.json` 增 targetExpr 用例，前后端逐例对齐（夹具用例需能表达 B 行 + 全局变量上下文）。
- E2E：配置路径——开"改用公式"、插入 A/B 字段 + 运算符、确认、断言 chip 回显。

## 8. 向后兼容

- 现有 `target: string` 的 token 不变，照常工作。`targetExpr` 为可选新增字段。
- 三视图一致性纪律不变（双引擎对等 + 共享夹具）。

## 9. 已知风险 / 注意点

- 新增 `b_field` token 是一次**字段类型/token 协议变更**，按 AP-44/AP-55 须在两个引擎 token switch + FormulaZone 回显 + 校验白名单同步登记。
- 全局变量 §5 的 B 行作用域是有意裁剪，spec 须明确，避免被当 bug。
- 夹具需扩展数据结构以携带 currentRow(B) + 全局变量上下文；保持前后端 JSON 同内容。
