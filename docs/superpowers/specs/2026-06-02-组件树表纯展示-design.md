# 组件树表(纯展示)设计

- 日期:2026-06-02
- 状态:设计已确认,待写实现计划
- 关联:组件管理 / 报价单渲染 / 核价单渲染 / 详情页渲染
- 反模式约束:AP-37 / AP-40 / AP-44 / AP-50 / AP-51

## 1. 背景与目标

料号之间存在父子关系。当前组件渲染是**平铺表格**,无法体现这种层级。需求:在「组件管理」中,允许把某个组件设为**树表**,由用户指定**两列**(本行 ID 列、父 ID 列)的父子关系,渲染时把这批行重排成树并支持缩进 + 折叠展开。

### 目标(已与用户逐条确认)

1. **配置端定规则 + 渲染端真正生效**(不仅是组件管理内预览)。
2. **父子关系模型 = 邻接表**:一列是本行 ID(料号),另一列是父 ID(父料号);父 ID 的值等于某行 ID 值 → 那行是其父。
3. **两列数据来源 = driver / SQL 视图**:展开时每行自带料号与父料号字段,系统读现成值算树(非销售手填)。
4. **纯展示**:只做缩进 + 折叠展开;不做父行汇总子行;不改变任何数值;折叠的子行仍照常参与 Tab 小计 / 公式。
5. **仅网页渲染**:Excel / PDF 导出维持现状(本期不做树状导出)。
6. **三视图全生效**:报价单填写视图(可编辑)、核价单视图、详情页(只读),由组件配置统一决定是否展示为树。
7. 渲染实现落在**渲染边界**(方案 A),不动数据权威层。

### 非目标(YAGNI)

- 不做父行对子行的金额/数量聚合(纯展示)。
- 不做 Excel/PDF 树状导出 / outline 分级。
- 不持久化用户手动展开/折叠状态。
- 不新增 `field_type`,不走 AP-44 字段类型联动矩阵。
- 不在 driver 展开层重排行序(见方案 B 否决理由)。

## 2. 父子关系模型

邻接表(adjacency list):

- **idField**:本行 ID 列(料号)
- **parentField**:父 ID 列(父料号)
- 行 X 的 `parentField` 值 == 行 Y 的 `idField` 值 → Y 是 X 的父。

两列值均来自 driver 展开行(BASIC_DATA / DATA_SOURCE),渲染层只读不写。

## 3. 存储 — 组件级 `treeConfig`

在 `ComponentItem`(前端 `cpq-frontend/src/pages/component/types.ts`)及后端组件实体的 config JSONB 新增:

```ts
treeConfig?: {
  idField: string;          // 本行 ID 列字段 key(料号)
  parentField: string;      // 父 ID 列字段 key(父料号)
  defaultExpanded?: boolean; // 默认全展开(true)/全折叠(false),缺省 true
}
```

- 与 `dataDriverPath`、`rowKeyFields` **同一层级** —— 组件级「渲染行为元数据」。
- **不长在 `FieldItem` 上** → 完全绕开 AP-44 字段类型联动 17 处检查点;它只是渲染层「拿到这批行后怎么排序 + 缩进」的指令。
- **单一真相源**:`treeConfig != null` 即「此组件是树表」。

### 校验

- 开启树表时:`idField`、`parentField` 均必填;且 `idField !== parentField`(不能同列)。
- 关闭树表时:清空 `treeConfig`(置 `undefined`/null)。
- 两个字段 key 必须存在于当前组件已配置字段中(配置改名/删字段时若失配,渲染层按「无 treeConfig 兜底平铺」降级,不报错不丢行)。

## 4. 配置 UI — 组件设置区

在组件管理设置区(`dataDriverPath` 配置附近)新增:

- 「树表」开关(Ant Design `Switch`)。
- 开关打开后展开:
  - **ID 列** 下拉(`Select`),选项 = 当前组件已配置字段(`fields`)。
  - **父 ID 列** 下拉(`Select`),同上。
  - **默认展开** 开关(对应 `defaultExpanded`)。
- 遵循项目 UI 规范:配置交互在抽屉 / 设置面板内,不用 Modal 表单。
- 关闭开关 → 清空两列选择与 `treeConfig`。

## 5. 配置传播(写后端 + 进快照)

`treeConfig` 跟随 `dataDriverPath` 的**现有传播链**,逐点带上:

1. **后端**:组件实体/DTO + `ComponentService` 存取(config JSONB 序列化/反序列化)。
2. **模板快照**:`TemplateService.refreshSnapshotsByComponent` 写 `componentsSnapshot` 时带上 `treeConfig`;整份快照结构 v2(`CardStructure` 的 tab)序列化带上。
3. **前端回填**:
   - 模板路径 `enrichComponentData.ts`(同 `dataDriverPath` 的回填处,约 line 161~176 / 266)。
   - 整份快照路径 `buildComponentDataFromStructure`(`enrichComponentData.ts:266` 附近,与 `dataDriverPath` 同处)。
4. AP-37 纪律:同 `componentId` 多实例(同组件挂不同 driver)时,`treeConfig` 随各 tab 实例独立带入,不跨实例污染。

## 6. 渲染 — 共享纯函数 + 折叠 hook(方案 A)

### 6.1 隔离模块

新增 `cpq-frontend/src/pages/quotation/treeTable.ts`(纯函数,无 React 依赖,可单测):

```ts
interface TreeLayout {
  order: number[];                       // 重排后的原始行下标序列(子行紧跟父行)
  depthByIndex: Record<number, number>;  // 每个原始行下标 → 缩进层级(根=0)
  hasChildren: Set<number>;              // 含子节点的行下标(显示折叠箭头)
  parentByIndex: Record<number, number | null>; // 每行的父行下标(根=null;用于折叠隐藏判定)
}

function buildTreeRows(
  rows: Record<string, any>[],
  treeConfig: { idField: string; parentField: string },
): TreeLayout
```

**输入是原始行数组(原序),输出只是「怎么排 + 怎么缩进」的指令。不改 `rows`、不改 `rowCount`、不改快照、不改行序权威源。**

### 6.2 折叠状态 hook

`useTreeCollapse(defaultExpanded: boolean)`:

- 会话内 `useState<Set<number>>`(存「被折叠的行下标」)。
- 不持久化、不进快照;刷新/重进恢复默认。
- `defaultExpanded === false` → 初始把所有 `hasChildren` 行放入折叠集。
- 暴露 `isCollapsedHidden(index)`(祖先链上任一节点折叠 → 该行隐藏)、`toggle(index)`。

### 6.3 三视图接线

报价单 `QuotationStep2.tsx`、详情页 `ReadonlyProductCard.tsx`、核价单卡片渲染入口,统一改为:

- `treeConfig` 存在 → 用 `buildTreeRows` 的 `order` 迭代;跳过 `isCollapsedHidden` 为真的行;首列单元格前插入「缩进(按 depth)+ 折叠箭头(`hasChildren` 才显示)」。
- `treeConfig` 不存在 → 完全走现有平铺迭代(`rowCount` 循环),**零行为变化**。
- 三处共享同一 `treeTable.ts` + `useTreeCollapse`,避免 AP-50「各视图各写一份导致漏分支」。

### 6.4 与现有迭代逻辑的关系

- 现有渲染按 `rowCount` 平铺迭代(`QuotationStep2.tsx:644 / 1456`)。树模式只是把「迭代顺序」从自然下标改为 `order`,并加「折叠隐藏跳过」+「首列缩进/箭头」。
- 行数权威仍是 driver `rowCount`(AP-51);`buildTreeRows` 不得改变行集合,只能重排与标注。
- 折叠隐藏的子行仍计入 Tab 小计/公式(纯展示,数值不受树影响)。

## 7. 建树边界规则(逐条已确认)

1. **根节点判定**:某行 `parentField` 为空,**或** `parentField` 有值但在当前批次行里找不到匹配 `idField` → 当作顶层根(不因父缺失而消失/丢行)。
2. **多根**:允许多个根并列。
3. **环检测**:检测到父子成环(A→B→A 或更长链)→ 把环上的行降级为根平铺显示 + `console.warn`,其余行不受影响,不死循环。
4. **料号重复**:同一 `idField` 值出现多行时,找父按「第一条匹配的同 ID 行」;重复行全部照常显示,不丢行。
5. **展开/折叠状态**:纯前端会话态,不持久化;初始按 `defaultExpanded`;刷新恢复默认。
6. **同级排序**:同一父下子行保持 driver 返回原序,不额外排序。

## 8. 测试

纯函数单测覆盖 6 条边界规则:

- 正常单根多层树:order 子行紧跟父、depth 正确、hasChildren 正确。
- 父为空 → 根。
- 父值在批次内缺失 → 根(不丢行)。
- 多根并列。
- 成环 → 降级为根 + 不死循环。

## 9. 方案取舍记录

- **方案 A(采纳)** 渲染边界纯展示重排:不碰数据权威层,规避 AP-37/40/51;树逻辑隔离成共享模块,改动面集中、可单测;缺点是三视图各接一次 wiring(逻辑共享)。
- **方案 B(否决)** 在 `useDriverExpansions` 预排成树:消费端零改动,但动了行序/`rowCount` 数据权威源,会牵动快照 `baseRows`、`rowKeyFields` 对齐、后端整份快照,高概率引发 AP-37/40 隐性 bug,违背「纯展示不动数据」。
- **方案 C(否决)** 新 `field_type` / 新组件类型:过度设计,要趟 AP-44 的 17 处协议传播,纯展示需求不值得。

## 9.5 与「Excel 视图卡片引用公式」设计的关系(跨设计不变量)

关联文档:`docs/superpowers/specs/2026-06-02-Excel视图卡片引用公式配置-design.md`(实施中:`ExcelViewService`/`CardDataProvider`/`TemplateFormulaService` + `LinkedExcelView`/`CardFormulaDrawer`/`ExcelViewConfigTab`)。两份设计**无冲突**,作用层正交:Excel 公式 = 在「Excel 视图层」消费卡片已算好的 `rowData` + 小计再算列公式;本设计 = 在「产品卡片渲染层」对卡片自己的行做缩进/折叠。

记录两条持久不变量:

1. **方案 A 主动规避了唯一潜在冲突 —— 树表不得改 `rowData` 行序/行集**。
   Excel 的 `CardDataProvider` 把聚合(`SUM_OVER` 等)与单值取行(`FIRST_ROW` / `ROW_WHERE`)的数据源切到「某页签实例的 `rowData` 行集」,其结果**依赖 `rowData` 的原始行顺序与行集合**。本设计采用方案 A(渲染边界纯展示重排),`buildTreeRows` 只产出渲染顺序/缩进指令,**绝不改 `rowData` / `rowCount` / 行序权威源**,故 Excel 聚合与 FIRST_ROW 不受任何影响。
   > 若改用方案 B(driver 展开层排成树)会直接改掉 `rowData` 行序 → Excel「首行」从「driver 第一行」漂移成「树根行」、聚合行序也变,那才是真冲突。否决方案 B 的 AP-37/40 理由顺带消掉了此冲突。**后续任何"把树序下沉到数据层"的优化都必须先评估对 Excel CardDataProvider 的破坏,默认禁止。**

2. **Excel `FIRST_ROW`「首行」= 数据原序第一行,不是树的视觉根行**。
   若某树表组件同时被 Excel 列以 `[页签.字段]` FIRST_ROW 引用,「首行」(driver 原序第一行)与「树根」(本设计视觉重排后的顶层行)可能不是同一行。这是设计自洽的(Excel 走数据序、树走视觉序),实现与文档不得假设二者对齐。

**实施协调点(流程,非设计冲突)**:
- 两者都属 AP-44 渲染层协议改动且共跑 `quotation-flow.spec.ts` → 树表实现应在 Excel 这批提交之上 rebase 后开工并复跑 E2E 基线。
- 模板快照序列化两边都要扩(Excel 加 `excel_view_config` 模板级、本设计加 `treeConfig` 组件/页签级),不同 key、additive;若改到同一段 `CardStructure` 序列化代码需手动并段。
- 并发分支纪律:提交时只 `git add` 树表相关文件,勿扫入 Excel 未提交工作。
- 共用文件仅 `QuotationStep2.tsx`,且代码区不相交(Excel 在 line 13/2213/2272 import+渲染 `LinkedExcelView`;树表在 line ~644/1456 卡片行渲染循环),合并风险低、无逻辑耦合。
