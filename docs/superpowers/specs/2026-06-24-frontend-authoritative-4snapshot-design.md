# 报价单「前端单引擎权威 + 4 快照」架构设计方案

> 立项 2026-06-24。目标:把报价侧 / 核价侧(各含 Excel 视图)的全部计算移到前端单引擎,
> 后端退化为「存储 + 审批校验」,saveDraft 只落库 4 个快照 JSON。彻底消除 saveDraft 里
> ~145s 的服务端逐行重算(实测:snapshotQuotation 12s + 逐行 cardValues 132s)。

---

## 1. 现状 vs 目标

| | 现状 | 目标 |
|---|---|---|
| 报价 card values | 后端 `buildCardValues` 算 | **前端单引擎算** |
| 报价 Excel | **前端算**(已权威,quoteExcelValues) | 前端算(不变) |
| 核价 card values | 后端 `buildCostingCardValues` 算(~44s/170行) | **前端单引擎算** |
| 核价 Excel | 后端 `buildExcelValues` 算 | **前端单引擎算** |
| saveDraft | 主存 + 服务端重算 4 份(~145s) | **只落库前端算好的 4 份 JSON** |
| 提交/导出 | (待确认)读快照 | **读快照(快照即权威)** |
| 审批 | 直接审批 | **后台异步重算校验,校验过才能审批** |

## 2. 核心原则

1. **前端单引擎 = 唯一计算真相源**(渲染 + 落库都用它);后端不在 saveDraft 热路径算。
2. **后端 = 存储 + 审批关卡校验**;Java 引擎保留但移出热路径(仅审批校验 + 独立核价单 CostingSummary 用)。
3. **快照即权威**:渲染/提交/导出一律读 4 个快照 JSON,不重算。
4. **精度前后端逐位一致**(见 §4,本方案命门)。
5. **产品卡片间独立**(仅产品内页签互引)→ 保存粒度 = 整张产品卡(含其所有 tab)。

## 3. 数据契约

### 3.1 四个快照(均 per line item,前端算、后端原样存)
- `quote_card_values`(报价卡片值,新增前端权威)
- `quote_excel_values`(报价 Excel,已前端权威)
- `costing_card_values`(核价卡片值,新增前端权威)
- `costing_excel_values`(核价 Excel,新增前端权威)

### 3.2 新增「分组 raw 数据」接口(前端计算输入)
打开报价单时,后端一次返回**按产品卡片分组**的计算所需原料:
- 各产品 × 各组件的 driver 展开行(报价 + 核价两套模板)= 现有 `batch-expand` 的数据,grouped;
- 各模板结构(tabs/fields/formulas/列定义)+ 全局变量当前值。
- 复用/扩展现有 `batch-expand`,新增"核价模板侧"展开 + 一次性分组返回。

### 3.3 SaveDraftRequest 扩展
- `LineItemDraft` 增加 `quoteCardValues` / `costingCardValues` / `costingExcelValues`(`quoteExcelValues` 已有);后端**原样落库,不重算**。
- 新增**单产品卡增量保存**端点:`PUT /quotations/{id}/line-items/{lineItemId}/card-snapshot`,只更这张卡的 4 份 JSON(编辑场景)。

## 4. 精度一致性方案(命门)

**问题**:JS `number`(IEEE754 浮点)vs Java `BigDecimal` → 同公式必然有舍入/精度差 → 审批重算逐位比会天天报假不一致。

**方案:统一"十进制计算契约",两端逐位等价。**

1. **前端弃用原生浮点,改用十进制库**(decimal.js / big.js),语义对齐 Java `BigDecimal`:
   - 相同 `scale`(小数位)+ 相同 `RoundingMode`(全工程统一,如 `HALF_UP`);
   - 中间计算保持高精度,**只在"落值点"按口径舍入**。
2. **舍入口径沿用** `[[cpq-decimal-display-policy]]`:计算列 / 列小计 / 页签合计 = 4 位;最终产品小计 + 对外导出总额 = 2 位;取数列保留原精度;COMPUTED_FALLBACK 三处=4。两端在**完全相同的节点**用**完全相同的位数+模式**舍入。
3. **统一公式语义规范**(两端各一份实现,但规范同源):运算符优先级、函数语义(SUM/IF/ROUND/...)、null/空值处理、除零处理、字符串↔数字转换,逐条写死,配**共享测试向量**(同一批 input → 期望 output,前端 JS 测 + 后端 Java 测都跑)。
4. **比较口径**:审批重算后,**先按 §4.2 口径规范化(round 到目标 scale)再逐字段比**,容差 = 0(规范化后应完全相等)或极小 epsilon 兜底浮点残差;**只比落库快照里的单元格值**(不比中间量)。
5. **CI 守卫**:一份"黄金数据集"(若干真实单的 input + 期望 output),前端引擎单测 + 后端引擎单测**都必须命中同一期望**;任一端改公式 → 两端测试同时跑,防漂移。

> 落地顺序上,§4 必须**先做**:没有精度契约,审批校验(§5.4)就是噪音。

## 5. 流程设计

### 5.1 首次打开报价单(导入后第一次)
```
GET 报价单 → 后端返 分组 raw 数据(报价+核价 driver 行 + 模板结构 + 全局变量)
 → 前端单引擎算 4 份快照(报价 card/excel + 核价 card/excel,eager 两边一起算)
 → saveDraft 一次落库 4 份 JSON
```
**eager 两边一起算**(非"切到核价才算"):因 §6.1 提交/导出读快照,必须保证任何时刻 4 份都在,否则"没看过核价就提交"→ 核价快照空 → 导出/审批校验拿空。

### 5.2 用户编辑(报价侧)
- 改某产品卡 → 前端**重算该产品卡全部页签**(产品内页签互引,整卡一致)→ **单卡增量保存**(只发这张卡的快照)。
- 产品间独立 → 不影响别的卡。

### 5.3 核价侧「从基础刷新」(仅核价侧开放)
- 点击 → 后端从库重插核价侧基础数据 + 返回该单核价 raw → 前端重算核价两份 → 保存。

### 5.4 提交 → 审批校验(后台异步)
```
提交 → 状态进入"待审批/校验中"(此时不可审批通过)
 → 后台异步任务:Java 引擎重算 4 份 → 按 §4 口径与前端快照比
 → 一致:放行,可审批; 不一致:通知用户 + 标记差异单元格,阻止审批
```
- 后台任务**独立 request context 隔离 + 连跑两次确定**(避 9/73 类竞态)。
- 低频、非热路径,可接受异步。

### 5.6 渲染(再次打开 / 详情 / 列表)
- 直接读 4 份快照 JSON 渲染,**不算、不展开、无远程往返** → 秒回。
- 混合态:存量单(后端快照)与新单(前端快照)**JSON 结构必须一致**,渲染层统一读。

## 6. 改动点清单

### 6.1 后端
- `QuotationResource.saveDraft`:**移除** `snapshotQuotation(id,true)` + `snapshotLineValuesWithUnion` 循环;改为持久化请求里的 4 份快照。
- `SaveDraftRequest.LineItemDraft`:+`quoteCardValues`/`costingCardValues`/`costingExcelValues`。
- 新增**分组 raw 数据**端点(扩展 batch-expand:加核价模板展开 + 分组返回 + 全局变量)。
- 新增**单卡增量保存**端点。
- 新增**核价从基础刷新**端点(重插核价基础 + 返 raw)。
- 新增**审批重算校验**后台任务 + 状态机增加"校验中"态 + 差异通知。
- 确认**提交/导出读快照**(SnapshotCollectorService / QuotationExportService);若有重算改为读快照。
- `buildCardValues`/`buildCostingCardValues`/`buildExcelValues`:移出 saveDraft 热路径;保留供审批校验 + 独立核价单。

### 6.2 前端
- 单引擎扩展:从"算 quote excel + 渲染后端 card 快照"→ **产出全部 4 份快照**(报价 card + 核价 card + 两侧 excel)。
- 打开流程:加载分组 raw → eager 算 4 份 → saveDraft。
- 编辑:单卡重算 + 单卡增量保存。
- 核价侧"从基础刷新"按钮。
- 引入十进制库 + 统一舍入口径(§4)。

### 6.3 契约/质量
- 精度契约 + 共享黄金测试向量(§4.5)。
- 快照 JSON shape 前后端逐键对齐(混合态)。
- 下游消费方字段穷举(渲染/提交/导出/对比/折扣 读啥前端都得算)。

## 7. 结果评估

### 7.1 性能收益
| 操作 | 现状 | 改后 |
|---|---|---|
| saveDraft(首存) | ~145s(服务端重算) | **后端只落库 ~亚秒**;首存耗时=前端算+传输(见风险) |
| 再次打开渲染 | batch-expand 22s+ | **读 4 快照,秒回** |
| 单卡编辑保存 | 全量风险 | **只算+存一张卡,快** |

### 7.2 风险与成本
| 项 | 等级 | 说明 |
|---|---|---|
| **精度前后端漂移** | 🔴 高 | 命门;靠 §4 十进制契约+黄金向量+CI 守卫。不做好→审批校验全是噪音 |
| **客户端性能(首存搬家)** | 🔴 高 | eager 两边算 170 行 + 传全量 driver 数据到浏览器,瓶颈可能从"后端算"变"传输+客户端算"。**必须先实测**(§9) |
| **快照完整性** | 🟠 中 | 漏算某下游字段→该下游拿空;需穷举消费方 |
| **双引擎维护** | 🟠 中 | Java 引擎为审批校验+核价单保留,与前端须长期一致(黄金向量兜底) |
| **信任/篡改** | 🟢 低-中 | 快照权威=信任前端;由 §5.4 审批校验兜底(合同生效点) |
| **混合态** | 🟢 低 | 存量后端快照 + 新前端快照共存;JSON shape 对齐即可 |

### 7.3 总评
- **结构上彻底解决 saveDraft 慢**(后端不再热路径重算),且渲染秒回、编辑轻量。
- 成败压在**两个实测/契约**上:**①精度契约(§4)②客户端首存性能(§9)**。这两个不过关,方案不成立。
- 是**架构迁移级**改动(非快赢):前端引擎要产出全部 4 份 + 精度契约 + 后端 rewire + 审批校验。

## 8. 落地阶段(建议顺序)

- **P0 实测验证可行性**(不写生产码):①客户端算 170 行+传输耗时(§9)②前端 vs 后端同公式精度差现状。两项过关再投入。
- **P1 精度契约**:十进制库 + 统一舍入 + 黄金测试向量(前后端都过)。
- **P2 后端只存**:SaveDraftRequest 扩 4 份 + saveDraft 移除重算 + 分组 raw 端点;前端 eager 算 4 份 + 落库。新单走前端权威,存量不动。
- **P3 编辑增量**:单卡重算 + 单卡保存端点。
- **P4 核价从基础刷新** + **审批异步校验**(依赖 P1 精度契约)。
- 每阶段:前端 vs 后端黄金向量等价 + E2E。

## 9. 必须先做的实测(P0,决定方案成立与否)
1. **客户端首存性能**:打开 170 行单,前端算报价+核价两边 + 传输的真实耗时/体积。若客户端算+传输 > 可接受阈值 → eager 改 lazy(+提交守卫)或分批。
2. **精度差现状**:取若干真实单,前端 JS 引擎 vs 后端 Java 引擎对同公式输出,看差异分布 → 定 §4 舍入口径与容差。
