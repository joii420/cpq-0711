## 0. 背景、目标与依赖

### 0.1 目标

核价基础数据 Excel（24 Sheet）新增「销售料号」列，语义 = **报价料号** `XXXX-YYMMNNNNNN`，全系统唯一、贯穿报价/核价。要求：

1. 19 个"按料号" Sheet 的目标表**旁加 `sales_part_no` 列**（生产料号原落库列 `material_no`/`finished_material_no`/`code` 不动）。
2. 销售料号进各表**唯一键**（复合键末尾追加 `COALESCE(sales_part_no,'')`）。
3. 销售料号进**升版 groupKey**，实现"同生产料号、不同销售料号各自独立升版、is_current 互不翻转"。
4. P05 在 `material_customer_map` 登记**三码**（销售料号 / 生产料号 / 客户料号）。
5. 5 个全局/元素级 Sheet 不加此维度。

### 0.2 硬前置依赖：报价料号 Spec1 必须先落 🔒

`material_customer_map` 由**两个 spec 共同改造**，本 spec 的三码登记（§5）**依赖报价 Spec1 已加的 `system_type` 列**（P05 行须打 `PRICING` 标才能与报价 Q02 的 `QUOTE` 行按行隔离）。因此：

| 事项 | 报价 Spec1（先） | 本 spec（后） |
|---|---|---|
| `material_customer_map` DDL | 加 `system_type`/`production_no`、DROP `customer_product_no` NOT NULL、重建 uq（`uq_mcm_composite` NULLS NOT DISTINCT + 两个 QUOTE 分区索引） | 加 `sales_part_no`（进 DO UPDATE SET，**不进冲突键 arbiter**） |
| `MaterialCustomerMapRepository.upsert/upsertChunk` | 11 列 → 13 列（+`system_type`+`production_no`），冲突键+客户守卫改造 | 13 列 → 14 列（+`sales_part_no`），步长 `i*13`→`i*14` |
| `P05CustomerMapHandler` | 写 `system_type='PRICING'` | 读「销售料号」「生产料号」、写 `sales_part_no` |
| Flyway 号段 | V308 + V310（原 V309 已并入 V308） | **V311+**（现役 max=V307） |

**结论**：本 spec 在报价 Spec1 合入 master 后开工；两者若并行，`material_customer_map` DDL 与 Repository 会硬冲突。实现计划 Task 0 须先校验 `system_type` 列已存在。

### 0.3 四项已决策（2026-07-07 确认，见 V6.2 §0）

1. 销售料号 = 复合唯一键 + 升版 groupKey 维度（进 19 个按料号表，与生产料号并存）。
2. 生产料号原落库列不动，旁加 `sales_part_no`。
3. P05 在 `material_customer_map`（PRICING 行）登记三码。
4. 5 个全局/元素级 Sheet（P01/P02/P03/P21/P04）不加销售料号维度。

**三码关系**（对齐报价 Spec §1）：销售料号 ↔ 客户料号 = 1:1；生产料号 ↔ 销售料号 = 1:N。

## 1. 数据模型（DDL）

### 1.1 加列（V311，11 张表）

```sql
-- 列规格统一：VARCHAR(32) NULL（报价料号定长 15，留余量；列级可空以兼容共享表全局行）
ALTER TABLE unit_price            ADD COLUMN sales_part_no VARCHAR(32);  -- 费用行有值；P01/P02 全局行 NULL
ALTER TABLE material_bom          ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE material_bom_item     ADD COLUMN sales_part_no VARCHAR(32);  -- 随主表镜像
ALTER TABLE element_bom           ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE element_bom_item      ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE capacity              ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE labor_rate            ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE production_energy     ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE auxiliary_energy      ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE tooling_cost          ADD COLUMN sales_part_no VARCHAR(32);
ALTER TABLE material_customer_map ADD COLUMN sales_part_no VARCHAR(32);  -- 见 §5，进 SET 不进 arbiter
```

- **`material_master` 不加列 / 保持生产料号主表**（2026-07-07 用户复确认）：单重/主数据按生产料号唯一，而生产料号↔销售料号=1:N，加列破坏 1:N；且现有 174 行含 74 个 BOM 组成料号（子件生产料号，无销售料号），无法用销售料号作键。销售↔生产↔客户映射统一由 `material_customer_map` 承载（§5）；销售料号的全局登记表是 `material_customer_map` 的 QUOTE 行（报价 Spec1 已建）。
- **列级为何可空**：`unit_price` 被全局行（P01 元素价格 / P02 材料价格，无销售料号）与费用行共用；全局行落 NULL，费用行必填由 **handler 校验**保证（§6），uq 用 `COALESCE(sales_part_no,'')` 中性化，二者不冲突。

### 1.2 唯一键重建（V311，附各表**实测**现役 uq）

各表在现有 uq **末尾追加** `COALESCE(sales_part_no,'')`（不改原列顺序，保护存量 ON CONFLICT 表达式）：

| 表 | 现役 uq（实测·迁移出处） | 加维后 |
|---|---|---|

| `capacity` | `uq_capacity`：`(system_type, material_no, process_no, resource_group_no, COALESCE(calc_version,''))`（V290，**含 system_type**——V6.2 §5 漏写，以此为准） | + `COALESCE(sales_part_no,'')` |
| `labor_rate` | `uq_labor_rate`：`(version_no, process_no, COALESCE(material_no,''), COALESCE(labor_grade,''))`（V220） | + `COALESCE(sales_part_no,'')` |
| `production_energy` | `uq_production_energy`：`(material_no, process_no, COALESCE(equipment_no,''), COALESCE(calc_version,''))`（V220，**无 system_type**——P09/P10 专属） | + `COALESCE(sales_part_no,'')`（P09/P10 合并契约 4→5 维，见 §4.3） |
| `auxiliary_energy` | `uq_auxiliary_energy`：`(material_no, process_no, COALESCE(calc_version,''))`（V220） | + `COALESCE(sales_part_no,'')` |
| `tooling_cost` | `uq_tooling_cost`：`(material_no, process_no, seq_no, tooling_no)`（V220，全 NOT NULL 无 COALESCE） | + `COALESCE(sales_part_no,'')` |
| `material_bom` | `uq_material_bom_v6`：`(system_type, customer_no, material_no, bom_version, COALESCE(characteristic,''))`（V219）——唯一性由 is_current 版本机制承载 | sales_part_no 进 **groupKey**（§3）即隐含进版本维度；uq 由升版机制保证不撞 |

## 2. 参数注入：销售料号如何进 handler

现状 `PricingImportService.importExcel` 构造 `ImportContext ctx`：`ctx.customerNo=null`（核价无客户上下文）、`ctx.systemType="PRICING"`。销售料号是**行级值**（每行 Excel 都带），**不进 ctx**，由各 handler 从行读：

```java
String salesPartNo = row.getStr("销售料号");
```

生产料号列兼容读（原「宏丰料号」改名「生产料号」）：

```java
String materialNo = row.getStr("生产料号", "宏丰料号");   // 优先新表头，回退旧名
```

**例外**：P07 物料与元素BOM 的料号列真实表头是「物料料号」（未改名），保持 `row.getStr("物料料号")`，不套用上面的回退（见 V6.2 §4「P07 料号列例外」）。

## 3. 升版 groupKey 变更（P06 / P07 / P08 + 一步到位：unit_price 费用类走 `VersionedV6Writer`）

`VersionedV6Writer` 是表通用的：groupKey 是 `Map<String,Object>`，其列集定义"版本组"——组内 is_current 翻转、版本号 max+1。**给 groupKey 加 `sales_part_no` 键 → 不同销售料号即使生产料号相同也各自成组、各自 2000 起版、互不翻转**。groupKey 列会被 `insertRowGeneric` 写进主/子行（§1 已建列，白名单已含表）。

| 表 | Handler | 现役 masterGroupKey（**实测**） | 加维后 |
|---|---|---|---|
| `material_bom`(+item) | `P06MaterialBomHandler`（`:91-100` set-based / `:120-129` 逐行） | master `{system_type:PRICING, customer_no:_GLOBAL_, material_no, bom_type:MATERIAL}`；child `{system_type, customer_no, material_no, characteristic:null}` | master + child 各 `put("sales_part_no", salesPartNo)`；**聚合键 `childByMat` 由 `material_no` 改 `(material_no, salesPartNo)`** |
| `element_bom`(+item) | `P07ElementBomHandler` | master groupKey `{system_type:PRICING, customer_no:_GLOBAL_, material_no}`，**versionColumn=`characteristic`**（`characteristic` 是版本列，不是 groupKey 成员——勿写进 gk）；child groupKey 同 master | master + child groupKey 各加 `sales_part_no`（不动版本列 characteristic）；聚合键加 salesPartNo |
| `capacity`(+labor_rate) | `P08CapacityHandler`（`:86-91` / `:171-174`） | `{system_type:PRICING, material_no, resource_group_no:PRICING_DEFAULT}`，`VERSION_TRIGGER=[process_no]`，按 `capByMat`(material_no) 聚合；labor 走裸 SQL 冲突键 `(version_no, process_no, COALESCE(material_no,''), COALESCE(labor_grade,''))`（`:146`/`:184`） | ① gk `put("sales_part_no", salesPartNo)`；② **`capByMat`/`laborByMat`/`gkByMat` 三个聚合 Map 键由 `material_no` 改 `(material_no, salesPartNo)`**；③ labor_rate 裸 SQL：DB 冲突键 + INSERT 列 + 绑定加 `sales_part_no`；④ **⚠️ `laborByKey` 内存去重键（`:112` `ck=(version, processNo, materialNo, "")`）必须追加 `nz(salesPartNo)`**——见下方 M1 |

> **关键**：P06/P08 现按 `material_no` 单键在内存聚合成组（`childByMat`/`capByMat`）。加销售料号维度后，**聚合键必须升为 `(material_no, sales_part_no)` 复合键**，否则同生产料号跨销售料号的行会被错误并入同一版本组（丢隔离）。这是本 spec 最易漏的改点。

> **⚠️ M1（必现丢行·AP-51 类协议漏点）**：P08 labor_rate 在**入库前**有一层内存去重 `laborByKey`（`P08CapacityHandler:104-127`），键 `ck=(version, processNo, materialNo, "")`。因每个独立升版组**首版号恒为 `2000`**（`VersionedV6Writer` `newVersion` 首版逻辑），同生产料号 M 的 S1/S2 两组 labor 行的 `version` 都是 `2000` → `ck` 完全相同 → **内存里先被折成一行**，DB uq 虽已含 sales 能容两行也救不回来，S1 人工单价永久丢。故 `ck` 必须追加 `nz(salesPartNo)`；同时 `gkByMat.get(materialNo)`（`:108`）关联查找须改用复合键。DB 冲突键加维 ≠ 内存 dedup 键加维，**两处都要改**。

> **m4（P06 material_master 消费点）**：`P06MaterialBomHandler:145` `masterRepo.upsertBatchMaterialNoOnly(new ArrayList<>(childByMat.keySet()), ...)` 依赖 `childByMat` 外层键是纯 `String materialNo`。聚合键复合化为 `(material_no, salesPartNo)` 后，此处须**从复合键回抽 material_no 并去重**（material_master 按生产料号唯一，不含 sales 维），否则登记入拼接垃圾键。

### 3.1 unit_price 费用类系统升版（一步到位，v3.0 新增）

**背景**：P13/P14/P15–P20/P22/P23 现走 `UnitPriceWriter` 覆盖式 upsert（P15–P20/P23 版本恒 `V_DEFAULT`、P13/P14/P22 读 Excel 版本直存），无 is_current、无版本历史。一步到位改为 `VersionedV6Writer` 系统升版（`unit_price` 已在写入器 `ALLOWED_TABLES` 白名单，机制现成）。

**统一 groupKey**（对齐落库方案 §5.2；`code`/`finished_material_no`/`operation_no` 是业务主体，`sales_part_no` 是隔离维度）：
```
groupKey = { system_type:'PRICING', price_type, cost_type, code,
             finished_material_no(缺省''), operation_no(缺省''), sales_part_no }
```
**content（组内明细 + 值字段，参与指纹比对，随内容变化升版）**：`seq_no, pricing_price, cost_ratio, currency, unit, defect_rate, market_ref_price, recovery_discount` 等**该 handler 实际写入的非键列**（逐 handler 取其写的列子集；`seq_no`/`discount_order`/`item_seq` 是组内明细维度→进 content 不进 groupKey）。

> 每个费用 handler 按其现有 `UnitPriceWriter.newRow(...)` 所设字段，把「固定/键字段」归 groupKey、「值/明细字段」归 content，逐 handler 定 CONTENT 常量（与 P08/P21 现写法一致）。

**迁移坑清单（实现必查，否则运行时炸或静默丢）**：

1. **`created_at`/`updated_at` DB 默认值（高危）**：`VersionedV6Writer.insertRowsBatched` 是**原生 INSERT，绕过 Hibernate `@CreationTimestamp`**，只写 groupKey+content+version+is_current 列。若 `unit_price.created_at`/`updated_at` 是 `NOT NULL` 且**无 DB 级 DEFAULT** → 插入直接报 `null value in column`。**实现 Task 首步须实测** `unit_price` 这两列的 DB 默认（`\d unit_price`）；无默认则 V311 里 `ALTER COLUMN created_at SET DEFAULT now()`（updated_at 同）——否则费用类系统升版第一条 INSERT 就崩。P08/P21 能跑说明 capacity/plating_scheme 有默认，unit_price **须单独确认**。
2. **P01/P02 共表**：`writeVersionedGroups` 按 `system_type='PRICING'` 常量前缀 `loadCurrentByPrefix`（`WHERE prefix AND is_current=TRUE`）会**载入 P01/P02 全局行**，但它们 price_type=ELEMENT/MATERIAL_PRICE、不在费用 handler 传入的 `groups` 里 → 只被载入比对、不被 flip/insert，**不受影响**（安全，但确认全局行 is_current 默认 TRUE 不被误翻）。
3. **动态 cost_type 前缀非常量**：P16/P17/P19/P20 的 cost_type 逐行变 → 该 sheet 的 constPrefix 退化为 `{system_type}`（仍非空，合法）；P13/P14/P15/P18/P22/P23 cost_type 固定 → 前缀更紧。均满足 `writeVersionedGroups` 的"前缀非空"要求。
4. **effective_date**：写入器不写该列 → NULL（见 §4.1，顺带修翻倍隐患）。
5. **is_current 列**：`unit_price` 已由 V277 加 `is_current`（费用行升版翻转，全局行保持默认 TRUE）。
6. **`unit_price` uq 仍需 14 维（含 sales）**：系统升版靠 version_no 区分版本行、is_current 标当前，但**物理唯一仍由 uq_unit_price 保证**（§1.2），故 §1.2 的 uq 加维照做，不因走 VersionedV6Writer 而省。

## 4. 写入器 / handler 改造契约（逐类）

### 4.1 `unit_price` 费用类（P13/P14/P15/P16/P17/P18/P19/P20/P22/P23）→ `VersionedV6Writer` 系统升版（一步到位，v3.0）

> 见 §3.1 的 groupKey/content 契约与迁移坑清单。本节给 handler 级改造要点。

- **write 路径切换**：这 10 个 handler 由「逐行 `UnitPriceWriter.upsert`」改为「按 §3.1 groupKey 累积成组 → sheet 末尾一次 `versionedWriter.writeVersionedGroups("unit_price", "version_no", CONTENT, TRIGGER, groups)`」（与 P08/P21 现有写法同构）。version_no 由写入器系统生成（2000/2001…），**不再读 Excel 版本列、不再落 `V_DEFAULT`**。

## 7. 存量迁移

**已定（2026-07-07 用户确认）＝全删核价存量 + 重导**（对齐报价 Spec1）。理由：核价基础数据从 Excel 整簿周期性重导、非手工累积；旧存量无 `sales_part_no`，回填信息不在旧数据里，只能靠重导带入。全删重导比"回填 + 兼容空值"简单且无脏值风险。
