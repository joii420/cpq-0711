## 1. 背景与需求

报价/核价的公式配置目前要求用户手写 token（`[页签#N]`、`cross_tab_ref` 匹配键、`SUM_OVER(... WHERE ...)`、`@gvar:CODE`…）。实战反馈最大卡点是**不会写表达式语法**——用户知道要算什么（如「按料件关联，求和所有满足行的 来料.组成用量 × 元素.含量 × 元素.单价」），但拼不对 token / 匹配键 / 聚合写法。

本方案把公式配置改造为**纯可视化构建器**：用户点选「要算什么 → 选 Tab/列 → 选关联键 → 拼算式 → 选聚合」，系统自动生成底层 token，用户不接触语法。

### 1.1 需求共识（9 问澄清结论）

| 维度 | 结论 | 对设计的含义 |
|---|---|---|
| 目标用户 | 实施管理员 / 销售报价员 / 技术顾问 **三类全兼顾** | **分层**：简单模式（默认）+ 高级模式 |
| 范围 | 整个 Excel / 卡片公式配置 | 公式配置整体可用性，非单一场景 |
| 最大痛点 | 不会写表达式语法 | 核心目标=**消灭手写 token** |
| 达成方式 | 纯可视化构建器 | 点选搭积木，系统生成 token |
| v1 覆盖 | 本行四则 / 跨页签取单值 / 跨页签求和(SUMPRODUCT) / 引用小计·全局变量 | 四类积木全要 |
| 关联键 | **只列可选项、不智能提示** | 构建器只管「拼公式」，不校验键是否对得上（数据对齐在构建器外） |
| 条件过滤 | 要全套（多条件 AND/OR + IF 分支） | IF 需引擎函数化 → **必然分期** |
| 入口形态 | **两个抽屉各自增强、不合并** | CrossTabRefDrawer（组件字段）+ CardFormulaDrawer（Excel 列）分别做 |
| 原始公式 | **保留，可视化为主 + 可切原始文本（双向同步）** | 双模式 |
| 本轮交付 | 先出路线图 + 正式设计文档；P1 先做 CrossTabRefDrawer | 当前=设计；plan 与实施在后 |

## 2. 现状能力盘点（分期依据）

**关键事实：用户要的能力里，只有 IF 分支需要动引擎；其余后端均已具备**——P1/P2 主要是把已具备能力包成可视化积木（低风险），P3 才是引擎级硬骨头。

| 积木 | 后端现状 | 缺口 |
|---|---|---|
| 本行四则 `[X] ± × ÷` | ✅ 经典 FORMULA / 卡片公式 | 仅可视化 |
| 跨页签取单值（按键 VLOOKUP） | ✅ cross_tab_ref（match + agg=NONE，单列 target） | 仅可视化引导 |
| 跨页签求和 / SUMPRODUCT | ✅ cross_tab_ref（agg=SUM + targetExpr `A列×B列`），语义=逐行先乘再求和 | 仅可视化引导 |
| 引用小计 / 全局变量 | ✅ component_subtotal / global_variable token | 仅可视化 |
| 条件求和 SUMIF / 多条件 AND·OR | ✅ SUM_OVER 动态 WHERE（condRows：字面量 / 产品字段 / 已算列 RHS，AND/OR） | 仅可视化「加条件」UI |
| **IF 分支（满足取 A 否则取 B）** | ❌ 卡片公式引擎**无函数调用机制** | **前后端双引擎函数化（大工程）** |

> 两套引擎边界（不在本期合并，用户已确认）：
> - **组件字段** → cross_tab_ref（`FormulaCalculator` / 前端 `formulaEngine.ts`），UI = `CrossTabRefDrawer`。
> - **Excel 卡片列** → SUM_OVER / 卡片公式（`TemplateFormulaService` / `CardFormulaEvaluator`），UI = `CardFormulaDrawer`。

## 3. 分期路线图

### P0 · 数据准备（前置，在构建器之外）
- 跨页签关联前提：两个 Tab 要有「对得上的键」。典型坑：`来料.料件 = 料件名称(material_name)` vs `元素.料件 = 料号(material_no)`，直接等值匹配零命中。
- 解法：视图层补公共键列（如 `ll_view` 增 `子料号 = mbi.component_no`），由配置管理员/技术顾问一次性准备。
- 用户已明确**构建器不做智能键提示**，故此步刻意留在构建器外。是否做独立「关联键体检」只读小工具 → 排 P2 可选，**不进 P1**。

### P1 · 可视化积木构建器 v1 — 先做 CrossTabRefDrawer（组件字段）
- **本期落地用户的跨页签求和场景**（详见 §4）。
- CrossTabRefDrawer 由「插入式表单」重构为「引导式 + 分层 + 双模式」可视化构建器。
- CardFormulaDrawer 增强（本行四则 / 引用小计 / 全局变量 / 无条件 SUM_OVER 可视化）随后排入 P1.2。

### P2 · 条件 / 过滤可视化（SUMIF + 多条件 AND/OR）
- 复用已落 condRows 动态 WHERE，构建器加「+ 加条件」→ 条件行（左列 / 运算符 / 右值来源：字面量·产品字段·已算列）→ AND/OR。
- 直接覆盖「非银点」过滤（类型 ≠ 银点）等。
- 可选并入「关联键体检」只读提示。

### P3 · IF 分支 / 卡片引擎函数化（大工程，单独立项）
- 前后端双公式引擎加**函数调用 token** + 移植函数库（IF / ROUND / 汇率…）+ 共享夹具锁前后端一致。
- 函数化后构建器加「条件分支（满足取 A 否则取 B）」积木，补齐「全套条件逻辑」。

## 4. P1 详细设计：CrossTabRefDrawer 可视化重构

### 4.1 现状（已有基础，复用不重写底层）
`cpq-frontend/src/pages/component/CrossTabRefDrawer.tsx`（562 行）已具备：
- 源组件下拉（`sourceId`）、关联键 a/b 双下拉（`matchPairs[{a,b}]`，**已是"只列可选项"形态**，符合需求）。
- 聚合下拉（`agg`：NONE/SUM/AVG/COUNT/MAX/MIN）。
- 单列 target 下拉 + 「改用公式」(`useFormula`) 的 targetExpr chip 构建器（插入 A 字段 / B 字段 / 运算符 / 数字 / 全局变量）。

**token 结构不变（纯前端 UX 改造，无协议变更，向后兼容）**：
```jsonc
{ "type":"cross_tab_ref", "source":"<A componentId>", "sourceLabel":"来料",
  "target":"<单列>", "targetExpr":[/* ExpressionToken[] */],
  "match":[{"a":"A列","b":"B列"}], "agg":"NONE|SUM|AVG|COUNT|MAX|MIN" }
```

### 4.2 目标交互：引导式四步（默认简单模式）
把现状的扁平表单重排为一条「**要算什么 → 从哪取 → 怎么对上 → 算什么 × 怎么汇总**」的引导主线（Steps / 分段卡片，单抽屉内不跳页）：

1. **第 1 步「要算什么」**——业务措辞的操作选择（替代直接选 agg 术语）：
   - 「从另一个页签**取一个值**」（→ agg=NONE，单匹配）
   - 「把满足条件的行**求和**」（→ agg=SUM）
   - 「求**平均 / 最大 / 最小 / 计数**」（→ AVG/MAX/MIN/COUNT，折叠在"更多汇总方式"里，给高级用户）
2. **第 2 步「从哪个页签取」**——源组件下拉（`sourceLabel` 显示中文名）。
3. **第 3 步「两个页签怎么对上」**——关联键 a/b 双下拉（**纯列举、不智能提示**，符合需求；可加多对 AND）。措辞：「源页签的 [A列] 等于 本页签的 [B列]」。
4. **第 4 步「算什么」**——
   - 简单模式默认：单列下拉（取一个列值）。
   - 「要逐行相乘/相加」开关 → 展开 targetExpr 积木条：点选【源页签列】【本页签列】【× ÷ + − ( )】【数字】【全局变量】，chip 回显 + 中文可读预览（复用现 `tokenToText`）。
   - 文案点明语义：求和时「逐行先按此式算、再求和」（SUMPRODUCT），避免误解为先各自求和再乘。

### 4.3 三类用户分层（简单 / 高级 一键切）
- **简单模式（默认）**：只露「取一个值 / 求和」两种操作 + 单列或一条乘积式 + 中文标签；隐藏 COUNT/MAX/MIN、多关联对、token 术语。
- **高级模式**：露全部聚合方式、多关联对、targetExpr 全积木、**原始文本切换**（§4.4）。
- 实现：抽屉头部一个「简单 / 高级」`Segmented`；模式状态决定区块可见性，不影响生成的 token。

### 4.4 双模式：可视化 ↔ 原始公式文本（双向同步，高级模式）
- 高级模式提供「查看/编辑原始公式」`TextArea`，与可视化区**双向同步**：
  - 可视化变更 → 实时序列化为可读 token 文本（复用 `tokenToText` / FormulaZone 风格）。
  - 文本手改 → 解析回 token 结构回填可视化区（复用现有反解析基础；解析失败 → 行内报错 + 阻止确定，不静默）。
- 满足「保留原始公式入口、可视化为主」的需求；防错靠**只给合法选项**（Tab/列/聚合枚举），文本模式是高级逃生口。

### 4.5 关联键处理（明确不做智能提示）
- 仅把源页签列、本组件列分别列入 a/b 下拉，用户自行选。
- **不**做"键体检/零命中预警/自动补键"（用户明确选择）。数据层对齐（P0）由管理员负责。
- 仅保留最低限度结构校验：关联键至少一对填全、targetExpr 非空（非 COUNT），沿用现 `buildInsertResult` 校验。

### 4.6 范围裁剪（P1 不含）
- 条件 / 过滤（SUMIF WHERE、多条件 AND/OR）→ P2。
- IF 分支 / 函数 → P3。
- CardFormulaDrawer（Excel 列）可视化 → P1.2（本期先 CrossTabRefDrawer）。
- 关联键体检小工具 → P2 可选。

## 5. 落地映射：QT-20260609-1652（你的跨页签求和）

目标式：`Σ over (来料,元素) where 来料.料件=元素.料件 of (来料.组成用量 × 元素.含量 × 元素.单价)`。

- **P0 前置**：`ll_view` 增 `子料号 = mbi.component_no`，使 `来料.子料号` 与 `元素.料件(=ebi.material_no)` 可等值匹配。
- **P1 用 CrossTabRefDrawer 配**（在「元素」或「来料」组件的小计字段）：
  - 操作=「求和」（agg=SUM）；源页签=来料；关联键=`来料.子料号 = 元素.料件`；
  - 算什么=逐行式 `来料.组成用量 × 含量 × 单价`（targetExpr：field 组成用量 × b_field 含量 × b_field 单价）；
  - 小计字段 `is_subtotal=true` → 引擎跨行求和 = 目标总额。
- **口径提醒**：`含量` 若为百分数（如 90）需 ÷100，与现有元素小计口径一致；是否含 `单重` 由业务确认。
- Excel 模板那一列再引用该小计（component_subtotal）即可。

## 6. 设计原则（贯穿全程）
1. **中文业务措辞**：抽屉文案不出现 token/JEXL 术语；术语仅在高级模式原始文本里出现。
2. **三类用户分层**：同一抽屉简单/高级一键切，不另起界面。
3. **防错靠枚举**：Tab/列/聚合/操作均下拉选择，杜绝自由文本拼写错（高级原始文本除外）。
4. **协议纪律**：本期 token 结构不变（向后兼容）；若 P2/P3 新增 token/字段类型，走 AP-44/AP-55 全套（17 检查点 + 双引擎对等 + 共享夹具 + 双 spec E2E）。

## 8. 向后兼容 / 已知风险
- **token 结构零变更** → 历史 cross_tab_ref 配置不迁移、照常工作；新 UI 仅改"怎么配"，不改"配出来是什么"。
- 风险：双向同步的文本→token 反解析需健壮（解析失败要可恢复、不丢已配内容）；P1 文本模式定位为高级逃生口，解析失败=行内报错 + 维持可视化态，不破坏已配。
- 风险：引导式 Steps 重排不得削弱"多关联对 / 全 agg / targetExpr 全积木"既有能力——高级模式必须 100% 暴露现状全部能力。
