## 0. 一句话目标

把核价单渲染从「**每卡片 × 每递归组件各跑一次 BOM 闭包 + 逐 occurrence 用 spineKeys 挂数据**」重构为「**一次可配置递归 SQL 跑全量生产料号 + 每页签 SQL 只跑一次 + 后端按 `material_no` 分组到各卡片**」。总 SQL 执行次数 = 1(递归) + N(页签数),其余全是后端内存分组。

**实现结果必须与本规格逐条一致。** 任何偏离在实现前回本文确认。

---

## 1. 范围与非目标

### 1.1 范围
- **仅核价侧(COSTING)**。核价模板里**含核价树页签**时,整单走本新管线;不含则核价单按现有平铺渲染,不受影响。
- **彻底替换**现有:`BomClosureService`(逐根闭包)+ `SpineKeysMacro`/`SpineKeysContext`(spineKeys 三元组)+ `buildSpineBaseRows`(逐 occurrence 按 `hf_part_no` 挂数据)+ `expandForPartSet`。

### 1.2 非目标(明确不动)
- 报价侧(QUOTATION)渲染管线完全不变。SqlViewExecutor 是共享的,报价侧的 customer/lineItem 等上下文注入**保持不变**——只是新核价管线**不设**这些上下文(见 §5)。
- 下游机制不变:公式引擎、页签小计/合计、行编辑(手工输入行/改值)、`costing_card_values` 快照。本方案**只改「每个页签的 baseRows 从哪来」**,拿到分好组的 baseRows 之后与现在完全一致。

### 1.3 存量策略:硬切换、不迁移(用户裁决 2026-07-02)
- **硬切换**:删除旧引擎(`BomClosureService` / spineKeys 等),核价侧统一走新管线,**不保留双路径**。
- **不写任何存量迁移**:不迁 `bom_recursive_expand` 字段、不迁 `template.components_snapshot`、不重写存量核价 `component_sql_view`。
- **新规则只对新数据生效**:`bom_recursive_expand` 新语义、§11「一模板最多一个树页签」校验、页签 SQL 新契约(输出 `material_no` + 用 `:total_material_no`)**只在新建 / 新保存时生效与校验**。
- **存量核价报价单 / 模板 / 视图不受支持**:打开旧核价单可能报错或空。**接受此后果,必要时直接删除存量核价数据**(建议清理范围见 §12)。这条是采用「仅 material_no 匹配 + 不迁移」简化路线的前提。

## 2. 领域模型:确认过的完整语义

| 概念 | 定义 |
|---|---|
| 产品卡片 | = 一个生产料号(1 卡 = 1 `product_part_no`) |
| seed(递归根集合) | 本报价单**所有 line item 的 `product_part_no`**(非空、去重) |
| 递归 SQL | **全系统唯一一条**、可配置;跑一次全量 seed |
| `root_no` | 递归行归属的根生产料号(隐藏列,分组用) |
| `material_no` | 当前料号(显示) |
| `bom_version` | 当前料号版本号(显示) |
| `parent_no` | 上级料号(隐藏,建父子层级) |
| `card_material_no[卡片]` | `distinct(material_no where root_no = 该卡片生产料号)`,**含根本身** |
| `total_material_no` | `distinct(所有 material_no)` = 各卡片 `card_material_no` 的并集 |
| 匹配键 | **仅 `material_no`**(版本号不参与匹配) |
| 落选行 | 页签结果行的 `material_no` 不属任何卡片 → **直接丢弃** |
| 多 occurrence | 同一卡片内同料号挂多父 → 树里多个节点,业务数据按 `material_no` 同时挂到各节点(**保留,不去重**) |
| 一行进多卡 | 同一 `material_no` 落在多个卡片的 `card_material_no` 里 → 该行进多个卡片 |

> **多 occurrence 的小计语义(用户裁决 2026-07-02,明确不是 bug)**:同一料号在一张卡片内挂多个父件时,其业务行按 `material_no` 复制到每个 occurrence 节点,**页签小计对每个 occurrence 各算一次**。这是业务预期(料号确实被用在多处,成本本就该按位置各计)。因此匹配键**只用 `material_no`、不带父件维度**是正确的,不引入 spineKeys 式三元组。`is_current` 保证同料号不会因多版本产生额外重复行。

## 3. 数据模型(Flyway 新表)

```sql
CREATE TABLE costing_bom_tree_config (
    id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    name         text NOT NULL,              -- 配置名(区分多条候选)
    sql_template text NOT NULL,              -- 递归 SQL 全文
    is_active    boolean NOT NULL DEFAULT false,  -- 生效行,全局唯一
    created_at   timestamptz NOT NULL DEFAULT now(),
    updated_at   timestamptz NOT NULL DEFAULT now()
);

-- 全局同一时刻最多一条生效
CREATE UNIQUE INDEX ux_cbt_active ON costing_bom_tree_config (is_active) WHERE is_active;
```

- 「设为生效」= 单事务内先把当前 active 置 false,再把目标置 true(避开部分唯一索引冲突)。

## 4. 契约(逐字约定,实现与文档必须一致)

### 4.1 递归 SQL(配置员手写)
- **输入**:`:production_part_nos`(`text[]`),作为递归 seed 的根集合。
- **输出列名(逐字一致)**:
  - `root_no` —— 根生产料号(隐藏,分组用)
  - `material_no` —— 当前料号(显示)
  - `bom_version` —— 当前料号版本号(显示;叶子自身无 BOM 版本可为 NULL)
  - `parent_no` —— 上级料号(隐藏,建层级;根行为 NULL)
- 示例骨架(仅示意,具体分区条件由配置员写):
```sql
WITH RECURSIVE bom AS (
  SELECT p AS root_no, p AS material_no,
         (SELECT bom_version FROM material_bom_item x
           WHERE x.material_no = p AND x.customer_no='_GLOBAL_'
             AND x.system_type='PRICING' AND x.is_current LIMIT 1) AS bom_version,
         CAST(NULL AS varchar) AS parent_no
  FROM unnest(:production_part_nos) AS p
  UNION ALL
  SELECT b.root_no, ch.component_no,
         ch.bom_version, ch.material_no
  FROM material_bom_item ch
  JOIN bom b ON ch.material_no = b.material_no
  WHERE ch.customer_no='_GLOBAL_' AND ch.system_type='PRICING'
    AND ch.is_current AND ch.component_no IS NOT NULL
) CYCLE material_no SET is_cyc USING pth
SELECT root_no, material_no, bom_version, parent_no FROM bom;
```
> 注:`bom_version` 语义 = 当前料号(节点自身)版本;实现细节由配置员的 SQL 决定,后端只按列名取值,不干预算法。

### 4.2 页签 SQL(组件 `$view`,树页签与普通页签同契约)
- **输入**:`:total_material_no`(`text[]`);SQL 内写 `WHERE <料号列> = ANY(:total_material_no)`(或等价 IN)。
- **输出**:**必含 `material_no` 列**(逐字);其余为业务列。
- 分区条件(PRICING/_GLOBAL_/is_current 等)由 SQL 自己写死(见 §5,不注入上下文变量)。

## 5. 变量注入(`SqlViewExecutor`)

- 新增两个数组绑定入口:
  - 执行**递归 SQL** 时:`:production_part_nos` → `text[]`(seed)。
  - 执行**页签 SQL** 时:`:total_material_no` → `text[]`。
- 数组绑定复用 commit `6d9787b` 已修的 `List→text[]` NULL-safe 逻辑(保留 SQL NULL,不转字符串 `"null"`)。
- **不注入** `customer_id` / `quotation_id` / `lineItemId` 等上下文变量(确认取舍 A):页签/递归 SQL 只吃料号类变量,分区自理。
- 保存期 dry-run 校验时,这两个变量用 `ARRAY[]::text[]` 字面量占位(仿 `SpineKeysMacro.expandForValidation`),不污染 `required_variables`。

## 6. 编排服务 `CostingTreeRenderService`(整单一次)

新建服务;`CardSnapshotService` 在「核价侧 + 模板含核价树页签」时**整单委托**给它。算法:

```
输入: quotation, 所有 lineItems, 核价模板 templateId

1. seed = distinct( lineItems 中非空的 productPartNo )
   若 seed 为空 → 返回空(无卡片可渲染)

2. 取生效递归 SQL(costing_bom_tree_config.is_active=true)
   无生效行 → 抛业务错(见 §11),不静默空树
   跑 1 次,绑 :production_part_nos = seed
   → treeRows[]: { root_no, material_no, bom_version, parent_no }

3. 后端内存分组(不查库):
   cardMaterialNo : Map<root_no, LinkedHashSet<material_no>>   // 含根;保序
   totalMaterialNo: distinct( 所有 treeRows.material_no )
   treeRowsByRoot : Map<root_no, List<treeRow>>                // 树页签建树用,保序

4. 对模板每个 driver 组件页签 comp(有 $view 的):
   跑其 $view 1 次,绑 :total_material_no = totalMaterialNo
   → rowsByMaterial[comp]: Map<material_no, List<业务行>>       // 一料号可多行;落选(不在任何卡)自然不进任何卡片
   注:一个组件在模板里多实例(同 compId 多 sortOrder)按现有 tabKey=compId:sortOrder 维度分别持有

5. 对每张卡片 li(root = li.productPartNo)、每个页签 comp:
   - 普通页签:
       out = flatten( material ∈ cardMaterialNo[root] 的 rowsByMaterial[comp][material] )
       平铺;落选行天然不在集合内 → 丢弃
   - 树页签(comp 勾了核价树):
       遍历 treeRowsByRoot[root] 每个节点 node(保留多 occurrence):
         biz = rowsByMaterial[comp][node.material_no]
         有行 → 对每条 biz 输出一行,注入系统列(见 §7)
         无行 → 输出空行 + 系统列

6. 产出 Map<lineItemId, Map<compId(:sortOrder), baseRows: ArrayNode>>
   回填现有快照/公式管线(与逐卡 expandTemplateDriverBaseRows 的返回结构一致)
```

- **执行次数**:1(递归) + N(页签数)。步骤 3/5/6 纯内存。
- **确定性**:分组用 `LinkedHashSet` / 稳定排序(沿用现有 `stableRowKey` 思路),满足 AP-51「刷新行数稳定」。

## 7. 树页签:三系统列 + 层级

后端为树页签每行注入(`__` 前缀系统列,协议与现有前端一致):

| 系统列 | 值 |
|---|---|
| `__hfPartNo` | 当前料号(= node.material_no) |
| `__bomVersion` | 版本号(= node.bom_version) |
| `__parentNo` | 上级料号(= node.parent_no,隐藏) |
| `__nodeId` | 节点唯一 id(由 `root_no` + 层级路径/出现序生成,保多 occurrence 唯一) |
| `__parentId` | 父节点 `__nodeId`(根行为 null) |
| `__lvl` | 层级深度(根=1) |

- **前端渲染层保持不变**:沿用现有「核价侧 && 行含 `__sys.nodeId` → 渲染树 + 固定带出『料号 / 版本』两列 + 按 `__parentId→__nodeId` 建层级」。`__parentNo` 仅用于建层级,不单独显示为列。
- 因系统列协议逐字沿用,前端 `QuotationStep2` 树渲染分支**理论上零改动**;但 `QuotationStep2.tsx` 归一化对 nodeId 规则敏感(根 `nodeId=''`→`__bomroot__`、DAG occurrence 各自独立),实现时**须 E2E 实证零改动**,不能只声称。

### 7.1 `__nodeId` 单一真理源(关键约束)
- `__nodeId` / `__parentId` 的生成算法**只能有一处**(在 `CostingTreeRenderService`,写进 `costing_card_values` 快照)。
- **两条渲染面(产品卡片 + Excel 视图)都从快照读同一套 `__nodeId`,禁止任何一方再自行重算树骨架**(现状 Excel 侧 `buildLineTreeRows` 自己重算 = 本方案要根除的隐患,见 §8)。
- occurrence 唯一性:同料号挂多父的多个节点必须有各自唯一的 `__nodeId`(由 `root_no` + 父路径/出现序生成),否则前端建层级会塌。

## 8. 集成点(两条渲染面共用同一棵树)

核价树在系统里有**两条并列渲染面**,新方案下**都消费同一棵树**(由 `CostingTreeRenderService` 产出、落进 `costing_card_values` 快照的 `__nodeId` 骨架):

### 8.1 产品卡片视图 `CardSnapshotService`
- 判定入口:核价侧渲染时,查模板是否含「勾了核价树(`bom_recursive_expand=true`)的组件」。
  - 是 → 整单调 `CostingTreeRenderService`,拿回 `Map<lineItemId, Map<compId, baseRows>>`,替代逐 li 的 `expandTemplateDriverBaseRows(..., closure, ...)` closure 分支。
  - 否 → 走现有非递归平铺路径(不变)。
- 拿到 baseRows 后:公式引擎 / 小计合计 / 行编辑 / `costing_card_values` 快照**完全复用,不变**。

### 8.2 Excel 视图核价树 `ExcelViewService.buildLineTreeRows`(必改,用户确认 Excel 也是树模式且用核价树 SQL 做骨架)
- **现状(要根除)**:`ExcelViewService.java:~252` 自己再调一次旧引擎 `BomClosureService.compute(...)` 重算 spine,再按 `node.nodeId` 用 `CardEffectiveRows.filterByNodeId` 去卡片快照对号取行。
- **新做法**:改为**直接读 `costing_card_values` 快照里由 `CostingTreeRenderService` 写好的树骨架(`__nodeId/__parentId/__lvl/__hfPartNo/...`)**,不再自行重算闭包。Excel 与产品卡片因此共用**同一 `__nodeId` 真理源**,渲染逐节点一致。
- 若删旧引擎而不改此处 → 编译不过 / Excel 核价树整列空,故此项与 §9 删除**必须同批完成**。

### 8.3 详情页 `ReadonlyProductCard`(待核,可能是第三渲染面)
- 实现时用 `codegraph` 核实它是否也读 `__sys.nodeId` 树协议;若是,同样只读快照骨架、不重算,并纳入回归。

## 9. 删除清单(旧 spine 机制)

实现时移除/改造:

- **删**:`BomClosureService`(`CLOSURE_SQL` / `compute` / `partSet` / `spine`)、`BomClosureResult`(及 `SpineNode`)。
- **删**:`SpineKeysMacro`、`SpineKeysContext`、`ComponentDriverService.expandForPartSet`、`CardSnapshotService.buildSpineBaseRows` / `spineRowNode` / `expandTemplateDriverBaseRows` 的 closure 分支与相关重载。
- **删**:`SqlViewExecutor` 里 spineKeys(`__skP/__skPP/__skV`)注入路径。
- **改**:`ExcelViewService.buildLineTreeRows` —— 从「自调 `BomClosureService.compute` 重算」改为「读快照树骨架」(见 §8.2),否则删旧引擎后编译不过。
- **删**:旧引擎相关测试 —— 至少 `BomClosureServiceTest`,以及注入了 `bomClosureService` 的 `SaveDraftProfileTest` / `S3SegmentProfileTest`(改写或清理,避免编译失败)。删除前用 `codegraph_callers bomClosureService` 拉全注入点。
- **保留、改语义**:组件字段 `bom_recursive_expand` —— 语义由「该组件走 BOM 闭包递归」改为「**该组件页签 = 核价树页签**(渲染树 + 三系统列)」;删除它触发老闭包的所有逻辑。**存量该字段的历史值不迁移**(§1.3),只影响新数据。
- **文档**:改写 `docs/核价树页签组件配置指南.md`(去 spineKeys/4A-4B 二选一;改为「递归 SQL 全局配置 + 页签 SQL 用 `:total_material_no` + 输出 `material_no`」);相关旧 spec 标记 superseded。

> **删除前一律用 `codegraph_impact` / `codegraph_callers` 逐一核对引用面。** 评审已核实:
> - `ExcelViewService.buildLineTreeRows` **确有**依赖 → 按 §8.2 改。
> - `ConfigureSnapshotService` **不依赖** BomClosure(其 driver 展开走独立路径)→ **不在删除/迁移清单内,勿误动**。
> - impact 里出现的 `VersionedV6Writer` 等为同名符号误命中,与本方案无关。

## 10. 时机与快照

- **打开核价单**:有 `costing_card_values` 快照 → 优先读快照(不实时算);无 → 实时跑新管线 → 落快照。
- **存草稿 / 提交**:跑新管线固化快照(与现有落快照点一致)。
- 触发/失效边界沿用现有草稿刷新逻辑(改配置/改视图/改字段后须重算,与现状一致)。
- **新增缓存失效钩子**:切换 `costing_bom_tree_config` 生效行 / 改递归 SQL 后,必须让相关核价快照失效重算(否则改配置后打开仍读旧快照)。现有 `ComponentDriverService.evictForLineItem` 是 li 维度,新整单编排需定义自己的失效入口(至少:改生效行时清核价 draft 快照 / 标记需重算)。改页签 `$view` 沿用现有视图改动失效路径。

## 11. 校验规则

| 场景 | 行为 |
|---|---|
| 一个核价模板 > 1 个「勾核价树」组件 | 保存模板 / 渲染前校验 → 报错(最多一个) |
| 模板有树页签但无生效递归 SQL | 渲染时明确报业务错(不静默空树) |
| 页签 SQL 缺 `material_no` 输出列 | 保存期 dry-run 拦截 |
| 递归 SQL 缺 4 个约定输出列 / 未引用 `:production_part_nos` | 保存 `costing_bom_tree_config` 时 dry-run 拦截 |
| 设为生效 | 单事务先清旧 active 再置新 active(唯一索引保证) |

> **校验只在新建 / 新保存时生效**(§1.3):这些校验拦的是新数据。存量模板即便违反(如有多个 recursive 组件)也不迁移、不回补校验——存量核价数据不受支持,必要时清理(§12)。

## 12. 边界情况

- line item 无生产料号 → 不进 seed。
- 根料号无子件 → 树仅根一行。
- 页签行 `material_no` 落空(不属任何卡) → 丢弃。
- 同料号跨多卡共享 → 递归天然按 `root_no` 分别产行,各卡 `card_material_no` 独立命中。
- 同料号卡内挂多父(多 occurrence) → 树多节点,业务行按 `material_no` 复制挂到各节点,小计按位置各算一次(§2 已定,为预期)。
- 两个 line item 生产料号相同(重复行) → seed 去重后一次递归;两卡片 `card_material_no` 相同,各自独立渲染。

### 12.1 硬切换后的存量清理(可选,用户已授权必要时删除)
- 删旧引擎后,**存量核价报价单**(依赖旧 spine 快照 / 旧 spineKeys 视图的)打开可能报错或空。
- 建议清理范围(按需,由用户确认后执行):含旧 `__nodeId` 语义的 `costing_card_values` 快照、依赖旧 spineKeys 的存量核价 `component_sql_view`、含多个 `bom_recursive_expand=true` 组件的存量核价模板。
- **不做自动数据迁移**;清理是一次性运维动作,不进主流程。
