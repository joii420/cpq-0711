## 0. 术语与关键事实（读码结论）

| 术语 | 指 | 现役坐标 |
|---|---|---|
| 生产侧全局指纹 | `FingerprintCalculator.simpleFingerprint/compositeFingerprint`（sha256，`v1|SIMPLE|recipe|elem...`），落 `material_master.config_fingerprint`（`uq_material_master_fingerprint` **全局唯一**） | `FingerprintCalculator.java:42/58/87`；`ConfigureProductService.lookupHfByFingerprint:115-122` |
| 销售侧指纹（本方案新增） | 客户维度、落 `sel_part_signature`，与生产侧指纹**两码事** | 3b 新建 |
| 报价料号 | `XXXX-YYMMNNNNNN`（15 位），`QuoteMaterialNoAllocator.mintAndRegister(customerNo, yyMm)` 铸号 + 登记 `material_customer_map` QUOTE 行 | `QuoteMaterialNoAllocator.java:35-48` |
| CFG- 料号 | 现役选配产出 `PartNoProvider.apply` → `CFG-{symbol}-{6位}` | `AutoAllocatePartNoProvider.java:34/42`；`PartNoContext.java` |
| hfPartNo | resolvePart 返回值 = 选配产出料号（今为 CFG-，本方案改报价料号），落 `material_master.material_no` + 各 V6 QUOTE 行 `material_no` + `quotation_line_item.product_part_no_snapshot` | `ConfigureProductService.resolvePart:180`、`insertLineItem:1010` |

**关键读码事实（决定 3b/3c 全部设计）**：

1. **现役 custom 分支已「无前置料号从零建料号 + material_master + element/material/工序」**——`resolvePart:262/263/268/255` 已调 `insertMaterialMasterV6 / insertElementBomV6 / insertMaterialBomItemV6 / insertProcessSimpleUnitPriceV6`，均按 `(customerCode, hfPartNo)` 落 QUOTE 行。**AP-53 防断链在现役已成立**，3c 不是从零搭，而是「换料号来源 + 补 sales_part_no 维度 + handler 化分发」。
2. **料号发号仅 2 处**：SIMPLE custom `resolvePart:247-248`、COMPOSITE 父 `configure:796-797`，都是 `partNoProvider.apply(PartNoContext(...))`。3b 只需替换这两处。
3. **`material_master.config_fingerprint` 全局唯一是硬约束**。现役 custom 分支先 `lookupHfByFingerprint`（生产侧全局）命中即**跨客户复用**同一 hfPartNo。这与「报价料号客户维度、号里嵌客户码」**根本冲突**——同物质不同客户必须各自发号，不能全局复用。故 3b 必须让选配 custom 分支**改用销售侧客户维度指纹判复用，且写 material_master 时 config_fingerprint 置 NULL**（否则两客户同物质 → 同 config_fingerprint → 撞全局唯一索引，INSERT 崩）。这是本方案最易踩、需求 spec 未点破的落地硬点。
4. `mintAndRegister` 的第一个入参语义 = `material_customer_map.customer_no` 的取值；现役 V6 mirror 视图按 `customer_no = customer.code` 过滤，`ConfigureProductService.getCustomerCodeFromCustomerId:850` 已派生 `customerCode`。故发号传 `customerCode`，与 spec「传 V6 customer.code」一致，且与 mirror 过滤同源，闭环。
5. `isCfg` 仅 2 个调用点（`MaterialBomMergeHandler.java:72/106`），`isCfg:262-264` = `startsWith("CFG-")`。改前缀后它对报价料号选配号**不再命中**。

## 1. 集成总图（custom 分支改造前后）

### 1.1 SIMPLE custom 分支：现状（`resolvePart:223-274`）

```
custom 请求(recipeCode, elements, processIds, unitWeightGrams)
  → validateCustomPart                                    :228
  → fp = FingerprintCalculator.simpleFingerprint(生产侧)   :235
  → existing = lookupHfByFingerprint(fp)  // material_master 全局  :240
  → existing!=null ? 复用 existing : partNoProvider.apply(CFG-)     :242-248
                                        └ insertProcessSimpleUnitPriceV6 :255
  → insertMaterialMasterV6(hfPartNo,...,fp)   // config_fingerprint=fp  :262
  → insertElementBomV6 / insertMaterialBomItemV6                        :263/268
  → return hfPartNo
```

问题：跨客户复用（生产侧全局指纹）+ CFG- 发号，与报价料号客户维度/统一发号相悖。

### 1.2 SIMPLE custom 分支：目标态

```
custom 请求 + 销售配置上下文(customerNo=customerCode, industryCode, 启用参数集, 选值)
  → validateCustomPart                                    (不变)
  → sig = SalesFingerprintCalculator.compute(customerCode, structureVersion, 启用参数投影)   ← 3b 新
  → hit = SalesSignatureRepository.lookup(customerCode, structureVersion, sig.hash)          ← 3b 新
  → hit!=null ? 复用 hit.quotePartNo（跳过发号+跳过全部落库，数据已存在）
             : quotePartNo = QuoteMaterialNoAllocator.mintAndRegister(customerCode, yyMm)     ← 3b 发号 swap
               + SalesSignatureRepository.insert(customerCode, structureVersion, sig, quotePartNo)  ON CONFLICT DO NOTHING
  → [未命中时] persist：按 sel_param_type.persist_handler_key 分发（3c）
        MATERIAL_RECIPE_BIND → insertMaterialMasterV6(quotePartNo, ..., config_fingerprint=NULL) + insertMaterialBomItemV6
        ELEMENT_OVERRIDE     → insertElementBomV6
        PROCESS_LIST         → insertProcessSimpleUnitPriceV6
      三者写 V6 QUOTE 行时 material_no = sales_part_no = quotePartNo
  → return quotePartNo
```

要点：
- **生产侧 `FingerprintCalculator` + `lookupHfByFingerprint` 在 custom 判复用链路里退出**（保留类本身，`lookupFingerprint` 端点见 §4.3 处置）。
- **复用命中 = 完全跳过**发号与落库（幂等，数据在首次已落）。
- `material_master.config_fingerprint` 对选配报价料号一律 **NULL**（不参与生产侧全局去重）。

### 1.3 数据流（目标态）

```
报价单 → quotation.customer_id → customer.code(=customerCode) + customer.industry_code
                                          │
   SelTemplateService.getByIndustry(industryCode) → 启用参数集+限值（无则 __DEFAULT__ 回退）  ← 3a 运行时
                                          │
   前端渲染选值(MAT/ELE/PROC) → configure 请求(PartRequest + 销售配置上下文)
                                          │
   ConfigureProductService.configure → PASS1 resolvePart(每子件)
        3b: 算销售指纹 → sel_part_signature 命中?  ── 命中 → 复用 quotePartNo
                                                  └ 未命中 → mintAndRegister → 登记 sel_part_signature
        3c: persist_handler 分发 → material_master(fp=NULL) + element_bom_item + material_bom_item + unit_price
                                    (material_no = sales_part_no = quotePartNo)
                                          │
   PASS3 buildLineItems → quotation_line_item.product_part_no_snapshot = quotePartNo
                        → 卡片渲染走现役 mirror 视图（JOIN material_master.material_no）→ 四 Tab 有数据
```

COMPOSITE：子件逐个走上面链路；父组合体在 `configure:792-813` 处**同构改造**——组合体也算销售指纹（对 childQuotePartNos 排序集合 + customerCode）→ 命中复用 / 未命中 `mintAndRegister` 铸父报价料号，`insertMaterialMasterV6(parent, config_fingerprint=NULL)`。

## 2. Plan 3b 设计：销售侧指纹 + 去重 + 报价料号发号

### 2.1 `sel_part_signature` 建表 DDL（V314）

```sql
-- V314__sel_part_signature.sql
-- 销售侧客户维度指纹去重登记。独立于 material_master.config_fingerprint（生产侧全局）。
CREATE TABLE sel_part_signature (
    id                   UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    customer_no          VARCHAR(50)  NOT NULL,   -- = customer.code（与 material_customer_map.customer_no / mirror 过滤同源）
    structure_version    VARCHAR(10)  NOT NULL,   -- 结构/算法版本，如 'v1'（与 FingerprintCalculator.VERSION 命名统一，不双轨）
    config_fingerprint   CHAR(64)     NOT NULL,   -- sha256 hex（比对用）
    config_signature_text TEXT        NOT NULL,   -- 可读规范串（含全部启用槽位 + 空哨兵），= 该料号 schema 快照
    quote_part_no        VARCHAR(32)  NOT NULL,   -- 命中/新发报价料号 XXXX-YYMMNNNNNN（→ material_customer_map QUOTE 行 material_no）
    product_type         VARCHAR(16)  NOT NULL DEFAULT 'SIMPLE',  -- SIMPLE/COMPOSITE（同库分辨独立体 vs 组合体指纹）
    created_at           TIMESTAMPTZ  NOT NULL DEFAULT NOW(),
    CONSTRAINT uq_sel_part_signature UNIQUE (customer_no, structure_version, config_fingerprint)
);
CREATE INDEX idx_sel_part_signature_quote ON sel_part_signature(quote_part_no);
```

- **唯一键** = `(customer_no, structure_version, config_fingerprint)`（需求 spec §5.3）。`product_type` 不进唯一键——独立体与组合体的规范串结构天然不同（COMPOSITE 串是 childQuotePartNo 排序集合），指纹不会撞，`product_type` 仅作审计/查询辅助列。
- **不加选配列到 `material_customer_map`**（需求 spec 决策 #17，共享表不污染）。报价料号权威登记仍是 `mintAndRegister` 写的 QUOTE 行；`sel_part_signature` 只是「指纹→报价料号」的旁路去重索引。

### 2.2 指纹算法：归属类 + 伪代码

**新建 `com.cpq.configure.SalesFingerprintCalculator`（`@ApplicationScoped`）**，与生产侧 `FingerprintCalculator` 并列、职责分离（不改后者）：

```java
public static final String STRUCTURE_VERSION = "v1";   // 与 FingerprintCalculator.VERSION 命名统一

/** 启用参数投影 —— 由 3a 运行时按本次实际使用模板的 enabled 参数 + 选值构造。 */
public record EnabledParam(String paramTypeCode,          // MATERIAL / ELEMENT / PROCESS（sel_param_type.code）
                           String materialCode,            // MATERIAL: recipe/配比码；否则 null
                           List<ElementPct> elements,      // ELEMENT: 元素码+含量；否则 null
                           List<String> processCodes) {}   // PROCESS: 工序码集合（无序）；否则 null

public record Signature(String hash, String text) {}

public Signature computeSimple(String customerNo, List<EnabledParam> enabled) {
    // 1. 按 paramTypeCode 排序（稳定槽位）
    // 2. 每参数渲染规范 token（启用但未选 → 显式空哨兵，不省略）：
    //    MATERIAL → "MAT=" + code            （未选 → "MAT=∅"）
    //    ELEMENT  → "ELE=" + 元素码升序 join "码:normalize(含量)"（去尾零，stripTrailingZeros().toPlainString()；未选 → "ELE=∅"）
    //    PROCESS  → "PRC=" + 工序码升序 join ","（排序集合，不区分顺序；未选 → "PRC=∅"）
    // 3. 串首带结构版本 + 客户号：text = STRUCTURE_VERSION + "|CUST=" + customerNo + "|" + tokens.join("|")
    // 4. hash = sha256(text)
    return new Signature(sha256(text), text);
}

public Signature computeComposite(String customerNo, List<String> childQuotePartNos) {

    //  （组合工艺若纳入指纹：追加 "|CPROC=" + defCode 排序集合；见 §2.7 待定）
}
```

设计取舍（对齐需求 spec §5.2）：
- **客户号进串**（`CUST=`）→ 天然实现「客户维度」；同物质不同客户指纹不同，各自发号。这是与生产侧 `FingerprintCalculator`（无客户维度）的本质差异。
- **不哈希模板 ID**：启用参数集不同 → 槽位不同 → 指纹天然不同；模板只改可选值范围（未动启用项）→ 规范串结构不变 → 相同选值仍命中复用。
- **元素含量 `normalize` 复用生产侧口径**（`stripTrailingZeros().toPlainString()`，`FingerprintCalculator.java:97-100`），微调值计入。
- **工序排序集合**（不区分顺序），与需求 spec §5.2 token 规则一致。

> **备选（列 2 个）**：
> (A) 直接扩展 `FingerprintCalculator` 加 `customerNo` 参数 —— 否决：会污染生产侧全局指纹语义、两套去重耦合，需求 spec §5.1 明确「两码事、不改它」。
> (B) 指纹里哈希 `template_id + version` —— 否决：模板每发一版就把同一实体分裂成多个报价料号，去重失效（需求 spec §5.2 已论证）。

### 2.3 `resolvePart` 精确改造点（SIMPLE custom 分支）

现役 `resolvePart(PartRequest pr, UUID operatorId, UUID customerId, String customerCode, List<String> reused)`（`:180`）。为携带「启用参数投影 + structureVersion + yyMm」，**签名扩展**：追加一个 `SalesConfigContext salesCtx`（由 `configure` 从请求 + `SelTemplateService.getByIndustry` 组装，见 §2.6）。custom 分支（`:223-274`）逐行改法：

| 现役行 | 现役代码 | 目标改法 |
|---|---|---|
| `:235` | `String fp = fingerprintCalc.simpleFingerprint(pr.recipeCode, elems);` | **保留但降级为「兜底/审计」**（可选，用于 P2 端点兼容，不再驱动复用判定）。复用判定改用销售指纹。 |
| `:240` | `String existing = lookupHfByFingerprint(fp);` | **删除此行的复用作用**。新增：`var sig = salesFp.computeSimple(customerCode, salesCtx.enabledParams()); String hit = sigRepo.lookup(customerCode, STRUCTURE_VERSION, sig.hash());` |
| `:242-244` | `if (existing != null){ reused.add(existing); hfPartNo = existing; }` | `if (hit != null){ reused.add(hit); hfPartNo = hit; /* 跳过发号+落库，直接 return */ return hfPartNo; }` |
| `:245-257` | `else { hfPartNo = partNoProvider.apply(PartNoContext(recipe.symbol,"SIMPLE",operatorId)); if(processIds...) insertProcessSimpleUnitPriceV6(...); }` | `else { hfPartNo = quoteAllocator.mintAndRegister(customerCode, salesCtx.yyMm()); sigRepo.insert(customerNo, STRUCTURE_VERSION, sig, hfPartNo); }`（发号 swap；工序落库挪进 3c persist 分发，见 §3） |
| `:262` | `insertMaterialMasterV6(hfPartNo, recipe.symbol, pr.unitWeightGrams, recipe.id, fp);` | `insertMaterialMasterV6(hfPartNo, recipe.symbol, pr.unitWeightGrams, recipe.id, null);` —— **config_fingerprint 传 NULL**（选配报价料号不进生产侧全局去重，避免跨客户同物质撞 `uq_material_master_fingerprint`）。方法签名不变，实参改 null。 |
| `:263/268` | `insertElementBomV6 / insertMaterialBomItemV6` | 由 3c persist_handler 分发调用（行为不变，补 sales_part_no 维度，见 §3.2）。 |

> **顺序纪律**：命中 `hit` 时**在 `:262` 之前 return**（跳过所有落库）。现役命中 existing 仍会走到 `:262` 补 V6（因为跨客户复用需补当前客户数据）——本方案命中是**同客户同结构**复用，数据必已存在，直接 return，不重复落库。

**custom 分支必须处理的边界**：`customerCode` 为空（quotation 无 customer）→ 无法发报价料号（`mintAndRegister` 需 customerNo）。现役 `:251-254` 已对「processIds 但无 customerCode」抛错；本方案把校验前移：custom 选配**强制要求 customerCode 非空**（报价料号号内嵌客户码，无客户码不能发号），否则 `recordError`/抛 `IllegalArgumentException`。

### 2.5 发号 swap 细节

- 注入 `@Inject QuoteMaterialNoAllocator quoteAllocator;`，替换 `partNoProvider` 在 `:247-248/:796-797` 的调用。`PartNoProvider`/`PartNoContext` **保留**（其它模块/测试仍引用；仅选配链路停用）。
- `yyMm` 取值时点：需求 spec §9 列为待定（报价创建日 vs 发号 now()）。**本方案定：发号 now()**（与 `MaterialBomMergeHandler.java:59` `YearMonth.now()` 现役口径一致，避免跨表两套 YYMM）。由 `salesCtx.yyMm()` 承载，`configure` 入口统一 `java.time.YearMonth.now().format("yyMM")`。
- `mintAndRegister` 自带 `material_customer_map` QUOTE 行登记（`:43-46`），选配无需另调 `ensureRegistered`。

### 2.6 上下文装配：`SalesConfigContext`（由 3a 运行时喂入）

`configure(quotationId, req, operatorId)`（`:771`）入口新增：
```
industryCode = SELECT industry_code FROM customer WHERE id = customerId   // Plan1 列
template = selTemplateService.getByIndustry(industryCode) ?? getByIndustry("__DEFAULT__")   // 回退，不报错

### 2.7 唯一约束 / 并发

- `sel_part_signature` `uq_sel_part_signature` + **`INSERT ... ON CONFLICT (customer_no, structure_version, config_fingerprint) DO NOTHING`**；`sigRepo.insert` 返回受影响行数，0 → 竞态败者，**回读既有 `quote_part_no` 复用**（对齐需求 spec §5.5，与 `QuoteMaterialNoAllocator` 的竞态败者回读同构）。
- `mintAndRegister` 自身并发安全（`quote_material_no_seq` 的 `ON CONFLICT DO UPDATE RETURNING`，`:37-40`）。
- **两阶段并发窗口**：同客户并发两次相同选配，可能都 miss → 都 mint（各得一个报价料号）→ 一个 `sigRepo.insert` 成功、另一个 ON CONFLICT DO NOTHING 败者回读到**先赢者的** quote_part_no，于是败者 mint 出的号成为**孤儿**（登记在 map 但无 sel_part_signature 指向、无 lineItem 引用）。可接受（号空洞，与四位码序列空洞同级），需求 spec §5.5 未要求回收。**实现须在 sigRepo.insert 败者分支放弃自己 mint 的号、改用回读号建 lineItem**。
- `sel_part_signature` 的写入与 `mintAndRegister`、各落库同处 `configure` 的 `@Transactional`（`:771`）—— 单事务内，指纹登记与料号/落库原子成败。

### 2.8 `isCfg` 处理（承接 Spec2）

- 现役 `MaterialBomMergeHandler.isCfg:262-264` = `startsWith("CFG-")`，`:72/:106` 用于**拒绝把系统生成的选配料号当宏丰料号导入 BOM**。
- 选配号从 `CFG-` 改 `XXXX-YYMMNNNNNN` 后，`isCfg` 对新号**不再命中** → 新选配号可被 BOM 导入当普通料号接受。这是**期望行为**：报价料号统一后，选配产出号本就是合法报价料号，`MaterialNoResolver.resolve`（Spec1 §4）「料号有值→返回」会把它当文件值正常处理并 `ensureRegistered`。
- **本方案对 `isCfg` 的处置**：**保持 `isCfg` 原样（继续拦 CFG- 历史号）**，**不**把 `XXXX-YYMMNNNNNN` 加进拒绝集。理由：报价料号是一等可导入料号；把选配号加进拒绝集会误伤 Spec1 的正常报价料号导入。
- **须核实的连带点**：`codegraph_callers isCfg` 确认仅 `MaterialBomMergeHandler` 2 处；若 Spec2 另有「按 CFG- 前缀路由选配料号」的逻辑（如候选查询、BOM 合并特判），前缀改变后须同步 —— 本方案范围内 grep 结论是无其它 isCfg 引用，低风险。**这是 Spec2 BL-0017 与本方案的交接项**（§6）。

## 3. Plan 3c 设计：按 sales_part_no 落库 handler

### 3.1 persist_handler 分发

`sel_param_type.persist_handler_key` 三值（`V313:14-16`）：`MATERIAL_RECIPE_BIND` / `ELEMENT_OVERRIDE` / `PROCESS_LIST`。设计一个 **SPI 分发接口**，把现役 `ConfigureProductService` 的私有落库方法收编为可注册 handler：

```java
public interface SelPersistHandler {
    String key();   // MATERIAL_RECIPE_BIND / ELEMENT_OVERRIDE / PROCESS_LIST
    void persist(SelPersistContext ctx);   // ctx: quotePartNo, customerCode, salesPartNo(=quotePartNo), 选值, recipe, lineItemId...
}
```

分发点：`resolvePart` custom 分支「未命中→已 mint」之后，遍历 `salesCtx.enabledParams()`，按各参数 `persist_handler_key` 取 handler 执行（`sel_param_type` 查一次缓存成 Map）。落库 handler 到现役方法映射：

| persist_handler_key | 现役落库方法（复用，补 sales_part_no） | material_no / sales_part_no |
|---|---|---|
| `MATERIAL_RECIPE_BIND` | `insertMaterialMasterV6(:390)`（material_recipe_id 绑定 + config_fingerprint=NULL）+ `insertMaterialBomItemV6(:442)`（材质自指物料行，供 `composite_child_materials_mirror` 渲染） | `material_no = quotePartNo`；`material_bom_item.sales_part_no = quotePartNo` |
| `ELEMENT_OVERRIDE` | `insertElementBomV6(:410)`（元素配比 → `element_bom_item`，characteristic='2000'） | `material_no = quotePartNo`；`element_bom_item.sales_part_no = quotePartNo` |
| `PROCESS_LIST` | `insertProcessSimpleUnitPriceV6(:647)`（工序 → `unit_price` 自制加工费，`VersionedV6Writer`）+ per-quote `insertQuotationLineProcesses(:737)` | `code=finished_material_no=quotePartNo`；`unit_price.sales_part_no = quotePartNo` |

> 分发只是**结构化包装**——现役已在 custom 分支顺序调这三组方法（`:262/263/268` + `:255`）。3c 的价值 = ①按模板 enabled 参数动态决定调哪些（模板没启用工序就不落工序）；②统一补 `sales_part_no` 维度；③COMPOSITE 复用同 handler。**若嫌 SPI 过重，退化方案**：不引 SPI，直接在 custom 分支按 `enabledParams` 的 paramTypeCode `switch` 调对应方法（列为 3c 备选 A，见 §3.5）。

### 3.2 补 `sales_part_no` 维度（对齐核价销售料号维度落库）

`V311` 已给 `element_bom_item / material_bom_item / unit_price / capacity / material_bom / labor_rate` 加 `sales_part_no` 列且 uq 末尾含 `COALESCE(sales_part_no,'')`。3c 落库须**填该列**，否则与核价侧视图（按 `(material_no, sales_part_no)` 分组）口径不一致（AP-53 家族「共 N 项」）。逐方法改点：

| 方法 | 现役 INSERT/groupKey | 补 sales_part_no |
|---|---|---|
| `insertElementBomV6:414-424` | `INSERT INTO element_bom_item (system_type, customer_no, hf_part_no, material_no, characteristic, seq_no, component_no, content)` | 加列 `sales_part_no`，值 = partNo（= quotePartNo） |
| `insertMaterialBomItemV6:444-453` | `INSERT INTO material_bom_item (id, system_type, customer_no, material_no, characteristic, seq_no, component_no, component_usage_type, ...)` | 加列 `sales_part_no = :p` |
| `insertProcessSimpleUnitPriceV6:677-686` | `VersionedGroupSpec("unit_price","version_no", gk{system_type,price_type,cost_type,customer_no,code,finished_material_no}, ...)` | groupKey `put("sales_part_no", hfPartNo)`（进版本组，与核价 §3.1 一致；`unit_price` 已在 `VersionedV6Writer` 白名单，写进主行） |
| COMPOSITE：`writeCombomaterialBomV6:515` / `insertProcessUnitPriceV6:595` / `insertCompositeProcessCapacityV6:696` | 各 `bomGroupKey/gk` | 分别 `put("sales_part_no", childPn / parentHfPartNo)`（子件行=子件报价料号，组合工艺行=父报价料号） |

> **选配语境下 material_no ≡ sales_part_no ≡ quotePartNo**（见 §3.4），故填列是「同值冗余」，但为让核价侧新视图（按 sales 分组）统一命中、且未来生产料号后补时不破裂，**必须显式填**，不能留 NULL 靠 COALESCE 兜底（NULL 会与核价费用行分组语义混淆）。

### 3.3 material_master 建行防断链（AP-53 硬约束）

- 现役 `insertMaterialMasterV6:390` 已 `INSERT INTO material_master (material_no, material_type, unit_weight, material_recipe_id, config_fingerprint) ... ON CONFLICT DO NOTHING`。3c 仅改 `config_fingerprint` 实参为 **NULL**（§2.3）。
- **为何必须建**：产品卡片四 Tab（材质/元素/工序/…）的 mirror 视图（`composite_child_materials_mirror` / `composite_child_elements_mirror` 等）**JOIN `material_master.material_no`** 取数。无此行 → JOIN 空 → 四 Tab 空（AP-53）。需求 spec §4.4/§9 已点名此为「唯一会阻塞渲染的硬约束」。
- 复用命中（§2.3 return）时不重复 INSERT——首次已建，`ON CONFLICT DO NOTHING` 幂等亦安全。

### 3.4 生产料号(material_master) ↔ 报价料号 1:N 落库时的主键归属

需求 spec §4.4 与核价 spec 反复强调「生产料号↔报价料号 = 1:N」「material_master 按生产料号唯一、不加 sales 列」。**但选配语境是特例，必须厘清，否则落库主键归属含糊**：

- **核价导入语境**：`material_master.material_no` = **真生产料号**（ERP 号 / 宏丰料号），一个生产料号可挂 N 个报价料号 → `sales_part_no` 作**旁列**加在 `element_bom_item` 等按料号表（V311），material_master 不加列。1:N 成立。
- **选配语境（本方案）**：选配「从零配」时**尚无 ERP 生产料号**（`production_no` 后补）。选配为报价料号**新建一行 `material_master`，其 `material_no` = 报价料号 quotePartNo**（现役 `insertMaterialMasterV6` 已如此，只是料号今为 CFG-）。此刻 material_master 这一行的主键就是报价料号，`config_fingerprint=NULL`，`production_no` 关系走 `material_customer_map` QUOTE 行的 `production_no`（后补，Spec1 §1.1）。
- **结论（落库主键归属，写进本方案硬口径）**：
  1. 选配 material_master 行 **PK/唯一键 = 报价料号**（`material_no=quotePartNo`），**不复用**任何既有生产料号行（避免跨客户/跨报价料号污染 config_fingerprint 全局唯一）。
  2. 各按料号 V6 QUOTE 行（element/material/unit_price）`material_no = sales_part_no = quotePartNo`，**自洽 1:1**（选配阶段生产=报价同号）。
  3. 未来同一实体接入 ERP 生产料号时，1:N 由 `material_customer_map.production_no`（多个 QUOTE 报价料号行填同一 `production_no`）承载，**不回改** material_master 主键。
  4. 因此选配落库**不触碰**核价侧「生产料号 material_master 行」——两者是不同的 material_master 行（生产料号行 vs 报价料号行），`sales_part_no` 列在选配行里 = 自身料号，在核价费用行里 = 旁挂的成品报价料号。语义不冲突。

### 3.5 3c 备选

- **备选 A（轻量，推荐先行）**：不引 `SelPersistHandler` SPI，custom 分支按 `enabledParams` 的 paramTypeCode `switch` 直调现役三方法 + 补 sales_part_no。改动最小、E2E 面最小。
- **备选 B（SPI 化，§3.1）**：为「一期固定 3 类、未来开发才加第 4 类」预留扩展点。需求 spec §2「固定三类、加第四类需开发改代码」——SPI 收益有限，**建议 3c 先做备选 A，SPI 化列入 Backlog**。

## 6. 与「报价料号统一 Spec2 (BL-0017)」的边界
