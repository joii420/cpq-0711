## 一、方案定位

### 1.1 业务背景

CPQ 系统当前的报价流程：**销售选 partNo → 选模板 → 填字段 → 生成报价单**。这个流程假设 **partNo 已存在**（mat_part 表里有）。

但对于**组合产品**（CFG-COMBO-XXX）和**可配置产品**（如电机 + 不同绕组 + 不同壳体），客户实际的购买决策是"**先配置后定价**"：

> 客户："我要一台电机，绕组用 Ag85Cu15，壳体加防腐镀层，连接器选 M12 螺纹..."
> 销售：每次选一项 → 系统实时显示 3D 模型 + 价格变化

这就是 **3D 产品选配器**的业务价值。

### 1.2 参考产品类比

| 产品 | 类似形态 |
|---|---|
| 特斯拉 / 蔚来 / 小鹏 "配置你的车" | 颜色 / 轮毂 / 内饰 / 套件，实时 3D 渲染 |
| 苹果 / 戴尔 电脑配置购买页 | CPU / 内存 / 硬盘 / 屏幕，实时价格 |
| 宜家家居配置 | 沙发布料 / 颜色 / 配件 |
| 工业品在线配置器（机柜 / 机床 / 电梯）| 行业典型，CPQ 主要受众 |

### 1.3 v0.4 单一主线（2026-05-26 整合）

v0.4 是**唯一主线**：独立全屏配置器，从 3D 选配开始生成报价单。**没有任何 mode 分支硬编码** — 一切都由配置驱动：
- 配置选项 → 配置 3D 渲染规则 → 配置价格规则 → 自动生成 line_item
- 不同产品的差异（颜色 vs 几何形态完全不同）**只是配置数据形态不同**，**不是代码分支**

## 三、核心概念模型

### 3.1 五大核心实体

```
ProductConfigTemplate (配置模板)
  ├── 关联 mat_part (基础产品)
  ├── 关联 mat_part_model (基础 3D 模型)
  └── ConfigOption[] (选项树)
       ├── ConfigOptionValue[] (每个选项的可选值)
       │    └── ConfigOption3DRule (该值对 3D 的渲染影响)
       │    └── ConfigOptionPriceRule (该值对价格的影响)
       └── ConfigConstraint[] (跨选项的约束/依赖)

ConfigInstance (用户的当前选配)
  ├── 选了哪些 OptionValue
  ├── 当前总价
  ├── config_fingerprint (生成 SHA256 用于幂等 + 去重)
  └── 生成的 line_item (报价单)
```

### 3.2 配置选项的层级（Option Tree）

典型 4-5 层（按业务复杂度可调）：

```
配置模板：接触片产品 (ContactStrip)

├── 选项 1: 型号 (REQUIRED, exclusive)
│   ├── 标准款 (¥1,000)
│   ├── 增强款 (¥1,500)
│   └── 高端款 (¥2,200)
│
├── 选项 2: 材质 (REQUIRED, exclusive, 依赖型号)
│   ├── Ag85Cu15 (+ ¥580)
│   ├── Cu90Ag10 (+ ¥200)
│   └── 纯 Cu     (+ ¥0,标配)
│
├── 选项 3: 表面处理 (OPTIONAL, exclusive)
│   ├── 无处理     (+ ¥0)
│   ├── 镍镀层 0.5μ (+ ¥140)
│   ├── 银镀层 1μ   (+ ¥320)
│   └── 钝化处理    (+ ¥80)
│
├── 选项 4: 接口规格 (REQUIRED, exclusive)
│   ├── M6 螺纹
│   ├── M8 螺纹 (默认)
│   ├── M10 螺纹
│   └── 焊接连接
│
└── 选项 5: 附加部件 (OPTIONAL, multi-select)
    ├── 加固卡扣 (+ ¥80)
    ├── 防尘罩   (+ ¥45)
    └── 接地线   (+ ¥30)
```

### 3.3 选项类型枚举

| 类型 | 说明 | UI 控件 |
|---|---|---|
| `EXCLUSIVE` | 互斥单选（型号、材质）| 卡片单选 / 单选下拉 |
| `MULTI_SELECT` | 多选（附加部件）| 多选 checkbox 卡片 |
| `NUMERIC` | 数值（长度、数量）| 数字输入 + 滑块 |
| `TEXT` | 文本（刻字、备注）| 文本输入 |
| `COLOR` | 颜色（外观）| 色卡 |

### 3.4 配置约束（Constraint）

跨选项的依赖关系：

| 约束类型 | 含义 | 示例 |
|---|---|---|
| `REQUIRES` | 选 A 必须先选 B | "银镀层 1μ" 要求 "材质 = Ag85Cu15" |
| `EXCLUDES` | 选 A 不能选 B | "高端款型号" 不能选 "无处理" 表面 |
| `IMPLIES` | 选 A 自动选 B | "焊接连接" 自动选"接地线"附件 |
| `HIDES` | 选 A 时隐藏选项 B | "标准款"型号 时隐藏"附加部件"全组 |
| `NUMERIC_RANGE` | 数值范围约束 | "长度" 与 "型号" 关联：标准款 50-100mm |

### 3.5 选项 → 3D 渲染规则（ConfigOption3DRule）

每个 OptionValue 控制 3D 模型的视觉表现：

```json
{
  "option_value": "Ag85Cu15",
  "rules": [
    {
      "action": "REPLACE_MATERIAL",
      "target_mesh": "mesh_main_body",
      "material": {
        "diffuse": "#E8E8EC",
        "metallic": 0.9,
        "roughness": 0.15
      }
    },
    {
      "action": "SHOW_MESH",
      "target_mesh": "mesh_silver_badge"
    }
  ]
}
```

3D 规则的 5 种 Action：

| Action | 说明 | 参数 |
|---|---|---|
| `SHOW_MESH` | 显示某个 mesh | target_mesh |
| `HIDE_MESH` | 隐藏某个 mesh | target_mesh |
| `REPLACE_MATERIAL` | 替换材质（颜色 / PBR 参数）| target_mesh + material |
| `SWAP_MESH` | 用 A mesh 替换 B mesh（如 M6 螺纹 → M8 螺纹）| from_mesh + to_mesh_url |
| `TRANSFORM_MESH` | 平移/旋转/缩放（如不同长度螺栓）| target_mesh + transform |

### 3.7 全实体状态枚举与术语表（v0.4 P0 修复，2026-05-26）

> 跨原型 / 跨章节状态命名必须严格按本表，不允许 PUBLISHED 与 ACTIVE 混用、DRAFT 与 EDITING 混用。

| 实体 | 草稿 | 中间态 | 已上线 / 可用 | 不可用 / 归档 | 删除 |
|---|---|---|---|---|---|
| `product_config_template` 选配模板 | `DRAFT` | `EDITING`（v1.3-rc1 之类） | `PUBLISHED` | `ARCHIVED` | 软删（保留引用） |
| `product_config_instance` 选配实例 | `DRAFT` | `SUBMITTED` | `LINKED`（绑了报价单）| `EXPIRED`（30 天未提交）| 硬删 |
| `cpq_feature_group` 特征群组 | `DRAFT` | — | `ACTIVE` | `ARCHIVED` | 软删 |
| `cpq_feature_field` 特征字段 | — | — | `is_active=true` | `is_active=false` | 软删 |
| `cpq_feature_value` 特征值 | — | — | `is_active=true` | `is_active=false` | 软删 |
| `product_config_share` 分享链接 | — | — | `ACTIVE` | `EXPIRED` / `REVOKED` | 硬删 |
| `product_config_approval` 审批单 | — | `PENDING` | `APPROVED` | `REJECTED` / `WITHDRAWN` | — |
| `customer_lead` 客户线索 | — | `PENDING_REVIEW` | `CONVERTED`（已转 customer）| `REJECTED` | 软删 |

**为什么模板叫 PUBLISHED 而特征群组叫 ACTIVE**：
- 模板有"发布行为"语义（PM 主动审核后发布），状态变化由审批驱动
- 特征群组是主数据字典性质，由"是否启用"驱动 — 没有"发布"动作
- 此差异是有意保留的语义区分，不要统一为同一词

**状态机转换示意**（选配实例为例）：
```
            ┌──────────┐
   开始选配 │  DRAFT   │ ── 30 天未操作 ──> EXPIRED
   ───────> └────┬─────┘
                 │ 提交
                 ▼
            ┌──────────┐ ── 关联到报价单 ──> ┌────────┐
            │SUBMITTED │                    │ LINKED │
            └────┬─────┘ ←── 解绑报价单 ──── └────────┘
                 │
                 │ 销售放弃 / 客户失联
                 ▼
              (硬删)
```

## 四、3D 渲染模型设计

### 4.1 基础场景 + 动态变体

3D 模型分两部分：

**A. 基础模型（base.glb）**：所有可能配置的 mesh **全部预加载**到场景里。每个 mesh 默认 visible 状态由"默认配置"决定。

**B. 配置规则触发的运行时变换**：
- 选项变化 → 调用 ConfigOption3DRule → 执行 SHOW_MESH / HIDE_MESH / REPLACE_MATERIAL / SWAP_MESH / TRANSFORM_MESH
- Babylon 即时 re-render

```
┌────────────────────────────────────────────────────────┐
│ base.glb (预加载所有 mesh)                              │
│  ├── mesh_main_body          (始终显示)                 │
│  ├── mesh_silver_badge       (材质=Ag 时显示)           │
│  ├── mesh_thread_m6          (接口=M6 时显示)           │
│  ├── mesh_thread_m8          (接口=M8 时显示，默认)     │
│  ├── mesh_thread_m10         (接口=M10 时显示)          │
│  ├── mesh_clip               (附加=加固卡扣 时显示)     │
│  ├── mesh_dust_cover         (附加=防尘罩 时显示)       │
│  └── mesh_ground_wire        (附加=接地线 时显示)       │
└────────────────────────────────────────────────────────┘
                ↓
            (选项变化)
                ↓
┌────────────────────────────────────────────────────────┐
│ 运行时变换:                                              │
│  - HIDE_MESH(mesh_thread_m6)                            │
│  - SHOW_MESH(mesh_thread_m8)                            │
│  - REPLACE_MATERIAL(mesh_main_body, Ag85Cu15_PBR)       │
│  - SHOW_MESH(mesh_clip)                                 │
└────────────────────────────────────────────────────────┘
```

### 4.2 性能优化策略

| 策略 | 实现 | 何时用 |
|---|---|---|
| **预加载所有 mesh** | base.glb 含全部部件 | 部件数 < 50 时 |
| **按需加载** | SWAP_MESH 时从 CDN 拉 sub-glb | 部件数 > 50 / 大型组件 |
| **LOD（多层级细节）** | 远距离用低面数 mesh | 复杂场景 |
| **材质共享** | StandardMaterial 复用 + 仅改 diffuse | 材质数 < 20 |
| **离屏渲染** | requestIdleCallback 错峰渲染 | 选项快速变化 |
| **去抖动** | 300ms debounce 选项变化 | 滑块连续拖动 |

### 4.3 选项值的 3D 配置（v0.4 收敛 — 选项树即特征树）

> **2026-05-26 收敛**：v0.2 mat_feature 系列 4 张表已废弃，**选项值表 (`product_config_option_value`) 直接吸收特征语义** —— feature_type / attributes / tags 字段下沉。详见 §七.4。

```
ConfigOptionValue: "材质 = Ag85Cu15"
  ├── feature_type: 'MATERIAL'                        ← 替代 mat_feature.feature_type
  ├── attributes: {"composition":{"Ag":85,"Cu":15}}   ← 替代 mat_feature.attributes
  ├── tags: ['high-conductivity']                      ← 替代 mat_feature.tags
  ├── 3D 规则 (rules 数组):
  │    - REPLACE_MATERIAL(mesh_main_body, "#E8E8EC")
  │    - SHOW_MESH(mesh_silver_badge)
  ├── 价格规则:
  │    - DELTA_FIXED +580
  └── 业务实体引用 (product_config_value_reference):
       - PROCESS / MRO-AS-0010 (镀银工序)
       - MATERIAL / MAT-AGCU-85-15 (原料供应)
```

### 4.4 3D 渲染规则统一模型（配置驱动，零硬代码）

> **核心宗旨**：所有 3D 联动行为**完全由数据配置驱动**，没有"模式 A / 模式 B"硬代码分支。后端 evaluate API 不区分类型，统一返回 `render3DCommands` 数组；前端渲染引擎按统一规则处理。

#### 4.4.1 配置由两个维度组成（任意组合）

| 配置维度 | 数据来源 | 覆盖场景 |
|---|---|---|
| **维度 1: base 模型内 mesh 操作** | `product_config_3d_rule` 表 | 颜色 / 显隐 / 变换（同形态变体）|
| **维度 2: 关联独立子模型** | `product_config_option_value.sub_model_part_no` + `attach_mode` | 几何完全不同（独立 GLB 文件）|

**两个维度可任意组合**：
- 仅维度 1：颜色变化（如材质选 Ag85Cu15）
- 仅维度 2：型号完全不同（如电机标准/防爆）
- **同时用**：材质换色 + 附加散热器子模型（如：维度 1 改主体颜色 + 维度 2 OVERLAY 加散热器）

#### 4.4.2 维度 1 — base mesh 操作（5 种 Action）

`product_config_3d_rule.action` 枚举值：

| Action | 用途 |
|---|---|
| `SHOW_MESH` | 显示某 mesh |
| `HIDE_MESH` | 隐藏某 mesh |
| `REPLACE_MATERIAL` | 改 mesh 材质 / 颜色 |
| `SWAP_MESH` | 同位置 mesh 互换（M6 ↔ M8 ↔ M10）|
| `TRANSFORM_MESH` | 位置 / 旋转 / 缩放变化 |

#### 4.4.3 维度 2 — 关联独立子模型（3 种挂载模式）

`product_config_option_value.attach_mode` 枚举值：

| attach_mode | 含义 | 何时配置 |
|---|---|---|
| `OVERLAY` | 子模型加载后叠加到 base 上 | 选项值含**独立的附加部件**（如散热器 / 防尘罩）|
| `SWAP` | 隐藏 base 上某 mesh，加载子模型填入对应位置 | 选项值含**同位置不同规格**的部件 |
| `REPLACE_BASE` | 完全替换整个 base 模型 | 选项值的**整体几何完全不同** |

#### 4.4.4 evaluate API 的统一处理（不区分维度）

```
用户选项变化 → POST /configurator/instances/{id}/evaluate
                ↓
后端 evaluate 服务（不区分维度，统一遍历配置）:
  for opt in selectedValues:
    val = option_value 表
    # 维度 1 规则
    rules = product_config_3d_rule.findByValueId(val.id)
    render3DCommands.addAll(rules)
    # 维度 2 配置（若有 sub_model_part_no）
    if val.sub_model_part_no:
      render3DCommands.add({
        action: 'LOAD_SUB_MODEL',
        modelUrl: lookupModelUrl(val.sub_model_part_no),
        attachMode: val.attach_mode,
        attachPosition: val.attach_position
      })
                ↓
前端渲染引擎（不区分维度，按 action 类型分发）:
  for cmd in render3DCommands:
    switch cmd.action:
      SHOW_MESH / HIDE_MESH / REPLACE_MATERIAL / SWAP_MESH / TRANSFORM_MESH → 同步执行
      LOAD_SUB_MODEL → 异步加载（LRU 缓存 / 并行 / AbortController）
```

**关键不变量**：
- ✅ 不论配置多少维度 1 规则 + 多少维度 2 子模型，**都走同一套数据 + API + 渲染引擎**
- ✅ 加新维度（如未来加"动画播放""特效"）只需在 evaluate 输出新 action，不改前端引擎结构
- ❌ 不存在"if 是策略 A 则 ...；else if 是策略 B 则 ..."的硬代码分支

#### 4.4.5 子模型加载性能策略

- **预加载**：进入选配页时按"高频选中"统计预加载常用子模型（管理员配置）
- **LRU 缓存**：最近 5 个子模型保留在内存
- **并行加载**：多选项时并发拉取子 GLB
- **取消加载**：用户快速切换选项时取消进行中的子模型加载（AbortController）
- **加载状态**：UI 显示骨架屏 + 圆形进度（**禁止"加载中..."文字**，AP-31 规范）

#### 4.4.6 配置选择决策（业务管理员使用）

管理员在 `v0.4-3D选配模板管理端原型.html` 配置某个 OptionValue 的 3D 行为时：

| 业务场景 | 配置哪些字段 |

## 五、配置流程设计

### 5.1 引导式 vs 自由式

| 模式 | 适用场景 | UI 表现 |
|---|---|---|
| **引导式（Stepper）** | 复杂产品 / 客户自助 / 不熟悉的销售 | 顶部步骤指示器 + 强制顺序 + 完成度进度 |
| **自由式（Tab/Accordion）** | 简单产品 / 经验丰富销售 / 频繁切换 | 右侧 Tab 任意切换 + 可跳过非必选 |

**v0.3 默认走自由式 + 进度提示**：右侧 Accordion 各选项可任意展开 / 折叠，顶部一个进度条显示必选项完成度。

### 5.2 实时联动（核心）

每次用户改选项：

```
1. UI 即时反馈（按钮选中态）
2. POST /api/cpq/configurator/{instanceId}/evaluate
   Body: { "选项 X": "新值" }
3. 后端:
   a) 应用约束（REQUIRES / EXCLUDES / IMPLIES）
      若有冲突，返回冲突列表
   b) 计算价格（累加 ConfigOptionPriceRule.delta）
   c) 收集 3D 渲染指令（ConfigOption3DRule）
   d) 计算 config_fingerprint
4. 前端:
   a) 应用 3D 渲染指令到 Babylon 场景
   b) 更新底部价格
   c) 如有冲突提示，弹 modal "选 A 与已选 B 冲突，是否：①切换到 A 自动取消 B / ②保持 B 取消选 A / ③取消"
```

### 5.3 冲突解决 UX

3 种冲突处理方式：

| 类型 | 用户操作 | 系统行为 |
|---|---|---|
| **自动消解** | 选互斥单选时自动切换 | 透明，仅 toast 提示 |
| **弹 Modal 选择** | 跨选项 EXCLUDES 冲突 | 提供 3 个解决路径 |
| **硬阻止** | 配置模板配错 / 数据完整性 | 错误提示 + 不让选 |

### 5.4 完成度跟踪

- **必选项完成数 / 总必选数** = 进度百分比
- 100% 时"生成报价单"按钮变高亮可点
- 中途用户可保存草稿（ConfigInstance.status = 'DRAFT'），下次从 URL `/configurator/{instanceId}` 恢复

### 5.5 提交流程（v0.4 P0 修复：两步式 — 先建实例，再决定后续）

**修改理由（2026-05-26 P0 修复）**：原"一键提交生成报价单"违反 §17 弱关联设计 — 选配实例应能独立存在（销售内部库 / 客户尚未确定 / 客户自助提交待销售对接）。

**新流程（两步式）**：

```

│    - status = 'SUBMITTED'                                             │
│    - template_id + template_version snapshot                          │
│    - selected_values JSONB                                            │

│ 4. 返回 { instance_id, instance_code, fingerprint, computed_price }   │
│   ↓                                                                  │

└──────────────────────────────────────────────────────────────────────┘

┌─ 步骤 2: 用户选后续动作（三选一） ───────────────────────────────────┐

│    a) 生成或复用 mat_part (config_fingerprint 唯一索引保证幂等)         │
│    b) 生成 quotation + quotation_line_item                              │
│    c) UPDATE instance SET status='LINKED', linked_quotation_id=新ID,    │
│           linked_at=NOW(), linked_by=user_id,                           │

│    d) 返回 { quotation_id, redirect_url: '/quotations/{id}/edit' }      │
│    前端跳转报价单编辑页                                                  │
│                                                                       │
│ ② 💾 保存为草稿（稍后处理）                                              │
│    无后端调用（instance 已是 SUBMITTED 状态）                            │

│    弹"选择报价单"列表（同客户的 DRAFT 报价单）                            │

│    b) 校验 quotation.status IN ('DRAFT','EDITING')                      │
│    c) 生成 line_item INSERT 到 existing quotation                       │
│    d) UPDATE instance SET status='LINKED', linked_quotation_id=...      │

**幂等性保证**：
- `config_fingerprint UNIQUE` 索引（按 customer + fingerprint 唯一）— 同一客户重复提交同一配置返回同一 instance

```

## 六、UG NX → STEP → GLB 转换工作链路（v0.4 主线）

> **2026-05-25 调整**：业务实际使用的是西门子 UG NX，原 CAD 通用工作流改为 **UG NX 主线** + 双文件保存（`.prt` 源 + `.stp` 中性）+ 服务端 STEP 转 GLB。

### 6.1 完整三段链路

```
段 1: UG NX 工程师设计
    ↓ 工程师在 UG NX 中保存为 .prt（西门子专属，含完整设计树/参数化历史）
.prt 源文件
    ↓ UG 导出 → 工程师在 UG NX 内 [Export → STEP] 选 AP203 / AP214 / AP242
.stp / .step 中性格式（行业通用）
    ↓ 工程师上传 .prt + .stp 双文件到 CPQ
                                              ↓
                                ┌─────────────┴─────────────┐
                                ▼                           ▼
                       ┌─────────────────┐         ┌──────────────────┐
                       │ 桶 1: cpq-ugnx- │         │ 桶 2: cpq-stp-   │
                       │ source          │         │ source           │
                       │ .prt 永久保存    │         │ .stp 永久保存    │
                       └─────────────────┘         └──────────────────┘
                                                            ↓ 后端 Worker
                                                  ┌──────────────────────┐
                                                  │ STEP → GLB 转换流水线 │
                                                  │ FreeCAD CLI (.stp →   │
                                                  │  .stl) → Blender      │
                                                  │  headless (.stl →     │
                                                  │  .glb + Draco 压缩)   │
                                                  └──────────┬───────────┘
                                                             ▼
                                                  ┌──────────────────────┐
                                                  │ 桶 3: cpq-3d-glb     │
                                                  │ .glb 永久保存        │
                                                  └──────────┬───────────┘
                                                             ▼
                                                  浏览器 Babylon 加载 .glb
```

### 6.2 为什么 .prt 和 .stp 都要存

| 文件 | 价值 | 应用场景 |
|---|---|---|
| **`.prt`（UG 源文件）** | 包含完整**设计树 / 参数化历史 / 特征定义** | 工程师返工 / 设计变更 / 标注关联 / 与 UG 同步更新 |
| **`.stp`（STEP 中性格式）** | 跨 CAD 软件**几何 + 拓扑**信息 | 服务器转换 / 其他 CAD 软件打开 / 长期归档（百年标准）|
| **`.glb`（渲染格式）** | **浏览器可加载** + Draco 压缩后 < 5MB | Babylon 实时渲染 / 移动端 |

**关键场景**：设计变更时，工程师改 `.prt` → 重新导出 `.stp` → 上传 → 新 GLB 版本生成 → 报价单页面渲染新模型。**任一文件丢失都阻断回环**。

### 6.3 转换方案对比

| 方案 | 实现 | 优点 | 缺点 | 推荐度 |
|---|---|---|---|---|
| **A. 后端 FreeCAD + Blender CLI** | Java 调子进程 / 流水线脚本 | 开源 / 格式全 / Draco 压缩可控 | 需服务器装两个软件 ~1.1GB / 慢 (20-90s) | ⭐⭐⭐⭐⭐ |
| **B. 后端 OCCT (Open CASCADE)** | Java 直调 C++ 库 / JNI / 容器化 | 比 FreeCAD 快 (5-30s) / 内存可控 | 集成复杂 / 调用栈陡峭 | ⭐⭐⭐⭐ |
| **C. 商业 Autodesk Forge API** | HTTP API 云端转换 | 质量最高 / 无运维 | 收费 / .prt 不支持（仅 STEP）/ 文件出境 | ⭐⭐ |
| **D. 前端 occt-import-js (STEP only)** | 浏览器 WebAssembly | 即时预览 / 无后端 | 大文件慢 / 浏览器内存限 | ⭐⭐⭐ (做预览补充) |

**v0.3 默认 A**（FreeCAD + Blender Docker 镜像）+ D 作前端预览快速反馈。

**重要**：`.prt` 是 UG 专属格式，**只有 UG 本身或商业 PLM 系统（如 Teamcenter）能解析**。后端服务**不直接转 `.prt`**，工程师必须自己在 UG 内导出 `.stp`。

### 6.4 转换流程详解

```
1. 工程师 UG NX 设计 → File → Export → STEP (AP203/AP214/AP242)
   生成: model.prt + model.stp 两份文件
   ↓
2. 管理员在 CPQ 上传向导:
   - Step 1: 上传 .prt
   - Step 2: 上传 .stp
   - Step 3: 系统校验两文件来源一致（MD5 / 时间戳）
   - Step 4: 加元数据 (UG 版本 / 设计师 / 项目)
   ↓
3. 后端 POST /admin/parts/{partNo}/source/upload (multipart)
   ↓
4. 并行保存到 3 个桶:
   - cpq-ugnx-source/{partNo}/v1/model.prt
   - cpq-stp-source/{partNo}/v1/model.stp
   - INSERT mat_part_source_file (含两条记录: PRT 和 STP)
   ↓
5. 发消息到转换队列: { partNo, version, stpFilePath }
   ↓
6. 异步 Worker 拉队列消息
   ↓
7. Worker 容器内:
   a. 下载 .stp 到临时目录
   b. FreeCAD: STEP → STL (高密度 mesh)
   c. Blender headless: STL → GLB + Draco 压缩 + 自动 LOD
   d. 解析 GLB → 提取 mesh 列表 + 顶点数 + 包围盒
   e. gltf-validator 校验
   ↓
8. 上传 GLB 到 cpq-3d-glb:
   - cpq-3d-glb/{partNo}/v1/model.glb
   - cpq-3d-glb/{partNo}/v1/thumbnail.png (Blender 渲染缩略图)
   ↓
9. INSERT mat_part_model (DRAFT 状态)
10. INSERT mat_part_glb_conversion (转换记录)
11. **从 STEP 自动提取几何特征**（用于 mat_feature.geometry_ref）:
    a. FreeCAD 解析 STEP → 识别孔/螺纹/槽等特征
    b. 提取每个 feature 的包围盒 / 位置 / 直径 / 深度
    c. 自动建议（不自动创建）mat_feature 候选列表给管理员审核
    d. 管理员确认后批量 INSERT mat_feature + mat_part_mesh_feature
12. 通知工程师: "CFG-XXX v1 转换完成 (耗时 38s) / 失败"
    ↓
管理员在 /admin/parts/{partNo}/3d 看到结果 + 审核自动识别的特征
```

### 6.5 转换脚本示例（Docker 容器内）

```bash
#!/bin/bash
# cpq-backend/scripts/convert-stp-to-glb.sh
INPUT_STP=$1
OUTPUT_GLB=$2
WORKDIR=$(mktemp -d)

# Step 1: STEP → STL (FreeCAD)
freecadcmd -c "
import FreeCAD, Mesh, Import
Import.open('$INPUT_STP')
doc = FreeCAD.ActiveDocument
Mesh.export(doc.Objects, '$WORKDIR/model.stl', tessellation=1.5)
"

# Step 2: STL → GLB + Draco 压缩 (Blender headless)
blender --background --python /scripts/stl-to-glb.py -- \
        --input "$WORKDIR/model.stl" \
        --output "$OUTPUT_GLB" \
        --draco-compression 7 \
        --decimate 0.4

# Step 3: 生成缩略图
blender --background --python /scripts/render-thumb.py -- \
        --input "$OUTPUT_GLB" \
        --output "${OUTPUT_GLB%.glb}-thumbnail.png" \
        --size 256

# Step 4: 校验
gltf-validator "$OUTPUT_GLB" || exit 1

rm -rf "$WORKDIR"
echo "[OK] Converted $INPUT_STP → $OUTPUT_GLB"
```

### 6.6 STEP 自动特征识别（v0.4 新能力）

FreeCAD 解析 STEP 时可识别**几何特征**（孔 / 槽 / 螺纹 / 倒角等），自动建议 mat_feature 候选：

```python
# /scripts/extract-features.py
import FreeCAD, Part

doc = FreeCAD.open('/tmp/model.stp')
features = []

for obj in doc.Objects:
    if hasattr(obj, 'Shape'):
        shape = obj.Shape
        # 识别孔特征
        for face in shape.Faces:
            if face.Surface.TypeId == 'Part::GeomCylinder':
                radius = face.Surface.Radius
                features.append({
                    'feature_type_code': 'HOLE',   # 或 'THREAD' 按规格判断
                    'attributes': {
                        'diameter_mm': radius * 2,
                        'depth_mm': estimate_depth(face),
                    },
                    'geometry_ref': {
                        'bbox': face.BoundBox.serialize(),
                        'center': face.CenterOfMass,
                    }
                })

# 输出 JSON 给后端，管理员审核后 INSERT mat_feature
print(json.dumps(features))
```

**关键约束**：自动识别**不自动写入** mat_feature 表，必须管理员在 `/admin/parts/{partNo}/3d` 审核确认后批量 INSERT（避免特征字典污染）。

### 6.7 三个对象存储桶设计

| 桶名 | 内容 | 路径规范 | 保留期 | 加密 |
|---|---|---|---|---|
| `cpq-ugnx-source` | UG `.prt` 源文件 | `{partNo}/v{version}/model.prt` | **永久** | SSE-S3/OSS |
| `cpq-stp-source` | STEP `.stp` 中性格式 | `{partNo}/v{version}/model.stp` | **永久** | SSE-S3/OSS |
| `cpq-3d-glb` | GLB 渲染格式 + 缩略图 | `{partNo}/v{version}/model.glb` + `thumbnail.png` | 跟随 GLB 版本归档（旧版本 ARCHIVED）| SSE-S3/OSS |

**访问权限**：
- `cpq-ugnx-source` / `cpq-stp-source`：仅 `SYSTEM_ADMIN` / `UG_ENGINEER` 可下载（签名 URL 5 分钟）
- `cpq-3d-glb`：所有登录用户可读（CDN 加速）

### 6.8 转换失败处理

| 错误类型 | 处理 |
|---|---|
| `.stp` 文件损坏 / AP 版本不支持 | 状态 `FAILED` + 告警工程师 "请用 AP214 重新导出" + **保留 .prt 和 .stp 原文件**不删 |
| `.prt` 上传但 `.stp` 缺失 | 拒绝上传（两文件必须配对） |
| 两文件 MD5 不一致 / 时间戳异常 | 警告但允许（人工确认）|
| 转换超时（> 5 分钟）| Worker kill + 状态 `TIMEOUT` + 可手动重试 |
| 内存溢出（复杂装配体）| 自动降级 tessellation 精度 + 重试 |
| 转换后 GLB 校验失败 | gltf-validator 检测 + 失败 ROLLBACK + 保留 STEP |

### 6.9 转换队列与并发

- 队列：RabbitMQ / Redis Stream（项目已有 → 复用，没有 → 用 Postgres LISTEN/NOTIFY 简化）
- Worker 池：3 个并发（按 CPU 调整，单 worker 转换需 ~1GB RAM）
- 任务超时：5 分钟（复杂装配体可配 10 分钟）
- 重试策略：失败留人工干预入口，**不自动重试**（避免雪崩 + 工程师可能要先修 STEP 文件）

### 6.10 前端实时预览（D 方案补充）

工程师上传 `.stp` 后，**先用前端 occt-import-js 做快速低质量预览**（10 秒内）：

```
工程师上传 .prt + .stp
    ↓
浏览器内 occt-import-js 解析 .stp → 立即显示低质量 3D（无 Draco / 全顶点）
    ↓
工程师确认视觉无误 → 点"确认提交"
    ↓
触发后端正式高质量转换 + 入库（异步 ~ 30-60s）
    ↓
转换完成邮件通知 + 浏览器内自动刷新加载高质量 GLB
```

**好处**：工程师不用等 60 秒就知道文件对不对，发现错误立即重新导出，节省服务器算力。

### 7.4 product_config_option_value (选项取值 · v0.4 收敛版 — 选项树即特征树)

> **2026-05-26 重大收敛**：v0.2 查看模式废弃后，原 4 张特征表 (`mat_feature_type` / `mat_feature_category` / `mat_feature` / `mat_feature_reference_type` / `mat_feature_reference` / `mat_part_mesh_feature`) **全部废弃**。特征语义并入选项值表 + 新增 `product_config_value_reference` 替代 `mat_feature_reference`。

```sql
CREATE TABLE IF NOT EXISTS product_config_option_value (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    option_id       UUID NOT NULL REFERENCES product_config_option(id) ON DELETE CASCADE,
    code            VARCHAR(64) NOT NULL,                -- AGCU85 / RED / M8 / STD / ENH / PRM

    /*  示例 (feature_type=THREAD):
        {"thread_spec":"M8","depth_mm":15,"tolerance":"6H"}

        {"composition":{"Ag":85,"Cu":15},"density_kg_m3":10500}

    tags            TEXT[] NOT NULL DEFAULT '{}',        -- 灵活打标签 (['critical','custom'])
    geometry_ref    JSONB DEFAULT '{}',                  -- 几何信息（包围盒/位置，可从 .stp 自动提取）

    attach_position      JSONB,                            -- 子模型挂载位置 {"position":[x,y,z],"rotation":[x,y,z],"scale":[x,y,z]}
    replace_base_mesh    VARCHAR(128),                    -- attach_mode=SWAP 时要替换的 base mesh 名

    metadata        JSONB DEFAULT '{}',
    UNIQUE (option_id, code),
    CONSTRAINT chk_value_attach_mode CHECK (attach_mode IN ('OVERLAY','SWAP','REPLACE_BASE'))
);
CREATE INDEX idx_value_feature_type ON product_config_option_value(feature_type);
CREATE INDEX idx_value_tags ON product_config_option_value USING GIN(tags);
CREATE INDEX idx_value_attributes ON product_config_option_value USING GIN(attributes);
CREATE INDEX idx_value_sub_model ON product_config_option_value(sub_model_part_no) WHERE sub_model_part_no IS NOT NULL;
```

**字段说明**：

| 字段 | 来源 | 用途 |
|---|---|---|
| `feature_type` | 原 mat_feature.feature_type 字段 | 标识该取值的特征类型（用户自定义，开放枚举）|
| `attributes` JSONB | 原 mat_feature.attributes | 灵活属性（按 feature_type 填规格/尺寸/工艺参数）|
| `tags` TEXT[] | 原 mat_feature.tags | 灵活打标签便于检索 |
| `geometry_ref` JSONB | 原 mat_feature.geometry_ref | 几何包围盒等，可从 STEP 自动提取 |
| **`sub_model_part_no`** | v0.4 新增 | **关联独立子模型 partNo**（不同选项绑不同 3D 模型，反馈 2）|
| **`attach_mode`** | v0.4 新增 | OVERLAY=叠加 / SWAP=替换 base 某 mesh / REPLACE_BASE=替换整个 base 模型 |
| **`attach_position`** | v0.4 新增 | 子模型在场景中的位置 / 旋转 / 缩放 |

**两种 3D 联动策略**：

| 策略 | 适用 | 实现 |
|---|---|---|
| **A. 单一 base.glb + 5 种 Action** | 同形态变体（颜色/部件显隐）| `product_config_3d_rule` (§7.6) — SHOW_MESH / HIDE_MESH / REPLACE_MATERIAL |
| **B. 子模型动态加载** | 完全不同的几何 | `option_value.sub_model_part_no` — 加载独立 GLB 合成场景 |

**实际使用**：两种策略可**混用**。如电机产品：
- "型号" 选项三个取值（标准 / 防爆 / 高效）→ 各绑**独立子模型**（壳体完全不同）
- "颜色" 选项五个取值 → 复用 base 模型，仅 REPLACE_MATERIAL
- "附加部件" 选项 → 复用 base 模型，SHOW/HIDE 即可

|---|---|---|---|
| `UGNX_SOURCE` | `prt` | `cpq-ugnx-source` | UG 工程师源文件 — 永久保存 |
| `STP_NEUTRAL` | `stp` | `cpq-stp-source` | STEP 中性格式 — 永久保存 + 转换源 |
| `GLB_RENDER` | `glb` | `cpq-3d-glb` | 浏览器渲染 + Draco 压缩 |

**关联 (`related_file_id`)**：
- STP_NEUTRAL.related_file_id → UGNX_SOURCE.id（STEP 来自哪个 UG 源）
- GLB_RENDER.related_file_id → STP_NEUTRAL.id（GLB 来自哪个 STEP）

**优势**：
- 未来加格式（Parasolid `.x_t` / CATIA `.CATPart`）只需加 `file_role` 枚举值（DML 一行），不改表结构
- 同一 partNo 不同版本可保留全部三文件历史
- 查询某 partNo 当前 UG 源文件：`SELECT * FROM mat_part_source_file WHERE part_no='X' AND file_role='UGNX_SOURCE' AND is_current=true`

## 十、与现有系统衔接

### 10.1 与报价单（quotation）的关系（v0.4 P0 修复，2026-05-26）

```
v0.4 选配 → 提交建实例 (SUBMITTED) → 三选一:
                                      ① NEW_QUOTATION: 生成 mat_part + quotation + line_item → 实例 LINKED
                                      ② SAVE_DRAFT: 仅入库实例（不触报价单）
                                      ③ LINK_EXISTING: 添加 line_item 到已有报价单 → 实例 LINKED
                ↓
              后续走现有报价单编辑流程（不动）
```

**关键**：
- v0.4 选配实例是**弱关联**到报价单 — 实例可独立存在（SUBMITTED 但未 LINKED），完成选配后所有现有功能（公式 / 模板 / 审批 / 导出）零变化
- **报价单内 3D 不展示**（v0.4 移除 §15）— 报价单仍是表格 + 公式 + Excel/PDF 输出形态
- 报价单 line_item 通过 `instance_id` 反查可看到原始选配实例（追溯）— 但报价单 PDF / Excel 不渲染 3D
- 销售想看 3D 配置 → 通过 line_item.instance_id 跳转到「选配实例列表」→ 实例详情

### 10.2 与组件管理 / 模板管理的关系

| 概念 | v0.3 配置模板 | 现有报价模板 |
|---|---|---|
| 主表 | `product_config_template` | `template` |
| 配置粒度 | 选项 / 取值 / 3D 规则 / 价格规则 | 字段 / 公式 / 子组件 |
| 服务对象 | 客户选购前阶段 | 报价单生成阶段 |
| 共用数据 | 共用 mat_part / mat_feature | 共用 component |

两者**独立维护**，通过"配置模板的 `base_part_no` → 生成的子料号 → 报价单 line_item.product_part_no"链接。

### 10.3 与 mat_part 的关系（关键）

每次用户选配提交生成的 mat_part 子料号：

```
mat_part 表新行:
  part_no:              CFG-CONTACT-STRIP-A2F8X (自动生成，hash-based)
  part_name:            "接触片 增强款 Ag85Cu15 镍镀层 M8"
  product_type:         COMPOSITE (符合现有约束)
  material_recipe_id:   FK 到选配的材质配方
  config_fingerprint:   sha256 (复用 V167 已加列)
```

**`config_fingerprint` 全局唯一**（V167 unique index 已加）：相同选配组合**永远复用同一个 partNo**，避免主数据膨胀。

### 10.4 与 v0.2 共享数据通道

```
v0.3 ConfigOptionValue --(option) feature_id--> mat_feature (v0.2)
                                                    ↓
                                            mat_feature_reference
                                                    ↓
                                            业务实体 (process/material/part)
```

选配提交生成的 line_item 自动写入 mat_feature_reference（已有特征），后续报价单内查看 3D 时（v0.2 模式）能正确显示。

### 10.5 与公式引擎的关系

- 价格规则 `FORMULA` 类型可复用 CPQ 公式引擎 token（含 `global_variable`）
- 例：表面处理价格 = 表面积 × 镀层厚度 × 银价（global_variable）
- 公式 token 存于 `product_config_price_rule.formula_token` JSONB

### 13.1 为什么需要版本管理

同一个选配实例可能经历多次变更：
- 销售草拟 → 客户反馈"加防尘罩" → 销售改 → 客户又要"换镀层" → 提交 → 后续报价单还要改

### 17.5 客户身份处理 SOP（v0.4 P1 修复，2026-05-26）

**问题背景**：公网客户自助提交时只填了"姓名 + 电话 + 邮箱 + 留言"，没有 customer_id。直接 INSERT 到 customer 主表会污染主数据（同一人多次提交 = 多个客户）。

**SOP**：引入 `customer_lead`（客户线索）中间层，待销售审核后再决定"绑定现有客户 / 新建客户 / 拒绝"。

#### `customer_lead` (客户线索)

```sql
CREATE TABLE IF NOT EXISTS customer_lead (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    lead_code       VARCHAR(40) NOT NULL UNIQUE,      -- LEAD-{yyyyMM}-{seq:04d}
    -- 来源
    source_type     VARCHAR(32) NOT NULL,             -- CUSTOMER_SELF / SHARED_LINK / IMPORT_BATCH
    share_token     VARCHAR(64),                       -- 客户自助来源时填
    -- 客户提交的信息（未经核实）
    contact_name    VARCHAR(128) NOT NULL,
    contact_phone   VARCHAR(40) NOT NULL,
    contact_email   VARCHAR(128),
    company_name    VARCHAR(255),
    note            TEXT,
    -- 审核状态
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING_REVIEW',
                                                       -- PENDING_REVIEW / CONVERTED / REJECTED
    reviewed_by     UUID,
    reviewed_at     TIMESTAMPTZ,
    review_action   VARCHAR(32),                       -- BIND_EXISTING / CREATE_NEW / REJECT
    bound_customer_id UUID REFERENCES customer(id),    -- review_action=BIND_EXISTING 时填
    review_note     TEXT,
    -- 时间
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    CONSTRAINT chk_lead_status CHECK (status IN ('PENDING_REVIEW','CONVERTED','REJECTED'))
);
CREATE INDEX idx_customer_lead_status ON customer_lead(status);
CREATE INDEX idx_customer_lead_phone ON customer_lead(contact_phone);
CREATE INDEX idx_customer_lead_share ON customer_lead(share_token) WHERE share_token IS NOT NULL;
```

#### 客户线索审核流程

```
公网客户提交 (POST /share/configurator/{token}/submit)
   ↓
后端事务:
1. INSERT customer_lead (PENDING_REVIEW) → 返回 lead_code = LEAD-yyyyMM-NNNN
2. INSERT product_config_instance:
   - status = 'SUBMITTED'
   - customer_id = NULL
   - customer_lead_id = <new>
   - share_token = <token>
3. 触发通知: 销售在「客户线索 / 实例列表」可见
   ↓
销售在「⚙️ 系统设置 / 客户线索」打开 PENDING_REVIEW 列表
   ↓
3 种审核动作（POST /customer-leads/{id}/review）:

① BIND_EXISTING (绑定现有客户)
   - 销售搜 customer 主表，找到匹配的客户（如电话匹配）
   - 后端: lead.bound_customer_id = <existingCustomerId>, status='CONVERTED'
   - 后端: instance.customer_id = <existingCustomerId> (UPDATE 关联)
   - 后端: instance.customer_lead_id 保留作为来源追溯

② CREATE_NEW (新建客户)
   - 销售填补完整信息（公司全称 / 行业 / 客户分类 / 信用额度等）
   - 后端: INSERT customer (新主档)
   - 后端: lead.bound_customer_id = <newCustomerId>, status='CONVERTED'
   - 后端: instance.customer_id = <newCustomerId> (UPDATE 关联)

③ REJECT (拒绝)
   - 销售判断: 同一客户重复提交 / 不真实 / 测试单
   - 后端: lead.status='REJECTED', review_note=<原因>
   - 后端: instance.status='EXPIRED' (软关闭)
```

#### 反复提交去重

- 后端在 INSERT customer_lead 前先查：同 share_token + 同 contact_phone 在过去 7 天是否已有 PENDING_REVIEW 线索 → 有则更新已有 lead 的 updated_at + 不新建（或返回已有 lead_code 复用）
- 销售看 PENDING_REVIEW 列表时按 contact_phone 分组高亮 → 一次审核绑定该客户的所有 instance

#### 销售审核 UI（暂无原型，后期补）

「⚙️ 系统设置 / 客户线索」列表页：
- 列：lead_code / 来源类型 / 联系人 / 电话 / 邮箱 / 公司 / 提交时间 / 关联实例数 / 状态 / 操作
- 工具栏：按状态筛选 / 按 source 筛选 / 搜索电话/邮箱
- 行内操作：详情 / 绑定客户 / 新建客户 / 拒绝
- 详情 Drawer 显示该 lead 的所有关联 instance（一个客户可能多次提交）

## 十八之前补充 · 特征库（方案 B 快照复制，2026-05-26 立）

> **背景**：参考 ERP（鼎捷风格）已有的 5 张"料件特征群组"表（imsba / imsbal / imsbb / imsbd / imsbdl），CPQ 引入轻量"特征库"作为选配模板的**可复用主数据**。模板加载时**复制成快照**写入 `product_config_option` / `product_config_option_value`，保留来源指针仅作追溯 — 模板独立稳定（不被 ERP 改动惊吓），与 §十三 版本管理天然对齐。
>
> **方案 B 决策理由**（不选 A 直接引用 / 不选 C 混合）：
> - AP-39 教训：PUBLISHED 模板被外部主数据改动冲击 → 历史报价单语义漂移
> - §十三 版本管理 = 快照模式天然契合（拉取新版特征 = 新草稿版本）
> - ERP 同步可选不强依赖 — 首版手工录入，先在 CPQ 内验证业务
> - 后期想升级到 C 混合，加 `override_*` 字段即可平滑过渡

    data_source_ref   VARCHAR(80),                              -- ERP imsbb13 — 关联 master_data_source.code
    partno_prefix     VARCHAR(20),                              -- ERP imsbb14
    partno_suffix     VARCHAR(20),                              -- ERP imsbb15
    extra_attrs       JSONB,                                    -- 替代 udXX
    created_at        TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at        TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE(group_id, code)
);
CREATE INDEX idx_cpq_ff_group ON cpq_feature_field(group_id);

-- 3) 特征值（字段下的可选值，对应 ERP imsbd）
CREATE TABLE cpq_feature_value (
    id                BIGSERIAL PRIMARY KEY,
    field_id          BIGINT NOT NULL REFERENCES cpq_feature_field(id) ON DELETE CASCADE,
    code              VARCHAR(40) NOT NULL,                     -- ERP imsbd04
    label             VARCHAR(255) NOT NULL,                    -- 显示名称（来自 imsbdl）
    description       TEXT,                                     -- 全称
    sort_order        INTEGER NOT NULL DEFAULT 0,               -- ERP imsbd03
    partno_include    BOOLEAN NOT NULL DEFAULT TRUE,            -- ERP imsbd05 — 是否参与子料号拼接
    is_active         BOOLEAN NOT NULL DEFAULT TRUE,            -- ERP imsbdacti
    extra_attrs       JSONB,
    created_at        TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at        TIMESTAMP NOT NULL DEFAULT NOW(),
    UNIQUE(field_id, code)
);
CREATE INDEX idx_cpq_fv_field ON cpq_feature_value(field_id);
CREATE INDEX idx_cpq_fv_active ON cpq_feature_value(is_active);
```

### 18A.3 数据类型 × 赋值方式矩阵

参考 ERP 的 `imsbb04 类型` × `imsbb05 赋值方式` 二维设计，明确每种组合的语义：

| 数据类型 \ 赋值方式 | MANUAL（手工输入） | SELECT（枚举选择） | COMPUTED（公式计算） |
|---|---|---|---|
| **STRING** 字符 | 文本框 / partno_prefix+输入+partno_suffix | 下拉 / 多选 / 单选（用 cpq_feature_value） | 公式输出字符串（如拼接子料号） |
| **NUMBER** 数值 | 数字输入框 + min/max 校验 + decimal_places 限制 | 离散数值下拉（如厚度档位 0.3/0.5/0.8/1.0） | 公式输出数值（如基价 × 系数） |
| **DATE** 日期 | 日期选择器 | 较少用（如"季度选择"） | 公式输出日期（如交货期 = 今天 + LEADTIME） |
| **BOOLEAN** 布尔 | 单选 ✓/✗ | (退化为 MANUAL) | 公式输出布尔（如"是否需要审批"） |

- `MANUAL + STRING + partno_prefix='-L-'` → 输入 "100mm" → 子料号片段 `-L-100`
- `SELECT + STRING + values=[STD,ENH,PRM]` → 选 PRM → 子料号片段 = value.code = `PRM`
- `COMPUTED + NUMBER` → 公式 `BASE_PRICE * 1.13` → 输出 169.5 → 不参与子料号

### 18A.4 模板加载快照协议（核心：方案 B 复制语义）

`product_config_option` / `product_config_option_value` 加 2 个追溯字段（不强制外键，群组归档/删除不影响模板）：

```sql
ALTER TABLE product_config_option ADD COLUMN source_feature_field_id BIGINT;
ALTER TABLE product_config_option ADD COLUMN source_feature_snapshot_at TIMESTAMP;
ALTER TABLE product_config_option_value ADD COLUMN source_feature_value_id BIGINT;
ALTER TABLE product_config_option_value ADD COLUMN source_feature_snapshot_at TIMESTAMP;

CREATE INDEX idx_pco_src_field ON product_config_option(source_feature_field_id);
CREATE INDEX idx_pcov_src_value ON product_config_option_value(source_feature_value_id);
```

**快照写入算法（伪代码）**：

```pseudo
function importFeatureFieldsIntoTemplate(templateId, fieldIds: List<Long>):
    template = product_config_template[templateId]
    assert template.status IN ('DRAFT', 'EDITING')  # PUBLISHED 不允许直接改

    snapshot_ts = now()
    for fieldId in fieldIds:
        field = cpq_feature_field[fieldId]
        values = cpq_feature_value WHERE field_id = fieldId AND is_active = TRUE

        # 写入 option（快照复制，独立 PK，外键仅追溯）
        opt = INSERT INTO product_config_option (
            template_id      = templateId,
            code             = field.code,           # 复制（可能后续被改）
            label            = field.name,
            sort_order       = field.sort_order,
            data_type        = field.data_type,
            assign_mode      = field.assign_mode,
            is_required      = field.is_required,
            default_value    = field.default_value,
            min_value        = field.min_value,
            max_value        = field.max_value,
            partno_prefix    = field.partno_prefix,
            partno_suffix    = field.partno_suffix,
            source_feature_field_id    = fieldId,
            source_feature_snapshot_at = snapshot_ts
        )

        # 复制每个 value
        for val in values:
            INSERT INTO product_config_option_value (
                option_id      = opt.id,
                code           = val.code,
                label          = val.label,
                sort_order     = val.sort_order,
                partno_include = val.partno_include,
                source_feature_value_id    = val.id,
                source_feature_snapshot_at = snapshot_ts
            )

    return imported_count
```

**关键不变量**：
- 写入后 option/value 是**独立行**，外部 cpq_feature_* 改动不会自动影响
- 模板可局部修改（label / sort / is_required），无需告知特征库
- 特征库改后想同步 → 用户主动点"📋 重新拉取"（开新草稿版本）

### 18A.5 重新拉取协议（同步差异检测）

```pseudo
function refreshTemplateFromFeatureLibrary(templateId):
    options_with_source = product_config_option
        WHERE template_id = templateId AND source_feature_field_id IS NOT NULL
    diffs = []
    for opt in options_with_source:
        src_field = cpq_feature_field[opt.source_feature_field_id]
        if src_field is None:
            diffs.append({type: 'SOURCE_DELETED', option: opt})
            continue
        # 字段级 diff（label / type / required / default / min / max / prefix / suffix）
        field_diff = compareFields(opt, src_field)
        # 取值级 diff（新增 / 删除 / label 变更 / 排序）
        value_diff = compareValues(opt.values, src_field.values)
        if field_diff or value_diff:
            diffs.append({type: 'CHANGED', option: opt, field_diff, value_diff})

    return diffs   # UI 让用户逐项选「保留模板版本 / 采用特征库版本」
```

**触发条件**：用户手动点"📋 重新拉取" → 进入差异列表 → 选择性合并 → 提交后**创建模板新草稿版本**（不污染当前 PUBLISHED）

**禁止行为**：
- ❌ 后端定时自动同步 — 违反方案 B 决策（PUBLISHED 不被惊吓）
- ❌ 跨过差异 UI 直接整体覆盖 — 业务方需要逐项审核

### 18A.6 引用统计与影响面查询

```sql
-- 一个特征字段被多少个模板/版本引用
SELECT COUNT(DISTINCT pct.id) AS template_count,
       COUNT(DISTINCT pct.id || '@' || pct.version) AS version_count
FROM product_config_option pco
JOIN product_config_template pct ON pct.id = pco.template_id
WHERE pco.source_feature_field_id = :fieldId;

-- 一个特征群组的所有引用模板
SELECT DISTINCT pct.code, pct.name, pct.status
FROM product_config_option pco
JOIN product_config_template pct ON pct.id = pco.template_id
JOIN cpq_feature_field cff ON cff.id = pco.source_feature_field_id
WHERE cff.group_id = :groupId;
```

特征库管理页用这两个 SQL 在群组/字段/取值行显示「N 个模板引用 / N 个版本引用」。

### 18A.7 删除/归档保护

- 删除群组：检查是否仍有 PUBLISHED 模板的 option 引用 — 有则拒绝，要求先归档
- 归档群组：允许，但已加载到模板的快照不受影响（独立行）
- 删除特征字段/值：同上检查
- 通过 source_feature_*_id 外键设计是 `BIGINT` 不带 FK 约束，因此特征库删除后只是 `source_feature_*_id` 指向无效 ID — UI 上显示「源已删除」标识
