# 报价模板生成规则

> **用途**：把「客户 Excel 组件模板」转成 CPQ 组件管理里的一套配置（目录 → 组件 → 字段 → 组件 SQL 视图）的**唯一操作规则**。
> **读者**：配置实施者 / AI 协作者。拿到一份 `组件模板 - <客户>.xlsx`，照本文即可端到端建好。
> **权威来源**：以**系统代码为准**（`ComponentService` / `SqlViewValidator` / `component/types.ts`）+ 现网权威模型「报价模板V2」(dir 4007d593) / 「罗克韦尔模板」(dir f4e2bca1)。
> **同源约束**：改 `ComponentService.VALID_FIELD_TYPES` / `SqlViewValidator.FORBIDDEN_TABLE_TOKENS` / `default_source` 结构时同步更新本文。
> **配套**：字段类型/公式三层配置总论见 `docs/配置方法论-合并版.md`；V6 基础表清单见 `docs/方案制定前必读.md §V6`。

## 0. 客户模板 → 组件的映射心智

一份客户模板（`页签——<客户>.xls` 或 `组件模板 - <客户>.xlsx`）：

- **每个 sheet（页签）= 一个组件**（`component`，`componentType=NORMAL`；小计性质的 sheet 建 `SUBTOTAL`）。
- **每个列（表头字段）= 一个字段**（`fields[]` 里一项）。**⚠️ 个别组件还要加模板里没有的「料号列」字段**（材料成本+材质料号、外购件成本+组成件料号），因 tab_type 的 `partNoField` 必须指真实料号（§5.5/§7）。
- **每个组件配一个组件 SQL 视图**（`component_sql_view`，`$name`）当取数源；多行组件用它当 `data_driver_path` 驱动。
- **组件按页签语义配 `tabType`**（主件/材质元素/外购件…，费用页签留空，§5.5）。
- **模板目录 = 客户名称**（`component_directory.name = <客户名>`）。

模板只给「结构」（组件数 + 字段名），**数据来源由本文 §5 的 V6 表映射规则补齐；每种页签的完整配方见 §7；生成 JSON 的流程见 §6.0**。

## 1. 用户要求规则（本项目硬约束，优先级最高）

| # | 规则 | 说明 |
|---|---|---|
| **R1** | **字段统一用 `INPUT_TEXT`（文本栏位）/ `INPUT_NUMBER`（数字栏位）+ `default_source` 绑基础数据** | 不用只读 `BASIC_DATA`。连主键/标识字段（销售料号/元素/项次）也是 `INPUT_*`。列语义是文本→`INPUT_TEXT`，是量/率/金额→`INPUT_NUMBER`。 |
| **R2** | **禁止在数据库建物理视图（无 `CREATE VIEW` / DDL）** | 一律用「组件 SQL 视图功能」(`component_sql_view`，`$name` inline 子查询，运行时拼成子查询执行，不产生 DB 视图对象)。 |
| **R3** | **组件 SQL 视图 `sql_template` 一律 `FROM` V6 基础表** | 不依赖现网 `v_q_*_merged` 等物理视图（那是 DB 建的视图，违反 R2）。dry-run 会拦 V44/V76 废弃表。 |
| **R4** | **所有 SQL 列别名加 `_` 前缀** | 如 `_销售料号` / `_材质` / `_元素`。目的：业务列不被隐式 JOIN 误当谓词注入。字段 `default_source.path` 对应写 `$view._列`。 |
| **R4-例外** | **`hf_part_no` 是唯一不带 `_` 的别名** | 它是驱动键，隐式 JOIN 专门注入它做料号收窄；改名（如 `_hf_part_no`）会让视图返全表。 |
| **R5** | **字段名（`name`）保持中文业务原名，不带 `_`** | `_` 只加在 SQL 列别名 + `default_source.path` 上；字段显示名不变（如字段名 `损耗率%` 绑 `$mc_view._损耗率`）。 |
| **R6** | **金额性质字段标 `is_amount=true`** | 费用/单价/金额类。否则 footer 小计行不带 `¥`、不走货币格式化。 |

## 2. 系统代码约束（不满足会 400 / 异常 / 静默失败）

| # | 约束 | 出处 |
|---|---|---|
| **C1** | `field_type` 必须 ∈ `{FIXED_VALUE, INPUT_TEXT, INPUT_NUMBER, BASIC_DATA, DATA_SOURCE, FORMULA, LIST_FORMULA}` | `ComponentService.VALID_FIELD_TYPES` |
| **C2** | **多行 driver（`data_driver_path` 非空）+ 任一可编辑字段（`INPUT_TEXT/INPUT_NUMBER/LIST_FORMULA`）→ 必须声明 `rowKeyFields`**，否则**新建时抛异常** | `ComponentService.validateRowKeyConfig(hard=true)`。因 R1 统一用 `INPUT_*`，凡多行组件都命中 → 都要 `rowKeyFields`。 |
| **C3** | `rowKeyFields` 存**字段名**（如 `["项次","元素"]`），不是 SQL 列名；无稳定键用哨兵 `["__seq_no__"]` 按行号对齐 | 与现网模型一致（field name 命名空间） |
| **C4** | 组件 SQL 视图 dry-run（create/update 同步校验，失败 400）：必须 `SELECT`/`WITH` 开头；禁 DDL/DML（`INSERT/UPDATE/DELETE/CREATE/DROP/ALTER/TRUNCATE/DO`…）；**禁 V44/V76 废弃表**；禁 `:hfPartNo` 标量（用 `:hfPartNos` 数组）；`:__sk*`/`:__vf*` 前缀保留 | `SqlViewValidator.validate` |
| **C5** | `sqlViewName`：小写字母/下划线/数字，同组件内唯一；`scope` 默认 `COMPONENT`（跨组件引用才用 `GLOBAL` + `$$code.name`） | `CreateComponentSqlViewRequest` |
| **C6** | `default_source` **后端不强校验**（仅 `warnDefaultSourcePaths` 软告警列名）——路径写错不报错，但运行期取不到默认值，需自查 | `ComponentService.warnDefaultSourcePaths` |
| **C7** | 组件保存后自动 `refreshSnapshotsByComponent`（H1），改字段引用它的模板即时生效 | `ComponentService.update` |

**禁引废弃表清单**（`SqlViewValidator.FORBIDDEN_TABLE_TOKENS`）：
`mat_part / mat_bom / mat_process / mat_fee / plating_plan / mat_customer_part_mapping / element_price / element_daily_price / customer_tax / costing_part_*`。

## 3. 字段配置规则（`fields[]` 一项的结构）

```jsonc
{
  "name": "销售料号",             // 中文业务名，不带 _（R5）；作 rowKeyFields 引用键
  "field_type": "INPUT_TEXT",     // 文本→INPUT_TEXT，数值→INPUT_NUMBER（R1）
  "content": "",                  // 静态兜底值（数字列可填 "0"/"100"）
  "is_amount": false,             // 金额列 true（R6）
  "is_subtotal": false,           // 需被「小计」聚合的列 true
  "notes": "",
  "sort_order": 0,                // 列序，按模板列顺序
  "default_source": {             // 绑基础数据（R1）
    "path": "$cp_view._销售料号", // $视图._列（R4：列带 _；hf_part_no 除外）
    "type": "BASIC_DATA"
  }
}
```

**运行期取值三级兜底**：`用户输入 row[key] → default_source(解析 $view._列) → content 静态兜底 → 0`。用户在报价单改的值只活在当前草稿，不回写基础数据。

`default_source.type` 取值：`BASIC_DATA`（`$view._列` 形态，支持中文列，本项目主用）/ `BNF_PATH`（物理路径）/ `GLOBAL_VARIABLE`（`code`+`key_field_refs`）。

## 4. 组件 SQL 视图编写规则

```sql
SELECT
  <料号列>          AS hf_part_no,     -- 驱动键：销售料号别名成 hf_part_no（R4-例外，不带 _）
  <料号列>          AS _销售料号,       -- 业务列一律 _ 前缀（R4）
  <列>              AS _材质,
  ...
FROM <V6 基础表>                       -- R3：必须 V6 表
  LEFT JOIN <V6 表> ...
WHERE <表>.system_type = 'QUOTE'        -- 报价侧固定 QUOTE
  AND <表>.is_current                   -- 取当前版本
  AND <表>.customer_no = :customerCode  -- 按客户收窄（运行时注入）
ORDER BY ...
```

要点：
1. **`hf_part_no` 必出**且不带 `_`——报价侧料号即**销售料号**，把 `material_no`（或 `finished_material_no`）别名成 `hf_part_no` 接隐式 JOIN。
2. **`:customerCode`** = 报价单客户的 `customer_no`，框架运行时注入；dry-run 时按 NULL 兜底（返 0 行不报错）。
3. 业务列别名全部 `_` 前缀（R4）；`default_source.path` 对应 `$view._列`。
4. **一对多会让驱动行翻倍**时，用**相关子查询 `string_agg`** 把多值聚合成单值（见 §5「材质」）。
5. 只 `SELECT`/`WITH`；不写 DDL/DML；不引废弃表（C4）。
6. 🚨 **版本化表禁止出现在带 `is_current` 的 `LEFT JOIN ON` 里 —— 用标量子查询取值**（2026-07-22 罗克韦尔来料树事故）：
   - **背景**：DRAFT 报价单渲染时，框架 `QuotePendingRewriter` 会把版本化白名单表（`unit_price` / `material_bom_item` / `material_master` / `capacity` 等有 `pending_quotation_id` 列的表）**整表 token 替换成带 `:pq` 的子查询**，让 pending（待生效）行也被看到。
   - **坑**：当版本化表写成 `LEFT JOIN unit_price up ON ... AND up.is_current=true`（`is_current` 在 **JOIN ON 复合条件**里）时，pending 改写会产出**参数错位的畸形 SQL**，运行期报 `The column index is out of range: N, number of columns: M`（driver 执行失败，静默返 0 行 / 节点数据全空）。dry-run（EXPLAIN，无 pending 上下文）**照样通过**，只在真实渲染暴露。
   - **规避**：把该 `LEFT JOIN <版本化表>` 改成**相关标量子查询**取值——子查询 WHERE 里带 `is_current` 是安全的（同 `NOT EXISTS` 子查询）：
     ```sql
     -- ❌ 触发 pending 改写 bug：
     LEFT JOIN unit_price up ON up.finished_material_no=x.material_no AND up.code=x.component_no
       AND up.price_type='INCOMING_MATERIAL_PROCESS' AND up.is_current=true
     -- ✅ 改标量子查询：
     (SELECT up.base_value FROM unit_price up
       WHERE up.finished_material_no=x.material_no AND up.code=x.component_no
         AND up.price_type='INCOMING_MATERIAL_PROCESS' AND up.is_current LIMIT 1) AS _加工费值
     ```
   - 普通位置的 `is_current`（主 `WHERE` / 子查询 `WHERE` / `NOT EXISTS`）不受影响；只有**版本化表的 `LEFT JOIN ON` 内含 `is_current`** 才中招。
7. 🔢 **多行页签要按「项次/序号」数字正序显示 → 用组件 `sort_field`，别指望视图 `ORDER BY`**（2026-07-22 task-0722）：
   - **坑**：DRAFT 报价单渲染时，`QuotePendingRewriter` 把视图包成带锚点列的复杂子查询、执行器再套 `SELECT * FROM (...) inner_q`（**无外层 ORDER BY**）→ 视图里写的 `ORDER BY` **PostgreSQL 不保证透传**，多行页签会按 driver 返回序（常见倒序）显示。视图 `ORDER BY` 只对**已提交/冻结**报价单有效。
   - **正解**：给组件设 **`sort_field`**（= 本组件 `fields[].name` 之一，如「项次」「序号」）。快照组装层（`ConfigureSnapshotService`）按该字段做**数字感知升序**（数字段数字序、文本段字典序），DRAFT + 冻结都稳。树页签（`tabType=BOM`）按树序，不受 `sort_field` 约束。
   - **配法**：REST `PUT /components/{id}` 传 `sortField:"项次"`；或直接 `UPDATE component SET sort_field='项次' WHERE ...`。设后 `refresh-snapshot` 重算即生效。
   - 本项目已配：`其他费用/电镀费用/自制加工费/组装加工费 → sort_field=项次`、`元素 → sort_field=序号`。

## 5. V6 数据表映射规则（字段来源权威表）

> 料号口径：**报价侧「料号」= 销售料号 = `material_no`**（`material_master`/`material_customer_map`/`element_bom_item`/`material_bom_item` 的 `material_no`）。`production_no` 为生产料号，本报价场景不用作键。

### 5.1 产品信息（单行/成品级）——源 `material_customer_map` + `material_master`

| 模板字段 | V6 来源 |
|---|---|
| 销售料号 / hf_part_no | `material_master.material_no`（或 `material_customer_map.material_no`） |
| 客户料号名称 | `material_customer_map.customer_material_name` |
| 客户产品编号 | `material_customer_map.customer_product_no` |
| 客户图号 | `material_customer_map.customer_drawing_no` |
| 报价货币 | `material_customer_map.quote_currency` |
| 基础货币 / 汇率 / 付款方式 | `material_customer_map.base_currency / exchange_rate / payment_method` |
| 品名 / 规格 / 单重 | `material_master.material_name / specification / unit_weight` |

JOIN：`material_customer_map.material_no = material_master.material_no AND material_customer_map.customer_no = :customerCode`。

### 5.2 材料成本（多行/逐元素）——源 `element_bom_item`（材质经**本行 `material_part_no`** → `material_recipe` + `unit_price` 取元素单价）

**层级结构（务必理解）**：
- `element_bom_item` 是元素行驱动表：`.material_no` = **销售料号（成品）本身**，`.component_no` = **元素**（Ag/Cu/Ni…），`.seq_no` = 项次。
- `element_bom_item.material_part_no` = **本行元素所属的材质料号**（= 材质库配方码，如 991/99901）。
- `material_recipe.code` = 材质料号，`.name` = **材质名**（如 `H65带`/`Cu/Ag/Cu带`）；`material_master.material_no`（=material_part_no）的 `.material_name` 可作兜底名源。

🚨 **「一号多材质」铁律（2026-07-16 森萨塔 QT-20260716-2046 事故沉淀 → COMP-0063/0066/0070 修复）**：
- **一个销售料号在同一客户下可合法对应多个 `material_part_no`**（多材质链，各一条独立 element BOM 版本链、各留一个 `is_current=TRUE`；导入按材质料号分链是正确行为，**勿改升版/Q04ElementBomHandler**）。
- **材质名必须逐行取自本行 `material_part_no`**（`LEFT JOIN material_recipe mr ON mr.code = ebi.material_part_no`）。**严禁**用 `material_bom_item` 子查询把整单聚合成一个材质值——那样多材质链的元素行会变成「值相同、无从区分的匿名重复行」，且客户 `material_bom` 为空时材质恒空（旧方案的双重坑）。
- **`material_part_no` 进 driver 排序**（`ORDER BY material_no, material_part_no, seq_no`）让同材质连续成组。
- **🔑「材质」字段必须进 `rowKeyFields`**（如 `["销售料号","元素","材质"]`）。否则多材质链下 `(项次,元素)` 撞键 → 可编辑行 editRows 串行 / 末值×行数塌缩（见 `docs/反模式.md` 行键唯一化；记忆 `cpq-rowkey-uniqueness-disambiguation`）。这是纯改视图 SQL **不够**、必须连组件 rowKeyFields 一起改的点。
- **不要** DISTINCT/GROUP BY 压成一行（丢一种真实材质成本），也**不要**随便取一个材质料号只留一行（两种材质都是有效数据）。
- 「只显示客户实际报价的那一种材质」需另加锚点（母件 BOM `material_bom_item` INNER JOIN `ON child.component_no = ebi.material_part_no`，或报价行加"材质选择"字段），非改视图 SQL 能解决 → **另立需求**。

> 📌 **旧方案已弃**：早期 `$mc_view`/`$cl_view` 用 `string_agg(material_recipe.name) via material_bom_item.component_no = material_recipe.code` 取材质（本节历史版本）。已证有「一号多材质匿名重复 + material_bom 空则材质空」双缺陷，2026-07-16 统一改为上方 `material_part_no` 逐行方案。顺带备案：`material_bom_item.characteristic` 有 `RECIPE`/`ASSEMBLY`/`(空)` 三态，其中 `RECIPE` 只是 task-0708 repair-2 开发期临时代码遗留脏数据（当前 `MaterialBomMergeHandler` 只写 `null`/`ASSEMBLY`，`git log -S"RECIPE"` 零命中），**任何按 characteristic 过滤都不可取**。

| 模板字段 | V6 来源 |
|---|---|
| 销售料号 / hf_part_no | `element_bom_item.material_no` |
| **材质** | `COALESCE(mr.name, mm2.material_name)`，`mr.code = ebi.material_part_no` / `mm2.material_no = ebi.material_part_no`（**逐行取，见铁律**） |

#### 5.2.1 元素单价接入客户价格策略（`f_customer_element_price`，task-0722）

**目标**：给元素类组件的「单价」列接客户价格策略，报价/核价渲染时**按客户自动取元素单价**（不让用户手填）。**只改组件 `$view` 的 `sql_template` + 字段取数配置，不改字段类型**（不触发 AP-44）。

**核心机制**：价格逻辑封装在 PG 表函数（**已落库**）：
```
f_customer_element_price(客户编号 TEXT, 基准日 DATE)
  → TABLE(element_code 元素符号, unit_price 最终单价(含系数加价), currency 货币, price_unit 价格计价单位)
```
一次返回该客户全部有价元素（非逐行调用，无 N+1）。视图里 `LEFT JOIN` 它，用**元素符号**关联。

**三步配置**：
1. 驱动视图 `FROM` 加 JOIN（**两侧客户参数不同**）：
   ```sql
   -- 报价侧(system_type='QUOTE')：传运行时真实客户
   LEFT JOIN f_customer_element_price(:customerCode, :priceBaseDate) cep ON cep.element_code = ebi.component_no
   -- 核价侧(system_type='PRICING')：传字面量 '_GLOBAL_'
   LEFT JOIN f_customer_element_price('_GLOBAL_', :priceBaseDate) cep ON cep.element_code = ebi.component_no
   ```
2. `SELECT` 加输出列，中文别名与组件字段名**逐字一致**：`cep.unit_price AS 单价, cep.currency AS 货币`（⛔ 不输出「计价单位」，见硬约束 4）。
3. `component_sql_view.required_variables` 声明：报价侧 `["customerCode","priceBaseDate"]`；核价侧 `["priceBaseDate"]`（不含 customerCode，传的是字面量）。字段取数：报价侧字段 `default_source` 指 `$view.单价`/`$view.货币`（类型仍 `INPUT_*`）；核价侧 `basic_data_path` 指 `$view.单价`（类型 `BASIC_DATA`）。**字段类型一个字都不改**。

**🚨 硬约束（违反即出 bug，逐条自检）**：
1. **必须 `LEFT JOIN`，禁 `INNER JOIN`** —— 否则无价元素整行消失、行数变少、页签小计错。
2. **JOIN 键用元素符号列** `element_bom_item.component_no`（值如 Ag/Cu/301），**不是元素编号** —— 用错恒不命中，全表无价。
3. **禁 `COALESCE(cep.unit_price, 0)` 兜底** —— 无价必须留 `NULL` 触发「留空手填」；填 0 会静默算 0 元成本、不报错。
4. **⛔ 不要绑「计价单位」字段**。报价单「计价单位」列是 **BOM 发料单位** `ebi.issue_unit`（g/KG），被「毛用量/净用量」以 `unit_source_field` 引用做 g→KG 换算；改指价格 `price_unit`(元/kg) 会让换算读到非法 token、静默算错。**本期只绑「单价」+「货币」两字段**。
5. **核价侧传字面量 `'_GLOBAL_'`，且视图 WHERE 里不要加 `customer_no=:customerCode`** —— 核价基础数据 `customer_no` 恒 `'_GLOBAL_'`，加客户过滤一行查不到。
6. **别名与字段名逐字一致**（含全半角/空格）—— 否则路径解析不到、渲染空白。
7. **不为「源名/命中日期/取值方式」加额外输出列** —— 无消费方，徒增复杂度。

**前提与边界**：
- 客户没配价格策略 → 表函数返 0 行 → 元素行照常显示、单价留空手填（**设计如此，不报错**）。*（当前现网实测 `_GLOBAL_`/`CUST-1269` 均返 0 行 = 尚无策略数据，接上后暂全留空，待业务配策略后自动出价。）*
- `:priceBaseDate` 后端自动注入（=报价单创建日期；配置期预览回退今天），视图直接写占位符，不用自己算。已在 `SqlViewExecutor`（task-0722 §11.2）接好。
- 字段类型不变 → **不触发 AP-44**。

**自检**：`expand-driver` 端点或直接 SQL 实测——有价元素出数、无价元素返 `NULL`（不是 0、不是整行消失）；换一个没配策略的客户，元素行数不变、单价全空。

🔧 **要素名称=工序名称规则（2026-07-19 用户明确）**：

- 现网**绝大多数客户**的电镀数据在 `unit_price` `price_type='PLATING'`，且是**行式**：每料号两行（`cost_type='电镀加工费'` / `'电镀材料费'`，值在 `pricing_price`），**料号在 `code` 列**（`finished_material_no` 常为空）。`plating_fee`（费用列式）只有个别客户（如 8000143）有数据。

```sql
SELECT
  up.code AS hf_part_no,
  up.code AS _销售料号,
  NULL::varchar AS _电镀元素名称,     -- unit_price 无此维度→NULL（有 plating_scheme 数据的客户另议）
  NULL::numeric AS _镀层厚度,

  MAX(CASE WHEN up.cost_type = '电镀材料费' THEN up.pricing_price END) AS _电镀材料费,
  MAX(up.currency) AS _货币,
  MAX(up.unit) AS _计价单位
FROM unit_price up
WHERE up.system_type='QUOTE' AND up.price_type='PLATING' AND up.is_current AND up.customer_no = :customerCode
```

### 5.6 🚨 报价树渲染的**两种页签契约**（树页签 vs 数据页签，绑定变量不同，不可混——2026-07-22 QT-2085 事故）

配了 `tabType=BOM` 树页签的模板，报价侧渲染（`ConfigureSnapshotService.snapshotLines`）**分两趟路由，两趟注入的绑定变量完全不同**：

| Pass | 覆盖的页签 | 引擎 | 框架注入的绑定变量 | 视图必须用的过滤契约 |
|---|---|---|---|---|
| **Pass 2 树渲染** | **仅 `tab_type='BOM'`**（判据 `BomTreeRenderService.isQuoteTreeTabType`） | `BomTreeRenderService.render(usage=QUOTE)` | **`:total_material_no`**（整棵 spine 的料号数组）+ `versionFilter` 宏 | `WHERE <料号列> = ANY(:total_material_no)`，且输出 `material_no`(本节点子件号) + `parent_no`(父号) 做边键匹配 |
| **Pass 1 普通平铺展开** | **其余所有数据页签**（材质元素/零件/外购件/费用页签…） | `componentDriverService.expand`（既有报价平铺路径） | **`:customerCode`** + 行料号（隐式 JOIN 注入 `hf_part_no`） | `WHERE ... customer_no = :customerCode`，输出 `<表>.material_no AS hf_part_no`（驱动键，接隐式 JOIN） |

**铁律**：
1. **`:total_material_no` 只在 Pass 2（BOM 树页签）注入**。数据页签走 Pass 1，`:total_material_no` **永不注入** → 若数据页签视图写 `= ANY(:total_material_no)`，运行时该变量无绑定 → **整表 0 行**（不报错、静默空）。这正是 QT-2085「只有产品页签有数据、材料成本/加工费全空」的根因。
2. **树页签**（产品/BOM）：视图用 `:total_material_no` + `material_no`/`parent_no` 边契约（见 §7.0）。
3. **数据页签**（材质元素/零件/外购件/费用）：视图**照常用 §4/§7 的平铺契约**（`hf_part_no` + `:customerCode`），**不要**因为"模板里有树"就改成 `:total_material_no`。它们在报价侧渲染为**独立平铺页签**（每行项目一份数据），**不挂到树节点上**。
4. 树的深度由 `costing_bom_tree_config` 的递归 SQL 决定（递归 `material_bom_item` 子件）。若产品的子件是 `characteristic='RECIPE'`（材质），递归条件**不能限死 `='ASSEMBLY'`**，否则树只剩根节点。默认递归全子件（RECIPE 材质 + ASSEMBLY 组成件都成节点）。

> 一句话：**只有 BOM 树页签换契约（`:total_material_no`），其余页签一律保持 `:customerCode` 平铺契约。** 把数据页签也改成树契约 = 静默返 0 行。

## 7. 组件配方大全（每种页签的完整配置，照抄即可）

> **用法**：拿到客户模板，按 sheet 名对号入座；每个配方给全 `tabType` / `driver=$view` / `rowKeyFields` / 逐字段（类型+绑定）/ 完整 SQL 视图。
> **字段类型判定**：标识/名称/单位/货币/料号 → `INPUT_TEXT`；量/率/金额/厚度/汇率 → `INPUT_NUMBER`；金额性质加 `is_amount=true`（数字列 `content="0"`，汇率 `"1"`）。
> **⚠️ 有的组件要加模板里没有的「料号列」字段**（材料成本加「材质料号」、外购件成本加「组成件料号」）——因为 `tabType` 的 `partNoField` 必须指真实料号（§5.5），模板只有名称列。
> 全部视图 `_` 前缀别名（`hf_part_no` 除外）、`FROM` V6 表、带 `system_type='QUOTE'`/`is_current`/`:customerCode`。
> **例外——§7.0 树页签视图**：`tabType=BOM` 的树页签走另一套契约（`:total_material_no` + `material_no`/`parent_no`），**不用** `hf_part_no`/`:customerCode`（详见 §5.6）。其余所有数据页签（§7.1~§7.7）一律用平铺契约。

### 7.0 🌳 产品（BOM 树页签，`tabType=BOM`）——树契约，非平铺

> **仅当产品页签要渲染成 BOM 树时用本配方**（否则产品用 §7.1 平铺主件）。走 Pass 2 树渲染，绑定变量是 `:total_material_no`（**不是** `:customerCode`），输出 `material_no`(本节点子件号)/`parent_no`(父号) 做树边匹配（§5.6）。
>
> 前置：① 该组件 `tabType=BOM`（保存自动 `bomRecursiveExpand=true`；被 COSTING 引用会 400 → 须新建报价专用组件）；② 建一条 `costing_bom_tree_config`（`usage=QUOTE`, active）的递归 SQL，递归 `material_bom_item` **全子件**（RECIPE 材质 + ASSEMBLY 组成件都成节点，别限死 ASSEMBLY），输出 `root_no/material_no/bom_version/parent_no/node_path`，入参 `:production_part_nos` + 客户传播列 `_cust`。

`driver=$cp_view` · `rowKeyFields=["销售料号"]` · `partNoField=销售料号` · 字段绑定同 §7.1（`$cp_view._销售料号` 等）。

```sql
-- $cp_view（树契约：边式全子件 + 根分支；material_no=子, parent_no=父）
SELECT
  mbt.component_no AS material_no,          -- 本节点（子件号）
  mbt.material_no  AS parent_no,            -- 父节点
  mbt.component_no AS _销售料号,
  mcm.customer_material_name AS _客户料号名称, mcm.customer_product_no AS _客户产品编号,
  mcm.quote_currency AS _报价货币, mm.standard_unit AS _单位, mcm.exchange_rate AS _汇率
FROM material_bom_item mbt
LEFT JOIN material_master mm        ON mm.material_no  = mbt.component_no
LEFT JOIN material_customer_map mcm ON mcm.material_no = mbt.component_no
WHERE mbt.system_type='QUOTE' AND mbt.is_current
  AND mbt.component_no = ANY(:total_material_no)
UNION ALL   -- 根分支：无 ASSEMBLY/子件父边的成品自身（树根）
SELECT mm.material_no, NULL::text,
  mm.material_no, mcm.customer_material_name, mcm.customer_product_no,
  mcm.quote_currency, mm.standard_unit, mcm.exchange_rate
FROM material_master mm
LEFT JOIN material_customer_map mcm ON mcm.material_no = mm.material_no
WHERE mm.material_no = ANY(:total_material_no)
  AND NOT EXISTS (SELECT 1 FROM material_bom_item x
                  WHERE x.component_no = mm.material_no AND x.system_type='QUOTE' AND x.is_current)
```

> ⚠️ 树页签只出树结构 + 主件表头；材质/工序等**数据仍在各自数据页签（§7.2/§7.6）平铺渲染**，不挂到树节点上。
