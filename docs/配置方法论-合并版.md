# 第一部 · 配置操作手册 📘

## 1. 四层架构（看图定位）

```
L0  物理层  ──  V6 基础资料表（23 张）+ costing_* 核价表 + v_costing_* / v_composite_child_* 视图
               ↑ 维护入口：「核价基础数据」/「全局变量配置」/ 数据导入
               ⚠️ V44 老表（mat_part / mat_bom / mat_process / mat_fee / plating_plan /

L1  注册层  ──  global_variable_definition + global_variable_value（单表）— 给 L0 数据命名、暴露给公式

L2  组件层  ──  component（字段定义：8 种 field_type + 公式 token 数组 + 可选内嵌 SQL 视图）

—— 运行时 ——

```

**语义边界（改哪层影响谁）**：

| 改动 | 影响范围 |
|---|---|
| 改 L0 数据 | L1~L3 实时反映（除已冻结的快照） |
| 改 L2 组件字段 | 保存即自动同步到引用模板（H1）；**已发布报价单不动**（snapshot 冻结，见 §10） |

## 2. 决策树：先定层，再选来源

### Q1：我要配什么？

| 需求 | 选哪层 | 配置入口 |
|---|---|---|
| 字段「取一个值」 | **L2 组件** `BASIC_DATA` | 组件管理 → 字段类型 BASIC_DATA + 路径 |
| 字段「算一个值」 | **L2 组件** `FORMULA` | 组件管理 → 字段类型 FORMULA + token 表达式 |
| 字段「按外部/多类型数据源取值」 | **L2 组件** `DATA_SOURCE` | 组件管理 → DATA_SOURCE + 选 4 子类型之一 |
| 字段「列表 × 逐项条件分支公式」 | **L2 组件** `LIST_FORMULA` | 组件管理 → LIST_FORMULA + 绑配置模板 |
| 报价/核价单「加一列数据展示」 | **L3 Excel 模板** `VARIABLE` | Excel 模板配置 → 加列 → VARIABLE |
| 报价/核价单「加一列计算结果」 | **L3 Excel 模板** `FORMULA`/`EXCEL_FORMULA`/`CARD_FORMULA`/`TAB_JOIN_FORMULA` | Excel 模板配置 → 加列 → 选对应来源（见 §4.2） |
| 比对视图「加一个差异指标」 | **L0 ComparisonTag + L3 列 tag** | SQL 注册标签 + Excel 列设 comparison_tag |
| 公式取「参考价/汇率/系数」 | **L1 全局变量** | 全局变量配置 → 注册 + 维护数据 |
| 公式「跨组件汇总」 | **L2 component_subtotal token** | 公式编辑器 →「其他数据源」→ 选组件小计 |
| 公式「按当前行元素动态查表」 | **L1 全局变量动态 key 模式** | 全局变量 picker →「⚡ 动态查表」 |
| B 页签「VLOOKUP/SUMIF 同卡片 A 页签」 | **L2 cross_tab_ref token** | 组件管理 → FORMULA →「跨页签引用」 |

### Q2：字段值是单值还是多行？

| 场景 | 答案 | 实操 |
|---|---|---|
| 整张单一个值（汇率、税率） | **单值** | 全局变量静态 key 或带谓词 BNF 路径 |
| 每料号一行（BOM、工序成本） | **多行** | 组件 `data_driver_path` 设主表/视图，字段路径不带谓词靠隐式 JOIN |
| 嵌套行（料号下每电镀方案行） | **多行** | driver 视图带 `hf_part_no` + 业务键，隐式 JOIN 自动收窄 |

### Q3：跨组件 / 跨页签取值？

| 场景 | 答案 |
|---|---|
| 单值跨组件小计（总成本卡引用电镀材料费 ∑） | `component_subtotal` token，对方组件需有 `is_subtotal=true` 字段 |
| 同卡片 A 页签按列匹配取值/聚合（VLOOKUP/SUMIF） | `cross_tab_ref` token（§3.4） |
| Excel 列汇总单卡多页签内容 | Excel `TAB_JOIN_FORMULA` 列（§4.2） |
| 跨表按当前行元素查表 | 全局变量动态 key（路径不带谓词，靠隐式 JOIN 注入 driver row 的 element_name） |

## 3. 组件字段配置（L2）

「组件管理」→ 编辑组件 → 字段配置表（`FieldConfigTable`）。每个字段由 `field_type` 决定行为，「内容/配置」列按类型动态切换 UI。

### 字段配置表列

| 列 | 说明 |
|---|---|
| 字段名 | `name` — 同时作为公式 token 的引用 key（`{"type":"field","value":"<name>"}`） |
| 字段类型 | `field_type` — 见 §3.1 |
| 内容/配置 | 按类型动态：固定值框 / 路径徽章 / 数据源按钮 / 默认值来源 / 公式提示 |
| 金额 | `is_amount` — 渲染按货币格式化 |
| 小计 | `is_subtotal` — 公式聚合用；**可多个**，每个被标记字段各算一列总计（Plan 2，`validateFields` 不再限制数量） |
| 备注 | `notes` — 配置者留言，渲染不展示 |

> 💰 **金额字段配置通则（2026-06-18）**：凡**金额性质**的字段（费用 / 金额 / 单价 / 金额类小计等），配置时**必须标 `is_amount=true`**，否则报价/核价表 footer 小计行不显示 `¥` 前缀、且单元格不走货币格式化。
> - 渲染规则见《三大核心模块基线》§4.6：footer **严格按 `field.is_amount === true`** 决定是否加 `¥`，与 `is_subtotal` 无关（小计列不会"默认带 ¥"）。
> - 这是**按字段语义判断的通则**，由配置者对当前模板里**实际存在**的金额字段逐一判断标注，**不针对任何具体字段名/具体模板**——字段可被用户改名/增删，规则不枚举字段。

### 3.1 字段类型全集（7 种 UI 可选）

> **权威来源**：前端 `FIELD_TYPE_OPTIONS`（`cpq-frontend/src/pages/component/types.ts`）暴露 **7 种**；后端 `ComponentService.VALID_FIELD_TYPES` 额外保留一个 legacy `INPUT`（旧数据兼容，不在 UI 出现）。新配置只用下表 7 种。

| field_type | 数据来源 | UI 渲染 | 关键 JSON 字段 |
|---|---|---|---|
| `FIXED_VALUE` | 字面量 | 只读单元格 | `content` |
| `INPUT_TEXT` | 用户输入（文本） | 文本输入框 | `content`、`default_source` |
| `INPUT_NUMBER` | 用户输入（数字） | 数字输入框 | `content`、`default_source` |
| `BASIC_DATA` | BNF 路径查物理表/视图/SQL 视图 | 只读单元格 | `basic_data_path`、`global_variable_code` |
| `DATA_SOURCE` | 4 种数据源（见 §3.3） | 只读 / 下拉 | `datasource_binding.{type,...}` |
| `FORMULA` | 公式引擎计算 | 计算结果只读 | `formula_tokens` |
| `LIST_FORMULA` | 配置模板列表 × 逐项条件分支公式 | 列驱动多行 | `list_formula_config` |

#### FIXED_VALUE — 固定字面量
直接输入 `content`，按字面量显示。常用于标签 / 单位 / 说明文字。
**坑**：`is_amount=true` 时 `content` 必须是数字字面量，否则货币格式化报错。

#### INPUT_TEXT / INPUT_NUMBER — 用户输入（三级兜底链）
报价单可编辑单元格。核心能力 `default_source` 默认值机制。行值为空时按序解析：

```
用户输入 row[key]
  → default_source.GLOBAL_VARIABLE → basicDataValues['@gvar:CODE']
  → default_source.BNF_PATH        → basicDataValues[{path}] / pathCache
  → field.content                  → 静态字面量兜底
  → 0
```

UI：「内容/配置」列左侧 `content` 静态兜底框 + 右侧「+ 默认值来源」→ `DefaultSourceEditor` 抽屉（选 type → 填配置 → 填测试 driverRow JSON →「测试解析」调 `POST /api/cpq/data-sources/resolve` 实时验证 → 保存）。
报价单 placeholder：命中 default_source → `默认 X · CODE`；仅 content → `默认 X`；用户输入即覆盖（覆盖值只活在当前草稿）。

#### BASIC_DATA — BNF 路径查表
只读，按 BNF 路径解析；`basic_data_path` 经 ImplicitJoinRewriter 隐式 JOIN driver row。
入口两个：「配置物理表路径」→ `PathPickerDrawer`；「🌐 全局变量」→ `GlobalVariablePickerDrawer`（编译为 BNF path + 设 `global_variable_code` 徽章）。
**设计原则**：BASIC_DATA 只查不算；**不要写表达式**（`mat_part.weight * mat_bom.pct` 禁止，拆成两个 path token 在 FORMULA 里组合）。

#### FORMULA — 公式引擎计算
公式 token 在模板编辑器 / FormulaZone 拖拽配置（见 §5.1 token 全集）。

#### LIST_FORMULA — 配置模板 + 条件分支公式
绑定**独立配置模板**（一级菜单「配置模板」维护，三表 `config_template`/`config_category`/`config_item`），按选定**大类**的明细项数 expand 行；每项配**多条件分支**（IF-ELSE-IF 链），运行时第一个 `condition=true` 的 branch 取其 formula 求值。

```jsonc
{
  "name": "工序成本", "field_type": "LIST_FORMULA", "is_amount": true,
  "list_formula_config": {
    "config_template_id": "...", "config_template_code": "STD_RULES",
    "category_code": "PROCESS",
    "per_item_rules": {
      "RIVET": {
        "branches": [
          { "condition": "[厚度] > 5 AND [材质] = 'AgCu85'", "formula": "[厚度] * 2.5" },
          { "condition": "[厚度] > 2", "formula": "[厚度] * 1.8" }
        ],
        "default_formula": "[厚度] * 1.5"
      },
      "LASER_WELD": { "branches": [], "default_formula": "100" }
      // 未显式配的项 → 走 config_item.default_value
    }
  }
}
```

求值时序：`useConfigTemplates` 收集 config_template_id → 拉详情 → 行数 = `category.items.length` → 每行按 `per_item_rules[item.code].branches` 顺序求条件 → 第一个 true 取 formula → 全不命中 `default_formula` → 再无 `item.default_value` → 都空渲染 `—`。
约束：配置模板须 **PUBLISHED** 才能引用（DRAFT 在 Drawer 过滤）；同组件多个 LIST_FORMULA 字段必须绑同 `(config_template_id, category_code)`（行数语义对齐）；条件语法 `[字段] op X` + AND/OR（不支持括号嵌套/NOT/IN/函数）；模板加新明细项 → 引用组件自动新增行（无需改字段配置）。

LIST_FORMULA 公式还可引用 BNF 路径参与计算，规则：
- `{表.列}`（含 `.`）→ BNF 数据库路径，从 driver 行级 `basicDataValues` 取，缺值 fallback partNo 级 pathCache；
- `{GV_CODE}`（不含 `.`）→ 全局变量；
- `[字段名]` → 本行其他字段；`[表名.列名]` → 列表项原生列。
- **坑**：`{mat_bom.net_qty}`（无谓词）返多行 → 当 null → 求值 0；必须用 `{...[element_name='Sn'].net_qty}` 单值形态。

### 3.2 数据源引用语法（物理表 / 本组件 SQL / 跨组件 SQL）

BNF 路径不仅能引用物理表/视图，还能引用**组件管理里配置的内嵌 SQL 视图**：

| 场景 | 写法 | 示例 |
|---|---|---|
| 物理表/视图 | `<sheet>[谓词].<col>` | `costing_summary[summary_no='CS-0001'].quote_currency` |
| **本组件 SQL 视图** | `$<sql_view_name>[谓词].<col>` | `$element_view[bom_type='ELEMENT'].composition_pct` |
| **跨组件 GLOBAL SQL 视图** | `$$<componentCode>.<sql_view_name>[谓词].<col>` | `$$COMP-FEE.assembly_fee[...].fee_value` |

`PathPickerDrawer` 的「SQL 视图」Tab 列出本组件可用视图 + 跨组件 GLOBAL 视图。`$`/`$$` 解析由后端 `com.cpq.datasource.sqlview` 包（`SqlViewPathRewriter` + `SqlViewExecutor`）做 BNF 解析层别名替换、拼成 inline subquery，再交由 `com.cpq.formula.dataloader.DataLoader` 执行。

🚨 **V44 老表禁用规则（AP-53）**：组件 `data_driver_path` + 字段 `basic_data_path` **禁止直引** V44 废弃表/PG 视图名（`mat_part`/`mat_bom`/`mat_process`/`mat_fee`/`plating_plan` 等），必须改走 `$<sql_view_name>` 引用，且 `component_sql_view.sql_template` 内部必须 FROM **V6 表**。废弃表清单与 V6 对照见**附录 B**。

用户 SQL 纪律：
- ❌ 禁用 `:hfPartNo` 标量占位符（由外层 batch 注入）；❌ 禁止 DDL/DML（保存时 EXPLAIN 校验）。
- ⚠️ 双层冻结：模板 PUBLISHED 冻结至 `template.sql_views_snapshot`，报价单 SUBMITTED 冻结至 `quotation_component_sql_snapshot`。

### 3.3 DATA_SOURCE — 4 种数据源子类型

| type | 配置字段 | 解析行为 | 适用 |
|---|---|---|---|
| `DATABASE_QUERY`（缺省） | `datasource_id` + `params` | `DataSourceService.execute` | 数据源管理页注册的查询 |
| `GLOBAL_VARIABLE` | `global_variable_code` + `key_field_refs` | 按 driver row 解 → `global_variable_value` 单表 | 引用配置中心变量值 |
| `BNF_PATH` | `bnf_path` | `DataLoader.loadByPath` | 与 BASIC_DATA 同源（数据源形态） |
| `HTTP_API` | `api_config.{url_template, response_path, auth_token_env}` | 外部 REST（严格安全） | 集成外部系统 |

后端 `ComponentService.validateFields` 按 `datasource_binding.type` 分发校验必填字段（缺则保存 400）：DATABASE_QUERY→`datasource_id`；GLOBAL_VARIABLE→`global_variable_code`；BNF_PATH→`bnf_path` 非空；HTTP_API→`api_config.url_template`。
**HTTP_API 的安全配置（默认拒绝、allowed-hosts 白名单、token 走环境变量等）见 `HTTP_API_安全配置.md`「默认拒绝原则」与「不变量」章节，本文不再复述。**

### 3.4 跨页签引用 cross_tab_ref（VLOOKUP / SUMIF）

B 页签 FORMULA 字段按「A.列 = B.列（可多列 AND）」匹配同卡片（同目录）内 A 页签已算行，取值（NONE）或聚合（SUM/AVG/COUNT/MAX/MIN）。
配置：组件管理 → B 组件 → FORMULA 字段 → 公式区「跨页签引用」→ `CrossTabRefDrawer`（选源组件 A → 配匹配列对 → 选目标列/聚合方式）。

```json
{ "type": "cross_tab_ref", "source": "<A组件componentId>", "sourceLabel": "A组件名",
  "target": "目标列名", "match": [{"a": "A列", "b": "B列"}], "agg": "NONE" }
```

语义边界：0 行匹配→0；NONE 多行匹配→整公式报错按 0；聚合非数字→按 0；空/纯空白键不参与匹配；COUNT 无需 target 列。
范围：**仅同卡片（同目录）**其他组件，禁跨目录；禁循环依赖（模板 publish 时 `TemplateService` 拓扑校验，有环拒绝）；计算顺序按 `CrossTabComponentOrder.topoOrder`（Kahn BFS，A 先于 B）。
**目标公式 targetExpr**（进阶）：目标列可改用公式，内含 `field`（A 匹配行列）、`b_field`（B 当前行列）、四则、`global_variable`；对每匹配 A 行先算再按 agg 聚合（SUMPRODUCT 式）；`targetExpr` 非空时优先于 `target`。

#### 3.4.1 多 source 链式 SUM + KSUM 嵌套预聚合（2026-06-13）

在 `SUM(...)`（及 AVG/MAX/MIN/COUNT）的行级表达式里，除引用驱动页签列外，还可：

**① 多 source 链式 SUM（v1）** —— 一个外层 `SUM(...)` 内引用**多个非宿主 source 页签**。合法前提：各 source 与宿主的 `row_key_fields` **两两可比**（`⊆`/`⊇`，含相等）。
- **驱动** = 最细 source（`row_key_fields` 集合最大者），结果 token `source` 镜像 `sources[0]`，并写有序 `sources`（最细→最粗）。
- **更粗 source 按公共行键广播**：对每个驱动行，按 `match` 在更粗 source 命中行 —— **0 命中 → 该项=0（不报错）**；**1 命中 → 取该行列值**；**>1 命中 → 报错引导改用 KSUM**。
- 互不包含（不可比）→ 序列化期报错（提示改用 KSUM 聚合更细/不可比页签）。
- 例：`SUM([元素.单价] + [来料.组成用量])`（驱动=元素`[料件,元素]`，来料`[料件]` 按料件广播）。

**② KSUM/KAVG/KMAX/KMIN/KCOUNT（v2 降维投影预聚合）** —— 内层算子，对**单个被聚合页签**按**宿主行键塌缩成标量**，绕开"互不包含维度→笛卡尔积"硬限制（如每料件多要素行的外购件费用合计）。
- 只能写在**外层 SUM/AVG/MAX/MIN/COUNT 内**（顶层裸 KSUM 报错 = 决策 M）；不得 K 套 K（决策 J，双端拒）。
- **inner 白名单（§2.4 同源）**：KSUM 内只允许 `field`（限被聚合页签自己的列）/ `operator` / `number` / `bracket` / `global_variable`(实际 token `type=path`)。**拒**：宿主列(`b_field`) / 跨页签 `field` / `component_subtotal` / `quotation_field` / `product_attribute` / `previous_row_subtotal` / 嵌套 `cross_tab_ref`。前端序列化期 + 后端 `TokenMappabilityValidator` 双端镜像拒绝。
- **决策 K 空集分流**：`KSUM`/`KCOUNT` 命中 0 行 → **0（静默）**；`KAVG`/`KMAX`/`KMIN` 命中 0 行 → **null → 整外层表达式塌 0 + ⚠ 错误标记**（"无定义"，不静默给 0）。
- **同页签二选一（决策 I2）**：同一页签不能在同一 SUM 内既被 KSUM 聚合又被裸引用（语义二义）→ 报错。
- 单列 KSUM（如 `KSUM([外购件.费用])`）**强制折叠成 targetExpr**（`target=''`），**不复用单列 shortcut**（C2）。
- 折叠 token 形态：`{ "type":"cross_tab_ref", "projectToHostKey":true, "source":"<被聚合页签id>", "agg":"SUM", "match":[...], "targetExpr":[{"type":"field","value":"费用","source":"<被聚合页签id>"}] }`，嵌入外层 token 的 `targetExpr`。
- 例（近 E2E 真实公式）：`SUM([元素.重量(g)]/1000*[元素.单价]*(1+[元素.损耗率]/100) + KSUM([外购件.费用])) * [组成用量]`。

**③ `K SUM` 不可拆写（C3）**：`KSUM` 等必须连写，写成 `K SUM(...)` 触发专门文案"不能拆写，请连写"。

**④ Excel 模型 B 差异（降级）**：Excel 列模型（`TabJoinPlanEvaluator`）**暂不支持** KSUM 降维 / 多 source 链式 SUM，遇此类 token **显式抛错**（非静默少算），上层降级为该列空值 + warn 日志。改用页签连表渲染（模型 A）。

**⑤ 配色即时反馈（insideKsum）**：编辑器对 KSUM 区间内的违规块（宿主列 / 第二被聚合页签）即时标红 `insideKsum-illegal`，无需保存即暴露。

> token 模型：`FormulaToken` 纯新增 `sources`（多 source 链）+ `projectToHostKey`（KSUM），**无 DB 列、无 snapshot 迁移**，缺省即旧 token。前后端求值由共享夹具 `cross-tab-cases.json` 锁一致。

## 4. Excel 模板列配置（L3）

「Excel 视图」按预定义列结构展示报价单/核价单数据，**每个 lineItem（产品行）渲染一行**。

### 4.1 入口、状态机与反查

- 入口：`Excel 模板配置` → 列表 → 点模板名 → 列配置详情（`/costing-templates/:id`）。
- 编辑约束：`DRAFT` 可编辑；`PUBLISHED` / `ARCHIVED` 只读（改 → 列表「派生新草稿」）。
- 反查渲染源（同时满足）：`costing_template.linked_template_id = <报价单 customer_template_id 或 costing_card_template_id>` + `status='PUBLISHED'` + 优先 `is_default=true`，否则取最新。
- **同一 `linked_template_id` 内 `is_default=true + PUBLISHED` 唯一**（DB 约束）。换默认先置旧为 false 或归档。

### 4.2 列的字段 + 8 种列来源

列基本字段：`col_key`（公式中 `[X]` 引用名，建议 A/B/C…）、`title`、`source_type`、来源配置（`variable_path`/`formula`/`field_key`/`fixed_value` 等）、`comparison_tag`（比对分组，可空）、`hidden`。

> **权威来源**：后端 `ExcelViewService.buildRowData` 的 `source_type` switch 支持以下 **8 种**列来源。配置时按需选用：

| source_type | 含义 | 取值来源 |
|---|---|---|
| `VARIABLE` | 取数列（BNF 路径 / `{code}` 简写 / 全局变量 / `$name` 模板 SQL 视图） | 前端 hook 已算好，后端取 `componentRowData[col_key]` |
| `FORMULA` | 模板公式列，`[名称]` 引用，**后端**求值 | `evaluateFormulaColumn`（先查模板公式 Map，再 fallback cachedCells） |
| `EXCEL_FORMULA` | `[X]` 列引用公式，**前端**求值（旧 `LinkedExcelView` 路径） | 后端原样返回公式串给前端 |
| `CARD_FORMULA` | 卡片公式列，跨列依赖批量拓扑求值 | `cardFormulaEvaluator`（按节点聚合，异常吞为 null 不 500） |
| `TAB_JOIN_FORMULA` | 页签连表公式（§4.4） | `TabJoinPlanEvaluator.evaluateColumn` |
| `PRODUCT_ATTRIBUTE` | 产品属性值 | `productAttrs[field_key]` |
| `COMPONENT_FIELD` | 取某组件字段值 | `componentRowData[field_key]` |
| `FIXED_VALUE` | 固定值 | `col.fixed_value` |

#### VARIABLE 列三种写法

| 写法 | 示例 | 何时用 |
|---|---|---|
| **`{code}` 简写**（不查 DB） | `{hf_part_no}` `{product_name}` | lineItem 内存已有字段，性能最好 |
| **BNF 物理路径** | `v_costing_summary_full.material_cost` | 查物理视图（隐式按料号过滤） |
| **`$name` 模板 SQL 视图**（推荐新配方式） | `$summary_full.material_cost` | 模板私有、发布冻结、隔离性最强、dry-run 可校验拒 V44（见下） |

`{code}` 简写（前端 `LinkedExcelView.resolveVariable` 解析）常用映射：`{hf_part_no}`→productPartNo、`{product_name}`→productName、`{specification}`→hfPartInfo.specification、`{customer_drawing_no}`、`{customer_part_no}`、`{subtotal}`、`{base_currency}`（硬编码 USD）、`{system_date}`、其他 `{xxx}`→productAttributeValues['xxx'] 兜底。

#### `$name` 模板 SQL 视图（V249/V250 起，推荐）
模板层独立 `template_sql_view` 表，模板内维护专属 SQL 视图，用 `$<name>.<col>` 引用。对比直引物理视图：发布冻结（历史可复现）、隔离（本模板私有）、dry-run 拒 V44 老表（AP-53）、UI 可维护可预览。
操作：模板配置 → 派生草稿 →「SQL 视图」Tab（`TemplateSqlViewsTab`）→ 新建视图 → dry-run 预览 → 保存 →「列配置」Tab → VARIABLE 列点「🗄 SQL 视图」→ 选视图字段 → 写回 `$name.col`。
纪律：`sql_template` 禁查 V44 老表；禁 `$$comp.view.col` 跨组件语法（模板上下文强阻断）；PUBLISHED 视图只读。

### 4.3 Excel 公式语法（EXCEL_FORMULA 前端求值）

前端 `LinkedExcelView.evaluateFormula`：

| 语法 | 含义 |
|---|---|
| `[X]` | 引用本行 X 列已求值数值 |
| `{code}` | 引用 VARIABLE 列已 resolved 的值 |
| `+ - * / ( )` / `< > <= >= == != && \|\| ? :` | 四则 + 比较 + 逻辑 + 三元 |
| `=` 前缀 | 兼容 Excel 习惯，自动剥离 |

**不支持**：字符串字面量（`"KG"` → 返 `—`）、Excel 函数（`SUM`/`IF`/`VLOOKUP`）、跨行引用。安全闸：表达式只允许 `[\d+\-*/().,\s%<>=!&|?:]`，含其他字符立即返 `—`。
**要显示固定文本**（如 "KG"）→ 改用 VARIABLE 列指向视图硬编码字段（`v_costing_summary_full.weight_unit` 写死 `'KG'::varchar`）。

**求值顺序**（重要）：`LinkedExcelView` 每行两遍扫描——第一遍处理 VARIABLE 列（`{code}` 同步 resolve / BNF path 异步调 `/formulas/evaluate` 进 pathCache）；第二遍处理 FORMULA/EXCEL_FORMULA 列（替换 `[X]`/`{code}` 后运行表达式）。**含义**：EXCEL_FORMULA 只能引用 VARIABLE 列，不能引用其他公式列（无第三遍）；嵌套公式须平铺成单行。

### 4.4 TAB_JOIN_FORMULA — 页签连表公式（2026-06-10）

一列的值由**单卡片内多个页签内容**按行键对齐后算出**一个单值**。列来源选「页签连表公式」→「配置公式」打开 `TabJoinFormulaDrawer`。

```json
{ "col_key":"L", "source_type":"TAB_JOIN_FORMULA",
  "expression":"[投料.金额] * [加工.工时] + [回料(总计)]",
  "tabs":[ {"alias":"投料","tabKey":"<componentId>:0","rowKeyFields":["物料编码"]},
           {"alias":"加工","tabKey":"<componentId>:1","rowKeyFields":["物料编码"]},
           {"alias":"回料","tabKey":"<componentId>:2","rowKeyFields":["物料编码","工序"]} ] }
```

令牌三类：明细 `[别名.字段]`；小计列总计 `[别名.列名(总计)]`；页签总计 `[别名(总计)]`。
行键对齐：页签按各自 `rowKeyFields`（组件里配的行键字段）完全相等分「行键类」；同类页签按行键全外连对齐（行键唯一，不会笛卡尔放大）；**只有同行键类页签的明细才能在一个表达式里逐行运算**。
求值（按顶层 `+ -` 拆加减项）：项内有裸明细字段→整项按对齐行逐行算再求和（默认自动求和，无需写 SUM）；明细全在聚合函数内 / 整项纯标量→只算一次。函数 `SUM/AVG/MIN/MAX/COUNT`；缺值→0、除数 0 或缺→按 1。
试算：构建器选一张已有卡片样本 → 试算（`POST /api/cpq/templates/{id}/excel-view-config/dry-run-tab-formula`）。后端求值器 `com.cpq.quotation.service.tabjoin.TabJoinPlanEvaluator`。

### 4.5 `:spineKeys` 核价复合键过滤（2026-06-06）

仅核价（COSTING）模板的 `template_sql_view.sql_template`。按卡片 BOM 树三元组 `(子件料号, 父件料号, 子件自身当前 BOM 版本)` 精确过滤，解决单列 `:hfPartNos` 无法区分同料号在不同父件/版本下多 occurrence。

```sql
WHERE :spineKeys(<子件料号列>, <父料号列>, <版本列>)
```

框架自动展开为 NULL-safe `EXISTS + unnest + IS NOT DISTINCT FROM`（三数组绑定，绕开 PG 65535 参数上限）。三元组由 `BomClosureService` 从卡片 BOM 闭包派生并环境注入：核价卡片展开上下文有值；其他上下文（报价单渲染 / standalone dry-run）三数组为空 → `EXISTS` 恒 false → 匹配 0 行（安全降级）。
版本口径：每节点取自身 `is_current` BOM 版本；叶子节点 NULL。
限制：`:__sk*` 前缀框架保留禁自定义；实参不支持含括号外层逗号的字符串字面量；同 sql_template 可多次使用共享三数组；dry-run 展开为 `ARRAY[]::text[]` 校验形，不进 `required_variables`。

### 4.6 实操：按 23 列样例配核价汇总模板

对应导入 Excel「汇总」页签 1:1。步骤：① 派生草稿 → ② 关联核价模板 → ③ 配列 → ④ 保存 → 发布。关键列对照（节选）：

| col_key | title | source_type | variable_path / formula |
|---|---|---|---|
| A | 宏丰料号 | VARIABLE | `{hf_part_no}` |
| B | 品名 | VARIABLE | `{product_name}` |
| L | 材料成本 | VARIABLE | `v_costing_summary_full.material_cost`（推荐改 `$summary_full.material_cost`） |

| **U** | **总成本** | **EXCEL_FORMULA** | `=[L]+[M]+[N]+[O]+[P]+[Q]+[R]+[S]+[T]`（9 列求和，前端求值） |

## 5. 公式体系总览（三套语法的边界）

系统有**三套独立公式语法**，互不通用，按配置位置选：

| 公式 | 配置位置 | 语法 | 引擎 |
|---|---|---|---|
| **组件 FORMULA token** | L2 组件 FORMULA 字段 | token 数组（见 §5.1） | 前端 `formulaEngine.ts` + 后端 `FormulaCalculationService`/`FormulaCalculator` |
| **Excel `[X]` 公式** | L3 EXCEL_FORMULA 列 | `[列key]` + 四则三元 | 前端 `LinkedExcelView.evaluateFormula` |
| **LIST_FORMULA 条件分支** | L2 LIST_FORMULA 字段 | `[字段] op X` 条件 + `[字段]`/`{path}`/`{GV}` 公式 | `conditionEngine.ts` + `evaluateListFormulaString` |

### 5.1 组件公式 token 完整清单（14 种）

> **权威来源**：前端 `formulaEngine.ts` 的 token case（最全 14 种，含内部 `b_field`）；后端 `FormulaCalculationService` 实现其中 9 种（field/number/operator/bracket_open/bracket_close/component_subtotal/datasource_field/global_variable/product_attribute），`cross_tab_ref` / `previous_row_subtotal` / `quotation_field` / `path` / `b_field` 由卡片引擎 `FormulaCalculator` 等求值。前后端语义对等。

| token type | 用途 | 必填字段 |
|---|---|---|
| `field` | 引用同行其他字段 | `value`（字段名） |
| `number` | 数字常量 | `value` |
| `operator` | `+ - * /` | `value` |
| `bracket_open` / `bracket_close` | 括号 | - |
| `path` | BNF 路径标量直取 | `path` |
| `component_subtotal` | 跨组件小计 | `component_code` 或 `tab_name` |
| `previous_row_subtotal` | driver 上一行 subtotal（行 0 兜底） | `fallback_component_code` |
| `cross_tab_ref` | 跨页签 VLOOKUP/SUMIF（§3.4） | `source` + `match` + (`target` 或 `targetExpr`) + `agg` |
| `b_field` | B 当前行某列（**仅用于 `cross_tab_ref` 的 `targetExpr` 内部**，非顶层可配） | `value`（列名） |
| `product_attribute` | 产品属性 | `attribute_name` |
| `quotation_field` | 报价单级字段（税率等） | `value` |
| `datasource_field` | 同行 DATA_SOURCE 字段解析值 | `name`（字段名） |
| `global_variable` | 全局变量 | `code` + (`key_values` 或 `key_field_refs`)，运行时走编译产物 `path` |

> **重要**：FORMULA token 引用 DATA_SOURCE 字段必须用 `datasource_field`（不要用 `field`）；`global_variable` 的 `path` 由前端编译，不要手写。

### 症状 → 根因速查

| 症状 | 根因 | 处理 |
|---|---|---|
| `首值（共N项）` 后缀 | ImplicitJoinRewriter 没注入 partNo → 全表多行 | 视图 DDL 后重启 Quarkus 清进程缓存；或视图缺收窄键加 PIVOT（§9/§12） |
| 产品卡 cell `(共N项)` 但数据真单行 | 渲染层 fallback 漏读 `row[fieldKey]` | `ComponentCell` BASIC_DATA 分支补 row 优先级（AP-22 类别 B） |
| 数值全 `0` 但语义是「无数据」 | 视图 `COALESCE(_, 0)` 遮蔽 NULL | 视图改 NULL 传递，前端识别全 NULL 行整行清空 |
| 比对 tag 全归「其他」组 | `comparison_tag` 表未注册 | SQL 注册 (code, label, group_name, sort_order) |
| 列「加载中…」永不消失 | `/formulas/evaluate` 报错 / 路径未预热 | F12 Network 看 500/超时；查路径采集（AP-31/AP-37） |
| FORMULA / EXCEL_FORMULA 永远 `—` | 公式含非法字符（`"KG"`/`SUM`/`IF`） | 改用 VARIABLE 取数 |
| 改组件字段报价单还是旧配置 | snapshot 没同步 | H1 自动；或手工 `POST /components/{id}/refresh-template-snapshots`（§3.7） |

```
[后端] FormulaEvaluateResource → FormulaEngine.evaluate → resolvePathValue
   ↓ EvaluationContext {dataLoader, partNo, customerId, driverRow, bindings}
   ↓ dataLoader.loadByPath(path, driverRow, partNo, customerId)
[Rewriter] ImplicitJoinRewriter.rewriteWithContext
   - 合并 effective driver: {hf_part_no:partNo, customer_id:customerId} ∪ driverRow
   - 查目标视图 information_schema.columns（V112 起空集不缓存）
   - 注入「目标视图存在 + 原谓词未用 + 非黑名单」的列
   ↓ rewritten: v_costing_summary_full[hf_part_no='3100080003'].pure_material_cost
[Compiler] CachedSqlCompiler → SQL → [JDBC] PreparedStatement → PostgreSQL
   ↓ 单行→标量 / 单行多列→Map / 多行→List<Map>
[前端] formatPathValue 兜底: 单值直返 / List>1→「首值（共N项）」 / null→「—」
```

## 9. 隐式 JOIN 机制 + 系统列黑名单 + 缓存纪律

`ImplicitJoinRewriter`（`com.cpq.formula.dataloader`）渲染时把 effective driver（`{hf_part_no, customer_id}` ∪ driverRow）里**目标视图存在、原谓词未用、非黑名单**的列注入为 AND 谓词，实现「不写谓词自动按料号/客户/行键收窄」。

**系统列黑名单（实际 12 列，`SYSTEM_COLUMN_DENYLIST`）**：
```
# 审计追溯
id, created_at, updated_at, created_by, updated_by, version, deleted_at, is_deleted
# 导入溯源
import_record_id, imported_by
# 生命周期 / 状态（不是业务键，注入会过度收窄）
status, is_current
```
（这些不是业务键，注入会把多行误收窄成驱动行自己。）

**缓存纪律（视图 DDL 后必须重启 Quarkus）**：`ImplicitJoinRewriter` 的列缓存、`CachedSqlCompiler`、`CachedPathParser` 都是进程级 ApplicationScoped 缓存。`DROP VIEW ... CASCADE` 瞬间若有请求触发 `getColumns` → 旧版本会缓存空集 → 永久不再注入谓词 → 视图返全表 N 行 →「(共N项)」。V112 后空集不缓存可自愈，但残留旧 JVM 进程仍需重启。**任何 schema DDL 后 touch java 强制重启**。

## 10. snapshot 冻结与同步机制

三处持久化形成「源 → 冻结」链：

| 层 | 列 | 行为 |
|---|---|---|
| 组件源 | `component.fields` | 唯一可编辑源 |
| 模板冻结 | `template.components_snapshot` + `template.sql_views_snapshot` | publish 定格 |
| Tab 级覆盖 | `template_component.fields_override` | 每 Tab 独立覆盖 |
| 报价冻结 | `quotation_*_snapshot` | SUBMITTED 定格，历史可复现 |

- **H1 自动同步**：`ComponentService.update` → `TemplateService.refreshSnapshotsByComponent`，刷所有引用方 snapshot。
- ⚠️ **同 cid 多 tc 实例必须按 sortOrder 精确匹配**（AP-40），禁用 `firstResult()`——否则 A Tab 的 `fields_override` 反向污染 B Tab，配置永久丢失。
- schema 迁移改字段形态时三处 JSONB 列必须同步（AP-39）。

## 11. 字段类型联动协议（AP-44 核心规范）

### 组合产品（SIMPLE/COMPOSITE）现状

## 12. 多行 driver 展开链路 + 反模式索引

多行 driver 在「配置 → 求值 → 渲染」三段任一断裂都显示 `X (共N项)`。四个收窄关卡：

```
SQL 物理表/视图        ← 关卡1: WHERE 谓词收窄
   ↓ ImplicitJoinRewriter  ← 关卡2: 隐式注入 hf_part_no/customer_id/driver row 同名列
   ↓ useDriverExpansions    ← 关卡3: driver 展开后每行单独 evaluate
   ↓ ComponentCell 渲染      ← 关卡4: BASIC_DATA cell 四级回退链（row[fieldKey] > globalPathCache）
   ↓ formatPathValue 兜底    ← list>1 → "首值（共N项）"
```

配置一个新多行 driver 组件 checklist：driver 视图含 `hf_part_no` 列；driver 行含字段路径目标表业务键（否则写带谓词 path）；跨表字段路径目标视图含 driver 行同名列；`rowCount=0` 时 BASIC_DATA cell 走 `—` 兜底（不盲降 globalPathCache，AP-38）。
完整反模式见 `反模式.md` AP-22（多行展示族 4 类）/ AP-31（加载中族）/ AP-37/38/40~54。
