# 配置中心架构

> 2026-05-18 V190 重构后的新基准。本文是项目"唯一真理源"原则的承载文档：所有业务渲染（报价单 / 核价单 / Excel 视图）必须从配置中心读，零硬编码。

## 一、三层模型

```
┌────────────────────────────────────────────────────────┐
│              配置中心（唯一真理源）                  │
├────────────────────────────────────────────────────────┤
│  全局变量定义 + 值（global_variable_definition +     │
│                      global_variable_value）          │
│  组件配置（component.fields JSON）                   │
│  模板配置（template.components_snapshot JSON）       │
│  公式定义（formula_tokens）                          │
└────────────────────────────────────────────────────────┘
                          ↓
        ┌─────────────────┼─────────────────┐
        ↓                 ↓                 ↓
   报价单 渲染      核价单 渲染      Excel 视图渲染
   (零业务逻辑)     (零业务逻辑)     (零业务逻辑)
```

**核心原则**：业务渲染层不做任何字段语义推断、不硬编码字段映射、不二次解析配置。配置层定义→运行时按定义渲染。

## 二、字段类型 × 数据来源（field_type 矩阵）

| field_type | 数据来源 | 渲染形态 | 关键配置 |
|---|---|---|---|
| `FIXED_VALUE` | 字面量 | 只读单元格 | `content` |
| `INPUT_TEXT` | 用户输入 | 文本框 | (可选 `default_source` 默认值) |
| `INPUT_NUMBER` | 用户输入 | 数字框 | `content` 静态兜底 + 可选 `default_source` |
| `BASIC_DATA` | BNF 路径查物理表/视图 | 只读单元格 | `basic_data_path` |
| `DATA_SOURCE` | 多类型数据源 | 只读 / 下拉选 | `datasource_binding.type` + 各类型配置 |
| `FORMULA` | 公式引擎计算 | 计算结果只读 | `formula_tokens` |

### INPUT_NUMBER / INPUT_TEXT 默认值兜底链

```
用户输入(row[key]) → default_source 解析 → field.content 静态兜底 → 0
```

`default_source` 三种类型：

```jsonc
// 1. 引用全局变量（最常见，自动按 driver row 取动态 key）
{
  "type": "GLOBAL_VARIABLE",
  "code": "PROCESS_DEFAULT_YIELD",
  "key_field_refs": {}  // 留空 = 按 def.keyColumns 同名取 driver row 字段
                        // 显式映射: {"process_code": "工序代码"}
}

// 2. 引用 BNF 路径（与 BASIC_DATA 同源）
{
  "type": "BNF_PATH",
  "path": "v_costing_element_price[element_code='Cu'].costing_price"
}

// 3. HTTP API 调用（Phase D 引入骨架, 暂未实现）
{
  "type": "HTTP_API",
  "api_config": { /* method/url/headers/body 模板 */ }
}
```

## 三、全局变量（V190 单表模型）

### Schema

```sql
-- 定义表（5 字段决定一切）
global_variable_definition (
    code         VARCHAR(64) PK,        -- 唯一业务编码
    name         VARCHAR(100),          -- 中文名
    var_type     VARCHAR(20),           -- LOOKUP_TABLE | SCALAR
    key_schema   JSONB,                 -- LOOKUP 时声明 key 列结构, SCALAR 时 []
    value_type   VARCHAR(20),           -- NUMBER | STRING (TODO)
    unit         VARCHAR(20),
    value_source_type VARCHAR(32),      -- KV_TABLE | COSTING_VIEW
    visibility   VARCHAR(32),           -- PUBLIC | COSTING_INTERNAL
    source_view  VARCHAR(100),          -- COSTING_VIEW 时填; KV_TABLE 时空
    ...
)

-- 值表（单一通用 KV 存储 — 所有变量值共用这张表）
global_variable_value (
    var_code   VARCHAR(64) FK,
    key_id     VARCHAR(200),     -- SCALAR='_'; LOOKUP=key 列拼接（'Z350' / 'CNY:USD'）
    key_values JSONB,            -- 结构化保留（{"process_code":"Z350"}）
    value_number NUMERIC(20,4),  -- value_type=NUMBER 用这列
    value_text   TEXT,           -- value_type=STRING 用这列
    note         TEXT,
    updated_at   TIMESTAMPTZ,
    PRIMARY KEY (var_code, key_id)
)

-- 变更日志（保留 V106 现状）
global_variable_change_log (...)
```

### value_source_type 两态

| value_source_type | 用途 | 数据物理位置 | UI 维护 |
|---|---|---|---|
| `KV_TABLE` | 轻量配置（业务无关） | `global_variable_value` 单表 | 全局变量管理页 |
| `COSTING_VIEW` | 核价价格主数据（业务相关、版本化） | `costing_*_price` 物理表 + 视图 | 核价模块页面 |

**判断准则**：
- 有版本化（PUBLISHED/DRAFT 切换） + 跟特定业务（核价）强绑 → `COSTING_VIEW`
- 简单 KV 配置（业务无关 / 默认值 / 阈值 / 系数） → `KV_TABLE`

### visibility 两态

| visibility | 全局变量页 | Picker 选择器 | 维护数据 |
|---|---|---|---|
| `PUBLIC` | ✅ 默认显示 | ✅ 显示 | ✅ 可编辑 |
| `COSTING_INTERNAL` | ❌ 默认隐藏（toggle 开关查看） | ✅ 显示 + 「核价」tag | ❌ 禁用 + 提示去核价模块 |

## 四、报价单渲染数据流（成材率字段示例）

```
template.components_snapshot
  └─ COMP-CFG-PROCESS 组件
      └─ fields[i]: { name:"成材率", field_type:"INPUT_NUMBER",
                       content:"100",  ← 静态兜底
                       default_source:{type:"GLOBAL_VARIABLE", code:"PROCESS_DEFAULT_YIELD"} }
                                       ↓
                                       
[1] enrichComponentData (QuotationWizard) 把 default_source 透传到 lineItem.fields
                                       ↓
                                       
[2] useDriverExpansions 调 batch-expand 后端
                                       ↓
                                       
[3] ComponentDriverService.expand:
    - 解析 fields[i].default_source.GLOBAL_VARIABLE 任务
    - 对每个 driver row (mat_process 行) 调
      globalVariableService.resolveValue('PROCESS_DEFAULT_YIELD', {process_code: driverRow.process_code})
    - 结果塞入 basicDataValues['@gvar:PROCESS_DEFAULT_YIELD'] = 解出的值
                                       ↓
                                       
[4] GlobalVariableService.resolveValue:
    - 按 def.value_source_type 分发
      · KV_TABLE: SELECT value_number FROM global_variable_value WHERE var_code=? AND key_id=?
      · COSTING_VIEW: SELECT value_column FROM source_view WHERE key=?
                                       ↓
                                       
[5] 前端三级兜底链 (computeAllFormulas + placeholder 渲染):
    用户输入 → basicDataValues['@gvar:PROCESS_DEFAULT_YIELD'] → field.content="100" → 0
```

## 五、加新全局变量的标准 SOP

### 形态 1：纯 UI 操作（90% 场景）— G1~G3 已落地

1. 打开「全局变量管理」页 → 点「**新建变量**」按钮
2. 填写 code (大写英文+数字+下划线) / 名称 / 类型 (LOOKUP_TABLE 或 SCALAR) / key 列名 / 单位
3. 在「维护数据」抽屉添加 (key, value) 行
4. 打开「组件管理 → 字段配置」，找到目标 INPUT_NUMBER/INPUT_TEXT 字段，点「**+ 默认值来源**」按钮
5. 选 type=GLOBAL_VARIABLE，选刚创建的变量；key_field_refs 留空走同名映射
6. 保存组件 + 同步 template.components_snapshot

**零 Flyway，零 Java**。

### 形态 2：Flyway 注册（暂行方案，UI「新建变量」按钮上线前）

```sql
-- 1. 注册变量定义
INSERT INTO global_variable_definition
  (code, name, var_type, key_schema, value_type, unit, value_source_type, visibility,
   key_columns, value_column)
VALUES
  ('YOUR_CODE', '你的变量', 'LOOKUP_TABLE', '[{"name":"key1","label":"键 1","type":"STRING"}]'::jsonb,
   'NUMBER', '%', 'KV_TABLE', 'PUBLIC',
   '["key1"]'::jsonb, 'value_number');

-- 2. Seed 数据
INSERT INTO global_variable_value (var_code, key_id, key_values, value_number)
VALUES ('YOUR_CODE', 'k1_value', '{"key1":"k1_value"}'::jsonb, 100.0000);

-- 3. 组件字段 JSON 加 default_source
UPDATE component SET fields = ... WHERE id = ...;

-- 4. 同步 template snapshot
DO $$ ... END $$;
```

## 六、Template Snapshot 同步机制 (H1)

### 设计原则

**配置中心是真理源, snapshot 是缓存视图。** 当组件配置 (fields / formulas / dataDriverPath) 发生平台级变化时, 所有引用该组件的 template.components_snapshot 必须立即同步, 不能让"已发布模板的版本快照不可变"约定阻挡架构演进。

### 触发方式

| 触发点 | 行为 | 适用场景 |
|---|---|---|
| **组件保存** (`PUT /api/cpq/components/{id}`) | 自动调 `TemplateService.refreshSnapshotsByComponent` | 日常组件配置修改 |
| **手工管理端点** (`POST /api/cpq/components/{id}/refresh-template-snapshots`) | 同上 | 历史模板修复 / 数据迁移补偿 |
| **Flyway 迁移** (V184~V187 历史) | 手工 `DO $$` 循环 | **不再推荐** — 用上面两种 |

### 行为细节

```
refreshSnapshotsByComponent(componentId):
  1. SELECT id FROM template WHERE components_snapshot::text LIKE %componentId%
  2. for each template:
       parseJson(components_snapshot) → list of {componentId, fields, formulas, ...}
       匹配 componentId 的项 → 覆盖 fields/formulas/data_driver_path/componentType/componentName/componentCode
       toJson 写回 template.componentsSnapshot
  3. 返回受影响 template.id 列表
```

**不变量**：
- template.version 不变（保留版本号语义）
- 模板创建时间 / publish 状态 / FK 关系不变
- 只重写 components_snapshot 内字段元数据

**事务**：与组件 update 同事务，失败不阻断组件保存（仅 LOG.warn）。

### 模板级覆盖（V200/V204）— `template_component.fields_override` / `data_driver_path_override`

**动机**：同一 component 在不同模板/不同 Tab 需要不同字段集或不同 driver。典型场景：
- 同 componentId `e42185ec`（COMP-CFG-MATERIAL-RECIPE）既挂在 COMPOSITE 模板的"材质" Tab（多 1 列"子件"，driver = `v_composite_child_materials` 聚合子件），又挂在同模板的"选配-材质" Tab（5 列基础字段，driver = `v_part_material_recipe`）。
- 同模板 v1.9 中有 4 对这种 cid 复用关系（材质/工序/元素含量/组合工艺 × 标准/选配-*）。

**两列语义**：
| 列 | 类型 | null 含义 | 非空含义 |
|---|---|---|---|
| `fields_override` | jsonb（字段数组 JSON） | 走 `component.fields`（默认） | 全量覆盖：publish 时把这数组直接放进 components_snapshot |
| `data_driver_path_override` | text | 走 `component.dataDriverPath`（默认） | publish 时用这字符串覆盖 |

**publish 规则**（`TemplateService.java:221-228`）：
```java
String effectiveFields = (tc.fieldsOverride != null && !tc.fieldsOverride.isBlank())
        ? tc.fieldsOverride : comp.fields;
String effectiveDriverPath = (tc.dataDriverPathOverride != null && !tc.dataDriverPathOverride.isBlank())
        ? tc.dataDriverPathOverride : comp.dataDriverPath;
```

**clone 升版本**（`TemplateService.java:387`）：`newTc.fieldsOverride = stc.fieldsOverride;` — override 原样复制。所以：在组件管理改了 component.fields，clone 出来的 v2.0 草稿如果原本有 fields_override，**新加的字段不会进 v2.0 snapshot**，必须先清空 override（或编辑 override）才能让组件变更生效。

**UI 入口（V204 新增）**：模板编辑页 → Tab 顶部工具栏 → 「⚙ 编辑字段 / Driver 覆盖」按钮 → `OverridesDrawer`：
- Driver Path 开关：关闭=走组件默认；打开=显示输入框 + "重置为组件默认" 按钮
- 字段定义开关：关闭=走组件默认字段集；打开=显示 `FieldConfigTable`（复用组件管理同款编辑器），含「复制组件默认字段重新开始」快捷按钮
- 仅 DRAFT 可改，PUBLISHED 只读查看
- DATABASE_QUERY 类数据源（datasource_id）仍需去组件管理配，其他字段类型（LIST_FORMULA / FORMULA / BNF_PATH / GLOBAL_VARIABLE / FIXED_VALUE / INPUT…）抽屉内可直接编辑

**API**：
| Endpoint | 行为 |
|---|---|
| `GET /api/cpq/templates/{id}` | 返回的 `components[i]` 含 `fieldsOverride` / `dataDriverPathOverride`（V204 起 TemplateComponentDTO 暴露） |
| `GET /api/cpq/templates/{tid}/components` | 同上 |
| `PATCH /api/cpq/templates/{tid}/components/{tcId}/overrides` | body `{ fieldsOverride?, dataDriverPathOverride? }`. 键缺省=不动；显式 null=清空走默认；非空=设置。DRAFT 才允许；PUBLISHED 返 400 |

**Tab 头视觉**：当 tc 有 override 时，Tab 标签自动显示橙色「覆盖」徽章；Tab 内顶部工具栏显示当前 effective driver 路径 + 「已覆盖」标签。

**反模式提醒**：不要靠改 component.fields 来"让某个 Tab 字段集变化"——如果该 Tab 已有 fields_override，组件改动对它无效。正确做法是去模板编辑里改该 Tab 的 fields_override，或先清空 override。

```json
{
  "global_variables": {
    "definitions_total": 5,
    "definitions_kv_table": 2,
    "definitions_costing_view": 3,
    "values_total": 30

  "structure": {
    "components_total": 112,

  },
  "status": "OK"

```

用途：
- 运维监控（接入 Prometheus / Grafana 之类）
- PR 自检：迁移前后对照 definitions_total / values_total 变化

### 一键刷新 — `POST /api/cpq/config-center/refresh-all-snapshots`

仅 `SYSTEM_ADMIN` 可调。遍历所有 template.components_snapshot 含到的 componentId 集合，逐个调用 `refreshSnapshotsByComponent`。

返回：
```json
{
  "components_processed": 8,
  "templates_touched_total": 12,
  "errors": 0,
  "per_component": { "<componentId>": <touched_count>, ... }
}
```

适用场景：
- schema 大变更后批量修复（替代 V184~V187 手工 `DO $$`）
- 数据迁移补偿
- 偶发 snapshot 不一致诊断

## 八、公式 token datasource_field (K1)

公式 token 现支持引用同行 DATA_SOURCE 字段的解析结果：

```json
{
  "field_type": "FORMULA",
  "formula_tokens": [
    { "type": "datasource_field", "name": "实时汇率" },
    { "type": "operator", "value": "*" },
    { "type": "field", "value": "美元单价" }
  ]
}
```

求值期 fieldValues 应已含 DATA_SOURCE 字段的解析值（前端 `computeAllFormulas` 在公式求值前先解 DATA_SOURCE 写入 row）。

## 九、禁止事项

❌ **不要为新全局变量建独立物理表** — 一律走 `global_variable_value` 单表  
❌ **不要硬编码 PHYSICAL Map** — V190 已删, 不再有 Java 路由表  
❌ **不要在业务渲染层做字段语义推断** — 字段渲染必须按 field_type 走通用分发  
❌ **不要让数据源类型固化在代码** — 用 `datasource_binding.type` 显式区分  
❌ **不要绕过缓存失效闭环** — 全局变量写入必走 `GlobalVariableService.upsertEntry`, 该方法已含 `invalidateDependentCaches`  
❌ **不要让核价价格表混进「全局变量」UI 编辑** — 核价模块独立维护, 通过 `COSTING_INTERNAL` 标记隔离  
❌ **不要再写 `DO $$` 循环刷 template snapshot** (H1 已自动化) — V184~V187 历史包袱不要重演

## 九、同模板双轨支持组合产品 (2026-05-20)

模板被同时用于 SIMPLE 独立产品 + COMPOSITE 组合产品时, BASIC_DATA 字段 + `data_driver_path` 必须**双轨配置** — JSONB 内单字段加 `_composite` 后缀, 运行时按 `lineItem.compositeType` 切换 effective path. 后端透明 (前端发送时 path 已确定), 无 Flyway。

### 存储格式

```jsonc
// 字段层
{
  "basic_data_path": "mat_process.process_code",                       // SIMPLE
  "basic_data_path_composite": "v_composite_child_processes.process_code"  // COMPOSITE
}

// template_component 层
{
  "data_driver_path": "mat_process",                                  // SIMPLE
  "data_driver_path_composite": "v_composite_child_processes"         // COMPOSITE
}
```

- `POST /api/cpq/templates/admin/{id}/delete-tcs` — 删冗余 Tab (体)
- `POST /api/cpq/templates/admin/{id}/patch-composite` — 注入双轨字段 (patch / replace 双模式)
