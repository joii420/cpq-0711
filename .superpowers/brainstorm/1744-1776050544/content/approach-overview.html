CPQ V1 实施方案：单人全栈开发策略

三种实施路径的核心差异在于模块切分和风险管控方式

A

垂直切片法（推荐）

核心思路：按业务价值链垂直切割，每个阶段交付一个可运行的端到端功能闭环

6 个阶段：
P0 项目骨架 + 认证 → P1 主数据 + 数据源 → P2 配置中心（组件→模板→关联）→
P3 计算引擎 POC + 集成 → P4 报价全流程（生成→审批→输出）→ P5 收尾打磨

优势

  • 每阶段可演示、可测试
  • 风险前置（P3 引擎验证）
  • 单人开发节奏可控

挑战

  • 需要严格控制每阶段不回头改
  • 前期看不到报价核心功能
B

核心优先法

核心思路：先用最小数据做通报价全流程骨架，再逐步丰富各模块

路径：
骨架 + 硬编码数据 → 报价生成器原型 → 逐步替换为真实模块 → 补全管理功能

优势

  • 最快看到核心报价流程
  • 早期可获取用户反馈

挑战

  • 大量返工（硬编码→真实数据）
  • 单人开发返工成本高
  • 架构容易变形
C

按 PRD 里程碑原序

核心思路：严格按 PRD M0-M7 顺序，但调整为单人节奏

路径：
M0→M1→M2→M2b→M3→M4a→M4b→M5→M6→M7（与PRD一致，去除并行假设）

优势

  • 与PRD完全对齐，沟通成本低
  • 里程碑清晰

挑战

  • PRD排期假设团队分工，单人会拉长
  • M2b数据源管理排在配置中心前面，但实际可以延后
推荐方案 A（垂直切片法）的理由：
  • 单人开发最忌返工 — 方案B的"先原型后替换"对团队合理，对单人是双倍工作量
  • PRD 已足够详细 — 不需要通过原型验证需求，可以直接按 PRD 建设
  • 数据管道自底向上 — 组件→模板→报价的依赖链决定了必须先建基础
  • 引擎风险需要早期隔离 — Drools POC 在 P3 独立验证，不阻塞前序工作
