# CPQ 规则引擎

一个为工业 CPQ 设计的、零依赖、纯 TypeScript 的规则解释器。
**故意不引入 OR-Tools / Z3 / Drools**——按你们的"BOM 加法 + 公式计算"业务模型,这就是最合适的工具。

## 为什么是这个,而不是 solver

> 见前面架构讨论:你们的模型是范式 A(BOM 累加 + Excel 公式),不是范式 B(组合搜索)。
> Solver 解决的 reverse search / 自动优化 / bin packing 问题,你们前 12 个月碰不到。

这个引擎做且只做四件事:
1. **正向配置 + 校验**(给定已选,过滤剩余可选,标记禁用原因)
2. **BOM 累加**(规则触发时往 BOM 添加工序行)
3. **3D 显隐控制**(BOM 驱动 Babylon 场景重建)
4. **价格调整**(基础公式之外的 markup / 折扣)

定价主体走客户 Excel + HyperFormula,这个引擎只负责**规则触发后的离散副作用**。

## 核心设计决策

### 1. Kleene 三值逻辑 (true / false / unknown)

部分配置时,二值逻辑会让 `not (material == stainless)` 在 material 未选时错误地求值为 true,
触发本不该触发的规则。Kleene 保证 `not unknown = unknown`,**规则只在条件 definitely true 时触发**。

```typescript
evalCondition(C.eq('a', 1), {})              // → 'unknown'
evalCondition(C.not(C.eq('a', 1)), {})       // → 'unknown'  (不是 true!)
evalCondition(C.and(C.eq('a', 1), C.eq('b', 2)), { a: 1 })  // → 'unknown'
```

### 2. 来源追溯 (source + version)

每条规则必须声明 `source: 'excel' | 'manual' | 'ai_suggested' | 'system'` 和 `version`。
评估结果里每个 effect 都关联到具体规则 id,**可以一路追溯到 Excel 哪个单元格或哪个 AI 建议**。

这是审计、回放、AI 规则人工 review 的基础设施。

### 3. 冲突显式检测

四种冲突会被显式报告,**不会静默吞掉**:

| 冲突类型 | 触发场景 |
|---|---|
| `require_conflict` | 两条规则要求同一属性的不同值 |
| `require_vs_disable` | 一条规则要求的值被另一条禁用 |
| `visibility_conflict` | 同一 mesh 被 show 又被 hide |
| `unit_cost_mismatch` | 同一 SKU 在不同规则里给了不同单价 |

冲突不阻止评估完成——继续给出尽量完整的结果,UI 决定怎么提示用户。

### 4. JSON 友好

所有类型都是 JSON 可序列化:
- 规则可以存数据库、Git、Excel 反推出来
- 评估结果可以缓存、回放、做审计快照
- AI 生成规则时直接输出这个 JSON 结构,不需要中间表示

### 5. 纯函数

`evaluate(rules, selection)` 是纯函数——同输入永远同输出,无副作用。
适合做 React/Vue 的 derived state、做服务端缓存、做单元测试。

| 规则数 | 评估耗时 (V8) |
|---|---|

## API 速览

### 写规则

```typescript
import { rule, C, E } from './engine';

const r = rule({
  id: 'R-FAST-001',
  description: '304 不锈钢不可镀锌',
  when: C.eq('material', 'SS304'),
  then: [E.disable('finish', ['zinc_white', 'zinc_color'], '电化学不兼容')],
  source: 'manual',
  version: '1.0.0',
  tags: ['compatibility'],
});
```

### 求值

```typescript
import { evaluate } from './engine';

const result = evaluate(rules, { material: 'SS304', size_mm: 10 });

// result.firedRules        → 触发的规则列表
// result.disabled          → 哪些属性的哪些值被禁用,带原因
// result.required          → 哪些属性被强制为某值
// result.bom               → 累加的 BOM 行
// result.priceAdjustments  → 价格调整列表
// result.visibility        → 3D mesh 显隐
// result.conflicts         → 检测到的规则冲突
// result.pending           → 因属性缺失未决的规则 (UI 提示用)
```

### UI 友好查询

```typescript
import { validOptionsFor, isConfigurationValid, applyPriceAdjustments } from './engine';

// 给配置器下拉框用
const finishOptions = validOptionsFor('finish', allFinishes, result);
// → [{ value: 'zinc_white', valid: false, reason: '电化学不兼容' }, ...]

// 下单前最终校验
const { valid, reasons } = isConfigurationValid(selection, result);

// 应用价格调整 (fixed 先 → percent 后)
const { finalPrice, breakdown } = applyPriceAdjustments(basePrice, result);
```

## 五种 Effect

| Effect | 用途 | 合并语义 |
|---|---|---|
| `disable` | 禁用某属性的某些值 | 列表并集,保留第一个 reason |
| `require` | 强制某属性必须为某值 | 同值合并,异值 → require_conflict |
| `addLineItem` | 往 BOM 追加行 | 三种模式: unique / sum / replace |
| `setVisibility` | 控制 3D mesh 显隐 | 同值合并,异值 → visibility_conflict |
| `priceAdjust` | 加价/减价 (fixed 或 percent) | 全部累加 |

## 后续扩展点 (按优先级)

短期 (前 3 个月):
- [ ] **规则 lint 工具**: 检测规则集里的死规则、永远冲突的规则对、覆盖率低的属性
- [ ] **Excel → 规则的反推管线**: vision LLM + 模板,从客户 Excel 自动生成 ai_suggested 规则
- [ ] **规则人工审核 UI**: 让产品工程师查看 AI 建议、确认后改 source 为 'excel'
- [ ] **审计快照**: 每次报价存档当时的规则集 hash + selection + result,合规可回放

中期 (3-6 个月):
- [ ] **增量评估**: 只重算受变化属性影响的规则 (规模超 1000 条后才需要)
- [ ] **规则解释链**: "为什么 finish 被禁用?" → 给出从触发条件到 effect 的完整推理链
- [ ] **规则覆盖率报告**: 历史报价里多少规则从未触发过 → 候选删除

长期 (6 个月+,且仅当真的需要):
- [ ] **反向求解适配层**: 仅当某个客户付费要求 "我给需求,你给方案" 时,接入 OR-Tools/Z3
- [ ] **自动排布**: 仅当做电气柜业务且承诺 auto-layout 时,接入 bin packing solver

**这些扩展点都不应阻塞 MVP。**先用现在这个引擎跑通一个真客户。
